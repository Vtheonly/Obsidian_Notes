# 5. Cost Estimation & Calculations

To verify if a plan is better, we calculate its **Cost**. In databases, cost is usually measured in **I/O (Input/Output)** units (pages read/written) or cardinality (number of tuples processed).

## Basic Formulas

Let $|R|$ be the number of rows in table $R$.
Let $|S|$ be the number of rows in table $S$.

### 1. Cartesian Product ($R \times S$)

- **Result Size:** $|R| \times |S|$
- **Example:** If $|R|=1,000$ and $|S|=1,000$, result = $1,000,000$ rows.
- _This is why we hate Cartesian products._

### 2. Selection ($\sigma$)

- **Cost:** Depends on **Selectivity** (probability a row satisfies the condition).
- **Formula:** $|Result| = |R| \times Selectivity$
- **Example:** Filtering for a unique ID? Selectivity $\approx 1/|R|$. Result = 1 row.

### 3. Join ($R \bowtie S$)

- **Result Size:** Difficult to predict exactly without statistics, but generally:
  $0 \le |Result| \le |R| \times |S|$
- **For Foreign Key Joins:** The result size usually equals the size of the "Child" table (the one with the foreign key).

## Calculation Example (From TD N°4)

**Scenario:**

- `Train` Table: 60,000 rows.
- `Wagon` Table: 200,000 rows.
- Query: Find the wagon type for Train #4002.

**Approach A (Naive / Bad): Join then Filter**

1.  **Join:** Join `Train` (60k) and `Wagon` (200k).
    - Assuming 1 Train has roughly $200,000 / 60,000 \approx 3.3$ wagons.
    - The join processes $60,000 + 200,000$ reads (best case) or $60k \times 200k$ comparisons (worst case nested loop).
    - Intermediate Result: 200,000 rows (all wagons attached to trains).
2.  **Filter:** Scan 200,000 rows to find Train #4002.
3.  **Total Rows Processed:** Massive overhead.

**Approach B (Optimized): Filter then Join**

1.  **Filter `Train`:** Selection $\sigma_{NoTrain=4002}(Train)$.
    - Input: 60,000 rows.
    - Output: **1 row** (since Train ID is likely unique or highly selective).
2.  **Join:** Join that **1 row** with `Wagon`.
    - Using an index on `Wagon.NoTrain`, we find the matching wagons instantly.
    - Intermediate Result: ~3 or 4 rows (wagons for that train).
3.  **Total Rows Processed:** Negligible.

**Conclusion:**
Pushing the selection down reduced the input to the join from 60,000 to 1. This is an exponential performance gain.
