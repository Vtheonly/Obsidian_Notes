# 1. The Analysis Lifecycle (Domain vs. Application)

A class diagram in the Analysis phase is fundamentally different from a diagram in the Implementation phase. This is the **Process** of modeling.

### 1.1 The Domain Class Model (Chapter 12)
This is about capturing the "real-world essence" of the problem.
*   **Finding Classes:** You don't just "guess." The book provides a formal **Noun Filtering** technique. You list candidate classes (nouns), then systematically eliminate them based on criteria:
    *   **Redundant:** (e.g., `Customer` and `User`).
    *   **Irrelevant:** (e.g., `Cost` apportioned to banks).
    *   **Vague:** (e.g., `SecurityProvision`).
    *   **Implementation Constructs:** (e.g., `TransactionLog`, `CommunicationsLine`).
*   **The Data Dictionary:** The model is useless without a data dictionary. This is a vital class diagram component. It forces you to write a precise description for *every single class* to ensure all stakeholders have the same mental model.

### 1.2 The Application Class Model (Chapter 13)
Once the domain model is stable, you add **Application Classes**.
*   **Boundary Classes:** Classes that mediate between the system and the external world (e.g., `ATMStation`, `CashCardBoundary`). These aren't "real world" but are required for the application to function.
*   **Controller Classes:** Active objects that manage control logic (e.g., `TransactionController`). These classes manage the state transitions of the business logic.

# 2. Design Optimization & Refactoring (Chapters 14 & 15)

In Design, you don't just "draw" classes; you "engineer" them. This is where you prepare the diagram for actual performance and reuse.

### 2.1 Bridging the Design Gap
The distance between "Desired Features" and "Available Resources" is the **Design Gap**.
*   **Intermediate Elements:** You must invent new classes that do not exist in the real world to act as "bridges" to your implementation environment.
*   **Algorithm-Driven Modeling:** You design the class diagram specifically to accommodate the algorithms you need. If a standard `List` structure is too slow, your class model must introduce a `hashedSet` or `Indexed` class to support the algorithm.

### 2.2 Refactoring Class Models
*   **Coherence:** Every class should have a single major theme. If a class has more than 10 attributes, 10 associations, or 20 operations, the book recommends "splitting" it.
*   **Separating Policy from Implementation:** 
    *   **Policy:** Context-dependent methods (handles business logic, I/O).
    *   **Implementation:** Pure, reusable algorithms (handles data manipulation). 
    *   *Design Pattern:* Ensure your class model distinguishes these. Policy methods call Implementation methods; Implementation methods never call Policy methods.

### 2.3 Optimization Strategies
*   **Adding Redundant Associations:** Sometimes, adding a "derived" association (e.g., a shortcut path in a graph) is necessary to avoid "nested loop" performance bottlenecks.
*   **Caching Derived Values:** If calculation is expensive, model a cache as a class attribute or a "SuspiciousUpdateGroup" class, and add an update mechanism to the model.

# 3. Implementation Modeling (Chapters 17, 18, 19)

This is the final stage where the class diagram is mapped to physical code and database structures.

### 3.1 Association Realization (The "Glue")
You must formulate a strategy for every association:
1.  **One-way:** Simple pointer or reference attribute.
2.  **Two-way:** Requires two pointers (bi-directional synchronization) or a separate **Association Object**.
3.  **Advanced (Qualified/Ordered/Bag):** Requires specialized collection objects (e.g., HashMaps, Lists, or specialized Dictionary structures).

### 3.2 Database Mapping Rules (The RDBMS Schema)
The book provides an explicit translation table for mapping class diagrams to SQL:

*   **Class:** `Table`
*   **Attribute:** `Column`
*   **Association (Many-to-Many):** `Junction Table` + `Composite Primary Key`.
*   **Generalization (Inheritance):** 
    *   Create a table for the superclass and each subclass.
    *   Ensure the subclass table has a `FOREIGN KEY` referencing the superclass, with `ON DELETE CASCADE` enabled.
    *   **Use Check Constraints:** Use SQL `CHECK` constraints to enforce the `generalization set name` (e.g., `CHECK (type IN ('Checking', 'Savings'))`).

You were not necessarily missing "new notation," but you were missing the **iterative refinement workflow** of the class diagram. Here is how the book guides that cycle:

1.  **Analysis (Conceptual):** Start with Domain Class Model (identify classes, attributes, associations).
2.  **Interaction Modeling:** Find use cases, which expose *missing* classes and attributes. (Iterate and update class diagram).
3.  **Application Analysis:** Add boundary and controller classes. (Update class diagram).
4.  **System Design:** Partition into subsystems/packages. (Update class diagram).
5.  **Class Design (Refinement):** Add algorithms, refactor, optimize, add association classes for speed. (Finalize class diagram).
6.  **Implementation:** Map to SQL/Language code. (The physical realization of the class diagram).
