# 3. Physical Operators & Algorithms

Once the logical plan is decided, the DBMS must decide _how_ to execute it physically. This involves choosing access paths and algorithms.

## 1. Access Paths (How to read data)

### A. Sequential Scan (Table Scan)

- **Mechanism:** Reads every single block/page of the table from disk.
- **When to use:** When the table is small, or when you need to read a large percentage (>15-20%) of the rows.
- **Cost:** Proportional to the total number of pages in the table.

### B. Index Scan

- **Mechanism:** Uses a B-Tree or Hash Index to find pointers to the specific rows, then fetches those rows.
- **When to use:** When looking for a specific value (e.g., `WHERE ID = 500`) or a small range.
- **Cost:** Proportional to the depth of the tree + number of matching records.

## 2. Join Algorithms (How to combine tables)

The choice of join algorithm drastically affects performance.

### A. Nested Loop Join (Boucles Imbriquées)

- **Concept:** For every row in the Outer Table (R), scan the entire Inner Table (S).
- **Complexity:** $O(N \times M)$
- **Best for:** Small tables.
- **Visual:**
  ```mermaid
  graph LR
  Outer[Row from Table A] -->|Check every row| Inner[Table B]
  ```

### B. Sort-Merge Join (Tri-Fusion)

- **Concept:** Sort both tables by the join key first. Then, iterate through both lists simultaneously (like zipping a zipper).
- **Complexity:** Cost of Sorting + $O(N + M)$.
- **Best for:** Large tables where the join key is already sorted (e.g., clustered index) or when the output needs to be sorted.

### C. Hash Join (Hachage)

- **Concept:** Create a Hash Map (in memory) of the smaller table. Then scan the larger table and probe the hash map for matches.
- **Complexity:** $O(N + M)$ (Assuming the hash table fits in memory).
- **Best for:** Large, unsorted tables. Usually the fastest for massive data equality joins.

---

> [!NOTE] Key Concept: Pipelining vs. Materialization
>
> - **Materialization:** One step finishes completely, writes results to a temporary table on disk, then the next step reads it. (Slow, safe for low memory).
> - **Pipelining:** The result of one operator is passed immediately to the parent operator row-by-row. (Fast, requires no disk I/O for intermediate steps). **Optimized trees usually rely on pipelining.**
