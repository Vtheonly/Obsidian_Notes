
## 3.4. Code Modularity and Packaging

### 1. Modular Organization
Organizing code into modules (individual Python files) and packages (directories containing modules and an `__init__.py` file) helps maintain clean boundaries between different functional parts of a project.

```
project_root/
├── utils/                  # Utility functions package
│   ├── __init__.py         # Marks directory as an importable package
│   ├── helpers.py          # String extraction and cleaning utilities
│   └── math_ops.py         # Computational math sub-module
└── main.py                 # Core application entry point
```

### 2. Package Initialization (`__init__.py`)
The `__init__.py` file runs automatically when a package is imported. It is used to expose package APIs and define which modules are exported publicly.

```python
# Content of utils/__init__.py
from .helpers import clean_lines, parse_products
from .math_ops import calculate_means

__all__ = ['clean_lines', 'parse_products', 'calculate_means']
```

This allows clean imports in `main.py` without needing to navigate internal folders:
```python
# Importing exposed APIs directly from the package
from utils import clean_lines, parse_products
```

### 3. Understanding `__name__ == "__main__"`
The `__name__` variable is a built-in Python attribute. When a script runs directly, `__name__` is set to `"__main__"`. If the script is imported as a module by another program, `__name__` is set to the script's actual file name.

This mechanism is used to separate runnable scripts (e.g., tests, utility scripts) from importable library modules.

```python
# File: calculations.py
def add(a, b):
    return a + b

# The following block runs ONLY if the script is executed directly
if __name__ == "__main__":
    print("Running calculations self-test...")
    assert add(2, 2) == 4
    print("Self-test passed.")
```
