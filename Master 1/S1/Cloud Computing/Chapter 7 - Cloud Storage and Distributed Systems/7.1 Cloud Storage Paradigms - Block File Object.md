## 6.1. Cloud Storage Paradigms - Block File Object

Cloud providers offer three primary types of storage, designed for different performance and scaling requirements:

### 6.1.1. Detailed Technical Matrix

| Dimension | Block Storage | File Storage | Object Storage |
| :--- | :--- | :--- | :--- |
| **Data Structure** | Raw sectors, blocks of fixed size. | Hierarchical folder tree with nested subdirectories. | Flat namespace; data is stored as independent, unnested objects. |
| **Access Methods** | Low-level storage commands (SCSI, NVMe). | File API commands (Open, Read, Write, Seek). | HTTP/HTTPS REST API commands (GET, PUT, POST, DELETE). |
| **Metadata** | Minimal (contains only block address maps). | Standard filesystem attributes (permissions, creation dates, size). | Highly customizable, unlimited key-value metadata tags. |
| **Scalability** | Limited (restricted to maximum disk partition limits). | Moderate (limited by filesystem index tables and inodes). | Massively Scalable (virtually unlimited scaling across clusters). |
| **Latency** | Extremely Low (1-5 milliseconds). | Moderate (network file protocol delays). | High (network request overhead). |
| **Update Capability** | Supports partial block updates. | Supports partial file updates and appends. | Immutable; updates require rewriting the entire object. |
| **Primary Use Cases** | Database files, Virtual Machine system disks. | Shared filesystems, legacy app migrations, user directories. | Big Data analytics, media assets, backups, archives. |
| **Industry Examples** | AWS EBS, Azure Disk, Google Persistent Disk. | AWS EFS, Azure Files, Google Filestore. | AWS S3, Azure Blob, Google Cloud Storage. |
