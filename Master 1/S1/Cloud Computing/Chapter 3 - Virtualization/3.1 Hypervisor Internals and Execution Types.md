# Hypervisor Internals and Execution Types

A **hypervisor** is the core technology that makes virtualization possible. It acts as the control layer between the physical hardware and one or more virtual machines (VMs) -- managing, scheduling, and isolating them as if each were a separate computer. Also known as a **Virtual Machine Monitor (VMM)**, the hypervisor is a specialized software or firmware that allows multiple operating systems to run simultaneously on the same physical hardware, each behaving as if it had the machine to itself.

---

## 1. The "Rings" of Privilege

To understand why different virtualization types exist (Full, Para, Hardware-assisted), you must understand CPU Rings. Standard x86 processors have privilege levels from **Ring 0** (highest privilege, meant for the OS kernel) to **Ring 3** (lowest privilege, meant for user applications).

Normally, an OS installs itself in Ring 0. But in virtualization, the **Hypervisor** needs Ring 0. So where does the Guest OS go? This problem birthed the three types of virtualization. The privilege ring architecture is the fundamental reason why virtualization is not simply "running multiple OSes at the same time" -- it requires careful orchestration of who is allowed to execute which instructions at which privilege level. Without a mechanism to manage this, the Guest OS would attempt to execute privileged hardware commands that conflict with the Hypervisor's own control of the system, causing crashes, data corruption, or security breaches.

---

## 2. The Core Job of a Hypervisor

At its core, the hypervisor does three main things: virtualization, resource management, and isolation.

### Virtualization

The hypervisor **emulates hardware** for each VM. When a VM asks for a CPU, memory, or disk, the hypervisor intercepts the request, maps it to actual physical resources, and returns a virtualized version of that hardware. This makes each VM believe it has its own CPU, RAM, storage, and network interface -- even though it is all being shared under the hood. Every hardware access a Guest OS attempts is mediated by the hypervisor, which translates virtual device commands into real hardware operations.

### Resource Management

The hypervisor **allocates and manages hardware resources** among all active VMs. For example, if your server has 8 CPU cores and 32 GB RAM, the hypervisor might give VM1 2 cores and 8 GB RAM, VM2 4 cores and 16 GB RAM, and VM3 2 cores and 8 GB RAM. It monitors performance, prevents one VM from starving others, and can dynamically reallocate resources as needed. This fair-scheduling capability is what enables multi-tenant cloud environments where dozens of customers share the same physical silicon without interfering with each other.

### Isolation and Security

Each VM is fully isolated from others. One VM cannot read or modify another's memory or disk. A crash in one VM does not affect others. The hypervisor enforces strict separation, acting as a firewall between them. This isolation is crucial for multi-tenant systems like public clouds, where thousands of users' VMs run on the same physical hardware. Without hypervisor-enforced isolation, one compromised VM could access the data of every other VM on the host, which would make cloud computing fundamentally insecure and unusable.

## 3. Types of Virtualization

### Full Virtualization (Software Emulation)

The Guest OS is pushed to Ring 1. Since it does not know it is virtualized, it tries to execute privileged hardware commands. The Hypervisor must **intercept and translate** (emulate) every single privileged instruction down to the actual hardware. This results in extremely slow performance due to the massive overhead of translating instructions on the fly. This round-trip overhead makes Full Virtualization the slowest of the three approaches.

### Paravirtualization (The "Aware" OS)

The Guest OS source code is physically modified (e.g., Xen). The OS **knows** it is virtualized. Instead of executing hardware commands, it issues special API calls called **Hypercalls** directly to the hypervisor. This is much faster than Full Virtualization because the hypervisor does not need to intercept and decode instructions -- the Guest OS cooperatively communicates its needs. However, there is a significant limitation: you cannot paravirtualize closed-source operating systems like standard Windows because you cannot modify the kernel code. Paravirtualization requires access to the OS source, making it practical only for open-source operating systems like Linux.

### Hardware-Assisted Virtualization (The Modern Standard)

Intel and AMD added new physical instructions to CPUs (Intel VT-x, AMD-V) that created a "Ring -1" (Ring Minus One). The Hypervisor sits in Ring -1, allowing the Guest OS to sit in Ring 0 natively. The CPU hardware itself handles the interception. No OS modification is needed, and performance is near bare-metal. This is the dominant approach in modern data centers and cloud platforms. The CPU's built-in virtualization extensions handle the complex privilege-level transitions in silicon rather than software, eliminating the translation overhead that plagued Full Virtualization and the source-code requirement of Paravirtualization.

## 4. Hypervisor Types: Type 1 vs Type 2

### Type 1 (Bare-Metal)

Installs directly on the server motherboard. It acts as both the OS and the Hypervisor. Maximum performance because there is no intermediary operating system layer. Every I/O request travels directly from the VM through the hypervisor to the hardware. Examples: VMware ESXi, Microsoft Hyper-V, Proxmox (KVM), Xen.

```
+--------------------------------+
| Physical Hardware              |
|--------------------------------|
| Hypervisor (e.g., KVM)         |
|--------------------------------|
| VM1 | VM2 | VM3                |
+--------------------------------+
```

### Type 2 (Hosted)

Installs as a standard software application on top of an existing OS (Windows/macOS). High overhead because an I/O request must travel: `App -> Guest OS -> Hypervisor -> Host OS -> Hardware`. Each layer adds latency and overhead. Examples: VirtualBox, VMware Workstation, Parallels.

```
+--------------------------------+
| Host OS (Windows / Linux)      |
|--------------------------------|
| Hypervisor (VirtualBox)        |
|--------------------------------|
| VM1 (Ubuntu) | VM2 (Fedora)    |
+--------------------------------+
```

| Type | Description | Where It Runs | Examples |
|------|-------------|---------------|----------|
| **Type 1 (Bare-Metal)** | Runs directly on hardware, no underlying OS. Used in data centers. | Directly on the physical machine | VMware ESXi, Microsoft Hyper-V, KVM, Xen |
| **Type 2 (Hosted)** | Runs on top of an operating system, like a normal app. Used on desktops. | Inside a host OS | VirtualBox, VMware Workstation, Parallels |

## 5. How It Actually Works (Simplified)

When a VM runs an instruction like: "Write this data to memory address 0xFF1A," the hypervisor steps in:

1. Intercepts the instruction (using CPU virtualization features like Intel VT-x or AMD-V)
2. Translates the virtual address to a real physical address
3. Performs the actual write on behalf of the VM

The same logic applies for CPU scheduling, disk I/O, and network access -- all go through the hypervisor layer, which mediates and controls them. The hypervisor is not merely a passive relay; it actively remaps every virtual resource reference to its physical counterpart, ensuring that two VMs never accidentally write to the same physical memory location or issue conflicting commands to the same hardware device.

## 6. Additional Capabilities of Modern Hypervisors

Beyond the basics, hypervisors in modern cloud and server environments do much more:

| Feature | Description |
|---------|-------------|
| **Live Migration** | Move a running VM from one host to another without shutting it down. |
| **Snapshotting** | Save the full state of a VM (RAM, disk, CPU) for rollback or cloning. |
| **Cloning** | Duplicate a VM quickly for scaling. |
| **Resource Overcommitment** | Allocate more virtual resources than physically exist (since not all VMs use them fully at once). |
| **Hardware Virtualization Support** | Uses CPU features (Intel VT-x / AMD-V) for efficiency. |
| **Nested Virtualization** | Run hypervisors inside VMs. |

---

## 7. Common Misconception: Emulation vs. Virtualization

Students often confuse **Emulation** with **Virtualization**. **Virtualization** divides existing local hardware resources among VMs (e.g., x86 hardware running an x86 VM). **Emulation** uses software to mimic entirely different hardware architectures (e.g., using an x86 PC to run a Nintendo Wii ARM processor game). Emulation is massively slower because every instruction must be translated from one architecture to another in real time, whereas virtualization simply partitions the same architecture with minimal overhead.

---

## 8. Analogy

Think of the hypervisor as the **operating system for virtual machines**. The same way your OS schedules and isolates processes on a CPU, the hypervisor schedules and isolates entire operating systems (VMs). So: **Processes** are to the **OS** what **Virtual Machines** are to the **Hypervisor**.

## 9. Quick Summary

| Concept | Description |
|---------|-------------|
| **Hypervisor** | Software layer that creates and manages virtual machines. |
| **Purpose** | Virtualize hardware, allocate resources, isolate VMs. |
| **CPU Rings** | Ring 0 (kernel), Ring 3 (user), Ring -1 (hypervisor with hardware-assisted virtualization). |
| **Full Virtualization** | Guest OS in Ring 1, hypervisor intercepts and translates all privileged instructions. Slowest. |
| **Paravirtualization** | Modified Guest OS uses Hypercalls. Fast, but requires source code access. |
| **Hardware-Assisted** | CPU Ring -1 for hypervisor. Guest OS runs natively in Ring 0. Near bare-metal speed. Modern standard. |
| **Type 1** | Bare-metal, runs directly on hardware. Used in data centers. |
| **Type 2** | Hosted, runs on top of a host OS. Used on desktops. |
| **Isolation** | Prevents VMs from interfering with one another. |
| **Real-world role** | Foundation of cloud computing -- allows multiple customers to share the same hardware safely. |
