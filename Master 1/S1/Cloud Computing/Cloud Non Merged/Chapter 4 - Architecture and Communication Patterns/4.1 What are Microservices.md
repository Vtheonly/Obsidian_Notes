# What are Microservices

## Definition

**Microservices** are an **architectural approach** where a large application is divided into many small, **independent services** -- each responsible for a *specific function* (like authentication, payments, user profiles, notifications, etc.).

Each microservice:

- Has its **own codebase**
- Runs **independently** (can be on its own server, container, or VM)
- Communicates with others through **APIs** (usually REST or gRPC)
- Can be **developed, deployed, and scaled separately**

---

## The Core Idea

Instead of building **one big monolithic app**, you break it into **many self-contained mini-apps** that together form the whole system.

Each one focuses on doing **one job well**.

This architectural philosophy stems from the Unix design principle: do one thing, and do it well. In a monolithic application, every feature is intertwined within a single deployable unit, which means a change in the notification system can potentially break the authentication flow. Microservices eliminate this class of problem by enforcing hard boundaries between functional domains. Each service owns its data, its logic, and its release cycle, making the overall system far more resilient to change and failure.

---

## Example

Let us take a simple SaaS platform as an example:

| Function        | Microservice        | Example Tech        |
| --------------- | ------------------- | ------------------- |
| User Management | `auth-service`      | Node.js + JWT       |
| Payments        | `payment-service`   | Python + Stripe API |
| Notifications   | `notify-service`    | Go + RabbitMQ       |
| Analytics       | `analytics-service` | Java + Kafka        |

Each runs in its own container (Docker, Kubernetes pod, etc.), possibly even on separate machines or cloud instances. The critical point here is that each microservice is independently deployable. If the payment service needs a hotfix, you can deploy just that service without touching the authentication or notification services. This isolation extends to the database layer as well: each service typically owns its own database or schema, preventing tight coupling at the data level.

---

## How They Talk to Each Other

Communication usually happens through:

- **REST APIs** (`/api/users`, `/api/payments`)
- **Message Queues** (Kafka, RabbitMQ, NATS)
- **Event-driven patterns** (publish/subscribe model)

That is how they stay **loosely coupled** -- they do not share databases or internal code, only **messages or contracts**. The choice between synchronous communication (REST, gRPC) and asynchronous communication (message queues, events) has significant implications for system reliability. Synchronous calls create temporal coupling: if the payment service is down, the order service cannot complete its workflow. Asynchronous communication via message queues eliminates this coupling by allowing services to emit events and continue processing, trusting that downstream services will consume those events when they become available.

---

## Analogy

Think of a **city**:

- Each department (police, fire, transport, health) works independently.
- They all serve the same city but communicate via official channels.
- If one department fails, the others still run.

That is how microservices keep large systems resilient and scalable. The city analogy also highlights an important trade-off: coordination between departments requires well-defined protocols and communication channels, just as microservices require well-defined APIs and messaging contracts. Without these, the system descends into chaos despite its modularity.

## Monolith vs Microservices

| Feature           | Monolithic App             | Microservices           |
| ----------------- | -------------------------- | ----------------------- |
| Structure         | One big codebase           | Many small services     |
| Deployment        | All at once                | Each service separately |
| Scalability       | Scale whole app            | Scale only needed parts |
| Failure impact    | One crash = whole app down | Isolated failures       |
| Team organization | One large team             | Multiple smaller teams  |
| Communication     | Internal function calls    | Network APIs or queues  |

The transition from monolith to microservices is not a simple refactor. It requires a fundamental shift in how teams are organized (often following the Conway's Law pattern where team boundaries mirror service boundaries), how code is deployed (each service needs its own CI/CD pipeline), and how the system is monitored (distributed tracing becomes essential when a single user request may traverse five or more services).

---

## How They Run in the Cloud

Microservices are often deployed in:

- **Containers (Docker)**
  Each service packaged with its dependencies, ensuring consistent behavior across development, staging, and production environments.
- **Orchestrators (Kubernetes, ECS)**
  Manages scaling, networking, and discovery automatically, allowing services to be elastically scaled based on demand.
- **Serverless runtimes (AWS Lambda, Cloudflare Workers)**
  Some microservices can even be serverless if they are small and event-based, paying only for actual execution time rather than reserved compute capacity.

The cloud is the natural habitat for microservices because it provides the elastic infrastructure needed to independently scale each service. A payment service handling a surge during a flash sale can scale out to twenty instances while the notification service remains at two, something impossible in a monolithic deployment model.

---

## Advantages

- Flexibility in tech stack per service (use Go for performance-critical services, Python for ML pipelines, Node.js for real-time features).
- Easier scaling and isolation (scale only the bottleneck, not the entire application).
- Independent deployment pipelines (deploy the auth service ten times a day without touching the payment service).
- Better fault tolerance (a crash in the analytics service does not take down the checkout flow).
- Ideal for distributed teams (each team owns one or more services end-to-end).

---

## Disadvantages

- More complex networking and monitoring (latency between services, distributed tracing, and service mesh overhead).
- Harder debugging (many services talking together makes it difficult to reproduce issues locally).
- Needs proper DevOps setup (Kubernetes, observability, CI/CD) -- the operational complexity is significantly higher than managing a single monolithic application.
- Data consistency challenges (without a shared database, maintaining consistency across services requires patterns like sagas or event sourcing).

---

## In Short

> **Microservices** = breaking a large application into a collection of small, independent services that communicate over the network -- each doing one thing well, and together forming the full system.
