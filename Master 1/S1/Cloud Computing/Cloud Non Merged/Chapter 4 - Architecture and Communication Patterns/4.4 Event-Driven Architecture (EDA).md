# Event-Driven Architecture (EDA)

## What is Event-Driven Architecture?

**Event-Driven Architecture (EDA)** is a software design paradigm in which the flow of the program is determined by **events** -- significant changes in state that the system detects, communicates, and responds to. Unlike traditional request-driven architectures where a client explicitly invokes a service and waits for a response, EDA inverts this model: services react to events as they occur, without requiring a direct command from the triggering source.

An **event** is an immutable record of something that has happened. It carries a timestamp, a type, and a payload describing the state change. For example, an `OrderPlaced` event might contain the order ID, customer ID, items ordered, and total price. Crucially, an event is a statement of fact about the past -- it is not a command instructing a service to do something. This distinction is fundamental: commands are imperative ("Process this payment"), while events are declarative ("A payment was processed"). This declarative nature is what enables the radical decoupling that makes EDA so powerful in distributed systems.

In an EDA system, there are three primary roles:

- **Event Producers** -- Services or components that detect a state change and emit an event. They do not know or care who consumes the event.
- **Event Channels** -- The infrastructure (message brokers, event buses) that transports events from producers to consumers.
- **Event Consumers** -- Services that subscribe to specific event types and react to them by executing business logic, updating state, or emitting new events.

---

## EDA vs Request-Driven Architecture

Understanding EDA requires understanding what it is not. The dominant architectural pattern in most web applications is **request-driven (or request-response) architecture**, where communication is synchronous and direct.

| Aspect                  | Request-Driven Architecture              | Event-Driven Architecture                      |
| ----------------------- | ---------------------------------------- | ---------------------------------------------- |
| Communication model     | Synchronous (call and wait)              | Asynchronous (fire and react)                  |
| Coupling                | Tight (caller must know the callee)      | Loose (producer knows nothing about consumers)  |
| Failure propagation     | Cascading (callee failure blocks caller) | Isolated (consumer failure does not affect producer) |
| Scalability model       | Scale the entire service                 | Scale consumers independently                  |
| Data consistency        | Strong (immediate)                       | Eventual (delayed propagation)                 |
| Typical protocol        | HTTP REST, gRPC                          | MQTT, Kafka, AMQP                              |
| Debugging complexity    | Lower (single call chain)                | Higher (distributed event flows)               |

In a request-driven system, when a user places an order, the order service must synchronously call the payment service, the inventory service, and the notification service. If the payment service is slow, the entire operation is slow. If it is down, the entire operation fails. In an EDA system, the order service simply emits an `OrderPlaced` event and returns immediately. The payment, inventory, and notification services each consume this event independently, at their own pace, and handle their own failures without affecting the order service or each other.

## The Event Lifecycle

An event in an EDA system follows a well-defined lifecycle:

1. **Occurrence** -- Something happens in the real world or in the system: a user clicks "Buy," a sensor reads a temperature, or a timer fires.

2. **Detection** -- A producer component detects the occurrence and constructs an event object containing the relevant data.

3. **Emission** -- The producer publishes the event to an event channel (broker, bus, or stream).

4. **Routing** -- The event channel determines which consumers should receive the event based on subscriptions, topics, or content-based filters.

5. **Delivery** -- The event is transported to the appropriate consumers.

6. **Processing** -- Each consumer processes the event according to its own business logic, potentially producing new events (creating an event chain).

7. **Acknowledgment** -- Consumers acknowledge successful processing, allowing the broker to manage delivery guarantees (at-least-once, exactly-once).

## Event Schema and Patterns

### Event Notification

The simplest pattern. The event contains minimal information -- just enough to signal that something happened. Consumers that need more data must query the source system.

```
{
  "type": "OrderPlaced",
  "orderId": "ORD-12345",
  "timestamp": "2025-01-15T10:30:00Z"
}
```

This pattern minimizes event size and coupling, but it introduces a temporal dependency: the consumer must be able to reach the source system at query time. If the source system is down, the consumer cannot complete its processing.

### Event-Carried State Transfer

The event carries the full state needed by consumers, eliminating the need for them to query the source. This pattern trades larger event payloads for complete autonomy.

```
{
  "type": "OrderPlaced",
  "orderId": "ORD-12345",
  "customerId": "CUST-789",
  "items": [{"sku": "WIDGET-1", "qty": 2, "price": 29.99}],
  "total": 59.98,
  "shippingAddress": {...},
  "timestamp": "2025-01-15T10:30:00Z"
}
```

This pattern is particularly useful when consumers need to maintain their own materialized views of data. For example, a customer service dashboard might consume `OrderPlaced` events and build a local read-optimized view of all orders, enabling fast queries without hitting the order service's database.

### Event Sourcing

Instead of storing just the current state of an entity, event sourcing stores **every state-changing event** that has ever occurred. The current state is derived by replaying the event sequence from the beginning.

For example, a bank account is not stored as `{"balance": 1500}`. Instead, the event log contains:

1. `AccountOpened` with initial deposit of 1000
2. `DepositMade` of 500
3. `WithdrawalMade` of 200
4. `DepositMade` of 200

The balance of 1500 is computed by replaying these events. This pattern provides a complete audit trail, enables time-travel debugging (what was the balance on Tuesday at 3 PM?), and makes it possible to retroactively create new views of historical data. However, it requires careful handling of event schema evolution, as old events must remain replayable even as the system evolves.

## CQRS (Command Query Responsibility Segregation)

CQRS is a pattern that naturally pairs with EDA and event sourcing. It recognizes that the read and write sides of an application have fundamentally different requirements:

- **Write side (Command model):** Optimized for validation, business rules, and consistency. It processes commands and emits events.
- **Read side (Query model):** Optimized for fast queries. It consumes events and builds materialized views tailored to specific query patterns.

In a traditional CRUD application, the same data model serves both reads and writes, leading to compromises. A database schema optimized for writing (normalized, transactional) is often terrible for complex queries (joins across many tables). CQRS splits these concerns entirely. The write side might use a relational database with strict ACID transactions, while the read side might use Elasticsearch for full-text search or a wide-column store for analytics. Events flow from the write side to the read side, eventually synchronizing the two models.

## Sagas

In a microservices architecture, a single business transaction often spans multiple services. In a monolithic application, a database transaction ensures atomicity: either all changes succeed or none do. In a distributed system, there is no single database, so traditional transactions do not work.

A **saga** is a pattern for managing distributed transactions across microservices. It breaks the overall transaction into a sequence of local transactions, each owned by a single service. If any step fails, the saga executes **compensating transactions** to undo the work completed by previous steps.

There are two orchestration styles:

- **Choreography** -- Each service emits events that trigger the next step. There is no central coordinator. The flow emerges from the event chain. This is simpler but harder to debug for complex workflows.
- **Orchestration** -- A central saga orchestrator sends commands to each service and handles responses. This provides clearer visibility and control but introduces a single point of coordination.

## Benefits of EDA

- **Loose coupling** -- Producers and consumers are completely independent. A new consumer can subscribe to existing events without any changes to the producer.
- **Scalability** -- Consumers can be scaled independently based on their processing load. A slow consumer does not block a fast producer.
- **Resilience** -- If a consumer fails, events accumulate in the broker and are processed when the consumer recovers. No data is lost.
- **Extensibility** -- New functionality can be added by introducing new event consumers without modifying existing services.
- **Audit trail** -- Events provide a natural, immutable log of everything that has happened in the system, enabling compliance, debugging, and replay capabilities.

---

## Challenges of EDA

- **Eventual consistency** -- The system is not immediately consistent. It takes time for events to propagate and for all consumers to process them. This requires careful UI design to manage user expectations.
- **Debugging complexity** -- A single user action might trigger a chain of events across five services. Tracing the full flow requires distributed tracing tools and correlation IDs.
- **Event ordering** -- Ensuring events are processed in the correct order is non-trivial, especially when events come from multiple producers or partitions.
- **Schema evolution** -- As the system evolves, event schemas change. Consumers must be able to handle both old and new event formats, requiring careful versioning strategies.
- **Duplicate processing** -- At-least-once delivery guarantees mean consumers might receive the same event twice. Idempotent processing (ensuring the same event processed twice produces the same result) is essential.

---

## Relationship to Microservices

EDA and microservices are complementary but independent patterns. Microservices define *how an application is decomposed* into independently deployable units. EDA defines *how those units communicate*. A microservices architecture can use synchronous REST communication, asynchronous event-driven communication, or a mix of both.

However, EDA is often considered the natural communication model for microservices because it preserves the independence that microservices aim to achieve. If microservices communicate synchronously, they form a distributed monolith: a failure in one service cascades through the call chain, and deploying one service requires verifying compatibility with all its dependents. Event-driven communication preserves the autonomy of each service, making the microservices architecture truly resilient and independently deployable.

## Real-World Examples

- **E-commerce platforms** -- When an order is placed, events flow to payment processing, inventory management, shipping, and notification services. Each operates independently and can fail or scale without affecting the others.
- **IoT systems** -- Thousands of sensors emit temperature, humidity, and pressure readings as events. Analytics services consume these streams in real-time to detect anomalies and trigger alerts.
- **Financial trading** -- Market data feeds produce millions of price-change events per second. Trading algorithms consume these events, make decisions, and emit trade events, all within milliseconds.
- **Social media** -- A user posting a status update triggers events for feed generation, push notifications, analytics tracking, and content moderation, each handled by separate consumer services.
