# Kubernetes Networking, Services, and Configuration Management

## Background Context: The Ephemeral IP Problem

In traditional IT, a database server might have the IP address `10.0.1.5` for five years. In Kubernetes, **Pods are ephemeral (mortal)**. If a Pod uses too much memory, Kubernetes ruthlessly kills it and creates a new one. The new Pod gets a completely different IP address.

If your Frontend application is hardcoded to talk to `10.0.1.5`, the moment that Pod dies, your entire application breaks.

This is not an edge case -- it is the normal operational mode of Kubernetes. Pods are designed to be disposable. They are created and destroyed constantly: during scaling events, rolling updates, node failures, and even routine health check failures. A production Kubernetes cluster might recreate hundreds of pods per hour. If every pod recreation broke dependent services, the system would be unusable. This is why Kubernetes introduces the Service abstraction -- to provide a stable network identity that outlives any individual pod.

---

## The Solution: Kubernetes Services

A `Service` in Kubernetes is an abstraction layer -- a permanent, stable network endpoint that sits in front of a group of Pods. Even if the Pods behind the Service die and are reborn with new IPs a hundred times a minute, the Service IP and DNS name *never change*.

A Service uses label selectors to determine which Pods belong to it. When you create a Service, you specify a selector like `app: backend`. The Service automatically discovers all Pods with that label and load-balances traffic across them. As pods are created or destroyed, the Service's endpoint list is updated automatically -- no manual intervention required.

---

### Types of Services

#### 1. ClusterIP (The Default)

- *Scope:* Strictly internal.
- *Mechanic:* Creates a stable IP address that is only reachable from *inside* the Kubernetes cluster.
- *Use Case:* The Frontend talking to the Backend API. The public internet has no business talking to your internal API directly.

ClusterIP is the most commonly used service type because most communication in a Kubernetes cluster is internal. When the frontend pod needs to call the backend API, it uses the ClusterIP Service. The DNS name `backend-api.default.svc.cluster.local` resolves to the Service's ClusterIP, and traffic is load-balanced across all healthy backend pods. This internal routing is handled entirely by kube-proxy and iptables rules on the node, without any external load balancer involvement.

#### 2. NodePort

- *Scope:* External access (Basic).
- *Mechanic:* Kubernetes physically opens a port (between 30000 and 32767) on **every single Worker Node** in the cluster. If you hit `Any_Node_IP:30050`, the traffic is forwarded to the Service, and then down to the Pod.
- *Use Case:* Testing, or small-scale setups where you do not want to pay for a Cloud Load Balancer.

NodePort builds on top of ClusterIP. It allocates a ClusterIP internally and additionally opens the specified port on every worker node. This means you can reach your service from outside the cluster by connecting to any node's IP address on the NodePort. The limitation is that the port range (30000-32767) is not user-friendly, and if a node goes down, you cannot reach the service through that node's IP. For these reasons, NodePort is rarely used directly in production -- it is typically used as a building block by LoadBalancer services.

#### 3. LoadBalancer

- *Scope:* External access (Production).
- *Mechanic:* It creates a NodePort internally, but it goes a step further: it reaches out to the Cloud Provider (AWS, Azure) via APIs and commands them to provision a physical/virtual Load Balancer. The Cloud Load Balancer gets a public IP and routes traffic to your cluster.
- *Use Case:* Standard production web applications.

When you create a Service of type LoadBalancer on AWS, Kubernetes uses the AWS Cloud Controller Manager to call the Elastic Load Balancing API and provision a Classic Load Balancer, Network Load Balancer, or Application Load Balancer (depending on annotations). The cloud load balancer gets its own public IP address and forwards traffic to the NodePort on your cluster's worker nodes. From the user's perspective, they simply connect to the load balancer's public IP, and traffic flows seamlessly to the pods. The main drawback is cost: each LoadBalancer service provisions a separate cloud load balancer, which can be expensive at scale.

## Deep Dive: How `kube-proxy` Makes Services Work

The `Service` is not a physical router; it is a virtual concept. The component that makes it real is **kube-proxy**.

- `kube-proxy` runs on every single Worker Node.
- It constantly listens to the API Server for new Services.
- When you create a Service, `kube-proxy` writes `iptables` (or IPVS) firewall rules directly into the Linux kernel of its node.
- These rules say: "If any traffic tries to hit the Service IP, immediately intercept it, randomly pick one of the healthy Pod IPs, and redirect the traffic there."

kube-proxy operates in three modes:

- **iptables mode (default):** Uses Linux iptables rules for packet interception and NAT. It is efficient because packet filtering happens in kernel space, but it can become slow with thousands of services due to linear rule traversal.
- **IPVS mode:** Uses the Linux IP Virtual Server, which is designed for load balancing and uses hash tables for O(1) lookup. It is the preferred mode for large clusters.
- **Userspace mode (legacy):** The oldest mode, where kube-proxy itself processes packets in user space. It is the slowest and is rarely used today.

## Configuration Management: ConfigMaps and Secrets

A core principle of Docker is that **Images must be immutable and portable**.

If you bake your database connection string into the container image, you cannot use that same image for Development and Production. You would have to rebuild the image just to change the URL.

### ConfigMap

A Kubernetes object that stores non-sensitive configuration data (like `ENV_NAME=Production` or `API_URL=http://backend`). ConfigMaps decouple configuration from container images, allowing you to use the same image across multiple environments (development, staging, production) simply by changing the ConfigMap values.

### Secret

Similar to a ConfigMap, but used for passwords, SSH keys, and API tokens. It is base64 encoded and (ideally) encrypted at rest in `etcd`. Secrets provide a mechanism to hold sensitive data separately from the application code, reducing the risk of accidental exposure in version control or container image layers.

### How It Works

When a Pod boots up, Kubernetes intercepts the ConfigMap and injects the variables directly into the Pod as Environment Variables or mounted files. This allows you to use the exact same `nginx:latest` image everywhere, simply feeding it different ConfigMaps depending on the environment.

There are several ways to consume ConfigMaps and Secrets in Pods:

1. **Environment Variables:** The simplest method. Kubernetes injects key-value pairs as environment variables into the container.
2. **Mounted Volumes:** Kubernetes creates a file for each key in the ConfigMap/Secret and mounts them as a directory in the container. This is useful for configuration files (like `nginx.conf` or `application.properties`).
3. **Command-line Arguments:** Use ConfigMap values as arguments to the container's entrypoint command.

The mounted volume approach has a significant advantage: when the ConfigMap is updated, Kubernetes automatically updates the mounted files in the container (with a small delay), allowing applications to reload configuration without restarting. Environment variables, by contrast, are set at container creation time and cannot be updated without restarting the pod.
