# Dockerfile Layering and Image Build Mechanics

## Background Context: The Dockerfile

A `Dockerfile` is essentially an automation script. However, understanding *how* Docker parses this script is the secret to building fast, lightweight containers. A Dockerfile defines a sequence of instructions that Docker executes to build a container image. Each instruction creates a new layer in the image, and the final image is the stacked result of all these layers combined by UnionFS.

Understanding the build mechanics is not just an academic exercise -- it has direct practical consequences for build speed, image size, and security. A poorly structured Dockerfile can result in images that are gigabytes larger than necessary, builds that take minutes instead of seconds, and sensitive data (like API keys) permanently baked into the image layers.

---

## Core Instructions

- **`FROM`**: Defines the base image. **Must** be the first line. (e.g., `FROM python:3.10-slim`). Every Docker image is built on top of another image. The `FROM` instruction specifies the starting point. If you choose a minimal base like `alpine` or `slim`, your final image will be significantly smaller than if you start from a full Ubuntu image.

- **`WORKDIR`**: Sets the working directory inside the container. If it does not exist, Docker creates it. All subsequent `COPY` and `RUN` commands execute here. Using `WORKDIR` is preferred over `RUN cd /app && ...` because it is explicit, does not create an extra layer, and works correctly with relative paths in subsequent instructions.

- **`COPY`**: Takes files from the physical host machine and physically copies them into the container's filesystem. It creates a new layer containing exactly the files that were copied. The `ADD` instruction is similar but has additional features (automatic tar extraction and URL support) that are generally discouraged because they make the build behavior less predictable.

- **`RUN`**: Executes a command *during the build process*. (e.g., `RUN pip install -r requirements.txt`). It is used to install dependencies, create directories, and configure the environment. Each `RUN` instruction creates a new layer. This is why it is important to chain multiple commands in a single `RUN` instruction using `&&` when they logically belong together -- it reduces the number of layers.

- **`CMD`**: Defines the default command that will execute *when the container starts*. (e.g., `CMD ["python", "app.py"]`). There can only be one `CMD` per Dockerfile. If you specify multiple, only the last one takes effect. The `CMD` can be overridden at runtime with `docker run image command`.

- **`ENTRYPOINT`**: Similar to `CMD` but harder to override. When used together, `ENTRYPOINT` defines the executable and `CMD` provides default arguments. This pattern is common for images that act as executables, such as `ENTRYPOINT ["python"]` with `CMD ["app.py"]`, allowing the user to override the script while keeping the Python interpreter.

> [!WARNING] The difference between RUN and CMD
> This is a highly tested concept.
> - `RUN` happens at **Build Time**. It permanently changes the image (adds a new layer).
> - `CMD` happens at **Run Time**. It does not build anything; it just tells the container what program to launch when it wakes up.

## The Mechanics of Layer Caching

Every single line in a Dockerfile creates a new, read-only layer via UnionFS.

When you run `docker build`, Docker checks its cache. If line 1, 2, and 3 have not changed since the last build, Docker skips building them and just uses the cached layers. It only rebuilds the layer where a change occurred, *and every layer underneath it*.

**Why this matters for Developers:**
You must put the things that change *frequently* at the very bottom of the Dockerfile.

*Bad Dockerfile (Slow):*
```dockerfile
FROM python:3.9
COPY . /app            # You copy all your code (changes constantly)
WORKDIR /app
RUN pip install flask  # This layer is invalidated EVERY time you change code!
```

*Good Dockerfile (Fast):*
```dockerfile
FROM python:3.9
WORKDIR /app
COPY requirements.txt .  # Only copy the dependency list first
RUN pip install flask    # This layer is cached perfectly!
COPY . /app              # Now copy the code. Only this layer rebuilds.
```

In the bad example, every time you change a single line of Python code, Docker invalidates the `COPY . /app` layer and everything after it -- including the `pip install` step. This means every code change triggers a full reinstallation of all Python packages, which can take minutes. In the good example, `requirements.txt` changes rarely (only when you add a new dependency), so the expensive `pip install` layer is almost always cached. Only the final `COPY . /app` layer rebuilds, which takes seconds.

## Build Context and `.dockerignore`

When you run `docker build`, the first step is sending the **build context** (the directory containing your Dockerfile) to the Docker daemon. This means every file in that directory is packaged and transferred, including `node_modules`, `.git`, virtual environments, and build artifacts.

A `.dockerignore` file (similar to `.gitignore`) excludes files from the build context. This is critically important for two reasons: it reduces the size of the build context (speeding up the build), and it prevents sensitive files from being included in the image. A typical `.dockerignore` might exclude:

```
node_modules
.git
.env
__pycache__
*.pyc
.docker
```

## Multi-Stage Builds

One of the most powerful Dockerfile features is multi-stage builds, which allow you to use multiple `FROM` statements in a single Dockerfile. Each `FROM` starts a new build stage with a fresh environment. You can copy artifacts from one stage to another, leaving behind everything you do not need in the final image.

```dockerfile
# Stage 1: Build
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Stage 2: Production
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
```

In this example, the first stage installs all Node.js dependencies (which can be hundreds of megabytes) and builds the application. The second stage starts from a minimal Nginx image and copies only the build output (the `dist` directory) from the first stage. The Node.js runtime, npm packages, source code, and build tools are all discarded. The final image contains only Nginx and the static files -- typically under 50MB instead of hundreds.

## Best Practices Summary

| Practice                           | Reason                                                              |
| ---------------------------------- | -------------------------------------------------------------------- |
| Use minimal base images (`slim`, `alpine`) | Smaller attack surface, faster pulls, less disk usage          |
| Order instructions from least to most frequently changing | Maximizes layer cache hits during rebuilds                |
| Combine `RUN` commands with `&&`   | Reduces the number of layers and the total image size               |
| Use `.dockerignore`                | Prevents unnecessary files from entering the build context and image |
| Use multi-stage builds             | Separates build dependencies from runtime dependencies              |
| Never store secrets in images      | Use build args with `--secret` flag or runtime environment variables |
| Pin image versions (`python:3.9.18-slim` not `python:latest`) | Ensures reproducible builds                               |
