# Linux Kernel Internals (Namespaces, Cgroups, UnionFS)

## Background Context: Why Containers?

VMs require a full Guest OS (which consumes GBs of storage and RAM just to boot). Containers strip away the Guest OS. They are simply standard Linux processes wrapped in an "illusion" of isolation. If you run `htop` on a Docker Host, you will see the containerized processes running directly on the host kernel.

The key insight behind containers is that the Linux kernel already provides the primitives necessary for isolation -- you do not need a separate operating system to achieve it. A virtual machine achieves isolation by running a complete separate OS instance, complete with its own kernel, init system, device drivers, and system libraries. This provides strong isolation but at enormous cost: each VM consumes gigabytes of disk space for the OS image, hundreds of megabytes of RAM just to boot, and seconds of startup time. Containers take a radically different approach. They share the host kernel and use kernel-level features to create the illusion of isolation. The result is that a container can start in milliseconds, consume only megabytes of disk space, and use only the memory required by the application itself.

## The Three Pillars of Containerization

### 1. Namespaces: The "Illusion of Isolation"

Namespaces manipulate what a process can *see*. They are a Linux kernel feature that partitions kernel resources so that one set of processes sees one set of resources while another set of processes sees a different set. There are several namespace types, each isolating a different aspect of the system:

- **PID (Process ID):** A containerized app might think it is PID 1 (the master init process). In reality, on the host OS, it is PID 34582. The PID namespace creates this translation. This means the container has its own process tree, and processes inside the container cannot see or signal processes outside it. The init process inside the container (PID 1 from the container's perspective) has special responsibilities: it must reap zombie processes and handle signal propagation, which is why many container images use `tini` or `dumb-init` as the entrypoint rather than running the application directly as PID 1.

- **NET:** Gives the container its own isolated network stack (its own eth0, IP address, and routing table). Each network namespace has its own set of network interfaces, routing tables, firewall rules, and port numbers. This is why two containers can both listen on port 80 without conflicting -- they are in separate network namespaces. Docker creates a virtual ethernet pair (veth) to connect each container's network namespace to the host's network bridge, enabling communication between containers and the outside world.

- **MNT (Mount):** Isolates the file system. The container only sees its own root directory `/`, and cannot see the host's `/var` or `/home`. Mount namespaces ensure that mount and unmount operations inside the container do not affect the host's filesystem, and vice versa. This is what makes it possible to give each container its own root filesystem (provided by the container image) without containers interfering with each other's files.

- **UTS:** Allows the container to have its own Hostname. The UTS (Unix Time-Sharing System) namespace isolates the hostname and NIS domain name. This is useful for applications that rely on the hostname for identification or clustering, allowing each container to have a unique identity.

### 2. Cgroups (Control Groups): The "Resource Police"

If Namespaces dictate what a process can *see*, Cgroups dictate what a process can *use*.

Without Cgroups, a buggy containerized app could consume 100% of the host's RAM, crashing the whole server. Cgroups enforce strict hardware quotas: "Container A can only use 1 CPU core and 512MB of RAM." If it exceeds RAM, the kernel kills the process (OOM Killer).

Cgroups provide accounting, limiting, and prioritization of resource usage across the following subsystems:

- **CPU:** Limit the CPU shares or absolute CPU time a cgroup can consume. This prevents a single container from monopolizing the host's CPU cores.
- **Memory:** Set hard and soft limits on RAM usage. When a container exceeds its hard memory limit, the OOM killer terminates the offending process.
- **Block I/O:** Throttle the read/write bandwidth to disk, preventing one container from saturating the storage subsystem.
- **Network:** In combination with traffic control (tc), limit the network bandwidth available to a cgroup.
- **PIDs:** Limit the number of processes a cgroup can create, preventing fork bombs.

Cgroups are hierarchical: a child cgroup cannot exceed the limits of its parent. This enables nested resource allocation, which is used by container orchestration systems like Kubernetes where a namespace (in the Kubernetes sense) might have a total resource quota, and individual pods within that namespace are allocated slices of that quota.

### 3. UnionFS / OverlayFS: Layered File Systems

This is the magic behind Docker Images. Images are built in layers to save massive amounts of disk space.

- **Lowerdir (Read-Only layers):** When you download `ubuntu:latest`, it is stored on the host disk as read-only. If 10 containers use `ubuntu:latest`, they all share the exact same physical files on the disk.

- **Upperdir (Read-Write layer):** When you launch a container, Docker creates a thin, empty Read/Write layer on top.

- **Copy-on-Write (CoW):** If Container A wants to modify a system file located in the Read-Only base, OverlayFS *copies* that file up into the Read/Write layer, and modifies the copy. The base image remains untouched for everyone else.

OverlayFS is the default storage driver for modern Docker installations. It works by presenting a merged view of the upper and lower layers to the container. When the container reads a file that exists in the lower layer, OverlayFS serves it from there. When the container writes to a file that exists in the lower layer, OverlayFS copies the file to the upper layer first (the copy-on-write operation) and then modifies the copy. When the container creates a new file, it goes directly into the upper layer. When the container deletes a file from the lower layer, OverlayFS creates a "whiteout" marker in the upper layer that hides the file from the merged view.

> [!TIP] Troubleshooting Reminder
> Because the Upperdir (R/W layer) is ephemeral (temporary), if you delete the container, all data inside it is wiped. To save data permanently in Docker, you must mount a **Volume** (which bypasses UnionFS and writes directly to the host disk).
