### 3.3 User Datagram Protocol (UDP) Architecture

#### 1. UDP Operational Philosophy
The User Datagram Protocol (UDP) provides a simple, direct interface between the network layer and application processes. UDP does not implement any of the reliability features found in TCP, such as connection establishment, flow control, congestion management, or data reordering. 

Instead, it functions as a lightweight, low-latency transmission protocol. It is ideal for real-time applications that can tolerate occasional packet loss but require minimal delay, or for simple query-response protocols (like DNS) that want to avoid the overhead of setting up a connection.

---

#### 2. UDP Packet Header Format
The simplicity of UDP is reflected in its header. While a TCP header is at least 20 bytes long, a UDP header is fixed at just **8 bytes**, split into four 16-bit fields:

```mermaid

    block:SrcPort["Source Port<br/>(16 bits)"]:2
    block:DstPort["Destination Port<br/>(16 bits)"]:2
    block:Length["Length<br/>(16 bits)"]:2
    block:Checksum["Checksum<br/>(16 bits)"]:2
```

* **Source Port (16 bits):** Identifies the sending application's port on the source host. This port is also used as the destination port for return traffic.
* **Destination Port (16 bits):** Specifies the receiving application's port on the destination host.
* **Length (16 bits):** Specifies the total length of the UDP datagram (Header + Payload) in bytes.
  $$\text{Minimum Length} = 8 \text{ bytes (Header only)}$$
* **Checksum (16 bits):** An optional field used to verify the integrity of the UDP header and payload. It also covers a "pseudo-header" containing the source and destination IP addresses, helping to detect packets that have been misrouted.
