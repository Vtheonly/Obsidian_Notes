
# 4.4. Routing Regular Expression Matchers

## 1. When to Use Regular Expressions for Routing
While standard path converters (`int`, `slug`, `uuid`) cover most use cases, complex routing rules require more advanced matching logic. For example, if you need to enforce specific string lengths, numeric ranges, or composite dynamic paths, you should use regular expression matchers.

To use regular expressions, define your patterns using `re_path` instead of `path`.

## 2. Regular Expression Syntax in Routing
`re_path` uses named capturing groups to extract parameters from URLs and pass them to views as keyword arguments:
```python
(?P<name>pattern)
```
* **`?P`**: Declares a named capturing group.
* **`<name>`**: Defines the parameter name passed to your view.
* **`pattern`**: The regular expression rule applied to the capture group.

### Routing Example: Archival Records Matcher
This route requires a 4-digit year, a 2-digit month, and a 2-digit day format:
```python
from django.urls import re_path
from . import views

urlpatterns = [
    # Matches URLs like: '/archives/2026/05/22/'
    # Rejects URLs like: '/archives/26/5/22/'
    re_path(
        r'^archives/(?P<year>[0-9]{4})/(?P<month>[0-9]{2})/(?P<day>[0-9]{2})/$', 
        views.archive_view, 
        name='archive-records'
    ),
]
```

## 3. View Implementation
The captured arguments are passed to the view as keyword arguments:
```python
from django.http import HttpResponse

def archive_view(request, year, month, day):
    # The arguments are passed as strings and must be parsed or handled as needed
    return HttpResponse(f"Archival query for date format: {year}-{month}-{day}")
```

## 4. Comparison: `path()` vs. `re_path()`

| Routing Matcher | Readability | Validation Level | Performance | Maintenance |
| :--- | :--- | :--- | :--- | :--- |
| **`path()`** | High | Basic (Limited to type checks) | High | Simple |
| **`re_path()`** | Medium-Low | High (Enforces length, range, and character constraints) | High | Requires regex knowledge |
```
