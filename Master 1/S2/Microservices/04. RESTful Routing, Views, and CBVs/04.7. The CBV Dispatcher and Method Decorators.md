
# 4.7. The CBV Dispatcher and Method Decorators

## 1. Deep-Dive: The `dispatch` Method
The **`dispatch`** method is the central entrypoint of a Class-Based View. When a CBV is executed, the `as_view()` handler calls the `dispatch()` method first.

`dispatch` performs two key actions:
1. It inspects the HTTP request method (e.g., `GET`, `POST`, `PATCH`).
2. It routes the request to the matching method defined on the class if it exists (e.g., routing a `GET` request to the view's `get()` method). If the method is not defined or not supported by the view, it returns an automatic `HTTP 405 Method Not Allowed` response.

```python
# Conceptual representation of dispatch logic inside View:
def dispatch(self, request, *args, **kwargs):
    if request.method.lower() in self.http_method_names:
        handler = getattr(self, request.method.lower(), self.http_method_not_allowed)
    else:
        handler = self.http_method_not_allowed
    return handler(request, *args, **kwargs)
```

## 2. Applying Decorators to Class-Based Views
Decorators modify the behavior of a view function (e.g., requiring authentication or restricting access). Since CBVs are classes rather than functions, applying decorators to them requires a different approach.

To apply a decorator to a CBV, use Django's **`@method_decorator`** utility.

### Method 1: Decorating Individual Methods
You can apply decorators directly to specific methods (like `get()` or `post()`) inside your class:
```python
from django.views import View
from django.utils.decorators import method_decorator
from django.contrib.auth.decorators import login_required

class SecureView(View):
    # Apply login_required to the GET handler only
    @method_decorator(login_required)
    def get(self, request):
        return render(request, 'dashboard.html')
```

### Method 2: Decorating the Class Definition (Best Practice)
You can also apply decorators to the class definition. This is the cleanest approach because it keeps your view methods decoupled from security or access controls. You must specify the `name` parameter to target the target view method (such as `dispatch`):
```python
from django.views import View
from django.utils.decorators import method_decorator
from django.contrib.auth.decorators import login_required, permission_required

# Apply decorators to the class definition
@method_decorator(login_required, name='dispatch')
@method_decorator(permission_required('clinical.view_patient', raise_exception=True), name='dispatch')
class SecurePatientDashboard(View):
    def get(self, request):
        # This view inherits all class decorators automatically
        return render(request, 'dashboard.html')
```

## 3. Important Reminders & Student Traps
* **Decorating `dispatch` covers all methods**: Applying a decorator to the `dispatch` method (using `@method_decorator(..., name='dispatch')`) ensures the decorator runs for **all** HTTP request methods (`GET`, `POST`, `PUT`, etc.).
* **Failing to specify the `name` argument**: Attempting to apply `@method_decorator` directly to a class definition without specifying the `name` target parameter (e.g., writing `@method_decorator(login_required)` on top of a class) will raise a configuration error.
