
# 9.5. Programmatic Client Integrations with JWTs

## 1. Interacting with the Token Endpoints
Client-side applications (such as frontend SPAs or external microservices) must handle requesting, storing, and sending tokens to secure endpoints.

Below is a complete implementation using Python's `requests` library showing how a client connects to a secure API using JWTs.

```python
import requests
import sys

BASE_URL = "http://127.0.0.1:8000"

def run_client_pipeline():
    # 1. Authenticate and retrieve token pair (Login)
    login_url = f"{BASE_URL}/api/token/"
    credentials = {
        "email": "doctor@hospital.org",
        "password": "SecurePassword123"
    }
    
    print("[Client] Sending credentials to login endpoint...")
    response = requests.post(login_url, json=credentials)
    
    if response.status_code != 200:
        print(f"[Error] Authentication failed: {response.text}")
        sys.exit(1)
        
    tokens = response.json()
    access_token = tokens["access"]
    refresh_token = tokens["refresh"]
    print("[Client] Successfully retrieved Access and Refresh tokens.")

    # 2. Make an authorized request to a protected endpoint
    # The access token is sent in the Authorization header using the 'Bearer' prefix
    protected_url = f"{BASE_URL}/api/patients/"
    headers = {
        "Authorization": f"Bearer {access_token}",
        "Content-Type": "application/json"
    }
    
    print("[Client] Making authorized request to secure endpoint...")
    api_response = requests.get(protected_url, headers=headers)
    
    if api_response.status_code == 200:
        print("[Client] Data retrieved successfully:")
        print(api_response.json())
    elif api_response.status_code == 401:
        print("[Client] Access token expired or invalid.")
    else:
        print(f"[Error] Request failed with status code {api_response.status_code}")

if __name__ == "__main__":
    run_client_pipeline()
```

## 2. Standard Client-Side Storage Options
How you store tokens on the client depends on your application's security requirements:

| Storage Location | Security Vulnerabilities | Best Use Case | Implementation Details |
| :--- | :--- | :--- | :--- |
| **Local Memory (JS State)** | Cleared on page refresh. Safe from CSRF and XSS. | Access Tokens in Single-Page Applications (SPAs). | Keep the token in a local JavaScript variable. |
| **`HttpOnly` Cookie** | Safe from XSS. Vulnerable to CSRF (requires CSRF protection headers). | Refresh Tokens in secure web environments. | Set by the backend server during authentication. |
| **LocalStorage / SessionStorage** | Vulnerable to Cross-Site Scripting (XSS). Anyone with access to the console can read the tokens. | **Avoid in production** for sensitive credentials. | Read and write using `localStorage.setItem('token', value)`. |
