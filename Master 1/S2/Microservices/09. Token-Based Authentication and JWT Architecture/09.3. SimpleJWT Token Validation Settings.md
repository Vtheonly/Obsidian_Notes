
# 9.3. SimpleJWT Token Validation Settings

## 1. Configuring the `SIMPLE_JWT` Setting Dictionary
You can customize token lifetimes, algorithms, and validation behaviors in `settings.py` using the `SIMPLE_JWT` configuration dictionary.

Add the following configuration block to **`settings.py`**:

```python
# settings.py
from datetime import timedelta
import os

SIMPLE_JWT = {
    # 1. Token Lifetimes
    'ACCESS_TOKEN_LIFETIME': timedelta(minutes=15),
    'REFRESH_TOKEN_LIFETIME': timedelta(days=7),
    
    # 2. Token Security and Rotation Policies
    # If True, requesting a new access token using a refresh token 
    # will also return a brand new refresh token, resetting its lifetime.
    'ROTATE_REFRESH_TOKENS': True,
    # If True, old refresh tokens are immediately blacklisted 
    # after they are used to generate a new access token.
    'BLACKLIST_AFTER_ROTATION': True,
    'UPDATE_LAST_LOGIN': True,

    # 3. Cryptographic Signature Rules
    'ALGORITHM': 'HS256',
    'SIGNING_KEY': os.getenv('SECRET_KEY', 'django-insecure-fallback-key-123'),
    'VERIFYING_KEY': None,
    'AUDIENCE': None,
    'ISSUER': None,
    'JSON_ENCODER': None,

    # 4. Token Location Headers
    # The HTTP Header name and prefix required for API requests
    # Example Header: "Authorization: Bearer <token>"
    'AUTH_HEADER_TYPES': ('Bearer',),
    'AUTH_HEADER_NAME': 'HTTP_AUTHORIZATION',
    'USER_ID_FIELD': 'id',
    'USER_ID_CLAIM': 'user_id',
    'USER_AUTHENTICATION_RULE': 'rest_framework_simplejwt.authentication.default_user_authentication_rule',

    # 5. Token Properties Setup
    'AUTH_TOKEN_CLASSES': ('rest_framework_simplejwt.tokens.AccessToken',),
    'TOKEN_TYPE_CLAIM': 'token_type',
    'TOKEN_USER_CLASS': 'rest_framework.permissions.WrappedUser',

    # 6. Blacklist configuration
    'JTI_CLAIM': 'jti',
}
```

## 2. Managing Token Lifetimes and Security Trade-offs
* **Access Tokens (`ACCESS_TOKEN_LIFETIME`)**: Because access tokens are stateless and cannot be easily revoked before they expire, they should have a **short lifetime** (e.g., 5 to 15 minutes). This limits the window of opportunity for an attacker if a token is intercepted.
* **Refresh Tokens (`REFRESH_TOKEN_LIFETIME`)**: Refresh tokens are used to request new access tokens without requiring the user to re-enter their credentials. They are stateful (stored in the database) and can have a **longer lifetime** (e.g., 7 to 30 days) because they can be blacklisted if compromised.

## 3. Best Practices for Rotation Policies
Enabling `ROTATE_REFRESH_TOKENS` and `BLACKLIST_AFTER_ROTATION` creates a security pipeline known as **Refresh Token Rotation**. 

Every time a client requests a new access token:
1. They submit their current refresh token.
2. The server invalidates the old refresh token and returns a new access token paired with a new refresh token.
3. If an attacker attempts to reuse a blacklisted refresh token, the server can detect the reuse and revoke all tokens associated with that user session immediately.
