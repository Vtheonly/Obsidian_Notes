
# 8.6. Authentication Status Helpers in View Controllers

## 1. Check User Connection Status
Django automatically attaches a user object (`request.user`) to every incoming request. You can check whether a user is logged in or browsing anonymously by accessing their connection status properties:

* **`request.user.is_authenticated`**: Returns `True` if the user is logged in. Returns `False` if the user is browsing anonymously.
* **`request.user.is_anonymous`**: Returns `True` if the user is browsing anonymously. Returns `False` if the user is logged in.

### Example: View Controller Check
```python
from django.shortcuts import render

def home_view(request):
    if request.user.is_authenticated:
        # User is logged in. Render their personal dashboard.
        return render(request, 'dashboard.html', {'username': request.user.username})
    else:
        # Anonymous visitor. Render the generic public landing page.
        return render(request, 'public_landing.html')
```

## 2. Protecting Views Using Decorators and Mixins
To prevent unauthorized users from accessing sensitive views, you can protect your views using decorators (for Function-Based Views) or mixins (for Class-Based Views).

### Protecting Function-Based Views (FBVs)
Use the **`@login_required`** and **`@permission_required`** decorators:
```python
from django.contrib.auth.decorators import login_required, permission_required
from django.shortcuts import render

@login_required(login_url='/accounts/login/')
@permission_required('clinical.change_patient', raise_exception=True)
def edit_patient_view(request):
    # This view will only run if the user is logged in AND has the required permission
    return render(request, 'edit_patient.html')
```

### Protecting Class-Based Views (CBVs)
For Class-Based Views, inherit from Django's built-in mixins: **`LoginRequiredMixin`** and **`PermissionRequiredMixin`**:
```python
from django.contrib.auth.mixins import LoginRequiredMixin, PermissionRequiredMixin
from django.views import View
from django.shortcuts import render

# Class-Based Views must inherit mixins FIRST, before inheriting from View
class PatientSecureDashboard(LoginRequiredMixin, PermissionRequiredMixin, View):
    # Configure required properties
    login_url = '/accounts/login/'
    redirect_field_name = 'next'
    
    # Configure permission rules
    permission_required = 'clinical.view_patient'
    raise_exception = True  # Raises an HTTP 403 Forbidden response instead of redirecting

    def get(self, request):
        return render(request, 'secure_dashboard.html')
```

## 3. Important Reminders & Student Traps
* **Inheritance Order Matters**: When protecting Class-Based Views, you must list your mixins **before** the base `View` class in your inheritance list (e.g., `class MyView(LoginRequiredMixin, View)`). Listing them in the wrong order can cause the mixin security checks to be bypassed.
* **Avoid Silent Redirects for APIs**: For REST APIs, you should avoid redirecting unauthorized users to a HTML login page. Instead, configure your view or permission classes to raise an exception (`raise_exception=True`), which returns an appropriate `HTTP 403 Forbidden` response to the client.
