
# 01.4. Distributed Systems Component Overview

> [!info] The Microservice Ecosystem
> Breaking a monolith into microservices solves many problems, but it introduces network complexity. To manage this, a microservice ecosystem relies on specific infrastructure components.

## Standard Components of the Ecosystem

### 1. The Services (Stateless & Stateful)
The actual application code (e.g., Django, Spring Boot). 
* **Stateless**: The service does not store any local data between requests. This allows the service to be destroyed and recreated instantly (ideal for auto-scaling).
* **Stateful**: Services that maintain continuous state, typically databases or cache instances.

### 2. API Gateway
The front door of the system. Clients do not talk to microservices directly; they talk to the API Gateway, which routes the request to the correct internal service. (See [[01.5. API Gateway Pattern and Responsibilities]]).

### 3. Message Broker / Event Bus
Used for asynchronous communication. Instead of Service A waiting for Service B to respond via HTTP, Service A drops a message in a broker (like RabbitMQ or Apache Kafka) and moves on. Service B picks up the message when it is ready.

### 4. Service Registry and Discovery
Because microservices are constantly scaling up and down, their IP addresses change dynamically. A Service Registry (like HashiCorp Consul) acts as a dynamic phonebook. Services register themselves when they boot up, and others query the registry to find them.

### 5. Database per Service
A core tenet of microservices. To prevent tight coupling, services should not share a database schema. If Service A needs data from Service B, it must make an API request. This prevents schema changes in Service B from accidentally breaking Service A.

### 6. Observability (Monitoring, Logging, Tracing)
Because a single user action might travel through 5 different microservices, debugging is incredibly difficult.
* **Logging**: Centralized logs (e.g., ELK Stack) gather logs from all services into one place.
* **Tracing**: Tools (e.g., Jaeger) attach a unique `Trace-ID` to a request so you can visually see how much time it spent in every single microservice along its journey.
