
# 11.7. Managing Peer Consensus and Client Nodes

## 1. Verifying Cluster Integrity
Once you have started your nodes, verify that your cluster is stable, healthy, and has successfully reached a quorum.

## 2. Diagnostic Commands

### I. `consul members`
Displays a list of all nodes currently participating in the LAN gossip pool:
```bash
consul members -http-addr=http://127.0.0.1:8500
```

#### Expected Terminal Output:
```text
Node              Address         Status  Type    Build   Protocol  DC   Segment
leader1           127.0.0.1:8301  alive   server  1.19.0  2         dc1  <all>
follower1         127.0.0.1:8304  alive   server  1.19.0  2         dc1  <all>
client-agent-1    127.0.0.1:8307  alive   client  1.19.0  2         dc1  <default>
```

### II. `consul operator raft list-peers`
Displays Raft consensus details, including voter status and which server is currently the elected Leader:
```bash
consul operator raft list-peers -http-addr=http://127.0.0.1:8500
```

#### Expected Terminal Output:
```text
Node       ID                                    Address         State     Voter  RaftProtocol
leader1    0f10c14b-ef54-4680-ba65-9ba444455555  127.0.0.1:8300  leader    true   3
follower1  99ab9901-cc11-4466-bbff-1a2b3c4d5e6f  127.0.0.1:8305  follower  true   3
```

## 3. Key Diagnostic Metrics
* **`Status = alive`**: Indicates that the node is healthy and actively communicating with the gossip pool.
* **`State = leader`**: Indicates which server node is currently coordinating write requests.
* **`Voter = true`**: Indicates that the node has voting rights and participates in Leader elections. Client nodes are not voters, so they do not appear in Raft peer lists.
