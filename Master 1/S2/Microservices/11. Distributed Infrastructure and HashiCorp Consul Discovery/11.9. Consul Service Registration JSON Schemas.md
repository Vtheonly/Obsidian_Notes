
# 11.9. Consul Service Registration JSON Schemas

## 1. The Structure of Service Definitions
You can register services with Consul dynamically using HTTP APIs or by writing static JSON configuration files.

Below is a static JSON configuration schema for registering a Django application service with Consul:

```json
{
  "ID": "django-service-node-1",
  "Name": "django-clinical-service",
  "Tags": ["primary", "api-layer"],
  "Address": "127.0.0.1",
  "Port": 8000,
  "Meta": {
    "version": "1.4.2",
    "environment": "production"
  },
  "Check": {
    "HTTP": "http://127.0.0.1:8000/api/health/",
    "Interval": "10s",
    "Timeout": "3s",
    "DeregisterCriticalServiceAfter": "1m"
  }
}
```

## 2. Registering Services Using raw HTTP APIs
To register a service dynamically, send a POST request containing your service schema to the local Consul client agent's registration endpoint (`/v1/agent/service/register`):

```bash
curl --request PUT \
  --url http://127.0.0.1:8502/v1/agent/service/register \
  --header 'Content-Type: application/json' \
  --data '{
    "ID": "django-service-node-1",
    "Name": "django-clinical-service",
    "Address": "127.0.0.1",
    "Port": 8000,
    "Check": {
      "HTTP": "http://127.0.0.1:8000/api/health/",
      "Interval": "15s"
    }
  }'
```

## 3. Querying Registered Services
To retrieve a list of healthy instances for a specific service:
```bash
# Query Consul for all registered instances of 'django-clinical-service'
curl http://127.0.0.1:8502/v1/catalog/service/django-clinical-service
```
