
# 10.1. Modifying Base Identities The AbstractBaseUser Class

## 1. Why Define a Custom User Model?
Django includes a built-in user model out of the box. However, real-world applications often have requirements that the default model does not support:
* Logging in with an **email address** instead of a username.
* Adding application-specific fields directly to the user model (such as a profile picture, user role, or telephone number).
* Changing how passwords or user states are validated and saved.

You can modify Django's user engine by inheriting from one of two classes:

* **`AbstractUser`**: Inherits all standard fields and behaviors from the default `User` model (such as first name, last name, and username).
  * *When to use*: If you are happy with the default username-based login flow, but want to add a few custom fields (e.g., phone number).
* **`AbstractBaseUser`**: Provides only the bare-minimum authentication requirements (such as password hashing, password reset helpers, and login tracking). It does not include standard fields like first name, last name, or username.
  * *When to use*: If you want complete control over your user schema (such as switching to an email-based login flow or using custom permission logic).

## 2. Setting Up a Custom User Model
To configure a custom user model, you must inherit from two base classes:

```python
from django.contrib.auth.models import AbstractBaseUser, PermissionsMixin

class CustomUser(AbstractBaseUser, PermissionsMixin):
    pass
```
* **`AbstractBaseUser`**: Provides core authentication features like password hashing and session tracking.
* **`PermissionsMixin`**: Adds support for Django's built-in group and permission structures. This allows you to continue using standard features like `request.user.has_perm()` and assigning users to groups in the admin panel.
