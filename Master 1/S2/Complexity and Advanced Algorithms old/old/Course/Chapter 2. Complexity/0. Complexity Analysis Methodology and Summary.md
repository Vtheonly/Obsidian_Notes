**1. Counting exact operations for Best and Worst Cases (Page 2)**
To find the complexity of an algorithm with nested loops (like searching an $n \times n$ matrix), you break down the mathematical operations:
*   **Best Case:** Count the exact number of operations if the algorithm stops on the very first try. The document shows this yields a fixed number (e.g., 16 operations). A constant number is defined as **$O(1)$**.
*   **Worst Case:** You calculate the formula for the number of operations ("Nbr d'op") from the inside of the loop to the outside:
    1.  **Inner Loop:** Sum the operations inside the loop (e.g., $1 + 2 + 3$) and multiply by the number of times it runs ($n$). *Result: $6 \times n$.*
    2.  **Outer Loop:** Take the inner loop's total ($6n$), add the outer loop's operations ($1 + 2 + 3$), and multiply that entire sum by the outer loop's iterations ($n$). *Result: $(1 + 6n + 5) \times n = 6n^2 + 6n$.*
    3.  **Total:** Add any constant operations that happen outside the loops to get your final polynomial (e.g., $6n^2 + 6n + 3$).
    4.  **Big-O:** Take the highest power of $n$ from your final polynomial. Because the highest term in $6n^2 + 6n + 4$ is squared, the complexity is **$O(n^2)$**.

**2. Calculating the Average Case Complexity / $T_{moy}$ (Page 3)**
Average complexity is counted using probability and summations:
*   **Assume Uniformity:** The probability of an element being in any position is equal. For a matrix of size $n^2$, the probability is **$1/n^2$**.
*   **Define cost per position $C(k)$:** The number of comparisons to find an item at the $k^{th}$ position is exactly $k$.
*   **Use the Summation Formula:** Multiply the probability by the cost, and sum it for all possible positions: 
    $T_{moy} = \sum (\text{probability} \times C(k))$
*   **Solve the Math:** Replace the summation of $k$ with the mathematical rule $\frac{n^2(n^2 + 1)}{2}$. Multiply this by your probability ($1/n^2$) to get the final average formula: **$\frac{n^2 + 1}{2}$**.

**3. Direct Mapping based on the Problem Type (Page 4)**
For standard algorithmic problems, the document does not count operations but assigns complexity based on a memorized lookup table:
*   **$O(1)$**: Inserting at the head of a linked list, inserting at the end of an array.
*   **$O(n)$**: Sum of two vectors, calculating a factorial, iterative sum of $1$ to $n$, inserting at the head of an array, inserting at the end of a linked list, testing if a number is even/odd.
*   **$O(n^2)$**: Sum of two square matrices of order $n$.
*   **$O(n^3)$**: Product of two square matrices of order $n$.

**4. Proving a Complexity Formula is Correct (Page 5)**
To verify if a complexity claim like $f(n) \in O(g(n))$ is mathematically true or false, you must prove it using the Big-O inequality formula:
*   **Set up the inequality:** Write **$f(n) \le c \cdot g(n)$**.
*   **Simplify using algebra:** Break down exponents. For example, turn $2^{n+1}$ into $2^n \times 2^1$, or turn $2^{2n}$ into $(2^n)^2$.
*   **Find constant $c$:** Try to find a constant, positive number for $c$ that makes the inequality true for any $n \ge 1$.
    *   *If True:* If you can isolate a constant (e.g., $2^n \times 2 \le c \cdot 2^n$ works perfectly if $c=2$), the formula is correct ("Vrai").
    *   *If False:* If the math leaves you with a growing variable on the left side that a constant $c$ cannot cover (e.g., $(2^n)^2 \le c \cdot 2^n$), the formula is incorrect ("Faux").
