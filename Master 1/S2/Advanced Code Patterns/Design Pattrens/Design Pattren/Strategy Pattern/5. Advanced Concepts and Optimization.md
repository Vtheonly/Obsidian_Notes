### Do we even need Subclasses?

In the original inheritance model, we had classes like `MallardDuck`, `RedheadDuck`, `RubberDuck`.

With the Strategy Pattern fully implemented, you might realize something profound:
**Do we actually need a `MallardDuck` class?**

If a "Mallard Duck" is simply a `Duck` configured with:

1.  `SimpleFlying` behavior
2.  `LoudQuack` behavior
3.  `VisualMallard` display behavior

Then `MallardDuck` is just a **configuration**, not necessarily a distinct class type.

### The "Display" Strategy

We can apply the Strategy pattern to the `display()` method as well.

- Create `IDisplayBehavior`.
- Implement `GraphicDisplay`, `TextDisplay`, `3DDisplay`.

If we do this, the `Duck` class becomes a purely composed object:

```java
Duck mallard = new Duck(new SimpleFlying(), new LoudQuack(), new GraphicDisplay());
```

We have eliminated the class explosion. We don't need hundreds of `Duck` subclasses; we just need instances of `Duck` configured with different combinations of strategies.

### Naming Conventions

Naming is crucial in Strategy Pattern to avoid confusion.

- **Bad:** `SimpleFly`, `Jet`, `No`. (Unclear what they represent).
- **Good:** `FlyWithWings`, `FlyWithRocket`, `FlyNoWay`. (Describes the strategy clearly).
- **Tip:** Ensure your interface names imply behavior (e.g., `IFlyBehavior` or `FlyStrategy`), and your concrete classes describe _how_ they implement that behavior.
