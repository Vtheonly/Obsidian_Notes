### Core Principles Learned

1.  **Encapsulate what varies:** If a behavior (like flying) changes between objects, move it out of the main class.
2.  **Favor Composition over Inheritance:** "Has-A" is more flexible than "Is-A". It allows you to change behavior at runtime and reuse code across unrelated classes.
3.  **Program to an Interface, not an implementation:** The Duck class interacts with `IFlyBehavior`, not `JetFlying`. It doesn't care _how_ it flies, only _that_ it flies.

### Benefits of Strategy Pattern

- **Eliminates Code Duplication:** Behaviors are written once in their own classes and reused by any client.
- **Clean Code:** Removes massive `if (type == rubber) { ... } else if (type == mallard) { ... }` blocks.
- **Open/Closed Principle:** The system is open for extension (you can add a `TeleportFlying` behavior) but closed for modification (you don't have to touch the `Duck` code to add it).

### Common Pitfalls

- **Class Explosion:** If you have 50 variations of flying that are only slightly different, you might end up with 50 strategy files.
- **Over-Engineering:** If the behavior never changes and all subclasses do exactly the same thing, you do not need the Strategy Pattern. Don't apply it blindly to every method.

### Final Memory Aid

Think of the Strategy Pattern like a **Video Game Character's Weapon**.

- The Character is the **Client**.
- The Weapon is the **Strategy Interface**.
- The Sword, Bow, and Axe are the **Concrete Strategies**.
- You can "equip" (compose) the character with a Bow.
- When you press "Attack" (delegate), the Character doesn't know how to shoot an arrow; the _Bow_ knows how to shoot.
- You can swap the Bow for an Axe in the middle of the game (Runtime change).
