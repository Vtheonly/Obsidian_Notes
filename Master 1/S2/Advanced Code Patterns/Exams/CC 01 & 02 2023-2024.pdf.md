# Part 1: Continuous Assessment MCA — Telemedicine Platform "TelMedUMBB"

## 1. Use Case Diagram (Functional Requirements View)

### Methodological Rules Applied (From Lecture Slides)
* **Naming Convention**: Every Use Case is named with an infinitive verb followed by a direct object/complement (e.g., *Consulter dossier*, *Planifier rendez-vous*).
* **Actor Identification**:
  * **Principal Actors** (Slide 11): Those who directly interact with the system to initiate and benefit from a use case (e.g., `Patient`, `Réceptionniste`, `Infirmière`, `Médecin`).
  * **Secondary Actors** (Slide 11): Those who are requested by the system to perform a transaction or provide information.
* **Stereotyped Associations**: When secondary actors are introduced, associations are stereotyped as `<<principal>>` or `<<secondaire>>` (Slide 11).
* **Include Relationship (`<<include>>`)**: Applied for mandatory dependencies (e.g., authentication `S'authentifier` is an absolute pre-requisite for accessing and modifying the EHR/DMI).

## 2. Class Diagram (Logical View)

### Methodological Rules Applied (From Lecture Slides)
* **Encapsulation & Visibility**: Attributes are declared private (`-`), operations are public (`+`) (Slide 11).
* **Typing**: Attributes and methods are strictly typed with standard types (e.g., `String`, `Date`, `bool`, `void`) (Slide 12).
* **Multiplicity**: Annotated on every association end (e.g., `0..*`, `1`) to denote limits precisely (Slide 28).
* **Inheritance/Generalization**: The logical view sets up an inheritance tree where clinic personnel and patients inherit generic identification attributes from an abstract class `Utilisateur` (Slide 59).

# Part 2: Evaluation 2 — Daily Flight Theft Reports System ("Rapports Quotidiens de Vol - RQV")

## 1. Structural Class Diagram

### Methodological Rules Applied (From Lecture Slides)
* **Inheritance on Entities**: Stolen property (`Property`) is modeled using generalization/specialization into `Bicycle` and `MotorVehicle` (Slide 59) because they have distinct identifiers (`numero de serie` vs `matricule`).
* **Attributes Grouping**: Addresses are grouped into an object structure class (`Address`) to avoid bloated attributes in the `Declaration` class.
* **Association Class / Qualifier Options**: The direct mapping shows a clear dependency where declarations are linked to the daily reports (RQV).

```mermaid
classDiagram
    direction TB

    class Utilisateur {
        <<abstract>>
        -cin: String
        -nom: String
        -prenom: String
        -adresse: String
        -telephone: String
    }

    class Declarant {
        -role: String
        +creerDeclaration() Declaration
        +modifierDeclaration(id: String) void
        +supprimerDeclaration(id: String) void
    }

    class AgentPolice {
        -badgeId: String
        -motDePasse: String
        +authentifier() bool
        +creerRapportQuotidien(date: Date) RQV
        +modifierStatutDeclaration(id: String, statut: String) void
    }

    class Property {
        <<abstract>>
        -marque: String
        -couleur: String
    }

    class Bicycle {
        -numSerie: String
    }

    class MotorVehicle {
        -matricule: String
        -descriptionGenerale: String
    }

    class Address {
        -numRue: String
        -ville: String
        -codePostal: String
        -detailsSupplementaires: String
    }

    class Declaration {
        -idDeclaration: String
        -dateTheft: Date
        -heureTheft: Time
        -dateDerniereModif: DateTime
        -statut: String
        -estSauvegarde: bool
        +ajouterDetails() void
        +supprimer() void
        +sauvegarder() void
    }

    class RQV {
        -dateRapport: Date
        +genererRapport() void
        +obtenirNouvellesDeclarations() List~Declaration~
        +obtenirDeclarationsModifiees() List~Declaration~
        +obtenirDeclarationsResolues() List~Declaration~
    }

    Utilisateur <|-- Declarant
    Utilisateur <|-- AgentPolice

    Property <|-- Bicycle
    Property <|-- MotorVehicle

    Declaration "1" *-- "1" Property : concerne >
    Declaration "1" *-- "1" Address : a_eu_lieu_a >
    Declarant "1..*" -- "0..*" Declaration : declare >

    RQV "0..*" -- "0..*" Declaration : documente >
    AgentPolice "1" -- "0..*" RQV : redige >
```

## 2. Sequence Diagrams

### a) Creating a New Declaration (Créer une nouvelle déclaration)

```mermaid
sequenceDiagram
    autonumber
    actor D as Declarant (Victime/Témoin)
    participant UI as Interface Déclaration
    participant Ctrl as ControleurDéclaration
    participant DB as Base de Données / Système

    D->>UI: Demander la création d'une déclaration
    UI->>Ctrl: initialiserDeclaration()
    Ctrl->>DB: genererIdentifiant()
    DB-->>Ctrl: idDeclaration
    Ctrl-->>UI: Formulaire de déclaration vide (idDeclaration)
    UI-->>D: Afficher le formulaire

    loop Saisie et Modification (avant sauvegarde)
        D->>UI: Renseigner informations (Rôle, Propriété, Lieu, Date, Heure)
        UI->>Ctrl: ajouterDetailsTemporaires()
        Ctrl-->>UI: Confirmer modification temporaire
    end

    alt Optionnel: Supprimer avant sauvegarde
        D->>UI: Annuler / Supprimer la déclaration
        UI->>Ctrl: annulerDeclaration()
        Ctrl-->>UI: Formulaire réinitialisé / fermé
    end

    D->>UI: Valider et Sauvegarder la déclaration
    UI->>Ctrl: validerDeclaration(donnees)
    Ctrl->>Ctrl: definirDateDerniereModif(DateTime.now)
    Ctrl->>DB: sauvegarder(Declaration)
    DB-->>Ctrl: Confirmation de sauvegarde
    Ctrl-->>UI: Message de succès (Déclaration enregistrée)
    UI-->>D: Afficher la confirmation
```

### b) Creating a Daily Flight Report (Créer un rapport quotidien de vol - RQV)

```mermaid
sequenceDiagram
    autonumber
    actor AP as Agent Policier
    participant UI as Interface Police
    participant Ctrl as ControleurRQV
    participant DB as Base de Données

    AP->>UI: Authentification (badgeId, password)
    UI->>Ctrl: authentifier(badgeId, password)
    Ctrl->>DB: verifierIdentite(badgeId, password)
    DB-->>Ctrl: authentificationOk
    Ctrl-->>UI: Acces autorise
    UI-->>AP: Afficher le tableau de bord de l'agent

    AP->>UI: Demander création RQV (dateCible)
    UI->>Ctrl: genererRapportQuotidien(dateCible)
    
    Ctrl->>DB: recupererDeclarationsParDate(dateCible)
    DB-->>Ctrl: listeToutesDeclarations
    
    Ctrl->>Ctrl: filtrerNouvelles(listeToutesDeclarations)
    Ctrl->>Ctrl: filtrerModifiees(listeToutesDeclarations)
    Ctrl->>Ctrl: filtrerResolues(listeToutesDeclarations)
    
    Ctrl->>Ctrl: creerRapport(dateCible, listesFiltrees)
    Ctrl->>DB: sauvegarderRapport(RQV)
    DB-->>Ctrl: Rapport enregistré
    Ctrl-->>UI: Afficher le rapport généré (3 sections distinctes)
    UI-->>AP: Présentation du RQV finalisé
```
