# 1.1 HPC Basics and Speedup

## The Golden Rule of HPC

In High Performance Computing, **algorithmic efficiency is the very first lever of performance**. Before you even think about buying a supercomputer, adding more RAM, or writing parallel code to utilize multiple cores, you must ensure your sequential (single-thread) algorithm is optimal. Parallelizing a poorly written, wasteful algorithm just gives you a bad algorithm that consumes more electricity — it will still be bad, just bad across more cores simultaneously.

> [!important] Core Principle
> A slow algorithm parallelized is still a slow algorithm. Always optimize the sequential version *first*, then parallelize.

The reason this matters is fundamental: parallelism provides at most a **linear speedup** (e.g., 4 cores can at best make your program 4x faster). But an algorithmic improvement — such as moving from $O(N^2)$ to $O(N \log N)$ — can provide speedups that grow *with the input size*, easily reaching factors of 10,000x or more for large inputs. No hardware upgrade on the market competes with that.

## Measuring Performance: Wall-Clock Time vs. CPU Time

To know if our optimization works, we must measure time. There are two distinct types of time in computing, and understanding the difference is essential for accurate benchmarking.

### Wall-Clock Time (Real Time)

The actual real-world time that elapses from the moment you start the program to the moment it finishes. If you start a script and go make a coffee for 5 minutes, the wall-clock time is 5 minutes, even if the CPU was only thinking for 1 second and waiting for a network download the rest of the time. This is the time a user *experiences* and the metric that ultimately matters for real-world performance.

Wall-clock time captures everything: CPU computation, I/O waits (disk reads/writes, network requests), sleep calls, and even time spent waiting for other processes that the OS scheduled on the same core. This makes it the most honest metric for end-to-end program performance.

### CPU Time

The amount of time the CPU actually spent processing instructions for your program. CPU time is split into two sub-categories:
- **User time:** Time spent executing your code (e.g., Python logic, mathematical computations).
- **System time:** Time the kernel spent doing work on your behalf (e.g., allocating memory, system calls like `print()` or file I/O).

CPU time can actually be *larger* than wall-clock time in multi-threaded programs, because multiple cores may be working simultaneously, each accumulating CPU time.

### Which One Do We Use?

> [!tip] In this course, we evaluate our gains using **Wall-Clock Time**.
> It reflects the total user experience. However, comparing CPU time to wall-clock time can reveal if your program is I/O-bound (CPU time ≪ wall-clock time) or compute-bound (CPU time ≈ wall-clock time).

## The Speedup Metric

To quantify exactly how much better our new code is compared to the old one, we use the **Speedup ($S$)** metric:

$$S = \frac{T_{\text{naive}}}{T_{\text{improved}}}$$

- $T_{\text{naive}}$: The execution time of the original, unoptimized baseline code.
- $T_{\text{improved}}$: The execution time of your new, optimized code.

### Interpreting Speedup Values

| Speedup Value | Meaning |
|:---:|:---|
| $S > 1$ | The improved version is **faster**. $S = 5$ means "5x faster." |
| $S = 1$ | No change in performance. The optimization had no measurable effect. |
| $S < 1$ | The improved version is **slower**! Your "optimization" actually made things worse. |

> [!warning] Common Mistake
> If your naive code took 10 seconds and your improved code takes 2 seconds, $S = 10 / 2 = 5$. This is a **5x speedup** — the improved code is 5 times faster. Do *not* say "it's 80% faster" (that would mean it takes 80% less time, which is 2 seconds vs 10 seconds, i.e., 5x, but the phrasing is ambiguous). Always report speedup as a ratio.

### Speedup in Parallel Contexts

When we later discuss parallelism (Chapter 3), speedup takes on an additional meaning. **Parallel Speedup** is:

$$S_p = \frac{T_{\text{serial}}}{T_{\text{parallel}}}$$

The theoretical maximum speedup with $P$ processors is $S_p = P$ (linear speedup). In practice, due to overhead from thread creation, synchronization, and Amdahl's Law (the serial portion of code that cannot be parallelized), we almost always observe $S_p < P$.

## Big-O Complexity — The Language of Algorithm Scaling

Before diving into specific exercises, it is crucial to understand **asymptotic time complexity**, commonly expressed using Big-O notation. Big-O describes how the running time of an algorithm grows as the input size $N$ grows toward infinity. It ignores constant factors and lower-order terms, focusing purely on the dominant growth rate.

| Big-O | Name | Example | If N doubles... |
|:---:|:---|:---|:---:|
| $O(1)$ | Constant | Hash table lookup | Time stays the same |
| $O(\log N)$ | Logarithmic | Binary search | Time increases slightly |
| $O(N)$ | Linear | Scanning a list | Time doubles |
| $O(N \log N)$ | Linearithmic | Merge sort | Time slightly more than doubles |
| $O(N^2)$ | Quadratic | Nested loop over list | Time quadruples |
| $O(2^N)$ | Exponential | Brute-force subset generation | Time squares! |

> [!note] Why Big-O Matters for HPC
> Moving one row up in this table (e.g., from $O(N^2)$ to $O(N \log N)$) can yield speedups that no hardware upgrade can match. This is why [[1.3 Algorithmic Optimization Exercises|algorithmic optimization]] must come before parallelism.
