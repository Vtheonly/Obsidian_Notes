# 2.2 Remote Execution and File Transfer

## Establishing Connection

The syntax for connecting to a remote machine via SSH is always:

```bash
ssh user@hostname_or_ip
```

- `user`: Your username on the remote machine (it may differ from your local username).
- `hostname_or_ip`: Either a fully qualified domain name (e.g., `cluster.univ-boumerdes.dz`) or an IP address (e.g., `192.168.1.50`).

**Example:**

```bash
ssh student@192.168.1.50
```

On first connection, you will see the authenticity prompt (as described in [[2.1 SSH and HPC Environments|2.1]]). Type `yes` to store the host key in `~/.ssh/known_hosts`. Future connections to the same host will skip this prompt.

Once connected, your terminal behaves exactly as if you were physically typing on the remote machine. Your shell prompt will change to reflect the remote hostname, and all commands you type execute on the remote machine using its CPU, memory, and filesystem.

### Identifying Your Environment

After connecting, it's good practice to verify where you are:

| Command | What It Prints | Why It Matters |
|:---|:---|:---|
| `hostname` | The machine's network name | Confirms you're on the right cluster node |
| `uname -a` | OS name, kernel version, architecture | Verify software compatibility (e.g., x86_64 vs ARM) |
| `whoami` | Your current logged-in username | Confirms you authenticated as the correct user |
| `pwd` | Current working directory | Usually starts in your home directory `~` |
| `nproc` | Number of CPU cores available | Essential for [[3.1 OpenMP and the Fork-Join Model|OpenMP]] thread planning |

## Running Commands Remotely Without an Interactive Session

Sometimes you want to execute a single command on the remote machine without opening a full interactive shell. Append the command directly:

```bash
ssh student@192.168.1.50 "hostname && nproc"
```

This connects, runs `hostname` and `nproc`, prints the output, and disconnects immediately. This is especially useful in shell scripts and automated workflows where you need to trigger remote computations or check remote status programmatically.

You can also chain complex commands:

```bash
ssh student@192.168.1.50 "cd ~/project && gcc -fopenmp program.c -o program && ./program"
```

## Secure Copy Protocol (SCP)

You cannot drag and drop files into an HPC cluster through a file explorer. The standard method for transferring files is **SCP (Secure Copy Protocol)**, which uses the same SSH encryption under the hood. All data in transit is fully encrypted.

### Syntax

```bash
scp [source] [destination]
```

The critical convention: a **colon (`:`)** after a hostname separates the machine address from the file path on that machine.

### Common Patterns

**Local → Remote (Upload):**
```bash
scp script.py student@192.168.1.50:~/
# Copies script.py to the home directory on the remote machine
```

**Remote → Local (Download):**
```bash
scp student@192.168.1.50:~/results.txt ./
# Downloads results.txt to the current local directory
```

**Copying an Entire Directory:**
```bash
scp -r ./project_folder student@192.168.1.50:~/workspace/
# The -r flag recursively copies the entire folder and its contents
```

**Specifying a Different Port:**
```bash
scp -P 2222 script.py student@192.168.1.50:~/
# Capital -P for SCP (lowercase -p is used for preserving file permissions)
```

> [!warning] SCP vs. SFTP vs. rsync
> - **SCP:** Simple, one-shot file transfer. Best for quick single-file copies.
> - **SFTP:** Interactive file transfer session (like FTP but encrypted). Allows browsing, resuming transfers. Use `sftp user@host`.
> - **rsync:** The most efficient option for large or repeated transfers. It only copies the *differences* (delta transfer), saving massive bandwidth. Use: `rsync -avz ./local_dir/ user@host:~/remote_dir/`

## Passwordless Authentication (SSH Keys)

Typing passwords repeatedly is slow, error-prone, and insecure for automated scripts. Moreover, many HPC clusters disable password authentication entirely, requiring key-based login. We use **Asymmetric Cryptography** to solve this.

### Step 1: Generate a Key Pair

```bash
ssh-keygen -t rsa -b 4096
```

This creates two keys in `~/.ssh/`:

- **Private Key** (`id_rsa`): **NEVER share this file.** It stays on your local machine only. If someone obtains your private key, they can impersonate you on any server that trusts the corresponding public key. Set its permissions to `chmod 600 ~/.ssh/id_rsa`.
- **Public Key** (`id_rsa.pub`): You can share this freely. It is placed on remote servers to verify your identity.

During generation, you'll be prompted for a **passphrase** — an optional password that encrypts your private key on disk. If you set one, you must enter it each time you use the key (unless you use `ssh-agent` to cache it). For HPC lab environments, an empty passphrase is common for convenience, but for production systems, always use a passphrase.

### Step 2: Copy Your Public Key to the Remote Server

```bash
ssh-copy-id student@192.168.1.50
```

This appends your public key to the remote server's `~/.ssh/authorized_keys` file. It will ask for your password one last time.

### Step 3: Test Passwordless Login

```bash
ssh student@192.168.1.50
```

If everything is configured correctly, you will be logged in **without being asked for a password**. Here's what happens behind the scenes:

1. Your client tells the server: "I want to authenticate as `student` using my RSA key."
2. The server looks up your public key in `~/.ssh/authorized_keys`.
3. The server encrypts a random **challenge** (a block of random data) using your public key and sends it to you.
4. Your client decrypts the challenge using your private key (only your private key can do this) and sends back the result.
5. The server verifies the response. If correct, you're authenticated — no password ever transmitted over the network.

### Troubleshooting

- **"Permission denied (publickey)":** The server doesn't recognize your key. Check that `~/.ssh/authorized_keys` on the remote contains your public key, and that file permissions are correct (`chmod 700 ~/.ssh` and `chmod 600 ~/.ssh/authorized_keys` on the remote).
- **"Agent admitted failure to sign":** Run `ssh-add` to add your key to the SSH agent.
- **"Host key verification failed":** The server's host key changed (could indicate a security issue or a server reinstall). Remove the old entry from `~/.ssh/known_hosts` if you're sure it's safe.

## SSH Configuration File (~/.ssh/config)

For convenience, you can create a config file that defines shortcuts:

```
Host cluster
    HostName 192.168.1.50
    User student
    IdentityFile ~/.ssh/id_rsa
    ForwardX11 yes
```

Now you can simply type `ssh cluster` instead of the full command. This is invaluable when managing multiple remote machines.
