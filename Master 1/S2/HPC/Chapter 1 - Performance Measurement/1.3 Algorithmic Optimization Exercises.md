# 1.3 Algorithmic Optimization Exercises

This section covers **Lab 1, Part I**. Here we apply our theory of Big-O complexity and mathematical insight to replace naive algorithms with dramatically faster ones. Each exercise follows the same pattern: understand the naive approach, identify its complexity, then discover the mathematical or data-structural insight that reduces the complexity class.

> [!important] The Pattern
> 1. Measure the naive implementation's time.
> 2. Analyze its Big-O complexity to understand *why* it's slow.
> 3. Apply a mathematical formula or better data structure.
> 4. Measure the improved implementation's time.
> 5. Compute speedup $S = T_{\text{naive}} / T_{\text{improved}}$.

---

## Exercise 1: Sum from 1 to N

### Problem

Compute the sum of integers from 1 to $N = 10{,}000{,}000$.

### Naive Implementation

```python
N = 10_000_000
total = 0
for i in range(1, N + 1):
    total += i
```

**Analysis:** This loop runs $N$ times. Its Time Complexity is $O(N)$ — linear in the input size. If $N$ grows 10x, the execution time grows 10x. If $N$ grows 1,000x, the time grows 1,000x. This is acceptable for small $N$, but becomes increasingly wasteful as $N$ grows, because we are performing $N$ additions when a single mathematical formula can produce the answer instantly.

### Improved Implementation — Gauss's Formula

**Mathematical Concept:** Carl Friedrich Gauss discovered as a child that the sum of an arithmetic progression can be calculated instantly using a closed-form formula:

$$\text{Sum} = \frac{N(N+1)}{2}$$

**Why it works:** Pair up the numbers from both ends: $(1 + N) + (2 + (N-1)) + (3 + (N-2)) + \ldots$ Each pair sums to $N + 1$, and there are $N/2$ pairs. Hence $(N+1) \times N/2$.

**Improved Code:**

```python
N = 10_000_000
total = (N * (N + 1)) // 2  # Use // for integer division
```

**Analysis:** This operation executes in exactly **1 step**, regardless of how large $N$ is. Its Time Complexity is $O(1)$ (Constant time). Whether $N$ is 10 million or 10 trillion, the computation takes the same tiny fraction of a microsecond. The speedup $S$ will be massive — for $N = 10{,}000{,}000$, you can expect speedups in the range of hundreds of thousands of times faster.

> [!tip] Why `//` instead of `/`?
> The `//` operator performs integer (floor) division in Python 3, returning an `int`. The `/` operator always returns a `float`, which can lose precision for very large $N$ (floats have ~15-16 significant digits). For $N = 10{,}000{,}000$, the product $N(N+1) = 100{,}000{,}010{,}000{,}000$ — well within `int` range but potentially imprecise as a `float`.

## Exercise 2: Prime Number Detection

### Problem

Find all prime numbers up to $N = 100{,}000$.

### Naive Implementation

```python
def is_prime_naive(n):
    if n < 2: return False
    for i in range(2, n):  # Checks EVERY number up to n-1
        if n % i == 0:
            return False
    return True

primes = [n for n in range(2, N+1) if is_prime_naive(n)]
```

**Analysis:** To check if a single number $n$ is prime, this tests every divisor from 2 to $n-1$, performing up to $n-2$ modulo operations. For the number 99,999, it checks 99,997 divisors. The overall complexity for finding all primes from 1 to $N$ is roughly $O(N^2)$, making this approach impractical for large $N$.

### Improved Implementation 1: The Square Root Bound

**Mathematical Concept:** If a number $n$ has a divisor $d$ greater than $\sqrt{n}$, then $n/d$ must be a corresponding divisor *smaller* than $\sqrt{n}$. For example, for 36: the factor pairs are $2 \times 18$, $3 \times 12$, $4 \times 9$, $6 \times 6$. Once you check divisors up to 6, you have already seen all possible factor pairs — checking 8, 9, 12, or 18 would be redundant because their partners (4.5, 4, 3, 2) were already checked. Therefore, we only *need* to check up to $\sqrt{n}$.

**Improved Code:**

```python
import math

def is_prime_improved(n):
    if n < 2: return False
    if n == 2: return True
    if n % 2 == 0: return False  # Skip even numbers entirely
    for i in range(3, int(math.isqrt(n)) + 1, 2):  # Only odd divisors
        if n % i == 0:
            return False
    return True
```

**Analysis:** Checking a single number now takes $O(\sqrt{n})$ instead of $O(n)$. For $n = 99{,}999$, we check up to ~316 instead of 99,997 — a **316x improvement** per number. The overall complexity for all primes up to $N$ drops to roughly $O(N \sqrt{N})$.

> [!note] The extra optimization of skipping even numbers (`n % 2 == 0` early return + `step=2` in range) cuts the loop iterations in half again, yielding a further ~2x practical speedup. These constant-factor optimizations matter in practice even though they don't change the Big-O class.

### Improved Implementation 2: Sieve of Eratosthenes (The "All at Once" Strategy)

The lab asks: *"Can we use a strategy that builds the list of primes all at once?"* Yes — the **Sieve of Eratosthenes**, one of the oldest known algorithms (circa 240 BC).

**Concept:**

1. Create a boolean array `is_prime[0..N]` with all values set to `True`.
2. Mark `is_prime[0]` and `is_prime[1]` as `False` (0 and 1 are not prime).
3. Starting from $p = 2$ (the first prime), mark all multiples of $p$ as `False`: 4, 6, 8, 10, ...
4. Move to the next `True` index (3), and mark all multiples of 3 as `False`: 6, 9, 12, 15, ...
5. Repeat up to $\sqrt{N}$. Any index still marked `True` is a prime.

**Why stop at $\sqrt{N}$?** Any composite number $n \leq N$ has at least one prime factor $\leq \sqrt{n} \leq \sqrt{N}$. So by the time we finish sieving with all primes up to $\sqrt{N}$, every composite has been marked.

**Code:**

```python
def sieve_of_eratosthenes(N):
    is_prime = [True] * (N + 1)
    is_prime[0] = is_prime[1] = False

    for p in range(2, int(math.isqrt(N)) + 1):
        if is_prime[p]:
            # Mark all multiples starting from p*p
            # (smaller multiples were already marked by smaller primes)
            for multiple in range(p * p, N + 1, p):
                is_prime[multiple] = False

    return [i for i in range(2, N + 1) if is_prime[i]]
```

**Analysis:** The Sieve has complexity $O(N \log \log N)$, which is nearly linear. It is dramatically faster than both the naive and sqrt-bound approaches because it completely eliminates repetitive division operations and replaces them with simple memory writes (addition-based stride through the array). For $N = 100{,}000$, the Sieve will finish in milliseconds whereas the naive approach can take minutes.

> [!tip] Optimization: Start marking from `p*p`
> Multiples of $p$ smaller than $p \times p$ (like $2p$, $3p$, etc.) were already marked as composite when we processed smaller primes (2, 3, etc.). Starting from $p \times p$ avoids redundant work.

## Exercise 3: Intersection of Two Large Lists

### Problem

Find elements appearing in both List A and List B (size 100,000 each).

### Naive Implementation

```python
intersection = []
for a in list_A:
    for b in list_B:
        if a == b:
            intersection.append(a)
            break
```

**Analysis:** This is a nested loop. For every element in A, we scan the entirety of B. The worst-case number of element comparisons is $|A| \times |B|$. If both are 100,000, that's $10{,}000{,}000{,}000$ (10 billion) comparisons! Time Complexity is $O(N \times M)$ — quadratic when $N \approx M$.

Even with the `break` statement (which exits the inner loop as soon as a match is found), the worst case occurs when there are no common elements or when matches only exist at the end of list B. In practice, the average case is still $O(N \times M / 2)$, which is the same complexity class.

### Improved Implementation — Hash Set Lookup

**Data Structure Concept:** Python `list` lookups (`x in my_list`) require scanning element by element from the beginning, which is $O(N)$. Python `set` uses a **Hash Table** under the hood. When you look up whether an item exists in a set, Python computes the item's hash value and jumps directly to the corresponding bucket in memory — no scanning required. This makes lookup $O(1)$ constant time on average.

**Improved Code:**

```python
# Convert list_B to a set first
set_B = set(list_B)    # Takes O(|B|) time — one-time cost
intersection = []
for a in list_A:        # Loops |A| times
    if a in set_B:      # O(1) average lookup instead of O(|B|)!
        intersection.append(a)
```

**Analysis:** We build a set from list B in $O(|B|)$ time, then perform a single loop over list A where each lookup is $O(1)$. Total complexity is $O(|A| + |B|)$ — linear. Down from 10 billion operations to roughly 200,000 operations. The speedup here will be extreme, typically 10,000x–50,000x for lists of size 100,000.

> [!warning] Set Caveats
> - Sets do not preserve order. If you need the intersection in the order elements appear in list A, the loop approach above naturally preserves it.
> - Sets eliminate duplicates. If list A contains the same value multiple times and it exists in B, the loop above will include it multiple times. If you want unique elements only, use `set(list_A) & set(list_B)`.
> - Sets require elements to be **hashable** (immutable types like `int`, `str`, `tuple`). Lists of lists would need a different approach.

### Even Shorter Python Solution

```python
intersection = list(set(list_A) & set(list_B))  # Set intersection operator
```

This is the most Pythonic one-liner, but understanding *why* it's fast (hash tables) is what matters for HPC.

## Summary: Optimization Principles

| Exercise | Naive Complexity | Improved Complexity | Key Insight |
|:---:|:---:|:---:|:---|
| Sum 1 to N | $O(N)$ | $O(1)$ | Mathematical closed-form formula |
| Prime Detection | $O(N^2)$ | $O(N \log \log N)$ | Math bound + algorithmic restructuring (Sieve) |
| List Intersection | $O(N \times M)$ | $O(N + M)$ | Hash table data structure (set) |

> [!abstract] The Three Weapons of Optimization
> 1. **Mathematics:** Replace iteration with formulas (Gauss's sum, square root bound).
> 2. **Algorithmic Restructuring:** Change the fundamental approach (trial division → Sieve).
> 3. **Data Structures:** Choose the right container for the operation (list → set for lookups).
