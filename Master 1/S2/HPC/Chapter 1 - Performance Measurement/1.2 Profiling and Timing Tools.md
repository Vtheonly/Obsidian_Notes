# 1.2 Profiling and Timing Tools

Before optimizing, we must find out exactly *where* the program is spending its time. This is called **Profiling** — the process of measuring where execution time is allocated across different functions and code paths. Without profiling, optimization is guesswork; you might spend hours speeding up a function that only accounts for 1% of total runtime while ignoring the true bottleneck.

> [!quote] Donald Knuth
> "Premature optimization is the root of all evil." Profile first, optimize second.

---

## Basic Timing

In Python and Linux, we have basic tools to time the entire execution of a script. These are useful for quick checks but insufficient for locating bottlenecks in larger programs.

### Python `time` Module

We use `time.time()`, which returns the current wall-clock time in seconds since the Unix Epoch (January 1, 1970, 00:00:00 UTC). We record it before and after the code block, and subtract to get the elapsed duration.

```python
import time

t_start = time.time()
# ... your code here ...
t_end = time.time()
print(f"Elapsed: {t_end - t_start:.4f} seconds")
```

> [!tip] For more precise timing in Python, use `time.perf_counter()` instead of `time.time()`. It uses the highest-resolution timer available on the OS and is not affected by system clock adjustments (e.g., NTP sync).

### Linux `time` Command

You run `time python3 script.py` in the terminal. It outputs three values:

| Field | Meaning | What It Measures |
|:---:|:---|:---|
| `real` | Wall-clock time | Total elapsed real-world time (what we care about) |
| `user` | CPU time (user-mode) | Time CPU spent executing your Python/C code |
| `sys` | CPU time (kernel-mode) | Time OS kernel spent on your behalf (memory allocation, I/O) |

**How to interpret:** If `real` is much larger than `user + sys`, your program is I/O-bound (waiting on disk, network, or sleep). If `user + sys` ≈ `real`, your program is compute-bound (CPU is the bottleneck).

## Advanced Profiling (Lab 3, Part A)

When scripts get large, simple start-to-finish timing isn't enough. We need to know which *specific function* is the **bottleneck** — the slowest part that is slowing everything else down. Python provides two built-in profilers for this purpose.

### The `profile` Module (Pure Python)

This profiler is written entirely in Python. It works by intercepting every function call and return, recording timestamps.

- **Pros:** Easy to use, generates detailed call statistics (number of calls, total time, cumulative time per function).
- **Cons:** Introduces **massive overhead** — it can slow down your program 5x–10x just by watching it. More critically, it **cannot accurately time C-extension functions** (like NumPy, math, or `time.sleep`). Because it operates at the Python interpreter level, C-extensions execute so fast underneath it that they appear "nearly free" or take 0.000 seconds in the report. This creates a dangerously misleading profile.

### The `cProfile` Module (C Implementation)

This profiler is written in C and integrated directly into the Python interpreter.

- **Pros:** Very low overhead (typically 2–5% slowdown). Accurately profiles both Python code and C-extensions. Suitable for real, long-running code in production environments. **Always use this over `profile`.**
- **Cons:** Output format is slightly less detailed than `profile` for recursive call chains, but for HPC purposes this never matters.

### Running a Profile

```bash
# Using profile (NOT recommended for real work)
python -m profile -s cumtime simulation.py

# Using cProfile (ALWAYS use this)
python -m cProfile -s cumtime simulation.py
```

The `-s cumtime` flag sorts the output by **cumulative time** (total time spent in each function and all functions it calls). This immediately surfaces the bottleneck.

## Profiling Exercise Breakdown (Lab 3 Exercise 1)

### The Scenario

You have a script named `simulation.py` with two functions doing mathematical operations:

1.  **`fast_task`:** Uses `numpy.sum(data)` to sum an array.
2.  **`slow_task`:** Uses a pure Python `for` loop to calculate the sum of squares.

### The Observation

If you run `python -m profile -s cumtime simulation.py`, it will **incorrectly** report `fast_task` as taking almost no time (e.g., 0.001 seconds), because `numpy` is a C-extension that executes below the Python interpreter's observation layer. Meanwhile, `slow_task` will appear as the dominant cost.

If you run `python -m cProfile -s cumtime simulation.py`, it gives an **accurate** reading, clearly showing that `slow_task` is a severe bottleneck consuming the vast majority of execution time. However, `fast_task`'s time is now also visible and fairly reported.

### The "Why" — Interpreter Overhead vs. Vectorization

Why is `slow_task` so slow despite doing the same mathematical operation? The answer lies in the fundamental architecture of Python:

Python is an **interpreted, dynamically typed language**. In a pure Python loop (`for x in data:`), the interpreter must perform the following *every single iteration*:
1. **Type check:** Determine the type of `x` (is it an `int`? a `float`? something else?).
2. **Method dispatch:** Look up the `__add__` method for that type.
3. **Object allocation:** Create a new Python object for the result.
4. **Reference counting:** Increment/decrement reference counts for garbage collection.
5. **Bounds checking:** Verify array access is within limits.

That is an enormous amount of overhead *per number* — easily 50–100x slower than raw machine computation.

**NumPy** (`fast_task`), on the other hand, relies on **Vectorization**. The array is stored as a contiguous block of raw C numbers in memory (no Python objects). The sum operation is performed in a heavily optimized, compiled C loop with no type-checking overhead, no object allocation, and excellent CPU cache utilization. The CPU can even use **SIMD (Single Instruction, Multiple Data)** instructions to process multiple numbers simultaneously.

### The Fix — Rewrite Using Vectorization

```python
import numpy as np

def slow_task_fixed(data):
    # Vectorized NumPy operation: squares the whole array at once, then sums.
    # One C-level operation replaces millions of Python interpreter steps.
    return np.sum(data * data)
    # Equivalently: return np.sum(data ** 2)
    # Even better:    return np.dot(data, data)  # Uses optimized BLAS
```

> [!note] Key Takeaway
> Whenever you have a Python loop performing numerical operations on arrays, ask yourself: "Can NumPy do this in one vectorized call?" If yes, the speedup will typically be 10x–100x.

## Profiling in C with `gprof`

For the C/OpenMP portion of the course (Chapter 3), the equivalent profiling tool is `gprof`. The workflow differs from Python:

1. **Compile with profiling flag:** `gcc -pg -fopenmp program.c -o program`
2. **Run the program:** `./program` (this silently creates a `gmon.out` file)
3. **Analyze:** `gprof program gmon.out > analysis.txt`

The output shows flat profile (time per function) and a call graph. This is essential for identifying which loop or function in your C code deserves parallelization effort.
