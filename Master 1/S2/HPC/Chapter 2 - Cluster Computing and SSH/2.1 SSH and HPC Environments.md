# 2.1 SSH and HPC Environments

## What is SSH?

**SSH (Secure Shell)** is the backbone of remote computing. Supercomputers and HPC clusters do not have monitors, keyboards, or mice attached to them — they live in temperature-controlled data centers, often in racks dozens of units tall. You must access them over the network, and that access must be secure. SSH provides an **encrypted channel** over an unsecured network, ensuring that your commands, passwords, and data cannot be intercepted by attackers on the same network.

SSH relies on a **Client-Server architecture**:

- **SSH Server (Daemon — `sshd`):** Runs on the host machine (the supercomputer/cluster), continuously listening for incoming connections on a specific port (usually **Port 22**). When a client connects, the daemon handles authentication and spawns a remote shell session.
- **SSH Client:** Your local laptop terminal (e.g., the `ssh` command-line tool, PuTTY on Windows, or Terminal on macOS/Linux).

> [!info] Why Not Telnet or FTP?
> Older protocols like Telnet and FTP transmit everything — including your password — in plain text. Anyone on the same network can read your credentials with a simple packet sniffer. SSH encrypts all traffic end-to-end using strong cryptographic algorithms (AES, ChaCha20, etc.), making it the standard for any serious computing environment.

## SSH Architecture Deep Dive

### The Handshake Process

When you type `ssh user@host`, the following happens in seconds:

1. **TCP Connection:** Your client connects to the server on port 22.
2. **Protocol Version Exchange:** Client and server agree on the SSH protocol version (SSH-2 is the modern standard; SSH-1 is deprecated due to security flaws).
3. **Algorithm Negotiation:** Both sides agree on which encryption, MAC (Message Authentication Code), and key exchange algorithms to use.
4. **Key Exchange:** Using Diffie-Hellman (or Curve25519), both sides derive a **shared secret** without ever transmitting it. This forms the basis for the session encryption key.
5. **Server Authentication:** The server proves its identity using its **host key**. On first connection, SSH asks: "The authenticity of host X can't be established. RSA key fingerprint is ... Are you sure you want to continue connecting?" This is your chance to verify you're connecting to the right machine (not a man-in-the-middle impostor).
6. **User Authentication:** You prove your identity — typically via password or [[2.2 Remote Execution and File Transfer|SSH key]].
7. **Session:** An encrypted tunnel is now active. All subsequent data (your keystrokes, command output, file transfers) flows through this tunnel.

## Setting Up the Environment (Lab 1, Part II — Step 0)

To use SSH, the daemon must be installed, running, and allowed through the firewall. This section walks through the complete setup on a Linux machine (Ubuntu/Debian).

### Installation

```bash
sudo apt update
sudo apt install openssh-server openssh-client
```

- `openssh-server`: Installs the `sshd` daemon so your machine can *accept* SSH connections.
- `openssh-client`: Installs the `ssh` command so you can *initiate* connections to other machines.

### Service Management

```bash
# Start the SSH daemon right now
sudo systemctl start ssh

# Tell systemd to start SSH automatically on every future reboot
sudo systemctl enable ssh

# Check if the daemon is currently running
sudo systemctl status ssh
```

> [!tip] If `status` shows "inactive (dead)", the daemon isn't running. Use `start` and `enable` as shown above. If it shows "active (running)", you're good to go.

### Firewall Configuration

By default, Ubuntu's Uncomplicated Firewall (`ufw`) blocks incoming connections. You must explicitly allow SSH traffic:

```bash
sudo ufw allow ssh       # Opens port 22
sudo ufw status          # Verify the rule was added
```

If you are on a campus or corporate network, ensure that the network firewall (not just the local `ufw`) also allows traffic on port 22. Network administrators sometimes block SSH for security reasons.

### Finding Your Machine's IP Address

```bash
ip a                     # Shows all network interfaces and their IPs
hostname -I              # Quick one-liner showing only the IP addresses
```

Your IP will typically look like `192.168.x.x` (private LAN) or `10.x.x.x` (campus network). The loopback address `127.0.0.1` always refers to your own machine — useful for local testing.

## HPC Cluster Environment — What to Expect

When you SSH into a real HPC cluster (like the one at University of Boumerdes), you are typically logging into a **login node** (also called a head node or front-end node). This is not where the heavy computation happens. Key characteristics:

- **Login Node:** Shared among all users. Used for editing code, compiling, submitting jobs, and lightweight tasks. **Never** run long or CPU-intensive computations here — it affects everyone.
- **Compute Nodes:** The actual heavy-lifting machines. You don't SSH into them directly; instead, you submit jobs via a **job scheduler** (like SLURM, PBS, or SGE).
- **Shared Storage:** Your home directory is typically on a network file system (NFS) accessible from all nodes. Files you create on the login node are available on compute nodes automatically.
- **Modules System:** Scientific software is loaded via `module load` commands (e.g., `module load gcc/12.1 openmpi/4.1`), which set up your `PATH` and library paths dynamically.
