# 3.3 Loop Scheduling and Load Balancing

## The `#pragma omp parallel for` Directive

Instead of manually dividing work based on Thread IDs (which is tedious and error-prone), OpenMP provides the `parallel for` directive that handles loop distribution automatically:

```c
#pragma omp parallel for
for (int i = 0; i < N; i++) {
    // work for iteration i
}
```

This single directive combines two actions:
1. **`parallel`:** Forks a team of threads.
2. **`for`:** Distributes the loop iterations among the team members.

But the critical question is: **How exactly does it divide the $N$ iterations among threads?** The answer lies in **Scheduling** — the strategy OpenMP uses to assign work to threads. Choosing the wrong scheduling strategy can leave most of your cores idle while one thread does all the heavy lifting.

---

## Workload Balance — The Core Problem

Before understanding scheduling strategies, we must understand the nature of the workload inside the loop.

### Balanced Workload

Every loop iteration takes approximately the **same** amount of time. For example:

```c
for (int i = 0; i < N; i++) {
    C[i] = A[i] + B[i];  // Simple addition — same cost every iteration
}
```

In this case, any reasonable division of work will result in all threads finishing at roughly the same time. The workload is **embarrassingly parallel** — no synchronization needed, and scheduling barely matters.

### Unbalanced Workload

Some iterations take **much longer** than others. For example:

```c
for (int i = 2; i < N; i++) {
    if (is_prime(i)) count++;  // Checking 99991 is far slower than checking 7
}
```

Checking if 7 is prime requires testing divisors up to $\sqrt{7} \approx 2.6$ — essentially 1 division. Checking if 99,991 is prime requires testing divisors up to $\sqrt{99991} \approx 316$ — up to 157 divisions. The cost varies by two orders of magnitude across iterations. If Thread 3 gets assigned the "heavy" iterations while Thread 0 gets the "light" ones, Thread 0 finishes instantly and sits completely idle while Thread 3 struggles — wasting 75% of your computing power.

This problem is called **Load Imbalance**, and it is the primary performance killer in parallel programs.

## Scheduling Strategies (Lab 3, Exercise 6)

OpenMP allows you to control loop division using the `schedule(type, chunk_size)` clause:

```c
#pragma omp parallel for schedule(type, chunk_size)
```

### 1. `schedule(static)` — The Default

**How it works:** Divides iterations into roughly equal contiguous blocks and assigns them to threads **before the loop starts**. No decisions are made during execution.

For 4 threads and 12 iterations: Thread 0 gets {0,1,2}, Thread 1 gets {3,4,5}, Thread 2 gets {6,7,8}, Thread 3 gets {9,10,11}.

With `schedule(static, chunk_size)`, iterations are assigned in round-robin fashion in blocks of `chunk_size`. For 4 threads, 12 iterations, chunk_size=2: Thread 0 gets {0,1,6,7}, Thread 1 gets {2,3,8,9}, etc.

**Pros:**
- Zero runtime overhead (assignment is computed once at the start).
- Excellent cache locality — each thread works on contiguous memory regions.
- Deterministic — same assignment every run.

**Cons:**
- In our Prime checking code, Thread 0 gets low numbers (finishes instantly) while Thread 3 gets the highest numbers (takes forever). The others sit completely idle while Thread 3 does all the heavy lifting. This is a **severe load imbalance**.

> [!tip] When to Use `static`
> Use `static` when iterations have **roughly equal cost** (e.g., array addition, matrix multiplication). It's the fastest scheduling option when the workload is balanced.

### 2. `schedule(dynamic, chunk_size)` — Work Stealing

**How it works:** Iterations are placed in a global queue. When a thread finishes its current chunk, it returns to the queue and takes the next available chunk. Threads that finish quickly automatically get more work; threads that are slow take less. No thread sits idle as long as there's work remaining.

For 4 threads, 12 iterations, chunk_size=2:
- Initially: All threads take their first chunk (Thread 0: {0,1}, Thread 1: {2,3}, etc.)
- When Thread 0 finishes its chunk first (because low numbers are cheap to check), it takes {8,9} from the queue.
- Thread 0 finishes again, takes {10,11}. Meanwhile Thread 3 is still working on its first chunk of high numbers.

**Pros:**
- Excellent load balance — fast threads automatically pick up more work.
- Ideal for unbalanced workloads like prime checking.

**Cons — The Overhead Trade-off:**
- Every time a thread needs new work, it must access the shared queue, which requires **synchronization** (a lock on the queue). This overhead grows with the number of queue requests.
- `schedule(dynamic, 1)` means threads come back to the queue for **every single number**. For 100,000 iterations, that's 100,000 lock acquisitions — high overhead.
- `schedule(dynamic, 1000)` means they take 1000 numbers at a time. Fewer queue visits = less overhead, but potentially slightly less balanced at the very end (one thread might get stuck with a chunk of expensive numbers).

> [!important] Choosing `chunk_size` for Dynamic Scheduling
> The chunk_size is a **trade-off knob**:
> - **Smaller chunks** → Better balance, more overhead.
> - **Larger chunks** → Less overhead, potentially worse balance.
> - **Rule of thumb:** Start with `chunk_size ≈ N / (4 * num_threads)` and adjust based on timing measurements.

### 3. `schedule(guided)` — The Best of Both Worlds

**How it works:** Starts dynamically with **large chunks** (reducing overhead) but the chunk sizes **shrink exponentially** as the remaining work decreases, ensuring increasingly fine-grained balance as the loop nears completion.

The chunk size at each step is approximately: `remaining_iterations / num_threads`. So for 100,000 iterations with 4 threads: first chunk ≈ 25,000, next ≈ 18,750, then ≈ 14,062, etc., shrinking until the minimum chunk size (default 1).

**Pros:**
- Low overhead at the start (large chunks = few queue visits).
- Excellent balance at the end (small chunks = fine-grained distribution).
- No need to manually tune `chunk_size`.

**Cons:**
- Slightly less cache-friendly than `static` (chunks shrink, reducing contiguity).
- Slightly more overhead than a perfectly-tuned `dynamic` with optimal chunk_size.

### 4. `schedule(auto)` — Let the Compiler Decide

The compiler/runtime chooses the scheduling strategy. Implementation-dependent — GCC's `auto` might choose differently from Intel's. Useful when you don't know the workload characteristics, but not recommended when you *do* know (explicit is better than implicit).

## Practical Benchmarking Advice

> [!tip] How to Choose the Right Schedule
> 1. **Start with `static`** — it's the default and has zero overhead.
> 2. **If you observe load imbalance** (one thread finishing much later than others), switch to `dynamic` with a moderate chunk size.
> 3. **If you're unsure or the workload varies unpredictably**, use `guided` as a safe default.
> 4. **Always measure** — the only way to know for sure is to time your actual code with your actual input data. Theoretical analysis can guide you, but empirical results are the truth.

```c
// Example: Benchmarking different schedules
#pragma omp parallel for schedule(static)
// ... time it ...

#pragma omp parallel for schedule(dynamic, 1000)
// ... time it ...

#pragma omp parallel for schedule(guided)
// ... time it ...

// Choose the fastest one for your specific workload
```
