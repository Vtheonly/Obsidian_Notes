## Exercise 3: Vector Addition

### Task

Calculate $C[i] = A[i] + B[i]$ for arrays of size $N$ (e.g., 100 million elements) in parallel.

### Implementation

```c
#include <stdio.h>
#include <omp.h>
#include <stdlib.h>

#define N 100000000

int main() {
    double *A = malloc(N * sizeof(double));
    double *B = malloc(N * sizeof(double));
    double *C = malloc(N * sizeof(double));

    // Initialize arrays (omitted for brevity)

    double t0, t_par;

    // Serial version
    t0 = omp_get_wtime();
    for (int i = 0; i < N; i++) {
        C[i] = A[i] + B[i];
    }
    double t_serial = omp_get_wtime() - t0;

    // Parallel version
    t0 = omp_get_wtime();
    #pragma omp parallel for
    for (int i = 0; i < N; i++) {
        C[i] = A[i] + B[i];
    }
    t_par = omp_get_wtime() - t0;

    printf("Serial:   %.4f s\n", t_serial);
    printf("Parallel: %.4f s\n", t_par);
    printf("Speedup:  %.2fx\n", t_serial / t_par);

    free(A); free(B); free(C);
    return 0;
}
```

### Theory: Does Speedup Scale Linearly?

If you have 4 threads, do you get exactly 4x speedup? **No.** Several factors prevent linear scaling:

1. **Amdahl's Law:** Even in this simple loop, there is a small serial fraction (array initialization, memory allocation, timing). This limits the maximum achievable speedup.
2. **Memory Bandwidth Limitation:** Vector addition is **memory-bound**, not compute-bound. Each iteration performs only 1 addition but reads 2 values and writes 1 value from/to RAM. The CPU can compute much faster than RAM can deliver data. With 4 threads all demanding data simultaneously, they may saturate the memory bus, and the CPU cores spend time waiting for data rather than computing.
3. **Thread Creation Overhead:** Forking and joining threads takes time (typically 10–100 microseconds). For a loop that takes only a few milliseconds, this overhead can dominate. For large $N$ (millions of elements), the overhead becomes negligible relative to the computation time.
4. **Cache Effects:** On the positive side, each thread works on its own contiguous portion of the arrays, which is cache-friendly. False sharing (where threads modify adjacent cache lines) is not a concern here because each `C[i]` is independent.

> [!note] Typical Results
> For $N = 100{,}000{,}000$ on a 4-core machine, you might observe a speedup of 2.5x–3.5x rather than the theoretical 4x. The gap is primarily due to memory bandwidth saturation.

## Exercise 4: Numerical Integration of Pi

### Task

Approximate $\pi$ using numerical integration of $f(x) = \frac{4}{1 + x^2}$ over $[0, 1]$, parallelizing the sum loop using `reduction`.

### The Mathematics

The integral $\int_0^1 \frac{4}{1+x^2} dx = \pi$. We approximate this using the **rectangle rule**: divide $[0,1]$ into $N$ strips of width $step = 1/N$, evaluate the function at the midpoint of each strip, and sum the areas.

### Implementation

```c
#include <stdio.h>
#include <omp.h>

#define N 100000000

int main() {
    double step = 1.0 / N;
    double sum = 0.0;

    #pragma omp parallel for reduction(+:sum)
    for (int i = 0; i < N; i++) {
        double x = (i + 0.5) * step;
        sum += 4.0 / (1.0 + x * x);
    }

    double pi = sum * step;
    printf("Pi = %.15f\n", pi);
    return 0;
}
```

### Key Design Decisions

**Why `reduction(+:sum)` and not `critical`?**

With `critical`, every thread must wait to enter the critical section for every single iteration. For $N = 100{,}000{,}000$, that's 100 million lock acquisitions. The overhead would be catastrophic, and the parallel version would likely be **slower** than the serial version.

With `reduction`, each thread accumulates into its own private `sum` variable with **zero synchronization overhead** during the loop. Only at the very end does OpenMP merge the 4 (or however many) private sums into the global `sum`. The merge cost is negligible — $O(\log P)$ additions.

> [!warning] The Variable `x` — Declared Inside the Loop
> Notice that `x` is declared *inside* the loop body. This is critical. In OpenMP, variables declared inside a parallel region are inherently **private** to that thread. Each thread gets its own `x` on its own stack. If `x` was declared **before** the `#pragma omp parallel for`, it would be **shared** by default, and threads would overwrite each other's `x` values between the read and the write of `sum`, creating a massive race condition on `x` itself. Always declare loop-local temporary variables inside the parallel region.

### Alternative (Wrong) Implementation for Understanding

```c
// WRONG — Race condition on sum
#pragma omp parallel for
for (int i = 0; i < N; i++) {
    double x = (i + 0.5) * step;
    sum += 4.0 / (1.0 + x * x);  // DATA RACE! Multiple threads writing sum
}

// WRONG — Correct result but terrible performance
#pragma omp parallel for
for (int i = 0; i < N; i++) {
    double x = (i + 0.5) * step;
    #pragma omp critical
    {
        sum += 4.0 / (1.0 + x * x);  // Serializes the loop — no parallel benefit
    }
}
```

## Exercise 5: Maximum Value in Array

### Task

Find the maximum value in a large array using `reduction(max:max_val)`.

### Implementation

```c
#include <stdio.h>
#include <omp.h>
#include <float.h>

#define N 100000000

int main() {
    double *data = malloc(N * sizeof(double));
    // Initialize data (omitted)

    double max_val = -DBL_MAX;  // Start with smallest possible value

    #pragma omp parallel for reduction(max:max_val)
    for (int i = 0; i < N; i++) {
        if (data[i] > max_val) {
            max_val = data[i];
        }
    }

    printf("Maximum: %.6f\n", max_val);
    free(data);
    return 0;
}
```

### Why `reduction(max:max_val)`?

Without the reduction clause, a race condition occurs. Consider Thread 1 finding a new maximum of 42 and Thread 2 finding 37 simultaneously. Both read the old `max_val` (say, 30), both compare, both try to write back. Thread 2 might write 37 after Thread 1 writes 42, **overwriting the true maximum with a lower value**. The result would be wrong — and wrong in a way that underestimates the true maximum, which is the most dangerous kind of error (silent, no crash, just an incorrect answer).

The `reduction(max:max_val)` clause creates a private `max_val` for each thread, initialized to the smallest possible double value (`-DBL_MAX`). Each thread independently finds the maximum in its portion. At the end, OpenMP takes the maximum of all thread-local maxima to produce the global maximum.

> [!tip] Why Initialize to `-DBL_MAX`?
> The identity value for `max` reduction is the smallest possible value in the domain. This ensures that the first real element encountered will always be larger than the initial value. If you initialized to 0.0, an array of all negative numbers would incorrectly report the maximum as 0.0.

## Exercise 7: Histogram Generation

### Task

Count frequencies of random numbers in an array. Each random number falls into a bin, and we increment `histogram[bin]++`. Fix the race condition when multiple threads try to increment the same bin simultaneously.

### Implementation

```c
#include <stdio.h>
#include <omp.h>
#include <stdlib.h>

#define N 10000000
#define NUM_BINS 256

int main() {
    int *data = malloc(N * sizeof(int));
    int histogram[NUM_BINS] = {0};

    // Generate random data in range [0, NUM_BINS-1]
    for (int i = 0; i < N; i++) {
        data[i] = rand() % NUM_BINS;
    }

    #pragma omp parallel for
    for (int i = 0; i < N; i++) {
        int v = data[i];
        #pragma omp atomic
        histogram[v]++;  // Fixed the race condition!
    }

    // Print histogram (omitted)
    free(data);
    return 0;
}
```

### Why `#pragma omp atomic` Instead of `reduction`?

This is a common exam question. You might wonder: "Why not just use `reduction(+:histogram)`?" There are several reasons:

1. **OpenMP Version Compatibility:** Older versions of OpenMP (before 4.5) do **not** support reduction on arrays — only on scalar variables (`int`, `float`, `double`, etc.). If your compiler doesn't support OpenMP 4.5+, `reduction(+:histogram)` won't compile.

2. **Syntax Complexity:** Newer OpenMP versions (4.5+) *do* support array reductions, but the syntax requires specifying the array bounds explicitly: `reduction(+:histogram[:NUM_BINS])`. This tells OpenMP to create a `NUM_BINS`-element private array for every single thread.

3. **Memory Overhead:** If `NUM_BINS` is large (e.g., 65,536 for a full 16-bit histogram) and you have 8 threads, array reduction creates 8 copies of a 65,536-element array — that's 8 × 65,536 × 4 bytes = ~2 MB of extra memory. For `atomic`, there's no extra memory at all.

4. **Contention Patterns:** With 10 million random numbers distributed across 256 bins, the probability of two threads hitting the **same bin** at the same time is relatively low. `atomic` only blocks when there's an actual collision on the same memory address. Thread 1 incrementing `histogram[42]` and Thread 2 incrementing `histogram[97]` can proceed simultaneously with `atomic` — they only wait if both try to increment `histogram[42]` at the exact same instant.

> [!note] When Array Reduction IS Better
> If every thread increments **every bin** frequently (high contention on every array element), the constant atomic overhead becomes significant. In that case, array reduction with private copies becomes faster despite the memory cost, because threads never wait for each other during the computation phase.

## Summary: Choosing the Right OpenMP Construct

| Scenario | Construct | Why |
|:---|:---|:---|
| Sum/product of loop values | `reduction(+:sum)` | Zero-contention accumulation |
| Max/min of loop values | `reduction(max:var)` | Each thread finds local max, merged at end |
| Incrementing shared array elements | `#pragma omp atomic` | Low contention, simple operation |
| Complex multi-line critical section | `#pragma omp critical` | Protects related operations as a unit |
| Balanced loop iterations | `schedule(static)` | Zero overhead, good cache locality |
| Unbalanced loop iterations | `schedule(dynamic, C)` | Self-balancing, moderate overhead |
| Unknown/variable workload | `schedule(guided)` | Adapts automatically |

> [!abstract] The Complete Workflow
> 1. Write the **serial** version first. Verify it produces correct results.
> 2. **Profile** the serial version to confirm the loop is the bottleneck.
> 3. Add `#pragma omp parallel for` to the hot loop.
> 4. Identify shared variables that are written to → add `reduction`, `atomic`, or `critical`.
> 5. Check for variable scoping issues (declare temporaries inside the parallel region).
> 6. Choose appropriate scheduling based on workload balance.
> 7. **Measure** parallel speedup. Compare to serial version on the same hardware.
> 8. If speedup is poor, investigate: memory bandwidth? load imbalance? synchronization overhead?
