# 2.1 The Output Spatial Dimension Formula

When architecting a CNN mathematically, determining exactly how large your output tensor feature map will be after navigating a specific Convolutional or Pooling layer is a mandatory requirement. If you cannot track your tensor shape at all times, building complex architectures or interfacing correctly with a final Fully Connected classification layer is entirely computationally impossible. The shape of every tensor at every point in the network must be known in advance — not guessed, not approximated, but calculated with mathematical certainty. A single pixel off in your calculations will cascade through the network and cause runtime shape mismatch errors that are notoriously difficult to debug.

To compute this flawlessly, Deep Learning universally relies on a singular Master Spatial Formula. This formula applies equally to convolutional layers and pooling layers — any operation that slides a window over the spatial dimensions of a tensor follows this same mathematical law. Master it once, and you can compute output shapes for any architecture.

---

### The Foundational Mathematics

The exact formula for calculating the resultant spatial length along one dimension (e.g., computing the new Output Width ($O_W$) derived from the original Input Width ($W_W$)) is structurally defined as:

$$O = \left\lfloor \frac{W - K + 2P}{S} \right\rfloor + 1$$

This formula is applied **independently** to both the Height and Width dimensions. If the input is not square (e.g., $W_H \neq W_W$), or if the kernel is not square, or if asymmetric padding is used, you simply run the formula separately for each spatial dimension.

---

### Decoding Every Variable in Detail

Each variable in this formula has a precise meaning and a specific domain. Understanding each one deeply is essential for applying the formula correctly in all situations.

#### $O$ — Output Size

The final resulting dimension length along the axis being computed. This value is always a positive integer (you cannot have a fractional or zero-dimensional output). If this formula produces a value of zero or a negative number, it means your configuration is invalid — the kernel is too large for the input, or the stride is too aggressive, and no valid convolution positions exist. In practice, deep learning frameworks will throw an error in this case.

#### $W$ — Input Size

The incoming dimension length (Input Width or Input Height) along the axis being computed. For the very first convolutional layer, this is simply the spatial resolution of the input image (e.g., $W = 224$ for a $224 \times 224$ image). For subsequent layers, this is the output size of the previous layer, which was itself computed using this same formula. This cascading dependency is why you must compute shapes layer by layer, in order, from input to output.

#### $K$ — Kernel Size

The spatial reach or footprint length of the sliding filter block along the axis being computed. For a $3 \times 3$ filter, $K = 3$ for both the Height and Width calculations. For a $5 \times 7$ filter (non-square), $K_H = 5$ and $K_W = 7$. Note that $K$ refers only to the spatial dimensions of the filter — the channel depth dimension is irrelevant for this formula because the depth is entirely consumed by the summation process described in [[1.3 Processing 3D Volumes, Depth Summation, and Biases]].

#### $P$ — Padding

The distinct number of symmetrical zero-pixel rings explicitly concatenated externally across **one side** of the image perimeter. The formula uses $2P$ because padding is applied symmetrically — if $P = 1$, one ring of zeros is added to the left side and one ring to the right side (for the Width dimension), for a total of $2P = 2$ extra columns. Similarly, $P = 1$ adds one ring of zeros to the top and one to the bottom for the Height dimension. This symmetric addition means the image grows by $2P$ pixels along each dimension.

#### $S$ — Stride

The fixed integer distance dictating how aggressively the sliding filter physically steps forward after every single overlapping extraction sequence. A stride of $S = 1$ means the filter moves one pixel at a time, producing the maximum possible output size. A stride of $S = 2$ means the filter skips every other pixel, cutting the output size roughly in half. The stride applies identically in both horizontal and vertical directions unless explicitly specified otherwise (asymmetric strides are rare but possible in some frameworks).

#### $\lfloor ... \rfloor$ — The Floor Operation

The mathematical directive to rigorously round down any resulting decimal to the nearest valid integer. The floor operation ensures that the output dimension is always a whole number, which is a physical requirement — you cannot have a fractional number of pixels. The floor operation also silently discards any "remainder" pixels at the boundary of the input that cannot be fully covered by the sliding window, as explained in the Fractional Edge Cases section below.

---

### Standard Calculation Walkthrough

Let's work through the most common configuration step by step:

*Suppose you process an image with Width $W = 64$. You deploy a standard $K=3$ Kernel, using a base Stride $S=1$, and absolute zero Padding $P=0$.*

$$O = \frac{64 - 3 + 2(0)}{1} + 1 = \frac{61}{1} + 1 = 62$$

The output width drops cleanly to $62$. The tensor feature map is now $62 \times 62$ (assuming square input and kernel).

**Interpretation:** With no padding and a stride of 1, the $3 \times 3$ filter can be placed at $62$ valid positions along a 64-pixel-wide image. The first position covers columns 1–3, the second covers columns 2–4, and so on, until the last position covers columns 62–64. Each position produces one output pixel, giving us 62 output pixels total.

---

### Multiple Worked Examples

To build fluency with this formula, let's work through several configurations with different parameter combinations:

#### Example 1: Standard 3×3 Convolution with No Padding
- $W = 32$, $K = 3$, $P = 0$, $S = 1$
- $O = \lfloor (32 - 3 + 0) / 1 \rfloor + 1 = 29 + 1 = 30$
- **Result:** $32 \times 32 \rightarrow 30 \times 30$

#### Example 2: 5×5 Convolution with No Padding
- $W = 32$, $K = 5$, $P = 0$, $S = 1$
- $O = \lfloor (32 - 5 + 0) / 1 \rfloor + 1 = 27 + 1 = 28$
- **Result:** $32 \times 32 \rightarrow 28 \times 28$
- **Observation:** Larger kernels cause more shrinkage. A $5 \times 5$ kernel removes 4 pixels per dimension vs. only 2 for a $3 \times 3$ kernel.

#### Example 3: 3×3 Convolution with Same Padding
- $W = 32$, $K = 3$, $P = 1$, $S = 1$
- $O = \lfloor (32 - 3 + 2) / 1 \rfloor + 1 = 31 + 1 = 32$
- **Result:** $32 \times 32 \rightarrow 32 \times 32$ (dimensions preserved!)
- **Observation:** With $P = (K-1)/2 = 1$, the output size matches the input size. This is the "Same Padding" strategy explored in detail in [[2.2 Padding - Mathematics, Edge Effects, and Valid vs Same]].

#### Example 4: 3×3 Convolution with Stride 2
- $W = 64$, $K = 3$, $P = 1$, $S = 2$
- $O = \lfloor (64 - 3 + 2) / 2 \rfloor + 1 = \lfloor 63 / 2 \rfloor + 1 = 31 + 1 = 32$
- **Result:** $64 \times 64 \rightarrow 32 \times 32$ (exactly halved!)
- **Observation:** Stride 2 with appropriate padding halves the spatial dimensions, discarding 75% of the spatial data. This is a common downsampling strategy explored in [[2.3 Strides - Skipping Pixels and Downsampling]].

#### Example 5: 7×7 Convolution in the First Layer of a Network
- $W = 224$, $K = 7$, $P = 3$, $S = 2$ (as used in ResNet's first layer)
- $O = \lfloor (224 - 7 + 6) / 2 \rfloor + 1 = \lfloor 223 / 2 \rfloor + 1 = 111 + 1 = 112$
- **Result:** $224 \times 224 \rightarrow 112 \times 112$
- **Observation:** This aggressive initial downsampling reduces the computational cost of all subsequent layers. The padding of $P = 3$ (which is $\lfloor 7/2 \rfloor$) prevents excessive shrinkage beyond what the stride already causes.

### Handling Fractional Edge Cases (The Floor Function)

What strictly happens within the computational engine if the sliding sequence geometry math does not resolve perfectly evenly? This is a situation that arises frequently when using strides greater than 1, especially with odd-dimensional inputs.

*Suppose you pass in an uneven Image Width $W=5$. You implement a Stride $S=2$, Kernel $K=2$, no Padding $P=0$:*

$$O = \left\lfloor \frac{5 - 2 + 0}{2} \right\rfloor + 1 = \left\lfloor \frac{3}{2} \right\rfloor + 1 = \lfloor 1.5 \rfloor + 1 = 1 + 1 = 2$$

The final valid Output shape successfully registers as strictly $2 \times 2$.

**What happens to the discarded decimal structurally?**

Because the required Stride of $2$ dynamically forced the sliding $2 \times 2$ kernel explicitly "off the edge" into non-existent space while trying to process the absolute 5th final pixel column, modern deep learning frameworks standardly choose to completely **ignore and drop** that final unsymmetrical jagged edge spatial column from computation altogether to preserve tensor matrix alignment laws.

Let's trace this geometrically to see exactly what is discarded:

| Filter Position | Columns Covered | Valid? |
|---|---|---|
| Position 1 | Columns 1–2 |  Fully within image |
| Position 2 | Columns 3–4 |  Fully within image |
| Position 3 | Columns 5–6 |  Column 6 does not exist! |

The filter at position 3 would need to access column 6, which is outside the image boundary. Since we have no padding ($P=0$), this position is invalid and is simply skipped. The single pixel in column 5 that was never covered by any valid filter position is permanently lost — it contributes nothing to the output feature map.

> [!warning] Discarded Pixels Are Silently Lost
> This floor behavior means that when designing architectures, you should be mindful of odd-dimension inputs combined with strided operations, as a small amount of border information may be silently discarded. This is usually negligible for large feature maps (losing 1 pixel from a 224-pixel-wide image is immaterial), but it can become significant for small feature maps deep in the network. If you have a $7 \times 7$ feature map and apply a stride of 2, the output is $3 \times 3$ (not $3.5 \times 3.5$), and the extra column/row of information is simply dropped.

### Practical Implications for Architecture Design

The output dimension formula has several important practical consequences for how you design CNN architectures:

1. **Layer-by-layer shape tracking is mandatory.** You cannot skip computing the shape at any intermediate layer, because each layer's output becomes the next layer's input. A spreadsheet or a simple script that computes shapes layer by layer is an essential design tool.

2. **The transition to Fully Connected layers requires exact shape knowledge.** Before the classification head, the 4D tensor must be flattened into a 1D vector. The length of this vector is $N \times C_{final} \times H_{final} \times W_{final}$, and you must know this exact number to define the first fully connected layer's input dimension. Even a one-pixel error will cause a runtime crash.

3. **Architectural symmetry is a design choice.** Most architectures use square kernels ($K_H = K_W$), symmetric padding ($P_H = P_W$), and equal strides ($S_H = S_W$) to keep the spatial dimensions square. This simplifies reasoning about shapes and is almost universally followed in practice.

4. **The formula works for pooling layers too.** Max Pooling and Average Pooling with a window size of $K$ and stride $S$ follow the exact same formula. For example, a $2 \times 2$ Max Pooling with stride 2 applied to a $64 \times 64$ input gives $O = \lfloor(64 - 2 + 0)/2\rfloor + 1 = 32$.

> [!tip] Quick Mental Math for Common Configurations
> For the most common configuration ($K=3$, $S=1$, $P=0$), the output size is simply $W - 2$. This is because a $3 \times 3$ filter removes one pixel from each side of the dimension. For $K=5$, $S=1$, $P=0$, the output is $W - 4$. The general rule for stride-1, no-padding convolution: the output shrinks by $K - 1$ pixels per dimension.

---

### Summary

The output spatial dimension formula $O = \lfloor(W - K + 2P)/S\rfloor + 1$ is the single most important equation in CNN architecture design. It allows you to compute the exact size of every tensor at every point in the network. The five variables — input size $W$, kernel size $K$, padding $P$, stride $S$, and the floor operation — each play a distinct role in determining the output. The floor operation handles cases where the sliding window does not fit perfectly by silently discarding boundary pixels. Mastery of this formula, and the ability to apply it quickly and correctly, is a non-negotiable prerequisite for building any CNN architecture.
