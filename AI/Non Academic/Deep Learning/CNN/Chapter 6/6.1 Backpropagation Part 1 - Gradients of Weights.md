# 6.1 Backpropagation Part 1: Gradients of the Weights (Learning)

How exactly does the mathematical Chain Rule literally update a 2D spatial filter matrix deep inside a CNN? This question sits at the heart of understanding how CNNs actually learn, and the answer reveals a beautiful mathematical symmetry: **the backward pass of a convolution is itself a convolution operation.**

In a standard MLP, gradient routing relies strictly on basic back-connected matrix multiplication. In a CNN, because of rigid Structural Parameter Sharing, the backward pass to find weight gradients is actually calculated using **another Convolution operation itself.** This is not an approximation or a computational shortcut — it is a fundamental mathematical property of the convolution operation.

### The Core Logic

If a specific $3 \times 3$ filter dynamically produced a highly active output signal that explicitly actively confidently caused a wildly wrong final classification prediction, the network looks cleanly backward at the exact local $3 \times 3$ patch of the input pixel image that the faulty filter was scanning at that exact functional moment. It calculates an error ratio and directly adjusts those 9 specific weights directly in the absolute opposite vector direction of the explicitly calculated error.

To make the network "learn," we must update the filter values. To do this, we need to calculate the gradient of the Loss with respect to every single weight inside the filter ($\frac{\partial L}{\partial W}$). This gradient tells us: "If I nudge this particular weight up or down by a tiny amount, how much does the total error change?"

### Why Weight Gradients Are Convolutions

The key insight is understanding what happens because of **parameter sharing**. In a standard MLP, each weight connects one specific input neuron to one specific output neuron, so the gradient for each weight is a simple product of the input activation and the output gradient. But in a CNN, each weight in a $3 \times 3$ filter is shared across *every spatial position* where the filter is applied. If the input image is $64 \times 64$, a single filter weight is used at $62 \times 62 = 3,844$ different positions.

Therefore, the gradient for a single filter weight is the *sum of contributions* from every position where that weight was applied:

$$ \frac{\partial L}{\partial W_{i,j}} = \sum_{\text{all positions}} X_{\text{patch}}[i,j] \times \frac{\partial L}{\partial O_{\text{position}}} $$

Where $W_{i,j}$ is the weight at position $(i,j)$ in the filter, $X_{\text{patch}}[i,j]$ is the input value at the corresponding position in each local patch, and $\frac{\partial L}{\partial O_{\text{position}}}$ is the error gradient at each output position.

This summation over all spatial positions is mathematically identical to sliding the error gradient across the input — which is precisely a convolution operation.

### The Mathematics of Part 1 (Weight Updates)

To compute the gradient for a specific filter:

1. **Take the original Input Feature Map** (from the forward pass). This is the input tensor that was fed into the convolutional layer during the forward pass. It must be cached in memory because it is needed during backpropagation.
2. **Take the Incoming Error Gradient** (passed down from the subsequent layer). This is a 2D grid of the same spatial dimensions as the output feature map, where each value represents how much the loss changes with respect to that output pixel.
3. **Perform a valid Convolution:** Slide the error gradient across the input feature map and compute the convolution. **Input $\circledast$ Error Gradient = Weight Gradient**.

The convolution operation naturally sums the contributions from all spatial positions, producing the correct gradient for each weight in the filter. This is the same valid convolution operation used in the forward pass — the same sliding window, the same element-wise multiplication, the same summation.

*(The optimizer then subtracts this Weight Gradient from the current filter to update it using the learning rate.)*

### A Concrete Numerical Example

Consider a simplified scenario with a $4 \times 4$ input and a $3 \times 3$ filter producing a $2 \times 2$ output:

**Forward Pass:**
- The $3 \times 3$ filter slides over the $4 \times 4$ input at 4 positions (top-left, top-right, bottom-left, bottom-right)
- At each position, 9 multiplications are performed and summed
- The output is a $2 \times 2$ feature map

**Backward Pass (Weight Gradient):**
- The incoming error gradient is a $2 \times 2$ map: $\begin{bmatrix} g_1 & g_2 \\ g_3 & g_4 \end{bmatrix}$
- To compute the gradient for weight $W_{0,0}$ (top-left of the filter):
  - This weight was multiplied by input pixel $(0,0)$ when producing output $(0,0)$ with gradient $g_1$
  - This weight was multiplied by input pixel $(0,1)$ when producing output $(0,1)$ with gradient $g_2$
  - This weight was multiplied by input pixel $(1,0)$ when producing output $(1,0)$ with gradient $g_3$
  - This weight was multiplied by input pixel $(1,1)$ when producing output $(1,1)$ with gradient $g_4$
  - Therefore: $\frac{\partial L}{\partial W_{0,0}} = X_{0,0} \cdot g_1 + X_{0,1} \cdot g_2 + X_{1,0} \cdot g_3 + X_{1,1} \cdot g_4$

This is exactly what a convolution of the input with the error gradient computes.

> [!info] Computational Symmetry
> You rarely code this manually as autograd engines like PyTorch handle it, but knowing that the backward pass of a convolution is literally just another convolution mathematically proves why CNNs are so computationally symmetrical. The same hardware that accelerates forward-pass convolutions automatically accelerates backward-pass gradient computation. This symmetry is one of the reasons why CNNs are so well-suited to GPU acceleration — the same specialized convolution kernels can be used for both the forward and backward passes.
