# 7. Data Validation and Integrity

## The Importance of Validation Before Training

Training a deep learning model is an expensive and time-consuming process. For TAMER, a single training run on the Swin-Base encoder and Transformer decoder can take many hours on multiple GPUs. If the training crashes halfway through because of a corrupt image file, a malformed JSON entry, or an out-of-memory error caused by an unexpectedly large image, all of that compute time is wasted. Worse, if the data contains subtle errors that don't crash training but degrade model quality — such as mislabeled samples or truncated LaTeX — the problem may not be discovered until evaluation, by which point it's too late to fix without retraining from scratch.

**Data validation** is the process of checking data quality before training begins, catching errors early when they are cheap to fix. In the TAMER pipeline, validation is implemented as a comprehensive set of checks that verify every aspect of the dataset: image files exist and are readable, LaTeX strings are well-formed and not too long, JSON entries are valid, and pixel counts are within acceptable bounds. These checks are run automatically before every training run, and they produce a detailed report of any issues found.

The philosophy is simple: **fail fast, fail loudly**. It is far better to spend a few minutes validating data than to waste hours of GPU time on a training run that is doomed to fail or produce a degraded model.

## The Validator Module

The validator module in TAMER is a collection of functions that each check a specific aspect of data quality. These functions are designed to be composable — they can be called individually for debugging or together as part of a comprehensive validation suite. The module is organized around two main functions: `validate_samples()`, which checks individual data samples, and `validate_before_training()`, which runs the full validation suite on the entire dataset.

## Image Existence Check

The most basic validation is verifying that every image file referenced in the dataset actually exists on disk. Dataset metadata (typically stored in JSONL files) contains paths to image files, and if any of these paths are incorrect — due to a moved directory, a typo, or a failed download — the DataLoader will raise a FileNotFoundError during training.

```python
def check_image_exists(image_path: str) -> bool:
    """Verify that an image file exists on disk."""
    return os.path.isfile(image_path)
```

This check is trivial but essential. In practice, missing files are one of the most common data issues, especially when datasets are assembled from multiple sources or when files are transferred between machines. The validator checks every image path in the dataset and reports any missing files, allowing the user to fix the paths or remove the corresponding entries before training.

## Image Readability Check

Even if an image file exists, it may be corrupt or in an unsupported format. The readability check attempts to open each image with PIL (Pillow) and verifies that it can be decoded:

```python
from PIL import Image

def check_image_readable(image_path: str) -> bool:
    """Verify that an image can be opened and decoded by PIL."""
    try:
        img = Image.open(image_path)
        img.verify()  # Verify file integrity
        return True
    except Exception:
        return False
```

The `Image.verify()` method checks the file's integrity without fully decoding the pixel data, making it fast. Common issues caught by this check include truncated downloads (the file exists but is incomplete), format mismatches (a PNG file with a .jpg extension), and corrupt headers. A corrupt image will cause PIL to raise an exception when the DataLoader tries to load it, crashing the training run.

It's worth noting that `verify()` should be called on a freshly opened image object — calling it on an image that has already been partially read may not detect all corruption. After verification, the image must be reopened before it can be used for actual pixel operations.

## Pixel Count Validation

Images that are too large consume excessive VRAM during preprocessing and can cause out-of-memory errors. TAMER sets a hard limit of **4 megapixels** (4,000,000 pixels total) for any input image:

```python
MAX_PIXEL_COUNT = 4_000_000  # 4 megapixels

def check_pixel_count(image_path: str) -> bool:
    """Reject images larger than 4 megapixels."""
    try:
        img = Image.open(image_path)
        width, height = img.size
        return width * height <= MAX_PIXEL_COUNT
    except Exception:
        return False
```

At the standard TAMER resolution of 256×1024, an image has 262,144 pixels — well below the 4-megapixel limit. However, some datasets contain high-resolution scans or screen captures that exceed this limit. These images must be either resized or discarded. The validator flags them so they can be handled before training.

The 4-megapixel limit is chosen to provide a safety margin above the target resolution while preventing pathological cases. An image at 2000×2000 pixels (4 megapixels) would be resized to 256×1024 during preprocessing, but the intermediate memory required to load and resize it is still modest. Images much larger than this (e.g., 4000×4000 = 16 megapixels) would require significant memory just to load, which can cause issues on systems with limited RAM.

## Token Length Validation

LaTeX sequences that are too long create two problems: (1) they require more VRAM for the decoder's attention matrices, and (2) the model's accuracy degrades significantly on very long sequences because error accumulation in autoregressive generation compounds. TAMER rejects formulas that tokenize to more than **150 tokens**:

```python
MAX_TOKEN_LENGTH = 150

def check_token_length(latex_ids: list) -> bool:
    """Reject LaTeX sequences with more than 150 tokens."""
    return len(latex_ids) <= MAX_TOKEN_LENGTH
```

The 150-token limit is aligned with the maximum sequence length of 200 used during inference, providing a comfortable margin. Most mathematical formulas that a human would write fall well within this limit — a formula requiring 150+ tokens is typically either extremely complex (beyond the model's capability) or poorly written (e.g., containing redundant markup that should have been normalized away).

## JSON Validity Check

TAMER's datasets are stored in JSONL (JSON Lines) format, where each line is a valid JSON object. Malformed JSON — missing brackets, unescaped quotes, or encoding errors — can cause the dataset loader to crash or silently skip entries:

```python
import json

def check_json_validity(jsonl_path: str) -> list:
    """Check each line in a JSONL file for valid JSON."""
    invalid_lines = []
    with open(jsonl_path, 'r', encoding='utf-8') as f:
        for line_num, line in enumerate(f, 1):
            line = line.strip()
            if not line:
                continue
            try:
                json.loads(line)
            except json.JSONDecodeError:
                invalid_lines.append(line_num)
    return invalid_lines
```

This check is particularly important for datasets that have been manually annotated or processed through multiple conversion steps. Even a single invalid JSON line can break the DataLoader, since most JSONL readers don't gracefully handle parse errors.

## The validate_before_training() Function

The `validate_before_training()` function is the main entry point for pre-training validation. It runs all checks on the entire dataset and produces a comprehensive report:

```python
def validate_before_training(dataset_config) -> dict:
    """Run all validation checks before training begins."""
    report = {
        'total_samples': 0,
        'missing_images': [],
        'unreadable_images': [],
        'oversized_images': [],
        'long_sequences': [],
        'invalid_json_lines': [],
        'passed': True
    }

    # Check JSON validity
    report['invalid_json_lines'] = check_json_validity(dataset_config.jsonl_path)

    # Check each sample
    for sample in load_samples(dataset_config):
        report['total_samples'] += 1
        image_path = sample['image_path']
        latex_ids = sample['latex_ids']

        if not check_image_exists(image_path):
            report['missing_images'].append(image_path)
        elif not check_image_readable(image_path):
            report['unreadable_images'].append(image_path)
        elif not check_pixel_count(image_path):
            report['oversized_images'].append(image_path)

        if not check_token_length(latex_ids):
            report['long_sequences'].append(sample.get('id', 'unknown'))

    # Determine overall pass/fail
    if (report['missing_images'] or report['unreadable_images'] or
        report['invalid_json_lines']):
        report['passed'] = False

    return report
```

The function returns a dictionary with detailed information about every issue found. The `passed` flag is set to `False` if any critical issues are found (missing or unreadable images, invalid JSON), which should prevent training from starting. Non-critical issues (oversized images, long sequences) are reported but don't necessarily block training, since these samples can be filtered out by the dataset class.

## The validate_samples() Function

For more targeted validation, the `validate_samples()` function checks a specific list of samples:

```python
def validate_samples(samples: list) -> list:
    """Check a list of samples for common issues."""
    issues = []
    for i, sample in enumerate(samples):
        sample_issues = []

        if not check_image_exists(sample['image_path']):
            sample_issues.append('missing_image')
        if not check_image_readable(sample['image_path']):
            sample_issues.append('unreadable_image')
        if not check_pixel_count(sample['image_path']):
            sample_issues.append('oversized_image')
        if not check_token_length(sample['latex_ids']):
            sample_issues.append('long_sequence')

        if sample_issues:
            issues.append({
                'sample_index': i,
                'sample_id': sample.get('id', 'unknown'),
                'issues': sample_issues
            })

    return issues
```

This function is useful for debugging specific batches or for validating a subset of the data (e.g., a new dataset that is being added to the training mix).

## The manifest.json File

The preprocessing pipeline generates a `manifest.json` file that records what was processed, when, and with what parameters. This file serves as a cache key: if the preprocessing inputs haven't changed (same dataset versions, same normalization parameters), the cached results can be reused without re-running the entire pipeline.

```json

    "preprocessing_version": "1.3",
    "timestamp": "2024-01-15T10:30:00Z",
```

## Cache Invalidation

The preprocessor caches its output (tokenized sequences, resized images, complexity labels) to avoid repeating expensive computations. The cache is invalidated when:

1. **The input data changes**: A different version of a dataset, or new samples added to an existing dataset.
2. **The normalization parameters change**: For example, if the `\over` → `\frac` conversion is toggled on or off.
3. **The tokenizer changes**: If new tokens are added to the vocabulary or existing tokens are modified.
4. **The preprocessing version is bumped**: A manual override to force re-processing after a bug fix.

The cache check compares the current configuration against the `manifest.json` and re-runs preprocessing only if there is a mismatch. This saves significant time during iterative development, when the preprocessing logic may be changing frequently.

## Dataset Statistics

Understanding the composition of the training data is essential for debugging and improving model performance. The validator collects statistics that provide insight into the dataset's balance and potential biases:

- **Samples per dataset**: How many samples come from each source (im2latex, CROHME, etc.)
- **Samples per complexity level**: The distribution across simple/medium/complex
- **Average sequence length**: How many tokens the typical formula requires
- **Vocabulary coverage**: What fraction of the 522-token vocabulary is actually used in the data
- **Image size distribution**: Whether the images cluster around the target resolution or have significant variation

These statistics help identify issues like class imbalance (too many simple formulas, not enough complex ones), vocabulary gaps (tokens in the vocabulary that never appear in the data), or dataset-specific biases (one dataset contributing disproportionately to a complexity level).

## Summary

Data validation is the unsung hero of reliable machine learning pipelines. The TAMER validator catches common issues — missing files, corrupt images, oversized inputs, overly long sequences, and malformed JSON — before they can waste GPU time or silently degrade model quality. The `validate_before_training()` function runs all checks and produces a detailed report, while `validate_samples()` provides targeted checking for debugging. The `manifest.json` file enables efficient caching and cache invalidation, avoiding redundant preprocessing. Together with dataset statistics that reveal balance and bias, these tools ensure that the data entering the training pipeline is clean, consistent, and well-understood. Investing a few minutes in validation before each training run can save hours of wasted compute and prevent the frustration of discovering data issues only after a failed or suboptimal training run.
