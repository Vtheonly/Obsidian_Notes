# 6. LaTeX Normalization

## Why Normalize LaTeX?

LaTeX is a remarkably flexible typesetting system that often provides multiple ways to express the same mathematical formula. While this flexibility is valuable for human authors who have personal preferences and stylistic conventions, it creates a significant problem for machine learning models. If the model sees `\frac{a}{b}` in some training examples and `a \over b` in others, it must learn that these are semantically identical — but this is an implicit learning task that wastes model capacity and introduces unnecessary ambiguity into the training signal.

**LaTeX normalization** addresses this problem by converting all equivalent representations into a single canonical form before tokenization and training. This reduces the vocabulary of LaTeX constructs that the model needs to learn, eliminates spurious variation in the training data, and ensures that evaluation metrics like exact match are meaningful (a predicted `\frac{a}{b}` should match a ground truth of `a \over b`).

Without normalization, the model would need to learn a many-to-many mapping from visual appearances to LaTeX strings, which is far harder than a many-to-one mapping to a canonical form. Normalization effectively reduces the complexity of the output space, allowing the model to focus its capacity on the hard visual recognition task rather than on learning LaTeX syntax variants.

## Whitespace Normalization

The simplest and most impactful normalization is whitespace handling. LaTeX treats multiple spaces the same as a single space, and in many contexts, spaces are entirely ignored (e.g., between a command and its argument). However, tokenizers are sensitive to whitespace — `\frac {a}{b}` and `\frac{a}{b}` produce different token sequences even though they render identically.

The normalization process:

1. **Collapse multiple spaces** into a single space: `x  +  1` → `x + 1`
2. **Strip leading and trailing spaces**: ` x + 1 ` → `x + 1`
3. **Remove spaces after commands**: `\frac {a}{b}` → `\frac{a}{b}`
4. **Normalize spaces around operators**: `x+1` and `x + 1` should be canonicalized to the same form

This step alone eliminates a large fraction of the spurious variation in real-world LaTeX datasets, which often contain inconsistent whitespace from manual annotation or automated conversion tools.

## Fraction Normalization: \frac vs \over

One of the most important normalizations is unifying fraction syntax. LaTeX supports at least three ways to write fractions:

- `\frac{numerator}{denominator}` — the standard command form
- `{numerator \over denominator}` — the TeX primitive form
- `\dfrac{numerator}{denominator}` — display-style fraction (same as `\frac` in display mode)

The canonical form in TAMER is `\frac{...}{...}`. The normalization converts all `\over` constructions to `\frac` form:

```
Input:  {a + b \over c - d}
Output: \frac{a + b}{c - d}
```

This conversion requires careful parsing because `\over` is an infix operator that divides the current group into numerator and denominator, whereas `\frac` takes two explicit arguments. The normalizer must correctly identify the scope of the numerator and denominator, which may involve nested braces and subscripts.

## Bracket Normalization

LaTeX provides multiple ways to size delimiters:

- `(` and `)` — standard parentheses
- `\left(` and `\right)` — auto-sized parentheses
- `\big(`, `\Big(`, `\bigg(`, `\Bigg(` — manually sized parentheses

While these produce different visual sizes, they are semantically equivalent in most mathematical contexts. The normalizer converts all sized delimiters to the standard form:

```
\left( \frac{a}{b} \right)  →  ( \frac{a}{b} )
\big[ x + 1 \big]           →  [ x + 1 ]
```

This normalization is important because the visual appearance of `\left(` vs `(` depends on the context (the height of the enclosed content), but the semantic content is the same. Forcing the model to predict which sizing variant was used would add unnecessary complexity.

## Command Normalization

Several LaTeX commands have near-equivalent alternatives:

- **`\mathrm` vs `\text`**: Both produce upright text in math mode, but with different semantics. `\mathrm` is for multi-letter variables (like `\mathrm{sin}`), while `\text` is for descriptive text. In practice, many datasets use them interchangeably.
- **`\displaystyle` vs `\textstyle`**: These control the rendering style of math (display vs inline). Since the visual model doesn't know the rendering context, these are often normalized away.
- **`\dfrac` vs `\frac`**: As mentioned, `\dfrac` is equivalent to display-style `\frac`.
- **`\tfrac` vs `\frac`**: Similarly, `\tfrac` is equivalent to text-style `\frac`.

The normalizer replaces these variants with their canonical equivalents, reducing the number of unique tokens the model must learn.

## The normalize_latex() Function

The central normalization function applies all the above transformations in sequence:

```python
def normalize_latex(latex: str) -> str:
    """Canonicalize a LaTeX string to a standard form."""
    # Step 1: Whitespace normalization
    latex = re.sub(r'\s+', ' ', latex).strip()

    # Step 2: Convert \over to \frac
    latex = over_to_frac(latex)

    # Step 3: Remove \left and \right
    latex = latex.replace('\\left(', '(')
    latex = latex.replace('\\right)', ')')
    latex = latex.replace('\\left[', '[')
    latex = latex.replace('\\right]', ']')
    latex = latex.replace('\\left\\{', '\\{')
    latex = latex.replace('\\right\\}', '\\}')

    # Step 4: Command normalization
    latex = latex.replace('\\mathrm', '\\text')
    latex = latex.replace('\\dfrac', '\\frac')
    latex = latex.replace('\\tfrac', '\\frac')

    # Step 5: Remove style commands
    latex = latex.replace('\\displaystyle', '')
    latex = latex.replace('\\textstyle', '')

    # Step 6: Final whitespace cleanup
    latex = re.sub(r'\s+', ' ', latex).strip()

    return latex
```

Each transformation is applied in a specific order to avoid interference. For example, whitespace normalization must come before bracket normalization, because the presence of extra spaces could prevent pattern matching.

## The should_discard() Function

Not all LaTeX formulas are suitable for training. Some use commands that the tokenizer cannot handle, others are too long or too complex, and some are malformed. The `should_discard()` function filters out unsuitable samples:

```python
UNSUPPORTED_COMMANDS = [
    '\\tikz', '\\begin{tikzpicture}', '\\begin{array}',
    '\\begin{tabular}', '\\newcommand', '\\def',
    '\\includegraphics', '\\rule{'
]

def should_discard(latex: str) -> bool:
    """Return True if the formula should be excluded from training."""
    # Discard formulas with unsupported commands
    for cmd in UNSUPPORTED_COMMANDS:
        if cmd in latex:
            return True

    # Discard formulas that are too long
    if len(latex) > 500:
        return True

    # Discard formulas with unbalanced braces
    if latex.count('{') != latex.count('}'):
        return True

    return False
```

This function acts as a quality gate, ensuring that only well-formed, supported LaTeX strings enter the training pipeline. A single malformed formula can corrupt an entire batch if it causes a tokenization error or produces an invalid target sequence.

## The get_complexity() Function

The `get_complexity()` function scores a LaTeX formula on a three-point scale: **simple**, **medium**, or **complex**. This classification is used by the curriculum learning system (covered in Chapter 7, Note 4) to order training samples from easy to hard.

The complexity score is based on counting structural elements in the formula:

| Element | Weight | Example |
|---------|--------|---------|
| Braces `{}` | 1 each | `{a}` |
| Fractions `\frac` | 3 each | `\frac{a}{b}` |
| Superscripts `^` | 2 each | `x^2` |
| Subscripts `_` | 2 each | `x_i` |
| Square roots `\sqrt` | 3 each | `\sqrt{x}` |
| Matrices / `\begin{}` | 5 each | `\begin{pmatrix}` |
| Nested fractions | +3 extra | `\frac{\frac{a}{b}}{c}` |

```python
def get_complexity(latex: str) -> str:
    """Classify a LaTeX formula as simple, medium, or complex."""
    score = 0
    score += latex.count('{')           # Braces
    score += latex.count('\\frac') * 3  # Fractions
    score += latex.count('^') * 2       # Superscripts
    score += latex.count('_') * 2       # Subscripts
    score += latex.count('\\sqrt') * 3  # Square roots
    score += latex.count('\\begin') * 5 # Environments

    # Bonus for nested fractions
    if '\\frac' in latex:
        inner = latex.split('\\frac', 1)[1]
        if '\\frac' in inner:
            score += 3

    if score <= 8:
        return 'simple'
    elif score <= 20:
        return 'medium'
    else:
        return 'complex'
```

**Simple** formulas have few structural elements — they are typically single-level expressions like `x + 1` or `\alpha = \beta`. **Medium** formulas contain some structure, such as a single fraction or a few superscripts, like `\frac{x+1}{y-2}` or `e^{i\pi}`. **Complex** formulas involve matrices, nested fractions, aligned environments, or many structural elements, like `\begin{pmatrix} \frac{a}{b} & c \\ d & \frac{e}{f} \end{pmatrix}`.

The complexity classification directly supports curriculum learning: the model trains on simple formulas first, learning to recognize basic symbols and simple spatial relationships. Once it has mastered these, it graduates to medium and then complex formulas, building on the foundational representations it has already learned.

## Handling Unicode

Many LaTeX datasets, especially those scraped from the web, contain Unicode math symbols instead of LaTeX commands. For example, `α` instead of `\alpha`, `∑` instead of `\sum`, or `×` instead of `\times`. The normalizer converts these to their LaTeX equivalents:

```python
UNICODE_MAP = {
    'α': '\\alpha', 'β': '\\beta', 'γ': '\\gamma',
    '∑': '\\sum', '∏': '\\prod', '∫': '\\int',
    '×': '\\times', '÷': '\\div', '±': '\\pm',
    '≤': '\\leq', '≥': '\\geq', '≠': '\\neq',
    '∞': '\\infty', '√': '\\sqrt',
    # ... many more
}

def normalize_unicode(latex: str) -> str:
    for unicode_char, latex_cmd in UNICODE_MAP.items():
        latex = latex.replace(unicode_char, latex_cmd)
    return latex
```

This conversion is essential for two reasons: (1) the tokenizer's vocabulary is defined in terms of LaTeX commands, not Unicode characters, and (2) the evaluation metrics compare predicted LaTeX strings against ground truth, so any Unicode in the ground truth would cause false mismatches.

## Summary

LaTeX normalization is a critical preprocessing step that converts the rich and varied syntax of LaTeX into a single canonical form. By unifying equivalent representations — `\over` to `\frac`, `\left(` to `(`, `\mathrm` to `\text` — the normalizer reduces the ambiguity in the training signal and allows the model to focus on visual recognition rather than syntax memorization. The `should_discard()` function acts as a quality gate, removing formulas that would confuse the model or break the pipeline. The `get_complexity()` function provides the foundation for curriculum learning, enabling the model to learn from easy examples before tackling hard ones. Together, these normalization steps significantly improve training efficiency and model quality, and they are a major reason why TAMER can achieve high accuracy with a relatively modest dataset.
