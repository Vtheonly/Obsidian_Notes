# 5. Handling Variable-Length Sequences

## The Core Problem: Variable-Length LaTeX

One of the fundamental challenges in sequence-to-sequence models is that input and output sequences have different lengths, and — critically — different samples in the same batch have different output lengths. A simple formula like `x + 1` produces a short LaTeX sequence of just a few tokens, while a complex expression like `\frac{\partial^2 f}{\partial x^2} + \frac{\partial^2 f}{\partial y^2} = 0` requires dozens of tokens. This variability creates a cascade of engineering challenges that must be addressed at every stage of the pipeline: data loading, training, and inference.

Unlike images, which can be resized to a fixed resolution, sequences cannot simply be "resized." We must handle the variability explicitly through padding, masking, and careful management of sequence termination. Each of these mechanisms serves a distinct purpose, and understanding their interactions is essential for implementing a correct and efficient training pipeline.

## Padding: Making Sequences Uniform in a Batch

GPUs require fixed-shape tensors for batched operations. When we process multiple samples simultaneously (batch size B > 1), all sequences in the batch must have the same length. **Padding** solves this by appending special PAD tokens (typically with ID 0) to shorter sequences until they match the length of the longest sequence in the batch.

```python
from torch.nn.utils.rnn import pad_sequence

def collate_fn(batch):
    images = torch.stack([item['image'] for item in batch])
    latex_ids = [item['latex_ids'] for item in batch]

    # Pad sequences to the same length
    padded_ids = pad_sequence(latex_ids, batch_first=True, padding_value=0)

    return {
        'image': images,
        'latex_ids': padded_ids
    }
```

The `pad_sequence` function from PyTorch automatically determines the maximum length in the batch and pads all other sequences to that length. The `padding_value=0` argument specifies that PAD tokens have ID 0. The resulting tensor has shape `(B, max_seq_len)`, where `max_seq_len` is the length of the longest sequence in the batch.

### Why Not Pad to a Global Maximum?

A natural question is whether we should pad all sequences to a global maximum length (e.g., 200) rather than the per-batch maximum. The answer is that per-batch padding is more efficient because most batches contain sequences that are significantly shorter than the global maximum. Padding to the global maximum would waste computation on many unnecessary PAD positions, especially for batches dominated by short formulas. Per-batch padding minimizes the amount of wasted computation while still satisfying the GPU's requirement for fixed-shape tensors.

However, per-batch padding means that the sequence length varies across batches, which is perfectly fine for PyTorch — the model processes whatever shape it receives — but it means that VRAM usage fluctuates from batch to batch. This is why it's important to monitor peak VRAM usage during training, not average usage.

## The ignore_index Trick: Excluding PAD from the Loss

Padding tokens should not contribute to the training loss. A PAD token carries no information about the LaTeX content — it exists purely as a bookkeeping mechanism to satisfy the GPU's shape requirements. If PAD tokens contributed to the loss, the model would waste capacity learning to predict PAD, and the loss would be dominated by the majority of positions that are padding (especially for batches with one long sequence and many short ones).

PyTorch's `nn.CrossEntropyLoss` provides the `ignore_index` parameter for exactly this purpose:

```python
criterion = nn.CrossEntropyLoss(ignore_index=0)
```

When `ignore_index=0` is set, the loss function simply skips any position where the target token is PAD (ID 0). The gradient contribution from these positions is zero, so they do not affect the model's parameters. This is implemented internally as a mask that zeroes out the loss at ignored positions before averaging over the remaining positions.

This mechanism is clean and efficient — it requires no explicit masking code in the training loop, and it handles the gradient computation correctly without any manual intervention.

## The Autoregressive Shift and Padding

During training, the decoder learns to predict the next token given all previous tokens. This is implemented via the **autoregressive shift**: the input to the decoder is the target sequence with the last token removed (and SOS prepended), and the expected output is the target sequence shifted by one position.

When the target sequence is padded, the padding is shifted along with the content. For example, if the target is `[SOS, x, +, 1, EOS, PAD, PAD]`, the decoder input is `[SOS, x, +, 1, EOS, PAD]` and the expected output is `[x, +, 1, EOS, PAD, PAD]`. The PAD positions in the expected output are correctly ignored by `ignore_index=0`, so the model is never trained to predict PAD tokens.

This is an important subtlety: the model does learn that PAD tokens exist in the input (since it sees them in the decoder input), but it is never penalized for what it predicts at PAD positions. In practice, the model quickly learns to ignore PAD tokens in the input because they carry no useful signal.

## The Unfinished Mask in Greedy Decoding

During inference, the model generates tokens one at a time, and different sequences in the batch may finish at different steps. Once a sequence has produced an EOS token, it should not generate any further tokens. The **unfinished mask** tracks which sequences are still active:

```python
# Initialize: all sequences are unfinished
unfinished = torch.ones(B, dtype=torch.bool, device=device)

for step in range(max_len):
    # ... compute next_tokens from model output ...

    # If a sequence was already finished, force it to keep producing EOS
    next_tokens = torch.where(unfinished, next_tokens, eos_id)

    # Update unfinished: a sequence finishes when it produces EOS
    unfinished = unfinished & (next_tokens != eos_id)

    # Append tokens to output
    output = torch.cat([output, next_tokens.unsqueeze(1)], dim=1)

    # Early stopping: if all sequences are finished, break
    if not unfinished.any():
        break
```

The `torch.where` operation is crucial: for sequences that have already finished (`unfinished` is False), it replaces the model's prediction with `eos_id`. This ensures that finished sequences do not produce spurious tokens after their EOS. Without this guard, a finished sequence might generate random tokens after its EOS, which would corrupt the output.

### Early Stopping

The `if not unfinished.any(): break` check provides early stopping: if every sequence in the batch has produced an EOS token, there is no need to continue generating. This can significantly speed up inference, especially for batches of short formulas. In the worst case (all sequences have length close to `max_len`), the early stopping does not help, but it never hurts — the overhead of the check is negligible.

## Maximum Sequence Length

TAMER imposes a hard cap of **200 tokens** on the maximum generation length. This serves multiple purposes:

1. **VRAM budget**: The decoder's self-attention matrix grows as O(L²), where L is the sequence length. At L=200, the attention matrix for a single head and a single batch element is 200×200 = 40,000 entries, which is manageable. Without the cap, a degenerate case could produce thousands of tokens and exhaust VRAM.

2. **Quality control**: Formulas that require more than 200 tokens are extremely complex and likely beyond the model's capability. Rather than generating garbage, it's better to truncate and return a partial result.

3. **Inference time**: Generation is autoregressive, so the total inference time scales linearly with the output length. The cap ensures a bounded worst-case latency.

4. **Training alignment**: During training, sequences longer than 150 tokens are filtered out (see the data validation chapter), so the model is never trained on very long sequences. Asking it to generate beyond its training distribution would produce unreliable results.

## Post-Processing: Truncation and SOS Removal

After decoding, the raw output requires two post-processing steps:

1. **Truncate at EOS**: Remove all tokens after the first EOS token. This eliminates any padding or spurious tokens that may have been generated after the end of the formula.

2. **Remove SOS**: The first token is always SOS (start-of-sequence), which is a control token, not part of the LaTeX output. It should be stripped from the final result.

```python
def post_process(output_ids, sos_id=1, eos_id=2):
    results = []
    for seq in output_ids:
        # Find first EOS
        eos_pos = (seq == eos_id).nonzero(as_tuple=True)[0]
        if len(eos_pos) > 0:
            seq = seq[:eos_pos[0]]

        # Remove SOS if present
        if len(seq) > 0 and seq[0] == sos_id:
            seq = seq[1:]

        results.append(seq)
    return results
```

These steps are simple but essential. Without truncation at EOS, the output would include PAD tokens or random tokens that the model generated after finishing. Without SOS removal, the decoded LaTeX would start with an invalid token that would cause errors when rendering or evaluating.

## Summary

Handling variable-length sequences is one of the less glamorous but absolutely essential aspects of building a sequence-to-sequence model. Padding ensures that batched tensors have a uniform shape. The `ignore_index` mechanism in the loss function ensures that PAD tokens do not corrupt the training signal. During inference, the unfinished mask prevents finished sequences from generating spurious tokens, and early stopping avoids unnecessary computation. The maximum sequence length of 200 provides a safety net against VRAM overflow and degenerate generation. Finally, post-processing — truncating at EOS and removing SOS — ensures that the decoded output is clean LaTeX. Each of these mechanisms is simple in isolation, but their correct interaction is critical for a working pipeline. A single bug in any of these steps (e.g., forgetting to apply the unfinished mask, or not removing SOS from the output) can silently degrade performance or produce nonsensical results.
