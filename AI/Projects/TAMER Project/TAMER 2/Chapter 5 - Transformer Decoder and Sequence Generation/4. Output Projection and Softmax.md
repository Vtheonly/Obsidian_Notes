# 4. Output Projection and Softmax

## The Final Linear Layer: From Hidden States to Vocabulary

After the Transformer decoder has processed the encoder's visual tokens and the previously generated tokens through multiple layers of self-attention, cross-attention, and feed-forward networks, it produces a hidden state for each position in the output sequence. These hidden states live in the model's internal representation space — a 768-dimensional vector for each token position. The final step is to convert these rich but abstract representations into concrete predictions over the LaTeX vocabulary.

This conversion is performed by the **output projection layer**, also commonly called the **language model head**. It is a single linear transformation that maps from the model's hidden dimension (`d_model = 768`) to the vocabulary size (`vocab_size = 522`). The weight matrix of this layer has shape `(768, 522)`, which amounts to approximately **400,896 parameters** — a relatively small fraction of the total model parameters, but one that plays a critical role in determining the model's output.

```python
self.output_projection = nn.Linear(d_model, vocab_size, bias=False)
# Weight shape: (768, 522) ≈ 400K parameters
```

The linear projection computes:

```
logits = hidden_states @ W_output
```

where `hidden_states` has shape `(B, L, 768)` and `W_output` has shape `(768, 522)`, producing `logits` of shape `(B, L, 522)`. Each of the 522 values in the last dimension represents a raw, unnormalized score for the corresponding token in the vocabulary. These raw scores are called **logits**.

## Understanding Logits

Logits are the direct output of the linear projection before any normalization. They can range from negative infinity to positive infinity, and their absolute magnitudes are not meaningful — only their relative values matter. A higher logit for a particular token means the model considers that token more likely to be the correct next token, but the logit values themselves are not probabilities.

There are several important reasons for working with logits rather than converting immediately to probabilities:

1. **Numerical stability**: Many operations (particularly the cross-entropy loss computation) are more numerically stable when performed on logits rather than on probabilities. PyTorch's `nn.CrossEntropyLoss` internally combines the softmax and negative log-likelihood computations into a single, numerically stable operation using the log-sum-exp trick.

2. **Gradient flow**: Computing softmax and then taking the logarithm (as would be needed for the loss) involves a division by the sum of exponentials, which can lead to very small gradients when the distribution is peaked. The fused log-softmax approach avoids this problem.

3. **Flexibility**: Working with logits allows us to easily apply temperature scaling, top-k filtering, or other modifications before converting to probabilities, without having to convert back and forth between probability and logit space.

## Softmax: From Logits to Probabilities

The softmax function converts a vector of logits into a probability distribution:

$$\text{softmax}(z_i) = \frac{\exp(z_i)}{\sum_{j} \exp(z_j)}$$

The output of softmax is always a valid probability distribution: all values are between 0 and 1, and they sum to 1. The token with the highest logit receives the highest probability, but the exact probability depends on the relative magnitudes of all logits.

In the context of TAMER, softmax is applied along the vocabulary dimension (dim=-1) to the logits at each position, producing a probability distribution over the 522 possible tokens at each step of the generation process.

### Log-Softmax for Numerical Stability

In beam search and other decoding algorithms, we need log-probabilities rather than raw probabilities. Computing `log(softmax(z))` naively is numerically unstable because `softmax(z)` can produce values very close to 0, and `log(0) = -inf`. Instead, we use the **log-softmax** function, which computes the same result but in a numerically stable way:

```python
log_probs = F.log_softmax(next_logits, dim=-1)
```

PyTorch's `F.log_softmax` uses the log-sum-exp trick internally: it subtracts the maximum logit before exponentiating, which keeps all intermediate values within a safe numerical range. The result is a vector of log-probabilities that can be safely added (since adding log-probabilities corresponds to multiplying probabilities) without risk of underflow.

## Temperature in Softmax

The **temperature** parameter T controls the "sharpness" of the probability distribution produced by softmax:

$$\text{softmax}(z_i / T) = \frac{\exp(z_i / T)}{\sum_{j} \exp(z_j / T)}$$

- **T = 1** (default): The standard softmax, using the logits as-is.
- **T < 1** (e.g., 0.5): The distribution becomes **more peaked** — the highest-scoring token gets an even larger share of the probability mass. This makes the model more "confident" and deterministic.
- **T > 1** (e.g., 2.0): The distribution becomes **more uniform** — the differences between tokens are flattened, giving lower-scoring tokens a higher probability. This makes the model more "exploratory" and diverse.

In TAMER's beam search, a lower temperature can help focus the search on high-probability candidates, while a higher temperature might be useful for generating diverse alternatives. However, in practice, the beam width itself provides sufficient control over the diversity-determinism tradeoff, and temperature is typically left at 1.0.

## Why No Softmax During Training

A common point of confusion is why we don't apply softmax before computing the loss during training. The answer is that PyTorch's `nn.CrossEntropyLoss` expects **raw logits** as input, not probabilities. Internally, it computes:

```
loss = -log(softmax(logits)[target_class])
```

but does so using the log-sum-exp trick for numerical stability. If we applied softmax first and then passed the result to `nn.NLLLoss` (negative log-likelihood loss), we would lose numerical precision because the softmax output can contain values very close to zero. By passing logits directly to `CrossEntropyLoss`, we avoid this issue entirely.

```python
# Training: CrossEntropyLoss expects raw logits
criterion = nn.CrossEntropyLoss(ignore_index=0)
loss = criterion(logits.view(-1, vocab_size), targets.view(-1))
# logits: (B*L, 522), targets: (B*L,)
```

## Decoding Operations on Logits

During inference, the logits are used in different ways depending on the decoding strategy:

### Greedy Decoding

In greedy decoding, we simply take the token with the highest logit (equivalently, the highest probability) at each step:

```python
next_token = logits[:, -1, :].argmax(dim=-1)
```

The `argmax` operation finds the index of the maximum value along the vocabulary dimension, giving us the token ID of the most likely next token. No softmax is needed because argmax is invariant to monotonic transformations — the same index that maximizes the logits also maximizes the softmax probabilities.

### Beam Search

In beam search, we need actual (log-)probabilities to compare different hypotheses. The beam search algorithm maintains the top-k partial sequences ranked by their cumulative log-probability:

```python
log_probs = F.log_softmax(next_logits, dim=-1)
top_log_probs, top_indices = log_probs.topk(beam_width, dim=-1)
```

The `topk` operation selects the k highest values and their indices, providing the k most promising token extensions for each beam. The log-probabilities are then added to the cumulative scores of the beams, and the top-scoring beams are kept for the next step.

## Summary

The output projection layer is the bridge between the Transformer decoder's internal representations and the LaTeX vocabulary. It transforms 768-dimensional hidden states into 522-dimensional logit vectors, one for each position in the output sequence. During training, these logits are passed directly to `CrossEntropyLoss`, which handles the softmax computation internally for numerical stability. During inference, the logits are either argmaxed directly (greedy decoding) or converted to log-probabilities via `log_softmax` (beam search). The temperature parameter provides additional control over the sharpness of the probability distribution. Understanding this pipeline is essential for debugging generation quality — if the model produces the wrong tokens, the issue is almost always upstream in the attention or hidden states, not in the projection layer itself.
