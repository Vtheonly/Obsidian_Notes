# 4.2 LaTeX Grammar Constraints

A common failure mode of "vibe-coded" models is generating LaTeX that doesn't actually compile. For example, the model might output `\frac { a }` (missing the denominator) or `\sqrt { x + 1` (missing the closing brace).

TAMER handles this by applying **LaTeX Grammar Constraints** during the decoding process.

##  The Grammar Mask Concept
Normally, a Transformer Decoder predicts the probability of every token in its vocabulary. For example:
*   `\frac`: 80%
*   `x`: 5%
*   `}`: 15%

A "Grammar Mask" is a set of hard rules that **zero out** invalid tokens before the model picks the next one.
*   **Rule:** If the previous token was `\frac`, the *only* valid next token is `{`.
*   **Action:** Set the probability of all other tokens (like `x`, `+`, `}`) to -∞.

##  The Implementation (The Logic)
In your project, the `LaTeXGrammarConstraints` class maintains a "State Machine" of the current formula:

1.  **Brace Counter:** It keeps track of the "Brace Depth".
    *   If depth is 0, the next token *cannot* be `}`.
    *   If depth is > 0, the model *cannot* output the `<EOS>` (End of Sentence) token.
2.  **Structural Tokens:** After structural tokens like `\frac`, `\sqrt`, `^`, or `_`, the next token *must* be a `{`.
3.  **Delimiters:** Certain tokens like `.` or `,` cannot appear at the very beginning of an expression.

```python
# From your imagetomath_merged.ipynb:
class LaTeXGrammarConstraints:
    def get_valid_tokens_mask(self, generated_tokens):
        mask = torch.ones(self.vocab_size, dtype=torch.bool)
        
        # Rule: Brace depth must be balanced
        brace_depth = sum(1 if t == '{' else -1 if t == '}' else 0)
        if brace_depth <= 0: mask[rbrace] = False
        if brace_depth > 0: mask[eos] = False
        
        # Rule: Structural requirements
        if last == frac: mask[:] = False; mask[lbrace] = True
        
        return mask
```

##  Why this is important for your project
Without grammar constraints, the model relies entirely on "vibe" (statistical patterns). With them, it follows "law" (mathematical rules). This is the difference between a model that *looks* right and a model that *is* right.

###  Examples of Syntax Fixes
| Input Image | Raw Transformer (No Constraints) | TAMER (With Constraints) |
| :--- | :--- | :--- |
| $\frac{a}{b}$ | `\frac { a } b` (Invalid LaTeX) | `\frac { a } { b }` (Fixed) |
| $\sqrt{x+1}$ | `\sqrt { x + 1` (Incomplete) | `\sqrt { x + 1 }` (Fixed) |
| $x^2$ | `x ^ 2` (Often works, but messy) | `x ^ { 2 }` (Standardized) |

---
> [!IMPORTANT]
> **Key Concept: Hard vs. Soft Constraints.** These grammar rules are **Hard Constraints**. Even if the visual features strongly suggest a `}`, if the grammar says a `}` is impossible, the model *cannot* pick it.

> [!TIP]
> **Students Often Miss:** Grammar constraints don't just fix "bad" formulas; they also help the model **converge faster during training**. By eliminating impossible options, the model focuses its "learning brain" on the valid ones.
