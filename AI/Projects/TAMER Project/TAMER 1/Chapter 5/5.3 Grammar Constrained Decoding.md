# 5.3 Grammar Constrained Decoding

Neural networks don't inherently "know" mathematical grammar. They will happily generate impossible LaTeX like `\frac { x } \sqrt`. To prevent this, we apply **hard algorithmic rules** during the Beam Search decoding phase.

## 📐 The Masking Logic

We create a dynamic **Mask Tensor** of shape `[Vocab Size]`. Before the model makes a prediction, we check the previously generated tokens and "mask out" (set probability to $-\infty$) any token that would break the grammar.

### Key Rules in HMER
1.  **Mandatory Braces:** If the last token was `\frac`, `^`, or `\sqrt`, the next token *must* be `{`. All other tokens are masked.
2.  **Delimiter Matching:** If the last token was `\left`, the next token *must* be a delimiter like `(`, `[`, or `|`.
3.  **Brace Balancing:** The model is forbidden from outputting the `<eos>` (End of Sequence) token if the net count of open braces `{` vs. closed braces `}` is not exactly zero.

## 💡 Benefit: 100% Compile Rate

By forcefully applying these masks, it becomes mathematically impossible for the network to predict a syntactically invalid LaTeX structure. This ensures that every output from your model can be rendered perfectly by a LaTeX engine.

⚠️ **Pro Tip:** While grammar constraints prevent "illegal" syntax, they can't prevent "logical" errors (like writing $1=2$). The goal is **Structural Integrity**, not mathematical truth.

> [!NOTE]
> **Implementation:** In your Python code, this is often implemented as a `LogitsProcessor` or a custom loop inside the `generate()` function that modifies the probability distribution before the `Softmax` is applied.
