
# 2. System Orchestrator and Architecture

Your final Python implementation features extremely rigorous OOP (Object-Oriented Programming) isolation. This design pattern is what separates hobbyist scripts from professional, maintainable software.

## Decoupled Pipeline Interfaces
```mermaid
graph LR
    A[BaseLoader] -->|Yields FrameData| B[Pipeline Orchestrator]
    B -->|process_frame| C[BaseFusionEngine]
    C -->|get_fused_result| D[BaseReconstructor]
```

*   **Loaders (`DatasetLoader`, `SyntheticLoader`, `VideoLoader`):** They have a single absolute job: yield a `FrameData` object containing exactly one Color array, one Depth array, and one Extrinsic array. The rest of the codebase doesn't care if it came from YouTube or an Open3D dataset. This is polymorphism in action.
*   **Fusion Engines (`MultiScaleFusionEngine`, `TSDFFusionEngine`):** They sit isolated, continuously accepting `FrameData` packages and accumulating them into octrees or voxel grids. They adhere to the `BaseFusionEngine` contract.
*   **Reconstructors (`PoissonReconstructor`):** Takes the generic fused geometry output and converts it to a mesh. 

This "separation of concerns" means you can swap `strategy="multiscale"` with `strategy="tsdf"` on a whim, knowing the backend logic has standardized data boundaries that will not throw Python `AttributeErrors`. You could even write a new `NeuralFusionEngine` and plug it into the pipeline with minimal code changes, because it would simply implement the same `BaseFusionEngine` interface.
