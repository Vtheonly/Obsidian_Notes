
# 1. Video Downloading and Extraction

Taking a raw `.mp4` file and translating it into usable Spatial 3D Data requires preprocessing.

## Stage 1: Ingestion
Your script implements a dual-fetch methodology to be robust:
*   **Direct URLs:** Utilizes the standard `requests` library in streamed chunks (`resp.iter_content(chunk_size=8192)`). This is crucial because it prevents the entire video file from being loaded into RAM at once, allowing you to download gigabyte-sized files on a memory-constrained machine.
*   **YouTube/Social URLs:** Leverages the standalone command line tool `yt-dlp`. The command `bestvideo[height<=720][ext=mp4]` is a specific filter that requests a standard, widely-compatible `.mp4` file at a reasonable resolution, avoiding complex codecs like MKV or WebM that could cause issues with OpenCV.

## Stage 2: Fragmenting via Target FPS
Videos frequently run at 30 or 60 frames per second (FPS). Supplying a fusion engine with 60 images capturing a 1-second pan is a catastrophic waste of memory and compute time, as the camera's change in position is negligible.

The pipeline applies a `Config.VIDEO_FPS` setting (e.g., `2` FPS) to decimate the input stream.
```python
src_fps = cap.get(cv2.CAP_PROP_FPS) # Get original FPS (e.g., 30)
frame_interval = max(1, int(round(src_fps / target_fps))) # e.g., round(30/2) = 15
```
Using OpenCV's `VideoCapture`, the pipeline reads every frame but only processes frames where `frame_idx % frame_interval == 0`. This modular skip operation strictly retrieves frames that have enough physical baseline disparity (camera movement) to be useful for triangulation in the subsequent SfM step.
