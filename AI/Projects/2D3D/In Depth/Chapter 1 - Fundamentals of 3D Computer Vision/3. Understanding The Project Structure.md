## Why the `__init__.py` Files?
These empty files signal to the Python interpreter that a directory is a "package". A package is a collection of modules that can be imported.

Without `__init__.py`, an attempt like this would fail:
```python
# This fails if `src` is not a package
from src.core.types import FrameData 
```
Python wouldn't know to look inside the `src` folder. The `__init__.py` file turns the directory into a namespace, allowing for clean, hierarchical imports that make the entire project readable and maintainable.

## The Principle of "Separation of Concerns"
Your project structure is a textbook example of "Separation of Concerns," a core software engineering principle.

*   **`core/`**: Defines the "language" of your application. The `FrameData` and `ReconstructionResult` dataclasses are nouns that every other part of the system understands. The `interfaces.py` file defines the "contracts" (Abstract Base Classes) that all modules must follow.
*   **`interfaces/`**: Handles all data input/output (IO). Code here is responsible for *loading* data, not *processing* it.
*   **`processing/`**: Contains the core "verbs" of the pipeline—the algorithms that transform data (e.g., fusing point clouds).
*   **`reconstruction/`**: Focuses on the final step of turning intermediate data (like point clouds) into a final mesh.
*   **`utils/`**: Holds universally useful "helper" code, like the logger and configuration manager, that doesn't belong to any specific domain.
*   **`tests/`**: Contains code that verifies the other modules work as expected. This is crucial for preventing bugs when you change one part of the system.

By separating the code this way, you can change your `loader` from synthetic data to a real dataset without touching a single line of your `fusion` or `reconstruction` code. This modularity is the hallmark of professional, scalable software.
