**Detailed Explanation:**
JavaScript runs on a **single thread**. This means it can only execute one task at a time, line by line. If you ask it to download a massive file from a server, the entire website would **freeze** — no scrolling, no clicking, no animations — until the download finishes.

To solve this, JavaScript uses **asynchronous programming**. It allows the browser to say: "Start downloading this file in the background, and while we wait, keep running the rest of the website normally. Notify me when the download is done."

---

## Callbacks (The Old Way)

The original approach was to pass a function (a "callback") to be executed once the async task completes:

```javascript
setTimeout(function() {
    console.log("This runs after 2 seconds");
}, 2000);

console.log("This runs immediately");
// Output:
// "This runs immediately"
// "This runs after 2 seconds"
```

The problem: nesting callbacks inside callbacks creates messy, hard-to-read code known as **"callback hell"**.

---

## Promises (The Modern Way)

A Promise is an object that represents a future value — it is either:
*   **Pending:** Still working on it.
*   **Fulfilled (Resolved):** The operation succeeded.
*   **Rejected:** The operation failed.

```javascript
const myPromise = new Promise(function(resolve, reject) {
    const success = true;
    if (success) {
        resolve("Operation completed successfully!");
    } else {
        reject("Something went wrong.");
    }
});

myPromise
    .then(function(result) {
        console.log(result);  // "Operation completed successfully!"
    })
    .catch(function(error) {
        console.log(error);
    });
```

## Async/Await (The Cleanest Way)

`async/await` is syntactic sugar built on top of Promises. It makes asynchronous code look and read like regular, synchronous code.

```javascript
async function fetchWeather() {
    try {
        // 'await' pauses THIS FUNCTION until the data arrives
        // But the rest of the page continues to work normally
        const response = await fetch("https://api.weather.com/today");
        const data = await response.json();
        console.log(data);
    } catch (error) {
        console.log("Failed to fetch weather:", error);
    }
}

fetchWeather();
```

**Key Rules:**
1.  `await` can only be used inside a function marked with `async`.
2.  `await` pauses the function, not the entire page.
3.  Always wrap `await` calls in a `try/catch` block to handle errors gracefully.

## The Fetch API — Making HTTP Requests

`fetch()` is the modern, built-in way to make HTTP requests from JavaScript (replacing the older `XMLHttpRequest`).

**GET Request (retrieve data):**
```javascript
async function getUsers() {
    const response = await fetch("https://jsonplaceholder.typicode.com/users");
    const users = await response.json();
    console.log(users);
}
```

**POST Request (send data):**
```javascript
async function createUser() {
    const response = await fetch("https://jsonplaceholder.typicode.com/users", {
        method: "POST",
        headers: {
            "Content-Type": "application/json"
        },
        body: JSON.stringify({
            name: "Ali",
            email: "ali@example.com"
        })
    });
    const newUser = await response.json();
    console.log("Created:", newUser);
}
```

## Why This Matters

Almost every real-world web application needs async operations:
*   Fetching data from a backend API
*   Loading images and videos
*   Sending form data to a server
*   Real-time features (chat, notifications)
*   Interacting with third-party services (Google Maps, payment gateways)

Mastering `async/await` and `fetch()` is the bridge from building static pages to building **real, dynamic web applications**.
