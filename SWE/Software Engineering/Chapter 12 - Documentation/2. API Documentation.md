# 2. API Documentation

> **Tags:** #api #documentation #openapi #swagger #rest

API documentation is the contract between your API and its consumers. Good API documentation is **complete** (every endpoint documented), **accurate** (matches the actual behavior), and **discoverable** (easy to navigate). This note covers how to write and generate API docs.

---

## 12.2.1 What API Documentation Should Include

For every endpoint:

| Field | Description |
| --- | --- |
| **Method and path** | `GET /api/users/:id` |
| **Description** | What the endpoint does |
| **Parameters** | Path, query, header, body parameters with types |
| **Request body** | Schema with example |
| **Response** | Status codes, response body schema, example |
| **Errors** | Possible error responses |
| **Authentication** | What auth is required |
| **Rate limits** | If applicable |
| **Examples** | Curl, JavaScript, Python examples |

---

## 12.2.2 OpenAPI / Swagger

**OpenAPI** (formerly Swagger) is the standard specification for REST APIs. You write an OpenAPI document (YAML or JSON), and tools generate interactive documentation, client SDKs, and mock servers.

### A Minimal OpenAPI Document

```yaml
openapi: 3.1.0
info:
  title: User API
  version: 1.0.0
  description: API for managing users

paths:
  /users:
    get:
      summary: List all users
      parameters:
        - name: limit
          in: query
          schema:
            type: integer
            default: 20
        - name: offset
          in: query
          schema:
            type: integer
            default: 0
      responses:
        '200':
          description: A list of users
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/User'
    
    post:
      summary: Create a new user
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreateUserRequest'
      responses:
        '201':
          description: User created
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '400':
          description: Invalid input
  
  /users/{id}:
    get:
      summary: Get a user by ID
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: A user
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '404':
          description: User not found

components:
  schemas:
    User:
      type: object
      properties:
        id:
          type: integer
        name:
          type: string
        email:
          type: string
          format: email
      required:
        - id
        - name
        - email
    
    CreateUserRequest:
      type: object
      properties:
        name:
          type: string
        email:
          type: string
          format: email
      required:
        - name
        - email
```

### Tools for OpenAPI

| Tool | Purpose |
| --- | --- |
| **Swagger UI** | Interactive HTML documentation from OpenAPI |
| **Redoc** | Alternative HTML documentation (nicer default theme) |
| **Swagger Editor** | Online editor with live preview |
| **openapi-generator** | Generate client SDKs in 50+ languages |
| **Prism** | Mock server from OpenAPI spec |
| **Spectral** | Lint OpenAPI specs for best practices |

## 12.2.3 Generating OpenAPI from Code

Instead of writing OpenAPI by hand, many frameworks generate it from your code:

### FastAPI (Python)

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class User(BaseModel):
    id: int
    name: str
    email: str

class CreateUserRequest(BaseModel):
    name: str
    email: str

@app.get("/users", response_model=list[User])
def list_users(limit: int = 20, offset: int = 0):
    return get_users(limit=limit, offset=offset)

@app.post("/users", response_model=User, status_code=201)
def create_user(req: CreateUserRequest):
    return save_user(req)
```

FastAPI automatically generates OpenAPI docs at `/docs` (Swagger UI) and `/redoc` (Redoc).

### NestJS (TypeScript)

NestJS uses `@nestjs/swagger` decorators:

```typescript
@ApiTags('users')
@Controller('users')
export class UsersController {
  @Get()
  @ApiOperation({ summary: 'List all users' })
  @ApiResponse({ status: 200, type: [User] })
  listUsers(@Query('limit') limit = 20) {
    return this.usersService.findAll(limit);
  }
}
```

### Spring Boot (Java)

Spring Boot uses `springdoc-openapi`:

```java
@RestController
@RequestMapping("/users")
public class UserController {
    
    @GetMapping
    @Operation(summary = "List all users")
    public List<User> listUsers(@RequestParam(defaultValue = "20") int limit) {
        return userService.findAll(limit);
    }
}
```

## 12.2.4 Documentation-First vs Code-First

### Documentation-First

1. Write the OpenAPI spec first.
2. Review with the team.
3. Generate server stubs from the spec.
4. Implement the business logic.

**Pros:** the API is designed before implementation; the spec is always accurate; clients can be generated immediately.

**Cons:** spec can drift from implementation if not enforced; extra upfront work.

### Code-First

1. Write the code.
2. Generate the spec from the code.

**Pros:** less upfront work; spec matches code automatically.

**Cons:** API design may be rushed; spec reflects implementation, not intent.

For new APIs, **documentation-first** is recommended. It forces you to think about the API before implementation, leading to better designs.

## 12.2.6 Common Mistakes

- **No examples.** A schema without an example is hard to understand. Always include example requests and responses.
- **Outdated docs.** Docs that do not match the API are worse than no docs. Automate generation from code.
- **No error documentation.** Documenting only the happy path leaves consumers unprepared for errors.
- **No authentication docs.** How do consumers authenticate? What tokens do they need?
- **No versioning.** When you change the API, how do consumers know? Use version headers or URL paths.
- **Jargon without explanation.** Not every reader is an expert. Define terms.

---

## 12.2.7 Key Takeaways

- API docs should include: method/path, parameters, request/response schemas, errors, auth, examples.
- OpenAPI (Swagger) is the standard spec format for REST APIs.
- Tools: Swagger UI (interactive docs), Redoc (alternative), openapi-generator (SDKs), Prism (mock server).
- Many frameworks (FastAPI, NestJS, Spring Boot) auto-generate OpenAPI from code.
- Documentation-first design leads to better APIs.
- Always include examples, document errors, and keep docs in sync with code.
