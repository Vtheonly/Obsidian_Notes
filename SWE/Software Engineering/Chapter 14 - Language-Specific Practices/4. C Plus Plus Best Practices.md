# 4. C++ Best Practices

> **Tags:** #cpp #best-practices #tooling #conventions #memory-safety

This note covers C++-specific practices for testing, debugging, refactoring, and tooling. C++ is powerful but unforgiving; following best practices is essential for safe and maintainable code.

---

## 14.4.1 Modern C++ (C++17/20/23)

Use modern C++ features that make code safer and more concise:

### Smart Pointers (Avoid Raw `new`/`delete`)

```cpp
// BAD: raw pointers, manual memory management
User* user = new User("Alice");
// ... if exception here, memory leak
delete user;

// GOOD: smart pointers
#include <memory>

auto user = std::make_unique<User>("Alice");  // unique ownership
// Automatically deleted when user goes out of scope

auto shared = std::make_shared<User>("Bob");  // shared ownership
// Deleted when last reference goes out of scope
```

| Pointer type | When to use |
| --- | --- |
| `std::unique_ptr` | Single owner (default choice) |
| `std::shared_ptr` | Shared ownership (reference counted) |
| `std::weak_ptr` | Non-owning reference to a `shared_ptr` |
| Raw pointer | Non-owning observation (does not manage lifetime) |

### RAII (Resource Acquisition Is Initialization)

```cpp
// BAD: manual resource management
FILE* file = fopen("data.txt", "r");
// ... if exception, file not closed
fclose(file);

// GOOD: RAII wrapper
#include <fstream>
std::ifstream file("data.txt");
// Automatically closed when file goes out of scope
```

RAII ties resource lifecycle to object lifetime. When the object is destroyed (goes out of scope), the resource is released. This is the most important idiom in C++.

### `auto` and Range-Based For

```cpp
// Old: explicit types
std::vector<std::string> names = getNames();
for (std::vector<std::string>::iterator it = names.begin(); it != names.end(); ++it) {
    std::cout << *it << std::endl;
}

// Modern: auto and range-based for
auto names = getNames();
for (const auto& name : names) {
    std::cout << name << std::endl;
}
```

### `std::optional` (C++17)

```cpp
#include <optional>

std::optional<User> findUser(int id) {
    auto it = users.find(id);
    if (it != users.end()) {
        return it->second;
    }
    return std::nullopt;
}

auto user = findUser(42);
if (user) {
    std::cout << user->name << std::endl;
} else {
    std::cout << "Not found" << std::endl;
}
```

### Structured Bindings (C++17)

```cpp
// Destructure pairs and tuples
std::map<int, std::string> m = {{1, "Alice"}, {2, "Bob"}};
for (const auto& [id, name] : m) {
    std::cout << id << ": " << name << std::endl;
}
```

### `constexpr` and `consteval`

```cpp
// Compile-time computation
constexpr int factorial(int n) {
    return (n <= 1) ? 1 : n * factorial(n - 1);
}

int main() {
    constexpr int f5 = factorial(5);  // computed at compile time
    static_assert(f5 == 120);
}
```

## 14.4.2 Memory Safety

C++ gives you direct memory access, which is powerful and dangerous. Follow these rules:

1. **Never use raw `new`/`delete`.** Use `make_unique` and `make_shared`.
2. **Never use C-style arrays.** Use `std::vector` or `std::array`.
3. **Never use C-style strings.** Use `std::string` or `std::string_view`.
4. **Never return references to local variables.** They dangle.
5. **Use `const` aggressively.** Mark variables, parameters, and methods `const` when possible.
6. **Use `[[nodiscard]]`** to prevent ignoring return values.
7. **Enable compiler warnings.** `-Wall -Wextra -Werror`.
8. **Use AddressSanitizer and UndefinedBehaviorSanitizer** in debug builds.

```cpp
// BAD: dangling reference
std::string& getName() {
    std::string name = "Alice";
    return name;  // dangling: name is destroyed
}

// GOOD: return by value (move semantics)
std::string getName() {
    return "Alice";  // RVO or move
}
```

## 14.4.3 Testing with GoogleTest

```cpp
#include <gtest/gtest.h>
#include "Calculator.h"

class CalculatorTest : public ::testing::Test {
protected:
    Calculator calc;
    
    void SetUp() override {
        calc = Calculator();
    }
};

TEST_F(CalculatorTest, AddReturnsSumOfTwoNumbers) {
    EXPECT_EQ(calc.add(2, 3), 5);
}

TEST_F(CalculatorTest, AddHandlesNegativeNumbers) {
    EXPECT_EQ(calc.add(-1, -1), -2);
    EXPECT_EQ(calc.add(-1, 1), 0);
}

TEST_P(CalculatorTest, AddIsCommutative) {
    int a = GetParam().first;
    int b = GetParam().second;
    EXPECT_EQ(calc.add(a, b), calc.add(b, a));
}

INSTANTIATE_TEST_SUITE_P(
    CommutativeTests,
    CalculatorTest,
    ::testing::Values(
        std::make_pair(1, 2),
        std::make_pair(-1, 1),
        std::make_pair(0, 0)
    )
);
```

## 14.4.4 Build with CMake

```cmake
cmake_minimum_required(VERSION 3.20)
project(MyApp CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# Enable warnings
add_compile_options(-Wall -Wextra -Wpedantic)

# Debug flags with sanitizers
set(CMAKE_CXX_FLAGS_DEBUG "-g -fsanitize=address,undefined -fno-omit-frame-pointer")

# Executable
add_executable(myapp src/main.cpp src/calculator.cpp)
target_include_directories(myapp PRIVATE include)

# Tests
enable_testing()
add_executable(calculator_test tests/test_calculator.cpp src/calculator.cpp)
target_link_libraries(calculator_test PRIVATE GTest::gtest_main)
include(GoogleTest)
gtest_discover_tests(calculator_test)
```

## 14.4.6 Common C++ Mistakes

### Off-by-One in Loops

```cpp
// BAD: <= instead of <
for (int i = 0; i <= vec.size(); ++i) {  // out of bounds on last iteration
    std::cout << vec[i] << std::endl;
}

// GOOD: <
for (int i = 0; i < vec.size(); ++i) {
    std::cout << vec[i] << std::endl;
}

// BEST: range-based for
for (const auto& elem : vec) {
    std::cout << elem << std::endl;
}
```

### Slicing

```cpp
// BAD: derived object sliced to base
class Base { public: virtual void speak() { cout << "Base"; } };
class Derived : public Base { public: void speak() override { cout << "Derived"; } };

void print(Base b) {  // takes Base by value
    b.speak();  // always "Base" — Derived part is sliced off
}

Derived d;
print(d);  // prints "Base", not "Derived"

// GOOD: pass by reference
void print(const Base& b) {
    b.speak();  // correctly calls Derived::speak
}
```

### Not Using `override`

```cpp
// BAD: typo in method name, no error
class Derived : public Base {
public:
    void spaek() { ... }  // meant to override speak(), but it is a new method
};

// GOOD: override keyword catches the mistake
class Derived : public Base {
public:
    void speak() override { ... }  // compiler error if it does not actually override
};
```

## 14.4.8 Key Takeaways

- Use modern C++: smart pointers, RAII, `auto`, range-based for, `std::optional`, structured bindings.
- Memory safety: never raw `new`/`delete`, never C-style arrays/strings, use `const` aggressively.
- Test with GoogleTest; use `TEST_F` for fixtures and `TEST_P` for parameterized tests.
- Build with CMake; enable sanitizers in debug builds.
- Debug with GDB/LLDB, Valgrind, and sanitizers.
- Avoid common mistakes: off-by-one, slicing, missing `override`.
- Tools: CMake + Ninja (build), GCC/Clang (compile), GoogleTest (tests), ASan/UBSan (runtime checks), clang-format/clang-tidy (lint).
