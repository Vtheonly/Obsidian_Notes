# 2. Make and CMake

> **Tags:** #make #cmake #c #cpp #build-tools

Make and CMake are the standard build tools for C and C++ projects. Make is a direct build tool; CMake is a build system generator that produces Makefiles (or other build files).

---

## 11.2.1 Make

**Make** is the classic Unix build tool. It reads a `Makefile` and executes rules to build targets.

### A Simple Makefile

```makefile
# Variables
CC = gcc
CFLAGS = -Wall -g
TARGET = myapp
SOURCES = main.c utils.c io.c
OBJECTS = $(SOURCES:.c=.o)

# Default target
all: $(TARGET)

# Link the executable
$(TARGET): $(OBJECTS)
	$(CC) $(CFLAGS) -o $@ $^

# Compile each .c to .o
%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@

# Clean build artifacts
clean:
	rm -f $(OBJECTS) $(TARGET)

# Run tests
test: $(TARGET)
	./$(TARGET) --test

# Phony targets (not files)
.PHONY: all clean test
```

### Makefile Concepts

| Concept | Meaning |
| --- | --- |
| **Target** | A file to build (e.g., `myapp`) or a phony action (e.g., `clean`). |
| **Dependency** | Files the target depends on. If a dependency is newer than the target, the target is rebuilt. |
| **Recipe** | The shell commands to build the target. Must start with a tab. |
| **Variable** | `CC = gcc` — use with `$(CC)`. |
| **Automatic variables** | `$@` (target name), `$<` (first dependency), `$^` (all dependencies). |
| **Pattern rules** | `%.o: %.c` — how to build any `.o` from a `.c`. |
| **Phony targets** | `.PHONY: clean` — marks targets that are not files. |

### Running Make

```bash
make            # build the default (first) target
make myapp      # build a specific target
make clean      # run the clean target
make -j4        # build with 4 parallel jobs (faster)
make -n         # dry run: print commands without executing
make V=1        # verbose: print all commands
```

### Make's Strengths and Weaknesses

**Strengths:**
- Simple syntax for small projects.
- Incremental builds (only rebuilds what changed).
- Ubiquitous — available on every Unix system.

**Weaknesses:**
- Does not scale well to large projects.
- No built-in dependency management (you manage include paths and libraries manually).
- Cross-platform builds are painful.
- Tab-sensitive syntax is error-prone.

## 11.2.2 CMake

**CMake** is a build system generator. You write `CMakeLists.txt`, and CMake generates Makefiles, Ninja files, Visual Studio projects, or Xcode projects.

### A Simple CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.16)
project(MyApp C CXX)

# C++ standard
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# Executable
add_executable(myapp main.cpp utils.cpp io.cpp)

# Include directories
target_include_directories(myapp PRIVATE include/)

# Link libraries
target_link_libraries(myapp PRIVATE mylib)

# Subdirectory (for libraries)
add_subdirectory(lib)

# Install rules
install(TARGETS myapp DESTINATION bin)
install(FILES include/myapp.h DESTINATION include)

# Enable testing
enable_testing()
add_test(NAME myapp_test COMMAND myapp --test)
```

### CMake Concepts

| Concept | Meaning |
| --- | --- |
| `project()` | Declare the project name and languages. |
| `add_executable()` | Define an executable target. |
| `add_library()` | Define a library target (static or shared). |
| `target_include_directories()` | Add include paths for a target. |
| `target_link_libraries()` | Link libraries to a target. |
| `set()` | Define a variable. |
| `if() / elseif() / else()` | Conditional logic. |
| `find_package()` | Find and use an external library (e.g., Boost, OpenCV). |
| `add_subdirectory()` | Include another CMake project (e.g., a sub-library). |
| `install()` | Define installation rules. |

### Building with CMake

```bash
# Generate build files (out-of-source build)
mkdir build && cd build
cmake ..

# Build
cmake --build . --parallel 4

# Run tests
ctest

# Install
cmake --install .
```

**Out-of-source builds** keep build artifacts out of the source tree. Always build in a separate `build/` directory.

### CMake Build System Generators

CMake can generate different build systems:

```bash
cmake -G "Unix Makefiles" ..       # Makefiles (default on Linux/macOS)
cmake -G "Ninja" ..                # Ninja (faster than Make)
cmake -G "Visual Studio 17 2022" .. # Visual Studio project
cmake -G "Xcode" ..                # Xcode project
```

Ninja is recommended for faster builds: `cmake -G Ninja ..`.

## 11.2.3 Dependency Management with CMake

### find_package

`find_package` locates installed libraries:

```cmake
find_package(Boost 1.70 REQUIRED COMPONENTS filesystem system)

add_executable(myapp main.cpp)
target_link_libraries(myapp PRIVATE Boost::filesystem Boost::system)
```

### FetchContent (CMake 3.11+)

`FetchContent` downloads dependencies at configure time:

```cmake
include(FetchContent)

FetchContent_Declare(
  fmt
  GIT_REPOSITORY https://github.com/fmtlib/fmt.git
  GIT_TAG        10.1.1
)
FetchContent_MakeAvailable(fmt)

target_link_libraries(myapp PRIVATE fmt::fmt)
```

### vcpkg and Conan

For larger dependency management, use a package manager:

- **vcpkg**: Microsoft's C++ package manager. Integrates with CMake via a toolchain file.
- **Conan**: Decentralized C++ package manager. Integrates via `conan_cmake_run()`.

```bash
# vcpkg
vcpkg install boost
cmake -DCMAKE_TOOLCHAIN_FILE=/path/to/vcpkg/scripts/buildsystems/vcpkg.cmake ..

# Conan
conan install .
cmake ..
```

## 11.2.4 Make vs CMake

| Aspect | Make | CMake |
| --- | --- | --- |
| What it is | Build tool | Build system generator |
| Input | `Makefile` | `CMakeLists.txt` |
| Output | Builds directly | Generates Makefiles/Ninja/VS/Xcode |
| Cross-platform | Poor | Excellent |
| Dependency management | Manual | `find_package`, `FetchContent`, vcpkg, Conan |
| Best for | Small projects, quick tasks | Medium to large C/C++ projects |

For any non-trivial C/C++ project, use CMake. It handles cross-platform builds, dependency management, and scales to large codebases.

---

## 11.2.5 Key Takeaways

- **Make** reads `Makefile` rules and builds targets incrementally.
- **CMake** generates build files (Makefiles, Ninja, VS, Xcode) from `CMakeLists.txt`.
- CMake concepts: `project()`, `add_executable()`, `target_link_libraries()`, `find_package()`.
- Always do out-of-source builds: `mkdir build && cd build && cmake ..`.
- Use Ninja for faster builds: `cmake -G Ninja ..`.
- For dependencies: `find_package` (system-installed), `FetchContent` (download at configure time), vcpkg/Conan (package managers).
- Use CMake for any non-trivial C/C++ project.
