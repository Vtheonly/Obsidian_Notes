# 3. Naming Conventions

> **Tags:** #naming #clean-code #conventions #quality

Naming is the hardest part of software development. Good names make code self-documenting; bad names create confusion that no amount of comments can fix. This note covers naming conventions across languages and contexts.

---

## 3.1 Naming Styles by Language

| Language | Variables | Functions | Classes | Constants | Files |
| --- | --- | --- | --- | --- | --- |
| Python | `snake_case` | `snake_case` | `PascalCase` | `UPPER_SNAKE` | `snake_case.py` |
| JavaScript | `camelCase` | `camelCase` | `PascalCase` | `UPPER_SNAKE` | `kebab-case.js` or `camelCase.js` |
| Java | `camelCase` | `camelCase` | `PascalCase` | `UPPER_SNAKE` | `PascalCase.java` |
| C# | `camelCase` | `PascalCase` | `PascalCase` | `PascalCase` | `PascalCase.cs` |
| C++ | `snake_case` | `snake_case` | `PascalCase` / `snake_case` | `UPPER_SNAPE` / `kCamelCase` | varies |
| Go | `camelCase` / `PascalCase` | `camelCase` / `PascalCase` | `PascalCase` | `PascalCase` (exported) | `snake_case.go` |
| Rust | `snake_case` | `snake_case` | `PascalCase` | `UPPER_SNAKE` | `snake_case.rs` |

In Go, visibility is determined by case: `PascalCase` is exported (public), `camelCase` is unexported (private).

## 3.2 Principles of Good Names

### 1. Reveal Intent

The name should tell the reader what the variable represents, why it exists, and how it is used.

```python
# BAD
d = 7  # what is d?

# GOOD
elapsed_time_in_days = 7
```

### 2. Avoid Disinformation

Do not use names that imply something false.

```python
# BAD: "List" implies a specific data structure
account_list = ("checking", "savings")  # it is a tuple, not a list

# GOOD
accounts = ("checking", "savings")
```

### 3. Make Distinctions Meaningful

Do not add noise words just to make names unique.

```python
# BAD: what is the difference?
product_data = get_product_data()
product_info = get_product_info()

# GOOD: clear, distinct names
product = get_product()
product_metadata = get_product_metadata()
```

### 4. Use Pronounceable Names

You will discuss code out loud. Names you cannot pronounce make discussion painful.

```python
# BAD
genymdhms = generation_timestamp  # impossible to say

# GOOD
generation_timestamp = datetime.now()
```

### 5. Use Searchable Names

Single-letter names are hard to search for. Use them only for very short scopes (loop counters).

```python
# BAD: searching for "e" finds every letter 'e' in the codebase
for e in events:
    process(e)

# GOOD: searchable
for event in events:
    process(event)
```

### 6. Avoid Encodings

Do not encode type or scope information in names (Hungarian notation).

```python
# BAD: Hungarian notation
str_name = "Alice"
i_count = 0
m_users = {}  # m_ for "member"

# GOOD
name = "Alice"
count = 0
users = {}
```

## 3.3 Naming Specific Things

### Variables

Nouns describing what the variable holds:

```python
# Good
user_count = 42
is_authenticated = True
file_path = "/etc/config"

# Bad (verbs, vague)
do_thing = 42
data = True  # what data?
stuff = "/etc/config"
```

Boolean variables should be predicates: `is_`, `has_`, `can_`, `should_`.

```python
is_valid = True
has_permission = False
can_edit = True
should_retry = False
```

### Functions

Verbs describing what the function does:

```python
# Good
def calculate_total(items): ...
def send_email(to, subject): ...
def validate_user(user): ...

# Bad (nouns, vague)
def total(items): ...  # is it a variable or a function?
def email(to, subject): ...
def user(user): ...
```

For functions that return a boolean, use `is_`, `has_`, `can_`:

```python
def is_valid(email): ...
def has_permission(user, action): ...
def can_access(user, resource): ...
```

### Classes

Nouns describing what the class represents:

```python
# Good
class User: ...
class ShoppingCart: ...
class EmailSender: ...

# Bad (vague, "data"-suffixed)
class UserData: ...  # what kind of data?
class Manager: ...   # manages what?
class Processor: ... # processes what?
```

Avoid vague suffixes like `Manager`, `Processor`, `Handler`, `Helper`, `Util`. If you cannot name what the class manages, processes, or handles, it may be doing too many things (SRP violation).

### Constants

`UPPER_SNAKE_CASE` in most languages:

```python
MAX_RETRIES = 3
DEFAULT_TIMEOUT = 30
PI = 3.14159
```

## 3.4 Naming in Context

### Loop Counters

`i`, `j`, `k` are acceptable for simple nested loops. For anything else, use descriptive names:

```python
# OK for simple loops
for i in range(10):
    print(i)

# Better for complex loops
for row_index in range(rows):
    for column_index in range(columns):
        grid[row_index][column_index] = 0
```

### Callbacks and Event Handlers

Use `on_` prefix for event handlers, `_callback` or `_handler` suffix for callbacks:

```javascript
button.onClick = handleClick;          // event handler
fetch(url, { onSuccess: handleSuccess }); // callback
```

### Database Tables and Columns

- Table names: plural (`users`, `orders`, `products`) or singular (`user`, `order`) — pick one and be consistent.
- Column names: `snake_case`, descriptive (`created_at`, `user_id`, `is_active`).
- Foreign keys: `<table>_id` (`user_id`, `order_id`).
- Join tables: `<table1>_<table2>` (`user_role`, `order_product`).

## 3.5 Common Mistakes

- **Abbreviations that are not obvious.** `usr` for `user`, `cnt` for `count`. Spell it out.
- **Inconsistent naming.** `getUser` in one file, `fetch_user` in another, `retrieveUser` in a third. Pick one and use it everywhere.
- **Names that contradict the type.** A function named `getUsers` that returns a single user. A variable named `is_valid` that holds a string.
- **Names that are too specific.** `processUserAndSendWelcomeEmail` — split into two functions.
- **Names that are too vague.** `doStuff`, `handleData`, `process`.
- **Names with typos.** `recieve` instead of `receive`. Typos propagate; fix them immediately.

---

## 3.6 Key Takeaways

- Names reveal intent, avoid disinformation, and are pronounceable and searchable.
- Variables are nouns; functions are verbs; booleans are predicates (`is_`, `has_`).
- Constants are `UPPER_SNAKE_CASE`.
- Avoid vague suffixes (Manager, Processor, Helper, Util).
- Avoid abbreviations, encodings, and inconsistent naming.
- Fix typos immediately — they spread.
