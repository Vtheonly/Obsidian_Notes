# 2. Unit Testing

> **Tags:** #testing #unit-testing #tdd #frameworks

Unit testing is the foundation of a test suite. This note covers the mechanics of writing unit tests, the frameworks available in different languages, and the patterns that make unit tests effective.

---

## 2.1 What Makes a Good Unit Test

A good unit test:

1. **Tests one behavior.** If it fails, you know exactly what broke.
2. **Is fast.** Milliseconds, not seconds.
3. **Is isolated.** No dependency on other tests, the filesystem, the network, or a database.
4. **Is deterministic.** Same input always produces the same result.
5. **Has a descriptive name.** `test_calculateDiscount_returnsZeroWhenDiscountIs100Percent` not `test1`.

---

## 2.2 Test Structure

```python
# pytest example
import pytest
from myapp import calculate_discounted_price

class TestCalculateDiscountedPrice:
    def test_normal_discount(self):
        # Arrange
        price = 100
        discount = 20
        # Act
        result = calculate_discounted_price(price, discount)
        # Assert
        assert result == 80
    
    def test_zero_discount(self):
        assert calculate_discounted_price(100, 0) == 100
    
    def test_full_discount(self):
        assert calculate_discounted_price(100, 100) == 0
    
    def test_negative_discount_raises(self):
        with pytest.raises(ValueError):
            calculate_discounted_price(100, -10)
```

## 2.3 Frameworks by Language

### Python: pytest

```python
# test_calculator.py
import pytest
from calculator import add

def test_add_two_positive_numbers():
    assert add(2, 3) == 5

@pytest.mark.parametrize("a, b, expected", [
    (1, 1, 2),
    (0, 0, 0),
    (-1, 1, 0),
    (100, 200, 300),
])
def test_add_parametrized(a, b, expected):
    assert add(a, b) == expected
```

Run: `pytest test_calculator.py -v`

### JavaScript: Jest

```javascript
// calculator.test.js
const { add } = require('./calculator');

describe('add', () => {
  test('adds two positive numbers', () => {
    expect(add(2, 3)).toBe(5);
  });

  test.each([[1, 1, 2], [0, 0, 0], [-1, 1, 0]])(
    'adds %i + %i = %i',
    (a, b, expected) => {
      expect(add(a, b)).toBe(expected);
    }
  );
});
```

Run: `npx jest`

### Java: JUnit 5

```java
// CalculatorTest.java
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {
    @Test
    @DisplayName("Add two positive numbers")
    void addTwoPositiveNumbers() {
        Calculator calc = new Calculator();
        assertEquals(5, calc.add(2, 3));
    }

    @ParameterizedTest
    @CsvSource({"1, 1, 2", "0, 0, 0", "-1, 1, 0"})
    void addMultipleCases(int a, int b, int expected) {
        assertEquals(expected, new Calculator().add(a, b));
    }
}
```

Run: `mvn test`

### C#: xUnit / NUnit

```csharp
// CalculatorTests.cs
using Xunit;

public class CalculatorTests
{
    [Fact]
    public void Add_TwoPositiveNumbers_ReturnsSum()
    {
        var calc = new Calculator();
        Assert.Equal(5, calc.Add(2, 3));
    }

    [Theory]
    [InlineData(1, 1, 2)]
    [InlineData(0, 0, 0)]
    [InlineData(-1, 1, 0)]
    public void Add_MultipleCases_ReturnsExpected(int a, int b, int expected)
    {
        Assert.Equal(expected, new Calculator().Add(a, b));
    }
}
```

Run: `dotnet test`

## 2.4 Setup and Teardown

Tests often need common setup (creating objects, initializing state). Frameworks provide hooks:

```python
# pytest
import pytest

@pytest.fixture
def user():
    return User(name="Alice", email="alice@example.com")

def test_user_name(user):
    assert user.name == "Alice"

def test_user_email(user):
    assert user.email == "alice@example.com"
```

```javascript
// Jest
beforeEach(() => {
  // runs before each test
  initializeDatabase();
});

afterEach(() => {
  // runs after each test
  cleanDatabase();
});

beforeAll(() => {
  // runs once before all tests
});

afterAll(() => {
  // runs once after all tests
});
```

## 2.5 Parametrized Tests

Parametrized (or data-driven) tests run the same test logic with multiple inputs. This is one of the most powerful testing patterns:

```python
@pytest.mark.parametrize("input, expected", [
    ("hello", "HELLO"),
    ("World", "WORLD"),
    ("", ""),
    ("123", "123"),
])
def test_uppercase(input, expected):
    assert input.upper() == expected
```

One test function, four test cases. Each appears separately in the test report.

## 2.6 Testing Exceptions

```python
# pytest
def test_divide_by_zero_raises():
    with pytest.raises(ZeroDivisionError):
        divide(1, 0)

def test_divide_by_zero_raises_with_message():
    with pytest.raises(ZeroDivisionError, match="division by zero"):
        divide(1, 0)
```

```javascript
// Jest
test('throws on division by zero', () => {
  expect(() => divide(1, 0)).toThrow(ZeroDivisionError);
  expect(() => divide(1, 0)).toThrow('division by zero');
});
```

## 2.7 Mocking Dependencies

```python
# pytest with unittest.mock
from unittest.mock import Mock, patch

def test_send_welcome_email():
    email_service = Mock()
    user_service = UserService(email_service)
    
    user_service.register("alice@example.com")
    
    email_service.send.assert_called_once_with(
        "alice@example.com",
        "Welcome!"
    )
```

```javascript
// Jest
test('sends welcome email on registration', () => {
  const emailService = { send: jest.fn() };
  const userService = new UserService(emailService);
  
  userService.register('alice@example.com');
  
  expect(emailService.send).toHaveBeenCalledWith('alice@example.com', 'Welcome!');
  expect(emailService.send).toHaveBeenCalledTimes(1);
});
```

## 2.8 Test Naming Conventions

Good test names describe the behavior being tested:

| Bad name | Good name |
| --- | --- |
| `test1` | `test_add_returnsSumForTwoPositiveNumbers` |
| `testAdd` | `test_add_returnsFiveWhenAddingTwoAndThree` |
| `testError` | `test_register_throwsWhenEmailIsEmpty` |

A common pattern: `test_[method]_[scenario]_[expectedResult]`.

---

## 2.9 Common Mistakes

- **Testing too much in one test.** One assertion (or one logical group) per test.
- **Testing private methods.** Test the public interface; private methods are tested indirectly through public ones.
- **Sharing mutable state between tests.** Each test should set up its own state.
- **Asserting on implementation details.** "The loop ran 3 times" — no, assert the result.
- **Ignoring flaky tests.** A test that sometimes fails is worse than no test. Fix it or remove it.
- **Not running tests.** Tests you do not run are useless. Run them on every save (via watch mode) and on every push (via CI).

---

## 2.10 Key Takeaways

- Unit tests verify a single unit in isolation, fast and deterministically.
- Structure: Arrange, Act, Assert.
- Use parametrized tests for multiple input combinations.
- Use fixtures/setup for common initialization.
- Mock dependencies to isolate the unit under test.
- Name tests descriptively: `test_method_scenario_expected`.
- Test behavior, not implementation.
