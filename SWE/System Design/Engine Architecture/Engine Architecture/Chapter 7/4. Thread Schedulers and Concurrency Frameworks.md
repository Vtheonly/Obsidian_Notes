# 4. Thread Schedulers and Concurrency Frameworks

> "Multi-threaded engine code is hard — much harder than single-threaded code. The libraries in this note handle the hard parts (event loops, task graphs, lock-free data structures) so you can focus on the engine logic, not the synchronization primitives."

This is the fourth note in Chapter 7. The previous notes covered languages, data structures, and SIMD; this note covers the concurrency frameworks that enable multi-threaded engines.

---

## 7.4.1 Asynchronous Execution Runtimes

### Tokio (Rust)

**Tokio** is the dominant async runtime for Rust. It provides an event-driven execution model with work-stealing schedulers, async I/O, and timers.

**Features:**
- **Work-stealing scheduler.** Each worker thread has a local queue; idle threads steal work from busy threads. Provides good load balancing.
- **Async I/O.** Built on `epoll` (Linux), `kqueue` (BSD), `IOCP` (Windows). Non-blocking network and file I/O.
- **Timers.** Async sleep, timeouts.
- **Channels.** Multi-producer multi-consumer channels for inter-task communication.

**Usage:**
```rust
use tokio;

#[tokio::main]
async fn main() {
    let listener = tokio::net::TcpListener::bind("0.0.0.0:8080").await?;
    loop {
        let (socket, _) = listener.accept().await?;
        tokio::spawn(handle_connection(socket));
    }
}

async fn handle_connection(mut socket: tokio::net::TcpStream) {
    // ... handle the connection ...
}
```

**When to use:** for any Rust async I/O application. Tokio is the de facto standard.

**Performance:** comparable to hand-written C event loops; faster than most other languages' async runtimes.

### libuv (C, Cross-Platform)

**libuv** is the async I/O library originally developed for Node.js. It is now used by many C and C++ applications.

**Features:**
- **Event loop.** Single-threaded event loop with non-blocking I/O.
- **Thread pool.** For operations that cannot be made async (e.g., file I/O on most OSes).
- **Cross-platform.** Linux, macOS, Windows, BSD.
- **TCP/UDP, pipes, TTY, files, DNS.**

**Usage:**
```c
#include <uv.h>

void on_connect(uv_stream_t* server, int status) {
    // ... handle new connection ...
}

int main() {
    uv_loop_t* loop = uv_default_loop();
    uv_tcp_t server;
    uv_tcp_init(loop, &server);
    
    struct sockaddr_in addr;
    uv_ip4_addr("0.0.0.0", 8080, &addr);
    uv_tcp_bind(&server, (const struct sockaddr*)&addr, 0);
    uv_listen((uv_stream_t*)&server, 128, on_connect);
    
    return uv_run(loop, UV_RUN_DEFAULT);
}
```

**When to use:** for C/C++ applications that need a portable async I/O library. Particularly useful when portability across Linux/macOS/Windows is required.

### Boost.Asio (C++)

**Boost.Asio** is the standard C++ async I/O library. It is the foundation of many C++ network servers.

**Features:**
- **Async I/O.** TCP, UDP, timers, files (on some platforms).
- **Proactor pattern.** Async operations initiate a read/write; a callback is called when complete.
- **Coroutines (C++20).** Can be used with C++20 coroutines for cleaner async code.
- **Buffer management.** `boost::asio::buffer` for safe buffer handling.
- **SSL/TLS.** Built-in SSL support via OpenSSL.

**Usage:**
```cpp
#include <boost/asio.hpp>
using boost::asio::ip::tcp;

void handle_accept(tcp::socket* socket, const boost::system::error_code& ec) {
    // ... handle the connection ...
}

int main() {
    boost::asio::io_context io;
    tcp::acceptor acceptor(io, tcp::endpoint(tcp::v4(), 8080));
    
    tcp::socket* socket = new tcp::socket(io);
    acceptor.async_accept(*socket, std::bind(handle_accept, socket, std::placeholders::_1));
    
    io.run();
}
```

**When to use:** for C++ applications that need async I/O. Boost.Asio is mature, portable, and integrates well with the C++ ecosystem.

### Comparison

| Runtime | Language | Platform | Use Case |
|---|---|---|---|
| Tokio | Rust | Linux, macOS, Windows | Rust async I/O applications |
| libuv | C | Cross-platform | C/C++ applications needing portable async I/O |
| Boost.Asio | C++ | Cross-platform | C++ applications needing async I/O |

For **Rust** engines, use Tokio. For **C++** engines, use Boost.Asio. For **C** engines or projects needing maximum portability, use libuv.

## 7.4.2 Task-Graph Scheduling Engines

For CPU-bound multi-threaded workloads, task-graph schedulers provide better performance than raw threads.

### Intel TBB (Threading Building Blocks)

**Intel TBB** is a C++ library for task-based parallelism. It provides a work-stealing scheduler, concurrent containers, and high-level parallel patterns.

**Features:**
- **Work-stealing scheduler.** Each worker thread has a local deque of tasks; idle threads steal from the back of other threads' deques.
- **Parallel patterns.** `parallel_for`, `parallel_reduce`, `parallel_pipeline`, `parallel_sort`. High-level abstractions over the scheduler.
- **Concurrent containers.** `concurrent_hash_map`, `concurrent_vector`, `concurrent_queue`.
- **Flow graph.** A graph-based programming model where nodes are computations and edges are data dependencies.

**Usage:**
```cpp
#include <tbb/parallel_for.h>
#include <tbb/blocked_range.h>

void process(float* data, int n) {
    tbb::parallel_for(tbb::blocked_range<int>(0, n),
        [&](const tbb::blocked_range<int>& r) {
            for (int i = r.begin(); i < r.end(); i++) {
                data[i] = compute(data[i]);
            }
        });
}
```

**When to use:** for CPU-bound C++ applications that need parallelism. TBB is particularly good for data-parallel workloads (process N elements in parallel).

### OpenMP

**OpenMP** is a multi-platform shared-memory parallel programming standard. It uses compiler pragmas to express parallelism.

**Features:**
- **Pragmas.** `#pragma omp parallel for` parallelizes the next loop.
- **Tasks.** `#pragma omp task` creates a task.
- **Reductions.** `#pragma omp parallel for reduction(+:sum)` parallelizes a loop with a sum reduction.
- **SIMD.** `#pragma omp simd` forces SIMD vectorization.

**Usage:**
```c
#include <omp.h>

void process(float* data, int n) {
    #pragma omp parallel for
    for (int i = 0; i < n; i++) {
        data[i] = compute(data[i]);
    }
}

float sum(float* data, int n) {
    float result = 0;
    #pragma omp parallel for reduction(+:result)
    for (int i = 0; i < n; i++) {
        result += data[i];
    }
    return result;
}
```

**When to use:** for quick parallelization of existing C/C++ loops. OpenMP is simpler than TBB for basic parallelism, but less flexible.

### Rayon (Rust)

**Rayon** is a Rust library for data parallelism, similar to TBB but with Rust's safety guarantees.

**Features:**
- **Work-stealing scheduler.** Like TBB.
- **Parallel iterators.** `par_iter()` instead of `iter()` for parallel iteration.
- **Scoped threads.** Safely borrow data across threads.

**Usage:**
```rust
use rayon::prelude::*;

fn process(data: &mut [f32]) {
    data.par_iter_mut().for_each(|x| {
        *x = compute(*x);
    });
}

fn sum(data: &[f32]) -> f32 {
    data.par_iter().sum()
}
```

**When to use:** for any Rust application that needs data parallelism. Rayon is the Rust equivalent of TBB, with better safety guarantees.

### Comparison

| Library | Language | Model | Use Case |
|---|---|---|---|
| Intel TBB | C++ | Task graph | CPU-bound C++ parallelism |
| OpenMP | C, C++, Fortran | Pragmas | Quick parallelization of existing loops |
| Rayon | Rust | Parallel iterators | Rust data parallelism |

For **C++** engines, use TBB for sophisticated parallelism, OpenMP for quick parallelization. For **Rust** engines, use Rayon.

## 7.4.3 Lock-Free Data Structures

For multi-threaded engines, lock-free data structures avoid the overhead and contention of mutexes.

### Boost.Lockfree

**Boost.Lockfree** provides lock-free data structures in C++:

- **`boost::lockfree::queue`.** Multi-producer multi-consumer queue.
- **`boost::lockfree::stack`.** Multi-producer multi-consumer stack.
- **`boost::lockfree::spsc_queue`.** Single-producer single-consumer queue (fastest; no atomic operations needed for the queue itself).

**Usage:**
```cpp
#include <boost/lockfree/queue.hpp>

boost::lockfree::queue<int> q(1024);

// Producer thread
q.push(42);

// Consumer thread
int value;
while (!q.pop(value)) {
    // spin or sleep
}
```

### Folly Lock-Free Structures

**Folly** provides several lock-free structures:

- **`folly::MPMCQueue`.** Multi-producer multi-consumer queue.
- **`folly::AtomicHashMap`.** Fixed-size concurrent hash map.
- **`folly::ConcurrentHashMap`.** Growable concurrent hash map.
- **`folly::HazPtr`.** Hazard pointer implementation for safe memory reclamation in lock-free code.

### Crossbeam (Rust)

**Crossbeam** is the Rust library for lock-free data structures:

- **`crossbeam::queue::ArrayQueue` / `SegQueue`.** Multi-producer multi-consumer queues.
- **`crossbeam::channel`.** Multi-producer multi-consumer channels.
- **`crossbeam::epoch`.** Epoch-based memory reclamation for lock-free code.
- **`crossbeam::deque`.** Work-stealing deques.

### The Disruptor Pattern

The **Disruptor** pattern (Martin Thompson, LMAX) is a single-producer single-consumer ring buffer with no locks. It is the canonical pattern for HFT event processing.

**Implementation:**
```c
struct alignas(64) RingBuffer {
    Cell cells[SIZE];  // power-of-2 size
    alignas(64) std::atomic<uint64_t> cursor;
    alignas(64) std::atomic<uint64_t> consumer_cursor;
};

void publish(RingBuffer* rb, const Cell* cell) {
    uint64_t pos = rb->cursor.load(std::memory_order_relaxed);
    rb->cells[pos & (SIZE - 1)] = *cell;
    rb->cursor.store(pos + 1, std::memory_order_release);
}

bool consume(RingBuffer* rb, Cell* out) {
    uint64_t pos = rb->consumer_cursor.load(std::memory_order_relaxed);
    if (pos >= rb->cursor.load(std::memory_order_acquire)) {
        return false;  // empty
    }
    *out = rb->cells[pos & (SIZE - 1)];
    rb->consumer_cursor.store(pos + 1, std::memory_order_relaxed);
    return true;
}
```

The `alignas(64)` ensures the cursor and consumer_cursor are on separate cache lines, avoiding false sharing between the producer and consumer threads.

**Performance:** ~10 ns per event handoff, vs. ~1 μs for a `std::mutex`-protected queue — 100× faster.

## 7.4.4 Choosing a Concurrency Framework

The choice depends on the workload:

| Workload | Recommended framework |
|---|---|
| I/O-bound (network server) | Tokio (Rust), Boost.Asio (C++), libuv (C) |
| CPU-bound, data-parallel | Rayon (Rust), TBB (C++), OpenMP (any) |
| CPU-bound, task-graph | TBB (C++) |
| Inter-thread queue | Disruptor pattern, folly::MPMCQueue, crossbeam::queue |
| Concurrent hash map | folly::ConcurrentHashMap, crossbeam-skiplist |
| HFT event loop | Custom Disruptor + busy-polling |

For most engines, the combination of an async runtime (Tokio/Asio) for I/O and a task scheduler (Rayon/TBB) for CPU-bound work provides a good balance.

## 7.4.5 Common Pitfalls

### Pitfall 1: Using `std::mutex` in the Hot Path

`std::mutex` is ~20 ns per lock/unlock (uncontended); contended locks can be 1000+ ns. For hot-path code, use lock-free data structures or per-thread data.

### Pitfall 2: False Sharing

Two threads modifying different fields of the same cache line cause the cache line to bounce between cores, killing performance. Pad shared data to cache line boundaries (`alignas(64)`).

### Pitfall 3: Lock Contention

A single lock protecting a hot data structure becomes a bottleneck. Use sharded locks, lock-free structures, or per-thread data.

### Pitfall 4: Not Using Work-Stealing

A simple thread pool with a shared queue has lock contention on the queue. Work-stealing schedulers (TBB, Rayon, Tokio) have per-thread queues, reducing contention.

### Pitfall 5: Blocking the Event Loop

Async runtimes (Tokio, Asio) run on a small number of threads. Blocking one (e.g., with a long computation or a blocking syscall) prevents other tasks from running. Use `spawn_blocking` (Tokio) or a separate thread pool for blocking work.

### Pitfall 6: Not Pinning Threads to Cores

Thread migration loses cache state. Pin threads to specific cores with `pthread_setaffinity_np` (Linux) or `SetThreadAffinityMask` (Windows).

### Pitfall 7: Not Considering NUMA

On multi-socket systems, accessing remote memory is 2× slower than local memory. Use NUMA-aware allocation (`numa_alloc_onnode`).

### Pitfall 8: Too Many Threads

More threads than cores leads to context switching overhead. Use at most 1×–2× the number of cores.

### Pitfall 9: Not Using Hazard Pointers / Epoch-Based Reclamation

Naive lock-free code that frees memory immediately can cause use-after-free bugs. Use hazard pointers (Folly's HazPtr) or epoch-based reclamation (Crossbeam's epoch).

### Pitfall 10: Not Testing Under High Load

Concurrency bugs often appear only under high load. Test with realistic (and unrealistic) load; use tools like `stress-ng`, `wrk`, or `vegeta` for load testing.

## 7.4.6 Important Reminders

- **Use async runtimes (Tokio, Asio, libuv) for I/O-bound workloads.** They handle thousands of connections efficiently.
- **Use task schedulers (Rayon, TBB, OpenMP) for CPU-bound workloads.** They provide work-stealing and high-level parallel patterns.
- **Use lock-free data structures (Disruptor, folly::MPMCQueue, crossbeam::queue) for inter-thread communication.** 100× faster than mutex-protected queues.
- **Pad shared data to cache line boundaries.** Avoid false sharing.
- **Pin threads to cores.** Avoid migration.
- **Use NUMA-aware allocation on multi-socket systems.**
- **Do not use more threads than cores.** Context switching overhead dominates.
- **Use hazard pointers or epoch-based reclamation in lock-free code.** Avoid use-after-free.
- **Test under high load.** Concurrency bugs appear only under load.

---

## 7.4.7 Summary

Concurrency frameworks handle the hard parts of multi-threaded programming: event loops, task graphs, lock-free data structures.

For **async I/O**, use Tokio (Rust), Boost.Asio (C++), or libuv (C). For **CPU-bound parallelism**, use Rayon (Rust), TBB (C++), or OpenMP (any). For **inter-thread communication**, use the Disruptor pattern, folly::MPMCQueue, or crossbeam::queue.

Common pitfalls include using `std::mutex` in the hot path, false sharing, lock contention, not using work-stealing, blocking the event loop, not pinning threads, ignoring NUMA, using too many threads, not using hazard pointers, and not testing under load.

With the right concurrency framework, multi-threaded engines can scale to many cores with minimal synchronization overhead.
