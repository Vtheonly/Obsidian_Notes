# 1. Low-Level System Languages

> "Engines are not written in Python. They are not written in JavaScript. They are not written in Java (with rare exceptions). They are written in C, C++, or Rust — the languages that give the engineer control over memory layout, instruction selection, and hardware interaction. The choice of language is the choice of how close to the metal you can get."

This is the first note in Chapter 7 — the system engineer's tooling and library blueprint. We start with the foundation: the programming language. Everything else (libraries, frameworks, tools) is built on this choice.

---

## 7.1.1 C / C++

C and C++ are the **base layer** of almost every real engine. Trading engines, databases, search engines, compilers, game engines — the vast majority are written in C or C++.

### Why C/C++?

1. **Deterministic memory control.** The engineer decides when memory is allocated and freed. No garbage collector, no pauses, no surprises. This is essential for latency-sensitive engines.

2. **Manual cache optimization.** The engineer controls the layout of structs in memory — alignment, padding, packing. The compiler does what the engineer tells it to, not what it thinks is best.

3. **Inline assembly access.** For the hottest loops, the engineer can drop down to assembly language. This is rarely needed (compilers are good), but available when necessary.

4. **Direct compiler instruction tuning.** Compiler flags like `-march=native`, `-mavx2`, `-mavx-512ev` allow the engineer to target specific CPU features.

5. **Zero-cost abstractions.** C++ templates, inline functions, and constexpr provide abstraction without runtime cost. `std::sort<int>` is as fast as a hand-written int sort.

6. **Mature ecosystem.** 50+ years of libraries, tools, and best practices. Every hardware platform has a C/C++ compiler.

### Where C/C++ Is Used

- **Trading engines.** Latency-critical; C++ is the default. Examples: Jump Trading, Citadel Securities, Jane Street (uses OCaml for some components, but C++ for the hot path).
- **Databases.** PostgreSQL (C), MySQL (C/C++), MongoDB (C++), Redis (C), SQLite (C).
- **Search engines.** Elasticsearch (Java, but with native C++ components via JNI), Lucene (Java).
- **Compilers.** GCC (C/C++), Clang (C++), V8 (C++), HotSpot (C++).
- **Game engines.** Unreal (C++), Unity (C++ core, C# scripting), idTech (C++).
- **Chess engines.** Stockfish (C++), Komodo (C++), Houdini (C++).

### The C++ Trade-off

C++ gives the engineer maximum control — and maximum responsibility. The engineer must:
- Manage memory manually (or use smart pointers correctly).
- Avoid undefined behavior (signed overflow, use-after-free, data races).
- Handle errors explicitly (no exceptions in hot paths).
- Compile for each target platform (no "write once, run anywhere").

For engines, this trade-off is favorable. The control is worth the responsibility. But for less performance-critical applications, the trade-off is unfavorable — a managed language (Go, Java, Python) is more productive.

### Modern C++ (C++17, C++20, C++23)

Modern C++ has evolved significantly from C++98. Features that improve engine code:

- **`constexpr` and `consteval`.** Compute at compile time. Move constant expressions out of runtime.
- **`std::array` and `std::string_view`.** Fixed-size arrays and non-owning string references. No allocation overhead.
- **`std::optional` and `std::variant`.** Type-safe alternatives to pointers and unions.
- **Move semantics.** Avoid copies of large objects.
- **RAII.** Resource Acquisition Is Initialization. Deterministic destruction; no manual cleanup.
- **Templates and concepts.** Generic programming without runtime cost.
- **Coroutines (C++20).** Async programming without callbacks.

### C vs C++

C is simpler, smaller, and easier to reason about. C++ is more expressive, more powerful, and more complex.

For new engine projects, **C++ is usually the better choice.** The expressiveness (templates, RAII, exceptions) outweighs the complexity, and modern C++ is much safer than C++98.

For projects that must interoperate with C-only libraries or must run on platforms without a C++ compiler, C is appropriate.

## 7.1.2 Rust

Rust is the **modern alternative** to C++. It provides C++-level performance with strong memory safety guarantees, enforced at compile time.

### Why Rust?

1. **Zero-cost abstractions.** Like C++, Rust's abstractions (iterators, traits, generics) compile to the same machine code as hand-written code.

2. **Memory safety without GC.** Rust's ownership system enforces memory safety at compile time. No use-after-free, no double-free, no data races — without a garbage collector. The performance cost is zero (or near-zero).

3. **Strong concurrency model.** Rust's `Send` and `Sync` traits make data races a compile-time error. Multi-threaded Rust code is provably safe.

4. **Modern tooling.** `cargo` (build tool and package manager), `rustfmt` (formatter), `clippy` (linter), `rustdoc` (documentation generator). The tooling is more consistent and modern than C++'s.

5. **No undefined behavior (in safe Rust).** Safe Rust cannot exhibit undefined behavior. Unsafe Rust can (for cases where the engineer needs raw pointer access), but unsafe code is explicitly marked and auditable.

### Where Rust Is Used

- **High-performance backend systems.** Cloudflare's pingora, Discord's read states service.
- **Distributed systems.** TiKV (distributed key-value store), InfluxDB IOx (time-series database).
- **Web browsers.** Parts of Firefox's rendering engine (Servo).
- **Operating systems.** Redox OS (a Rust-based OS); Linux kernel modules (experimental).
- **Game engines.** Bevy (an ECS-based game engine); Fyrox.
- **Trading.** Some HFT firms are experimenting with Rust, though C++ remains dominant.

### The Rust Trade-off

Rust's ownership system is a **learning curve.** Engineers must understand ownership, borrowing, and lifetimes. Code that would be trivial in C++ (e.g., a doubly-linked list) requires careful design in Rust.

For engines, this trade-off is favorable. The safety guarantees prevent entire classes of bugs (use-after-free, data races) that are catastrophic in production. The learning curve is a one-time cost; the safety is a permanent benefit.

### Safe vs Unsafe Rust

Rust distinguishes **safe** and **unsafe** code:

- **Safe Rust.** The compiler guarantees memory safety and thread safety. The vast majority of engine code should be safe.
- **Unsafe Rust.** The engineer takes responsibility for safety. Used for:
  - Calling C/C++ libraries (FFI).
  - Implementing low-level data structures (linked lists, ring buffers).
  - Hardware-specific intrinsics (SIMD, atomics).
  - Performance-critical code where the safe abstraction is too slow.

Unsafe code is marked with the `unsafe` keyword, making it easy to audit. A well-engineered Rust codebase has < 5% unsafe code, confined to specific modules.

### Rust for Engines — Recommended Libraries

- **`tokio`** — async runtime (event loop, tasks, I/O).
- **`rayon`** — data parallelism (parallel iterators).
- **`crossbeam`** — lock-free data structures (queues, skip lists).
- **`parking_lot`** — fast synchronization primitives (mutexes, RwLocks).
- **`bytes`** — zero-copy byte buffers.
- **`serde`** — serialization/deserialization.
- **` ndarray`** — N-dimensional arrays (for linear algebra).
- **`simde`** or `std::simd` — portable SIMD.

## 7.1.3 Choosing Between C++ and Rust

| Criterion | C++ | Rust |
|---|---|---|
| Performance | Equal | Equal |
| Memory safety | Manual | Compile-time guaranteed |
| Concurrency safety | Manual | Compile-time guaranteed |
| Learning curve | Steep (but familiar) | Steeper (ownership) |
| Ecosystem maturity | Very mature | Maturing rapidly |
| Talent pool | Large | Smaller, growing |
| Tooling | Fragmented | Unified (cargo) |
| Interop with C | Native | Via FFI |
| Interop with other C++ | Native | Limited |

**Choose C++ when:**
- The team has deep C++ expertise.
- The project must interoperate with existing C++ code.
- The project needs libraries that are only available in C++.
- The project targets platforms without a mature Rust compiler.

**Choose Rust when:**
- The team is willing to learn Rust.
- Memory safety is critical (e.g., security-sensitive engines).
- Concurrency safety is critical.
- The project is new (no existing C++ to interop with).
- Modern tooling is important.

For **greenfield** engine projects today, Rust is increasingly the right choice. For **existing** C++ codebases, migrating to Rust is expensive and rarely worth it — improve the C++ instead.

## 7.1.4 Languages to Avoid for Engine Cores

Some languages are unsuitable for engine cores, despite being excellent for other purposes:

### Python

- **Pros:** Excellent for prototyping, scripting, data analysis, ML training.
- **Cons:** Interpreted; GIL prevents true multi-threading; dynamic typing prevents optimization.
- **Verdict:** Use for prototyping and tooling, not for the engine core. If the prototype is too slow, rewrite in C/C++/Rust.

### JavaScript / Node.js

- **Pros:** Excellent for I/O-bound web servers; large ecosystem.
- **Cons:** Interpreted (V8 JIT helps but is not enough for engine cores); garbage collection pauses.
- **Verdict:** Use for the web tier, not for the engine core.

### Java

- **Pros:** Mature ecosystem; JIT compilation can be fast; excellent tooling.
- **Cons:** Garbage collection pauses (even with ZGC); object layout is not engineer-controlled (header overhead, alignment).
- **Verdict:** Suitable for engines where occasional GC pauses are acceptable (e.g., Elasticsearch, Cassandra). Not suitable for HFT or other latency-critical engines.

### Go

- **Pros:** Simple; fast compilation; excellent concurrency primitives (goroutines).
- **Cons:** Garbage collection; less control over memory layout than C++/Rust.
- **Verdict:** Suitable for high-throughput servers (gRPC services, API gateways). Not suitable for engine cores where latency or memory layout matters.

### C#

- **Pros:** Mature ecosystem; JIT compilation; excellent tooling (.NET).
- **Cons:** Garbage collection; less control over memory layout.
- **Verdict:** Suitable for game engines (Unity) and Windows-only engines. Not suitable for cross-platform latency-critical engines.

### Swift

- **Pros:** Modern; safe; fast.
- **Cons:** Apple-centric; smaller ecosystem outside Apple platforms.
- **Verdict:** Suitable for Apple-platform engines. Less suitable for cross-platform server engines.

## 7.1.5 The Hybrid Approach

Many engines use a **hybrid approach**: a fast core in C/C++/Rust, with higher-level languages for surrounding functionality.

```mermaid
flowchart TB
    subgraph Hot Path C++ or Rust
        Core[Engine core: state, F, optimization]
    end
    subgraph Warm Path Java, Go, or C#
        App[Application logic: API, business rules]
    end
    subgraph Cold Path Python or JavaScript
        Tools[Tooling: data analysis, ML training, scripts]
    end
    Core -->|called via FFI or RPC| App
    App -->|invokes scripts| Tools
```

This pattern is common:
- **Chess engines:** C++ core; Python scripts for testing and analysis.
- **Search engines:** C++ index and query engine; Java/Go API tier; Python for ML training.
- **Trading engines:** C++ core for the hot path; Java/Python for risk monitoring and reporting.
- **Recommendation engines:** C++ inference engine; Python for model training; Go/Java for the API tier.

The hot path is in the fastest language; the cold path is in the most productive language. The boundary is well-defined (FFI or RPC), so the hot path can be optimized independently.

## 7.1.6 Common Pitfalls

### Pitfall 1: Using a Slow Language for the Core

Python, JavaScript, and other interpreted languages are too slow for engine cores. Even with JIT compilation, they cannot match C++/Rust for latency-critical workloads.

### Pitfall 2: Over-Using C++

C++ is appropriate for the engine core, but using it for the API tier, tooling, and ML training is often less productive than using a higher-level language. Use the right language for each layer.

### Pitfall 3: Not Learning Modern C++

C++98 idioms (raw pointers, manual memory management) are error-prone. Modern C++ (smart pointers, RAII, move semantics) is much safer. Learn and use modern C++.

### Pitfall 4: Fighting Rust's Borrow Checker

Engineers new to Rust often fight the borrow checker, trying to write C++-style code in Rust. The result is frustration. Learn Rust's idioms; design code around ownership, not against it.

### Pitfall 5: Mixing Too Many Languages

Each language adds complexity (build systems, FFI, deployment). Use at most 2–3 languages per project.

### Pitfall 6: Not Considering the Team's Expertise

A language the team does not know is a poor choice, even if it is technically superior. Training is a real cost.

### Pitfall 7: Premature Optimization of Language Choice

Do not choose C++/Rust for a prototype that may not need to be fast. Start with a productive language; rewrite the hot path in C++/Rust if needed.

---

## 7.1.7 Important Reminders

- **C, C++, and Rust are the engine languages.** Use them for the core.
- **Modern C++ (C++17+) is much safer than C++98.** Use smart pointers, RAII, move semantics.
- **Rust provides C++-level performance with memory safety.** Recommended for greenfield projects.
- **Avoid Python, JavaScript, and Java for the engine core.** They are too slow or have GC pauses.
- **Use the hybrid approach:** fast core in C++/Rust, productive languages for surrounding functionality.
- **Match the language to the team's expertise.** Training is a real cost.
- **Do not fight the language.** Learn the idioms; design code around them.
- **Premature language optimization is wasteful.** Start productive; rewrite the hot path if needed.

---

## 7.1.8 Summary

C, C++, and Rust are the languages for engine cores. They provide the memory control, hardware access, and zero-cost abstractions that engines require. C++ is the established choice; Rust is the modern alternative with strong safety guarantees.

Other languages (Python, JavaScript, Java, Go, C#) are unsuitable for engine cores due to interpretation overhead, garbage collection, or lack of memory control. They are appropriate for surrounding functionality (API tier, tooling, ML training).

The hybrid approach — fast core in C++/Rust, productive languages for the rest — is the dominant pattern in modern engine engineering.
