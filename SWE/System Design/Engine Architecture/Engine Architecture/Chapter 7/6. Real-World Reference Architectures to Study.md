# 6. Real-World Reference Architectures to Study

> "The best way to learn engine architecture is to study real engines. Not toy examples, not simplified models — the actual codebases that power production systems. Lucene, RocksDB, TensorRT, XGBoost, Bullet Physics: these are the masterworks of engine engineering, and their source code is available for you to read."

This is the sixth and final note in Chapter 7. The previous notes covered the tools and libraries; this note covers the real-world engines that exemplify the principles of this course. Studying these engines is the capstone of engine engineering education.

---

## 7.6.1 Lucene / Elasticsearch Core

**Lucene** is the leading open-source text search engine library. **Elasticsearch** is a distributed search server built on Lucene. Together, they power Wikipedia, GitHub, Netflix, and thousands of other services.

### What to Study

1. **Inverted index structure.** How Lucene represents the mapping from terms to documents. Look at:
   - `SegmentInfo` — the metadata for an index segment.
   - `TermsEnum` — iteration over terms in an index.
   - `PostingsEnum` — iteration over postings (documents containing a term).
   - The compression codecs (LZ4, FST — Finite State Transducers).

2. **Query evaluation.** How Lucene evaluates a query:
   - `IndexSearcher.search()` — the entry point.
   - `Weight.scorer()` — creates a scorer for each query clause.
   - `Scorer.iterator()` — iterates over matching documents.
   - The `BulkScorer` API for batch scoring.

3. **Analyzer pipeline.** How text is tokenized, normalized, and filtered:
   - `Tokenizer` — splits text into tokens.
   - `TokenFilter` — applies transformations (lowercasing, stemming, stopword removal).
   - `Analyzer` — combines tokenizer and filters.

4. **Lucene's codec architecture.** How Lucene abstracts the on-disk format:
   - `Codec` — the top-level format abstraction.
   - `PostingsFormat` — how postings are stored.
   - `DocValuesFormat` — how per-document values are stored.

5. **Segment merging.** How Lucene merges index segments in the background:
   - `LogMergePolicy` — when to merge.
   - `TieredMergePolicy` — better merge selection.
   - The merge thread pool.

### Why Study Lucene

Lucene is the canonical example of:
- **Inverted index design.** The most studied inverted index implementation.
- **Query evaluation optimization.** WAND, MaxScore, block-based scoring.
- **Compression codecs.** FST, LZ4, Roaring Bitmaps.
- **Concurrent indexing and searching.** Near-real-time search with segment immutability.

**Where to start:** the Lucene source code is on GitHub (apache/lucene). Start with `IndexSearcher.java`, then dive into `TermScorer.java` and `FuzzyQuery.java`.

## 7.6.2 LevelDB / RocksDB

**LevelDB** is Google's embedded key-value store, designed by Jeff Dean and Sanjay Ghemawat. **RocksDB** is Facebook's fork of LevelDB, optimized for flash storage and high-throughput workloads.

### What to Study

1. **LSM tree structure.** How RocksDB organizes data:
   - **Memtable.** In-memory sorted structure (skiplist).
   - **SSTables.** Immutable on-disk sorted files.
   - **Write-ahead log (WAL).** For durability.
   - **Levels.** SSTables are organized into levels (L0, L1, ..., Lmax).

2. **Compaction.** How RocksDB merges SSTables:
   - **Level compaction.** Merge SSTables from level N to N+1.
   - **Universal compaction.** Alternative strategy optimized for write amplification.
   - **FIFO compaction.** For time-series data.

3. **Bloom filters.** How RocksDB avoids disk reads for non-existent keys:
   - Per-SSTable Bloom filters.
   - Prefix Bloom filters for range queries.

4. **Block cache.** How RocksDB caches frequently-accessed data blocks in memory.

5. **Write batch.** How RocksDB supports atomic multi-key writes.

### Why Study LevelDB / RocksDB

LevelDB and RocksDB are the canonical examples of:
- **LSM tree design.** The write-optimized counterpart to B-trees.
- **Write-ahead logging.** How to achieve durability without sacrificing performance.
- **Bloom filter usage.** How to use Bloom filters to avoid expensive disk reads.
- **Background compaction.** How to do maintenance work without blocking the foreground.
- **Cache design.** How to cache hot data while keeping cold data on disk.

**Where to start:** the LevelDB source code is small (~10,000 lines) and well-commented. Start with `db/db_impl.cc`, then look at `table/table.cc` and `util/bloom.cc`. RocksDB adds significant complexity; study LevelDB first.

## 7.6.3 TensorRT

**TensorRT** is NVIDIA's high-performance deep learning inference engine. It takes a trained neural network and optimizes it for NVIDIA GPUs, achieving 2–10× speedup over frameworks like PyTorch.

### What to Study

1. **Graph optimization.** How TensorRT optimizes the neural network graph:
   - **Layer fusion.** Combine consecutive layers (e.g., Conv + ReLU + Bias) into a single kernel.
   - **Precision calibration.** Convert FP32 weights to FP16 or INT8 with minimal accuracy loss.
   - **Kernel auto-tuning.** Select the best GPU kernel for each layer from a library of implementations.

2. **Memory management.** How TensorRT minimizes GPU memory usage:
   - **Memory pooling.** Reuse memory across layers.
   - **Static memory planning.** Allocate all memory upfront; no allocation during inference.

3. **Kernel fusion.** How TensorRT combines operations to reduce kernel launch overhead:
   - Element-wise operations (add, multiply, ReLU) are fused into a single kernel.
   - Convolutions are fused with following activations and normalizations.

4. **INT8 quantization.** How TensorRT converts FP32 models to INT8:
   - Calibration: run the model on representative input; collect activation statistics.
   - Quantization: convert weights and activations to INT8 using the collected statistics.
   - Dequantization: convert INT8 outputs back to FP32 for the next layer.

### Why Study TensorRT

TensorRT is the canonical example of:
- **Inference engine design.** How to take a trained model and run it fast.
- **Graph optimization.** How to transform a computation graph for efficiency.
- **Kernel fusion.** How to combine operations to reduce overhead.
- **Quantization.** How to trade precision for performance.

**Where to start:** TensorRT is closed-source, but the documentation is extensive. Read the Developer Guide, then look at the samples in the `samples` directory of the TensorRT distribution.

For an open-source alternative, study **ONNX Runtime** (Microsoft) or **TVM** (Apache). Both provide similar functionality with open-source code.

## 7.6.4 XGBoost

**XGBoost** is the leading gradient-boosted decision tree library. It is used for tabular data classification and regression in finance, advertising, and many other domains.

### What to Study

1. **Tree construction.** How XGBoost builds decision trees:
   - **Histogram-based splits.** Pre-compute histograms of feature values; use them to find the best split.
   - **Sparsity-aware split finding.** Handle missing values efficiently.
   - **Approximate split finding.** Use quantiles for approximate splits on large datasets.

2. **Cache-aware access.** How XGBoost organizes data for cache-friendly access:
   - **Column blocks.** Store data in column-major order (one column at a time).
   - **Compressed columns.** Compress sparse columns.

3. **Out-of-core computation.** How XGBoost handles datasets larger than RAM:
   - **Block compression.** Compress blocks on disk.
   - **Sharding.** Distribute blocks across multiple disks.

4. **Distributed training.** How XGBoost scales to multiple machines:
   - **Data-parallel training.** Each worker has a subset of the data; gradients are aggregated.
   - **Feature-parallel training.** Each worker has a subset of the features; splits are aggregated.

### Why Study XGBoost

XGBoost is the canonical example of:
- **Cache-aware algorithm design.** How to lay out data for cache-friendly access.
- **Sparsity-aware computation.** How to handle missing values efficiently.
- **Out-of-core computation.** How to handle datasets larger than RAM.
- **Distributed training.** How to scale ML training to multiple machines.

**Where to start:** the XGBoost source code is on GitHub (dmlc/xgboost). Start with `src/tree/updater_histmaker.cc` (histogram-based splits), then look at `src/common/column_matrix.h` (cache-aware data layout).

## 7.6.5 Bullet Physics Engine

**Bullet Physics** is an open-source physics simulation engine, used in games (Grand Theft Auto, Red Dead Redemption), movies (Hugo, Sherlock Holmes), and robotics research.

### What to Study

1. **Broadphase collision detection.** How Bullet quickly eliminates pairs of objects that cannot be colliding:
   - **Sweep and prune.** Sort objects by their bounding box minima; only check pairs whose bounding boxes overlap.
   - **Dynamic AABB tree.** A bounding volume hierarchy that is updated as objects move.

2. **Narrowphase collision detection.** How Bullet computes the exact contact points for pairs of objects that may be colliding:
   - **GJK (Gilbert-Johnson-Keerthi) algorithm.** Computes the closest distance between two convex shapes.
   - **EPA (Expanding Polytope Algorithm).** Computes the penetration depth when shapes overlap.
   - **SAT (Separating Axis Theorem).** For convex polyhedra.

3. **Constraint solving.** How Bullet resolves collisions and other constraints:
   - **Sequential impulse solver.** Iteratively applies impulses to satisfy constraints.
   - **Projected Gauss-Seidel solver.** Alternative solver for more complex constraint systems.

4. **Continuous collision detection.** How Bullet handles fast-moving objects:
   - **Conservative advancement.** Iteratively advances the objects; checks for collision at each step.
   - **Time of impact.** Computes the exact time at which two objects will collide.

### Why Study Bullet Physics

Bullet Physics is the canonical example of:
- **Broadphase / narrowphase design.** The two-phase collision detection pattern used in all physics engines.
- **Constraint solving.** How to resolve multiple interacting constraints iteratively.
- **Continuous collision detection.** How to handle fast-moving objects without missing collisions.
- **Real-time simulation loops.** How to run a complex simulation within a fixed time budget.

**Where to start:** the Bullet source code is on GitHub (bulletphysics/bullet3). Start with `src/BulletCollision/BroadphaseCollision/btDbvt.cpp` (dynamic AABB tree), then look at `src/BulletDynamics/ConstraintSolver/btSequentialImpulseConstraintSolver.cpp` (constraint solving).

## 7.6.6 Other Engines Worth Studying

Beyond the five above, these engines are also worth studying:

### Stockfish (Chess Engine)

The strongest open-source chess engine. Exemplifies:
- **Bitboard-based state representation.**
- **Alpha-beta search with all modern pruning techniques.**
- **NNUE (efficiently updatable neural network) evaluation.**
- **Iterative deepening with any-time behavior.**
- **Parallel search with lazy SMP.**

**Where to start:** the Stockfish source code is on GitHub (official-stockfish/Stockfish). Start with `src/search.cpp`, then look at `src/evaluate.cpp` (NNUE) and `src/movegen.cpp` (bitboard move generation).

### Redis (In-Memory Database)

A high-performance in-memory key-value store. Exemplifies:
- **Single-threaded event loop.** All operations are O(1) or O(log N); no locks needed.
- **Efficient data structures.** Sorted sets (skiplist), hashes (ziplist for small, hash table for large), lists (quicklist).
- **Persistence.** RDB (snapshot) and AOF (append-only log).
- **Pipelining.** Batch multiple commands in a single network round-trip.

**Where to start:** the Redis source code is on GitHub (redis/redis). Start with `src/server.c` (event loop), then look at `src/t_string.c` (string commands) and `src/t_zset.c` (sorted set).

### PostgreSQL (Relational Database)

The leading open-source relational database. Exemplifies:
- **MVCC (Multi-Version Concurrency Control).** Readers do not block writers; writers do not block readers.
- **WAL (Write-Ahead Log).** For durability and replication.
- **Query planner.** Cost-based optimization of SQL queries.
- **B-tree index.** The primary index structure.
- **VACUUM.** Background cleanup of dead tuples.

**Where to start:** the PostgreSQL source code is on GitHub (postgres/postgres). Start with `src/backend/tcop/postgres.c` (top-level command processing), then look at `src/backend/optimizer/plan/planer.c` (query planner) and `src/backend/access/nbtree/nbtpage.c` (B-tree).

### Nginx (Web Server)

A high-performance web server and reverse proxy. Exemplifies:
- **Event-driven architecture.** Single-threaded per worker; handles thousands of connections.
- **Master-worker model.** Master process manages worker processes; workers handle connections.
- **HTTP pipeline.** Modular processing of HTTP requests.
- **Reverse proxy with caching.** Cache responses from upstream servers.

**Where to start:** the Nginx source code is at nginx.org. Start with `src/core/nginx.c` (main loop), then look at `src/http/ngx_http_request.c` (HTTP request handling).

### V8 (JavaScript Engine)

Google's JavaScript engine, used in Chrome and Node.js. Exemplifies:
- **JIT compilation.** Just-in-time compilation of JavaScript to machine code.
- **Garbage collection.** Generational GC with concurrent marking and sweeping.
- **Hidden classes.** Optimization for dynamic object shapes.
- **Inline caching.** Cache the result of method lookups.

**Where to start:** the V8 source code is on GitHub (v8/v8). Start with `src/d8/d8.cc` (the shell), then look at `src/objects/objects.h` (object model) and `src/compiler/` (JIT compiler).

## 7.6.7 How to Study an Engine

Studying an engine is a skill. The process:

### Step 1: Read the Documentation

Read the official documentation first. Understand what the engine does, what its API is, and how it is used. Do not dive into the source code until you understand the user-facing behavior.

### Step 2: Read the High-Level Architecture

Find a high-level architecture document (or a technical paper). Understand the major components and how they interact.

### Step 3: Read the Entry Points

Find the entry points — the functions that are called first. For a search engine, this is the query evaluation function. For a database, this is the command processing function. Read these functions to understand the high-level flow.

### Step 4: Trace a Single Request

Trace a single request through the engine. Use a debugger, add print statements, or read the code carefully. Understand each step the request goes through.

### Step 5: Dive into One Component

Pick one component that interests you (e.g., the inverted index, the B-tree, the query planner). Read its source code in depth. Understand its data structures, algorithms, and optimizations.

### Step 6: Modify and Experiment

Make a small modification (e.g., add a new metric, change a parameter, add a new feature). Build the engine with your modification. Run the tests. This is the deepest form of learning — you understand the code by changing it.

### Step 7: Write Up Your Findings

Write a blog post or a document explaining what you learned. Teaching is the best way to learn.

## 7.6.8 Common Pitfalls

### Pitfall 1: Trying to Understand Everything at Once

Engines are large. You cannot understand everything in one sitting. Focus on one component at a time.

### Pitfall 2: Skipping the Documentation

Diving into source code without understanding the user-facing behavior leads to confusion. Read the docs first.

### Pitfall 3: Reading Code Passively

Reading code without modifying it is like reading a textbook without doing the exercises. You will not retain the knowledge. Modify, experiment, build.

### Pitfall 4: Studying Only One Engine

Each engine has its own design philosophy. Studying multiple engines gives you a broader perspective and helps you recognize patterns.

### Pitfall 5: Not Studying the Tests

The test suite is documentation of how the engine is supposed to behave. Read the tests to understand the engine's contract.

### Pitfall 6: Copying Code Without Understanding

Do not copy code from these engines into your own without understanding it. You will inherit its bugs without understanding its design.

### Pitfall 7: Not Reading the Git History

The git history shows why the code is the way it is. Look at the commit messages and the issue tracker to understand the design decisions.

### Pitfall 8: Giving Up Too Early

Engines are complex. The first read-through will be confusing. Persist; the second read-through is much clearer.

- **Study real engines, not toy examples.** The principles are best learned from production code.
- **Lucene / Elasticsearch** for inverted indexes and query evaluation.

- **V8** for JIT compilation and garbage collection.
- **Read the documentation, then the entry points, then trace a request, then dive into one component, then modify and experiment.**
- **Engines are large; study one component at a time.**
- **Reading passively is not enough; modify and experiment.**

## 7.6.10 Summary

The best way to learn engine architecture is to study real engines. Lucene, LevelDB/RocksDB, TensorRT, XGBoost, and Bullet Physics are the canonical examples of their respective domains. Each exemplifies specific design patterns and optimization techniques.

Beyond these five, study Stockfish (chess), Redis (in-memory database), PostgreSQL (relational database), Nginx (web server), and V8 (JavaScript engine). Each is a masterwork of engine engineering.

The process: read the documentation, then the high-level architecture, then the entry points, then trace a single request, then dive into one component, then modify and experiment, then write up your findings.

Common pitfalls include trying to understand everything at once, skipping the documentation, reading passively, studying only one engine, not studying the tests, copying code without understanding, not reading the git history, and giving up too early.

Studying real engines is the capstone of engine engineering education. With this course's principles and these reference architectures, you have everything you need to build your own fast, correct, and scalable engines.
