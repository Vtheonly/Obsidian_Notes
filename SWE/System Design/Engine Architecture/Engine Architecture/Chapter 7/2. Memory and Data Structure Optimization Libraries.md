# 2. Memory and Data Structure Optimization Libraries

> "You do not write your own hash table. You do not write your own memory allocator. You do not write your own flat map. You use the libraries written by people who spent years getting them right — Google's Abseil, Meta's Folly, Boost, EA's EASTL. These libraries are the foundation of fast engine code."

This is the second note in Chapter 7. The previous note covered the programming language; this note covers the libraries that provide high-performance memory management and data structures.

---

## 7.2.1 High-Performance Memory Allocators

The default `malloc` (glibc) is general-purpose and slow. For engines, specialized allocators are 5–10× faster.

### Custom Allocators

**Arena allocator.** Allocates a large block upfront; bump-allocates from it. O(1) allocation; no per-allocation free (free the whole arena at once). Perfect for short-lived data.

```c
struct Arena {
    char* block;
    size_t capacity;
    size_t used;
};

void* arena_alloc(Arena* a, size_t size) {
    size = (size + 7) & ~7;  // align to 8 bytes
    void* ptr = a->block + a->used;
    a->used += size;
    return ptr;
}

void arena_reset(Arena* a) {
    a->used = 0;  // free everything at once
}
```

**Pool allocator.** Allocates fixed-size objects from a pre-allocated pool. O(1) allocation and deallocation. Perfect for many objects of the same type.

**Stack allocator.** Like arena, but supports scoped deallocation. Push a marker; allocate; pop back to the marker to free everything since. Perfect for temporary allocations within a function.

### jemalloc (Facebook)

**jemalloc** is Facebook's malloc implementation, originally developed by Jason Evans for FreeBSD. It is the default allocator in FreeBSD and is widely used in production.

**Key features:**
- **Thread-local caches.** Each thread has a small cache; allocations from the cache are lock-free.
- **Size-class-based allocation.** Allocations are rounded up to a size class; each size class has its own pool. Reduces fragmentation.
- **Multi-arena design.** Memory is partitioned into arenas; threads are assigned to arenas. Reduces lock contention.
- **Background thread.** Periodically purges unused pages, reducing memory footprint.

**Performance:** 5–10× faster than glibc malloc for multi-threaded workloads, primarily due to thread-local caches.

**Drop-in replacement:** link with `-ljemalloc` and most C/C++ programs automatically use jemalloc.

### tcmalloc (Google)

**tcmalloc** is Google's Thread-Caching Malloc. Similar design to jemalloc: thread-local caches, size classes, multi-arena.

**Performance:** comparable to jemalloc; both are 5–10× faster than glibc malloc.

**Drop-in replacement:** link with `-ltcmalloc`.

### mimalloc (Microsoft)

**mimalloc** is Microsoft's newer allocator. Competitive performance with jemalloc/tcmalloc; biased for short-lived allocations.

**Key features:**
- **Per-thread heaps.** Each thread has its own heap; allocations are lock-free.
- **First-fit allocation.** Reduces fragmentation for short-lived objects.
- **Delayed freeing.** Objects freed by other threads are deferred to the owning thread. Reduces cross-thread contention.

**Performance:** comparable to jemalloc; sometimes faster for short-lived allocation patterns.

### Choosing an Allocator

| Allocator | Best for | Notes |
|---|---|---|
| glibc malloc | Default; general purpose | Slow for multi-threaded workloads |
| jemalloc | Multi-threaded servers | Mature; widely used |
| tcmalloc | Multi-threaded servers | Similar to jemalloc |
| mimalloc | Short-lived allocations | Newer; competitive |
| Custom arena | Per-request data | O(1) allocation; no per-object free |
| Custom pool | Fixed-size objects | O(1) allocation and deallocation |
| Custom stack | Scoped temporaries | Push/pop markers |

**Recommendation:** for production engines, use **jemalloc** or **tcmalloc** as the default allocator, plus custom arenas/pools for specific patterns. The default glibc malloc should never be used in production.

## 7.2.2 Optimized Collections

The C++ Standard Template Library (STL) provides `std::vector`, `std::map`, `std::unordered_map`, etc. These are correct but not always optimal. Optimized collection libraries provide 2–10× better performance for specific use cases.

### Abseil (Google)

**Abseil** is Google's C++ library, providing drop-in replacements for many STL types.

**Key types:**

- **`absl::flat_hash_map` / `absl::flat_hash_set`.** Open-addressing hash maps/sets. 2–5× faster than `std::unordered_map` (which uses chaining). Cache-friendly (data is inline, not in separate nodes).

- **`absl::node_hash_map` / `absl::node_hash_set`.** Like `flat_hash_map` but with pointer stability (moving the map does not invalidate pointers to elements). Slightly slower than `flat_hash_map` but compatible with code that requires pointer stability.

- **`absl::btree_map` / `absl::btree_set`.** B-tree-based ordered maps/sets. More cache-friendly than `std::map` (which is a red-black tree). 2–5× faster for typical workloads.

- **`absl::Span`.** A non-owning view of a contiguous sequence (like `string_view` but for any type). Standardized in C++20 as `std::span`.

- **`absl::string_view`.** A non-owning view of a string. Standardized in C++17 as `std::string_view`.

**Usage:**
```cpp
#include "absl/container/flat_hash_map.h"
absl::flat_hash_map<std::string, int> map;
map["foo"] = 42;
```

### Folly (Meta)

**Folly** is Meta's C++ library, providing high-performance data structures and utilities.

**Key types:**

- **`folly::FBVector`.** A drop-in replacement for `std::vector` with better growth strategy (1.5× growth instead of 2×) and SIMD-optimized move/copy.

- **`folly::F14`.** A fast hash map/set. Uses SIMD for hash table operations; 2–3× faster than `absl::flat_hash_map` for large maps.

- **`folly::ConcurrentHashMap`.** A lock-free concurrent hash map. Scales to many cores without contention.

- **`folly::MPMCQueue`.** A multi-producer multi-consumer queue, lock-free. Used for inter-thread communication.

- **`folly::AtomicHashMap`.** A fixed-size concurrent hash map. Faster than `ConcurrentHashMap` for known-size workloads.

- **`folly::small_vector`.** A vector that stores small numbers of elements inline (no heap allocation) and falls back to heap allocation for larger sizes. Avoids allocation for the common case.

**Usage:**
```cpp
#include <folly/container/F14Map.h>
folly::F14FastMap<std::string, int> map;
map["foo"] = 42;
```

### Boost

**Boost** is the universal C++ utility library. It is large (100+ libraries), mature, and the testing ground for many features that later become standard.

**Key libraries for engines:**

- **`boost::container::flat_map` / `flat_set`.** Sorted-vector-based ordered containers. Faster lookups than `std::map` for small-to-medium sizes (binary search in contiguous memory vs. tree traversal).

- **`boost::lockfree::queue` / `stack`.** Lock-free data structures for inter-thread communication.

- **`boost::multi_index_container`.** A container that can be indexed by multiple keys. Useful for caches with multiple lookup patterns.

- **`boost::intrusive`.** Intrusive containers (the element itself contains the list/tree hooks, rather than the container allocating nodes). Lower overhead than non-intrusive containers.

- **`boost::asio`.** Async I/O and networking. The foundation of many C++ network servers.

### EASTL (Electronic Arts)

**EASTL** is Electronic Arts' STL replacement, optimized for game engines. The key difference from STL: EASTL allows specifying an allocator per container, has fixed-size containers, and is generally more cache-friendly.

**Key types:**

- **`eastl::vector`.** Like `std::vector` but with configurable allocator and faster growth strategy.
- **`eastl::fixed_vector`.** A vector with a fixed maximum size; no heap allocation.
- **`eastl::hash_map` / `hash_set`.** Open-addressing hash containers; faster than `std::unordered_map`.
- **`eastl::intrusive_list`.** Intrusive linked list; no node allocation.

**Usage:** primarily in game engines; less common in other domains. The fixed-size containers are useful in any engine with bounded working sets.

## 7.2.3 Choosing the Right Collection

The choice of collection depends on the access pattern:

| Access pattern | Recommended collection |
|---|---|
| Sequential scan | `std::vector`, `absl::Span` |
| Random access by index | `std::vector`, `std::deque` |
| Random access by key (small) | `absl::flat_hash_map`, `folly::F14` |
| Random access by key (large) | `folly::F14`, `absl::btree_map` |
| Ordered iteration | `absl::btree_map`, `boost::container::flat_map` |
| Concurrent access by key | `folly::ConcurrentHashMap` |
| Inter-thread queue | `folly::MPMCQueue`, `boost::lockfree::queue` |
| Small fixed-size list | `folly::small_vector`, `eastl::fixed_vector` |
| Multi-index lookup | `boost::multi_index_container` |

### Performance Comparison (Indicative)

For a 1-million-entry workload:

| Operation | `std::unordered_map` | `absl::flat_hash_map` | `folly::F14` |
|---|---|---|---|
| Insert | 200 ns | 80 ns | 50 ns |
| Lookup (hit) | 150 ns | 50 ns | 30 ns |
| Lookup (miss) | 200 ns | 60 ns | 40 ns |
| Erase | 180 ns | 70 ns | 50 ns |
| Iterate | 20 ns/element | 5 ns/element | 5 ns/element |

The differences come from:
- **Open addressing (flat_hash_map, F14)** vs. **chaining (unordered_map)**. Open addressing is more cache-friendly.
- **SIMD probing (F14)**. F14 uses SIMD to check 14 slots at once; flat_hash_map checks one slot at a time.
- **Inline storage (flat_hash_map, F14)**. Elements are stored inline in the table, not in separate nodes.

For engines, the choice of collection can give 5–10× speedup. Use `folly::F14` or `absl::flat_hash_map` instead of `std::unordered_map` whenever possible.

## 7.2.4 Specialized Data Structures

Beyond general collections, engines often need specialized data structures:

### Bloom Filters

A Bloom filter is a probabilistic data structure that tests set membership with false positives (but no false negatives).

**Use cases:**
- Pre-filtering before an expensive lookup. (e.g., "does this term appear in any document in this shard?")
- Caching presence information. (e.g., "is this user in the active set?")
- Distributed systems. (e.g., Bloom filters exchanged between nodes to avoid unnecessary RPCs.)

**Libraries:**
- `boost::bloom_filter` (Boost).
- `folly::BloomFilter` (Folly).

### Roaring Bitmaps

A Roaring Bitmap is a compressed bitmap for sets of integers. It is the standard for search engines and columnar databases.

**Use cases:**
- Document ID sets in search engines.
- Column values in columnar databases (Druid, ClickHouse).
- Set operations (intersection, union) on large integer sets.

**Libraries:**
- `CRoaring` (C library, with C++ bindings).

### HNSW (Approximate Nearest Neighbor)

HNSW is the standard data structure for approximate nearest neighbor search in high-dimensional spaces.

**Use cases:**
- Semantic search.
- Recommendation engines.
- Image similarity.

**Libraries:**
- `hnswlib` (C++ library).
- `faiss` (Facebook's library for similarity search; includes HNSW).
- `Milvus`, `Pinecone`, `Weaviate` (vector databases built on HNSW and similar).

### Lock-Free Queues

For inter-thread communication in multi-threaded engines.

**Libraries:**
- `boost::lockfree::queue` (Boost).
- `folly::MPMCQueue` (Folly).
- `moodycamel::ConcurrentQueue` (header-only library).

### B-Trees

For ordered data with frequent range queries.

**Libraries:**
- `absl::btree_map` (Abseil).
- `boost::container::flat_map` (Boost; sorted vector, faster for small sizes).
- `lmdb` (Lightning Memory-Mapped Database; a B-tree with memory-mapped file storage).

## 7.2.5 Common Pitfalls

### Pitfall 1: Using `std::unordered_map` in the Hot Path

`std::unordered_map` uses chaining, which causes cache misses on every lookup. Use `absl::flat_hash_map` or `folly::F14` instead.

### Pitfall 2: Using `std::map` for Small Ordered Maps

For small maps (< 100 elements), `boost::container::flat_map` (sorted vector) is faster than `std::map` (red-black tree) due to cache friendliness.

### Pitfall 3: Using `std::list` Anywhere

`std::list` is a doubly-linked list. Every operation is a cache miss. Use `std::vector` (or `folly::small_vector` for small sizes) instead.

### Pitfall 4: Allocating in the Hot Path

Even with `folly::F14`, allocation is not free. For maximum performance, pre-allocate all collections at startup; reuse them across iterations.

### Pitfall 5: Not Using Custom Allocators

For collections used in the hot path, specify a custom allocator (arena or pool). This eliminates per-element allocation.

### Pitfall 6: Using glibc malloc in Production

glibc malloc is general-purpose and slow. Use jemalloc, tcmalloc, or mimalloc in production.

### Pitfall 7: Not Benchmarking Alternative Collections

The choice of collection can give 5–10× speedup. Benchmark alternatives; do not assume one is best.

### Pitfall 8: Not Considering Concurrency

For multi-threaded engines, choose concurrent collections (`folly::ConcurrentHashMap`, `folly::MPMCQueue`). Wrapping a non-concurrent collection in a mutex is slower.

## 7.2.6 Important Reminders

- **Use jemalloc, tcmalloc, or mimalloc in production.** 5–10× faster than glibc malloc.
- **Use `absl::flat_hash_map` or `folly::F14` instead of `std::unordered_map`.** 2–5× faster.
- **Use `absl::btree_map` or `boost::container::flat_map` for ordered maps.** More cache-friendly than `std::map`.
- **Use `folly::small_vector` for small fixed-size lists.** Avoids allocation.
- **Use `folly::MPMCQueue` or `boost::lockfree::queue` for inter-thread communication.** Lock-free.
- **Use custom arena/pool allocators for hot-path collections.** Eliminates per-element allocation.
- **Use specialized data structures (Bloom filters, Roaring Bitmaps, HNSW) where appropriate.** They are 10–100× better than general collections for specific use cases.
- **Benchmark alternatives.** Do not assume one collection is best.
- **Do not use `std::list`.** Cache-unfriendly; use `std::vector` instead.

---

## 7.2.7 Summary

Memory and data structure libraries are the foundation of fast engine code. Use high-performance allocators (jemalloc, tcmalloc, mimalloc) instead of glibc malloc. Use optimized collections (`absl::flat_hash_map`, `folly::F14`, `absl::btree_map`) instead of STL collections. Use specialized data structures (Bloom filters, Roaring Bitmaps, HNSW) for specific use cases.

The choice of allocator and collection can give 5–10× speedup. Always benchmark alternatives; do not assume one is best. For hot-path collections, use custom allocators (arena, pool) to eliminate per-element allocation.
