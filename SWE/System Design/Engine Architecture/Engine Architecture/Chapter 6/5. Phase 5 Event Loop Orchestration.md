# 5. Phase 5 — Event Loop Orchestration

> "Phases 1–4 gave you a fast engine core. Phase 5 wraps that core in a runtime that handles the outside world: market events, search queries, user moves, configuration changes, signals. The core loop is fast; the event loop must be just as fast — and just as robust. A microsecond engine core is worthless if the event loop adds milliseconds of overhead."

This is the fifth and final note in the engine lifecycle. Phase 1 produced the state, Phase 2 produced a correct $F$, Phase 3 made $F$ structurally fast, Phase 4 micro-optimized the hot loop, and Phase 5 wraps everything in an event-driven runtime.

---

## 6.5.1 Implementing Continuous Processing Systems

The engine core is a function: `state ← F(state, context)`. The event loop is the runtime that calls this function repeatedly, feeding it events from the outside world.

### The Basic Event Loop

The simplest event loop:

```python
def event_loop(state, context):
    while not state.terminal:
        event = wait_for_event()
        state = handle_event(state, event, context)
    return state
```

This loop:
1. Waits for an external event (market data tick, search query, user move).
2. Handles the event (updates state, runs $F$, sends output).
3. Repeats until the engine terminates.

### Variants of the Event Loop

**Single-threaded event loop.** One thread handles all events sequentially. Simple, but cannot use multiple cores.

**Multi-threaded event loop.** Multiple threads handle events in parallel. Each thread has its own state; events are partitioned across threads. Scales to multiple cores, but requires thread-safe state.

**Event loop with worker pool.** The main thread receives events and dispatches them to a worker pool. Workers process events in parallel; the main thread collects results. Good for I/O-bound workloads (web servers).

**Reactive event loop.** The loop is driven by callbacks or promises. Events trigger callbacks; the loop's job is to schedule callbacks. Used in Node.js, Nginx, Twisted.

**Busy-polling event loop.** The loop spins on the event source (e.g., NIC's receive ring), never blocking. Lowest latency; highest CPU usage. Used in HFT.

### Choosing the Right Variant

| Workload | Recommended variant |
|---|---|
| CPU-bound, single-threaded | Single-threaded event loop |
| CPU-bound, multi-threaded | Multi-threaded event loop with sharded state |
| I/O-bound, many connections | Event loop with worker pool (or reactive) |
| Latency-critical | Busy-polling event loop |
| Mixed | Hybrid: busy-polling for hot events, worker pool for cold events |

## 6.5.2 Integrating External Events

External events come in many forms: network packets, file changes, signals, timer expirations, IPC messages. The event loop must integrate all of these into a single queue.

### Event Sources

**Network sockets.** The most common event source. Data arrives on a socket; the loop must read it.

- **Blocking I/O.** `recv()` blocks until data arrives. Simple, but the thread cannot do anything else while waiting.
- **Non-blocking I/O with polling.** `recv()` with `O_NONBLOCK` returns immediately. The loop polls all sockets in a round-robin. Inefficient for many sockets.
- **`select` / `poll`.** The OS tells the loop which sockets have data. Works for up to ~1000 sockets.
- **`epoll` (Linux) / `kqueue` (BSD).** Efficient event notification for thousands of sockets. The OS maintains a list of watched sockets and notifies the loop when one is ready.
- **`io_uring` (Linux 5.1+).** Modern asynchronous I/O. The loop submits I/O requests to a ring buffer; the kernel completes them asynchronously. Higher throughput than `epoll`.
- **Kernel bypass.** DPDK, Solarflare OpenOnload, AF_XDP. The application reads packets directly from the NIC, bypassing the kernel. Lowest latency; highest complexity.

**Files.** File I/O is similar to network I/O, but typically slower (disk seek time).

**Signals.** POSIX signals (e.g., `SIGINT`, `SIGTERM`) are asynchronous interrupts. The loop must handle them gracefully.

**Timers.** The loop must call certain functions at specific times (e.g., check the deadline every 1 ms).

**IPC.** Inter-process communication (pipes, shared memory, message queues) is another event source.

### The Unified Event Queue

The event loop integrates all event sources into a single queue:

```python
def event_loop(state, context):
    while not state.terminal:
        # Wait for any event source to be ready
        ready = poll_all_sources(timeout=compute_timeout(state))
        
        if ready.timer_expired:
            handle_timer(state, context)
        if ready.network_data:
            handle_network(state, context)
        if ready.signal:
            handle_signal(state, context)
        if ready.ipc_message:
            handle_ipc(state, context)
```

The `poll_all_sources` function uses `epoll` (or equivalent) to wait for any of the sources to be ready, with a timeout.

### Event Priorities

Not all events are equal. A market data tick in HFT is more urgent than a configuration update. The event loop should prioritize:

```python
def event_loop(state, context):
    while not state.terminal:
        # High-priority events first
        if has_market_data():
            handle_market_data(state, context)
        elif has_order_ack():
            handle_order_ack(state, context)
        # Medium-priority events
        elif has_timer_event():
            handle_timer(state, context)
        # Low-priority events
        elif has_config_update():
            handle_config_update(state, context)
        elif has_log_flush():
            handle_log_flush(state, context)
```

This ensures that high-priority events are handled with minimum latency, even when low-priority events are queued.

## 6.5.3 Handling Cancellation Signals

The event loop must handle cancellation: the engine may need to stop a long-running computation before it completes. This is essential for any-time algorithms (Chapter 5).

### Cooperative Cancellation

The simplest cancellation model: the long-running computation checks a "should I stop?" flag periodically and exits if set.

```python
def long_running_search(state, context, cancel_flag):
    while not state.terminal:
        if cancel_flag.is_set():
            raise CancelledError()
        # ... do work ...
```

The event loop sets the flag when cancellation is requested:

```python
def event_loop(state, context):
    while not state.terminal:
        event = wait_for_event()
        if event.type == CANCEL:
            cancel_flag.set()
            wait_for_search_to_finish()
        else:
            handle_event(state, event, context)
```

**Pros:** simple; no state corruption.
**Cons:** requires explicit checks; if a check is missing, the search may overrun.

### Preemptive Cancellation

The event loop kills the thread running the long-running computation.

```python
def event_loop(state, context):
    while not state.terminal:
        event = wait_for_event()
        if event.type == CANCEL:
            search_thread.kill()
            cleanup_state(state)
        else:
            handle_event(state, event, context)
```

**Pros:** works even if the search does not check the flag.
**Cons:** may corrupt state; resource leaks; not portable.

For engines, **cooperative cancellation is strongly preferred.** The search must check the cancel flag frequently — every 1000 iterations or every 1 ms.

### Cancellation in Any-Time Algorithms

Any-time algorithms (iterative deepening, MCTS) handle cancellation naturally: they always have a "best so far" answer ready. When cancelled, they return the best-so-far.

```python
def iterative_deepening(state, context, cancel_flag):
    best_so_far = None
    depth = 1
    while not cancel_flag.is_set():
        result = search_at_depth(state, depth, cancel_flag)
        if result is not None:
            best_so_far = result
            depth += 1
        else:
            break  # cancelled or failed
    return best_so_far
```

If the search is cancelled mid-depth, the partial result is discarded; the best-so-far from previous depths is returned.

## 6.5.4 Logging, Monitoring, and Observability

The event loop is also responsible for logging, monitoring, and observability. The engine's behavior must be visible to operators.

### Logging

**What to log:**
- Engine startup and shutdown.
- Configuration changes.
- Errors and warnings.
- Significant events (e.g., "started search for query X", "sent order Y").
- Performance metrics (latency, throughput).

**How to log:**
- Use a structured logger (e.g., JSON lines) for machine-readability.
- Include timestamps with microsecond precision.
- Include context (engine ID, thread ID, request ID).
- Async logging: the log call writes to a buffer; a separate thread flushes the buffer to disk. Avoids blocking the event loop.

**What NOT to log:**
- Per-event details in the hot loop (too much volume).
- Sensitive data (passwords, API keys).
- Anything that requires synchronous I/O in the hot loop.

### Monitoring

**Metrics to expose:**
- **Counters:** events processed, orders sent, errors.
- **Gauges:** queue depth, active connections, memory usage.
- **Histograms:** latency distribution, response size distribution.
- **Timers:** time spent in each phase of the engine.

**How to expose:**
- **Prometheus:** the de facto standard for metrics. Expose metrics via HTTP; Prometheus scrapes them.
- **StatsD:** a simpler protocol; metrics are sent via UDP.
- **OpenTelemetry:** a vendor-neutral standard for metrics, traces, and logs.

### Tracing

**Distributed tracing** tracks a single request across multiple services. Each service adds a span to the trace; the trace shows the entire request lifecycle.

For engines, tracing is useful for:
- Debugging slow requests.
- Understanding the call graph.
- Profiling in production.

**Tools:** Jaeger, Zipkin, OpenTelemetry, Datadog APM.

### Health Checks

The engine should expose a health check endpoint that reports its status:
- **Healthy:** the engine is running normally.
- **Degraded:** the engine is running but with issues (e.g., high latency, low cache hit rate).
- **Unhealthy:** the engine is not running or is in a critical state.

Load balancers use health checks to route traffic only to healthy instances.

## 6.5.5 Configuration and Dynamic Updates

The engine's configuration may need to change while it is running (e.g., adjusting risk limits, enabling/disabling features, switching models). The event loop must handle configuration updates without restarting.

### Configuration Sources

- **Static config file.** Read at startup; does not change.
- **Environment variables.** Read at startup; rarely change.
- **Command-line arguments.** Read at startup; do not change.
- **Configuration service.** Etcd, Consul, Zookeeper. The engine subscribes to changes; updates are pushed.
- **Database.** The engine periodically polls for changes.

### Hot Reload

Hot reload is the ability to update configuration without restarting:

```python
def event_loop(state, context):
    while not state.terminal:
        event = wait_for_event()
        if event.type == CONFIG_UPDATE:
            new_config = load_new_config()
            apply_config(context, new_config)  # update context, not state
        else:
            handle_event(state, event, context)
```

Configuration updates modify the *context*, not the *state*. The state continues to evolve; the context influences future decisions.

### Atomic Updates

Configuration updates must be atomic: the engine must not see a partially-updated configuration. Use copy-on-write:

```python
def apply_config(context, new_config):
    new_context = copy(context)
    new_context.config = new_config
    # Swap atomically
    atomic_swap(context, new_context)
```

The engine reads the context once per iteration; the swap ensures it sees either the old or new context, never a mix.

## 6.5.6 Common Pitfalls

### Pitfall 1: Blocking the Event Loop

The event loop must never block. If an event handler blocks (e.g., on I/O), no other events are processed. Use async I/O or offload blocking work to a separate thread.

### Pitfall 2: Per-Event Allocation

The event loop runs millions of times per second. Per-event allocation (e.g., creating a new string for each event) overwhelms the allocator. Pre-allocate buffers; reuse them across events.

### Pitfall 3: Synchronous Logging in the Hot Loop

Synchronous logging (write to disk per log entry) is too slow for the hot loop. Use async logging: write to a buffer; a separate thread flushes.

### Pitfall 4: Not Handling Cancellation

If the engine cannot be cancelled, it cannot be stopped gracefully. Always implement cooperative cancellation; check the cancel flag frequently.

### Pitfall 5: Not Handling Errors

Errors in event handlers must not crash the engine. Catch exceptions; log them; continue. The engine must be resilient to individual event failures.

### Pitfall 6: Not Handling Backpressure

If events arrive faster than the engine can process them, the queue grows unboundedly. Implement backpressure: when the queue is full, signal the source to slow down (or drop events).

### Pitfall 7: Not Testing Under Load

An event loop that works at 1000 events/second may fail at 100,000 events/second. Test under realistic (and unrealistic) load; verify graceful degradation.

### Pitfall 8: Not Monitoring

An unmonitored engine is a black box. Expose metrics; alert on anomalies; trace requests.

### Pitfall 9: Not Handling Configuration Changes

If the engine must be restarted to change configuration, it cannot meet high-availability requirements. Implement hot reload.

### Pitfall 10: Tight Coupling Between Event Loop and Engine Core

The event loop and the engine core should be loosely coupled. The event loop calls $F$; $F$ does not know about the event loop. This separation allows the engine core to be reused in different contexts (e.g., embedded in a service, run as a batch job, tested in isolation).

---

## 6.5.7 Important Reminders

- **The event loop must never block.** Use async I/O; offload blocking work.
- **Pre-allocate buffers.** No per-event allocation.
- **Async logging.** No synchronous I/O in the hot loop.
- **Cooperative cancellation.** Check the cancel flag frequently.
- **Handle errors gracefully.** The engine must not crash on individual event failures.
- **Backpressure is mandatory.** Unbounded queues lead to OOM.
- **Test under load.** Realistic and unrealistic.
- **Monitor everything.** Counters, gauges, histograms, traces.
- **Hot reload for configuration.** No restarts for config changes.
- **Loose coupling.** The event loop and engine core should be independent.

## 6.5.8 Summary

Phase 5 of the engine lifecycle is event loop orchestration: wrapping the engine core in a runtime that handles external events. The event loop integrates multiple event sources (network, files, signals, timers, IPC) into a unified queue, prioritizes events, handles cancellation, and provides logging, monitoring, and observability.

The basic event loop is `while not terminal: event = wait(); handle(event)`. Variants include single-threaded, multi-threaded, worker pool, reactive, and busy-polling. The choice depends on the workload.

Key design decisions: event prioritization (high-priority events first), cancellation (cooperative preferred), logging (async, structured), monitoring (counters, gauges, histograms), tracing (distributed), health checks, and configuration (hot reload, atomic updates).

Common pitfalls include blocking the event loop, per-event allocation, synchronous logging, no cancellation, no error handling, no backpressure, no load testing, no monitoring, no hot reload, and tight coupling between event loop and engine core.

With Phase 5 complete, the engine is production-ready: fast, correct, observable, and resilient. This completes Chapter 6 — Practical Engine Construction and Engineering Lifecycle. Chapter 7 covers the tools and libraries that engine engineers reach for along the way.
