# 1. Phase 1 — High-Fidelity State Mapping

> "Every engine begins with a question: what is the *state*? Get this wrong, and no amount of algorithmic cleverness or hardware optimization will save you. Get it right, and the rest of the engineering is downhill. Phase 1 of the engine lifecycle is the most important phase, and it is the phase most engineers rush through."

This is the first of five notes covering the practical engineering lifecycle of building an engine. The lifecycle is divided into five phases:

1. **Phase 1: High-fidelity state mapping.** Define the state $S$ and context $C$.
2. **Phase 2: Build the transition function $F$.**
3. **Phase 3: Structural optimizations over brute force.**
4. **Phase 4: Profiling-driven micro-optimization.**
5. **Phase 5: Event loop orchestration.**

Each phase builds on the previous. Skipping or rushing a phase leads to problems in later phases that are expensive to fix.

---

## 6.1.1 Abstracting and Minimizing System Representation

The first task of Phase 1 is to **abstract** the real-world problem into a tractable computational representation. The real world is messy; the engine's state must be clean.

### The Abstraction Process

Abstraction proceeds in three steps:

1. **Identify the entities.** What are the "things" in the problem? (Chess: pieces, squares, side to move. Trading: orders, prices, positions. Search: documents, terms, queries.)
2. **Identify the relationships.** How do the entities relate? (Chess: pieces occupy squares; one side moves at a time. Trading: orders have prices and quantities; positions are net of trades. Search: documents contain terms; queries match terms.)
3. **Identify the dynamics.** How does the state change over time? (Chess: moves change piece positions. Trading: orders are added, modified, canceled, filled. Search: documents are indexed; queries are evaluated.)

The output of this process is a **conceptual model** of the problem — a mental picture of what the engine is reasoning about.

### Decoupling Physical Entity Representations from Working Runtime Data Structures

The conceptual model is *not* the same as the runtime data structure. The conceptual model is what the engine is reasoning about; the runtime data structure is how the engine represents it in memory.

For example, in chess:
- **Conceptual model:** 64 squares, each may contain a piece; one side moves at a time; certain special rules (castling, en passant).
- **Runtime data structure:** Eight 64-bit bitboards (one per piece type), plus 8 bytes of metadata.

The bitboards are a *compressed* representation of the conceptual model. The compression preserves all the information the engine needs (piece locations, side to move, special rules) while fitting in a small, cache-friendly footprint.

### The Compression Imperative

The runtime data structure should be **as small as possible** while preserving all the information the engine needs. Smaller is faster (fits in cache); smaller is cheaper (less memory); smaller is simpler (fewer things to update).

Compression techniques:

1. **Bit-packing.** Encode multiple values into a single machine word.
2. **Canonicalization.** Store only one representative of equivalent states.
3. **Differential encoding.** Store deltas from a reference state.
4. **Hashing.** Store hashes for comparison (not for reconstruction).
5. **Implicit representation.** Compute derived values on demand rather than storing them.

The goal: hot state fits in L1 cache (32 KB). Warm state fits in L2 (256 KB). Cold state can be in DRAM.

### The Information-Preservation Test

A compressed representation is correct if and only if it preserves all the information the engine needs to make its decisions. Test this by asking:

1. **Can the engine reconstruct the conceptual model from the runtime representation?** (Not necessarily literally, but functionally — can it answer any question about the conceptual model?)
2. **Are there two distinct conceptual states that map to the same runtime representation?** (If so, the compression is lossy in a way that may matter.)
3. **Are there runtime states that do not correspond to any conceptual state?** (If so, the representation has redundancies or invalid states.)

If the answer to (1) is "yes" and the answers to (2) and (3) are "no", the compression is correct.

## 6.1.2 Ensuring Memory Contiguity

The second task of Phase 1 is to ensure the state is laid out in memory **contiguously** — packed into a small number of cache lines, with no wasted space.

### The Cache-Line Budget

A cache line is 64 bytes. The hot state should fit in as few cache lines as possible:

| Cache lines | Bytes | Cache level | Access time |
|---|---|---|---|
| 1 | 64 | L1 | 1 ns |
| 4 | 256 | L1 | 1 ns |
| 32 | 2 KB | L1 | 1 ns |
| 512 | 32 KB | L1 (max) | 1 ns |
| 4096 | 256 KB | L2 (max) | 4 ns |
| 500K | 30 MB | L3 (max) | 12 ns |
| ∞ | ∞ | DRAM | 100 ns |

Design target: hot state fits in 1–4 cache lines (64–256 bytes). This gives 1 ns access time.

### Designing for Contiguity

To make the state contiguous:

1. **Pack fields into a single struct.** Do not use pointer-based linked structures.
2. **Align the struct to a cache line boundary.** `alignas(64) struct HotState { ... };`.
3. **Separate hot and cold fields.** If the struct has hot fields (touched every iteration) and cold fields (touched rarely), split into two structs.
4. **Use fixed-size arrays, not dynamic containers.** `Order orders[MAX_ORDERS]` is contiguous; `std::vector<Order>` may not be (and has allocation overhead).
5. **Avoid padding.** Order fields from largest to smallest to minimize padding.

### Example: Cache-Line-Aligned Chess State

```c
struct alignas(64) ChessState {
    // 8 bitboards (8 bytes each) = 64 bytes
    uint64_t white_pawns;
    uint64_t white_knights;
    uint64_t white_bishops;
    uint64_t white_rooks;
    uint64_t white_queens;
    uint64_t white_king;
    uint64_t black_pawns;
    uint64_t black_knights;
    // ... 4 more bitboards for black, but we're out of space in this cache line
};

// Better: split into two cache lines
struct alignas(64) ChessState {
    uint64_t bitboards[12];   // 96 bytes — spans 2 cache lines
    uint8_t side_to_move;     // 1 byte
    uint8_t castling_rights;  // 1 byte (4 bits used)
    uint8_t en_passant;       // 1 byte (square index, 0 = none)
    uint8_t halfmove_clock;   // 1 byte
    uint16_t fullmove_number; // 2 bytes
    uint32_t padding;         // 4 bytes
    uint64_t zobrist_hash;    // 8 bytes
    // Total: 96 + 8 + 8 = 112 bytes — 2 cache lines
};
```

112 bytes fits in 2 cache lines — 2 ns to read the entire state from L1.

### The Hot-Cold Split

If a struct has both hot and cold fields, split them:

```c
// Hot: touched every iteration of the search
struct alignas(64) ChessStateHot {
    uint64_t bitboards[12];   // 96 bytes
    uint64_t zobrist_hash;    // 8 bytes
    // Plus 56 bytes of padding to fill the second cache line
};  // 128 bytes — 2 cache lines

// Cold: touched rarely (e.g., for display, logging)
struct ChessStateCold {
    uint8_t side_to_move;
    uint8_t castling_rights;
    uint8_t en_passant;
    uint8_t halfmove_clock;
    uint16_t fullmove_number;
    char fen_string[80];     // for display
    MoveList move_history;   // for threefold repetition
};
```

The hot struct is in L1 every iteration. The cold struct is fetched only when needed (e.g., for displaying the position).

## 6.1.3 The State Design Process

Designing the state is an iterative process. The steps:

### Step 1: List All Fields

Start by listing every field the engine might need:

- Position (board, order book, document, etc.)
- Side / role / actor
- History (for undo, repetition detection, audit)
- Counters (move number, iteration number)
- Time budget remaining
- Cache pointers (transposition table, etc.)
- Statistics (nodes searched, cache hits, etc.)

### Step 2: Classify Each Field

For each field, classify:
- **Hot or cold?** Is it touched every iteration, or rarely?
- **Mutable or immutable?** Does it change during execution?
- **Derivable or stored?** Can it be computed from other fields?
- **Bounded or unbounded?** Does it have a known maximum size?

### Step 3: Eliminate Redundant Fields

If a field can be derived from other fields, eliminate it (unless caching the derivation is a deliberate optimization).

### Step 4: Pack the Hot Fields

Pack the hot fields into a cache-line-aligned struct. Use bit-packing to minimize size.

### Step 5: Pack the Cold Fields

Pack the cold fields into a separate struct. Order fields by size (largest first) to minimize padding.

### Step 6: Choose the Right Data Structures

For collections (lists, maps, sets), choose the right data structure:
- **Sequential access?** Flat array.
- **Random access by key?** Hash table (small) or B-tree (large).
- **Range queries?** B-tree or sorted array.
- **Similarity queries?** ANN structure (HNSW, IVF).

### Step 7: Document the Layout

Write a comment explaining the layout — why each field is where it is, what the alignment is, what the cache behavior is. Future you (and your colleagues) will thank you.

### Step 8: Benchmark

Measure the cache behavior of the state:
- `perf stat` to measure cache hit rate.
- Verify that the hot state fits in L1.
- If the cache hit rate is below 90%, redesign.

## 6.1.4 Common Pitfalls

### Pitfall 1: Object-Oriented State

Engineers trained in OOP tend to model state as objects with methods. This leads to pointer-based structures, virtual function calls, and poor cache behavior. Prefer plain structs with free functions.

### Pitfall 2: Storing Derivable Data

If a field can be derived from others, do not store it (unless derivation is expensive). Storing derivable data wastes memory and creates consistency bugs.

### Pitfall 3: Mixing Hot and Cold Data

Putting hot and cold fields in the same struct pollutes the cache with cold data. Split them.

### Pitfall 4: Pointer Chasing

Linked structures (linked lists, trees with pointer children) cause cache misses. Use array-based representations.

### Pitfall 5: Wrong Alignment

A struct that is 65 bytes wastes an entire cache line. Pad to 64, 128, or 192 bytes.

### Pitfall 6: No Documentation

State layout decisions are often subtle. Without documentation, future changes will break the layout (and the performance).

### Pitfall 7: Premature Optimization

Do not optimize the state layout before you have a working (if slow) engine. Get the engine working first, then optimize the state based on profiling.

### Pitfall 8: Not Testing State Operations

State operations (make_move, undo_move, serialize, deserialize) must be tested exhaustively. A bug in any of these corrupts the entire engine.

## 6.1.5 Important Reminders

- **State is the most important design decision.** Get it right first.
- **Hot state fits in L1 (32 KB).** Cold state can be in DRAM.
- **Use bit-packing.** Pack small fields into bytes/words.
- **Separate hot and cold fields.** Do not let cold data pollute the cache.
- **Use contiguous arrays, not linked structures.** Always.
- **Align structs to cache line boundaries.** `alignas(64)`.
- **Document the layout.** Future you will thank you.
- **Test state operations exhaustively.** Bugs here are catastrophic.
- **Profile cache behavior.** `perf stat`. Aim for > 90% L1/L2 hit rate.
- **Get a working engine first, then optimize.** Premature optimization wastes time.

---

## 6.1.6 Summary

Phase 1 of the engine lifecycle is high-fidelity state mapping: defining the state $S$ and context $C$ that the engine will reason about. This is the most important phase, because a poorly chosen state representation caps the engine's performance no matter how clever the algorithm.

The state design process has eight steps: list all fields, classify each field, eliminate redundant fields, pack the hot fields, pack the cold fields, choose the right data structures, document the layout, and benchmark. The goal is a cache-line-aligned, contiguous, minimal representation that preserves all the information the engine needs.

Common pitfalls include object-oriented state (pointer-based, poor cache behavior), storing derivable data (wastes memory), mixing hot and cold data (cache pollution), pointer chasing (cache misses), wrong alignment (wasted cache lines), no documentation, premature optimization, and insufficient testing.

With a well-designed state, the engine is ready for Phase 2: building the transition function $F$.
