# 3. Phase 3 — Structural Optimizations over Brute-Force

> "Phase 2 gave you a correct slow engine. Phase 3 makes it fast — not by micro-optimizing the existing code, but by *replacing* brute-force algorithms with structured ones. Precomputation. Caching. Indexing. These are the techniques that turn a 100-million-node search into a 1-million-node search."

This is the third note in the engine lifecycle. Phase 1 produced the state, Phase 2 produced a correct $F$, and Phase 3 replaces the brute-force parts of $F$ with structured optimizations. Phase 4 (next note) will handle micro-optimizations; Phase 3 is about *algorithmic* and *structural* changes.

---

## 6.3.1 Identifying Precomputation Opportunities

**Precomputation** is the technique of moving computation from runtime to compile time or initialization time. If a computation depends only on static data, do it once upfront and look up the result at runtime.

### The Precomputation Test

A computation is precomputable if:

1. **Its inputs are known at startup.** The inputs do not depend on runtime state.
2. **Its outputs fit in available memory.** Precomputation trades memory for compute; if the outputs do not fit, precomputation is not viable.
3. **Its outputs are accessed frequently.** The precomputation cost is amortized across many accesses; if the output is accessed once, precomputation is wasteful.

### Examples of Precomputation

**Chess engines:**
- **Attack tables.** For each (piece type, square), precompute the bitboard of squares attacked. 64 × 6 = 384 entries × 8 bytes = 3 KB. Fits in L1.
- **Magic bitboards.** For sliding pieces, precompute the attack sets for all relevant occupancies. ~800 KB. Fits in L2.
- **Zobrist hash keys.** Precompute random 64-bit numbers for each (piece, square) combination. Used for incremental hashing.
- **Endgame tablebases.** Precompute exact results for positions with ≤ 7 pieces. ~150 TB for 7-piece; stored on SSD.

**Search engines:**
- **Inverted index.** Precompute the mapping from terms to documents. Built once (or incrementally updated); queried billions of times.
- **PageRank.** Precompute the authority score for every document. Computed monthly.
- **Document embeddings.** Precompute the embedding for every document. Used for semantic search.

**Trading engines:**
- **Risk limit tables.** Precompute the per-instrument risk limits at startup.
- **Symbol resolution.** Precompute the mapping from symbol name to instrument ID.
- **Fee schedules.** Precompute the per-venue fee structure.

**Parsers:**
- **LL/LR parse tables.** Precompute the parser's transition table at parser-generation time.
- **Keyword recognition.** Precompute a perfect hash function for the language's keywords.

**Recommendation engines:**
- **Item embeddings.** Precompute the embedding for every item. Updated periodically.
- **ANN index.** Precompute the HNSW (or similar) index over item embeddings.
- **User cluster assignments.** Precompute which cluster each user belongs to (for coarse retrieval).

### The Precomputation Process

To precompute a value:

1. **Identify the static inputs.** What does the computation depend on?
2. **Compute the outputs offline.** Run the computation once, store the results.
3. **Load the outputs at engine startup.** Read from disk or embed in the binary.
4. **Look up at runtime.** Replace the runtime computation with a lookup.

```python
# Before precomputation
def get_knight_attacks(square):
    attacks = 0
    for dx, dy in [(-2,-1), (-2,1), (-1,-2), (-1,2), (1,-2), (1,2), (2,-1), (2,1)]:
        target = square + dx + dy * 8
        if 0 <= target < 64:
            attacks |= 1 << target
    return attacks

# After precomputation
KNIGHT_ATTACKS = [compute_knight_attacks(sq) for sq in range(64)]  # at startup

def get_knight_attacks(square):
    return KNIGHT_ATTACKS[square]  # O(1) lookup
```

### When Precomputation Is Not Viable

Precomputation is not viable when:

- **The inputs change at runtime.** If the computation depends on runtime state, it cannot be precomputed.
- **The outputs are too large.** If the outputs do not fit in memory (or in cache), precomputation does not help.
- **The outputs are rarely accessed.** If the output is accessed once, precomputation is wasteful.

In these cases, consider **caching** (compute on demand, cache the result) instead.

## 6.3.2 Designing Bounded Cache Layers

**Caching** is the technique of storing the results of expensive computations and returning the cached result on subsequent requests. Unlike precomputation (which is done upfront), caching is done on demand.

### Cache Design Questions

When designing a cache, answer these questions:

1. **What is the key?** The cache key must uniquely identify the input. For a chess transposition table, the key is the Zobrist hash. For a search query cache, the key is the query string.

2. **What is the value?** The cached result. For a transposition table, the value is `(best_move, score, depth, flag)`. For a query cache, the value is the ranked result list.

3. **What is the size?** How many entries can the cache hold? Size determines memory usage and hit rate.

4. **What is the replacement policy?** When the cache is full, which entry is evicted? LRU, LFU, depth-preferred, etc.

5. **What is the invalidation policy?** When does a cached entry become stale? TTL, event-based, or never (until evicted).

6. **What is the concurrency model?** How is the cache accessed from multiple threads? Lock-based, lock-free, or per-thread.

### Bounded Caches

A **bounded cache** has a fixed maximum size. When full, adding a new entry requires evicting an old one. Bounded caches are essential for production engines — unbounded caches will eventually run out of memory.

```python
class BoundedCache:
    def __init__(self, capacity, replacement_policy='lru'):
        self.capacity = capacity
        self.cache = {}
        self.replacement_policy = replacement_policy
    
    def get(self, key):
        if key in self.cache:
            self._touch(key)
            return self.cache[key]
        return None
    
    def put(self, key, value):
        if key in self.cache:
            self.cache[key] = value
            self._touch(key)
        else:
            if len(self.cache) >= self.capacity:
                self._evict()
            self.cache[key] = value
            self._touch(key)
    
    def _touch(self, key):

        pass
    
    def _evict(self):
```

### Replacement Policies

The choice of replacement policy depends on the access pattern:

- **LRU (Least Recently Used).** Best for temporal locality (recently-accessed items are likely to be accessed again). Simple to implement.
- **LFU (Least Frequently Used).** Best for skewed access patterns (a few items are accessed very frequently).
- **W-TinyLFU.** A modern policy that combines LRU and LFU. State-of-the-art for general-purpose caches.
- **Depth-preferred.** For transposition tables: evict the entry with the shallowest search depth. Deeper entries are more valuable.
- **ARC (Adaptive Replacement Cache).** Dynamically balances LRU and LFU based on workload. Used in ZFS.

### Concurrency Models

For multi-threaded engines, the cache must be thread-safe:

- **Lock-based.** A mutex protects the cache. Simple, but limits scalability.
- **Sharded.** The cache is partitioned into N shards, each with its own lock. Reduces contention.
- **Lock-free.** Atomic operations update the cache. Higher scalability, but tricky to implement.
- **Per-thread caches.** Each thread has its own cache; no synchronization needed. Periodically merge into a shared cache.

For most engines, sharded caches with 4–16 shards provide a good balance of scalability and simplicity.

### Cache Hit Rate

The cache hit rate is the fraction of accesses that hit the cache. A high hit rate is essential for the cache to be beneficial:

| Hit rate | Effective speedup |
|---|---|
| 0% | 0× (cache is overhead) |
| 50% | 2× |
| 90% | 10× |
| 99% | 100× |
| 99.9% | 1000× |

For a cache to be worth its memory cost, the hit rate should be > 50%. For hot-path caches, aim for > 90%.

### Measuring Cache Performance

Instrument the cache to track:
- **Hits.** Number of accesses that hit the cache.
- **Misses.** Number of accesses that missed.
- **Evictions.** Number of entries evicted.
- **Hit rate.** Hits / (Hits + Misses).
- **Average access time.** (Hits × cache_lookup_time + Misses × computation_time) / (Hits + Misses).

If the hit rate is too low, increase the cache size or improve the replacement policy.

## 6.3.3 Beyond Precomputation and Caching

Precomputation and caching are the two main structural optimizations. Others include:

### Indexing

Build an index over the data to enable fast lookup. Already covered in Chapter 4.

### Algorithm Replacement

Replace a brute-force algorithm with a more efficient one:

- **Brute-force search → alpha-beta.** $O(b^d)$ → $O(b^{d/2})$.
- **Linear scan → hash lookup.** $O(N)$ → $O(1)$.
- **Naive sort → merge sort.** $O(N^2)$ → $O(N \log N)$.
- **Recursive Fibonacci → memoized.** $O(2^N)$ → $O(N)$.

Algorithm replacement is the highest-leverage optimization, but it requires understanding the problem and the available algorithms.

### Data Structure Replacement

Replace a slow data structure with a faster one:

- **Linked list → array.** 10–100× faster for sequential access.
- **Hash map with chaining → hash map with open addressing.** 2–5× faster.
- **Binary search tree → B-tree.** Better cache behavior for large datasets.
- **Sorted array → Bloom filter.** 100× less memory, with false positives.

### Loop Restructuring

Restructure loops for better performance:

- **Loop fusion.** Combine two loops over the same data into one.
- **Loop fission.** Split one loop into two for better register allocation.
- **Loop unrolling.** Process multiple elements per iteration.
- **Loop tiling.** Restructure nested loops for better cache behavior.

Loop restructuring is usually done by the compiler, but manual restructuring can help in specific cases.

### Batch Processing

Replace per-item processing with batch processing:

- **Per-item function call → batch function.** Reduces call overhead.
- **Per-item I/O → batched I/O.** Reduces syscall overhead.
- **Per-item allocation → arena allocation.** Reduces allocation overhead.

## 6.3.4 The Optimization Process

Structural optimization is iterative:

### Step 1: Profile the Engine

Use a profiler (Linux `perf`, Intel VTune, gprof) to identify the hottest functions. Focus on the top 3–5 — these are where optimization effort pays off.

### Step 2: Identify the Bottleneck

For each hot function, identify the bottleneck:
- **Algorithm?** The algorithm is too slow (e.g., $O(N^2)$ where $O(N \log N)$ is possible).
- **Data structure?** The data structure is slow (e.g., linked list where an array would be faster).
- **Memory?** The function is memory-bound (cache misses dominate).
- **Compute?** The function is compute-bound (instruction count dominates).

### Step 3: Apply the Right Optimization

Based on the bottleneck:
- **Algorithm:** replace with a faster algorithm.
- **Data structure:** replace with a faster data structure.
- **Memory:** improve data layout (AoS → SoA, separate hot/cold, prefetch).
- **Compute:** use SIMD, branchless code, or other micro-optimizations.

### Step 4: Verify Correctness

After each optimization, verify the engine still produces correct results. Run the test suite; compare against the unoptimized version.

### Step 5: Measure the Improvement

Measure the engine's performance before and after the optimization. If the improvement is not measurable, the optimization did not work — revert it.

### Step 6: Repeat

Go back to Step 1 and profile again. The bottleneck has likely moved.

## 6.3.5 Common Pitfalls

### Pitfall 1: Optimizing Without Profiling

Optimizing based on intuition is wrong ~80% of the time. Profile first; optimize the bottleneck.

### Pitfall 2: Micro-Optimizing Before Structural

Micro-optimizations (SIMD, branchless code) give 2–5× speedup. Structural optimizations (algorithm replacement, caching) give 10–1000× speedup. Always do structural first.

### Pitfall 3: Over-Caching

Adding a cache that has a low hit rate wastes memory and adds overhead. Measure the hit rate; remove caches that do not earn their cost.

### Pitfall 4: Cache Invalidation Bugs

A cache that returns stale data is worse than no cache. Implement invalidation carefully; test invalidation logic.

### Pitfall 5: Premature Algorithm Replacement

Replacing an algorithm before understanding why it is slow can introduce bugs. Profile to understand the bottleneck; then choose the right replacement.

### Pitfall 6: Not Verifying Correctness

An optimization that breaks correctness is a regression, not an improvement. Always run the test suite after each optimization.

### Pitfall 7: Measuring the Wrong Thing

Measuring throughput when latency matters (or vice versa) leads to wrong conclusions. Know what you are optimizing.

### Pitfall 8: Not Iterating

One optimization often reveals the next bottleneck. Keep profiling and optimizing until the engine meets its performance target.

---

## 6.3.6 Important Reminders

- **Structural optimizations beat micro-optimizations.** 10–1000× vs 2–5×.
- **Profile before optimizing.** Intuition is wrong 80% of the time.
- **Precompute static data.** Lookup tables, attack tables, indexes.
- **Cache dynamic data.** Transposition tables, query result caches.
- **Size caches for > 90% hit rate.** Lower hit rates waste memory.
- **Choose the right replacement policy.** LRU, LFU, W-TinyLFU, depth-preferred.
- **Replace brute-force algorithms.** $O(N^2)$ → $O(N \log N)$, $O(N)$ → $O(\log N)$, etc.
- **Replace slow data structures.** Linked list → array, hash map with chaining → open addressing.
- **Verify correctness after each optimization.** Run the test suite.
- **Measure the improvement.** If it is not measurable, revert.
- **Iterate.** One optimization reveals the next bottleneck.

## 6.3.7 Summary

Phase 3 of the engine lifecycle is structural optimizations over brute force. The two main techniques are precomputation (moving computation to startup time) and caching (storing results of expensive computations for reuse). Other techniques include indexing, algorithm replacement, data structure replacement, loop restructuring, and batch processing.

The optimization process is iterative: profile to identify the bottleneck, apply the right optimization (algorithm, data structure, memory, or compute), verify correctness, measure the improvement, and repeat.

Common pitfalls include optimizing without profiling, micro-optimizing before structural, over-caching, cache invalidation bugs, premature algorithm replacement, not verifying correctness, measuring the wrong thing, and not iterating.

With structural optimizations applied, the engine is fast (10–1000× faster than the brute-force version). Phase 4 (next note) covers micro-optimizations to squeeze out the last 2–5×.
