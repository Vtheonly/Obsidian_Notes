# 4. Phase 4 — Profiling-Driven Micro-Optimization

> "Phase 3 made the engine 100× faster by replacing brute force with structure. Phase 4 squeezes out the last 2–5× by optimizing the hot loop at the cycle level — SIMD, branchless code, cache-line alignment, prefetching. This is the phase where 1% improvements matter, because by now the engine is fast enough that 1% is significant."

This is the fourth note in the engine lifecycle. Phase 1 produced the state, Phase 2 produced a correct $F$, Phase 3 made $F$ structurally fast, and Phase 4 micro-optimizes the hot loop. Phase 5 (next note) will cover event loop orchestration.

---

## 6.4.1 Locating Bottlenecks with Profiling Tools

The cardinal rule of Phase 4: **never optimize without profiling.** Intuition about bottlenecks is wrong ~80% of the time. Profile first; optimize the bottleneck.

### Profiling Tools

**Linux `perf`.** The standard profiler on Linux. Samples the CPU at a configurable rate and attributes samples to functions.

```bash
# Profile a program
$ perf record ./engine

# Report the hot functions
$ perf report

# Get statistics (cache misses, branch misses, IPC)
$ perf stat ./engine
```

`perf stat` output:
```
  1000000000 instructions
   200000000 cache misses
    50000000 branch misses
       2.5 GHz CPU
       3.5 IPC
       5.2 seconds wall time
```

An IPC (instructions per cycle) of 3.5 is good (the CPU is mostly executing, not waiting). An IPC of 0.5 is bad (the CPU is mostly waiting for memory).

**Intel VTune.** A more detailed profiler (commercial). Provides cache analysis, memory access patterns, and hot spots.

**AMD μProf.** AMD's equivalent of VTune.

**macOS Instruments.** The "Counters" instrument provides cache stats on macOS.

**Valgrind Callgrind / Cachegrind.** Simulates the CPU; very slow but very accurate. Useful for cache analysis.

**gprof.** A compiler-assisted profiler. Adds instrumentation to the code; provides function call counts and times.

### What to Profile

Profile the engine under realistic load — the same input it will see in production. Profiling under artificial load gives misleading results.

Specifically, profile:
- **Total runtime.** The bottom-line metric.
- **Hot functions.** Which functions take the most time?
- **Cache behavior.** L1, L2, L3 hit rates; cache misses.
- **Branch behavior.** Branch misprediction rate.
- **Memory allocation.** Number of allocations, total bytes allocated.
- **I/O.** Bytes read/written, syscall count.

### Reading a Profile

A typical profile shows:

```
Samples: 100K
  35.2%  search.c:alpha_beta        # hot function 1
  18.7%  search.c:evaluate          # hot function 2
  10.3%  movegen.c:generate_moves   # hot function 3
   8.1%  hash.c:tt_lookup           # hot function 4
   5.4%  ...
   3.2%  ...
   ...    (long tail)
```

Focus on the top 3–5 functions. These account for ~75% of runtime. Optimizing them gives the biggest speedup. The long tail (functions with < 1% each) is not worth optimizing — even eliminating all of them gives < 25% speedup.

### Profiling Multi-Threaded Engines

For multi-threaded engines, profile each thread separately. Look for:
- **Load imbalance.** Some threads are idle while others are busy.
- **Lock contention.** Threads spend time waiting for locks.
- **Cache coherence traffic.** Threads invalidate each other's cache lines.

`perf` can profile multi-threaded engines; use `perf record -t <thread_id>` to profile a specific thread.

## 6.4.2 Eliminating Speculative Micro-Optimizations

Phase 4 is about **measured** optimization. "Speculative" optimizations — changes made based on intuition without measurement — are the enemy of Phase 4.

### The Speculative Optimization Trap

An engineer thinks: "Function X must be slow because it does Y. Let me optimize it by doing Z." They spend two days implementing Z. They measure. The engine is 0% faster — function X was not the bottleneck.

This is the speculative optimization trap. It wastes time and adds complexity without benefit.

### The Anti-Pattern: "Clever" Code

Speculative optimization often takes the form of "clever" code:
- Bit-twiddling hacks that are not measurably faster.
- Cache-line-aligned structs that are not in the hot path.
- SIMD intrinsics in functions that are not bottlenecks.
- Lock-free data structures where locks were not the bottleneck.

"Clever" code is harder to read, harder to maintain, and harder to debug. It should be added only when measured performance justifies it.

### The Disciplined Approach

The disciplined approach to Phase 4:

1. **Profile.** Identify the top 3–5 hot functions.
2. **Hypothesize.** Form a hypothesis about why each function is slow (cache misses? branch mispredictions? algorithm?).
3. **Test the hypothesis.** Use targeted profiling (cache stats, branch stats) to verify.
4. **Optimize.** Apply the optimization that addresses the verified bottleneck.
5. **Measure.** Verify the optimization improved performance.
6. **Verify correctness.** Run the test suite.
7. **Keep or revert.** If the optimization helped, keep it. If not, revert.

If the optimization did not help, **revert it.** Do not keep "clever" code that does not improve performance.

### Case Study: The Branchless Optimization That Did Not Help

An engineer notices a hot function has a branch: `if (x < threshold) y = a; else y = b;`. They replace it with branchless code: `y = (x < threshold) ? a : b;` (which compiles to a conditional move).

They measure. The function is 1% faster, but the engine is 0% faster — the function was not the bottleneck. The branchless code is harder to read. They revert.

Lesson: micro-optimizations only matter in the hot loop. Outside the hot loop, they are noise.

## 6.4.3 The Micro-Optimization Toolkit

When profiling identifies a bottleneck, apply the appropriate micro-optimization:

### For Memory Bottlenecks

If the hot function is memory-bound (low IPC, high cache miss rate):

- **Improve data layout.** AoS → SoA; pack hot fields; align to cache lines.
- **Prefetch.** Use `__builtin_prefetch` to load data before it is needed.
- **Reduce working set.** Eliminate unnecessary fields; use smaller types.
- **Use cache-friendly data structures.** Arrays over linked lists; flat hash tables over chained hash tables.

### For Branch Bottlenecks

If the hot function has high branch misprediction rate:

- **Make branches predictable.** Reorder code so the common case is the fall-through.
- **Eliminate branches.** Use conditional moves, lookup tables, branchless arithmetic.
- **Use SIMD.** SIMD comparisons are branchless.

### For Compute Bottlenecks

If the hot function is compute-bound (high IPC, low cache miss rate):

- **Use SIMD.** AVX2, AVX-512, NEON. Process 4–16 elements per instruction.
- **Reduce instruction count.** Replace complex operations with simpler ones (e.g., multiply by 2 → shift left by 1).
- **Inline the function.** Eliminate call overhead.
- **Use compiler optimizations.** `-O3`, `-march=native`, `-funroll-loops`.

### For Allocation Bottlenecks

If the hot function spends time in `malloc`/`free`:

- **Pre-allocate.** Allocate all memory at startup.
- **Use arena/pool allocators.** O(1) allocation; no per-allocation free.
- **Avoid `std::string` and `std::vector` in hot paths.** Use flat arrays with known sizes.
- **Use a fast allocator.** jemalloc, tcmalloc, mimalloc.

### For Lock Bottlenecks

If the hot function spends time waiting for locks:

- **Reduce critical sections.** Move work outside the lock.
- **Use sharded locks.** Partition the data; each shard has its own lock.
- **Use lock-free data structures.** Atomic operations, RCU, hazard pointers.
- **Use per-thread data.** Each thread has its own copy; no synchronization needed.

## 6.4.4 Verification and Benchmarking

Every micro-optimization must be verified to actually improve performance. The verification process:

### Step 1: Establish a Baseline

Before the optimization, measure the engine's performance under realistic load. Use a representative benchmark (e.g., a chess engine's benchmark suite, a search engine's query log).

Record:
- Total runtime.
- Hot function runtimes (from profiler).
- Cache hit rates.
- Branch misprediction rates.
- IPC.

### Step 2: Apply the Optimization

Make the change. Keep the diff small — one optimization at a time. If you make multiple changes, you cannot tell which one helped.

### Step 3: Re-Measure

Run the same benchmark. Record the same metrics.

### Step 4: Compare

Compare before and after:
- Did total runtime improve?
- Did the hot function's runtime improve?
- Did cache hit rates improve?
- Did branch misprediction rates improve?

### Step 5: Verify Correctness

Run the test suite. The engine must produce the same output (or statistically equivalent output, for non-deterministic engines).

### Step 6: Keep or Revert

If the optimization improved performance and did not break correctness, keep it. Otherwise, revert.

### Benchmarking Best Practices

- **Run multiple times.** Single runs are noisy. Run the benchmark 5–10 times and take the median.
- **Use a realistic benchmark.** A benchmark that does not represent production load gives misleading results.
- **Pin the CPU frequency.** Modern CPUs scale frequency dynamically; this adds noise. Pin the frequency (e.g., `cpufreq-set -f 3GHz`) for consistent measurements.
- **Isolate the machine.** Other processes on the machine add noise. Run benchmarks on a dedicated machine.
- **Measure with the right tool.** Use `perf stat` for cache/branch stats, `time` or `perf record` for runtime. Do not trust the engine's internal timers — they may not include I/O or kernel time.

## 6.4.5 Common Pitfalls

### Pitfall 1: Optimizing Without Profiling

The cardinal sin. Profile first; optimize the bottleneck.

### Pitfall 2: Optimizing the Wrong Function

The top function in the profile is the bottleneck. The 10th function is not. Focus on the top.

### Pitfall 3: Keeping "Clever" Code That Does Not Help

If an optimization does not measurably improve performance, revert it. "Clever" code that does not help is technical debt.

### Pitfall 4: Not Verifying Correctness

An optimization that breaks correctness is a regression, not an improvement. Run the test suite.

### Pitfall 5: Making Multiple Changes at Once

If you make multiple changes and the engine gets faster, you do not know which change helped. Make one change at a time.

### Pitfall 6: Noisy Measurements

Single runs are noisy. Run multiple times and take the median. Pin CPU frequency.

### Pitfall 7: Wrong Benchmark

A benchmark that does not represent production load gives misleading results. Use a realistic benchmark.

### Pitfall 8: Over-Optimizing

Phase 4 gives 2–5× speedup. After that, further optimization gives diminishing returns. Know when to stop.

### Pitfall 9: Not Considering Maintainability

Micro-optimized code is harder to read and maintain. Weigh the performance gain against the maintenance cost.

### Pitfall 10: Ignoring the Compiler

The compiler is good at micro-optimization. Let it do its job with `-O3` and `-march=native`. Manual micro-optimization is needed only for cases the compiler cannot handle.

## 6.4.6 Important Reminders

- **Never optimize without profiling.** Intuition is wrong 80% of the time.
- **Focus on the top 3–5 hot functions.** The long tail is not worth optimizing.
- **One change at a time.** Otherwise you cannot tell what helped.
- **Measure before and after.** Use `perf stat`, `perf record`, VTune.
- **Verify correctness.** Run the test suite.
- **Revert optimizations that do not help.** "Clever" code that does not measurably improve is technical debt.
- **Match the optimization to the bottleneck.** Memory, branch, compute, allocation, or lock.
- **Use the compiler.** `-O3`, `-march=native`, `-funroll-loops`. Let it do its job.
- **Run multiple times.** Take the median. Pin CPU frequency.
- **Use a realistic benchmark.** Production-like load gives meaningful results.
- **Know when to stop.** Phase 4 gives 2–5×. Beyond that, diminishing returns.

## 6.4.7 Summary

Phase 4 of the engine lifecycle is profiling-driven micro-optimization. The cardinal rule: never optimize without profiling. Use `perf`, VTune, or similar tools to identify the top 3–5 hot functions, then apply the appropriate micro-optimization (memory, branch, compute, allocation, or lock) based on the verified bottleneck.

The disciplined approach: profile, hypothesize, test the hypothesis, optimize, measure, verify correctness, keep or revert. Never keep "clever" code that does not measurably improve performance.

Common pitfalls include optimizing without profiling, optimizing the wrong function, keeping code that does not help, not verifying correctness, making multiple changes at once, noisy measurements, wrong benchmarks, over-optimizing, ignoring maintainability, and ignoring the compiler.

With Phase 4 complete, the engine is as fast as it can reasonably be. Phase 5 (next note) covers event loop orchestration — wrapping the engine in a runtime that handles external events.
