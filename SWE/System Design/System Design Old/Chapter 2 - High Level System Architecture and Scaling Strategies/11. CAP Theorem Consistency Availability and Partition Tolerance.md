# CAP Theorem: Consistency, Availability, and Partition Tolerance

## Overview

The CAP theorem, also known as Brewer's theorem after computer scientist Eric Brewer, is one of the most influential principles in distributed systems design. It states that a distributed data store can provide at most two of three guarantees simultaneously: Consistency, Availability, and Partition Tolerance. Because network partitions are inevitable in any real distributed system, the practical choice is between consistency and availability when a partition occurs. Understanding the CAP theorem is essential for making informed architectural decisions, because it forces engineers to explicitly acknowledge the tradeoffs inherent in distributed systems and choose the properties that matter most for their specific use case.

## Key Concepts

### Consistency

In the context of CAP, consistency means that all nodes in the distributed system see the same data at the same time. When a write operation is acknowledged, any subsequent read operation (from any node) should return the most recently written value. This is equivalent to linearizability or strong consistency. If one user updates their profile picture on one server, every other server in the system must reflect that update before the write is considered complete. Achieving strong consistency typically requires coordination between nodes (such as consensus protocols like Paxos or Raft), which adds latency and reduces availability during network partitions. Relational databases with synchronous replication provide strong consistency guarantees.

### Availability

Availability means that every request received by a non-failing node in the system results in a response (not an error). The system continues to operate and respond to requests even when some nodes are unavailable. In a highly available system, there is no single point of failure, and the system can continue serving requests even if individual components fail. Achieving high availability typically requires replication: multiple copies of data exist across different nodes, so if one node fails, others can serve requests. However, if those replicas are not kept in sync (which would require coordination and reduce availability during partitions), they may return stale or inconsistent data.

### Partition Tolerance

Partition tolerance means the system continues to operate despite network partitions. A network partition occurs when communication between nodes is lost or delayed, preventing them from coordinating. In any real distributed system, network partitions are inevitable: cables get cut, routers fail, configuration errors isolate parts of the network, and congestion causes timeouts. Because partitions cannot be prevented, partition tolerance is not optional for distributed systems; it is a fact of life. The CAP theorem therefore reduces to a binary choice: when a partition occurs, the system must choose between consistency and availability. It cannot provide both.

### CP vs AP vs CA Systems

A CP system (Consistency and Partition Tolerance) prioritizes consistency over availability. When a partition occurs, the system refuses to serve requests that might return inconsistent data, effectively becoming unavailable for some operations. Examples include MongoDB with strong consistency settings, HBase, and Redis Cluster. An AP system (Availability and Partition Tolerance) prioritizes availability over consistency. When a partition occurs, the system continues serving requests from all nodes, even if some nodes return stale data. Examples include Cassandra, DynamoDB in eventual consistency mode, and CouchDB. A CA system (Consistency and Availability) provides both consistency and availability but only when there are no partitions. Since partitions are inevitable in distributed systems, CA systems effectively assume that partitions do not happen, which is only realistic for single-node systems. Strictly speaking, CA is not a meaningful category for distributed systems.

### PACELC Extension

The PACELC theorem extends CAP by considering the tradeoff even when there is no partition. The acronym stands for: If there is a Partition (P), the system chooses between Availability (A) and Consistency (C); Else (E), when running normally, the system chooses between Latency (L) and Consistency (C). This extension acknowledges that even in the absence of partitions, there is a tradeoff between latency and consistency. A system can respond quickly (low latency) by serving data from the nearest replica, but this may return stale data (lower consistency). Alternatively, a system can ensure strong consistency by coordinating with all replicas before responding, but this increases latency. PACELC provides a more nuanced framework for understanding the tradeoffs in distributed systems design.

## Practical Applications

When designing a banking system, consistency is paramount: a user's account balance must always be accurate, even if it means the system is temporarily unavailable during a network partition. This makes a CP architecture the appropriate choice. In contrast, a social media feed prioritizes availability: it is better to show slightly stale posts than to show an error page, making an AP architecture more suitable. A product catalog on an e-commerce site might use eventual consistency (AP) for browsing products but strong consistency (CP) for inventory and checkout. Many modern databases allow configurable consistency levels per operation, enabling applications to make different tradeoffs for different use cases within the same system.
