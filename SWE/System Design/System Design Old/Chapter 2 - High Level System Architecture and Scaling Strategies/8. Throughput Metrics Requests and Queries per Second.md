# Throughput Metrics: Requests and Queries per Second

## Overview

Throughput measures how much work a system can accomplish in a given period of time. In the context of web services, throughput is typically expressed as Requests Per Second (RPS) for servers or Queries Per Second (QPS) for databases. These metrics are fundamental to capacity planning, performance optimization, and cost estimation. Understanding how to measure, interpret, and optimize throughput is essential for system designers because it directly determines how much infrastructure is needed to handle a given load and how much that infrastructure will cost. At extreme scales, such as handling a million requests per second, even small improvements in throughput can translate to significant cost savings or the difference between viable and unviable architecture.

## Key Concepts

### RPS and QPS Definitions

Requests Per Second (RPS) measures the number of HTTP requests a server can handle in one second. This includes all request types: GET, POST, PUT, DELETE, and others. A higher RPS value indicates better server performance and the ability to handle more concurrent users. Queries Per Second (QPS) measures the number of database queries a database server can process in one second. Like RPS, a higher QPS value signifies better database performance. Both metrics are measured over a specific time window (typically 1 second) and are averaged over longer periods to smooth out variability. In practice, RPS and QPS are often used interchangeably, but it is important to be precise about whether you are measuring server throughput or database throughput.

### How to Measure Throughput

Throughput is typically measured using load testing tools that simulate concurrent user traffic. Tools like Apache JMeter, Locust, k6, and autocannon can generate thousands or millions of requests per second against a target server and report the achieved throughput. The measurement process involves sending a sustained load for a defined duration (for example, 20 seconds) with a specified number of concurrent connections, and then reporting the average throughput over that period. It is important to measure throughput under realistic conditions: with the actual data sizes, request patterns, and backend dependencies that the production system will encounter. A simple endpoint that returns "hello" might achieve 70,000 RPS, while a realistic API endpoint that queries a database and returns 32 KB of data might achieve only 8,000 RPS on the same hardware.

### Factors Affecting Throughput

Many factors influence throughput. CPU capacity is often the primary bottleneck: a single-threaded application can utilize only one CPU core, so adding more cores requires multi-threading or multi-processing. Network bandwidth limits how much data can be transferred per second; if each response is large, the network card may become saturated before the CPU is fully utilized. Database performance limits how quickly the application can retrieve and store data; if every request requires a database query, the database often becomes the bottleneck before the application server. Memory availability affects how much data can be cached in RAM, reducing the number of slow disk or network operations. I/O bandwidth determines how quickly data can be read from and written to storage.

### Throughput vs Latency Tradeoffs

Throughput and latency are inversely related in many scenarios. Batching multiple operations together increases throughput by amortizing fixed overheads, but it increases the latency of individual operations because each operation must wait for the batch to fill. Increasing the number of concurrent connections can increase throughput up to a point, but beyond that point, contention for shared resources (CPU, memory, database connections) increases latency for all requests. This tradeoff is captured by Little's Law, which states that the average number of requests in the system (L) equals the average arrival rate (throughput, lambda) multiplied by the average time each request spends in the system (latency, W): L = lambda * W. Understanding this relationship helps engineers find the optimal operating point where throughput is maximized without unacceptable latency increases.

### Data Throughput

In addition to request throughput, data throughput measures the volume of data transferred per second, typically expressed in bytes per second or megabytes per second. This metric is important for understanding network utilization and storage I/O requirements. A server handling 10,000 RPS with an average response size of 32 KB moves 320 MB of data per second, which requires a network link of at least 2.56 Gbps (320 MB/s * 8 bits/byte). At the scale of a million requests per second with similar response sizes, the data throughput reaches 32 GB/s, requiring multiple high-speed network links. Understanding data throughput helps engineers provision network infrastructure and identify when network bandwidth, rather than CPU or memory, becomes the limiting factor.

## Practical Applications

When benchmarking a new API, engineers typically start with a single instance and measure RPS under various configurations (single-threaded vs multi-threaded, with and without caching, different response sizes). The benchmark results inform capacity planning: if one instance handles 40,000 RPS and the target load is 1,000,000 RPS, approximately 25 instances are needed, plus additional capacity for traffic spikes and failover. Understanding the throughput characteristics of each endpoint also helps prioritize optimization efforts: endpoints that consume disproportionate CPU or database resources are candidates for caching, query optimization, or architectural changes that reduce their per-request cost.
