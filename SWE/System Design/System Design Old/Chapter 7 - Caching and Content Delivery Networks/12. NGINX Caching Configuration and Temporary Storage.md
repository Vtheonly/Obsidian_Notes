# NGINX Caching Configuration and Temporary Storage

## Overview
NGINX is one of the most widely used web servers and reverse proxies in production systems, and it includes powerful built-in caching capabilities. When configured as a reverse proxy, NGINX can cache responses from upstream servers and serve subsequent requests directly from the cache without contacting the backend. This dramatically reduces backend load and improves response times for end users. NGINX caching is configured through a set of directives in the nginx.conf file, which control where cached content is stored, how long it is kept, and which responses are eligible for caching. Additionally, NGINX uses temporary storage for request bodies and responses that are too large to hold in memory, which must be properly configured to avoid disk space issues and performance degradation. Understanding NGINX caching configuration is essential for engineers who deploy applications behind NGINX in production.

## Key Concepts

### Proxy Cache Configuration
The core of NGINX caching is the `proxy_cache` directive and its associated settings. To enable caching, you must first define a cache zone using the `proxy_cache_path` directive at the HTTP level. This directive specifies the directory where cached files are stored, the shared memory zone name and size for storing cache metadata (keys), and various parameters controlling cache behavior. For example: `proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m max_size=10g inactive=60m use_temp_path=off;` creates a cache zone named `my_cache` with 10 MB of shared memory for keys, a maximum disk size of 10 GB, and inactive entries expired after 60 minutes. Within a location block, the `proxy_cache my_cache;` directive activates caching for that location using the specified zone.

### Cache Key andproxy_cache_key
NGINX uses a cache key to determine whether a request matches a cached response. The default cache key is `$scheme$proxy_host$request_uri`, which includes the protocol, the upstream host, and the full request URI. For more control, you can customize the cache key using the `proxy_cache_key` directive. For example, `proxy_cache_key $scheme$host$request_uri` would vary the cache by the Host header rather than the upstream server, which is useful when multiple upstream servers serve identical content. You can also include additional variables such as `$cookie_user` to cache personalized content per user, though this must be done carefully to avoid cache pollution.

### Cache Validity and TTL
NGINX determines how long to cache a response based on several factors. The `proxy_cache_valid` directive sets the default caching duration for different HTTP status codes. For example, `proxy_cache_valid 200 302 10m;` caches 200 and 302 responses for 10 minutes, while `proxy_cache_valid 404 1m;` caches 404 responses for only 1 minute. NGINX also respects the `Cache-Control`, `Expires`, and `X-Accel-Expires` headers from the upstream server. The `proxy_ignore_headers` directive can be used to ignore specific headers (such as `Set-Cookie` or `Cache-Control: private`) that would otherwise prevent caching. The interaction between these configuration directives and upstream headers requires careful testing to ensure the desired caching behavior.

### Temporary Storage andproxy_temp_path
When NGINX proxies a request, it may need to temporarily store the request body or the response on disk if they are too large to hold in memory. The `proxy_temp_path` directive specifies the directory where these temporary files are stored. By default, this is a subdirectory of the NGINX installation prefix. For production systems, it is important to ensure that the temporary storage directory has sufficient disk space and is located on a fast disk (preferably SSD). The `proxy_temp_file_write_size` directive controls the size of data chunks written to temporary files. Setting `use_temp_path=off` in the `proxy_cache_path` directive causes temporary files to be written directly in the cache directory, avoiding an extra copy operation between the temp directory and the cache directory.

### Cache Bypass and Stale Content
NGINX provides several directives for handling cache bypass and stale content. The `proxy_cache_bypass` directive defines conditions under which the cache is skipped and the request is sent directly to the upstream server. This is commonly used to bypass the cache for authenticated users or for specific query parameters. The `proxy_no_cache` directive prevents the response from being saved to the cache even if the request is sent to the upstream. For stale content, `proxy_cache_use_stale` specifies when NGINX can serve stale cached responses, such as when the upstream server is unavailable (`proxy_cache_use_stale error timeout updating`) or while a fresh response is being fetched in the background (`proxy_cache_use_stale updating`). This provides graceful degradation during backend failures.

### Cache Locking and Race Conditions
When multiple concurrent requests arrive for the same uncached resource, NGINX can send all of them to the upstream server, causing unnecessary load. The `proxy_cache_lock` directive addresses this by allowing only one request at a time to fetch the resource from the upstream while other requests wait for the cache to be populated. The `proxy_cache_lock_timeout` directive sets a maximum wait time; if the lock is not released within this time, the waiting requests are sent to the upstream anyway. This prevents a single slow upstream response from blocking all concurrent requests indefinitely. Cache locking is especially important during cache warmup or after a cache purge, when many users may request the same uncached content simultaneously.

### Monitoring NGINX Cache
NGINX provides cache status information through variables like `$upstream_cache_status`, which can be logged or included in response headers for monitoring. Possible values are HIT (served from cache), MISS (not in cache, fetched from upstream), EXPIRED (cache entry expired, revalidated with upstream), STALE (served stale content due to upstream error), UPDATING (served stale content while fresh content is being fetched), and BYPASSED (cache was bypassed). Monitoring these values helps identify caching issues and tune the configuration for optimal hit rates.

## Practical Applications
NGINX caching is used extensively in production web architectures. A typical deployment places NGINX as a reverse proxy in front of application servers, caching API responses and static content. Media streaming platforms use NGINX to cache video segments at the proxy layer, reducing the load on origin servers during peak viewing hours. E-commerce sites use NGINX caching for product pages and category listings, with cache bypass for logged-in users who see personalized pricing. Content management systems use NGINX to cache rendered pages and invalidate them through purge requests when content is updated. In each case, the careful configuration of cache zones, TTLs, and stale-content handling ensures that users receive fast responses while the backend remains protected from overload.
