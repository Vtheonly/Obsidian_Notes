# Serving Static Assets versus Dynamic Content Routing

## Overview
A critical architectural decision in system design is how to serve static assets (such as images, CSS, JavaScript, and fonts) versus dynamic content (such as personalized pages, API responses, and real-time data). Static assets change infrequently and are identical for all users, making them ideal candidates for aggressive caching and CDN distribution. Dynamic content varies per user or per request and often cannot be cached at all, or only with short TTLs and careful invalidation strategies. Effective systems separate these two types of content at the infrastructure level, routing static assets through CDNs and caching layers while sending dynamic requests to application servers that can compute personalized responses. Understanding how to route and serve each type of content efficiently is essential for building performant and scalable systems.

## Key Concepts

### Static Assets
Static assets are files that do not change based on the user or the request context. Examples include CSS stylesheets, JavaScript bundles, images, fonts, and downloadable files. These assets are typically stored on disk or in object storage (such as Amazon S3) and served directly by a web server or CDN without any server-side processing. Because static assets are identical for all users and change infrequently, they benefit enormously from long TTL caching and CDN distribution. A single cached copy at a CDN edge server can serve millions of users, reducing the load on the origin server to near zero for these requests.

### Dynamic Content
Dynamic content is generated in real time based on the user's identity, request parameters, or current application state. Examples include personalized dashboards, search results, shopping cart contents, and real-time notifications. Dynamic content typically requires server-side computation, database queries, and business logic. It cannot be cached in the same way as static assets because the response varies per user or per request. However, some dynamic content can be partially cached: for example, the common header and footer of a personalized page might be cached while the user-specific content is generated fresh. This technique, known as fragment caching or edge-side includes (ESI), allows systems to combine the performance of cached content with the personalization of dynamic content.

### Content Routing Strategies
Effective content routing directs each type of request to the infrastructure best equipped to handle it. A common pattern is to use a CDN or reverse proxy (such as NGINX) to serve static assets directly from cache or object storage, while forwarding dynamic requests to the application server tier. This can be configured using URL path-based routing: requests for `/static/*` or `/assets/*` are served from the CDN, while requests for `/api/*` or `/dashboard/*` are proxied to the application servers. Domain-based routing is another approach: static assets are served from a separate subdomain (e.g., `static.example.com`) that points to the CDN, while the main domain routes to the application servers.

### Cache Invalidation for Static Assets
When static assets are updated, the CDN and browser caches may still serve the old version until the TTL expires. The industry-standard solution is content versioning through URL fingerprinting. Each static asset includes a hash of its content in its filename or URL path (e.g., `app.3a7b2c1d.js`). When the content changes, the hash changes, producing a new URL. Because the URL is different, the CDN and browser treat it as a new resource and fetch the updated version. The old URL continues to serve the old version for any cached references, but the new HTML that references the new URL will always get the latest content. This eliminates the need for cache purging and allows static assets to use extremely long TTLs.

### Edge Computing and Dynamic Content at the Edge
Modern CDNs increasingly support edge computing capabilities, which allow dynamic logic to be executed at the edge server rather than at the origin. Platforms like Cloudflare Workers, AWS Lambda@Edge, and Fastly Compute@Edge enable developers to run JavaScript or WebAssembly code at CDN edge locations. This enables personalization, A/B testing, geolocation-based content, and authentication at the edge, reducing the need to route dynamic requests all the way to the origin. While edge computing is not suitable for all dynamic content (especially content that requires complex database queries), it can handle many lightweight personalization tasks with much lower latency than origin-based processing.

### Performance Optimization Techniques
Several techniques optimize the delivery of both static and dynamic content. For static assets, minification (removing whitespace and comments from CSS and JS), compression (gzip or Brotli), and image optimization (WebP format, responsive images) reduce the size of transferred data. HTTP/2 and HTTP/3 enable multiplexed connections, allowing the browser to download many assets in parallel over a single connection. For dynamic content, server-side caching of expensive computations, database query caching, and pre-computed materialized views can reduce the latency of dynamic responses. Combining these techniques with effective content routing ensures that each type of content is delivered with optimal performance.

## Practical Applications
Large-scale web applications rely heavily on the separation of static and dynamic content routing. Spotify serves its JavaScript application bundles and album artwork through a CDN with long TTLs and fingerprinted URLs, while API requests for playlists and playback state are routed to application servers. Amazon serves product images and CSS from a CDN on a separate static domain, while dynamic content like pricing, recommendations, and cart state is generated by application servers. News websites use CDNs to serve article text and images (which are relatively static after publication) while handling comments and personalization through API calls to application servers. In each case, the separation allows the system to scale each tier independently based on its specific load characteristics.
