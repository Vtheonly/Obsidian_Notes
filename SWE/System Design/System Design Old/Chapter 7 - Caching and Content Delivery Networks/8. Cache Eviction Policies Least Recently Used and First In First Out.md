# Cache Eviction Policies Least Recently Used and First In First Out

## Overview
Cache eviction policies determine which data should be removed from the cache when it reaches its capacity limit. Since caches have finite memory, they cannot store all possible data indefinitely. When the cache is full and a new entry needs to be stored, an existing entry must be evicted to make room. The choice of eviction policy has a significant impact on cache hit rates and overall system performance. A good eviction policy keeps frequently accessed data in the cache while removing data that is unlikely to be needed again. The most common eviction policies are Least Recently Used (LRU), First In First Out (FIFO), and Least Frequently Used (LFU), each with different tradeoffs in terms of implementation complexity, memory overhead, and effectiveness for different access patterns.

## Key Concepts

### Least Recently Used (LRU)
LRU is the most widely used cache eviction policy. It evicts the entry that has not been accessed for the longest time. The rationale is that data that has not been used recently is less likely to be used in the near future, a principle known as temporal locality. LRU is implemented using a doubly-linked list combined with a hash map: every time an entry is accessed, it is moved to the head of the list, and the least recently used entry is always at the tail. When the cache is full, the entry at the tail is evicted. LRU provides excellent hit rates for most workloads because it naturally adapts to changes in access patterns. However, it has a weakness: a single scan through a large dataset can evict all the useful entries from the cache, a phenomenon known as cache pollution.

### First In First Out (FIFO)
FIFO is the simplest cache eviction policy. It evicts the entry that has been in the cache the longest, regardless of how frequently or recently it has been accessed. FIFO is easy to implement using a queue: new entries are added to the tail, and evictions remove entries from the head. The main advantage of FIFO is its low overhead -- it requires no metadata updates on cache hits, unlike LRU which must move entries in the linked list. The disadvantage is that FIFO does not consider access patterns at all. A frequently accessed entry that was inserted early can be evicted simply because it has been in the cache the longest, leading to poor hit rates for workloads with skewed access distributions.

### Least Frequently Used (LFU)
LFU evicts the entry with the lowest access frequency. Each cache entry maintains a counter that is incremented every time the entry is accessed, and the entry with the lowest count is evicted when the cache is full. LFU is effective for workloads with stable access patterns where some items are consistently more popular than others. However, LFU has a significant weakness: it does not adapt well to changes in access patterns. An item that was accessed many times in the past but is no longer needed will remain in the cache simply because of its high historical count, a problem known as the decay problem. To address this, some implementations use a windowed LFU that only counts recent accesses or a decaying counter that gradually reduces older counts.

### LRU Variants
Several variants of LRU have been developed to address its weaknesses. Approximate LRU (used by Redis) samples a small number of entries and evicts the least recently used among them, which is much more efficient than maintaining a precise ordering of all entries. Segmented LRU (SLRU) divides the cache into a probationary segment and a protected segment: new entries go into the probationary segment, and entries that are accessed again are promoted to the protected segment, which is not subject to eviction until the probationary segment is exhausted. Another variant is LRU-K, which considers the time of the K-th most recent access rather than just the most recent one, providing better resistance to scan-based cache pollution.

### Choosing the Right Eviction Policy
The choice of eviction policy depends on the workload characteristics and consistency requirements. For general-purpose caching with mixed access patterns, LRU or its approximations are usually the best choice. For workloads with very stable access patterns and clear hot/cold data separation, LFU may provide higher hit rates. FIFO is suitable when simplicity and low overhead are more important than optimal hit rates, or when the data has natural time-based expiration. In practice, many production systems use LRU as the default and switch to other policies only when profiling reveals specific access pattern characteristics that would benefit from a different approach.

### Monitoring Eviction Effectiveness
Regardless of the eviction policy chosen, it is essential to monitor cache hit rates, miss rates, and eviction counts. A high eviction rate relative to the total number of operations suggests that the cache is too small for the working set. A declining hit rate over time may indicate a shift in access patterns that the current eviction policy handles poorly. Tools like Redis INFO and Memcached stats provide detailed metrics on evictions, hit rates, and memory usage that can be used to tune the cache configuration and eviction policy.

## Practical Applications
LRU is the default eviction policy in Redis and is used extensively in web applications for session caching, API response caching, and object caching. CDNs like Cloudflare and Akamai use LRU-based eviction to manage their edge caches, evicting content that has not been recently requested to make room for new content. Operating systems use LRU variants for page cache management, evicting memory pages that have not been recently accessed. Web browsers use LRU for their disk and memory caches, removing the oldest unused resources when the cache reaches its size limit. In each case, the LRU policy effectively retains the most useful data while making room for new entries.
