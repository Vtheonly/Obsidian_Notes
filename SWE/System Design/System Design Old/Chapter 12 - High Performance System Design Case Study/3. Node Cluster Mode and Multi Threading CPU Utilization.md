# Node Cluster Mode and Multi Threading CPU Utilization

## Overview

Node.js runs JavaScript in a single thread by default, which means a single Node.js process can only utilize one CPU core at a time. On a machine with multiple cores -- which is virtually every modern server -- this leaves the majority of the CPU idle. Node's cluster module and process managers like PM2 address this limitation by spawning multiple instances of the application, each running in its own process with its own event loop. The master process distributes incoming connections across the worker processes, enabling the application to utilize all available CPU cores. Understanding how cluster mode works, how CPU utilization is measured, and how to optimize for maximum throughput is essential for building high-performance Node.js applications that can approach the limits of their hardware.

## Key Concepts

### Single-Threaded Execution

Node.js executes JavaScript in a single thread using an event-driven, non-blocking I/O model. While this model is excellent for I/O-bound workloads (handling many concurrent connections with minimal overhead), it means that a single Node.js process can only execute one piece of JavaScript code at a time. CPU-intensive operations -- such as parsing JSON, computing hashes, or processing arrays -- block the event loop, preventing the process from handling other requests until the computation completes. When benchmarking a single Node.js process running a CPU-intensive route, the process hits 100% CPU utilization on one core while the remaining cores sit idle. This is why a single instance on a 12-core machine might handle only 8,000 RPS while the machine has plenty of unused CPU capacity.

### Core and CPU Utilization Formulas

Core utilization measures how busy a single CPU core is: `(total_time - idle_time) / total_time * 100`. If a core was idle for 30 minutes out of the last hour, its utilization is 50%. CPU utilization aggregates core utilization across all cores. There are two methods: Method 1 adds all core utilizations together (so four fully utilized cores = 400%), while Method 2 divides by the total number of cores (so four fully utilized cores = 100%). Both methods are valid, but it is essential to know which one your monitoring tools use. In Activity Monitor on macOS, for example, the CPU percentage for a process uses Method 1, so a process utilizing all 12 cores reports approximately 1200% CPU usage.

### The Cluster Module and PM2

Node.js's built-in `cluster` module allows you to create a master process that forks multiple worker processes, each running its own instance of the application. The master process listens on the shared port and distributes incoming connections to the workers using round-robin scheduling (by default on most platforms). PM2 simplifies this by providing a process manager that automatically spawns the optimal number of worker processes based on the number of CPU cores. With `pm2 start ecosystem.config.js`, you can configure PM2 to run as many instances as there are cores (or a specific number). PM2 also handles process monitoring, automatic restarts on crashes, and zero-downtime reloads, making it the de facto standard for running Node.js applications in production.

### From Single Process to Cluster Mode

The performance impact of cluster mode is dramatic. In the benchmark scenario, a single Node.js process handling a complex route achieved approximately 8,000 RPS on a 12-core Mac Studio, with one core at 100% and the others mostly idle. Switching to cluster mode with PM2 running 12 instances increased throughput to approximately 36,000 RPS, with idle CPU dropping to around 30%. Further increasing the load by raising the number of AutoCannon worker threads pushed throughput to approximately 42,000-50,000 RPS with idle CPU at 0%, meaning all cores were fully utilized. This 5-6x improvement comes without any changes to the application code -- simply by utilizing more of the available hardware.

### Limits of Cluster Mode

Cluster mode is not a silver bullet. Each worker process has its own memory space, so the total memory consumption is multiplied by the number of workers. On a machine with 256 GB of RAM and 128 cores, running 128 instances of a Node.js process that uses 500 MB each would consume 64 GB of RAM just for the application, which is manageable. However, shared resources such as database connection pools, cache connections, and file handles are also multiplied, potentially overwhelming backing services. Additionally, inter-process communication between workers is limited, so coordination mechanisms (like shared state or distributed locks) must be handled through external systems like Redis. Finally, the master process itself can become a bottleneck if it cannot distribute connections fast enough, though this is rarely an issue in practice.

## Practical Applications

On the AWS C8i.32xlarge instance with 128 CPU cores, the application was run with PM2 in cluster mode with 128 instances of Node.js. This allowed the server to utilize all 128 cores, with each instance handling its share of incoming connections. When combined with AutoCannon generating traffic from 120 worker threads with 1000 connections and pipelining of 100, the server achieved over 6 million RPS for the simple route. The transition from a single-process, single-core setup to a multi-process, multi-core setup was the single most impactful optimization in the entire scaling journey, transforming a machine that could handle 8,000 RPS into one that could handle over 6 million RPS for the simplest route.
