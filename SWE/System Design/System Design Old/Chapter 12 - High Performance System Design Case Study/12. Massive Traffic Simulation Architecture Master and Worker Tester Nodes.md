# Massive Traffic Simulation Architecture Master and Worker Tester Nodes

## Overview

Simulating one million requests per second requires a traffic generation architecture that is itself a distributed system. A single client machine, no matter how powerful, may not be able to generate sufficient traffic to saturate a high-performance server cluster. The solution is a master-worker architecture where a master node coordinates multiple worker nodes, each generating a portion of the total traffic. This distributed approach enables the tester to scale horizontally, adding more worker nodes until the target traffic level is reached. Understanding how to design, deploy, and operate a massive traffic simulation architecture is essential for validating that a system can handle its expected peak load before it faces real traffic. The architecture must be carefully designed to ensure that the tester itself does not become the bottleneck, producing misleading results that understate the system's true capacity.

## Key Concepts

### Single Machine Limitations

In the initial benchmarking phase, both the server and the traffic generator (AutoCannon) ran on the same Mac Studio. This configuration is convenient for development but fundamentally limited because the client and server compete for the same CPU, memory, and network resources. The server's throughput was constrained not by its own capacity but by the client's ability to generate sufficient load. Even on a dedicated machine, a single AutoCannon instance running on a 12-core machine can only utilize 12 CPU cores for traffic generation, which limits the number of concurrent connections and the rate at which requests can be sent. For a server with 128 cores, a single 12-core client is woefully inadequate to generate enough traffic to saturate the server.

### Dedicated Tester Architecture

The first architectural improvement is to separate the tester from the server onto dedicated machines. In the benchmark scenario, two C8i.32xlarge instances were launched: one as the "power server" running the application with PM2 in cluster mode (128 Node.js instances), and one as the "power tester" running AutoCannon with 120 worker threads, 1000 connections, and pipelining of 100. This separation ensures that the server's resources are dedicated to handling requests, not generating them. The tester instance with 128 CPU cores has ample capacity to generate massive traffic -- at 120 worker threads, it utilizes most of its cores for traffic generation while leaving some headroom for the AutoCannon coordination process. The two instances communicate over AWS's private network, eliminating internet latency and variability from the benchmark.

### Master-Worker Coordination

For traffic simulations that exceed the capacity of a single tester machine, a master-worker architecture is required. The master node defines the test parameters (target URL, request rate, duration, request payload) and distributes these parameters to the worker nodes. Each worker node generates a portion of the total traffic, typically an equal share. The master node monitors the workers, collects aggregate results, and reports the combined throughput and latency statistics. This architecture can scale to any traffic level by adding more worker nodes. The master node's responsibility is coordination, not traffic generation, so it requires relatively few resources. The worker nodes are the ones generating traffic and should be sized with sufficient CPU cores and network bandwidth.

### Network Architecture for Testing

The network architecture of the traffic simulation is critical for producing valid results. The tester nodes and the server nodes should be on the same private network (VPC) to eliminate internet latency and variability. In the benchmark scenario, the power server, power tester, and database were all on the same AWS private network in the Ohio region, connected through their network interfaces. This ensured that the benchmark measured the server's application-level performance, not the vagaries of internet routing. For simulations that need to test geographic distribution, multiple tester nodes can be deployed in different regions, with the master node aggregating results across regions. However, for pure throughput benchmarking, a single-region deployment on a private network provides the most reliable and reproducible results.

### Monitoring During Traffic Simulation

During a massive traffic simulation, continuous monitoring of both the server and the tester is essential. On the server side, `mpstat` provides real-time CPU utilization, `free` shows memory usage, and `iostat` shows disk I/O. On the tester side, monitoring ensures that the tester is not the bottleneck -- if the tester's CPU is at 100% while the server's CPU has idle capacity, the results understate the server's true capacity. In the benchmark scenario, the presenter used separate SSH sessions for monitoring and operations, running `mpstat 1` in one terminal and AutoCannon in another, on both the server and tester machines. This dual monitoring ensured that the benchmark results were valid and that any bottlenecks were correctly attributed to the right component.

### Cost Management

Running a massive traffic simulation is expensive. In the benchmark scenario, the total infrastructure cost was approximately $30 per hour: $6 per hour for the power server, $6 per hour for the power tester, $6 per hour for the database, plus storage and data transfer costs. Running the simulation for a full day would cost approximately $720. This is why the presenter emphasized the importance of being careful with cloud resources: "if you mess up a little bit, you can end up with a massive bill at the end of the month." Best practices include using spot instances for non-critical tester nodes, setting billing alerts, terminating instances immediately after testing, and validating configurations on local machines before scaling to the cloud.

## Practical Applications

The benchmark scenario demonstrated a two-node architecture with dedicated power server and power tester machines, both running C8i.32xlarge instances on AWS. The power server ran 128 Node.js instances managed by PM2, while the power tester ran AutoCannon with 120 worker threads and 1000 concurrent connections. This architecture was sufficient to generate over 6 million requests per second for the simplest route, proving that the server could handle the target of one million requests per second with significant headroom. For more complex routes, the same architecture provided the load generation capacity needed to identify bottlenecks and validate optimizations. The experience demonstrated that at extreme scale, the traffic generation infrastructure is as important as the server infrastructure -- both must be carefully designed, provisioned, and monitored to produce valid benchmark results.
