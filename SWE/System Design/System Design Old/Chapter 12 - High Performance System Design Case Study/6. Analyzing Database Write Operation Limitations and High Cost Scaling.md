# Analyzing Database Write Operation Limitations and High Cost Scaling

## Overview

Database write operations are among the most expensive and difficult-to-scale aspects of any high-throughput system. Unlike reads, which can be cached and served from memory, writes must be persisted to durable storage to survive failures, introducing disk I/O latency and consistency constraints that fundamentally limit throughput. At the scale of one million requests per second, a traditional SQL database becomes prohibitively expensive to scale for writes, because each write operation requires disk I/O, transaction logging, and potentially replication to standby nodes. Understanding why database writes are slow, how much it costs to increase write capacity, and when to seek alternative architectures is essential for building systems that can handle extreme write loads without breaking the budget.

## Key Concepts

### Why Writes Are Expensive

A database write operation involves several steps, each of which adds latency. The database must parse the query, acquire locks, update indexes, write data pages to the buffer pool, flush the write-ahead log (WAL) to durable storage, and potentially replicate the change to standby nodes. The WAL flush is particularly costly because it requires a synchronous disk write -- the database cannot acknowledge the write until the WAL record is safely on disk, ensuring durability in the event of a crash. On a spinning disk, a single synchronous write takes approximately 10 milliseconds (limited by rotational speed and seek time), limiting throughput to approximately 100 writes per second per disk. Even on SSDs with thousands of IOPS, the aggregate write throughput is limited by the number of available IOPS and the overhead of transaction processing.

### IOPS and Write Throughput

Input/Output Operations Per Second (IOPS) is the key metric for database write performance. AWS provisioned IOPS SSDs (io2 volumes) can deliver up to 256,000 IOPS per volume, but at a significant cost premium. In the benchmark scenario, the PostgreSQL database was configured with approximately 3,000 IOPS on gp3 storage, which is relatively modest. At 3,000 IOPS, the database can handle approximately 3,000 write operations per second -- far short of the one million writes per second target. Scaling to even 100,000 IOPS would require io2 volumes and a significantly more expensive database instance, with costs scaling roughly linearly with IOPS provisioned. This linear cost scaling means that a database capable of handling one million writes per second would cost tens of thousands of dollars per month just in storage and IOPS charges.

### Write Amplification

Write amplification occurs when a single logical write operation results in multiple physical writes to disk. Index updates, WAL records, and data page writes all contribute to write amplification. A single INSERT statement that updates three indexes might result in four physical writes (one for the data and three for the indexes). The WAL entry adds another write. If replication is configured, each replica requires its own set of writes. This multiplication effect means that the actual disk I/O required can be many times the logical write rate, further constraining throughput. At scale, reducing write amplification -- by minimizing indexes, batching writes, or using append-only data structures -- can significantly improve effective write throughput.

### Connection Pool Exhaustion

Each database connection consumes memory on the database server (typically 5-10 MB per connection for PostgreSQL) and requires a dedicated process or thread to handle queries. When running 128 Node.js worker processes, each maintaining a connection pool of 10 connections, the total number of database connections is 1,280. A db.m5.16xlarge PostgreSQL instance with 256 GB of RAM can handle several thousand connections, but each active connection consumes CPU and memory resources on the database server. At high write rates, the connection pool can become a bottleneck not because of connection limits but because each connection's queries are queuing behind other connections' queries, increasing latency across the board.

### The Cost of Vertical Scaling

Vertically scaling a database instance -- upgrading to a larger instance with more CPU, RAM, and IOPS -- is the simplest way to increase write capacity, but it is also the most expensive. The db.m5.16xlarge instance with 64 CPU cores and 256 GB of RAM costs approximately $6 per hour. Upgrading to an even larger instance with more IOPS could double or triple this cost. And even the largest available instance has finite capacity. Horizontally scaling writes -- distributing write operations across multiple database instances -- is technically possible through sharding, but it introduces significant application complexity, including cross-shard transactions, distributed schema migrations, and rebalancing when shards grow unevenly. The fundamental insight is that at extreme scale, a relational database is not the right tool for high-throughput writes, and alternative architectures must be considered.

## Practical Applications

In the benchmark scenario, the PostgreSQL database with 3,000 IOPS became the immediate bottleneck for any route involving database writes. While the application server could handle tens of thousands of requests per second, the database could only process a few thousand writes per second, creating a severe mismatch. This mismatch illustrates the common adage that "the database is always the bottleneck," but the presenter made a crucial observation: "you don't even want your database to be your main bottleneck because the costs are just going to be absolutely unbelievable." The solution was not to scale the database further but to rethink the architecture, moving write operations to an in-memory store (Redis) that could handle orders of magnitude more operations per second at a fraction of the cost.
