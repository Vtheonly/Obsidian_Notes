# Cloud Infrastructure Sizing AWS EC2 Compute and Network Optimized Instances

## Overview

Cloud infrastructure sizing is the process of selecting the right virtual machine instances to match the computational, memory, and networking requirements of a workload. AWS EC2 offers over 600 instance types, each optimized for different use cases: compute-optimized instances for CPU-intensive workloads, memory-optimized instances for large datasets, network-optimized instances for high-throughput applications, and storage-optimized instances for database workloads. At the scale of one million requests per second, instance selection is not a theoretical exercise -- it directly determines whether the system can handle the load and how much it costs to run. The wrong instance type can mean either insufficient capacity (causing outages) or excessive cost (wasting money on unused resources). Understanding the instance landscape and how to map workload requirements to instance capabilities is a critical skill for building high-performance systems in the cloud.

## Key Concepts

### EC2 Instance Categories

AWS EC2 instances are organized into categories based on their hardware optimization. Compute-optimized instances (C-family, such as C7i, C8i) offer the highest ratio of CPU to memory, making them ideal for web servers, application servers, and batch processing. Memory-optimized instances (R-family, X-family) provide large amounts of RAM relative to CPU, suited for in-memory databases and large-scale data processing. Network-optimized instances offer enhanced networking capabilities with higher bandwidth and lower latency. Storage-optimized instances (I-family) provide high IOPS local storage for database workloads. General-purpose instances (M-family) balance CPU, memory, and networking for diverse workloads. The C8i.32xlarge instance used in the benchmark scenario is a compute-optimized instance with 128 CPU cores, 256 GB of RAM, and 50 Gbps network performance, costing approximately $6 per hour.

### Understanding vCPUs and Physical Cores

AWS uses the term vCPU (virtual CPU) to describe the compute capacity of an instance. On most instance types, one vCPU corresponds to one hyperthread on a physical core. This means that a "128 vCPU" instance may have 64 physical cores with two hyperthreads each. For CPU-bound workloads, the number of physical cores is more relevant than the number of vCPUs, because hyperthreads share execution units and do not provide the full throughput of a dedicated core. When running 128 Node.js processes on a 128-vCPU instance, each process gets its own vCPU, but pairs of processes share physical cores. In practice, the performance difference between hyperthreaded and dedicated core execution is workload-dependent, but it is worth understanding when squeezing maximum performance from an instance.

### Network Performance Tiers

EC2 instances are grouped into network performance tiers. Instances with "Up to 5 Gbps" networking share bandwidth with other instances on the same physical host, making their actual network performance variable. Instances with "12.5 Gbps" or higher provide dedicated network capacity. The C8i.32xlarge instance offers 50 Gbps (6.25 GB/s) of dedicated network bandwidth, which is essential for high-throughput workloads. At one million requests per second with an average response size of 1 KB, the network traffic is approximately 1 GB/s (8 Gbps), well within the 50 Gbps limit. However, with larger response sizes (32 KB), the same request rate would require 32 GB/s (256 Gbps), far exceeding any single instance's capacity and necessitating horizontal scaling across multiple instances behind a load balancer.

### Cost Analysis and Right-Sizing

The cost of cloud infrastructure at scale is significant and must be carefully managed. The C8i.32xlarge instance at approximately $6 per hour translates to approximately $4,380 per year, and running two such instances (one for the server and one for the tester) plus a powerful database instance (db.m5.16xlarge at approximately $6 per hour) brings the total to approximately $30 per hour or $20,000 per month. Right-sizing means selecting the smallest instance that can handle the workload with acceptable headroom. Over-provisioning by even one instance size can add thousands of dollars per year. Techniques for right-sizing include load testing at different instance sizes, using AWS Cost Explorer to analyze utilization patterns, and leveraging spot instances for non-critical workloads that can tolerate interruption.

### AMI and Instance Configuration

Creating a custom Amazon Machine Image (AMI) pre-configured with all necessary software (Node.js, PM2, application code) dramatically reduces the time to launch new instances. Instead of spending 30 minutes configuring each new server, a pre-built AMI can be launched and running in minutes. This is especially important during scaling events, where the ability to add capacity quickly can mean the difference between a brief degradation and a prolonged outage. The AMI should be regularly updated to include the latest application code, security patches, and configuration changes, and the AMI creation process should be automated as part of the CI/CD pipeline.

## Practical Applications

In the benchmark scenario, the transition from a Mac Studio to the AWS C8i.32xlarge instance represented a 10x increase in computing power. The C8i instance's 128 CPU cores (compared to 12 on the Mac Studio), 256 GB RAM (compared to 32 GB), and 50 Gbps network (compared to 10 Gbps) enabled throughput to scale from approximately 50,000 RPS to over 6 million RPS for the simplest route. Two C8i instances were launched: one as the "power server" running the application, and one as the "power tester" running AutoCannon. A db.m5.16xlarge PostgreSQL instance (64 CPU cores, 256 GB RAM, approximately $5-6 per hour) was also launched for database tests. The total infrastructure cost was approximately $18-20 per hour, plus storage costs.
