# Single Server Benchmarking Basics and AutoCannon Setup

## Overview

Benchmarking is the discipline of measuring a system's performance under controlled conditions, providing the quantitative foundation for all scaling decisions. Before deploying to production, and certainly before investing in expensive infrastructure, engineers must establish baseline performance numbers on a single server. This involves understanding how many requests per second a given application can handle, what resource bottlenecks limit throughput, and how different configurations affect performance. AutoCannon is a Node.js-based HTTP benchmarking tool that enables engineers to generate massive amounts of traffic against a target server and measure the resulting throughput, latency, and error rates. Mastering benchmarking fundamentals is essential because, at extreme scale, intuition about performance is unreliable -- only empirical measurement can guide architectural decisions.

## Key Concepts

### Why Benchmarking Matters

At the scale of one million requests per second, intuition about performance is dangerously misleading. What seems like a reasonable architectural choice on a development machine may prove catastrophic in production. Benchmarking provides the empirical evidence needed to make informed decisions about framework selection, infrastructure sizing, caching strategies, and language choices. Without benchmarking, teams risk over-provisioning (wasting money on unnecessary resources) or under-provisioning (causing outages during traffic spikes). The benchmarking process also reveals unexpected bottlenecks -- a database that seemed fast enough in development may become the limiting factor under load, or a framework's overhead may consume more CPU than anticipated.

### AutoCannon Setup and Installation

AutoCannon is installed globally via npm with the command `npm i -g autocannon`. Once installed, it can be invoked from the command line with a variety of options to control the benchmark. The basic syntax is `autocannon [options] URL`, where options specify the number of connections, duration, pipelining factor, number of worker threads, HTTP method, request body, and content type. AutoCannon generates HTTP traffic from one or more worker threads, each opening multiple TCP connections to the target server, and measures the server's throughput in requests per second. The tool reports average throughput, latency percentiles (p50, p95, p99), and error counts, providing a comprehensive view of server performance under load.

### Understanding AutoCannon Options

The key AutoCannon options are `-c` (number of concurrent connections), `-d` (test duration in seconds), `-p` (pipelining factor -- how many requests to send on each connection before waiting for a response), `-w` (number of worker threads), `-m` (HTTP method), `-b` (request body), and `-H` (headers). The number of concurrent requests at any given moment is calculated as connections multiplied by pipelining: `-c 6 -p 2` means 12 concurrent requests. The `-w` option determines how many CPU cores are utilized on the client machine to generate traffic. For benchmarking powerful servers with many cores, multiple worker threads are essential to generate sufficient load. Understanding these options is critical because misconfigured benchmarks produce misleading results.

### Interpreting Benchmark Results

AutoCannon reports several key metrics. The average requests per second (RPS) is the primary throughput metric. Latency percentiles indicate the distribution of response times: p50 (median), p95, and p99 reveal tail latency. Error counts indicate how many requests failed. Total data transferred shows the network impact. When interpreting results, the average RPS is the most important number for capacity planning, but tail latency is critical for understanding user experience. A server that handles 100,000 RPS at p50 but has p99 latency of several seconds may be unacceptable for real-time applications. Consistent results across multiple runs indicate reliable measurements, while high variance suggests that the server or network is experiencing instability.

### Common Benchmarking Pitfalls

Several pitfalls can invalidate benchmark results. Testing on the same machine as both client and server means the two compete for CPU and network resources, producing artificially low numbers. Insufficient warm-up time means the first few seconds of the test include JIT compilation and connection establishment overhead. Too few connections means the server is not fully utilized. Ignoring garbage collection pauses can produce misleading latency spikes. Not pinning CPU frequencies allows thermal throttling to affect results. To produce reliable benchmarks, use separate client and server machines, run multiple iterations, discard warm-up results, and monitor both client and server resource utilization throughout the test.

## Practical Applications

In the benchmark scenario, a Mac Studio with 12 CPU cores, 32 GB RAM, and 10 Gbps network performance served as the initial test platform. Running Express.js with a simple `/simple` route returning a JSON message, AutoCannon with 20 connections, 20-second duration, and pipelining of 2 yielded approximately 18,000 requests per second. Switching to Fastify improved this to approximately 66,000 RPS, and the custom Cpeak framework achieved around 73,000 RPS. These results demonstrate that framework choice alone can account for a 3-4x difference in throughput, validating the importance of empirical benchmarking over assumptions. When moving to AWS with a C8i.32xlarge instance (128 CPU cores, 256 GB RAM, 50 Gbps network), the same simple route handled over 6 million RPS with 1000 connections, 100 pipelining, and 120 worker threads.
