# Database Sharding Range Based Directory Based and Geographical

## Overview
Database sharding is a horizontal scaling technique that partitions data across multiple database servers, called shards, so that each shard holds a subset of the total data. Unlike replication, which copies the same data to multiple servers, sharding distributes different data to different servers, allowing the system to scale both storage capacity and throughput by adding more shards. Sharding is essential when a single database server cannot handle the data volume or query load of the application. The critical challenge in sharding is the shard key: the attribute used to determine which shard a given record belongs to. The three primary sharding strategies are range-based, directory-based (hash-based), and geographical. Each strategy has different tradeoffs in terms of data distribution, query routing, resharding complexity, and support for range queries. Choosing the right sharding strategy is one of the most important architectural decisions for large-scale systems.

## Key Concepts

### Why Sharding is Needed
Vertical scaling (adding more CPU, RAM, and storage to a single server) has hard limits: there is a maximum amount of hardware that can fit in a single machine, and the cost scales non-linearly. When an application outgrows the capacity of a single database server, horizontal scaling through sharding becomes necessary. Sharding allows the system to distribute data and load across many commodity servers, providing linear scalability for both reads and writes. It also improves fault tolerance: the failure of one shard affects only the subset of data stored on that shard, not the entire dataset. However, sharding introduces significant complexity in query routing, cross-shard operations, data rebalancing, and operational management.

### Range-Based Sharding
Range-based sharding assigns contiguous ranges of the shard key to specific shards. For example, user records might be partitioned so that users with IDs 1-1,000,000 go to Shard A, 1,000,001-2,000,000 to Shard B, and so on. Range-based sharding is intuitive and supports efficient range queries: a query for users with IDs between 500,000 and 1,500,000 needs to access only Shards A and B. However, range-based sharding is prone to hotspots: if new users are assigned sequential IDs and new users are the most active, Shard B (which holds the newest users) will receive disproportionately more traffic than Shard A. This uneven distribution can negate the benefits of sharding. Range-based sharding works best when the data is uniformly distributed across the key range or when the access patterns are not skewed toward a specific range.

### Hash-Based (Directory-Based) Sharding
Hash-based sharding applies a hash function to the shard key and uses the hash value to determine the target shard. For example, `shard = hash(user_id) % num_shards` evenly distributes users across shards, eliminating the hotspot problem of range-based sharding. The tradeoff is that range queries become expensive: a query for users with IDs in a range must be sent to all shards because the hash function scatters contiguous keys across different shards. Consistent hashing is a variant of hash-based sharding that minimizes data movement when shards are added or removed. In consistent hashing, each shard is assigned a position on a hash ring, and each key is assigned to the first shard whose position is clockwise from the key's hash position. Adding or removing a shard only affects the keys in the affected range, not the entire dataset.

### Directory-Based Sharding with a Lookup Service
Directory-based sharding uses a separate lookup service (a directory) that maps each shard key to the shard where the data resides. When the application needs to access a record, it first queries the directory to determine which shard holds the data, then queries that shard directly. This approach provides the most flexibility: the mapping can be arbitrary and can be updated without moving data (by updating the directory entry). However, the directory itself becomes a critical component that must be highly available and low-latency. If the directory goes down, the entire system is inaccessible. The directory is typically implemented as a distributed key-value store (like ZooKeeper or etcd) or cached in the application layer to avoid a lookup on every request.

### Geographical Sharding
Geographical sharding (also called geo-sharding or location-based sharding) partitions data based on the geographic location of the user or the data. For example, users in North America might be stored on shards in US data centers, while users in Europe are stored on EU data centers. This reduces latency for users by serving their data from nearby servers and can help with data residency regulations that require certain data to be stored within specific jurisdictions. Geographical sharding can be combined with other sharding strategies: within each geographic region, data might be further partitioned using hash-based or range-based sharding. The challenge is handling users who move between regions or cross-region queries that need to access data from multiple geographic shards.

### Resharding and Data Rebalancing
As the application grows, the number of shards may need to increase, requiring data to be moved from existing shards to new ones. This process, called resharding or rebalancing, is one of the most operationally complex aspects of sharding. During resharding, data is moved in the background while the system continues to serve requests. Consistent hashing simplifies resharding by minimizing the amount of data that needs to move. Range-based sharding can split a range into two sub-ranges, moving one sub-range to a new shard. Hash-based sharding typically requires doubling the number of shards (to maintain the modulo operation) and moving half the data from each existing shard to a new shard. The resharding process must be carefully orchestrated to avoid data loss, minimize downtime, and maintain consistency.

### Cross-Shard Queries and Joins
One of the most significant limitations of sharding is the difficulty of cross-shard queries and joins. A query that needs to access data from multiple shards must be executed on each shard separately, and the results must be combined at the application layer. This scatter-gather pattern is slower and more complex than a single-shard query. Joins across shards are particularly expensive and are generally avoided by denormalizing data or duplicating reference data across shards. Some databases (like CockroachDB and Google Spanner) support distributed joins transparently, but they incur higher latency than single-node joins. The shard key should be chosen to minimize cross-shard queries for the most common access patterns.

## Practical Applications
Sharding is used by virtually every large-scale application. Instagram shards its user data by user ID using hash-based sharding, distributing billions of records across many PostgreSQL instances. Slack uses range-based sharding for its message data, partitioning by workspace ID so that all messages for a workspace are on the same shard and can be queried efficiently. Uber uses geographical sharding for its trip data, storing trips in the region where they occurred to minimize latency for drivers and riders. Shopify uses directory-based sharding for its merchant data, with a lookup service that maps each shop to its shard. In each case, the sharding strategy is chosen based on the specific data distribution, access patterns, and scalability requirements of the application.
