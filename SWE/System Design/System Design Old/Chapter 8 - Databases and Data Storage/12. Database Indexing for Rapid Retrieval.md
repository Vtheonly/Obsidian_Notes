# Database Indexing for Rapid Retrieval

## Overview
Database indexing is a technique for dramatically accelerating data retrieval by creating auxiliary data structures that allow the database to find specific rows without scanning the entire table. An index is essentially a lookup table that maps the values of one or more columns to the physical locations of the corresponding rows. Without an index, the database must perform a full table scan, reading every row to find those that match the query criteria -- an O(n) operation that becomes prohibitively slow for large tables. With an appropriate index, the database can locate matching rows in O(log n) time or better, reducing query response times from seconds or minutes to milliseconds. However, indexes are not free: they consume additional storage, slow down write operations (because the index must be updated whenever data is inserted, updated, or deleted), and require ongoing maintenance. Understanding when and how to create indexes is one of the most impactful skills for database performance optimization.

## Key Concepts

### B-Tree Indexes
B-tree (Balanced Tree) indexes are the most common type of database index and the default in most relational databases. A B-tree is a self-balancing tree data structure where each node contains a sorted array of keys and pointers to child nodes. The tree is always balanced, meaning that the path from the root to any leaf node is the same length, guaranteeing O(log n) lookup time. B-tree indexes support efficient point lookups (finding a row with a specific key value), range scans (finding all rows with key values in a range), and ordered traversals (scanning rows in key order). When a query filters on an indexed column, the database uses the B-tree to quickly locate the matching rows without scanning the table. B-tree indexes are used by PostgreSQL, MySQL (InnoDB), Oracle, and SQL Server as their primary index structure.

### Hash Indexes
Hash indexes use a hash table to map key values to row locations. They provide O(1) average lookup time for exact-match queries (equality comparisons), which is faster than B-tree indexes for this specific use case. However, hash indexes do not support range queries, ordering, or prefix matching, because the hash function destroys the ordering of the keys. Hash indexes are used in PostgreSQL (for hash indexes), in-memory databases like Redis, and in some NoSQL databases for primary key lookups. In PostgreSQL, hash indexes were historically discouraged due to reliability issues, but they have been improved and can be useful for simple equality lookups on large tables.

### Composite Indexes
A composite index (also called a multi-column or concatenated index) is an index on two or more columns. The order of the columns in the index matters: a composite index on (last_name, first_name) can efficiently serve queries that filter on both columns, or queries that filter on last_name alone, but it cannot efficiently serve queries that filter on first_name alone. This is because the index is sorted first by last_name and then by first_name within each last_name group. The leftmost prefix rule states that a composite index can be used for any query that filters on a leftmost prefix of the indexed columns. Designing effective composite indexes requires understanding the application's query patterns and ordering columns by selectivity (most selective first) and usage frequency.

### Covering Indexes
A covering index is an index that includes all the columns needed by a query, so that the database can satisfy the query entirely from the index without accessing the table (a so-called "index-only scan"). For example, if a query selects `first_name` and `email` from the `users` table where `last_name = 'Smith'`, a composite index on (last_name, first_name, email) would be a covering index because all the requested columns are in the index. Covering indexes can dramatically improve query performance by eliminating the need for random I/O to fetch the actual table rows. PostgreSQL supports INCLUDE columns in indexes for this purpose, and SQL Server has INCLUDE as well. The tradeoff is that covering indexes are larger (they store more data) and are more expensive to maintain on writes.

### Clustered vs Non-Clustered Indexes
In a clustered index, the table rows are physically stored in the order of the index key. There can be only one clustered index per table because the rows can only be stored in one physical order. In MySQL's InnoDB engine, the primary key is a clustered index, and all other indexes are secondary indexes that contain the primary key value rather than a physical row pointer. In a non-clustered index, the index contains pointers to the actual row locations, and the table rows are stored separately in an arbitrary order. Non-clustered indexes are more flexible (multiple per table) but require an additional lookup to fetch the actual row data. Understanding the difference is important for performance tuning: clustered indexes are optimal for range scans on the primary key, while non-clustered indexes are better for point lookups on other columns.

### Index-Only Scan and Heap Fetch
When a query can be satisfied entirely from an index (an index-only scan), the database avoids the costly step of fetching the actual row from the table heap. However, for an index-only scan to work in PostgreSQL, all columns referenced in the query must be present in the index, and the visibility map must indicate that all pages in the table are visible to all transactions (to avoid needing to check the heap for row visibility). The performance difference between an index-only scan and an index scan with heap fetch can be significant, especially for wide tables where fetching a single row requires reading a large page from disk.

### When Not to Index
Not every column should be indexed. Indexes that are rarely or never used by queries waste storage and slow down writes without providing any benefit. Columns with low cardinality (few distinct values, such as a boolean flag) generally make poor index targets because the index is not selective: it returns a large fraction of the table, and the database may choose a full table scan instead. Very small tables may not benefit from indexes at all because a sequential scan is fast enough. Columns that are frequently updated but rarely queried may not justify the write overhead of maintaining an index. Indexes should be created based on actual query patterns, not speculative assumptions, and unused indexes should be identified and removed.

## Practical Applications
Database indexing is essential for any application that queries large datasets. An e-commerce platform indexes product IDs, category IDs, and price ranges to enable fast product searches. A social media platform indexes user IDs and timestamps to quickly load a user's feed. A logging platform indexes timestamps and severity levels to support efficient time-range queries and filtering. A financial application indexes account IDs and transaction dates to generate account statements quickly. In each case, the right indexes transform queries that would take seconds or minutes into queries that return in milliseconds, making the difference between a responsive application and an unusable one.
