# Key Value Stores Redis and Memcached

## Overview
Key-value stores are the simplest and fastest type of NoSQL database, storing data as a collection of key-value pairs where each key uniquely identifies a value. The value can be a simple string, a blob, or a more complex data structure depending on the implementation. Redis and Memcached are the two most widely used key-value stores, and they are primarily employed as in-memory caches to accelerate data access and reduce the load on primary databases. Both store data in RAM, enabling sub-millisecond read and write latency that is orders of magnitude faster than disk-based databases. While they share the same basic model, Redis and Memcached differ significantly in features, data structures, persistence options, and use cases. Understanding their strengths and tradeoffs is essential for designing high-performance caching and data storage layers.

## Key Concepts

### Redis Data Structures
Redis goes far beyond simple key-value storage by supporting a rich set of data structures. Strings are the basic type and can hold text, numbers, or binary data up to 512 MB. Hashes map string fields to string values, ideal for representing objects. Lists are ordered collections of strings that support push and pop operations from both ends, useful for queues and stacks. Sets are unordered collections of unique strings that support intersection, union, and difference operations. Sorted sets (ZSETs) associate each member with a score and maintain the set in score order, enabling efficient range queries and ranking operations. Redis also supports bitmaps, HyperLogLog (for cardinality estimation), geospatial indexes, and streams (for log-like data). This rich data model allows Redis to be used for much more than simple caching, including real-time leaderboards, rate limiting, pub/sub messaging, and session management.

### Memcached Simplicity
Memcached is designed to do one thing extremely well: cache arbitrary data in memory as key-value pairs. It supports only string keys and string values, with no complex data structures. This simplicity makes Memcached extremely fast and lightweight. It is multithreaded, meaning it can efficiently utilize multiple CPU cores on a single machine, and it uses a slab allocation model for memory management that reduces fragmentation. Memcached has no persistence, no replication, and no complex data operations. It is a pure cache, and if the server restarts, all data is lost. For many caching use cases, this simplicity is a feature rather than a limitation: there is less that can go wrong, and the system is easy to understand, deploy, and operate.

### Memory Management
Both Redis and Memcached are in-memory stores, but they manage memory differently. Memcached uses a slab allocator that divides memory into fixed-size chunks (slabs) and assigns items to the smallest slab that can hold them. This eliminates fragmentation but can waste space if items do not fit neatly into the available slab sizes. Redis uses a more traditional memory allocator (jemalloc by default) and supports virtual memory and LRU eviction when memory is full. Redis allows fine-grained control over memory policy through the `maxmemory` and `maxmemory-policy` configuration options, including LRU eviction, LFU eviction, TTL-based eviction, and volatile/random eviction strategies.

### Persistence in Redis
Unlike Memcached, Redis supports optional persistence, allowing cached data to survive server restarts. Redis offers two persistence mechanisms: RDB (Redis Database) snapshots and AOF (Append-Only File). RDB creates point-in-time snapshots of the dataset at configurable intervals, which are compact and fast to restore but may lose data written after the last snapshot. AOF logs every write operation to a file, providing a more durable guarantee at the cost of larger file sizes and slightly lower write throughput. Redis can use both mechanisms simultaneously, combining the compactness of RDB with the durability of AOF. For pure caching use cases, persistence is often disabled for maximum performance.

### Replication and High Availability
Redis supports master-slave replication, where one master node handles writes and one or more replica nodes maintain copies of the data for read scaling and failover. Redis Sentinel provides automatic failover: if the master fails, Sentinel promotes a replica to master and reconfigures other replicas to follow the new master. Redis Cluster provides a more comprehensive solution that supports automatic sharding and failover across multiple nodes, distributing data across the cluster using hash slots. This allows Redis to scale horizontally for both storage and throughput. Memcached, in contrast, has no built-in replication or high availability. It relies on the client to handle distribution across multiple nodes using consistent hashing, and failed nodes are simply removed from the pool.

### Use Cases and Choosing Between Redis and Memcached
The choice between Redis and Memcached depends on the use case. Memcached is ideal for simple, high-throughput caching of small objects where persistence and complex data structures are not needed. It excels at session caching, HTML fragment caching, and database query result caching. Redis is the better choice when the application needs data structures beyond simple strings (sorted sets for leaderboards, lists for queues, hashes for objects), persistence, replication, pub/sub messaging, or atomic operations on complex types. Redis is also preferred when the cache needs to survive restarts or when it serves as a primary data store for ephemeral data such as rate limiting counters or real-time analytics.

## Practical Applications
Redis and Memcached are used across virtually every industry for high-performance data access. Twitter uses Redis for its timeline generation and real-time notification system, leveraging sorted sets for ranking and pub/sub for pushing updates. GitHub uses Redis for rate limiting, session storage, and queue management. Stack Overflow uses Redis for caching and real-time updates. Pinterest uses Memcached for its object caching layer, storing serialized objects to avoid repeated database queries. Gaming companies use Redis for real-time leaderboards and player state management. In each case, the sub-millisecond latency of in-memory storage is essential for delivering responsive user experiences at scale.
