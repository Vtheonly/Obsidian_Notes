# ACID Properties Atomicity Consistency Isolation Durability

## Overview
The ACID properties are the cornerstone guarantees that relational databases provide for transactions. A transaction is a sequence of one or more database operations that are executed as a single logical unit of work. ACID stands for Atomicity, Consistency, Isolation, and Durability -- four properties that ensure database transactions are processed reliably even in the presence of system failures, concurrent access, and other error conditions. These guarantees are what make relational databases suitable for applications where data correctness is critical, such as financial systems, inventory management, and healthcare records. Understanding ACID deeply is essential for any engineer working with databases, as the tradeoffs between strict ACID compliance and performance are at the heart of many system design decisions.

## Key Concepts

### Atomicity
Atomicity guarantees that a transaction is treated as a single, indivisible unit. Either all operations within the transaction are successfully completed, or none of them are. If any operation fails, the entire transaction is rolled back, and the database is restored to its state before the transaction began. Consider a bank transfer: debiting one account and crediting another must happen as a single atomic operation. If the credit fails after the debit succeeds, the database rolls back the debit, ensuring that money is neither created nor destroyed. Atomicity is typically implemented using a write-ahead log (WAL), where the database records the intended changes before applying them. If a failure occurs, the database uses the WAL to undo incomplete transactions during recovery.

### Consistency
Consistency ensures that a transaction transforms the database from one valid state to another valid state. All defined rules, constraints, triggers, and cascades must be satisfied after the transaction completes. If a transaction would leave the database in an invalid state (for example, violating a foreign key constraint or a CHECK constraint), the transaction is rejected and rolled back. Consistency is the property that connects transactions to the business rules of the application. It is important to note that consistency in the ACID sense refers to the database's internal consistency rules, not the CAP theorem's consistency (which refers to replicated data consistency across distributed nodes).

### Isolation
Isolation ensures that concurrently executing transactions do not interfere with each other. Each transaction should appear to execute in isolation, as if it were the only transaction running on the system, even when many transactions are executing simultaneously. In practice, full isolation (serializability) can be expensive in terms of performance, so most databases offer multiple isolation levels with different tradeoffs between correctness and performance. The standard isolation levels, from weakest to strongest, are Read Uncommitted, Read Committed, Repeatable Read, and Serializable. At the Read Uncommitted level, a transaction can see uncommitted changes from other transactions (dirty reads). At the Serializable level, transactions appear to execute one at a time, with no interference whatsoever.

### Durability
Durability guarantees that once a transaction has been committed, its effects are permanent and will survive any subsequent system failures. Even if the database server crashes or loses power immediately after a commit, the committed data will be available when the system restarts. Durability is typically implemented through write-ahead logging and flushing changes to persistent storage (disk or SSD) before acknowledging the commit. Some databases also support synchronous replication, where committed data is written to multiple nodes before acknowledging, providing durability even if an entire storage device fails.

### Isolation Levels and Anomalies
Understanding the anomalies that can occur at different isolation levels is critical. Dirty reads occur when a transaction reads data written by an uncommitted transaction. Non-repeatable reads occur when a transaction reads the same row twice and gets different values because another transaction modified the row between the two reads. Phantom reads occur when a transaction re-executes a range query and gets a different set of rows because another transaction inserted or deleted rows in the range. Read Committed prevents dirty reads, Repeatable Read prevents dirty reads and non-repeatable reads, and Serializable prevents all three anomalies. Many databases implement Repeatable Read using snapshot isolation (MVCC), which provides a consistent view of the database at the start of the transaction.

### ACID in Distributed Systems
Implementing ACID guarantees in a distributed database is significantly more challenging than in a single-node database. Network partitions, node failures, and clock skew all complicate the picture. Distributed ACID transactions require coordination protocols such as Two-Phase Commit (2PC), which ensures that all participating nodes either commit or abort a transaction. However, 2PC is blocking: if the coordinator fails, participants may be left in an uncertain state, holding locks and unable to proceed. More recent approaches like Three-Phase Commit (3PC) and Paxos-based consensus protocols (used by systems like Google Spanner) provide better fault tolerance but at the cost of increased latency and complexity.

## Practical Applications
ACID properties are essential in any domain where data correctness is non-negotiable. In banking, a transfer between accounts must be atomic and durable; partial transfers or lost transactions would be catastrophic. In e-commerce, placing an order must atomically create the order, reserve inventory, and charge the payment method; if any step fails, the entire operation must roll back. In healthcare, patient records must maintain consistency with medical coding rules and regulatory requirements. In airline reservation systems, seat assignments must be isolated so that two customers cannot book the same seat simultaneously. These applications rely on ACID guarantees to ensure that the database always reflects a correct and consistent view of the real world.
