# Ports and Network Services

## Overview
While IP addresses identify devices on a network, ports identify specific processes or services running on those devices. A port is a 16-bit number ranging from 0 to 65535, and together with an IP address, it forms a socket, which is the endpoint of a bidirectional communication channel. Ports enable multiple network applications to operate simultaneously on a single device without interference, forming the backbone of multiplexed network communication.

## Key Concepts

### Port Number Ranges
Ports are divided into three ranges by IANA. Well-known ports, numbered 0 through 1023, are reserved for system-level services and require elevated privileges to bind on Unix-like operating systems. Examples include port 80 for HTTP, port 443 for HTTPS, port 22 for SSH, and port 53 for DNS. Registered ports, numbered 1024 through 49151, are assigned by IANA to specific applications upon request, such as port 3306 for MySQL and port 5432 for PostgreSQL. Dynamic or ephemeral ports, numbered 49152 through 65535, are used by client applications for temporary connections and are assigned by the operating system automatically.

### How Ports Enable Multiplexing
When a client connects to a server, it uses the server's IP address and a well-known port number to reach the desired service, while the operating system assigns an ephemeral port on the client side to identify the connection. This means a single server can handle thousands of simultaneous connections on port 443, with each connection uniquely identified by the combination of source IP, source port, destination IP, and destination port. This four-tuple ensures that incoming data is delivered to the correct process.

### Common Service Port Mappings
Understanding standard port assignments is essential for system design. Web traffic flows through ports 80 and 443. Email uses port 25 for SMTP, port 143 for IMAP, and port 993 for IMAPS. Database services occupy ports like 3306 for MySQL, 5432 for PostgreSQL, and 6379 for Redis. Message queue brokers use ports such as 5672 for AMQP and 9092 for Kafka. Firewalls and security groups must be configured to allow traffic only on the ports that legitimate services require.

### Port Security and Firewalls
Exposing unnecessary ports increases the attack surface of a system. Firewall rules should follow the principle of least privilege, allowing inbound connections only on ports that serve authorized services. Port scanning tools like Nmap can discover open ports, so administrators must regularly audit which ports are exposed and ensure that no unauthorized services are listening. In cloud environments, security groups act as virtual firewalls that control inbound and outbound traffic at the port level.

### Port Forwarding and NAT
In environments where private networks use NAT, port forwarding maps an external port on the public IP to an internal IP and port. This technique allows external clients to reach services running on private network devices. However, port forwarding must be configured carefully to avoid exposing internal services unintentionally, and it is generally preferable to use a reverse proxy or load balancer instead of direct port forwarding for production services.

## Practical Applications
When deploying a web application with a PostgreSQL database, the web server listens on ports 80 and 443, while the database listens on port 5432. The security group for the web server allows inbound traffic on 80 and 443 from the internet, while the database security group allows inbound traffic on 5432 only from the web server's private IP. This layered port-based access control ensures that the database is never directly accessible from the internet.
