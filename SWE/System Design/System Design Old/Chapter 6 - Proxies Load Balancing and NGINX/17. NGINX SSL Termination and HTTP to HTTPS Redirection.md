# NGINX SSL Termination and HTTP to HTTPS Redirection

## Overview
SSL termination (also called TLS offloading) is the process of decrypting HTTPS traffic at the NGINX reverse proxy and forwarding unencrypted HTTP traffic to the backend servers. As the source material explains, NGINX can handle SSL/TLS encryption and decryption, acting as the termination point for encrypted connections. This centralizes certificate management, offloads CPU-intensive encryption from backend servers, and enables the proxy to inspect decrypted traffic for security purposes. Additionally, NGINX can enforce HTTPS by redirecting all HTTP requests to their HTTPS equivalents, ensuring that all client connections are encrypted. Understanding SSL termination and HTTP-to-HTTPS redirection is essential for deploying secure web applications in production, where encrypted communication is a non-negotiable requirement.

## Key Concepts

### SSL Certificate Configuration
To enable SSL termination, NGINX must be configured with the SSL certificate and private key. The certificate is obtained from a Certificate Authority (CA) such as Let's Encrypt, DigiCert, or AWS Certificate Manager. In the NGINX configuration, the server block is configured with "listen 443 ssl" and two directives specify the certificate files: "ssl_certificate /path/to/certificate.crt" (the public certificate, often in PEM format which includes the full chain) and "ssl_certificate_key /path/to/private.key" (the private key). The private key must be kept secure and readable only by the root user. For Let's Encrypt certificates, the certbot tool automates certificate provisioning and renewal. In the source material, SSL is demonstrated by specifying the certificate and key paths in the server block and adding the "ssl" parameter to the listen directive. The certificate and key files must be in PEM format and must match each other.

### HTTP to HTTPS Redirection
To enforce HTTPS, all HTTP traffic on port 80 must be redirected to HTTPS on port 443. This is accomplished with a separate server block that listens on port 80 and returns a 301 redirect to the HTTPS URL. The configuration uses the "return 301 https://$host$request_uri" directive, which sends a permanent redirect response that preserves the original host and request URI. The "$host" variable contains the requested hostname, and the "$request_uri" variable contains the full request URI including query parameters. This ensures that users who type "http://example.com/path" are automatically redirected to "https://example.com/path". The source material demonstrates this pattern with two server blocks: one for the HTTP redirect and one for the HTTPS server. The redirect server block contains only the listen and return directives, making it lightweight and fast.

### SSL Protocol and Cipher Configuration
The default SSL configuration in NGINX may support outdated protocols and weak ciphers. For production security, the SSL protocol and cipher suite should be explicitly configured. The "ssl_protocols" directive specifies the allowed TLS versions: "ssl_protocols TLSv1.2 TLSv1.3" disables older, insecure protocols like SSLv3 and TLSv1.0/1.1. The "ssl_ciphers" directive specifies the allowed cipher suites, with a preference for authenticated encryption ciphers like AES-GCM and ChaCha20-Poly1305. The "ssl_prefer_server_ciphers on" directive ensures that NGINX's cipher preference order is used rather than the client's. Mozilla provides recommended SSL configurations for different security levels (modern, intermediate, and old) that are widely used as starting points. The "ssl_session_cache" and "ssl_session_timeout" directives configure session caching to improve TLS handshake performance for returning clients.

### HSTS and Security Headers
HTTP Strict Transport Security (HSTS) is a security mechanism that instructs browsers to only connect to the site via HTTPS, even if the user types "http://". The "add_header Strict-Transport-Security 'max-age=31536000; includeSubDomains' always" directive tells the browser to enforce HTTPS for one year, including all subdomains. Once a browser receives this header, it will automatically convert any HTTP URL to HTTPS before making the request, eliminating the opportunity for SSL stripping attacks. Additional security headers should be added at the NGINX level: "X-Frame-Options DENY" prevents clickjacking, "X-Content-Type-Options nosniff" prevents MIME type sniffing, "X-XSS-Protection '1; mode=block'" enables the browser's XSS filter, and "Content-Security-Policy" controls which resources the page can load. Centralizing these headers at the NGINX level ensures consistent security across all backend applications.

### Certificate Renewal and Automation
SSL certificates have a finite validity period (typically 90 days for Let's Encrypt and 1-2 years for commercial CAs). Automating certificate renewal is critical for production systems to avoid certificate expiration, which causes browsers to block access to the site. For Let's Encrypt, the certbot tool can be configured with a cron job or systemd timer to automatically renew certificates before they expire. After renewal, NGINX must be reloaded to load the new certificate: "sudo nginx -s reload". For cloud deployments, AWS Certificate Manager provides automatic renewal for certificates used with AWS services. Some NGINX configurations use the "ssl_certificate" and "ssl_certificate_key" directives with symlinks that point to the current certificate files, allowing renewal to update the symlink targets without modifying the NGINX configuration. Monitoring should alert on certificate expiration to catch cases where automated renewal fails.

## Practical Applications
Every production web application that handles sensitive data uses SSL termination at the reverse proxy or load balancer level. E-commerce platforms terminate SSL at NGINX to protect customer data in transit. SaaS applications use NGINX SSL termination combined with HSTS to enforce encrypted connections for all users. API services use SSL termination to secure API keys and bearer tokens transmitted in request headers. Internal applications behind a VPN may still use SSL termination for defense in depth. The source material demonstrates the complete SSL setup: configuring the server block with SSL directives, redirecting HTTP to HTTPS, and serving the application over an encrypted connection.
