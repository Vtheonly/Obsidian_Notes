# Load Balancer Health Checks and Self Healing

## Overview
Health checking is the mechanism by which a load balancer monitors the availability and health of backend servers, automatically removing unhealthy servers from the pool and adding them back when they recover. This self-healing capability is one of the most important functions of a load balancer because it ensures that traffic is never routed to a server that cannot process requests, without requiring manual intervention. As the source material explains, when one server goes down, the load balancer stops sending traffic to it and redirects all traffic to the remaining healthy servers until the failed server recovers. Understanding health checking mechanisms, their configuration, and their interaction with application design is essential for building resilient production systems.

## Key Concepts

### Active Health Checks
Active health checks (also called synthetic checks or out-of-band checks) are performed by the load balancer independently of client traffic. The load balancer periodically sends requests to each backend server at a configured interval and evaluates the response to determine the server's health. For HTTP health checks, the load balancer sends an HTTP request (typically GET or HEAD) to a specified URL (such as "/health") and checks the response status code, response body, and response time. If the server returns a 200 OK response within the expected time, it is considered healthy. If the server returns an error, times out, or refuses the connection, it is considered unhealthy. The load balancer marks a server as unhealthy after a configurable number of consecutive failures (the failure threshold) and marks it as healthy again after a configurable number of consecutive successes (the recovery threshold). In NGINX Plus (the commercial version), active health checks are configured using the "health_check" directive in the location block.

### Passive Health Checks
Passive health checks (also called in-band checks or real-time monitoring) evaluate server health based on actual client traffic rather than synthetic probes. The load balancer monitors the response status, latency, and error rate of each real client request and uses this data to assess server health. If a server starts returning errors (5xx status codes), timing out, or refusing connections for real requests, the load balancer marks it as unhealthy. Passive health checks have the advantage of requiring no additional requests to the backend and reflecting the actual client experience. However, they rely on real traffic to detect problems, which means unhealthy servers may receive some client requests before they are detected. Open-source NGINX supports passive health checks through the "max_fails" and "fail_timeout" directives in the upstream server configuration, which track the failure rate of real requests.

### Health Check Configuration Parameters
Health checks have several configurable parameters that affect their sensitivity and responsiveness. The check interval determines how often the load balancer probes the server (for example, every 5 seconds). A shorter interval detects failures faster but generates more health check traffic. The failure threshold (or unhealthy threshold) is the number of consecutive failed checks before a server is marked unhealthy (for example, 3 failures). A higher threshold reduces false positives caused by transient issues but delays detection of real failures. The recovery threshold (or healthy threshold) is the number of consecutive successful checks before a server is marked healthy again (for example, 2 successes). The timeout is the maximum time the load balancer waits for a response before considering the check failed. These parameters must be tuned based on the application's characteristics: a fast, stateless API can use aggressive settings (short interval, low thresholds), while a slow, stateful service may need more conservative settings to avoid false positives.

### Health Check Endpoint Design
The health check endpoint (typically "/health" or "/healthz") should accurately reflect the server's ability to process requests. A basic health check returns 200 OK if the server process is running, but this may not detect all problems. A comprehensive health check verifies all critical dependencies: the database connection, cache connectivity, message queue availability, and disk space. If any dependency is unavailable, the endpoint returns a 503 Service Unavailable status, signaling the load balancer to remove the server from the pool. However, overly aggressive health checks can cause cascading failures: if all servers report unhealthy simultaneously because a shared database is slow, the load balancer removes all servers and the entire service becomes unavailable. To prevent this, health checks should distinguish between degraded and unavailable states, and the load balancer should have a fallback mechanism (such as serving cached responses or a static error page) when no healthy servers are available.

### Circuit Breaker Pattern
The circuit breaker pattern complements load balancer health checks by providing application-level fault tolerance. When a service detects that a downstream dependency is failing, it trips the circuit breaker and stops making requests to that dependency for a cooldown period, returning a fallback response instead. This prevents cascading failures and allows the failing dependency time to recover. The circuit breaker has three states: closed (normal operation, requests flow through), open (failure detected, requests are short-circuited to fallback), and half-open (testing whether the dependency has recovered by allowing a limited number of requests). The circuit breaker pattern can be implemented at the application level (using libraries like Hystrix or resilience4j) or at the service mesh level (using Istio or Envoy). Combined with load balancer health checks, circuit breakers provide a comprehensive self-healing mechanism that operates at both the infrastructure and application levels.

## Practical Applications
NGINX Plus provides active health checks with configurable intervals, thresholds, and custom health check URIs. AWS ALB and NLB perform active health checks on registered targets, automatically deregistering unhealthy targets and re-registering them when they recover. Kubernetes uses liveness and readiness probes as health checks to determine whether a pod should receive traffic or be restarted. HAProxy supports both active and passive health checks with extensive configuration options. Service meshes like Istio implement circuit breakers that automatically detect and isolate failing service instances. Database clusters use health checks to determine which nodes are available for read and write operations, automatically failing over to a new primary if the current primary becomes unhealthy.
