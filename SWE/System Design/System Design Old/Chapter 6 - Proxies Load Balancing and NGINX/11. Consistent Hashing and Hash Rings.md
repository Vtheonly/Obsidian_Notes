# Consistent Hashing and Hash Rings

## Overview
Consistent hashing is a distributed hashing algorithm that minimizes the redistribution of keys when the number of hash buckets (servers) changes. Unlike traditional modulo-based hashing, where adding or removing a server causes nearly all keys to be remapped, consistent hashing ensures that only the keys assigned to the changed server need to be remapped. This property is critical for distributed caches, databases, and load balancers where the server pool changes dynamically due to scaling events, failures, or maintenance. Consistent hashing is visualized as a hash ring (also called a circle or continuum), where both servers and keys are mapped to positions on the ring, and each key is assigned to the first server encountered when moving clockwise from the key's position. Understanding consistent hashing is essential for designing horizontally scalable systems that require stable key-to-server mapping.

## Key Concepts

### The Hash Ring
The consistent hashing algorithm maps both server identifiers and data keys to positions on a circular space (the hash ring) using a hash function. The hash function's output range (for example, 0 to 2^32-1) forms the ring. Each server is placed on the ring at the position determined by hashing its identifier (IP address or hostname). When a key needs to be assigned to a server, the key is hashed to a position on the ring, and the algorithm walks clockwise from that position until it encounters a server. That server owns the key. For example, if servers A, B, and C are at positions 100, 300, and 700 on the ring, a key hashing to position 250 would be assigned to server B (the first server encountered clockwise), and a key hashing to position 800 would wrap around and be assigned to server A.

### Adding and Removing Servers
The key advantage of consistent hashing is that adding or removing a server affects only the keys in the affected segment of the ring, not all keys. When a new server D is added at position 500, only the keys that were previously assigned to server C (those between positions 500 and 700) are remapped to server D. All other keys remain assigned to their original servers. Similarly, when server B is removed, only the keys that were assigned to server B (those between positions 100 and 300) are remapped to the next server clockwise (server C). This contrasts with modulo hashing, where adding or removing a server changes the mapping for nearly all keys. In a system with n servers, consistent hashing remaps approximately 1/n of the keys on a server change, while modulo hashing remaps nearly all keys.

### Virtual Nodes
A problem with basic consistent hashing is uneven distribution. If only a few servers are placed on the ring, some servers may own disproportionately large or small segments, leading to unbalanced load. Virtual nodes (also called VNodes) solve this by placing multiple copies of each server at different positions on the ring. For example, instead of placing server A at one position, server A might be placed at 100 virtual positions spread around the ring. This increases the granularity of the ring and makes the distribution of keys more uniform, approaching an even distribution as the number of virtual nodes increases. When a server is added or removed, its virtual nodes are added or removed, and the affected keys are distributed more evenly across the remaining servers. The Ketama algorithm (used by libketama and memcached clients) popularized the virtual node approach with 160 virtual nodes per server by default.

### Weighted Consistent Hashing
Weighted consistent hashing extends the basic algorithm to support servers with different capacities. Servers with higher capacity should own a larger portion of the ring. This is achieved by assigning more virtual nodes to more powerful servers. For example, if server A has twice the capacity of server B, server A might have 200 virtual nodes while server B has 100. The proportion of the ring owned by each server is approximately proportional to its weight, ensuring that more capable servers receive more keys. Weight assignment is typically based on server capacity metrics such as CPU cores, memory, or measured throughput. In cloud environments with uniform instance types, weights are typically equal, but weighted consistent hashing becomes important when mixing instance types or when some servers have different resource allocations.

### Consistent Hashing in Distributed Systems
Consistent hashing is used extensively in distributed systems beyond load balancing. Distributed caches like Memcached and Redis Cluster use consistent hashing to distribute cache keys across nodes, ensuring that cache hit rates are maintained when nodes are added or removed. Distributed databases like Apache Cassandra and Amazon DynamoDB use consistent hashing (with VNodes) to partition data across nodes, enabling dynamic scaling without re-partitioning the entire dataset. Content delivery networks use consistent hashing to map content URLs to edge servers, ensuring that the same content is served from the same cache location. Load balancers use consistent hashing for session persistence, providing more stable mapping than IP hashing when the server pool changes. In each of these applications, the key benefit is minimizing the disruption caused by topology changes.

## Practical Applications
Memcached clients use consistent hashing (the Ketama algorithm) to distribute cache entries across multiple memcached servers, maintaining cache coherence when servers are added or removed. Apache Cassandra uses consistent hashing with VNodes to distribute data partitions across cluster nodes, enabling horizontal scaling without full data redistribution. Amazon DynamoDB uses consistent hashing internally for data partitioning across storage nodes. Discord uses consistent hashing to route users to chat server shards. Netflix uses consistent hashing to distribute viewing history across cache nodes. NGINX supports consistent hashing through the "hash" directive with the "consistent" parameter, providing stable request routing that minimizes disruption during scaling events.
