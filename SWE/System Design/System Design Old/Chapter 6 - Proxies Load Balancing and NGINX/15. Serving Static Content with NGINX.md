# Serving Static Content with NGINX

## Overview
Serving static content is one of NGINX's primary capabilities and one of the simplest use cases to configure. Static content includes HTML files, CSS stylesheets, JavaScript files, images, fonts, and any other files that are served to the client without server-side processing. NGINX excels at serving static content because it is designed from the ground up for high-performance file delivery, using efficient event-driven I/O and sendfile system calls to minimize CPU usage and maximize throughput. As the source material demonstrates, serving static content with NGINX requires only a few configuration directives: a server block with a listen directive, a location block with a root directive, and the static files in the specified directory. This simplicity makes NGINX an ideal choice for fronting web applications and offloading static content serving from application servers.

## Key Concepts

### Basic Static Content Configuration
The minimal configuration for serving static content consists of an HTTP context containing a server block with a location block. As shown in the source material, the server block specifies the port to listen on (for example, "listen 80") and the location block specifies the URL path and the document root. The "root" directive tells NGINX where to find the files on the filesystem. For example, with "location / { root /data/www; }", a request for "/index.html" serves the file "/data/www/index.html", and a request for "/images/logo.png" serves "/data/www/images/logo.png". The "root" directive appends the URL path to the specified directory, so the directory structure on disk mirrors the URL structure. NGINX automatically serves "index.html" as the default file when a directory is requested (this can be configured with the "index" directive).

### The root and alias Directives
The "root" and "alias" directives both map URL paths to filesystem paths, but they work differently. With "root", the URL path is appended to the root directory. For example, "location /app/ { root /data/www; }" maps "/app/index.html" to "/data/www/app/index.html". With "alias", the URL path is replaced by the alias directory. For example, "location /app/ { alias /data/other/; }" maps "/app/index.html" to "/data/other/index.html". The "alias" directive is useful when the URL path does not correspond to the filesystem directory structure. The source material demonstrates both: using "root" for the main location and "alias" for a different path that maps to a different filesystem directory. When using "alias", the location path must end with a slash, and the alias path should also end with a slash for consistent behavior.

### Multiple Location Blocks
NGINX allows multiple location blocks within a single server block, each handling a different URL path pattern. This enables serving different types of content from different directories. For example, the source material shows a configuration with "location /" serving files from "/data/www" and "location /server1" serving files from a different directory. Location blocks are matched in a specific order of precedence: exact matches ("location = /path") have the highest priority, followed by preferential prefix matches ("location ^~ /path"), then regular expression matches ("location ~ /pattern" or "location ~* /pattern" for case-insensitive), and finally prefix matches ("location /path"). Understanding this precedence is important for avoiding unexpected routing behavior when multiple location blocks could match a request.

### File Type Handling and MIME Types
NGINX uses the "include /etc/nginx/mime.types" directive (typically included in the http context) to map file extensions to MIME types, ensuring that the correct Content-Type header is sent with each response. Without this mapping, all files would be served with the default MIME type (application/octet-stream), which causes browsers to download files instead of rendering them. The "default_type" directive sets the MIME type for files that do not match any entry in the mime.types file. NGINX can also be configured to compress static files using gzip with the "gzip on" directive and related settings like "gzip_types" (file types to compress), "gzip_min_length" (minimum file size for compression), and "gzip_comp_level" (compression level). Compression significantly reduces the size of text-based files (HTML, CSS, JavaScript, JSON) and improves page load times.

### Caching and Performance Optimization
NGINX provides several directives for optimizing static content delivery. The "sendfile on" directive enables the use of the operating system's sendfile() system call, which transfers files directly from the filesystem to the network socket without copying data through user space, significantly reducing CPU usage. The "tcp_nopush on" directive optimizes the sending of HTTP response headers by combining them into a single packet. The "tcp_nodely on" directive disables Nagle's algorithm, reducing latency for small packets. For client-side caching, NGINX can set cache control headers using the "expires" directive (for example, "expires 30d" sets the Cache-Control and Expires headers to cache the file for 30 days). The "add_header Cache-Control" directive provides more granular control over caching behavior. ETags are generated automatically by NGINX, enabling conditional requests and reducing unnecessary data transfer.

## Practical Applications
Static websites and blogs are commonly served directly by NGINX without any application server. Single-page applications (SPAs) built with React, Angular, or Vue are served as static files by NGINX, with all routing handled client-side. Media serving applications use NGINX to deliver images, videos, and audio files with optimized caching and compression. Software download sites use NGINX to serve large binary files with rate limiting and resume support. Documentation sites are generated as static HTML and served by NGINX for maximum performance and minimal infrastructure cost. The source material demonstrates the most basic static content configuration, which is the starting point for all NGINX deployments.
