# IP Hashing and Session Persistence

## Overview
IP hashing is a load balancing algorithm that uses the client's IP address to determine which backend server should handle the request, providing session persistence (also called sticky sessions or session affinity). Unlike Round Robin or Least Connections, which may route the same client's requests to different servers on each request, IP hashing ensures that all requests from a given client IP are consistently directed to the same backend server. This is critical for applications that maintain local state on the server, such as shopping cart data, WebSocket connections, or session data stored in server-local memory. Understanding IP hashing and session persistence is essential for system designers because the choice between stateless and stateful architectures directly affects the load balancing strategy and the system's scalability characteristics.

## Key Concepts

### How IP Hashing Works
IP hashing computes a hash of the client's IP address and maps the hash value to one of the backend servers. The simplest implementation uses modulo arithmetic: hash(client_ip) % number_of_servers = server_index. For example, if there are three servers and the hash of a client's IP modulo 3 equals 1, the request is always sent to server 1. This provides deterministic routing: the same client IP always maps to the same server as long as the server pool does not change. In NGINX, this is configured using the "ip_hash" directive in the upstream block. When a server is added or removed from the pool, the modulo mapping changes, and some clients may be remapped to different servers, which can disrupt sessions. Consistent hashing (discussed separately) provides a more stable mapping when the server pool changes.

### Session Persistence Requirements
Session persistence is required when backend servers maintain state that is local to the server instance and not shared across the pool. Common examples include: in-memory session data (when sessions are not stored in a shared Redis or database), WebSocket connections (which require a persistent TCP connection to a specific server), server-side cached data (where the cache is local to the server instance), file uploads in progress (where the temporary file exists on one server), and legacy applications that were not designed for distributed deployment. When session persistence is required, the load balancer must ensure that all requests from a given client reach the same backend server for the duration of the session. IP hashing provides this by deterministically mapping the client IP to a specific server.

### Limitations of IP Hashing
IP hashing has several limitations. First, multiple users behind a NAT or corporate proxy share the same source IP address, so they are all mapped to the same backend server. This can create uneven load distribution because a large organization with many users may generate disproportionate traffic from a single IP. Second, clients with rotating IP addresses (such as mobile users switching between Wi-Fi and cellular) may be remapped to different servers, breaking session persistence. Third, adding or removing servers from the pool causes the hash mapping to change, potentially disrupting sessions for many clients simultaneously. Fourth, the hash distribution may not be perfectly uniform, especially with a small number of servers, leading to some servers receiving more traffic than others. These limitations make IP hashing best suited as a simple solution for session persistence, with more sophisticated methods preferred when higher precision is required.

### Alternative Session Persistence Methods
Beyond IP hashing, several alternative methods provide session persistence with different tradeoffs. Cookie-based persistence (also called insert or rewrite mode) uses the load balancer to set a cookie containing the backend server identifier. On subsequent requests, the load balancer reads the cookie and routes the request to the designated server. This provides more precise persistence than IP hashing because it operates at the individual client level rather than the IP level. However, it requires cookie support (which is not available for all types of traffic) and increases the request size. SSL session ID persistence routes clients with the same SSL session ID to the same backend, which works for HTTPS traffic without cookies. Custom header persistence uses an application-defined header for routing decisions. Application-level persistence uses a shared session store (like Redis) to make any server capable of serving any session, eliminating the need for load balancer-level persistence entirely.

### Best Practices for Session Persistence
The best practice for modern architectures is to design applications to be stateless, eliminating the need for session persistence at the load balancer level. State that must persist across requests should be stored in a shared data store (like Redis, Memcached, or a database) that all backend servers can access. This allows any server to serve any request, enabling true horizontal scaling and making the load balancer's routing decision irrelevant. However, when session persistence is unavoidable (for example, for WebSocket connections or legacy applications), IP hashing provides a simple and effective solution. When using IP hashing, the system should be designed to handle session disruption gracefully when servers are added or removed, and monitoring should track the distribution of clients across servers to detect imbalances.

## Practical Applications
NGINX provides the ip_hash directive for upstream blocks, ensuring that requests from the same client IP are routed to the same backend server. AWS ALB supports sticky sessions through application-controlled cookies or load-balancer-generated cookies. HAProxy provides multiple persistence methods including cookie-based, source IP-based, and stick table-based persistence. E-commerce applications use session persistence to maintain shopping cart state when sessions are stored in server-local memory. Real-time applications using WebSockets require session persistence to maintain the persistent connection between the client and a specific server. Gaming applications use IP hashing to route players to the same game server for the duration of a session.
