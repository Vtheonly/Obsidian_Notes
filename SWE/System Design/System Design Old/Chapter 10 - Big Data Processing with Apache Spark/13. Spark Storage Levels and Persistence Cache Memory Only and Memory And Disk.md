# Spark Storage Levels and Persistence Cache Memory Only and Memory And Disk

## Overview

Spark provides several storage levels that control how RDDs and DataFrames are persisted across the cluster, balancing the tradeoff between memory usage, CPU efficiency, and fault tolerance. When you call `persist` or `cache` on an RDD, Spark attempts to keep its partitions in memory across subsequent actions, avoiding the need to recompute the RDD from its lineage each time it is referenced. The choice of storage level determines whether data is kept in memory only, spilled to disk when memory is insufficient, replicated across multiple nodes for fault tolerance, or serialized for compact storage. Understanding storage levels is essential for optimizing the performance of iterative algorithms and multi-action Spark applications that reuse the same dataset.

## Key Concepts

### Cache versus Persist

The `cache` method is a shorthand for `persist(StorageLevel.MEMORY_ONLY)`, which stores the RDD as deserialized Java objects in memory. If the RDD does not fit in memory, some partitions will not be stored and will be recomputed from the lineage each time they are needed. The `persist` method provides more granular control, allowing you to specify any storage level. Both methods are lazy: the data is not actually cached until the first action that references the RDD is executed. Once cached, the RDD remains in storage until it is explicitly unpersisted or until the Spark application terminates. You should always cache RDDs that are reused across multiple actions to avoid redundant computation.

### MEMORY_ONLY

The `MEMORY_ONLY` storage level stores the RDD as deserialized Java objects in the JVM heap. This is the fastest storage level for reading, because the data is already in object form and does not require deserialization. However, it consumes the most memory, because Java objects have significant overhead compared to their serialized representation. If the RDD does not fit entirely in memory, the partitions that cannot fit are not stored and will be recomputed from the lineage when needed. This means that for RDDs larger than available memory, MEMORY_ONLY may result in frequent recomputation, which can be more expensive than using a storage level that spills to disk.

### MEMORY_AND_DISK

The `MEMORY_AND_DISK` storage level stores the RDD as deserialized Java objects in memory, and spills partitions that do not fit in memory to local disk. When a partition is needed, Spark first checks if it is in memory; if not, it reads it from disk. This provides a good balance between performance and reliability: partitions that fit in memory are accessed quickly, while partitions that spill to disk are still available without recomputation. MEMORY_AND_DISK is the recommended storage level for most production workloads, because it provides deterministic performance regardless of whether the dataset fits in memory, unlike MEMORY_ONLY which degrades unpredictably when memory is insufficient.

### MEMORY_ONLY_SER and MEMORY_AND_DISK_SER

The serialized storage levels, `MEMORY_ONLY_SER` and `MEMORY_AND_DISK_SER`, store the RDD as serialized byte arrays rather than deserialized Java objects. Serialization reduces memory consumption significantly, often by a factor of two to five, because serialized byte arrays are much more compact than Java object graphs. The tradeoff is that reading the data requires deserialization, which adds CPU overhead. When using serialized storage levels, choosing an efficient serialization format is important: Kryo serialization is much faster and more compact than Java serialization and is the recommended choice for most Spark applications. To enable Kryo, set `spark.serializer=org.apache.spark.serializer.KryoSerializer`.

### DISK_ONLY

The `DISK_ONLY` storage level stores the RDD entirely on local disk, without keeping any partitions in memory. This is useful when the dataset is too large to fit in memory but you want to avoid recomputing it from the lineage each time it is referenced. Reading from disk is slower than reading from memory but faster than recomputing from the lineage, especially when the lineage is deep or involves expensive operations. DISK_ONLY is rarely the optimal choice for production workloads, because the disk read overhead is typically comparable to the recompute overhead for simple transformations. However, it can be useful for very expensive computations that produce relatively small results.

### Replication Storage Levels

The `_2` variants of storage levels, such as `MEMORY_ONLY_2` and `MEMORY_AND_DISK_2`, replicate each partition on two cluster nodes instead of one. This provides fault tolerance at the storage level: if one node fails, the replicated copy on the other node is still available, and Spark can continue processing without recomputing the lost partition from the lineage. The tradeoff is that replication doubles the memory and disk consumption of the cached RDD. Replication is most useful for RDDs with expensive lineages that would be costly to recompute, or for streaming applications where recomputation latency is unacceptable. For most batch workloads, the default replication factor of one is sufficient because Spark's lineage-based recovery is fast enough.

## Practical Applications

A machine learning pipeline that trains a model iteratively over 100 iterations caches the training dataset with `MEMORY_AND_DISK`, ensuring that the data is read from memory on each iteration rather than recomputed from the source. If the dataset exceeds memory, partitions spill to disk but are still available without recomputation. A graph analytics application that repeatedly traverses the same graph structure caches the graph RDD with `MEMORY_ONLY_SER` using Kryo serialization, reducing memory consumption by 3x compared to deserialized storage while maintaining fast access. A data engineering pipeline that reads a slowly changing reference dataset and joins it with multiple event streams caches the reference dataset with `MEMORY_AND_DISK_2`, providing fault tolerance to avoid re-reading the reference data from the source system if an executor fails.
