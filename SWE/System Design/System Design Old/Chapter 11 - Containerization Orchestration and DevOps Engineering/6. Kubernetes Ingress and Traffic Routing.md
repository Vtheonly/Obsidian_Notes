# Kubernetes Ingress and Traffic Routing

## Overview

Kubernetes Ingress is an API object that manages external access to the services in a cluster, typically providing HTTP and HTTPS routing, TLS termination, and name-based virtual hosting. While Kubernetes Services provide stable internal networking, Ingress is the mechanism by which external traffic from the internet reaches the appropriate internal services. An Ingress controller -- such as Traefik, NGINX, or HAProxy -- watches for Ingress resources and dynamically configures the underlying proxy to route traffic according to the defined rules. This decoupling of routing configuration from the proxy implementation is a powerful pattern that enables automated, dynamic traffic management in microservices architectures.

## Key Concepts

### Ingress Resources and Rules

An Ingress resource defines a set of routing rules that map external requests to internal Services based on hostnames, paths, or both. For example, an Ingress rule might specify that requests to `api.example.com` should be routed to the API Service, while requests to `admin.example.com` should go to the back-office Service. Path-based routing allows further subdivision: `api.example.com/users` might route to the user Service, while `api.example.com/orders` routes to the order Service. These rules are expressed declaratively in YAML, version-controlled alongside application code, and automatically enforced by the Ingress controller.

### Ingress Controllers

An Ingress resource is simply a configuration object -- it has no effect without an Ingress controller running in the cluster. The Ingress controller is responsible for watching Ingress resources and configuring the underlying reverse proxy accordingly. Popular controllers include Traefik, NGINX Ingress Controller, and HAProxy Ingress. Each controller offers different features: Traefik is known for its automatic configuration and native support for multiple backends (Docker, Kubernetes, Mesos, Consul), NGINX offers battle-tested performance and extensive configuration options, and HAProxy provides advanced load-balancing algorithms. The choice of controller depends on the specific requirements of the deployment.

### Traefik as Ingress Controller

Traefik is a modern reverse proxy and load balancer that was specifically designed for dynamic, containerized environments. Unlike traditional reverse proxies that require static configuration files -- which can grow to hundreds of lines and become unmanageable -- Traefik listens to the APIs provided by orchestration systems and service discovery mechanisms, automatically configuring itself as services are added, removed, or scaled. Traefik supports Docker, Docker Swarm, Kubernetes, Mesos/Marathon, Consul, etcd, and Zookeeper as backends. It provides approximately 85% of the raw performance of NGINX while offering significantly more flexibility and automation. Additional features include circuit breakers, retry logic, rate limiting, CORS handling, WebSocket support, HTTP/2, SSL termination, and built-in Let's Encrypt support.

### TLS Termination and Certificate Management

Ingress controllers typically handle TLS termination, decrypting HTTPS traffic at the cluster edge and forwarding unencrypted traffic to internal Services. This centralizes certificate management and reduces the burden on individual services. Traefik's built-in Let's Encrypt support automates the entire certificate lifecycle: it can automatically provision, renew, and distribute TLS certificates based on the hostnames defined in Ingress resources. This eliminates the manual process of obtaining and renewing certificates, which is a significant operational burden at scale.

### The Problem with Static Configuration

In legacy architectures without containers and orchestration, reverse proxy configuration is often managed through large, static configuration files. As demonstrated by Reevoo's experience with Varnish, these files can grow to 465 lines or more, with complex routing rules that are difficult to understand and maintain. Deploying changes to these configurations requires ensuring that all backend servers exist with proper DNS setup, creating a tight coupling between the proxy configuration and the infrastructure state. This approach does not scale well with microservices, where services are created, destroyed, and scaled dynamically. Tools that attempt to automatically generate proxy configuration files from service metadata exist, but they often lack the robustness and testing needed for production use.

## Practical Applications

At Reevoo, the transition from a static Varnish configuration to Traefik as an Ingress controller dramatically simplified their traffic routing. By deploying Traefik to Kubernetes and defining Ingress resources for each service, the team eliminated the need for manual proxy configuration. When new Pods were deployed for the cheddar, stilton, or wensleydale services, Traefik automatically detected them through the Kubernetes API and began routing traffic to them without any manual intervention. This automation is critical in microservices environments where the number and location of service instances change frequently.
