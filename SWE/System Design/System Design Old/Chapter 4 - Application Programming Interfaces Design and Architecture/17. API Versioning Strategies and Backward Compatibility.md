# API Versioning Strategies and Backward Compatibility

## Overview
API versioning is the practice of managing changes to an API over time while maintaining compatibility with existing consumers. As APIs evolve, new features are added, existing behaviors are modified, and obsolete functionality is retired. Without a versioning strategy, changes to an API can break existing clients, causing outages and eroding consumer trust. A well-designed versioning strategy enables APIs to evolve safely, provides clear migration paths for consumers, and balances the cost of maintaining multiple versions against the risk of breaking changes.

## Key Concepts

### What Constitutes a Breaking Change
A breaking change is any modification that can cause existing consumers to fail. Removing a field from a response, changing a field's data type, renaming an endpoint, removing an endpoint, adding a required field to a request, and changing error response formats are all breaking changes. Non-breaking changes include adding new optional fields to responses, adding new endpoints, adding new enum values (if clients handle unknown values), and adding optional query parameters. Understanding this distinction is fundamental because versioning is primarily needed to manage breaking changes gracefully.

### URL Path Versioning
URL path versioning includes the version number in the URL path: /v1/orders, /v2/orders. This is the most visible and commonly used versioning strategy. It is easy to understand, easy to implement, and works well with API gateways and routing rules. Consumers explicitly opt into a version by choosing the URL, and different versions can be routed to different backend services. The main disadvantage is that URL versioning implies that different versions are entirely separate APIs, which can lead to code duplication and maintenance burden if the differences between versions are minor.

### Header Versioning
Header versioning uses a custom HTTP header to specify the API version. Common header names include API-Version, Accept (with a vendor MIME type like application/vnd.company.v2+json), and X-API-Version. Header versioning keeps URLs clean and allows the same URL to serve multiple versions, with the server selecting the response format based on the header. This approach is less visible than URL versioning, which can make debugging more difficult, and it requires consumers to include the header in every request.

### Query Parameter Versioning
Query parameter versioning uses a query parameter to specify the version: /orders?version=2. This is the simplest approach to implement but has several drawbacks. Query parameters are easily overlooked, they do not integrate well with caching (since caches may not differentiate based on query parameters), and they can be confused with functional query parameters. This approach is generally discouraged for production APIs but may be acceptable for internal or low-traffic APIs.

### Backward-Compatible Evolution
The most effective versioning strategy is to avoid versioning altogether by making only backward-compatible changes. New fields can be added to responses without breaking existing consumers, as long as consumers ignore unknown fields. New optional fields can be added to requests with sensible defaults. New endpoints can be added without affecting existing ones. This additive approach to API evolution, known as the Robustness Principle or Postel's Law, allows APIs to grow without requiring consumers to update. GraphQL naturally supports this approach because adding fields to types is non-breaking.

### Maintaining Multiple Versions
When breaking changes are unavoidable, maintaining multiple API versions becomes necessary. Each active version requires its own deployment, testing, documentation, and monitoring. The maintenance cost grows linearly with the number of active versions. To manage this cost, organizations should limit the number of simultaneously supported versions (typically 2-3) and enforce deprecation timelines that encourage consumers to migrate promptly. Shared libraries that encapsulate common logic between versions can reduce code duplication, and API gateways that transform requests between versions can simplify backend implementations.

### Consumer Migration Strategies
Migrating consumers from one API version to another requires careful planning and communication. Migration guides should document all changes between versions with before-and-after examples. Dual-running, where the consumer uses both versions simultaneously during the migration, allows for gradual switchover. Shadow testing, where the consumer sends requests to both versions and compares responses, validates the migration before cutting over. Feature flags in the consumer application can toggle between versions, enabling instant rollback if problems are discovered. Throughout the migration, the deprecated version must remain fully functional until all consumers have migrated.

## Practical Applications
A payment processing API uses URL path versioning with two active versions: v1 and v2. Version 2 introduced a breaking change where the amount field switched from a string to a structured object with value and currency fields. The API team published a migration guide 6 months before the v1 sunset date, added Deprecation headers to v1 responses, and sent email notifications to all registered API consumers. During the migration period, both versions were fully functional and monitored for errors. After 12 months, v1 traffic dropped below 1% and the team sunset v1 with a final 30-day notice.
