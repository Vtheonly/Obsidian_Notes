# Resource Modeling: Nouns versus Verbs

## Overview
Resource modeling is the process of identifying, naming, and structuring the entities that an API exposes. In RESTful API design, resources are represented as nouns in the URL path, while actions are expressed through HTTP methods. This noun-centric approach contrasts with RPC-style APIs, where URLs represent verbs or actions. Understanding the distinction between noun-based and verb-based URL design is critical for creating APIs that are intuitive, consistent, and aligned with REST principles.

## Key Concepts

### Resources as Nouns
In REST, every URL should identify a resource, which is a noun. Resources represent entities in your domain model: users, products, orders, invoices, comments. URLs are structured hierarchically to reflect relationships, such as /users/123/orders for the orders belonging to user 123. Using nouns in URLs makes the API intuitive because it mirrors how people think about domain objects. A developer seeing /products/456 immediately understands that this URL refers to a specific product, regardless of whether they intend to retrieve, update, or delete it.

### Actions as HTTP Methods
Actions on resources are expressed through HTTP methods, not through URL verbs. GET retrieves a resource, POST creates a new resource, PUT replaces a resource, PATCH partially updates a resource, and DELETE removes a resource. This separation of the resource identity (the URL) from the action (the method) allows a single URL to support multiple operations, making the API concise and consistent. For example, GET /orders/789 retrieves the order, while DELETE /orders/789 cancels it.

### When Verbs Are Necessary
Some operations do not map cleanly to CRUD actions on resources. Examples include sending an email, processing a payment, calculating a quote, or running a report. These operations are inherently action-oriented rather than resource-oriented. The recommended approach is to model these as resources that represent the action itself. For example, POST /emails creates an email resource (sending an email), POST /payments processes a payment, and POST /quotes generates a quote. By treating the action as a resource creation, you maintain consistency with REST principles while accommodating procedural operations.

### Avoiding Anti-Patterns
Common anti-patterns include using verbs in URLs (GET /getUser?id=123, POST /createOrder, DELETE /deleteProduct), embedding action names in paths (/users/123/activate), and using URLs that describe the technology rather than the domain (/json/users). These patterns break REST conventions, make APIs inconsistent, and confuse developers who expect resources to be nouns and actions to be HTTP methods. Rewriting these as GET /users/123, POST /orders, and DELETE /products/456 produces cleaner, more predictable APIs.

### Nested and Compound Resources
Complex domain relationships often require nested resource paths. /users/123/addresses represents the addresses belonging to user 123, and /orders/789/items represents the line items in order 789. Nesting should be limited to two levels to keep URLs manageable. For deeper relationships, use query parameters or separate top-level resources with filters (/items?order_id=789). Compound resources that aggregate data from multiple entities can be modeled as dedicated resources (/dashboard, /reports/sales-summary) that represent the aggregated view rather than the underlying data.

### Consistency and Naming Conventions
Consistent naming conventions are essential for API usability. Use plural nouns for collections (/users, not /user). Use lowercase with hyphens for multi-word names (/user-profiles, not /userProfiles). Avoid mixing singular and plural forms. Ensure that related resources follow the same patterns so that developers can predict URL structures without consulting documentation for every endpoint. Consistency reduces the learning curve and minimizes errors.

## Practical Applications
Designing a project management API, you would model resources as /projects, /projects/{id}/tasks, /tasks/{id}/comments, and /users. Creating a task is POST /projects/{id}/tasks, reassigning a task is PATCH /tasks/{id} with an assignee field, and completing a task is PATCH /tasks/{id} with a status field set to "completed". If you need a bulk operation like archiving multiple projects, you would POST /bulk-archive-requests with the list of project IDs, treating the bulk archive as a resource that can be created and tracked.
