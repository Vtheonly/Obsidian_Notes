# HTTP Status Codes: Success, Redirection, Client Error, and Server Error

## Overview
HTTP status codes are three-digit numbers returned by a server in response to a client's request, indicating the outcome of the operation. Status codes are grouped into five categories, each serving a distinct purpose in communicating the result of an API call. Proper use of status codes is one of the most important aspects of RESTful API design, as it allows clients to programmatically handle responses without parsing the response body, enabling consistent error handling and flow control across the entire API.

## Key Concepts

### 2xx Success Codes
The 2xx range indicates that the client's request was successfully received, understood, and accepted. 200 OK is the standard success response for GET, PUT, PATCH, and DELETE operations. 201 Created indicates that a new resource was successfully created, typically in response to a POST request, and should be accompanied by a Location header pointing to the new resource. 202 Accepted indicates that the request has been accepted for processing but has not yet been completed, used for asynchronous operations. 204 No Content indicates a successful operation with no response body, commonly returned after a DELETE or a PUT that does not return the updated resource.

### 3xx Redirection Codes
The 3xx range indicates that the client must take additional action to complete the request. 301 Moved Permanently indicates that the resource has been permanently moved to a new URL, and the client should update its bookmarks. 302 Found indicates a temporary redirect, where the client should continue using the original URL for future requests. 304 Not Modified indicates that the resource has not changed since the last request (based on If-None-Match or If-Modified-Since headers), allowing the client to use its cached copy. Proper use of 304 with ETag and Last-Modified headers significantly reduces bandwidth and improves performance for cacheable resources.

### 4xx Client Error Codes
The 4xx range indicates that the client made an error. 400 Bad Request indicates malformed syntax or invalid parameters. 401 Unauthorized indicates that authentication is required but has not been provided or has failed. 403 Forbidden indicates that the authenticated user does not have permission to access the resource. 404 Not Found indicates that the requested resource does not exist. 409 Conflict indicates that the request conflicts with the current state of the server, such as attempting to create a resource that already exists. 422 Unprocessable Entity indicates that the server understands the content type and syntax but cannot process the instructions, typically used for validation errors. 429 Too Many Requests indicates that the client has exceeded its rate limit.

### 5xx Server Error Codes
The 5xx range indicates that the server failed to fulfill a valid request. 500 Internal Server Error is a generic error indicating an unexpected condition on the server. 502 Bad Gateway indicates that the server, acting as a gateway, received an invalid response from an upstream server. 503 Service Unavailable indicates that the server is temporarily unable to handle the request due to maintenance, overload, or other transient conditions. 504 Gateway Timeout indicates that the server, acting as a gateway, did not receive a timely response from an upstream server. 5xx errors should be monitored closely as they indicate problems that the client cannot resolve.

### Choosing the Right Status Code
Selecting the correct status code requires careful consideration of the error condition. A missing authentication token should return 401, not 403. An authenticated user without permissions should return 403, not 401. A syntactically valid but semantically invalid request body should return 422, not 400. A resource that does not exist should return 404, not 200 with an error message in the body. Consistent and precise status code usage reduces ambiguity and makes APIs easier to consume and debug.

### Error Response Bodies
Status codes alone are often insufficient for effective error handling. Best practice is to return a structured error body that includes a machine-readable error code, a human-readable message, and optional details about which fields are invalid or what constraints were violated. For example, a 422 response might include a list of validation errors with field names and descriptions. This structured approach enables clients to display specific error messages to users and programmatically handle different error conditions.

## Practical Applications
When a client submits an order with an invalid credit card, the API returns 422 with a body detailing that the payment_method field is invalid. When the server's database is temporarily unavailable, it returns 503 with a Retry-After header. When a client exceeds the rate limit, the API returns 429 with headers like X-RateLimit-Limit, X-RateLimit-Remaining, and X-RateLimit-Reset. Monitoring dashboards alert on spikes in 5xx responses, which indicate infrastructure problems, while tracking 4xx responses helps identify client-side issues or API misuse.
