# API Query Parameters: Filtering and Sorting

## Overview
Query parameters are the primary mechanism for customizing API responses, enabling clients to filter, sort, and shape the data returned by a resource endpoint. Without query parameters, a GET request to a collection endpoint would return all records, which is impractical for large datasets. Well-designed filtering and sorting parameters give clients the flexibility to retrieve exactly the data they need while keeping the API intuitive and consistent. Proper design of these parameters is essential for building APIs that are both powerful and easy to use.

## Key Concepts

### Filtering with Query Parameters
Filtering allows clients to restrict results to records that match specific criteria. The simplest approach uses direct field matching: GET /products?category=electronics&in_stock=true returns only products that are both in the electronics category and in stock. For numeric and date fields, range filters are essential: GET /products?price_min=10&price_max=100 or GET /orders?created_after=2024-01-01. These parameter naming conventions should be consistent across the entire API to reduce the learning curve for developers.

### Advanced Filtering Patterns
Complex filtering requirements may need operators beyond simple equality and range. Common patterns include partial string matching (name_contains=headphones), set membership (status=in:draft,published), null checks (discount_is_null=true), and comparison operators (price_gt=50, created_before=2024-06-01). Some APIs implement a filter language using a structured query syntax like OData ($filter=price gt 50 and category eq 'electronics') or GraphQL-style filter objects. The key trade-off is between expressiveness and simplicity: highly expressive filter languages handle complex queries but are harder to learn and implement.

### Sorting Parameters
Sorting specifies the order in which results are returned. The most common convention uses a sort parameter with field names: GET /products?sort=price returns products sorted by price in ascending order, while sort=-price indicates descending order. Multiple sort fields can be separated by commas: sort=-created_at,name sorts by creation date descending, then by name ascending. The API should document which fields are sortable, as sorting on unindexed database fields can cause severe performance degradation for large datasets.

### Field Selection
Field selection parameters allow clients to specify which fields to include in the response, reducing payload size and improving performance for bandwidth-constrained clients. A common convention uses a fields parameter: GET /products?fields=id,name,price returns only the specified fields. This is particularly valuable for mobile clients and when resources have many fields that are not always needed. Field selection can also reduce database query time if the backend only fetches the requested columns.

### Combining Filters, Sorting, and Pagination
In practice, filtering, sorting, and pagination are almost always used together. A typical request might look like GET /products?category=electronics&sort=-price&limit=20&offset=0. The server applies filters first to determine the total matching set, then sorts the filtered results, and finally applies pagination to return the requested page. The response should include both the paginated data and metadata about the total number of matching records, enabling clients to render pagination controls accurately.

### Security Considerations
Query parameters appear in URLs and are logged by web servers, proxies, and browser history, so sensitive data like passwords or tokens should never be passed as query parameters. Additionally, allowing arbitrary filtering on database fields can expose internal schema details or enable denial-of-service attacks through computationally expensive queries. APIs should whitelist allowed filter and sort fields, validate parameter values, and set sensible defaults and maximum limits for result set sizes.

## Practical Applications
A product catalog API supports filtering by category, price range, and availability; sorting by price, name, or relevance; and pagination with cursor-based navigation. A customer support ticket API allows filtering by status, priority, assignee, and creation date, with sorting by last updated timestamp. Each filter combination enables a different view of the data that supports a specific workflow, from browsing products to triaging support tickets.
