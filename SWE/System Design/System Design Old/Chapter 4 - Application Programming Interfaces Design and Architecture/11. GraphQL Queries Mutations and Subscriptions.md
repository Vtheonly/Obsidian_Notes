# GraphQL Queries, Mutations, and Subscriptions

## Overview
GraphQL defines three root operation types that form the foundation of all client-server interactions: queries for reading data, mutations for writing data, and subscriptions for receiving real-time updates. Each operation type has distinct semantics and execution guarantees that shape how clients interact with the API. Understanding the differences between these operations, their execution model, and their practical trade-offs is essential for designing GraphQL APIs that are correct, efficient, and maintainable.

## Key Concepts

### Queries
Queries are the read operations of GraphQL, analogous to GET requests in REST. Queries are declarative: the client specifies the shape of the data it needs, and the server returns data in exactly that shape. Queries can be nested to any depth, allowing clients to fetch related data in a single request. For example, a query can fetch a user along with their posts and each post's comments in one round trip. Queries can include arguments for filtering, sorting, and pagination, and they can use aliases to request the same field multiple times with different arguments. Fragments allow query patterns to be reused across multiple queries, reducing duplication.

### Query Execution Model
When the server receives a query, the execution engine resolves each field by calling its resolver function. Fields without explicit resolvers use default resolvers that return the property of the parent object with the same name. The execution engine resolves fields on the same level concurrently, meaning sibling fields can be fetched in parallel. Nested fields are resolved sequentially, as each level depends on the results of the previous level. This execution model means that deeply nested queries can be slow if each level requires a separate database query or API call. DataLoader and similar batching tools address this by collecting and batching resolver calls within a single execution cycle.

### Mutations
Mutations are the write operations of GraphQL, analogous to POST, PUT, PATCH, and DELETE in REST. Unlike queries, mutations have important execution guarantees: the server processes mutations sequentially, not concurrently. This ensures that if a client sends multiple mutations in a single request, they are executed in the order they appear, preventing race conditions between dependent mutations. A mutation should return the modified data so that the client can update its local cache without making a follow-up query. For example, a createTask mutation should return the newly created Task object, including the server-generated id field.

### Mutation Payloads
Best practice for mutation design uses a dedicated payload type for each mutation that includes the modified data along with optional metadata. A CreateTaskPayload might include the created task, any validation errors, and the updated project to which the task was added. This pattern allows clients to update multiple cached objects in a single response, which is particularly important for Apollo Client and other GraphQL clients that normalize and cache data by type and ID.

### Subscriptions
Subscriptions enable real-time data delivery using a pub-sub model. When a client subscribes to an event, the server pushes updates to the client whenever the relevant data changes. Subscriptions are typically implemented over WebSocket connections, though the transport is not specified by the GraphQL specification. Common use cases include live chat messages, real-time dashboard updates, collaborative document editing, and notifications. Subscriptions use the same schema and resolver model as queries and mutations, making them consistent with the rest of the API.

### Subscription Execution and Scalability
Subscriptions maintain long-lived connections between the client and server, which poses scalability challenges. Each active subscription consumes server memory and a WebSocket connection. To scale horizontally, subscription state must be shared across server instances using a pub-sub broker like Redis or a message queue. When an event occurs, it is published to the broker, and all server instances that hold subscriptions for that event forward the update to their connected clients. Tools like Apollo Server and GraphQL WebSocket handle much of this complexity, but architects must plan for the resource requirements of sustained subscription connections.

### Operation Complexity and Depth Limiting
GraphQL's flexibility allows clients to construct arbitrarily complex queries, which can overwhelm the server. Deeply nested queries, queries with many fields, and recursive fragments can cause excessive computation and database load. To protect the server, APIs implement query complexity analysis, depth limiting, and rate limiting. Query complexity assigns a cost to each field and rejects queries that exceed a maximum cost. Depth limiting restricts the maximum nesting depth of queries. Persisted queries, which only allow pre-approved query strings, provide the strongest protection but reduce client flexibility.

## Practical Applications
A real-time collaboration tool uses queries to load the initial document state, mutations to apply edits, and subscriptions to receive edits from other users in real time. When a user types a character, a mutation is sent to the server, which validates the edit, persists it, and publishes a subscription event. Other users' clients receive the event through their subscriptions and update the document display. This combination of queries, mutations, and subscriptions creates a seamless real-time experience.
