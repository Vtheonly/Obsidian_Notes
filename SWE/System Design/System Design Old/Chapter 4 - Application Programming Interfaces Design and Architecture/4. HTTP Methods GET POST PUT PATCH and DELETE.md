# HTTP Methods: GET, POST, PUT, PATCH, and DELETE

## Overview
HTTP methods define the action that a client wishes to perform on a resource. In RESTful API design, these methods map directly to the fundamental CRUD operations: Create, Read, Update, and Delete. Understanding the semantics, idempotency, safety, and proper use of each HTTP method is essential for building APIs that behave predictably and conform to web standards. Misusing HTTP methods leads to APIs that are confusing, difficult to cache, and incompatible with standard HTTP infrastructure.

## Key Concepts

### GET
The GET method requests a representation of a specified resource. GET requests are safe, meaning they should not modify any server state, and they are idempotent, meaning multiple identical GET requests produce the same result. GET requests should never be used for creating, updating, or deleting resources, even if the implementation technically performs those actions. GET requests may include query parameters for filtering, sorting, and pagination, but they must not include a request body according to HTTP specifications (though some implementations tolerate it). Responses to GET requests are cacheable by default, making GET ideal for read-heavy operations.

### POST
The POST method submits data to be processed by the target resource. POST is neither safe nor idempotent. Each POST request typically creates a new resource, so sending the same POST request twice creates two separate resources. POST is the most versatile HTTP method and is used for creating resources, submitting forms, triggering actions, and any operation that does not fit the semantics of other methods. The URL for a POST request typically targets a collection resource (/orders), and the response should include a 201 Created status with a Location header pointing to the newly created resource.

### PUT
The PUT method replaces the entire target resource with the provided payload. PUT is idempotent, meaning sending the same PUT request multiple times produces the same result as sending it once. If the resource does not exist, PUT may create it. If it does exist, PUT completely replaces it with the new data, discarding any fields not included in the request. This full replacement behavior requires clients to send the complete resource representation, which can be inefficient for small updates. The URL for a PUT request typically targets a specific resource instance (/orders/123).

### PATCH
The PATCH method applies partial modifications to a resource. Unlike PUT, which replaces the entire resource, PATCH updates only the fields specified in the request body. PATCH is not guaranteed to be idempotent, though it can be if implemented carefully using strategies like JSON Merge Patch (RFC 7396) or JSON Patch (RFC 6902). JSON Merge Patch replaces specified fields with new values, which is idempotent for simple field updates. JSON Patch describes a sequence of operations (add, remove, replace, move, copy, test), where operations like "add" to an array are not idempotent. PATCH is the preferred method for partial updates because it reduces bandwidth and avoids accidentally overwriting fields that the client did not intend to change.

### DELETE
The DELETE method removes the specified resource. DELETE is idempotent: deleting a resource that has already been deleted should return 404 Not Found or 204 No Content, but the resource state after the first and subsequent DELETE requests is the same (the resource no longer exists). A successful DELETE typically returns 204 No Content or 200 OK with a confirmation message. Soft delete implementations that mark resources as inactive rather than physically removing them still use the DELETE method but should document this behavior.

### OPTIONS and HEAD
OPTIONS returns the HTTP methods supported by a resource, used by browsers for CORS preflight requests. HEAD retrieves only the headers of a resource without the body, useful for checking if a resource exists, verifying cache freshness, or getting metadata like Content-Length without downloading the full resource. While less commonly used in API design, these methods are important for CORS compliance and cache optimization.

## Practical Applications
In a task management API, the following method usage would be standard: GET /tasks retrieves the task list, POST /tasks creates a new task, GET /tasks/42 retrieves a specific task, PUT /tasks/42 replaces the entire task, PATCH /tasks/42 updates specific fields like status or assignee, and DELETE /tasks/42 removes the task. Using PATCH for the common case of updating task status avoids the risk of accidentally clearing fields like the task description that might not be included in a PUT request.
