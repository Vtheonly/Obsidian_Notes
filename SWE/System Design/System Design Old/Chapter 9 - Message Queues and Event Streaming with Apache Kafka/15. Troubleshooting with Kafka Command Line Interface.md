# Troubleshooting with Kafka Command Line Interface

## Overview

Kafka provides a comprehensive set of command-line interface tools that are indispensable for troubleshooting, monitoring, and administering a Kafka cluster. These tools allow operators to inspect topic configurations, view consumer group offsets and lag, produce and consume test messages, examine partition leadership and replication status, and perform cluster management operations. While production monitoring typically relies on dedicated observability platforms, the CLI tools remain the first line of defense for debugging issues quickly and directly. Mastery of these tools is essential for anyone operating or developing against Kafka, because they provide immediate visibility into the state of the cluster and its clients without requiring any additional infrastructure.

## Key Concepts

### kafka-topics.sh

The `kafka-topics.sh` tool is the primary utility for managing Kafka topics. It supports creating topics with specified partition counts and replication factors, listing all topics in the cluster, describing a topic's configuration including its partition count, replication factor, and leader assignments, and deleting topics when they are no longer needed. The describe command is particularly useful for troubleshooting: it shows which broker leads each partition, which brokers host replicas, which replicas are in the in-sync replica set, and whether any replicas are offline. This information is critical for diagnosing under-replicated partitions, leader imbalances, and offline partitions that can cause consumer errors or reduced fault tolerance.

### kafka-console-producer.sh and kafka-console-consumer.sh

The console producer and consumer tools allow you to interactively produce and consume messages from a Kafka topic. The console producer reads lines from standard input and publishes them as events to a specified topic, while the console consumer subscribes to a topic and prints events to standard output. These tools are invaluable for quick debugging: you can verify that a topic is accepting writes, confirm that events are being produced in the expected format, test consumer logic by manually producing test events, and inspect the contents of a topic to verify that producers are generating correct data. The consumer tool supports reading from the beginning of a topic with the `--from-beginning` flag, enabling you to review historical events.

### kafka-consumer-groups.sh

The `kafka-consumer-groups.sh` tool provides visibility into consumer group state, which is essential for diagnosing consumer lag, rebalancing issues, and offset management problems. The `--describe` command shows each consumer group's current offset, the log end offset, and the lag for every partition it consumes from. The `--list` command shows all consumer groups registered in the cluster. The `--reset-offsets` command allows you to reset a consumer group's offsets to a specific position, which is useful for replaying events or skipping problematic events. The `--delete` command removes a consumer group, which is necessary when a group with stale offsets is preventing partition rebalancing.

### kafka-run-class.sh

The `kafka-run-class.sh` script is a utility that sets up the classpath and JVM parameters for running any Kafka class. It is used internally by all other Kafka CLI tools and can also be used directly to run less common tools or custom classes. For example, you can use it to run the `kafka.tools.GetOffsetShell` class to quickly check the end offsets of partitions, or the `kafka.tools.DumpLogSegments` class to inspect the contents of Kafka log segments on disk, including viewing record offsets, timestamps, and payloads. This low-level access is invaluable for diagnosing data corruption, understanding partition layout, and verifying that events are being written correctly.

### Common Troubleshooting Scenarios

One common scenario is a consumer that appears stuck and is not processing new events. Using `kafka-consumer-groups.sh --describe`, you can check the consumer's lag: if the lag is growing, the consumer is falling behind; if the offset is not advancing at all, the consumer may have crashed or lost its partition assignment. Another common scenario is under-replicated partitions, where one or more replicas are not keeping up with the leader. Using `kafka-topics.sh --describe`, you can identify which partitions are under-replicated and which brokers host the lagging replicas. A third scenario is a broker that has left the cluster: checking the broker's logs and the controller's logs can reveal whether the broker crashed, lost its Zookeeper or KRaft session, or was explicitly shut down.

### Diagnostic Workflow

When troubleshooting a Kafka issue, a systematic approach is most effective. First, check the cluster health using `kafka-topics.sh --describe` to identify any offline partitions or under-replicated replicas. Second, check consumer group lag using `kafka-consumer-groups.sh --describe` to determine whether consumers are keeping up with producers. Third, use the console consumer with `--from-beginning` to inspect the actual events in a topic and verify that producers are generating correct data. Fourth, check broker logs for error messages related to the issue. Fifth, verify network connectivity between clients and brokers using the console producer and consumer. This structured approach narrows down the problem space efficiently and avoids the trap of making changes without understanding the root cause.

## Practical Applications

When a payment processing consumer reports that it has stopped receiving events, an operator uses `kafka-consumer-groups.sh --describe` to check the consumer's offset and lag. They discover that the lag is zero, meaning no new events are being produced. They then use `kafka-topics.sh --describe` to check the topic and find that the partition leader is on a broker that recently went offline. They restart the broker, and the controller elects it as the leader again, restoring the flow of events. In another scenario, a developer uses `kafka-console-consumer.sh --from-beginning --topic orders` to verify that the order events contain the expected JSON structure, discovering that a recent producer deployment introduced a schema change that broke the consumer's deserialization logic.
