# Kafka Graceful Shutdown and Resource Cleanup

## Overview

Graceful shutdown of Kafka brokers and consumer applications is critical for maintaining data integrity and minimizing disruption in a Kafka cluster. When a broker shuts down gracefully, it ensures that all in-flight writes are persisted, partition leadership is transferred to other brokers, and the controller is notified so that partitions can be reassigned without prolonged unavailability. Similarly, when a consumer shuts down gracefully, it commits its current offsets, releases its partition assignments, and notifies the group coordinator so that partitions can be reassigned quickly. Understanding how to implement graceful shutdown prevents data loss, reduces consumer lag spikes, and maintains cluster stability during planned maintenance and scaling operations.

## Key Concepts

### Broker Graceful Shutdown

When a Kafka broker receives a SIGTERM signal, it initiates a controlled shutdown sequence. First, the broker notifies the controller that it is shutting down, allowing the controller to elect new leaders for all partitions currently led by this broker. This proactive leadership transfer is faster and smoother than waiting for the controller to detect the broker's absence through session timeout, which could take tens of seconds. Second, the broker flushes all pending writes to disk, ensuring that no data is lost. Third, the broker closes its network connections and releases its resources. The controlled shutdown is enabled by setting `controlled.shutdown.enable=true` and configured with `controlled.shutdown.max.retries` and `controlled.shutdown.retry.backoff.ms`.

### Partition Leadership Transfer

During a graceful broker shutdown, the broker requests the controller to move leadership of its partitions to other brokers. The controller selects the first in-sync replica for each partition as the new leader, ensuring that no data is lost during the transfer. This process is much faster than an ungraceful shutdown, where the controller must wait for the session timeout to expire before it can elect new leaders, a period that might be configured as 30 seconds or more. By proactively transferring leadership, graceful shutdown minimizes the window of partition unavailability from tens of seconds to a few seconds.

### Consumer Graceful Shutdown

Consumer applications must handle shutdown gracefully to ensure that all processed records have their offsets committed and that the group coordinator is notified of the consumer's departure. When a consumer receives a shutdown signal, it should stop polling for new records, finish processing any in-flight records, commit the current offsets using `commitSync`, close the consumer with `consumer.close`, and allow the group coordinator to reassign its partitions. The `consumer.close` method handles many of these tasks automatically: it commits offsets if auto-commit is enabled, leaves the consumer group, and closes network connections. However, for at-least-once processing, the application should explicitly commit offsets before calling close.

### WakeupException Handling

When a consumer is blocked on a poll call and a shutdown signal is received, the application can call `consumer.wakeup` from a separate thread, which causes the poll to throw a `WakeupException`. The application should catch this exception in its main loop, check the shutdown flag, commit offsets if needed, and close the consumer. This pattern ensures that even a consumer that is blocked waiting for new records can be interrupted and shut down cleanly. Without wakeup handling, a consumer might remain blocked on a poll call indefinitely, preventing the application from shutting down within a reasonable time.

### JVM Shutdown Hooks

Both broker and consumer applications should register JVM shutdown hooks that execute the graceful shutdown logic when the JVM receives a SIGTERM signal. In Java, this is done by calling `Runtime.getRuntime().addShutdownHook` with a thread that performs the cleanup logic. In Python, the `signal` module can be used to register signal handlers for SIGTERM and SIGINT. In Kubernetes, when a pod is terminated, the container runtime sends SIGTERM to the process, waits for the `terminationGracePeriodSeconds` period (default 30 seconds), and then sends SIGKILL if the process has not exited. The shutdown hook must complete within this grace period.

### Docker and Kubernetes Considerations

When running Kafka in Docker, the container's entrypoint script must properly propagate signals to the Kafka process. Using the shell form of the ENTRYPOINT command can prevent signal propagation because the shell process becomes PID 1 and does not forward signals to child processes. The exec form of ENTRYPOINT or using a dedicated init system like tini ensures that signals reach the Kafka process. In Kubernetes, the `terminationGracePeriodSeconds` should be set high enough to allow the broker to complete its controlled shutdown, including partition leadership transfer and log flushing. A value of 60 to 120 seconds is typical for production Kafka deployments.

## Practical Applications

During a Kafka cluster upgrade, operators perform a rolling restart, shutting down one broker at a time and waiting for it to complete its controlled shutdown before restarting it and moving to the next broker. This ensures that the cluster remains available throughout the upgrade, with at most one broker unavailable at any time. When scaling consumer applications in Kubernetes, the Horizontal Pod Autoscaler terminates excess pods, which must shut down gracefully by committing offsets and leaving their consumer groups. During planned maintenance windows, operators drain brokers by moving partition leadership away before shutting down, minimizing the impact on client applications.
