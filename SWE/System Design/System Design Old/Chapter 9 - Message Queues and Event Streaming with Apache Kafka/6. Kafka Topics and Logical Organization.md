# Kafka Topics and Logical Organization

## Overview

Kafka topics provide the fundamental logical organization for events in a Kafka cluster. A topic is a named category or feed to which records are published and from which consumers subscribe. Topics serve a role analogous to tables in a relational database or channels in a messaging system, providing a way to group related events together so that producers and consumers can find and process them efficiently. The design of your topic topology, including how many topics you create and how you name and structure them, has significant implications for the maintainability, scalability, and clarity of your event-driven architecture.

## Key Concepts

### Topic Naming Conventions

Topic naming is an important operational concern that is often underestimated. A well-designed naming convention makes it easy for engineers to understand what data flows through a topic and how it should be used. Common conventions include using a hierarchical structure separated by dots or underscores, such as `orders.created`, `orders.cancelled`, `inventory.updated`, and `payments.processed`. This naming pattern encodes the domain entity and the event type, making it immediately clear what each topic contains. Some organizations also include the team name or data classification level in the topic name to enforce organizational boundaries and access policies.

### Topic Configuration

Each topic has a set of configuration parameters that control its behavior. The most important are the number of partitions, which determines the maximum parallelism for consumption, and the replication factor, which determines how many copies of the data are maintained across the cluster for fault tolerance. Other important configurations include `retention.ms`, which controls how long events are retained before being deleted; `cleanup.policy`, which can be set to delete old events or compact them by key; and `max.message.bytes`, which limits the maximum size of a single record. These settings allow you to tailor each topic to its specific use case.

### Retention Policies

Kafka topics can be configured with time-based retention, size-based retention, or log compaction. Time-based retention, controlled by `retention.ms`, deletes events older than the specified duration. Size-based retention, controlled by `retention.bytes`, deletes the oldest events when the total topic size exceeds the specified limit. Log compaction, enabled by setting `cleanup.policy=compact`, retains only the latest value for each key, effectively maintaining a snapshot of the current state for every key. This is useful for change data capture and materialized view patterns where you only need the most recent state rather than the full history of changes.

### Topic as a Contract

A topic represents a contract between producers and consumers. The schema of the events, the ordering guarantees, and the retention policy are all part of this contract. Changes to the topic schema must be backward-compatible to avoid breaking existing consumers. Tools like Confluent Schema Registry enforce compatibility rules that prevent producers from publishing events with incompatible schemas. Breaking the topic contract, for example by changing the event structure in a non-backward-compatible way, can cause widespread failures in downstream consumers and should be treated with the same caution as changing a database schema.

### Multiple Topics versus Multiple Event Types

A common design question is whether to use one topic per event type or to multiplex multiple event types into a single topic. Using one topic per event type provides clear separation and makes it easy to manage retention, monitoring, and access control independently for each type. However, it can lead to topic proliferation in large organizations with hundreds of event types. Multiplexing event types into a single topic reduces the number of topics but requires consumers to filter events by type, which adds complexity and can waste consumer capacity on irrelevant events. The one-topic-per-event-type approach is generally preferred for clarity and operational simplicity.

### Topic Creation and Management

Topics can be created explicitly using the Kafka CLI tools or the AdminClient API, or they can be created automatically when a producer first writes to a non-existent topic if `auto.create.topics.enable` is set to true on the broker. Automatic topic creation is convenient for development but dangerous in production, because it can lead to topics being created with default configurations that may not be appropriate, such as a single partition and no replication. Best practice is to disable automatic topic creation in production and manage topic creation through infrastructure-as-code pipelines that ensure consistent configuration.

## Practical Applications

In a food delivery application, you might create topics such as `food_orders_placed`, `food_orders_preparing`, `food_orders_delivering`, `food_orders_completed`, and `driver_location_updates`. Each topic serves a specific purpose and has its own retention and partitioning requirements. The driver location topic might have a short retention period since only recent locations are relevant, while the orders completed topic might have a long retention period for analytics and audit purposes. An e-commerce platform might have separate topics for order events, payment events, inventory events, and user activity events, each consumed by different microservices.
