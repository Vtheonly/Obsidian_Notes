# Kafka Consumers and Event Processing

## Overview

Kafka consumers are the client applications that subscribe to one or more topics and process the events stored within them. Unlike traditional message queue consumers that remove messages after consumption, Kafka consumers read events from a persistent log at their own pace, maintaining their own position through a mechanism called offsets. This design enables multiple independent consumers to read the same events without interfering with each other, supports replaying events from any point in the stream, and decouples the rate of production from the rate of consumption. Understanding consumer behavior, offset management, and delivery semantics is essential for building reliable event-driven applications.

## Key Concepts

### Consumer Architecture

A Kafka consumer connects to the Kafka cluster, subscribes to one or more topics, and begins polling for new events. The consumer reads events from the partitions assigned to it, processes them according to the application logic, and periodically commits its current offset to mark its progress. The poll loop is the core of the consumer: the application repeatedly calls the poll method, which returns a batch of records from the assigned partitions. The application processes each record and then commits the offset before the next poll cycle. If the consumer crashes, it resumes from the last committed offset, ensuring that no events are skipped.

### Offset Management

Offsets are monotonically increasing integers that uniquely identify each record within a partition. The current offset represents the next record to be consumed, while the committed offset represents the last successfully processed record. Consumers can manage offsets in two ways: automatic commits at regular intervals configured by `enable.auto.commit` and `auto.commit.interval.ms`, or manual commits using `commitSync` or `commitAsync`. Automatic commits are convenient but can lead to data loss if the consumer crashes after processing records but before the auto-commit interval elapses. Manual commits give the application full control over when offsets are committed, enabling at-least-once or exactly-once processing guarantees.

### Delivery Semantics

Kafka consumers can achieve three levels of delivery semantics depending on how offsets are managed. At-most-once semantics means the consumer commits offsets before processing records, so if it crashes during processing, those records are never reprocessed but may be lost. At-least-once semantics means the consumer commits offsets after processing records, so if it crashes after processing but before committing, records may be reprocessed but are never lost. Exactly-once semantics, the most difficult to achieve, requires idempotent processing logic or transactional consumption combined with transactional production, ensuring that each record is processed exactly once even in the face of failures and retries.

### Consumer Rebalancing

When a consumer joins or leaves a consumer group, Kafka triggers a rebalance event that reassigns partitions among the active consumers in the group. During a rebalance, consumers may lose their assigned partitions and be assigned new ones. This means that any in-flight processing must be completed or aborted, and state associated with the old partitions must be cleaned up. The cooperative sticky assignor minimizes the disruption of rebalances by only moving the partitions that must be reassigned, rather than revoking all partitions from all consumers and reassigning from scratch, which is what the older eager assignors do.

### Deserialization

Consumers must deserialize the byte arrays received from Kafka back into application objects. The deserializer must match the serializer used by the producer: if the producer uses StringSerializer, the consumer must use StringDeserializer; if the producer uses JSON serialization, the consumer must use the corresponding JSON deserializer. Schema management becomes important at scale, because producers and consumers must agree on the schema of the data. Confluent Schema Registry provides a centralized schema management service that ensures producers and consumers use compatible schemas, preventing deserialization errors caused by schema mismatches.

### Poll Loop Best Practices

The consumer poll loop must be carefully designed to balance processing time with liveness. If processing a batch of records takes too long, the consumer may exceed the `max.poll.interval.ms` configuration, causing Kafka to assume the consumer has died and trigger a rebalance. Conversely, processing too few records per poll cycle wastes network round trips. Best practices include keeping processing logic efficient, using a separate thread pool for heavy processing, and committing offsets only after processing is complete. The `max.poll.records` setting controls how many records are returned per poll, allowing you to bound the amount of work per iteration.

## Practical Applications

In a food delivery platform, the consumer might be a Kitchen Service that listens to the "new_orders" topic and prepares the food. The consumer reads each order event, updates the kitchen display, and sends progress updates to the customer. In an e-commerce system, multiple consumers independently process the same order event: one sends a confirmation email, one updates the inventory database, one generates an invoice, and one updates the analytics dashboard. Each consumer maintains its own offset, so if the email service is temporarily down, the other services continue processing normally, and the email service catches up when it recovers.
