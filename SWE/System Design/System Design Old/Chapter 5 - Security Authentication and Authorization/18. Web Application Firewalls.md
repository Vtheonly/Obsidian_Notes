# Web Application Firewalls

## Overview
A Web Application Firewall (WAF) is a security component that monitors, filters, and blocks HTTP traffic to and from a web application, protecting it from common web-based attacks. Unlike a traditional network firewall that operates at the transport and network layers (filtering based on IP addresses, ports, and protocols), a WAF operates at the application layer (Layer 7 of the OSI model) and inspects the content of HTTP requests and responses. WAFs are a critical component of defense-in-depth strategies, providing a protective barrier between the internet and the application that can detect and mitigate attacks before they reach the application code. Understanding WAF capabilities, deployment models, and limitations is essential for system designers responsible for production security.

## Key Concepts

### WAF Operation Model
A WAF sits between the client and the web application, intercepting all inbound HTTP requests and outbound HTTP responses. For each request, the WAF evaluates a set of rules (also called security policies) that define what constitutes malicious traffic. If a request matches a rule, the WAF can block the request, log the event, rate-limit the source, or flag it for review. If no rule matches, the request is forwarded to the application. The WAF also inspects outbound responses to detect data leakage, such as credit card numbers or social security numbers appearing in the response body. This bidirectional inspection provides comprehensive application-layer protection.

### Rule Sets and Detection Methods
WAFs use several detection methods. Signature-based detection compares request patterns against a database of known attack signatures -- for example, patterns that match SQL injection attempts (like "' OR 1=1 --") or XSS payloads (like "<script>alert(1)</script>"). Signature-based detection is effective against known attacks but cannot detect novel attack patterns. Anomaly-based detection builds a baseline of normal traffic patterns and flags deviations from the baseline, such as unusually long parameter values, unexpected character encodings, or abnormal request rates. Anomaly-based detection can catch zero-day attacks but may produce false positives. Behavior-based detection evaluates the sequence and context of requests over time, identifying patterns that suggest an attack campaign. Most modern WAFs combine all three methods for maximum effectiveness.

### OWASP Core Rule Set
The OWASP ModSecurity Core Rule Set (CRS) is the most widely used open-source WAF rule set. It provides generic attack detection rules that protect against the OWASP Top Ten vulnerabilities, including SQL injection, XSS, file inclusion, remote code execution, and session fixation. The CRS is designed to work with the ModSecurity WAF engine but has been ported to other platforms. It uses a scoring system where each matched rule adds to a transaction score, and requests exceeding a threshold are blocked. This scoring approach reduces false positives compared to a binary match/block strategy. The CRS is regularly updated to address new threats and should be supplemented with application-specific rules for custom vulnerabilities not covered by generic patterns.

### WAF Deployment Models
WAFs can be deployed in several models. A reverse proxy WAF sits in front of the web application, intercepting all traffic before it reaches the server. This is the most common deployment model and provides full visibility and control. A transparent WAF operates in bridge mode, inspecting traffic without modifying the network topology, which simplifies deployment but provides less control. Cloud-based WAFs (such as AWS WAF, Cloudflare WAF, and Akamai Kona Site Defender) are managed services that provide protection at the CDN edge, filtering malicious traffic before it reaches the origin infrastructure. These services offer easy setup, automatic rule updates, and global scalability but may provide less customization than self-hosted solutions. The choice of deployment model depends on the organization's infrastructure, expertise, and threat model.

### WAF Limitations and Bypass Techniques
WAFs are not a silver bullet and have several limitations. WAF bypass is an active area of security research, and attackers constantly develop new techniques to evade detection. Common bypass methods include encoding attacks in non-standard character sets (like double URL encoding or Unicode encoding), using HTTP parameter pollution to split attack payloads across multiple parameters, exploiting parser differentials between the WAF and the application server, and using chunked transfer encoding to hide payloads. WAFs also generate false positives that can block legitimate traffic, which creates pressure to weaken rules and reduce protection. Additionally, WAFs cannot protect against application logic vulnerabilities, authentication flaws, or server-side misconfigurations. A WAF should be part of a defense-in-depth strategy, not the sole security control.

## Practical Applications
E-commerce platforms use WAFs to protect against SQL injection and XSS attacks on product search, login, and checkout endpoints. Financial institutions deploy WAFs to comply with PCI DSS Requirement 6.6, which mandates application-layer protection. SaaS providers use cloud-based WAFs to protect multi-tenant applications from cross-tenant attacks. Government agencies use WAFs to protect public-facing web services from nation-state attack campaigns. Content management systems like WordPress benefit from WAF protection due to their large attack surface from plugins and themes. API gateways often integrate WAF capabilities to protect REST and GraphQL endpoints from injection and abuse.
