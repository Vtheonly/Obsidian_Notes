# OAuth 2 Framework and Delegated Authorization

## Overview
OAuth 2.0 is an authorization framework that enables a third-party application to obtain limited access to a user's resources on another service, without the user sharing their credentials. Defined in RFC 6749, OAuth 2.0 is not an authentication protocol itself -- it provides delegated authorization, allowing users to grant applications scoped, time-limited access to their data. OAuth 2.0 has become the standard for API authorization across the internet, powering everything from "Sign in with Google" to enterprise API access management. Understanding OAuth 2.0 is essential for system designers because it defines the flows, roles, and security considerations that underpin modern API security.

## Key Concepts

### OAuth 2.0 Roles
OAuth 2.0 defines four roles. The Resource Owner is the entity that can grant access to a protected resource (typically the end user). The Client is the application making requests on behalf of the resource owner. The Authorization Server issues access tokens to the client after successfully authenticating the resource owner and obtaining authorization. The Resource Server hosts the protected resources and accepts and validates access tokens. In many implementations, the Authorization Server and Resource Server are part of the same platform (for example, Google's auth server and Google's API servers), but they are logically distinct and can be deployed independently.

### Grant Types
OAuth 2.0 defines several grant types for different use cases. The Authorization Code Grant is the most secure and is used for server-side applications. The client redirects the user to the authorization server, the user authorizes the request, and the authorization server returns an authorization code. The client exchanges this code for tokens at the token endpoint, authenticating with its client secret. The Authorization Code Grant with PKCE (Proof Key for Code Exchange) is used for public clients like SPAs and mobile apps, replacing the client secret with a dynamically generated code verifier. The Client Credentials Grant is used for server-to-server communication where the client is also the resource owner. The Resource Owner Password Credentials Grant allows the client to directly exchange the user's username and password for tokens, but this is strongly discouraged because it requires the user to share credentials with the client. The Refresh Token Grant allows a client to obtain a new access token using a refresh token.

### Access Tokens and Scopes
Access tokens are credentials that the client uses to access protected resources on the resource server. They can be opaque (random strings referencing server-side data) or JWTs. The scope parameter in the authorization request defines the level of access the client is requesting. Scopes are defined by the resource server and can be granular (for example, "read:contacts write:contacts") or coarse ("profile"). The authorization server presents the requested scopes to the user for approval and includes the granted scopes in the issued access token. The resource server must validate that the access token's scopes are sufficient for the requested operation. This scope-based access control provides a flexible and user-transparent mechanism for limiting the permissions granted to third-party applications.

### Security Considerations
OAuth 2.0 has several well-documented security considerations. The authorization code is vulnerable to interception if the redirect URI is not strictly validated -- an attacker can craft a malicious redirect URI to capture the authorization code. To prevent this, the authorization server must enforce exact redirect URI matching. The Implicit Grant (now deprecated) was vulnerable to token leakage through the URL fragment and referrer headers. PKCE mitigates authorization code interception for public clients by requiring a code verifier that only the legitimate client can provide. CSRF attacks on the authorization endpoint are prevented by using the "state" parameter, which the client includes in the authorization request and validates in the callback. Token storage on the client must be secure, and all OAuth flows must be conducted over HTTPS.

### OAuth 2.0 Extensions and Best Current Practices
The IETF OAuth Working Group has published several Best Current Practice (BCP) documents that update and strengthen the original OAuth 2.0 specification. OAuth 2.0 for Browser-Based Apps (RFC 9706) recommends the Authorization Code Flow with PKCE for all browser-based applications and deprecates the Implicit Grant. OAuth 2.0 for Native Apps (RFC 8252) requires the use of PKCE and recommends using the system browser for authentication. JWT Profile for OAuth 2.0 Access Tokens (RFC 9068) defines a standard format for JWT access tokens. Token Binding and DPoP (Demonstration of Proof-of-Possession) provide sender-constrained tokens that prevent token theft and replay. These extensions address the evolving threat landscape and should be followed in all new implementations.

## Practical Applications
OAuth 2.0 is the foundation of third-party API access across the internet. When a user grants a calendar app access to their Google Calendar, OAuth 2.0 is the protocol that enables this delegated access. Social login buttons ("Sign in with Facebook/Twitter/Google") use OAuth 2.0 combined with OpenID Connect. In enterprise environments, OAuth 2.0 is used for API access management, allowing internal and partner applications to access resources without sharing credentials. Cloud platforms use OAuth 2.0 for service account authentication and cross-service authorization. API gateways enforce OAuth 2.0 scopes as part of their access control policies.
