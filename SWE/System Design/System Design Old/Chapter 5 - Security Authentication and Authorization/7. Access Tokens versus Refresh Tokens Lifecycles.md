# Access Tokens versus Refresh Tokens Lifecycles

## Overview
In modern authentication systems, the access token and refresh token pattern is a critical mechanism for balancing security and user experience. An access token is a short-lived credential that grants the client access to protected resources, while a refresh token is a longer-lived credential used to obtain new access tokens without requiring the user to re-authenticate. This dual-token approach minimizes the risk of token compromise by keeping access tokens short-lived while maintaining a seamless user experience through silent token renewal. Understanding the lifecycle of each token type, their security properties, and their interaction is essential for designing secure and user-friendly authentication systems.

## Key Concepts

### Access Tokens
Access tokens are the credentials that a client presents to access a protected resource. They are typically JWTs or opaque tokens with a short expiration time, usually between 5 and 30 minutes. The short lifespan limits the damage if the token is compromised -- an attacker can only use the token for a brief window before it expires. Access tokens should be treated as bearer tokens: anyone who possesses them can use them. They are included in the Authorization header of API requests (Authorization: Bearer \<token\>). Because access tokens are transmitted on every request, they should be kept as small as possible to minimize bandwidth overhead. Access tokens should never be stored in persistent client storage if possible, and they should always be transmitted over HTTPS.

### Refresh Tokens
Refresh tokens are credentials used to obtain new access tokens from the authorization server. They are long-lived, typically expiring after days, weeks, or even months, and they are used only with the authorization server's token endpoint -- never with resource servers. Because refresh tokens are long-lived, they are high-value targets for attackers. They should be stored securely, ideally in HttpOnly, Secure cookies on the client side. When a refresh token is used to obtain a new access token, the authorization server may also issue a new refresh token and invalidate the old one -- a practice called refresh token rotation. This rotation ensures that if a refresh token is stolen and used by an attacker, the legitimate client will detect the theft when its own refresh token no longer works.

### Token Lifecycle: From Issuance to Expiration
The lifecycle begins when the user authenticates successfully. The authorization server issues both an access token and a refresh token. The client uses the access token for API requests until it expires. When the access token expires, the client sends the refresh token to the token endpoint to obtain a new access token (and optionally a new refresh token). This process repeats until the refresh token itself expires or is revoked, at which point the user must re-authenticate. If a refresh token is detected as compromised (for example, if it is used after being rotated), the authorization server should revoke all refresh tokens associated with that user session, forcing a full re-authentication.

### Refresh Token Rotation
Refresh token rotation is a security mechanism where each time a refresh token is used, the authorization server issues a new refresh token and invalidates the previous one. This is specified in OAuth 2.0 for Browser-Based Apps (BCP) and is strongly recommended by security best practices. Rotation limits the window of opportunity for an attacker who steals a refresh token: once the legitimate client uses the refresh token and receives a new one, the stolen token becomes invalid. If both the attacker and the legitimate client attempt to use the same refresh token, the server detects the reuse and revokes the entire token family, forcing the user to log in again. This mechanism significantly reduces the impact of token theft.

### Revocation and Invalidation
Access tokens are difficult to revoke because they are self-contained and stateless -- a resource server validates them based on their signature and claims, not by checking a central store. This is why access tokens should be short-lived: revocation is effectively achieved by waiting for expiration. For immediate revocation, some systems maintain a token blacklist (a set of revoked token IDs stored in a fast lookup store like Redis) that resource servers check during validation. Refresh tokens are easier to revoke because they are validated by the authorization server, which can maintain a store of active refresh tokens and delete them as needed. When a user logs out, changes their password, or is administratively locked, all associated refresh tokens should be revoked.

## Practical Applications
The access/refresh token pattern is used by virtually all modern OAuth 2.0 and OpenID Connect implementations. Google, Microsoft, and Auth0 all issue short-lived access tokens and long-lived refresh tokens for their APIs. In mobile applications, refresh tokens enable persistent login without storing credentials on the device. In SPAs, the refresh token is stored in an HttpOnly cookie while the access token is held in memory, providing a balance between security and user experience. Enterprise identity platforms use refresh token rotation as a standard security control, and compliance frameworks like PCI DSS and NIST 800-63 recommend short access token lifetimes with secure refresh mechanisms.
