# Cross Origin Resource Sharing Policies

## Overview
Cross-Origin Resource Sharing (CORS) is a security mechanism implemented by browsers that controls whether a web application running at one origin can access resources from a different origin. An origin is defined by the combination of scheme (HTTP or HTTPS), host, and port. By default, browsers enforce the Same-Origin Policy (SOP), which prevents scripts from making requests to a different origin than the one that served the page. CORS provides a controlled way to relax this restriction, allowing servers to explicitly permit cross-origin requests through HTTP headers. Understanding CORS is essential for system designers because almost every modern web application with a separate frontend and backend must configure CORS correctly, and misconfiguration can lead to either broken functionality or serious security vulnerabilities.

## Key Concepts

### Same-Origin Policy
The Same-Origin Policy is the browser's default security model, which prevents a document or script loaded from one origin from interacting with resources from another origin. This policy prevents malicious websites from reading data from authenticated sessions on other websites. For example, if a user is logged into their bank at bank.com, the SOP prevents evil.com from making AJAX requests to bank.com and reading the responses. The SOP applies to XMLHttpRequest, fetch API, cookies, and DOM access. However, the SOP also prevents legitimate cross-origin requests, such as a SPA hosted on app.example.com calling an API at api.example.com. CORS was created to solve this problem by providing a controlled mechanism for cross-origin access.

### Simple Requests and Preflight Requests
CORS distinguishes between simple requests and requests that require a preflight. A simple request is one that uses GET, HEAD, or POST with standard content types (application/x-www-form-urlencoded, multipart/form-data, or text/plain) and does not set custom headers. For simple requests, the browser adds the Origin header and the server responds with CORS headers like Access-Control-Allow-Origin. For non-simple requests (for example, PUT or DELETE methods, custom headers like Authorization, or content types like application/json), the browser first sends a preflight request using the OPTIONS method. The server responds to the preflight with headers that indicate which methods, headers, and origins are permitted. If the preflight succeeds, the browser sends the actual request. If the preflight fails, the browser blocks the request entirely.

### CORS Headers
The key CORS headers include: Access-Control-Allow-Origin specifies which origins are permitted to access the resource (a single origin or "*" for any origin, but "*" cannot be used with credentials). Access-Control-Allow-Methods lists the HTTP methods allowed for cross-origin requests. Access-Control-Allow-Headers lists the request headers that are permitted in the actual request. Access-Control-Allow-Credentials indicates whether the request can include credentials like cookies or authorization headers. Access-Control-Max-Age specifies how long the preflight result can be cached. Access-Control-Expose-Headers lists which response headers the browser should make available to the JavaScript code. On the request side, the browser adds the Origin header to cross-origin requests and the Access-Control-Request-Method and Access-Control-Request-Headers headers to preflight requests.

### Security Risks of CORS Misconfiguration
CORS misconfiguration is a common and serious security vulnerability. The most dangerous misconfiguration is setting Access-Control-Allow-Origin to "*" while also setting Access-Control-Allow-Credentials to true, which the browser correctly blocks (this combination is not allowed by the spec). However, a subtler vulnerability occurs when the server dynamically reflects the request's Origin header in the Access-Control-Allow-Origin response header without validating it against an allowlist. This allows any origin to make credentialed cross-origin requests, effectively bypassing the Same-Origin Policy. Another misconfiguration is allowing overly broad methods or headers in the preflight response. To prevent these vulnerabilities, the server should maintain an allowlist of permitted origins and only set CORS headers for origins on that list.

### CORS in Microservices and API Gateways
In microservices architectures, CORS is typically handled at the API gateway or reverse proxy level rather than in individual services. The gateway inspects the Origin header, validates it against the allowed origins, and adds the appropriate CORS headers to the response. This centralizes CORS configuration and ensures consistency across all services. However, the gateway must correctly handle both preflight OPTIONS requests and actual requests. For preflight requests, the gateway should respond directly without forwarding to backend services. For actual requests, the gateway should add CORS headers to the backend's response. Some load balancers and CDNs also provide CORS configuration, allowing CORS headers to be added at the edge before the request reaches the application infrastructure.

## Practical Applications
SPAs calling REST APIs on a different domain require CORS configuration to function. For example, a React app at app.mycompany.com calling an API at api.mycompany.com must have the API server configured to allow the app's origin. Third-party integrations like embedded payment widgets or social login buttons use CORS to communicate with their parent services. CDN-hosted static assets that fetch data from an API origin require CORS headers on the API responses. Webhook providers must configure CORS if their management dashboards make cross-origin API calls. API gateways like Kong, AWS API Gateway, and NGINX provide built-in CORS configuration options for this purpose.
