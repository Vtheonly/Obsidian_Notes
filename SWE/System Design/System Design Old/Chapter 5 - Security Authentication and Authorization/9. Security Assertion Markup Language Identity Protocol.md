# Security Assertion Markup Language Identity Protocol

## Overview
The Security Assertion Markup Language (SAML) is an XML-based, open-standard protocol for exchanging authentication and authorization data between an Identity Provider (IdP) and a Service Provider (SP). Defined by OASIS, SAML 2.0 (the current and most widely used version) enables Single Sign-On (SSO) by allowing a user to authenticate once with the IdP and then access multiple service providers without re-authenticating. SAML uses digitally signed XML documents called assertions to communicate identity information, making it particularly well-suited for enterprise environments where browser-based SSO, strong security guarantees, and integration with existing LDAP or Active Directory infrastructure are required.

## Key Concepts

### SAML Architecture: IdP, SP, and User Agent
SAML involves three primary entities. The Identity Provider (IdP) is the trusted authority that authenticates users and issues SAML assertions. The Service Provider (SP) is the application or service that the user wants to access. The User Agent is typically the user's web browser, which facilitates the communication between the IdP and SP through HTTP redirects and POSTs. The trust relationship between the IdP and SP is established through metadata exchange, which includes the entity IDs, signing certificates, and endpoint URLs of each party. This metadata must be configured on both sides before SSO can function.

### SAML Assertions
A SAML assertion is an XML document that contains statements about an authenticated subject. There are three types of statements. Authentication statements assert that the subject was authenticated by a particular means at a particular time. Attribute statements assert that the subject has certain attributes (such as email, group membership, or role). Authorization decision statements assert that the subject is permitted to access a specific resource. Assertions are digitally signed by the IdP using XML Signature, which ensures their integrity and authenticity. The SP validates the signature using the IdP's public certificate before trusting the assertion's contents. Assertions can be encrypted using XML Encryption to protect sensitive attribute values in transit.

### SAML Protocol Flows
SAML supports several protocol flows, but the most common are SP-initiated and IdP-initiated SSO. In SP-initiated SSO, the user attempts to access a resource on the SP. The SP generates a SAML authentication request, encodes it, and redirects the user to the IdP via the browser. The IdP authenticates the user (or uses an existing session), generates a SAML assertion, and sends it back to the SP via an HTTP POST (the SAML response). The SP validates the assertion and grants access. In IdP-initiated SSO, the user first authenticates at the IdP and then selects an application from a portal. The IdP generates an unsolicited SAML assertion and posts it to the SP. Both flows rely on browser redirects and form posts, which means they are inherently synchronous and browser-based.

### SAML Security Considerations
SAML provides strong security when implemented correctly, but it has several known attack vectors. XML Signature wrapping attacks exploit the way some implementations parse signed XML by injecting additional elements that alter the document's meaning while preserving the signature's validity. To mitigate this, implementations must strictly follow the SAML specification for signature validation. SAML is also vulnerable to replay attacks if assertions are not checked for freshness using the NotOnOrAfter condition. Redirect-based flows expose the SAML request in the URL, which can leak information through browser history and referrer headers. Best practices include using POST bindings instead of redirects where possible, validating all assertion conditions, enforcing strict audience restrictions, and keeping library implementations up to date.

### SAML Metadata and Trust Configuration
SAML metadata is the configuration data that establishes trust between the IdP and SP. It includes the entity ID (a unique identifier for each party), the X.509 certificates used for signing and encryption, the SSO and SLO endpoint URLs, and the supported bindings and name ID formats. Metadata can be exchanged out-of-band (manually configured) or fetched dynamically via well-known URLs. In large federations, metadata aggregation services collect and distribute metadata for all participants. Proper metadata management is critical because incorrect or stale metadata will cause SSO failures, and a compromised signing certificate would allow an attacker to forge assertions. Metadata should be signed and verified, and certificates should be rotated on a regular schedule.

## Practical Applications
SAML is the dominant SSO protocol in enterprise environments. Organizations use SAML to integrate cloud applications like Salesforce, ServiceNow, and AWS with their corporate Active Directory or LDAP identity stores. In higher education, SAML (often via the Shibboleth implementation) enables federated identity across institutions, allowing students and researchers to access shared resources. Government agencies use SAML for cross-agency authentication. Enterprise IdP solutions like Okta, Azure AD, and Ping Identity support SAML as a core protocol, and virtually all enterprise SaaS products support SAML-based SSO for customer and employee access.
