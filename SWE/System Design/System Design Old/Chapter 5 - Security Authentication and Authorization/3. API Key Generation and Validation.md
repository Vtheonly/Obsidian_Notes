# API Key Generation and Validation

## Overview
API keys are unique identifiers used to authenticate and authorize clients accessing an API. Unlike user-facing authentication mechanisms, API keys are designed for programmatic access -- they are long-lived tokens that identify the calling application or service rather than an individual user. API keys are typically included in request headers, query parameters, or cookies, and the server validates them on each request to determine whether the client is authorized to access the requested resource. While API keys are simple to implement and widely used, they have significant security limitations and should be combined with other controls such as rate limiting, HTTPS enforcement, and short-lived tokens for sensitive operations.

## Key Concepts

### API Key Generation
An API key must be sufficiently random and long to resist brute-force and guessing attacks. Best practice is to generate keys using a cryptographically secure random number generator (CSPRNG) with at least 128 bits of entropy. The key is typically a hexadecimal or base64-encoded string, for example a 32-byte random value encoded as a 64-character hexadecimal string. Some systems use a prefix to identify the key type or environment (for example, "sk_live_" for a production secret key or "pk_test_" for a test public key), which makes keys easier to identify in logs and dashboards. Stripe is a well-known example of this prefix-based convention. Keys should never contain any information that could be used to derive the underlying random bytes.

### Key Storage and Distribution
API keys must be stored securely on both the server and the client side. On the server, keys should be hashed using a slow cryptographic hash function (such as bcrypt or Argon2) before being stored in the database, similar to how passwords are stored. This ensures that even if the database is compromised, an attacker cannot immediately use the keys. On the client side, keys must be stored in environment variables, secrets managers, or secure vaults -- never in source code, version control, or client-side JavaScript. When distributing keys to third-party developers, use a developer portal with authentication and audit logging to control access and track usage.

### Key Validation
On each API request, the server extracts the API key from the appropriate location (typically the Authorization header or a custom header like X-API-Key) and validates it against the stored records. Validation involves checking that the key exists, has not been revoked or expired, and is authorized for the requested operation. The validation process should be fast because it occurs on every request, which is why many systems cache active key metadata in Redis or another in-memory store. After validation, the server attaches the key's associated metadata (such as client identity, rate limit tier, and permission scope) to the request context for downstream processing.

### Key Lifecycle Management
API keys have a lifecycle that includes creation, rotation, revocation, and expiration. Keys should have an expiration date to limit the damage of a key leak, and systems should support key rotation -- the process of issuing a new key and deprecating the old one without downtime. Dual-key rotation strategies allow the old and new keys to be simultaneously valid for a transition period. Revocation should be immediate when a key is known to be compromised, and the system should support bulk revocation in case of a wide-scale breach. Audit logs should record every key lifecycle event for compliance and incident response.

### Security Limitations of API Keys
API keys are bearer tokens, meaning anyone who possesses the key can use it. They do not inherently provide identity verification, cannot enforce fine-grained permissions on their own, and are vulnerable to leakage through logs, URLs, or client-side code. Including API keys in query parameters is especially dangerous because URLs are logged by web servers, proxies, and browsers. API keys should always be transmitted over HTTPS, included in headers rather than query parameters, and never exposed in client-side applications. For public-facing APIs, API keys should be combined with additional security measures such as request signing, OAuth 2.0, or short-lived access tokens.

## Practical Applications
Public API providers such as Google Maps, Stripe, and Twilio use API keys to identify and meter client applications. In microservices architectures, service mesh implementations use API keys or mTLS certificates for inter-service authentication. API gateways like Kong, Apigee, and AWS API Gateway validate API keys as a first layer of access control before forwarding requests to backend services. For developer platforms, API keys are the standard mechanism for onboarding third-party integrations while maintaining auditability and usage tracking.
