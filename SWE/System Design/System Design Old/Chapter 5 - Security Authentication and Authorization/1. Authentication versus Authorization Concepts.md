# Authentication versus Authorization Concepts

## Overview
Authentication and authorization are two foundational pillars of system security that are frequently confused despite serving distinctly different purposes. Authentication, often abbreviated as AuthN, is the process of verifying the identity of a user or system -- answering the question "Who are you?" Authorization, abbreviated as AuthZ, is the process of determining what an authenticated identity is permitted to do -- answering the question "What can you do?" Understanding the distinction between these two concepts is critical for designing secure systems, as conflating them leads to both security vulnerabilities and overly restrictive access controls.

## Key Concepts

### Authentication (AuthN)
Authentication is the mechanism by which a system validates that a user, service, or device is who or what it claims to be. This is typically accomplished through one or more factors of evidence: something the user knows (a password or PIN), something the user has (a hardware token or mobile device), or something the user is (a biometric fingerprint or facial scan). When a user submits credentials, the authentication system verifies them against a stored record. If the credentials match, the system establishes an identity for that session. Multi-factor authentication (MFA) combines two or more of these factors to significantly increase the confidence in the claimed identity, making it substantially harder for an attacker to impersonate a legitimate user even if one factor is compromised.

### Authorization (AuthZ)
Authorization occurs after authentication succeeds. Once the system knows who the user is, authorization determines which resources and actions that user can access. Authorization policies define the rules governing access, and these policies can be simple or highly granular depending on the application's requirements. For example, a basic authorization policy might distinguish between admin and regular users, while a more sophisticated policy might enforce resource-level, attribute-based, or context-aware permissions. Authorization decisions are typically evaluated on every request, not just at login time, because permissions can change during a session.

### The Relationship Between Authentication and Authorization
Authentication and authorization are sequential and interdependent. Authorization cannot be performed without first authenticating the user, because the system must establish an identity before it can evaluate permissions. However, successful authentication does not imply full authorization -- a user may be authenticated but still denied access to specific resources. This separation of concerns allows each system to be designed, implemented, and scaled independently. For instance, an organization might use a centralized identity provider for authentication across all services while maintaining decentralized, application-specific authorization policies.

### Common Misconceptions
A frequent mistake is treating authentication as a substitute for authorization. If a system only verifies identity but does not enforce access controls, any authenticated user can access any resource, which is a serious security flaw. Conversely, implementing authorization without reliable authentication means the system cannot trust the identity on which access decisions are based. Both must be present and correctly implemented. Another misconception is that authentication is a one-time event; in practice, systems may re-authenticate users for sensitive operations or periodically refresh their authentication state.

### Defense in Depth
In production systems, authentication and authorization are part of a broader defense-in-depth strategy. This means they are combined with other security measures such as encryption, input validation, rate limiting, and logging to create multiple layers of protection. If one layer is compromised, the other layers continue to provide protection. Understanding the distinct roles of authentication and authorization within this layered approach is essential for building robust, secure architectures.

## Practical Applications
In a typical web application, a user logs in with a username and password (authentication), and the system then checks their role to determine which pages and API endpoints they can access (authorization). In a microservices architecture, each service independently verifies the authorization claims embedded in a token, while authentication is handled by a dedicated identity service. Cloud platforms such as AWS use IAM policies that separate identity verification from permission assignment, allowing fine-grained access control across resources.
