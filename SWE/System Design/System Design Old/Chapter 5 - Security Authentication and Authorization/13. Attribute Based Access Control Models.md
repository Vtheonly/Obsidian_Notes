# Attribute Based Access Control Models

## Overview
Attribute-Based Access Control (ABAC) is an authorization model that makes access decisions based on attributes of the subject (user), object (resource), action, and environment, combined through policies that evaluate these attributes. Unlike RBAC, which assigns static permissions through roles, ABAC evaluates dynamic conditions at access time, enabling fine-grained, context-aware access control. Defined in the NIST ABAC model (SP 800-162), ABAC addresses the limitations of RBAC in environments where access decisions depend on multiple contextual factors that cannot be captured by role assignments alone. ABAC is increasingly adopted in cloud platforms, zero-trust architectures, and regulated industries where fine-grained access control is required.

## Key Concepts

### ABAC Attribute Categories
ABAC evaluates four categories of attributes. Subject attributes describe the entity requesting access -- for example, user ID, department, security clearance, job title, or certifications held. Object attributes describe the resource being accessed -- for example, classification level, owner, creation date, or sensitivity tags. Action attributes describe the operation being performed -- for example, "read", "write", "delete", or "execute". Environment attributes describe the context of the access request -- for example, time of day, network location, device type, threat level, or regulatory jurisdiction. An access decision is made by evaluating a policy against the values of these attributes at the time of the request.

### ABAC Policy Language
ABAC policies are expressed as rules that combine attributes using logical operators. A simple policy might state: "Allow access if subject.department equals object.owner.department AND action equals 'read' AND environment.time is within business hours." More complex policies can include numerical comparisons, set membership, string matching, and custom functions. The eXtensible Access Control Markup Language (XACML) is an OASIS standard for expressing ABAC policies, though many implementations use simpler domain-specific languages. AWS IAM policies are a practical example of ABAC: they evaluate attributes like resource tags, principal tags, and conditions (such as IP range or time) to make access decisions. The policy language must be expressive enough to capture the organization's requirements while remaining understandable and auditable.

### Policy Decision, Enforcement, and Information Points
ABAC architectures typically follow the XACML reference architecture, which separates concerns into several components. The Policy Decision Point (PDP) evaluates policies against the available attributes and returns an access decision (permit, deny, not applicable, or indeterminate). The Policy Enforcement Point (PEP) intercepts access requests, queries the PDP, and enforces the decision. The Policy Information Point (PIP) provides attribute values that are not available in the request context -- for example, looking up a user's department from an HR system or a resource's classification from a metadata store. The Policy Administration Point (PAP) manages the creation, modification, and distribution of policies. This separation of concerns enables each component to be developed, deployed, and scaled independently.

### Comparing ABAC and RBAC
ABAC and RBAC are complementary models with different strengths. RBAC is simpler to understand and audit because permissions are grouped into roles with clear, static assignments. However, RBAC struggles with fine-grained access control, context-dependent decisions, and large-scale environments where role explosion occurs. ABAC provides fine-grained, context-aware access control without requiring a large number of role definitions, but it is more complex to design, implement, and audit because policies can be numerous and their interactions difficult to predict. In practice, many systems use a hybrid approach: RBAC for coarse-grained role assignment and ABAC for fine-grained, context-dependent decisions within each role. For example, a user might be assigned the "Doctor" role (RBAC), but ABAC policies further restrict which patient records they can access based on their department, the patient's consent, and the current time.

### ABAC in Zero Trust Architecture
ABAC is a natural fit for zero-trust security architectures, where every access request is evaluated based on multiple signals rather than relying on network perimeter trust. In a zero-trust model, attributes such as device health, user risk score, network location, and session context are evaluated continuously to determine whether to grant, deny, or step-up access. This continuous evaluation aligns with ABAC's dynamic, context-aware decision model. Cloud platforms like AWS, Azure, and GCP implement ABAC through conditional IAM policies that evaluate resource tags, principal attributes, and environmental conditions. As organizations move toward zero-trust models, ABAC is becoming the preferred authorization approach for sensitive and regulated workloads.

## Practical Applications
AWS IAM implements ABAC through tag-based access control, where policies evaluate resource tags and principal tags to make decisions. Azure implements ABAC through conditional access policies that evaluate device compliance, location, and risk. In healthcare, ABAC is used to enforce HIPAA access controls where patient data access depends on the provider's role, the patient's consent, the provider's relationship to the patient, and the purpose of access. In financial services, ABAC enforces trading restrictions based on the trader's department, the instrument's classification, the trade amount, and regulatory jurisdiction. Kubernetes implements a form of ABAC through its authorization webhook mechanism.
