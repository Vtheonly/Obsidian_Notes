# Basic and Digest Authentication Methods

## Overview
Basic and Digest authentication are two of the earliest HTTP-based authentication schemes defined in the HTTP specification. Both operate at the transport layer by including credentials in the HTTP request headers, but they differ fundamentally in how those credentials are transmitted and protected. Basic authentication sends credentials in plaintext (base64-encoded), while Digest authentication uses a challenge-response mechanism with cryptographic hashing to avoid transmitting the password in cleartext. Although both schemes have significant limitations by modern standards, understanding them is essential for system designers because they form the historical foundation of web authentication and are still encountered in legacy systems and constrained environments.

## Key Concepts

### Basic Authentication
HTTP Basic authentication is defined in RFC 7617 and is the simplest authentication scheme available. When a client attempts to access a protected resource, the server responds with a 401 Unauthorized status and a WWW-Authenticate header containing the challenge "Basic realm=\<realm\>". The client then concatenates the username and password with a colon, base64-encodes the result, and sends it in the Authorization header as "Basic \<base64-encoded-credentials\>". The server decodes the header and validates the credentials against its user store. The base64 encoding is merely an encoding scheme, not encryption, so the credentials are readable by anyone who intercepts the request. For this reason, Basic authentication must always be used over HTTPS to prevent credential exposure.

### Digest Authentication
HTTP Digest authentication, defined in RFC 7616, was designed to address the plaintext weakness of Basic authentication. Instead of sending the password directly, the server issues a challenge containing a nonce (a random value used once). The client responds with a hash computed from the username, password, nonce, HTTP method, and requested URI. This hash (typically MD5, though the spec allows stronger algorithms) is sent in the Authorization header. The server independently computes the same hash using its stored copy of the password and compares the results. Because the password itself is never transmitted, Digest authentication provides better protection against passive eavesdropping than Basic authentication, even over unencrypted connections.

### Limitations of Basic Authentication
Basic authentication has several critical weaknesses. First, base64 encoding provides no confidentiality -- any network observer can decode the credentials. Second, the credentials are sent with every request, increasing the attack surface. Third, there is no built-in mechanism for logout or session invalidation, as the browser caches the credentials for the duration of the session. Fourth, Basic authentication is vulnerable to credential replay attacks if HTTPS is not strictly enforced. Finally, the simplicity of the scheme means it lacks support for multi-factor authentication, token refresh, or any advanced identity features that modern applications require.

### Limitations of Digest Authentication
Digest authentication resolves the plaintext transmission problem but introduces its own set of weaknesses. The mandatory use of MD5 is a concern because MD5 is cryptographically broken for collision resistance, though it remains marginally acceptable for preimage resistance in this specific context. Digest authentication is vulnerable to man-in-the-middle attacks because an attacker can downgrade the scheme to Basic authentication. It also provides no integrity protection for the response body. Furthermore, the server must store passwords in a reversible or cleartext-equivalent form (or at minimum store the MD5 hash of username:realm:password), which violates modern password storage best practices that recommend irreversible bcrypt or Argon2 hashes.

### When to Use These Methods
In modern systems, neither Basic nor Digest authentication should be the primary authentication mechanism for user-facing applications. Basic authentication is still useful for machine-to-machine communication over secured channels, for simple internal APIs, or as a quick prototype mechanism. Digest authentication is largely obsolete and should be avoided in favor of token-based schemes like OAuth 2.0 or JWT-based authentication. However, both remain part of the HTTP standard and may be encountered in embedded devices, IoT systems, or legacy enterprise environments where upgrading is not immediately feasible.

## Practical Applications
Basic authentication is commonly used for simple API endpoints where HTTPS is enforced, such as webhook callbacks or internal service-to-service communication. It is also used in CI/CD pipelines where credentials are passed via environment variables. Digest authentication is rarely used in new systems but may be found in older enterprise applications or devices with limited processing capability that cannot handle TLS handshake overhead. When encountering these methods in production, the priority should be to ensure HTTPS enforcement and plan migration to more secure token-based mechanisms.
