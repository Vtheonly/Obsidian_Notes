# JSON Web Tokens Structure and Claims

## Overview
JSON Web Tokens (JWTs) are a compact, URL-safe, and self-contained token format defined in RFC 7519 for securely transmitting information between parties as a JSON object. A JWT consists of three parts -- a header, a payload, and a signature -- each base64url-encoded and concatenated with periods. The signature ensures the integrity and authenticity of the token, allowing the recipient to verify that the token has not been tampered with. JWTs have become the de facto standard for token-based authentication and authorization in web applications and APIs, though they come with important caveats regarding size, security, and revocation that system designers must understand.

## Key Concepts

### JWT Structure: Header, Payload, Signature
A JWT is structured as three dot-separated segments: header.payload.signature. The header specifies the token type ("typ": "JWT") and the signing algorithm ("alg": "HS256" or "RS256"). The payload contains claims -- key-value pairs that represent assertions about the subject. The signature is computed by base64url-encoding the header and payload, concatenating them with a period, and signing the result using the specified algorithm and a secret key (for HMAC) or a private key (for RSA/ECDSA). For example, an HS256 JWT might look like "eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjM0NTY3ODkwIn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c". The recipient validates the signature before trusting the claims in the payload.

### Registered Claims
The JWT specification defines a set of standard registered claims with three-letter keys to keep the token compact. The "iss" (issuer) claim identifies the principal that issued the token. The "sub" (subject) claim identifies the principal that is the subject of the token. The "aud" (audience) claim identifies the recipients that the token is intended for. The "exp" (expiration time) claim identifies the time after which the token must not be accepted. The "nbf" (not before) claim identifies the time before which the token must not be accepted. The "iat" (issued at) claim identifies the time at which the token was issued. The "jti" (JWT ID) claim provides a unique identifier for the token, which can be used to prevent replay attacks. These claims enable standard validation logic across implementations.

### Public and Private Claims
In addition to registered claims, JWTs can contain public claims and private claims. Public claims are defined in the IANA JSON Web Token Claims registry and are intended for broad interoperability -- for example, "name", "email", and "role". Private claims are custom claims agreed upon between the producer and consumer of the token, such as "tenant_id", "permission_level", or "subscription_tier". While private claims provide flexibility, they can create interoperability challenges and should be namespaced to avoid collisions. It is important to keep the payload as small as possible because the entire JWT is transmitted with every request, and large payloads increase bandwidth consumption and latency.

### Signing Algorithms: HMAC vs RSA vs ECDSA
JWTs support multiple signing algorithms, and the choice has significant security and architectural implications. HMAC-based algorithms (HS256, HS384, HS512) use a shared secret between the issuer and the validator. They are fast and simple but require the secret to be shared with every service that validates tokens, which increases the risk of key compromise. RSA-based algorithms (RS256, RS384, RS512) use asymmetric key pairs: the issuer signs with the private key, and validators verify with the public key. This is more secure for distributed systems because the private key never leaves the issuer. ECDSA-based algorithms (ES256, ES384, ES512) provide similar asymmetric security with smaller key and signature sizes, making them efficient for bandwidth-constrained environments. The "alg": "none" option should always be disabled because it allows unsigned tokens that bypass signature verification entirely.

### JWT Security Best Practices
Several security best practices must be followed when using JWTs. Always validate the "exp" claim to reject expired tokens. Validate the "iss" and "aud" claims to ensure the token was issued by a trusted authority and is intended for the current service. Never store sensitive data in the payload because it is only base64url-encoded, not encrypted -- anyone who intercepts the token can read the claims. Use JWE (JSON Web Encryption) if payload confidentiality is required. Keep the token size small by including only essential claims. Rotate signing keys periodically and support key rotation in validation logic by accepting multiple valid keys (identified by the "kid" header claim). Never accept tokens with "alg": "none". Use short expiration times for access tokens and longer-lived refresh tokens to minimize the window of exposure.

## Practical Applications
JWTs are used extensively in OAuth 2.0 as access tokens and ID tokens in OpenID Connect. In microservices architectures, each service validates the JWT locally using the signing key, enabling stateless authentication without a central session store. API gateways validate JWTs before forwarding requests to backend services. In SPAs, JWTs are stored client-side and included in API requests to authenticate the user. Enterprise identity providers like Auth0, Okta, and Keycloak issue JWTs as part of their authentication flows, and these tokens are consumed by applications across the organization.
