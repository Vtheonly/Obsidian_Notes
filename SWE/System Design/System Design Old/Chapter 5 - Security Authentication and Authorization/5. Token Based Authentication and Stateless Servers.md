# Token Based Authentication and Stateless Servers

## Overview
Token-based authentication is a stateless approach in which the server issues a cryptographically signed or encrypted token to the client after successful authentication. The client includes this token in subsequent requests, and the server validates the token's signature and claims without needing to look up session data in a store. This statelessness is the defining characteristic: the server does not maintain any per-client state, making the system inherently easier to scale horizontally because any server instance can validate any token independently. Token-based authentication has become the dominant pattern for APIs, single-page applications, mobile applications, and microservices architectures.

## Key Concepts

### How Token-Based Authentication Works
The flow begins when a client submits credentials (such as a username and password) to an authentication endpoint. The server validates the credentials and, if valid, generates a token containing claims about the authenticated identity (such as user ID, roles, expiration time, and issuer). The token is signed using a secret key (for HMAC-based signatures) or a private key (for asymmetric signatures like RSA or ECDSA). The signed token is returned to the client, which stores it (typically in localStorage, sessionStorage, or a secure cookie). On each subsequent request, the client includes the token in the Authorization header using the Bearer scheme (Authorization: Bearer \<token\>). The server verifies the signature, checks the expiration and other claims, and processes the request.

### Statelessness and Its Benefits
The primary advantage of token-based authentication is statelessness. Because all the information needed to authenticate and authorize a request is contained within the token itself, the server does not need to query a session store on every request. This eliminates the session store as a single point of failure and a bottleneck, simplifies horizontal scaling, and reduces latency. In a microservices architecture, each service can independently validate tokens without calling a central session service, which reduces inter-service communication and improves resilience. Stateless tokens also work seamlessly across different domains and services, which is essential for cross-origin and federated authentication scenarios.

### Stateless Limitations and Challenges
The statelessness of tokens introduces several challenges. Token revocation is the most significant: because the server does not maintain a list of active tokens, revoking a compromised token requires additional mechanisms such as a token blacklist, short expiration times with refresh token rotation, or version-based invalidation. Token size is another concern, especially for JWTs that contain many claims, because the token is sent with every request and increases bandwidth consumption. Tokens cannot be easily updated once issued; if a user's permissions change, the existing token will continue to reflect the old permissions until it expires or is refreshed. These tradeoffs must be carefully considered when designing a token-based system.

### Token Storage on the Client Side
Where and how the token is stored on the client has major security implications. Storing tokens in localStorage makes them accessible to JavaScript, which means they are vulnerable to XSS attacks. Storing tokens in HttpOnly, Secure cookies provides better protection against XSS but introduces CSRF vulnerability, which must be mitigated with CSRF tokens or SameSite cookie policies. Memory-only storage (storing the token in a JavaScript variable) is the most secure against XSS but loses the token on page refresh. Each approach has tradeoffs, and the best choice depends on the threat model. For high-security applications, a common pattern is to store access tokens in memory and refresh tokens in HttpOnly cookies.

### Token Types and Formats
Tokens can be opaque (random strings that reference server-side data) or self-contained (containing all claims within the token itself). Opaque tokens require a database or cache lookup on every request, which reintroduces statefulness but provides instant revocation. Self-contained tokens like JWTs are the most common format for stateless authentication. Other formats include SAML assertions (XML-based, used primarily in enterprise SSO), PASETO tokens (a safer alternative to JWT with versioned algorithms), and Macaroons (bearer credentials with caveats and third-party verification). The choice of token format depends on the ecosystem, interoperability requirements, and security properties needed.

## Practical Applications
Token-based authentication is the standard for REST APIs, where each request is independent and must carry all necessary context. OAuth 2.0 and OpenID Connect use token-based authentication as their foundation. In single-page applications (SPAs) built with React, Angular, or Vue, tokens enable API authentication without page reloads. Mobile applications use tokens because they cannot rely on cookies in the same way browsers do. In microservices architectures, services validate tokens locally using the signing key, eliminating the need for a central session store and reducing latency between services.
