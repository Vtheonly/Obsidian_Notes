# Single Sign On User Experience

## Overview
Single Sign-On (SSO) is an authentication scheme that allows a user to log in once and gain access to multiple independent applications or systems without re-entering credentials. From the user's perspective, SSO eliminates the friction of managing separate credentials for each application, reducing password fatigue and improving productivity. From the system design perspective, SSO centralizes authentication logic into a single identity provider, which simplifies credential management, enforces consistent security policies, and provides a single point of control for user access. SSO is the foundation of enterprise identity management and is implemented using protocols such as SAML, OpenID Connect, and OAuth 2.0.

## Key Concepts

### How SSO Works
In an SSO architecture, a central Identity Provider (IdP) is responsible for authenticating users and issuing authentication assertions. When a user attempts to access a Service Provider (SP) or Relying Party (RP), the application redirects the user to the IdP for authentication. If the user has an active session with the IdP, they are authenticated silently and redirected back to the application with an authentication token or assertion. If the user does not have an active session, they are prompted to log in at the IdP, and after successful authentication, a session is established at the IdP and the user is redirected back to the application. Subsequent visits to other applications in the SSO federation are automatically authenticated using the existing IdP session.

### SSO Protocols
SSO can be implemented using several protocols, each with different characteristics. SAML (Security Assertion Markup Language) is an XML-based protocol widely used in enterprise environments for browser-based SSO. It uses signed XML assertions called SAML tokens to convey authentication and authorization information between the IdP and SP. OpenID Connect (OIDC) is a modern, REST-friendly identity layer built on top of OAuth 2.0 that uses JWTs for identity tokens. OIDC is the preferred protocol for web and mobile applications. OAuth 2.0 itself can be used as a foundation for SSO when combined with OIDC, though OAuth 2.0 alone is an authorization framework, not an authentication protocol. Kerberos is another SSO protocol used primarily in on-premises Windows environments for network-level authentication.

### SSO Session Management
SSO introduces a hierarchical session model. The IdP maintains a master session, and each Service Provider maintains its own local session. When the user logs out of the IdP, the IdP should propagate the logout to all Service Providers with active sessions -- a process known as Single Logout (SLO). SLO is more complex than single sign-on because it requires coordination across all participants, and not all protocols handle it gracefully. SAML supports SLO through front-channel (redirect-based) and back-channel (SOAP-based) logout. OpenID Connect supports front-channel logout via iframes and back-channel logout via direct server-to-server calls. Failure to properly implement SLO can leave orphaned sessions on Service Providers, creating a security risk.

### Benefits of SSO
SSO provides significant benefits for both users and organizations. Users only need to remember one set of credentials, which reduces password fatigue and the temptation to reuse passwords across services. Fewer login prompts improve the user experience and increase productivity. Organizations benefit from centralized authentication management, which simplifies onboarding, offboarding, and compliance auditing. Security is improved because authentication policies (such as MFA requirements, password complexity, and session timeouts) are enforced consistently across all applications. The IdP also provides a central point for detecting and responding to suspicious authentication activity.

### Challenges and Risks of SSO
The centralization that makes SSO powerful also introduces risks. The IdP becomes a single point of failure -- if it goes down, users cannot access any application in the federation. The IdP is also a high-value target for attackers because compromising it grants access to all connected systems. For this reason, the IdP must be hardened with strong authentication (preferably MFA), robust monitoring, and high availability. Another challenge is that SSO can create a "keys to the kingdom" scenario: if a user's SSO credentials are compromised, an attacker gains access to all linked applications. This risk must be mitigated with MFA, anomaly detection, and session monitoring. Finally, implementing SSO across heterogeneous systems with different trust models and security requirements can be architecturally complex.

## Practical Applications
Enterprise organizations use SSO to give employees seamless access to internal tools like email, document management, CRM, and HR systems through a single corporate login. Google Workspace and Microsoft 365 provide SSO across their entire suite of productivity applications. Educational institutions use SSO (often via Shibboleth and SAML) to give students access to learning management systems, library resources, and campus services. Consumer-facing platforms like "Sign in with Google" or "Sign in with Apple" provide SSO across third-party applications, reducing registration friction while delegating authentication to a trusted provider.
