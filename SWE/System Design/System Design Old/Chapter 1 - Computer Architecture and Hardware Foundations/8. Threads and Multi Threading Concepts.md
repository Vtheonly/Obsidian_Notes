# Threads and Multi-Threading Concepts

## Overview

Threads are the fundamental unit of CPU scheduling and the primary mechanism for achieving parallelism within a single process. Understanding threads and multi-threading is essential for system design because modern CPUs have multiple cores, and utilizing them effectively requires concurrent execution. At scale, the difference between a single-threaded application that uses one core and a multi-threaded application that uses all available cores can be the difference between handling 8,000 requests per second and 42,000 requests per second on the same machine. However, multi-threading introduces complexity in the form of race conditions, deadlocks, and synchronization overhead that must be carefully managed.

## Key Concepts

### Threads vs Processes

A process is an instance of a running program with its own memory space, file descriptors, and system resources. A thread is a lightweight unit of execution within a process that shares the process's memory space and resources but has its own stack and program counter. Creating a new process (forking) is expensive because it requires duplicating the entire memory space, while creating a new thread is cheap because threads share memory. Multiple threads within a process can access the same global variables, heap memory, and file handles, which enables efficient communication but also requires careful synchronization to prevent data corruption. In practice, server applications use multi-threading to handle many concurrent requests within a single process, while multi-processing (running multiple instances) is used to scale beyond a single process.

### Context Switching

A context switch occurs when the operating system saves the state of one thread (its register values, stack pointer, and program counter) and loads the state of another thread so it can resume execution. Context switches happen when a thread blocks on I/O, when its time slice expires, or when a higher-priority thread becomes runnable. Each context switch incurs overhead: saving and restoring registers takes time, and the new thread may cause cache misses as it accesses different memory locations. Excessive context switching can significantly degrade performance, which is why server applications often use thread pools with a fixed number of threads rather than creating a new thread for every request.

### Hyper-Threading (Simultaneous Multi-Threading)

Hyper-threading is Intel's brand name for Simultaneous Multi-Threading (SMT), a technique that allows a single physical CPU core to appear as two logical cores to the operating system. Each physical core has duplicate architectural state (register sets) but shares the execution units, cache, and memory bus. When one thread stalls (for example, waiting for a cache miss or memory access), the other thread can use the idle execution units. Hyper-threading typically provides a 15-30% performance improvement for multi-threaded workloads, not the 2x improvement that two physical cores would provide. Understanding this distinction is important when provisioning servers: a 12-core CPU with hyper-threading presents 24 logical cores but cannot deliver 24 cores worth of compute performance.

### User Threads vs Kernel Threads

User-level threads are managed entirely by a user-space library without operating system involvement, making thread operations (create, switch, synchronize) very fast. However, if one user thread blocks on a system call, the entire process blocks because the kernel is unaware of the other user threads. Kernel-level threads are managed by the operating system, which can schedule each thread independently on different cores and handle blocking system calls correctly. Modern systems typically use a hybrid model (such as the 1:1 model in Linux) where each user thread is mapped to one kernel thread, combining the simplicity of kernel scheduling with the performance of user-space thread operations. Some runtime environments like Go use M:N models where many goroutines are multiplexed onto fewer OS threads.

### Thread Safety, Race Conditions, and Synchronization

Thread safety means that a data structure or function behaves correctly when accessed from multiple threads simultaneously. A race condition occurs when the outcome of a program depends on the timing of thread execution, leading to non-deterministic bugs that are difficult to reproduce and fix. For example, if two threads increment a shared counter without synchronization, some increments may be lost because the read-modify-write sequence is not atomic. Synchronization primitives like mutexes (mutual exclusion locks) ensure that only one thread can access a critical section at a time, while semaphores control access to a limited number of resources. However, excessive locking can serialize execution, eliminating the benefits of parallelism, and improper lock ordering can cause deadlocks where two or more threads wait indefinitely for each other. Designing thread-safe, deadlock-free code is one of the most challenging aspects of systems programming.

## Practical Applications

In a Node.js application, the single-threaded event loop can only utilize one CPU core. To scale across multiple cores, applications like PM2 launch multiple Node.js instances (a cluster), with a parent process distributing incoming connections across child processes. This is essentially multi-processing rather than multi-threading. In contrast, Java and Go applications can use true multi-threading within a single process, sharing memory and reducing the overhead of inter-process communication. When designing a high-throughput system, engineers must decide whether to use threads within a process or processes across cores, balancing the efficiency of shared memory against the simplicity and fault isolation of separate processes.
