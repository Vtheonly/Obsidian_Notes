# Kafka Partitions and Parallel Processing

## Overview

Partitions are the mechanism by which Kafka achieves horizontal scalability and parallel processing. Each topic is divided into one or more partitions, and each partition is an ordered, append-only log stored on a single broker. By splitting a topic into multiple partitions, Kafka allows multiple consumers to process events in parallel, with each consumer reading from a different partition. The number of partitions for a topic determines the maximum degree of parallelism: you cannot have more consumers in a single consumer group than there are partitions, because each partition can only be consumed by one consumer within a group at a time. Understanding partition design and routing is critical for building high-throughput Kafka applications.

## Key Concepts

### Partition as an Ordered Log

Each partition is an independent, ordered commit log. Events are appended to the partition in the order they arrive, and each event receives a sequential offset number. This ordering guarantee is strict within a partition: if event A is written before event B in the same partition, any consumer reading that partition will always see A before B. However, there is no ordering guarantee across different partitions. This means that events related to the same entity, such as all orders from the same customer, should be routed to the same partition if ordering matters for that entity.

### Key-Based Partitioning

When a producer sends an event with a non-null key, Kafka uses a hash function on the key to determine the target partition. This ensures that all events with the same key consistently land in the same partition, preserving their ordering. For example, if you use the customer ID as the key, all events for a given customer will be in the same partition and will be processed in order. This is essential for use cases like maintaining the correct sequence of account transactions or ensuring that status updates for the same order are applied in the correct sequence.

### Partition without Key

When an event is produced without a key, the producer uses a partitioning strategy that may be round-robin or sticky. Round-robin partitioning distributes events evenly across all partitions in strict rotation, ensuring balanced load distribution. Sticky partitioning, introduced in newer Kafka versions, batches events to the same partition for a period before switching, reducing the number of batch requests and improving throughput. Once events arrive at the broker, the lack of a key means there is no guarantee about which partition an event lands in, so ordering across events cannot be assured.

### Partition Count and Scalability

The number of partitions for a topic directly determines the maximum parallelism for consumption. If a topic has three partitions, at most three consumers in the same consumer group can process events concurrently, because each partition is assigned to exactly one consumer in the group. Adding more consumers beyond the partition count results in idle consumers that receive no assignments. Conversely, increasing the partition count allows more consumers to work in parallel, increasing throughput. However, more partitions also mean more files on disk, more memory for the broker to manage partition metadata, and potentially longer recovery times during broker failures.

### Partition Leader and Follower

Each partition has one broker designated as the leader and zero or more brokers designated as followers. All read and write requests for a partition go through the leader. Followers passively replicate the leader's log to maintain a copy of the data for fault tolerance. If the leader broker fails, one of the in-sync followers is elected as the new leader, and the partition becomes available again. This leader-follower model ensures that each partition is always available as long as at least one in-sync replica is alive, providing high availability without sacrificing consistency.

### Partition Reassignment

When a broker joins or leaves the cluster, Kafka may need to reassign partition leaders to distribute load evenly across the available brokers. Partition reassignment is a controlled operation that moves partition leadership and data from one broker to another. During reassignment, the partition remains available for reads and writes, but there may be a brief period of increased latency as the new leader catches up with the latest events. The Kafka administration tools provide commands for manually triggering partition reassignment and for monitoring the progress of ongoing reassignments.

## Practical Applications

In an e-commerce platform with global customers, the Orders topic might be partitioned by region: EU orders go to partition 0, US orders to partition 1, and Africa orders to partition 2. Each partition is consumed by a separate consumer instance, allowing parallel processing. When US orders spike during Black Friday, additional consumer instances can be deployed to process the US partition more quickly. In a ride-sharing application, driver location updates might be partitioned by driver ID, ensuring that each driver's location trail is maintained in order. An IoT platform might partition sensor data by device ID, enabling per-device analytics on separate consumers.
