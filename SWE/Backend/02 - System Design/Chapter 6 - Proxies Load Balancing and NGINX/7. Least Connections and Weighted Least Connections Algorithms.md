# Least Connections and Weighted Least Connections Algorithms

## Overview
Least Connections is a dynamic load balancing algorithm that routes each new request to the server with the fewest active connections at the time of the routing decision. Unlike Round Robin, which distributes requests blindly in rotation, Least Connections adapts to the actual state of the backend servers, sending more traffic to servers that are less busy and less traffic to those that are handling more concurrent requests. Weighted Least Connections extends this approach by factoring in server capacity weights, so that more powerful servers can handle more connections even when all servers have the same number of active connections. These algorithms are particularly effective when request processing times vary significantly, as they naturally balance the actual load rather than just the request count.

## Key Concepts

### Least Connections Algorithm
The Least Connections algorithm tracks the number of active connections (or concurrent requests) on each backend server. When a new request arrives, the load balancer selects the server with the lowest active connection count and forwards the request to it, then increments the server's connection count. When a connection is completed (the response is sent), the server's connection count is decremented. This approach is inherently self-balancing: if one server is processing long-running requests, its connection count stays high and new requests are directed to other servers. If another server finishes its requests quickly, its connection count drops and it receives more new requests. The algorithm adapts in real time to the varying processing requirements of different requests without requiring any configuration beyond the basic server pool definition.

### Weighted Least Connections Algorithm
Weighted Least Connections (WLC) extends the basic Least Connections algorithm by incorporating server weights. The weight represents the relative capacity of each server. Instead of selecting the server with the absolute fewest connections, WLC selects the server with the lowest ratio of active connections to weight. For example, if server A has weight 4 and 8 active connections (ratio 2.0), and server B has weight 2 and 3 active connections (ratio 1.5), the next request goes to server B even though it has a higher absolute connection count, because its weighted ratio is lower. This ensures that more powerful servers receive a proportionally larger share of traffic. In NGINX, this is configured using the "least_conn" directive in the upstream block combined with the "weight" parameter on each server.

### When to Use Least Connections
Least Connections is particularly effective in environments where request processing times vary significantly. Consider an API that handles both quick read queries (10ms) and expensive analytical queries (5s). Under Round Robin, a server that receives several expensive queries would fall behind because it continues to receive the same share of new requests. Under Least Connections, that server's connection count would increase, causing the load balancer to direct new requests to other servers until the expensive queries complete. This adaptive behavior makes Least Connections a better choice than Round Robin for most real-world workloads where request heterogeneity is the norm. It is also effective when server performance degrades unevenly, for example due to garbage collection pauses or resource contention.

### Connection Counting and Accuracy
The accuracy of the Least Connections algorithm depends on how accurately the load balancer tracks active connections. The simplest approach counts only the connections that are currently being processed by the load balancer, but this may not reflect the true state of the backend server -- the server may have additional connections from other sources or may be processing the request asynchronously. Some implementations allow the backend server to report its actual connection count to the load balancer through a side channel or health check endpoint. More sophisticated implementations track not just connection count but also request latency, error rate, and other metrics to make more informed routing decisions. The NGINX least_conn directive counts the number of requests currently being proxied to each server, which provides a reasonable approximation of server load for most workloads.

### Comparing LC and WLC with Round Robin
Round Robin is simpler and faster because it requires no state tracking beyond the current position in the server list. It works well when all servers have similar capacity and all requests require similar processing effort. Least Connections adds the overhead of tracking per-server connection counts but provides better load distribution for heterogeneous workloads. Weighted Least Connections adds the overhead of weight management but provides better distribution for heterogeneous server pools. The choice depends on the workload characteristics and operational complexity the team is willing to manage. In practice, many production systems start with Round Robin and switch to Least Connections when monitoring reveals uneven server utilization. The NGINX course material demonstrates Round Robin as the default but recommends Least Connections for most production scenarios.

## Practical Applications
NGINX supports the least_conn directive for upstream blocks, using it to route requests to the server with the fewest active connections. HAProxy implements Least Connections as the "leastconn" balance algorithm. Database load balancers like ProxySQL use least-connections algorithms to route queries to the replica with the fewest active queries, which is particularly important for databases with varying query costs. Real-time communication servers (WebSockets, long polling) benefit significantly from Least Connections because each connection can have a very different duration. Cloud load balancers like AWS NLB support Least Outstanding Requests (a variant of Least Connections) for TCP traffic, routing new connections to the target with the fewest outstanding requests.
