# Round Robin and Weighted Round Robin Algorithms

## Overview
Round Robin and Weighted Round Robin are the most fundamental and widely used load balancing algorithms. Round Robin distributes requests sequentially across all servers in the pool, cycling through them in order. Weighted Round Robin extends this concept by assigning a weight to each server, allowing more powerful servers to receive a proportionally larger share of requests. These algorithms are simple to understand, implement, and reason about, making them the default choice for many load balancers including NGINX. However, their simplicity also means they do not account for the actual load on each server or the varying processing requirements of different requests, which can lead to uneven distribution in practice.

## Key Concepts

### Round Robin Algorithm
The Round Robin algorithm maintains a circular list of servers and assigns each incoming request to the next server in the list, cycling back to the first server after reaching the end. For example, with servers A, B, and C, requests are assigned as A, B, C, A, B, C, and so on. This provides an equal distribution of requests across all servers, assuming all requests require similar processing effort and all servers have similar capacity. Round Robin is stateless -- the only state maintained is the current position in the server list, which requires minimal memory and computation. This makes Round Robin extremely fast and suitable for high-throughput environments. NGINX uses Round Robin as its default load balancing algorithm when no other method is specified.

### Weighted Round Robin Algorithm
Weighted Round Robin (WRR) extends Round Robin by assigning a weight to each server that determines the proportion of requests it should receive. If server A has weight 5, server B has weight 3, and server C has weight 2, then out of every 10 requests, server A receives 5, server B receives 3, and server C receives 2. The algorithm distributes requests in a sequence that respects these proportions, for example: A, A, A, A, A, B, B, B, C, C. In NGINX, weights are configured in the upstream block using the weight parameter: "server backend1.example.com weight=5; server backend2.example.com weight=3;". WRR is appropriate when backend servers have different capacities (for example, different CPU cores, memory, or network bandwidth) and the administrator wants to direct more traffic to more powerful servers.

### Smooth Weighted Round Robin
A naive implementation of WRR can produce bursty traffic patterns where a server receives several consecutive requests before the next server gets any. For example, with weights 5:1, the sequence would be A, A, A, A, A, B, which sends five consecutive requests to server A. Smooth Weighted Round Robin (also called smooth WRR or interleaved WRR) distributes requests more evenly by interleaving the assignments. Using the same weights 5:1, a smooth sequence might be A, A, B, A, A, B, which spreads the load more evenly over time. The NGINX implementation uses a smooth weighted round-robin algorithm that ensures a more uniform distribution of requests across the cycle, reducing the likelihood that a server receives a burst of requests that could overwhelm it momentarily.

### Advantages and Limitations
The primary advantage of Round Robin and Weighted Round Robin is their simplicity. They are easy to implement, configure, and debug. They require no communication between the load balancer and the backend servers (unlike least-connections algorithms), making them suitable for any backend technology. However, their simplicity is also their main limitation. They treat all requests as equal, regardless of their processing cost. A request that takes 100 milliseconds to process receives the same weight as one that takes 10 seconds. This can lead to imbalanced server loads when request types are heterogeneous. Similarly, if a server starts responding slowly due to garbage collection or resource contention, Round Robin continues sending it the same share of traffic, potentially overloading it further. For these reasons, Round Robin works best in environments where requests are relatively uniform and servers are similarly provisioned.

### Configuring Weights Effectively
Setting appropriate weights requires understanding each server's capacity relative to the others. A common approach is to set weights proportional to the server's capacity (for example, if server A has 4 CPU cores and server B has 2 CPU cores, server A might receive weight 4 and server B weight 2). However, capacity is not solely determined by CPU -- memory, disk I/O, network bandwidth, and the specific workload characteristics all affect a server's effective capacity. In practice, weights are often determined through load testing and empirical observation. Weights should be adjusted when servers are upgraded, when workload characteristics change, or when monitoring shows uneven resource utilization. In cloud environments with auto-scaling, all instances typically have the same capacity and Round Robin (without weights) is usually sufficient.

## Practical Applications
NGINX uses Round Robin as its default upstream balancing algorithm and supports the weight parameter for Weighted Round Robin. DNS round-robin uses the same principle to distribute traffic at the DNS level, returning different IP addresses for the same domain name in rotating order. Cloud load balancers like AWS ALB and GCP Load Balancer use Round Robin as a baseline algorithm, often enhanced with health checking and session persistence. Database read query distribution often uses Round Robin across read replicas when queries are relatively uniform in cost. Content delivery networks use Weighted Round Robin at the origin level to direct different proportions of traffic to different origin servers based on their capacity and geographic proximity.
