# NGINX Reverse Proxy Setup

## Overview
Configuring NGINX as a reverse proxy is one of its most powerful and commonly used capabilities. As the source material emphasizes, the reverse proxy pattern allows a single NGINX instance to sit between clients and multiple backend servers, routing requests to the appropriate backend based on URL paths or other request attributes. This centralizes routing, simplifies the client configuration (clients interact with a single endpoint), and enables NGINX to implement cross-cutting concerns like SSL termination, compression, and caching. The source material demonstrates using NGINX to combine a frontend application and a backend API into a single web application served from one port, with NGINX routing requests based on the URL path. Understanding the reverse proxy setup is the foundation for all advanced NGINX use cases including load balancing, SSL termination, and microservices routing.

## Key Concepts

### The proxy_pass Directive
The "proxy_pass" directive is the core mechanism for configuring NGINX as a reverse proxy. It specifies the protocol and address of the backend server to which the request should be forwarded. For example, "proxy_pass http://localhost:3000;" forwards the request to a server running on port 3000. The proxy_pass directive can be placed inside a location block, allowing different URL paths to be proxied to different backends. When NGINX receives a request matching the location, it establishes a new HTTP connection to the backend, forwards the request (with modified headers), receives the response, and returns it to the client. The source material demonstrates this with two location blocks: one for the frontend (proxying to localhost:3000) and one for the API (proxying to localhost:8000 with the /api prefix).

### Header Forwarding
When NGINX acts as a reverse proxy, the backend server sees the request as coming from NGINX rather than the original client. To preserve information about the original client, NGINX should be configured to forward relevant headers. The "proxy_set_header" directive modifies headers before forwarding the request. Common configurations include: "proxy_set_header Host $host" (passes the original Host header), "proxy_set_header X-Real-IP $remote_addr" (passes the client's IP address), "proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for" (appends the client IP to the X-Forwarded-For chain), and "proxy_set_header X-Forwarded-Proto $scheme" (passes the original protocol, http or https). These headers are critical for backend applications that need to know the original client's IP address (for logging, rate limiting, or geolocation) or the original protocol (for generating correct redirect URLs).

### Path-Based Routing
One of the most common reverse proxy patterns is path-based routing, where different URL paths are forwarded to different backend services. For example, "location / { proxy_pass http://frontend:3000; }" routes all root requests to the frontend, while "location /api/ { proxy_pass http://backend:8000; }" routes API requests to the backend. The source material demonstrates exactly this pattern. When using path-based routing, it is important to understand how NGINX handles the URL path in the proxy_pass directive. If proxy_pass includes a URI (even just a trailing slash), NGINX replaces the matched location prefix with the URI in proxy_pass. If proxy_pass does not include a URI, the full original URI is forwarded unchanged. This distinction affects how the backend receives the request path and must be configured correctly for each application.

### WebSocket Proxying
WebSocket connections require special handling in a reverse proxy because they upgrade the HTTP connection to a persistent, bidirectional TCP connection. NGINX supports WebSocket proxying with specific configuration directives. The "proxy_http_version 1.1" directive sets the proxy protocol version to HTTP/1.1 (required for WebSocket upgrade). The "proxy_set_header Upgrade $http_upgrade" directive forwards the Upgrade header from the client. The "proxy_set_header Connection 'upgrade'" directive sets the Connection header to "upgrade" to initiate the protocol upgrade. The "proxy_cache_bypass $http_upgrade" directive ensures that WebSocket upgrade requests are not cached. These directives must be included in any location block that handles WebSocket traffic. Without them, the WebSocket handshake will fail because the proxy will not forward the upgrade request to the backend.

### Buffering and Timeouts
NGINX by default buffers responses from the backend server, storing the entire response in memory or temporary files before sending it to the client. This buffering improves performance because NGINX can receive the response at the backend's speed and send it to the client at the client's speed, without blocking either side. However, buffering can increase latency for streaming responses or server-sent events. The "proxy_buffering off" directive disables buffering for real-time applications. Timeout directives control how long NGINX waits for the backend server: "proxy_connect_timeout" (time to establish a connection), "proxy_send_timeout" (time between two successive write operations), and "proxy_read_timeout" (time between two successive read operations). These defaults are typically 60 seconds but should be adjusted based on the backend's expected response times. For long-running operations, increasing the read timeout prevents NGINX from prematurely terminating the connection.

## Practical Applications
The source material demonstrates using NGINX as a reverse proxy to combine a frontend and backend into a single web application. Microservices architectures use NGINX to route requests to different services based on URL path, with each service running on a different port or host. API gateways built on NGINX proxy requests to backend API services while adding authentication, rate limiting, and transformation. Legacy application modernization often uses NGINX as a reverse proxy to gradually migrate traffic from a monolithic application to new microservices. Development environments use NGINX reverse proxy to provide a unified local development URL that routes to multiple services running on different ports.
