# API Pagination: Limits, Offsets, and Cursors

## Overview
Pagination is the technique of dividing a large dataset into smaller, manageable chunks or pages for delivery to clients. Without pagination, a request to a collection endpoint could return millions of records, overwhelming the client, consuming excessive bandwidth, and placing untenable load on the database. Pagination is one of the most important API design decisions, and the choice between offset-based and cursor-based approaches has significant implications for consistency, performance, and user experience.

## Key Concepts

### Offset-Based Pagination
Offset-based pagination uses limit and offset parameters to specify the page size and starting position. A request like GET /products?limit=20&offset=40 returns the third page of 20 products. The offset tells the database to skip the first 40 records and return the next 20. This approach is intuitive and allows clients to jump to any page directly. However, offset-based pagination suffers from two significant problems. First, if records are inserted or deleted between requests, the offset may point to a different set of records, causing items to be skipped or duplicated. Second, large offsets are inefficient because the database must scan and discard all preceding rows, making queries like OFFSET 1000000 extremely slow on large tables.

### Cursor-Based Pagination
Cursor-based pagination uses an opaque token, called a cursor, that marks the position in the result set. After the first request (GET /products?limit=20), the response includes a next_cursor token that the client uses to fetch the next page (GET /products?limit=20&cursor=abc123). The cursor typically encodes the values of the sort key from the last record on the current page, allowing the database to use an index seek instead of scanning from the beginning. Cursor-based pagination provides consistent results even when records are added or removed between requests, and it maintains consistent performance regardless of how deep into the dataset the client navigates.

### Keyset Pagination
Keyset pagination is a specific implementation of cursor-based pagination where the cursor is the value of the sort column from the last record. For example, if sorting by created_at descending, the cursor would be the timestamp of the last record, and the query would use WHERE created_at < cursor_value. This approach is highly efficient because it leverages database indexes and avoids the row scanning overhead of offset-based pagination. However, keyset pagination only supports forward and backward navigation, not random page access.

### Response Format
Paginated responses should include the data items along with metadata that enables navigation. A common format includes the items array, a total count of matching records (useful for rendering page indicators), and navigation links or tokens (next_cursor, prev_cursor). Some APIs use Link headers (RFC 8288) to provide pagination URLs, following web standards. The response should also include the limit that was applied and, for offset-based pagination, the current offset.

### Choosing Between Offset and Cursor Pagination
Offset-based pagination is appropriate when the dataset is relatively small, random page access is required (such as jumping to page 5 of search results), and occasional inconsistencies due to concurrent modifications are acceptable. Cursor-based pagination is preferred when the dataset is large, consistent iteration is required (such as syncing data between systems), and performance must remain constant regardless of position in the dataset. Many APIs offer both approaches, using offset pagination for the user interface and cursor pagination for data synchronization.

### Pagination and Rate Limiting
Pagination interacts with rate limiting in important ways. A client that paginates through an entire dataset of 100,000 records with a limit of 100 makes 1,000 API calls, which may exceed rate limits. APIs should document the expected pagination patterns and set rate limits accordingly. For bulk data access, dedicated export endpoints or streaming APIs may be more appropriate than iterating through paginated results. Some APIs offer a total_count field in paginated responses but disable it for performance reasons on very large datasets, requiring clients to paginate until an empty page is returned.

## Practical Applications
A social media feed API uses cursor-based pagination with the post ID as the cursor, ensuring that users see a consistent feed even as new posts are created while they scroll. An admin dashboard for managing a product catalog uses offset-based pagination because users need to jump to specific page numbers. A data synchronization service between a CRM and marketing platform uses keyset pagination on the last_modified timestamp to efficiently fetch only the records that have changed since the last sync.
