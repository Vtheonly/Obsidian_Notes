# GraphQL API Architecture and Single Endpoints

## Overview
GraphQL is a query language for APIs and a runtime for executing those queries against your data. Developed by Facebook in 2012 and released as open source in 2015, GraphQL offers a fundamentally different approach to API design compared to REST. Instead of exposing multiple endpoints for different resources, GraphQL exposes a single endpoint that accepts queries describing exactly what data the client needs. This architecture addresses key limitations of REST, particularly over-fetching and under-fetching, while providing strong typing, introspection, and a powerful developer experience.

## Key Concepts

### Single Endpoint Architecture
In a GraphQL API, all queries and mutations are sent to a single endpoint, typically /graphql. The client specifies the exact shape and fields of the data it needs in the request body, and the server returns a JSON response with that exact shape. This contrasts with REST, where the server determines the shape of the response, often returning more or less data than the client needs. The single endpoint simplifies client configuration, enables powerful tooling, and makes it straightforward to evolve the API without versioning.

### Query Language
GraphQL queries use a declarative syntax that mirrors the shape of the response. A query like `{ user(id: 123) { name email posts { title } } }` requests a user's name, email, and the titles of their posts. The response JSON has the same structure: `{ "user": { "name": "Alice", "email": "alice@example.com", "posts": [{ "title": "Hello World" }] } }`. This declarative approach eliminates the need for multiple round trips to fetch related data and gives clients complete control over the data they receive.

### The Schema
Every GraphQL API is defined by a schema that specifies the types, fields, and relationships available in the API. The schema is the contract between the client and server, serving a similar purpose to OpenAPI specifications for REST. The schema defines query types for reading data, mutation types for modifying data, and subscription types for real-time updates. Types can be scalar (String, Int, Float, Boolean, ID), object types with fields, enum types, input types for mutations, and interface or union types for polymorphism.

### Resolvers
Resolvers are functions that fetch the data for each field in a query. When a query is received, the GraphQL execution engine traverses the query tree and calls the appropriate resolver for each field. Resolvers can fetch data from databases, call external APIs, read from caches, or compute values on the fly. This resolver architecture decouples the API schema from the data sources, allowing a single GraphQL API to aggregate data from multiple backends, including REST APIs, databases, and microservices.

### Introspection
GraphQL APIs support introspection, which allows clients to query the schema itself. Tools like GraphiQL and Apollo Explorer use introspection to provide auto-complete, type validation, and interactive documentation. Introspection queries return information about all types, fields, arguments, and directives defined in the schema, making GraphQL APIs self-documenting. In production environments, introspection may be disabled for security reasons to prevent attackers from discovering the API structure.

### Batched Queries and Performance
GraphQL allows clients to batch multiple queries into a single request, reducing the number of HTTP round trips. However, this capability introduces the N+1 query problem: if a query requests posts and their authors, and the resolver fetches each author individually, the number of database queries grows linearly with the number of posts. DataLoader is a popular solution that batches and caches resolver calls within a single request, ensuring that each unique data item is fetched only once. Persisted queries, where the client sends a hash instead of the full query text, reduce bandwidth and improve caching.

## Practical Applications
A mobile e-commerce application uses GraphQL to fetch exactly the data needed for each screen: the product listing screen queries product names and thumbnails, while the product detail screen queries full descriptions, reviews, and related products. This eliminates over-fetching on the listing screen and under-fetching on the detail screen. The GraphQL API aggregates data from a product service, a review service, and an inventory service, providing a unified interface that shields mobile clients from the complexity of the backend microservices architecture.
