# Idempotency in API Requests

## Overview
Idempotency is a property of an operation whereby performing it multiple times produces the same result as performing it once. In the context of APIs, idempotency ensures that clients can safely retry failed requests without causing unintended side effects like duplicate charges, duplicate records, or inconsistent state. Understanding which HTTP methods are inherently idempotent, how to design idempotent operations, and how to implement idempotency keys is essential for building reliable distributed systems that gracefully handle network failures, timeouts, and retries.

## Key Concepts

### Idempotent HTTP Methods
The HTTP specification defines GET, PUT, DELETE, HEAD, and OPTIONS as idempotent methods. GET is idempotent because reading a resource multiple times returns the same data without changing server state. PUT is idempotent because replacing a resource with the same data multiple times results in the same final state. DELETE is idempotent because deleting a resource that no longer exists leaves the server in the same state as deleting it once (though the second DELETE may return 404 instead of 204). POST and PATCH are not inherently idempotent: POSTing the same data twice creates two resources, and PATCHing with array additions can produce different results on repeated application.

### Why Idempotency Matters
Network communication is inherently unreliable. Requests can timeout before the client receives a response, connections can be interrupted, and load balancers can drop connections during server failures. Without idempotency, a client that retries a failed payment request might charge the customer twice. An order submission that is retried might create duplicate orders. Idempotency allows clients to retry any failed request with confidence that the outcome will be the same as if the request had succeeded on the first attempt.

### Idempotency Keys
For operations that are not inherently idempotent (primarily POST requests), idempotency keys provide a mechanism to make them safe for retry. The client generates a unique key (typically a UUID) and sends it with the request in a header like Idempotency-Key. The server stores the key and the result of the first successful request. If a subsequent request arrives with the same key, the server returns the stored result without re-executing the operation. This approach is used by Stripe for payment processing: each charge request includes an idempotency key, ensuring that a network timeout followed by a retry does not result in a double charge.

### Implementing Idempotency Keys
The server must persist idempotency keys long enough to cover the retry window, which is typically 24 hours. Keys are stored in a database or cache with a unique constraint to prevent concurrent processing. When a request arrives, the server first checks if the key exists in the store. If it does, the server returns the stored response. If it does not, the server processes the request, stores the response with the key, and returns the result. Race conditions between concurrent requests with the same key must be handled through locking or conditional writes. Keys should be expired after the retention period to prevent unlimited storage growth.

### Designing Idempotent Operations
Beyond idempotency keys, individual operations can be designed to be naturally idempotent. Using unique identifiers for resource creation (such as client-generated order IDs) ensures that creating the same order twice results in the same order, not a duplicate. Using conditional updates with ETags or If-Match headers ensures that a PUT request only succeeds if the resource has not been modified since the client last read it, preventing lost updates. Using upsert patterns (update if exists, create if not) makes creation operations idempotent by default.

### Idempotency in Distributed Systems
In microservices architectures, idempotency is critical because requests may traverse multiple services, and any service in the chain can fail. If a request fails at the third service, the first two services may have already committed their changes. Without idempotency, retrying the request would cause the first two services to apply their changes again. Making each service's operation idempotent ensures that retries produce the correct final state. Saga patterns in event-driven architectures rely on idempotent compensating transactions to handle failures without creating inconsistent state.

## Practical Applications
A payment processing system requires idempotency keys on all charge endpoints. When a client submits a charge with key "abc-123", the server processes the payment and stores the result keyed by "abc-123". If the client retries due to a timeout, the server finds the existing key and returns the stored result without processing a second charge. In a task queue system, workers that process messages use message IDs as idempotency keys, ensuring that a message delivered twice due to a broker redelivery does not result in duplicate task execution.
