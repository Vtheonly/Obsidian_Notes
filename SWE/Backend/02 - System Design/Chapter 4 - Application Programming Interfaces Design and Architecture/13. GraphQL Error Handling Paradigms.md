# GraphQL Error Handling Paradigms

## Overview
Error handling in GraphQL follows a fundamentally different paradigm than REST. While REST uses HTTP status codes to indicate success or failure for the entire request, GraphQL always returns HTTP 200 OK (unless a server-level error prevents parsing the request) and communicates errors within the response body. This design reflects GraphQL's ability to partially succeed: a single query can request data from multiple fields, and some fields may succeed while others fail. Understanding how to structure error responses, handle partial failures, and communicate errors to clients is essential for building robust GraphQL APIs.

## Key Concepts

### The Errors Array
GraphQL responses include an optional errors array alongside the data object. Each error entry contains a message field, a locations array indicating the line and column of the offending query element, and a path array showing the field path to the error. The presence of errors does not necessarily mean the entire request failed; the data object may contain successfully resolved fields alongside null values for fields that encountered errors. This partial success model is one of GraphQL's key advantages, allowing clients to display whatever data is available rather than failing completely.

### Partial Failure Handling
In a REST API, a server error typically returns a 500 status code with no data. In GraphQL, if a query requests user data and their posts, and the user resolver succeeds but the posts resolver fails, the response includes the user data in the data object and an error in the errors array with a path pointing to the posts field. The posts field in the data object is set to null. This graceful degradation means that a client can still display the user's profile even if the posts feed is unavailable, providing a much better user experience than a complete failure.

### Custom Error Types
The basic GraphQL error format is limited to a message string and standard metadata. For richer error information, APIs implement custom error extensions using the extensions field in error objects. A validation error might include extensions with a code (VALIDATION_ERROR), a field path (email), and a detailed message (Email format is invalid). This structured approach allows clients to map error codes to user-facing messages, highlight invalid form fields, and take programmatic action based on the error type. Union types can also be used to return typed error objects as part of the data rather than in the errors array.

### Mutation Error Patterns
Mutations can fail for various reasons: validation errors, authorization failures, conflict errors, or internal errors. Two common patterns exist for handling mutation errors. The first pattern uses the errors array, where the mutation returns null in the data object on failure with details in the errors array. The second pattern uses a result type that can be either a success payload or an error payload, leveraging GraphQL's type system to make errors explicit in the schema. For example, a CreateTaskResult union type might resolve to CreateTaskSuccess on success or CreateTaskError on failure, with the error type containing specific error fields like code and message.

### Authentication and Authorization Errors
Authentication errors (missing or invalid credentials) and authorization errors (insufficient permissions) are typically handled in the errors array with specific error codes. Since GraphQL operates over HTTP, unauthenticated requests can also be rejected at the HTTP level before reaching the GraphQL execution engine, returning a 401 status code. However, for consistency with the GraphQL error model, most APIs return HTTP 200 with an authentication error in the errors array. This allows clients to use a single error handling path for all GraphQL errors.

### Error Logging and Monitoring
GraphQL errors must be logged and monitored carefully because the always-200 response pattern means that traditional HTTP monitoring tools will not detect application-level errors. Server-side logging should capture the full error details including the path, the original query, and the variables. Monitoring dashboards should track error rates by type, field, and operation. Alerting should trigger when error rates exceed thresholds, distinguishing between expected client errors (like validation failures) and unexpected server errors (like database connection failures).

## Practical Applications
An e-commerce GraphQL API uses typed error results for mutations: a PlaceOrderResult union resolves to Order on success or to PlaceOrderError on failure, where PlaceOrderError includes fields like code (OUT_OF_STOCK, PAYMENT_DECLINED, INVALID_ADDRESS) and affectedItems. On the query side, if the product catalog service is unavailable, the products field returns null with an error, while the user's cart and profile data are still returned successfully. The client displays the available data and shows an error message specifically for the products section, maintaining a functional experience despite the partial failure.
