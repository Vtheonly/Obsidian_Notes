# Over-Fetching and Under-Fetching Solutions

## Overview
Over-fetching and under-fetching are two of the most common data delivery problems in API design. Over-fetching occurs when an API returns more data than the client needs, wasting bandwidth, increasing latency, and consuming unnecessary server resources. Under-fetching occurs when an API returns less data than the client needs, forcing the client to make additional requests to gather the complete information. Both problems degrade application performance and developer experience, and they are primary motivations for adopting GraphQL and other modern API paradigms over traditional REST.

## Key Concepts

### Over-Fetching in REST
In a typical REST API, each endpoint returns a fixed data shape determined by the server. When a mobile app displays a list of product names and prices, it might call GET /products and receive complete product objects including descriptions, images, reviews, specifications, and inventory data. The client discards 90% of the response because it only needs the name and price. This wasted bandwidth is particularly harmful for mobile users on slow or metered connections. Over-fetching also increases server load because the database must query and the application must serialize fields that no client will ever use.

### Under-Fetching in REST
Under-fetching is the inverse problem: a client needs data from multiple related resources and must make separate requests for each one. A mobile app displaying a user profile with their recent posts and follower count might need to call GET /users/123, GET /users/123/posts, and GET /users/123/followers/count. These three sequential requests introduce significant latency, especially on high-latency mobile networks. This problem is sometimes called the N+1 request problem, where an initial request returns a list of items and the client must make N additional requests to fetch related data for each item.

### GraphQL as a Solution
GraphQL directly addresses both over-fetching and under-fetching. The client specifies exactly which fields it needs, and the server returns only those fields, eliminating over-fetching. The client can also query related data in a single request, eliminating under-fetching. A single GraphQL query can fetch the user's name, their 5 most recent post titles, and their follower count in one round trip. This precise data fetching makes GraphQL particularly valuable for mobile applications where bandwidth and latency are constrained.

### REST Mitigations
REST APIs can mitigate over-fetching and under-fetching through several techniques. Field selection parameters (GET /products?fields=name,price) allow clients to specify which fields to include in the response. Compound documents or sideloading (GET /users/123?include=posts,followers) allow clients to request related resources in a single response. Specialized endpoints for common use cases (GET /products/summary) return pre-defined lightweight representations. GraphQL-style nested endpoints (GET /users/123?expand=posts(limit:5),followers(count)) provide more granular control. Each of these approaches adds complexity to the REST API but can significantly improve performance.

### BFF Pattern
The Backend for Frontend (BFF) pattern addresses over-fetching and under-fetching by creating dedicated backend services for each client type. The mobile BFF aggregates data from multiple services and returns a response tailored to the mobile app's needs. The web BFF does the same for the desktop website. Each BFF makes multiple internal calls and composes the results into a single client-specific response. This pattern eliminates over-fetching and under-fetching for each client type without modifying the underlying service APIs. However, it introduces additional infrastructure to maintain and deploy.

### Caching Considerations
REST benefits from HTTP caching at multiple levels: browser cache, CDN cache, and proxy cache. Each resource URL can be cached independently with its own Cache-Control and ETag headers. GraphQL's single endpoint complicates caching because different queries to the same URL request different data. Normalized client-side caches (like Apollo Client) address this by caching data at the entity level rather than the query level. Server-side caching for GraphQL requires persisted queries with stable URLs or custom caching logic based on query analysis. The caching trade-off is one of the most important considerations when choosing between REST and GraphQL.

## Practical Applications
A mobile banking app uses GraphQL to fetch exactly the data needed for each screen: the account overview queries balances and recent transactions, while the transaction detail screen queries the full transaction with merchant information and categorization. This approach reduces the average response size by 70% compared to the previous REST API. For the web application, which has fewer bandwidth constraints, the same GraphQL endpoint serves richer queries that include charts and detailed analytics. The API team avoids creating separate REST endpoints for each screen by letting clients compose their own queries.
