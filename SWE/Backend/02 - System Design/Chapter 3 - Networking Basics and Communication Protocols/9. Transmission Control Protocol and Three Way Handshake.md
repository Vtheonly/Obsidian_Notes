# Transmission Control Protocol and Three-Way Handshake

## Overview
The Transmission Control Protocol (TCP) is the dominant transport layer protocol on the internet, providing reliable, ordered, and error-checked delivery of data between applications. TCP achieves this reliability through a combination of connection management, sequence numbering, acknowledgments, and retransmission mechanisms. The three-way handshake is the foundational process by which TCP establishes a connection between two endpoints before any data is transferred, ensuring that both parties are ready to communicate and agree on initial sequence numbers.

## Key Concepts

### The Three-Way Handshake
TCP connections are established through a three-step process. In the first step, the client sends a SYN (synchronize) packet to the server, which includes the client's initial sequence number. In the second step, the server responds with a SYN-ACK packet, acknowledging the client's SYN by sending an acknowledgment number equal to the client's sequence number plus one, and including the server's own initial sequence number. In the third step, the client sends an ACK packet acknowledging the server's SYN, and the connection is established. This three-step exchange ensures that both endpoints are reachable and agree on the starting sequence numbers for data transmission in both directions.

### Sequence Numbers and Acknowledgments
Every byte of data transmitted over TCP is assigned a sequence number. The sending host increments the sequence number by the number of bytes sent, allowing the receiver to reorder out-of-order packets and detect missing data. The receiver sends acknowledgment packets (ACK) that indicate the next expected sequence number. If the sender does not receive an ACK within a specified timeout period, it retransmits the unacknowledged data. This mechanism guarantees that all data eventually reaches the destination in the correct order.

### Flow Control with Sliding Windows
TCP implements flow control using a sliding window mechanism. The receiver advertises a window size in its ACK packets, indicating how many bytes of data it is willing to accept before the sender must wait for an acknowledgment. If the receiver's buffer is filling up, it reduces the window size, causing the sender to slow down. When the receiver has processed data and freed buffer space, it increases the window size, allowing the sender to transmit more data. This dynamic adjustment prevents fast senders from overwhelming slow receivers.

### Congestion Control Algorithms
TCP congestion control prevents senders from overwhelming the network itself. The slow start algorithm begins by sending a small amount of data and doubling the congestion window for each round trip until packet loss occurs. When loss is detected, TCP reduces the congestion window and enters congestion avoidance mode, which increases the window linearly. Fast retransmit and fast recovery allow TCP to recover from a single lost packet without waiting for a timeout, significantly reducing the delay in retransmission.

### Connection Termination
TCP connections are terminated through a four-way handshake. Each side independently closes its end of the connection by sending a FIN packet. The other side acknowledges the FIN with an ACK. After both sides have sent and acknowledged FIN packets, the connection is fully closed. The side that sends the first FIN enters a TIME_WAIT state that lasts for twice the maximum segment lifetime, ensuring that any delayed packets from the old connection are discarded before a new connection can use the same port pair.

### TCP Header Structure
The TCP header contains critical fields including source port, destination port, sequence number, acknowledgment number, data offset, control flags (SYN, ACK, FIN, RST, PSH, URG), window size, checksum, and urgent pointer. These fields enable the protocol to manage connections, ensure reliable delivery, and implement flow and congestion control.

## Practical Applications
Every HTTP request to a web server, every database query sent to PostgreSQL, and every SSH session relies on TCP for reliable data transport. Understanding TCP is essential for debugging network performance issues, such as high latency caused by excessive retransmissions or throughput limitations caused by small TCP window sizes. Tuning TCP parameters like the initial congestion window, keepalive intervals, and maximum segment size can significantly improve application performance.
