# Redis Clustering Strategies Masters Replicas and Slot Hashing

## Overview

When a single Redis instance cannot handle the throughput or memory requirements of a high-scale application, Redis Cluster provides a distributed solution that partitions data across multiple nodes. Redis Cluster uses hash slot-based sharding to distribute keys across a set of master nodes, with each master node responsible for a subset of the 16,384 hash slots. Replica nodes provide data redundancy and failover capability, ensuring that the cluster remains available if a master node fails. Understanding how Redis Cluster distributes data, how replicas interact with masters, and how to configure the cluster for optimal performance is essential for building Redis-backed systems that can scale beyond the capacity of a single instance. At the scale of one million requests per second, Redis clustering is not a luxury -- it is a necessity.

## Key Concepts

### Hash Slot Architecture

Redis Cluster divides the key space into 16,384 hash slots. Each key is assigned to a slot using the formula `CRC16(key) % 16384`. Each master node in the cluster is responsible for a subset of these slots. When a client sends a command to a node, the node checks whether the key's slot belongs to it. If it does, the node processes the command directly. If it does not, the node responds with a `MOVED` redirect, telling the client which node owns the slot. Modern Redis clients cache the slot-to-node mapping and route requests directly to the correct node, minimizing redirections. This architecture allows the cluster to scale horizontally: adding more master nodes redistributes the hash slots, increasing both throughput and memory capacity.

### Master and Replica Nodes

Each master node in a Redis Cluster can have one or more replica nodes. The replica continuously replicates the master's data, providing a hot standby that can be promoted to master if the original master fails. Redis Cluster uses an asynchronous replication model: the master sends write commands to replicas, but it does not wait for replicas to acknowledge before responding to the client. This means that in the event of a master failure, the replica may be slightly behind, and some recently written data may be lost. The trade-off between consistency and performance is configurable: using the `WAIT` command, a client can block until a specified number of replicas have acknowledged a write, at the cost of increased latency. For the write-buffer pattern, where Redis is used as a temporary queue, the slight risk of data loss on failover is usually acceptable.

### Slot Migration and Rebalancing

When nodes are added to or removed from the cluster, hash slots must be migrated to maintain a balanced distribution. Redis Cluster supports online slot migration: the source node gradually transfers key-value pairs to the destination node while continuing to serve requests. During migration, the source node acts as the slot owner, forwarding requests to the destination node if the key has already been migrated. Once all keys for a slot have been transferred, the slot's ownership is officially changed. This online migration capability allows the cluster to be resized without downtime, which is critical for production systems that cannot tolerate maintenance windows. The `redis-cli --cluster rebalance` command automates the rebalancing process.

### Cross-Slot Operations and Limitations

A significant limitation of Redis Cluster is that multi-key operations (such as MGET, MSET, or Lua scripts that access multiple keys) only work when all keys are in the same hash slot. To ensure that related keys are in the same slot, Redis supports hash tags: if a key contains curly braces (e.g., `user:{123}:name`), only the string inside the braces is used for hashing. This allows related keys to be colocated on the same node. For queue workloads where each operation typically involves a single key, this limitation is not a concern. However, for workloads that require multi-key transactions or complex Lua scripts, the hash tag mechanism must be used carefully to avoid data inconsistency.

### Cluster Performance and Scaling

A Redis Cluster with six nodes (three masters and three replicas) can handle three times the throughput of a single Redis instance, because each master processes one-third of the total slots. Adding more master nodes scales throughput linearly, up to the network and inter-node communication limits. For the benchmark scenario, a Redis Cluster with multiple master nodes could handle millions of operations per second, providing the write throughput needed to buffer one million requests per second. The key consideration is that each master node must have sufficient memory to hold its share of the data and sufficient CPU to process its share of the operations. Network bandwidth between cluster nodes is also a factor, as replication traffic and cluster gossip protocol messages consume bandwidth.

## Practical Applications

In the benchmark scenario, Redis was used as the primary write buffer, handling the write operations that PostgreSQL could not sustain. As the target throughput increased toward one million requests per second, the question of Redis clustering became relevant: a single Redis instance can handle hundreds of thousands of operations per second, but approaching one million may require a multi-node cluster. The hash slot architecture ensures that the load is distributed evenly across master nodes, while replicas provide the durability and failover capability needed for production reliability. The transition from a single Redis instance to a Redis Cluster is a scaling inflection point that enables the system to continue increasing throughput without hitting the limits of a single machine.
