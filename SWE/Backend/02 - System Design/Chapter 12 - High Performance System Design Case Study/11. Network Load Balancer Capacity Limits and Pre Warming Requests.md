# Network Load Balancer Capacity Limits and Pre Warming Requests

## Overview

Network load balancers sit at the edge of a distributed system, distributing incoming traffic across multiple backend servers to ensure no single server is overwhelmed. At the scale of one million requests per second, the load balancer itself can become a bottleneck if it is not properly sized and configured. Cloud providers like AWS offer managed load balancers with specific capacity limits, and exceeding these limits results in dropped connections, increased latency, or failed requests. Pre-warming is the process of notifying the cloud provider in advance about an expected traffic spike so that they can provision additional capacity before the spike arrives. Understanding load balancer capacity, when pre-warming is necessary, and how to architect around load balancer limitations is essential for building systems that can handle extreme traffic levels reliably.

## Key Concepts

### Types of Load Balancers

AWS offers several types of load balancers, each suited to different use cases. The Application Load Balancer (ALB) operates at Layer 7 (HTTP/HTTPS) and supports content-based routing, TLS termination, and WebSocket connections. The Network Load Balancer (NLB) operates at Layer 4 (TCP/UDP) and provides extremely high throughput with ultra-low latency, making it the preferred choice for high-performance applications. The Classic Load Balancer is the legacy option and is not recommended for new applications. For the benchmark scenario, the NLB is the appropriate choice because it can handle millions of connections per second and provides the raw throughput needed for one million requests per second. However, even NLBs have capacity limits that must be understood and managed.

### Load Balancer Capacity Limits

AWS load balancers are designed to scale automatically, but they have soft and hard limits. An ALB can handle approximately 100,000 connections per second per Availability Zone by default, with higher limits available on request. An NLB can handle significantly more -- up to millions of connections per second -- but the actual throughput depends on factors such as packet size, connection duration, and the number of targets. The key metric is the number of new connections per second, because establishing a TCP connection involves a three-way handshake and the allocation of resources on the load balancer. Short-lived connections (common in HTTP/1.0 and some HTTP/1.1 clients) stress the connection establishment capacity, while long-lived connections (HTTP/1.1 keep-alive, HTTP/2, WebSocket) stress the concurrent connection capacity.

### Pre-Warming Requests

Pre-warming is the process of requesting additional load balancer capacity in advance of an expected traffic spike. AWS requires customers to submit pre-warming requests through their support team, providing details about the expected traffic pattern: peak requests per second, expected traffic duration, source IP ranges, and target instance types. AWS then provisions additional capacity behind the scenes, ensuring that the load balancer can handle the spike without degradation. Pre-warming requests should be submitted at least 24-48 hours in advance, though AWS can sometimes accommodate shorter notice for premium support customers. Failing to pre-warm before a major traffic event (such as a product launch, flash sale, or major marketing campaign) can result in the load balancer scaling too slowly to handle the traffic surge, leading to dropped connections and user-facing errors.

### Load Balancer as a Bottleneck

When the load balancer reaches its capacity limit, it begins to drop connections or queue requests, increasing latency for all users. Unlike backend servers, which can be scaled horizontally by adding more instances, a single load balancer endpoint cannot be scaled beyond its maximum capacity. To handle traffic beyond a single load balancer's limits, the architecture must use multiple load balancers behind a DNS-based or anycast-based traffic distribution layer. Route 53 (AWS's DNS service) can distribute traffic across multiple load balancer endpoints using weighted routing, latency-based routing, or geolocation routing. CDN services like CloudFront can also absorb a significant portion of traffic by caching responses at edge locations, reducing the load on the origin load balancer.

### Health Checks and Target Registration

Load balancers continuously monitor the health of backend targets using health checks -- periodic requests to a designated health check endpoint. If a target fails the health check (returns a non-200 status code or does not respond within the timeout), the load balancer stops sending traffic to that target. When a new target is registered with the load balancer, it must pass the health check before traffic is routed to it. The deregistration delay controls how long the load balancer waits after deregistering a target before stopping traffic, allowing in-flight requests to complete. At one million requests per second, the health check interval and thresholds must be tuned carefully: too aggressive, and healthy targets may be incorrectly marked as unhealthy due to transient latency spikes; too lenient, and unhealthy targets may continue receiving traffic for too long.

## Practical Applications

In the benchmark scenario, the power server and power tester were on the same private network, eliminating the need for an external load balancer for the tests. However, in a real production deployment serving one million requests per second, one or more load balancers would be required to distribute traffic across multiple server instances. The AWS Network Load Balancer would be the natural choice, with pre-warming requested in advance to ensure sufficient capacity. The cost of the load balancer is relatively modest compared to the compute instances -- approximately $0.0225 per hour plus $0.008 per LCU-hour for an NLB -- but the cost of insufficient capacity (dropped connections, lost revenue) is far greater. The lesson is that load balancer capacity must be planned for explicitly, not discovered during a traffic spike.
