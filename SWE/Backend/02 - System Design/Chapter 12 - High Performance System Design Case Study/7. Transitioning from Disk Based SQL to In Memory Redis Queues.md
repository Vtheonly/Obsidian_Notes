# Transitioning from Disk Based SQL to In Memory Redis Queues

## Overview

When a relational database cannot keep up with write throughput, the architectural response is often to move write-heavy operations from disk-based SQL to in-memory data stores like Redis. Redis stores all data in RAM, eliminating disk I/O latency and enabling throughput of millions of operations per second on a single instance. By using Redis as a write-through queue or buffer, applications can accept writes at high speed and asynchronously persist them to the database in batches, decoupling the application's write throughput from the database's write capacity. This transition is one of the most impactful architectural changes available for high-throughput systems, but it introduces trade-offs around durability, consistency, and operational complexity that must be carefully managed.

## Key Concepts

### Why Redis Is Orders of Magnitude Faster

Redis achieves its performance by storing all data in RAM, where memory access times are measured in nanoseconds compared to the milliseconds required for disk I/O. A Redis instance running on modern hardware can handle 100,000 to 1,000,000 simple operations (GET, SET, LPUSH, etc.) per second on a single thread, because each operation involves only memory access and no disk writes during normal operation. Even with persistence enabled (AOF or RDB), Redis batches disk writes in the background, ensuring that the foreground operation completes in microseconds. This fundamental difference in storage medium -- RAM versus disk -- translates to a 1,000x or greater improvement in operation throughput for write-heavy workloads.

### Redis as a Write Buffer

The most common pattern for using Redis in a high-throughput system is as a write buffer between the application and the database. Instead of writing directly to PostgreSQL on every request, the application pushes data to a Redis list or stream. A background worker process reads from the Redis queue and batches writes to the database, inserting multiple records in a single transaction. This architecture decouples the application's write rate from the database's write rate: the application can handle millions of writes per second (limited only by Redis and network throughput), while the database processes writes at its own pace. The trade-off is that data in Redis that has not yet been flushed to the database is at risk of loss if the Redis instance fails before the background worker processes it.

### Redis Data Structures for Queuing

Redis provides several data structures suitable for queuing. Lists (LPUSH/BRPOP) offer simple FIFO queues with blocking operations that allow workers to wait for new items efficiently. Sorted Sets (ZADD/ZRANGEBYSCORE) enable priority queues where items are processed in order of their score. Streams (XADD/XREAD) provide a log-based queuing system with consumer groups, enabling multiple workers to process items from the same stream without duplication. For the benchmark scenario, Redis lists were the simplest and most performant option, providing atomic push and pop operations with minimal overhead. The choice of data structure depends on the specific requirements: if you need ordering by priority, use sorted sets; if you need message acknowledgments and consumer groups, use streams; if you need maximum simplicity and speed, use lists.

### Persistence and Durability Trade-offs

Redis offers two persistence mechanisms: RDB (point-in-time snapshots) and AOF (append-only file that logs every write operation). With no persistence, Redis is purely in-memory, and a restart loses all data. With RDB alone, you lose data written since the last snapshot. With AOF alone, you lose at most one second of data (with the default every-second fsync policy). With both RDB and AOF, you achieve the highest durability at the cost of some performance. For the write-buffer pattern, the durability requirement depends on the use case: if the data is ephemeral (like session state), no persistence is acceptable; if the data must not be lost, AOF with every-second fsync is a reasonable compromise. The key insight is that durability is not binary -- it is a spectrum, and the right level depends on the business requirements.

### Memory Management and Eviction

Because Redis stores all data in RAM, memory is a finite resource that must be managed carefully. Redis supports eviction policies that determine what happens when memory is full: noeviction (reject writes), allkeys-lru (evict least recently used keys), volatile-lru (evict least recently used keys with expiration), and others. For queue workloads, the data is typically transient -- items are pushed, processed, and removed -- so memory usage is determined by the queue depth, which depends on the rate of incoming writes versus the rate of processing. If the processing rate falls behind the write rate, the queue grows, consuming more memory. Monitoring queue depth and memory usage is essential to prevent out-of-memory conditions that would cause Redis to reject writes or evict data unexpectedly.

## Practical Applications

In the benchmark scenario, moving write operations from PostgreSQL to Redis was the critical architectural change that enabled the system to approach one million requests per second. The PostgreSQL database with 3,000 IOPS could handle only a few thousand writes per second, creating a severe bottleneck. By pushing data to Redis instead, the application could handle hundreds of thousands of write operations per second, limited only by the Redis instance's memory and CPU capacity. The background worker could then batch-flush data to PostgreSQL at a rate the database could sustain. This decoupling of write rate from database capacity is the standard pattern for high-throughput systems, used by companies like Uber, Netflix, and Amazon to handle massive write loads without overwhelming their databases.
