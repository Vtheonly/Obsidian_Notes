# CPU Cache Hierarchy: Level 1, Level 2, and Level 3

## Overview

The CPU cache hierarchy is one of the most important architectural features for system performance. Because RAM access takes roughly 100-200 CPU clock cycles, directly reading from RAM for every memory access would leave the CPU idle most of the time. The cache hierarchy bridges this speed gap by providing progressively faster but smaller storage layers between the CPU and RAM. Understanding the cache hierarchy is essential for writing performant code and designing systems that maximize hardware utilization. At scale, cache efficiency can be the difference between a system that handles millions of requests per second and one that collapses under far less load.

## Key Concepts

### Cache Levels and Sizes

Modern CPUs have three levels of cache, each with different size and speed characteristics. L1 cache is the smallest and fastest, typically 32-64 KB per core, with access latency of about 1-4 CPU cycles (roughly 1 nanosecond). L2 cache is larger, typically 256 KB to 1 MB per core, with access latency of about 4-10 cycles. L3 cache is the largest, typically 4-64 MB shared across all cores, with access latency of about 10-50 cycles. If data is not found in any cache level, the CPU must fetch it from RAM, which takes 50-100 nanoseconds (100-200 cycles). Each level captures a progressively larger subset of the working data, and the hit rate at each level determines overall memory performance.

### Cache Lines

Data is transferred between the cache and RAM in units called cache lines (or cache blocks), which are typically 64 bytes on modern processors. When the CPU reads a single byte from memory, it actually fetches the entire 64-byte cache line containing that byte. This design exploits spatial locality: if a program accesses one byte, it is likely to access nearby bytes soon. Understanding cache lines is critical for performance because data structures that fit within a single cache line can be accessed efficiently, while data structures that span multiple cache lines incur additional memory fetches. False sharing occurs when two threads modify different variables that happen to reside on the same cache line, causing the cache line to bounce between cores and severely degrading performance.

### Cache Hits and Misses

A cache hit occurs when the requested data is found in the cache, and a cache miss occurs when it is not. The cache hit rate is the percentage of memory accesses that result in a cache hit, and it directly determines the effective memory access time. For L1 cache, typical hit rates are 90-95% for well-optimized code. For L2, the hit rate for accesses that miss L1 is about 80-90%. L3 captures most remaining misses, with overall miss rates to RAM typically below 5% for workloads with good locality. Each cache miss that goes all the way to RAM costs roughly 100 nanoseconds, which at 3 GHz equates to about 300 wasted CPU cycles. In a tight loop processing millions of operations, even a small miss rate can dominate execution time.

### Spatial and Temporal Locality

Cache effectiveness depends on two principles. Temporal locality means that data accessed recently is likely to be accessed again soon, so keeping recently used data in the cache captures repeated accesses. Spatial locality means that data near recently accessed data is likely to be accessed soon, so fetching entire cache lines captures nearby accesses. Code that exhibits good locality performs well with the cache hierarchy, while code that accesses memory in random, scattered patterns (pointer chasing through linked lists, hash table probing) suffers frequent cache misses. Array-based data structures generally have better cache performance than pointer-based structures because arrays store data contiguously in memory.

### Inclusive vs Exclusive Caches

Some cache hierarchies use inclusive caches, where L3 contains a superset of all data in L1 and L2. This simplifies cache coherence because checking L3 is sufficient to determine whether another core might have a cached copy of a line. Other designs use exclusive caches, where L3 contains only data that is not in L1 or L2, effectively increasing the total cache capacity. Intel processors typically use inclusive L3 caches, while AMD processors have moved toward exclusive designs. The choice between inclusive and exclusive affects both performance and the behavior of cache coherence protocols in multi-core systems.

## Practical Applications

When designing a high-performance system, understanding the cache hierarchy helps with several critical decisions. Database engines organize index structures (like B-trees) to maximize cache line utilization, fitting frequently accessed node data within a single 64-byte cache line. In-memory caches like Redis are effective precisely because they keep hot data in RAM, which is many times faster than disk but still benefits from CPU cache optimization. When processing large datasets, iterating over arrays in sequential order is dramatically faster than random access because sequential iteration exploits spatial locality and allows the hardware prefetcher to load cache lines before they are needed. At the scale of handling hundreds of thousands of requests per second, even small improvements in cache hit rates can translate to significant performance gains and cost savings.
