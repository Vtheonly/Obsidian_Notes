# Rate Limiting Strategies Per IP Per User and Global

## Overview
Rate limiting is a defensive mechanism that controls the rate at which clients can make requests to a service, protecting the system from abuse, ensuring fair resource allocation, and maintaining availability under load. Rate limiting can be applied at multiple granularities: per IP address, per authenticated user, per API key, or globally across all clients. Each strategy addresses different threat models and has distinct tradeoffs. Without rate limiting, a single misbehaving client -- whether malicious (a DDoS attacker) or accidental (a buggy client in a retry loop) -- can consume disproportionate resources and degrade the experience for all other users. Understanding and implementing effective rate limiting is a fundamental responsibility of system designers building production-ready services.

## Key Concepts

### Per-IP Rate Limiting
Per-IP rate limiting restricts the number of requests that can originate from a single IP address within a given time window. This is the simplest rate limiting strategy and is effective against naive brute-force attacks and basic DDoS attempts. However, per-IP limiting has significant limitations. Multiple legitimate users behind a NAT or corporate proxy share the same IP address, so aggressive per-IP limits can unfairly penalize entire organizations. Conversely, sophisticated attackers can distribute requests across many IPs using botnets or proxy networks, rendering per-IP limits ineffective. Per-IP rate limiting is best used as a first layer of defense, combined with more granular strategies for authenticated traffic. Implementation typically uses the client's IP address (from the X-Forwarded-For header or the direct socket address) as the key in a rate limiting data structure.

### Per-User Rate Limiting
Per-user rate limiting restricts the number of requests an authenticated user can make, regardless of how many IP addresses they use. This strategy is more precise than per-IP limiting because it operates at the identity level rather than the network level. It enforces fair usage policies (for example, "each user can make 1000 API calls per hour") and prevents a single user from monopolizing resources. Per-user rate limiting requires authentication to be completed before the limit can be enforced, which means unauthenticated endpoints still need per-IP or global protection. The user identifier (user ID, API key, or token subject) serves as the rate limiting key. Tiered limits are common: free-tier users might be allowed 100 requests per hour, while premium users get 10,000.

### Global Rate Limiting
Global rate limiting sets an aggregate cap on the total number of requests the system will accept, regardless of the source. This protects the system's overall capacity and ensures that the service remains responsive even under extreme load. Global limits are typically set based on the system's provisioned capacity, performance benchmarks, and desired latency targets. If the global limit is reached, all subsequent requests are rejected with a 503 Service Unavailable response until capacity frees up. Global rate limiting is a last-resort protection mechanism and should be combined with more granular strategies to ensure fairness. Without per-user or per-IP limits, a few aggressive clients could consume the entire global budget, blocking all other users.

### Rate Limiting Algorithms
Several algorithms implement rate limiting with different characteristics. Fixed window counting divides time into fixed intervals (for example, one-hour windows) and counts requests within each window. It is simple but allows burst traffic at window boundaries (a client can send 2x the limit by making requests at the end of one window and the beginning of the next). Sliding window logging stores timestamps of individual requests in a sorted set and counts requests within a sliding time window, providing smooth rate limiting at the cost of higher memory usage. Sliding window counter combines fixed window efficiency with sliding window smoothness by weighting the current and previous window counts. Token bucket allows bursts up to the bucket capacity while maintaining a steady refill rate, making it well-suited for APIs that experience legitimate bursts. Leaky bucket smooths request processing to a steady rate, converting variable input into constant output.

### Rate Limit Response Patterns
When a rate limit is exceeded, the server should return a 429 Too Many Requests status code along with headers that inform the client about the limit state. Standard headers include "Retry-After" (seconds until the client can retry), "X-RateLimit-Limit" (the maximum allowed requests), "X-RateLimit-Remaining" (remaining requests in the current window), and "X-RateLimit-Reset" (seconds until the window resets). These headers enable well-behaved clients to implement exponential backoff and retry logic. For internal services, the server may queue excess requests rather than rejecting them, though this requires careful capacity management. In distributed systems, rate limit headers must reflect the global state, not just the local server's count, which requires coordination through a shared data store.

## Practical Applications
Public API providers like GitHub, Stripe, and Twitter use tiered per-user rate limits to manage API usage and enforce fair use policies. Cloud providers implement per-IP rate limiting at the CDN and load balancer layers to mitigate DDoS attacks. E-commerce platforms use rate limiting on login and checkout endpoints to prevent credential stuffing and inventory hoarding. Web application firewalls apply rate limiting as a rule type to protect against brute-force attacks. API gateways like Kong, AWS API Gateway, and Cloudflare implement configurable rate limiting at multiple granularities as a first-class feature.
