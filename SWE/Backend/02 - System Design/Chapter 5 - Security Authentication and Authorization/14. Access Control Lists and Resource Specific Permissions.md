# Access Control Lists and Resource Specific Permissions

## Overview
Access Control Lists (ACLs) are a foundational authorization mechanism that associates a list of permissions with a specific resource, specifying which subjects (users, groups, or roles) are allowed or denied particular operations on that resource. ACLs provide resource-centric access control: rather than asking "What can this user do?" (as in role-based models), ACLs answer "Who can do what with this resource?" This resource-centric perspective makes ACLs particularly effective for systems where access must be controlled at the individual resource level -- for example, a document that only specific users should be able to edit, or a database row that only certain teams can access. Understanding ACLs is essential because they underpin file system permissions, database access controls, and many application-level authorization systems.

## Key Concepts

### ACL Structure
An ACL is a data structure attached to a resource that contains a list of Access Control Entries (ACEs). Each ACE typically contains a subject identifier (a user, group, or role), a set of permissions (such as read, write, execute, delete), and an effect (allow or deny). When a subject requests access to a resource, the system evaluates the ACL by iterating through the entries and determining whether the requested permission is granted or denied. The order of evaluation matters: some systems use a "first match" approach where the first matching ACE determines the outcome, while others collect all matching entries and apply a deny-overrides rule. The specific semantics depend on the implementation, but the principle is the same: the ACL defines the access policy for its associated resource.

### Discretionary vs Mandatory Access Control
ACLs are the primary mechanism for Discretionary Access Control (DAC), where the resource owner has the discretion to set permissions. In DAC, the owner of a file or resource can grant or revoke access to other users, which provides flexibility but can lead to inconsistent security policies. In contrast, Mandatory Access Control (MAC) enforces a system-wide policy that users cannot override -- for example, a government system where documents with a "Secret" classification can only be accessed by users with a "Secret" clearance, regardless of the document owner's preferences. ACLs are typically used for DAC, while MAC requires a different mechanism such as security labels and clearance levels evaluated by the operating system kernel.

### ACL Evaluation Algorithms
The evaluation algorithm determines how ACL entries are processed to produce an access decision. Common algorithms include: first applicable (the first matching ACE determines the outcome, with an implicit deny at the end), deny-overrides (if any matching ACE denies access, the request is denied regardless of other allowing entries), allow-overrides (if any matching ACE allows access, the request is allowed regardless of other denying entries), and majority-based (the outcome is determined by the majority of matching entries). The deny-overrides algorithm is the most common because it is conservative and prevents accidental access. The choice of evaluation algorithm has significant security implications and must be clearly documented and consistently implemented.

### ACL Scalability and Performance
ACLs face scalability challenges when the number of resources or subjects grows large. If every document in a system has its own ACL, the total number of ACEs can become enormous, making management and evaluation slow. To mitigate this, systems use techniques such as group-based ACEs (assigning permissions to groups rather than individual users), ACL inheritance (where child resources inherit ACLs from parent resources with optional override), and ACL templates (standard ACL patterns that can be applied to categories of resources). Caching ACL evaluation results and using efficient data structures (such as bit vectors or bloom filters for membership checks) can improve performance. Despite these optimizations, ACL-based systems may struggle at the scale of millions of resources with per-resource permissions.

### ACLs in Distributed Systems
In distributed systems, ACL management becomes more complex because resources and authorization decisions may span multiple services. Centralized ACL stores provide a single source of truth but become a bottleneck and single point of failure. Replicated ACL stores provide availability and read performance but introduce consistency challenges. Some systems embed ACLs directly in the resource metadata (for example, in a database column or document field), which keeps authorization data close to the resource but makes cross-cutting policy changes difficult. In cloud environments, ACLs are often managed through infrastructure-as-code, where ACL definitions are version-controlled and deployed alongside application code. Consistent enforcement across all services is critical to prevent authorization bypass.

## Practical Applications
File systems like ext4 and NTFS use ACLs to control read, write, and execute permissions on files and directories. AWS S3 bucket policies and object ACLs control access to stored objects at both the bucket and individual object level. Database systems like PostgreSQL implement row-level security policies that function as ACLs on individual database rows. Content management systems like SharePoint and Confluence use ACLs to control document access at the folder and document level. Collaboration tools like Google Docs use ACLs to manage sharing permissions, allowing owners to grant view or edit access to specific users or groups.
