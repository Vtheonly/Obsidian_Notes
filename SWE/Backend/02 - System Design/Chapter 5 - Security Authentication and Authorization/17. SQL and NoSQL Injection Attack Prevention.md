# SQL and NoSQL Injection Attack Prevention

## Overview
SQL and NoSQL injection are attack techniques in which an attacker manipulates input data to execute unauthorized commands against a database. These attacks exploit insufficient input validation and the improper construction of database queries by concatenating user-supplied data directly into query strings. SQL injection has been a top security vulnerability for decades and remains prevalent in legacy systems. NoSQL injection is a newer variant that targets document-oriented, key-value, and graph databases, exploiting their different query syntax and evaluation models. Both types of injection can lead to data breaches, data manipulation, authentication bypass, and in severe cases, complete server compromise. Prevention requires a defense-in-depth approach combining parameterized queries, input validation, least-privilege database access, and ongoing security testing.

## Key Concepts

### SQL Injection Mechanics
SQL injection occurs when user input is concatenated directly into a SQL query string. For example, if a login query is constructed as "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'", an attacker can submit the username "admin' --" to produce the query "SELECT * FROM users WHERE username = 'admin' --' AND password = ''", which bypasses the password check entirely. SQL injection can be classified into several categories. In-band injection (the most common) uses the same channel for attack and data exfiltration. Blind injection infers results from the application's behavior (such as response time or error messages) without directly seeing query output. Out-of-band injection uses a different channel (like DNS or HTTP) to exfiltrate data. Union-based injection appends additional SELECT statements to extract data from other tables.

### NoSQL Injection Mechanics
NoSQL injection targets databases that use different query paradigms than traditional SQL. In MongoDB, for example, queries are typically constructed as JavaScript objects. If user input is directly embedded into a query object, an attacker can inject MongoDB query operators. For instance, if a login query uses {"username": username, "password": password}, an attacker can submit
```sql
{"$gt": ""} as the password value, producing {"username": "admin", "password": {"$gt": ""}}
```
  which matches any document where the password is greater than an empty string. In CouchDB and other databases that evaluate JavaScript, attackers can inject arbitrary JavaScript code for server-side execution. Redis and other key-value stores can be exploited through command injection if user input is concatenated into command strings. The diversity of NoSQL query languages means that each type of NoSQL database has its own injection vectors.

### Parameterized Queries and Prepared Statements
Parameterized queries (also called prepared statements) are the primary defense against SQL injection. Instead of concatenating user input into the query string, parameterized queries use placeholders that the database engine fills with the provided values. Because the database treats the placeholders as data rather than executable code, there is no possibility of injection. For example, using a parameterized query in Python with psycopg2: "cursor.execute('SELECT * FROM users WHERE username = %s AND password = %s', (username, password))". All major database drivers and ORM frameworks support parameterized queries. For NoSQL databases, the equivalent defense is using the driver's query builder API rather than constructing query objects from raw user input. In MongoDB, this means using the driver's filter construction methods rather than parsing JSON input directly into query objects.

### Input Validation and Sanitization
While parameterized queries are the primary defense, input validation provides an additional layer of protection. Validation ensures that input conforms to expected formats and ranges before it is processed. Whitelist validation (accepting only known-good patterns) is more secure than blacklist validation (rejecting known-bad patterns). For example, validating that a username contains only alphanumeric characters, or that an age field is a positive integer. Sanitization (also called encoding or escaping) transforms input to neutralize dangerous characters, but it should not be relied upon as the sole defense because it is error-prone and context-dependent. The correct approach is to validate input for format and range, use parameterized queries for database access, and apply output encoding when rendering data in responses.

### Defense in Depth for Injection Prevention
A comprehensive injection prevention strategy includes multiple layers. First, use parameterized queries or ORM frameworks for all database access -- never concatenate user input into queries. Second, validate and sanitize all input at the application boundary. Third, apply the principle of least privilege to database accounts: application accounts should only have the permissions necessary for their operations (for example, SELECT and INSERT but not DROP or GRANT). Fourth, use web application firewalls to detect and block injection patterns. Fifth, implement error handling that does not expose database error messages to users, as these messages can help attackers refine their attacks. Sixth, conduct regular security testing including automated vulnerability scanning, static code analysis, and manual penetration testing. Finally, keep database software and drivers updated to benefit from security patches.

## Practical Applications
Every web application that interacts with a database must implement injection prevention. E-commerce platforms must protect product search and login endpoints from SQL injection. Healthcare applications handling patient data must prevent injection to comply with HIPAA requirements. Financial applications are particularly sensitive because SQL injection can be used to manipulate transaction data. In the NoSQL space, MongoDB-based applications must validate input to prevent operator injection, while Redis-based applications must prevent command injection. ORM frameworks like Django ORM, Hibernate, and SQLAlchemy provide built-in parameterized queries, but developers must still avoid raw query methods that bypass this protection.
