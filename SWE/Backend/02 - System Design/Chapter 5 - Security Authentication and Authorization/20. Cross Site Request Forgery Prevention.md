# Cross Site Request Forgery Prevention

## Overview
Cross-Site Request Forgery (CSRF) is an attack that tricks an authenticated user's browser into sending an unintended request to a web application where the user is already authenticated. Because the browser automatically includes session cookies with each request to a domain, an attacker can craft a malicious page that causes the victim's browser to submit a state-changing request (such as a fund transfer, password change, or email update) without the user's knowledge or consent. The application processes the request as legitimate because it comes with the victim's valid session cookie. CSRF exploits the trust that a web application places in the user's browser, and it is distinct from XSS, which exploits the trust a user places in a web application. Preventing CSRF requires specific countermeasures because standard authentication mechanisms like cookies do not protect against it.

## Key Concepts

### How CSRF Works
A CSRF attack exploits the way browsers handle cookies. When a user is authenticated to a web application, the browser stores a session cookie for that domain. If the user then visits a malicious website, the attacker can include an HTML form or a JavaScript fetch that submits a request to the vulnerable application. The browser automatically includes the session cookie with this request, so the application processes it as if the user intentionally submitted it. For example, an attacker could create a page with a hidden form that POSTs to "https://bank.com/transfer" with parameters "to=attacker&amount=10000". When the victim visits the attacker's page, their browser submits the transfer request with their valid session cookie, and the bank processes the transfer.

### Synchronizer Token Pattern
The synchronizer token pattern (also called CSRF tokens) is the most common defense against CSRF. The server generates a cryptographically random token for each session (or each form) and embeds it in the HTML form as a hidden field. When the form is submitted, the server verifies that the token in the request matches the token stored in the session. Because the attacker's malicious page cannot read the token (due to the Same-Origin Policy), they cannot include it in their forged request, and the server rejects the request. CSRF tokens must be unique per session, generated using a CSPRNG, and validated on every state-changing request (POST, PUT, DELETE). GET requests should never perform state changes, because CSRF tokens cannot protect against GET-based attacks through image tags or link prefetching.

### SameSite Cookie Attribute
The SameSite cookie attribute is a browser-level defense against CSRF that has become the primary protection mechanism in modern applications. When a cookie is set with SameSite=Strict, the browser only includes the cookie in requests that originate from the same site that set the cookie. This provides strong CSRF protection but can degrade the user experience -- for example, following a link from an email to the application will not include the session cookie, so the user appears logged out. SameSite=Lax (the browser default in most modern browsers) includes the cookie in top-level navigation (following a link) but not in cross-origin POST requests or subresource loads. This provides a good balance between security and usability. SameSite=None allows the cookie to be sent with all cross-site requests but requires the Secure flag (HTTPS), making it suitable only for third-party integrations that genuinely need cross-site cookie access.

### Double Submit Cookie Pattern
The double submit cookie pattern is an alternative to the synchronizer token pattern that does not require server-side state. The server sets a CSRF token as a cookie, and the client includes the same token in a custom header (such as X-CSRF-Token) with each request. The server verifies that the cookie value matches the header value. This works because an attacker cannot read the cookie value from a cross-origin page (due to the Same-Origin Policy) and therefore cannot include it in the custom header. This pattern is commonly used in SPAs where the frontend reads the CSRF cookie and adds it to the request header via JavaScript. It requires the cookie to be set with the HttpOnly flag set to false (so JavaScript can read it) or the token to be delivered through another channel.

### Custom Request Headers
For AJAX-based applications, requiring a custom header (such as X-Requested-With: XMLHttpRequest) on all state-changing requests provides CSRF protection because browsers enforce the Same-Origin Policy for custom headers. A cross-origin request cannot include custom headers unless the server explicitly allows them through CORS preflight responses. Since the attacker's forged request cannot include the custom header, the server can reject requests that lack it. This approach is simple and effective for API-based applications but does not protect against CSRF through HTML form submissions, which cannot include custom headers. It should be combined with SameSite cookie attributes for comprehensive protection.

## Practical Applications
All major web frameworks provide built-in CSRF protection. Django includes CSRF middleware that implements the synchronizer token pattern. Rails uses an authenticity token embedded in forms. Angular and React applications commonly use the double submit cookie pattern with X-CSRF-Token headers. Banking applications implement multiple CSRF defenses (SameSite cookies, CSRF tokens, and re-authentication for sensitive operations) as part of regulatory compliance. OAuth 2.0 uses the "state" parameter to prevent CSRF on the authorization callback, ensuring that the callback corresponds to the original authorization request. APIs that use bearer token authentication (in the Authorization header) are inherently immune to CSRF because browsers do not automatically include custom headers in cross-origin requests.
