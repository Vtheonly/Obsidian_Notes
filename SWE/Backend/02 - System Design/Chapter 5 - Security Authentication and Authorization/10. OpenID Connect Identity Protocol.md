# OpenID Connect Identity Protocol

## Overview
OpenID Connect (OIDC) is an identity authentication protocol built on top of the OAuth 2.0 authorization framework. While OAuth 2.0 provides delegated authorization, OIDC adds a standardized identity layer that allows clients to verify the identity of the end user and obtain basic profile information. OIDC uses JSON Web Tokens (JWTs) to convey identity claims and is designed to be REST-friendly, lightweight, and suitable for web, mobile, and native applications. OIDC has largely replaced SAML for new implementations due to its simpler JSON-based format, better support for mobile and SPA use cases, and alignment with modern API-driven architectures.

## Key Concepts

### OIDC Architecture: Provider, Relying Party, and End User
OIDC defines three primary entities. The OpenID Provider (OP) is the identity provider that authenticates the end user and issues identity tokens. The Relying Party (RP) is the application that relies on the OP for authentication. The End User is the person being authenticated. The OP exposes several standard endpoints: the Authorization Endpoint (where the user authenticates), the Token Endpoint (where the RP exchanges an authorization code for tokens), and the UserInfo Endpoint (where the RP can retrieve additional profile information). The OP also publishes its configuration at a well-known discovery URL (.well-known/openid-configuration), which allows RPs to dynamically discover the OP's capabilities and endpoints.

### ID Tokens
The ID Token is the core innovation of OIDC. It is a JWT that contains claims about the authentication event and the authenticated user. Standard claims include "iss" (issuer identifier for the OP), "sub" (unique identifier for the user at the OP), "aud" (audience, the RP's client ID), "exp" (expiration time), "iat" (issued at time), "auth_time" (time when the user was authenticated), and "nonce" (a value that ties the ID token to a specific authentication request to prevent replay attacks). Additional claims such as "name", "email", and "picture" can be included or retrieved from the UserInfo endpoint. The ID token is signed by the OP and must be validated by the RP before the identity information is trusted.

### OIDC Flows
OIDC defines several flows for different application types. The Authorization Code Flow is the most secure and recommended flow for server-side applications. The RP redirects the user to the OP, the user authenticates, and the OP returns an authorization code to the RP. The RP exchanges this code for tokens at the Token Endpoint using its client secret. The Authorization Code Flow with PKCE (Proof Key for Code Exchange) extends this flow for public clients like SPAs and mobile apps by using a code verifier and code challenge instead of a client secret. The Implicit Flow (now deprecated by OAuth 2.1) returned tokens directly in the redirect URL and is no longer recommended due to security concerns. The Hybrid Flow returns some tokens in the redirect and others from the token endpoint, providing flexibility for specific use cases.

### Scopes and Claims
OIDC defines standard scopes that control which claims are returned. The "openid" scope is mandatory and indicates that the request is an OIDC authentication request. The "profile" scope requests access to default profile claims like name, family_name, and given_name. The "email" scope requests the email and email_verified claims. The "address" scope requests the user's postal address. The "phone" scope requests the phone_number and phone_number_verified claims. RPs can also request specific individual claims using the "claims" request parameter. This granularity allows RPs to request only the identity information they need, adhering to the principle of least privilege and improving user privacy.

### OIDC Discovery and Dynamic Registration
OIDC provides discovery and dynamic registration mechanisms that simplify integration. The OP publishes its configuration at the discovery endpoint, which includes the supported scopes, claims, response types, token endpoint authentication methods, and signing algorithms. RPs can fetch this configuration at runtime to automatically adapt to the OP's capabilities. Dynamic Client Registration allows RPs to register with the OP programmatically using a registration endpoint, eliminating the need for manual configuration. These features make OIDC highly interoperable and reduce the integration effort for both OPs and RPs. However, dynamic registration should be used with caution in production environments because it can be abused to register malicious clients.

## Practical Applications
OIDC is used by all major consumer identity providers. "Sign in with Google", "Sign in with Apple", and "Sign in with Microsoft" all implement OIDC for third-party authentication. Enterprise identity platforms like Okta, Auth0, and Azure AD expose OIDC endpoints for employee and partner SSO. In Kubernetes, OIDC is used as the authentication mechanism for the API server, allowing users to authenticate with their corporate identity provider. Cloud platforms use OIDC for federated access to resources, enabling users to assume roles based on their OIDC identity. Mobile applications use OIDC with AppAuth libraries to implement secure authentication flows.
