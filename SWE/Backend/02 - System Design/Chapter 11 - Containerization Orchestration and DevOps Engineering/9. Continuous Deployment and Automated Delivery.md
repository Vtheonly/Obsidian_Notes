# Continuous Deployment and Automated Delivery

## Overview

Continuous Deployment (CD) extends the principles of Continuous Integration by automatically deploying every change that passes the CI pipeline to production, without manual intervention. While Continuous Delivery -- a closely related practice -- ensures that code is always in a deployable state but requires a human decision to deploy, Continuous Deployment removes the human gate entirely: any commit that passes all automated checks is released to users. This approach demands a high degree of confidence in automated testing, monitoring, and rollback mechanisms, but it offers significant benefits in terms of deployment frequency, feedback speed, and reduced risk per deployment. The key insight is that small, frequent deployments are less risky than large, infrequent ones, because each deployment carries a smaller delta of change.

## Key Concepts

### Continuous Delivery versus Continuous Deployment

The distinction between Continuous Delivery and Continuous Deployment is subtle but important. Continuous Delivery means that every change is automatically built, tested, and staged for release, but a human makes the final decision to deploy to production. This provides the safety net of automation while retaining a human checkpoint. Continuous Deployment goes further: every change that passes the pipeline is automatically deployed to production. The choice between the two depends on the organization's risk tolerance, regulatory requirements, and confidence in its automated testing. Many organizations start with Continuous Delivery and graduate to Continuous Deployment as their testing and monitoring mature.

### Deployment Strategies

Effective CD systems support multiple deployment strategies to minimize risk. A rolling update gradually replaces old instances with new ones, ensuring that some instances are always available. A blue-green deployment maintains two identical production environments, routing traffic to the "blue" environment while deploying to the "green" environment, then switching traffic once the deployment is validated. A canary deployment routes a small percentage of traffic to the new version first, monitoring for errors before gradually increasing the traffic share. Feature flags provide another mechanism for controlled rollouts, enabling new functionality to be deployed in a disabled state and activated gradually for specific users or regions.

### Automated Rollback

A critical component of any CD system is the ability to automatically roll back a deployment when problems are detected. If monitoring systems detect an increase in error rates, latency spikes, or other anomalies after a deployment, the CD pipeline can automatically revert to the previous known-good version. This automated rollback capability is what makes Continuous Deployment feasible at scale -- without it, a bad deployment could affect users for the duration of time it takes a human to notice the problem and manually intervene. Effective rollback mechanisms require that deployments are atomic and reversible, and that the previous version's artifacts are always available.

### Environment Progression

A typical CD pipeline progresses through multiple environments: development, staging (or pre-production), and production. Each environment should be as similar to production as possible to catch environment-specific issues. The staging environment is used for final validation, including integration tests, performance tests, and user acceptance tests. Containerization plays a key role here -- the same container image that was tested in staging is deployed to production, ensuring that there are no environment-related differences. Infrastructure as Code ensures that the infrastructure in each environment is provisioned identically, reducing the risk of configuration drift.

### Observability and Deployment Confidence

The confidence to deploy automatically comes from robust observability. CD systems rely on real-time monitoring of application health metrics -- error rates, request latencies, throughput, CPU and memory utilization -- to determine whether a deployment was successful. Distributed tracing, centralized logging, and alerting systems provide the visibility needed to detect problems quickly. Without this observability layer, automated deployment is reckless rather than courageous. The investment in monitoring and alerting is a prerequisite for Continuous Deployment, not an optional enhancement.

## Practical Applications

The CI/CD pipeline described in the DevOps in 100 Seconds source material demonstrates the complete flow from code commit to production deployment. A developer pushes code to the master branch on GitHub, which triggers a GitHub Actions workflow. The workflow runs on a Linux container, checking out the code, setting up Node.js, installing dependencies, running tests, and then executing build and deploy commands. If all steps succeed, the new code is automatically delivered to users. If any step fails, the deployment is blocked and the team is notified. This automation provides two main benefits: it increases velocity by eliminating manual deployment steps, and it detects small problems early before they can grow into major disasters, resulting in higher code quality.
