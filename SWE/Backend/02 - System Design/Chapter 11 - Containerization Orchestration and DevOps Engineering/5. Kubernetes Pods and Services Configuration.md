# Kubernetes Pods and Services Configuration

## Overview

Pods and Services are the two most fundamental abstractions in Kubernetes, working together to provide both the execution unit and the stable networking interface for containerized applications. A Pod is the smallest deployable unit in Kubernetes, encapsulating one or more containers that share the same network namespace and storage volumes. A Service provides a stable IP address and DNS name that abstracts over the ephemeral nature of Pods, ensuring that clients can reliably reach the application regardless of which Pods are running at any given time. Understanding how to configure Pods and Services correctly is essential for building resilient, scalable applications on Kubernetes.

## Key Concepts

### Pod Configuration

A Pod is defined declaratively in a YAML manifest that specifies the containers to run, the resources they require, environment variables, volume mounts, and probes for health checking. Each container within a Pod specifies its image, port mappings, and resource requests and limits. Resource requests tell the scheduler the minimum CPU and memory the container needs, while limits cap the maximum resources it can consume. Liveness probes determine whether a container is still running and restart it if not, while readiness probes determine whether a container is ready to serve traffic. Properly configured probes are critical for Kubernetes to manage the Pod lifecycle correctly -- without them, Kubernetes cannot distinguish between a container that is running but not ready and one that has failed.

### Multi-Container Pods

While the most common pattern is one container per Pod, multi-container Pods are used for tightly coupled sidecar patterns. Common sidecar patterns include logging agents that ship container logs to a central system, proxy containers that handle authentication or TLS termination, and adapter containers that transform output from the main container. All containers in a Pod share the same network namespace (they can communicate via localhost) and can share volumes for data exchange. The sidecar pattern keeps the main application container focused on its primary responsibility while delegating cross-cutting concerns to the sidecar.

### Label Selectors and Service Routing

Kubernetes uses labels -- arbitrary key-value pairs attached to Pods and other resources -- and label selectors to determine which Pods a Service should route traffic to. When defining a Service, you specify a selector that matches the labels on your Pods. For example, a Service might select all Pods with the label `app: cheddar` to route traffic to the cheddar cheese application. This decoupling between the Service definition and the Pod definition means that you can update, scale, or replace Pods without modifying the Service configuration. The Service automatically updates its endpoints as Pods are created or destroyed.

### Service Types

Kubernetes supports several Service types, each suited to different networking scenarios. A ClusterIP Service (the default) exposes the service on a cluster-internal IP, making it reachable only from within the cluster. A NodePort Service exposes the service on each node's IP at a static port, allowing external traffic to reach the service. A LoadBalancer Service provisions an external load balancer from the cloud provider and routes traffic to the Service. A Headless Service, created by setting clusterIP to None, returns the IP addresses of the Pods directly rather than a single virtual IP, which is useful for StatefulSets and direct Pod-to-Pod communication.

### ReplicaSets and Deployments

While Pods are the execution unit, they are rarely created directly in production. Instead, Deployments manage ReplicaSets, which in turn manage Pods. A Deployment defines the desired number of Pod replicas and the Pod template to use. If a Pod fails, the ReplicaSet automatically creates a replacement. Deployments also manage rollouts -- when you update the Pod template (for example, to use a new container image), the Deployment creates a new ReplicaSet with the updated specification and gradually shifts traffic from the old Pods to the new ones, following a rolling update strategy. This ensures zero-downtime deployments when configured correctly.

## Practical Applications

In the Traefik and Kubernetes demo at Reevoo, Pods were deployed for multiple "cheese" applications (cheddar, stilton, wensleydale), each with its own Service defined using label selectors. The Services provided stable endpoints that Traefik could route to, even as Pods were created or destroyed. This architecture demonstrates the power of the Pod-Service abstraction: the routing layer does not need to know how many Pods exist or where they are running. It simply sends traffic to the Service, and Kubernetes handles the rest. This separation of concerns is what makes Kubernetes so effective at managing dynamic, containerized workloads at scale.
