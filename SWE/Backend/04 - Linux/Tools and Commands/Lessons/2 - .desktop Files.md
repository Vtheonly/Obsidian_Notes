# .desktop Files

In Linux systems, application menu entries and launchers are defined by `.desktop` files. These files instruct desktop environments (such as GNOME, KDE, XFCE, and Cinnamon) on how to display and launch applications, including their names, icons, and commands. Managing these `.desktop` files allows users to customize their system's application behavior, adjust the user interface, and even hide unwanted apps from the application menu. The `.desktop` file format is defined by the freedesktop.org Desktop Entry Specification, which ensures consistency across different desktop environments and Linux distributions.

A `.desktop` file is essentially a plain text configuration file with a standard key-value structure. Each file describes a single application, link, or directory entry that can appear in the application menu, on the desktop, or in a file manager. When you install an application through a package manager, it typically installs a `.desktop` file in one of the standard locations, which the desktop environment then discovers and incorporates into the application menu. Understanding how these files work empowers you to customize your desktop experience, create custom launchers for scripts and portable applications, and troubleshoot issues where applications do not appear in the menu as expected.

## Common Locations of .desktop Files

Depending on the installation type and scope (system-wide or user-specific), `.desktop` files can be found in various directories. The desktop environment searches these locations in a specific order, with user-specific entries taking precedence over system-wide ones, which allows users to override system defaults with their own customizations.

### 1. User-Specific Application Menu Entries

- **Location**: `~/.local/share/applications/`
- Stores `.desktop` files for applications installed by the user or customized launchers. This location is user-specific and does not affect other users on the system. When you create a custom `.desktop` file for a portable application (like an AppImage), this is the recommended place to put it.

### 2. System-Wide Application Menu Entries

- **Location**: `/usr/share/applications/`
- This directory holds `.desktop` files for applications installed system-wide, accessible to all users on the system. Most applications installed via APT place their `.desktop` files here.

### 3. Snap and Flatpak Application Menu Entries

#### Snap Applications
- **Location**: `/var/lib/snapd/desktop/applications/`
- Applications installed through Snap place their `.desktop` files here, affecting how they appear in the menu system. Snap manages these files automatically, and you should not modify them directly.

#### Flatpak Applications
- **Locations**:
  - **System-wide**: `/var/lib/flatpak/exports/share/applications/`
  - **User-specific**: `~/.local/share/flatpak/exports/share/applications/`
  These locations are specific to Flatpak applications, distinguishing system-wide from user-specific entries.

### 4. Wine Applications

- **Location**: `~/.local/share/applications/wine/Programs/`
- Applications installed through Wine (a compatibility layer for running Windows applications) place their `.desktop` files in this directory. Wine uses this folder to create launchers for Windows applications, making them accessible from the Linux desktop environment.

## Modifying .desktop Files

Customizing `.desktop` files can change how applications are displayed in the menu or even hide them entirely. Common modifications include altering the application's name, icon, or the command that launches it. When modifying a system-wide `.desktop` file, it is best practice to copy it to the user-specific directory (`~/.local/share/applications/`) and modify the copy rather than editing the original. This way, your changes will not be overwritten when the package is updated.

### Common Fields in .desktop Files

- **Name**: The display name of the application in the app menu.
- **Icon**: Path to the icon file associated with the application, or a freedesktop.org icon name that the system can resolve through the icon theme.
- **Exec**: The command to run when the application is launched. This can include command-line arguments and special field codes like `%f` (a single file name) or `%U` (a list of URLs).
- **Type**: The type of the desktop entry. For applications, this is always `Application`.
- **Categories**: Semicolon-separated list of categories under which the application should appear in the menu (e.g., `Development;IDE;`).
- **NoDisplay**: Set this to `true` to hide the application from the app menu without uninstalling it.
- **Terminal**: Set this to `true` if the application needs to run in a terminal window.
- **Comment**: A brief description of the application, displayed as a tooltip in many desktop environments.

### Editing Steps

1. **Locate the `.desktop` File**: Identify the correct file by navigating to one of the directories mentioned above.
2. **Copy to User Directory** (recommended): If the file is in a system directory, copy it to `~/.local/share/applications/` before editing.
3. **Open with a Text Editor**: Use editors like `nano`, `vim`, or `gedit` to open the `.desktop` file.
4. **Modify the Fields**: Make changes to fields like `Name`, `Icon`, `Exec`, or add `NoDisplay=true` to hide the app from the menu.
5. **Save and Refresh**: After saving the changes, refresh the desktop environment or restart the system for the modifications to take effect. On some desktop environments, you can force a refresh by running `update-desktop-database ~/.local/share/applications/`.

### Example .desktop File

```ini
[Desktop Entry]
Name=ExampleApp
Comment=A custom application launcher
Exec=/path/to/executable
Icon=/path/to/icon
Type=Application
Categories=Utility;
NoDisplay=true
```

- `Name=ExampleApp`: This defines how the application will be named in the app menu.
- `Comment=A custom application launcher`: A brief tooltip shown when hovering over the entry.
- `Exec=/path/to/executable`: This points to the command that launches the application.
- `Icon=/path/to/icon`: The location of the icon file to display.
- `Categories=Utility;`: The menu category under which the application appears.
- `NoDisplay=true`: This will hide the app from the menu but keep it installed and usable.

## Hiding Applications from the Application Menu

Sometimes you may want to hide an application from the app menu without uninstalling it (for example, if it is part of a package with dependencies or if you prefer to launch it only from the terminal). You can achieve this by modifying the corresponding `.desktop` file and adding the `NoDisplay=true` directive.

1. **Locate the `.desktop` File**: For example, Wine applications may have their `.desktop` files under `~/.local/share/applications/wine/Programs/`.
2. **Copy to User Directory** (if needed): Copy the system-wide `.desktop` file to `~/.local/share/applications/`.
3. **Edit the File**: Open the `.desktop` file and add the following line:
   ```ini
   NoDisplay=true
   ```
4. **Save the File** and refresh the desktop environment.

This method hides the application from the application menu but retains its functionality for use through the command line or other methods.

## Further Customization

To create custom launchers or advanced application configurations, you can reference the structure and fields of existing `.desktop` files. The freedesktop.org Desktop Entry Specification defines many additional fields for localization, startup notifications, DBus activation, and more. Modifying or creating new `.desktop` entries gives you full control over how applications behave and appear in your Linux desktop environment, making it possible to integrate portable applications, scripts, and custom workflows seamlessly into your desktop experience.
