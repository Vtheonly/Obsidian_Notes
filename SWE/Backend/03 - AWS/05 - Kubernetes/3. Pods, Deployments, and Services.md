# 3. Pods, Deployments, and Services

> [!info] Chapter Context
> Pods, Deployments, and Services are the three core resources you will work with daily in Kubernetes. This note covers their YAML structure, how they interact, and the patterns you need to deploy applications.

## 1. Pods

A **Pod** is the smallest deployable unit. It contains one or more containers that share networking (same IP, same port space) and storage volumes.

### 1.1 A Simple Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      ports:
        - containerPort: 80
      resources:
        requests:
          cpu: 100m
          memory: 128Mi
        limits:
          cpu: 200m
          memory: 256Mi
      livenessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 10
        periodSeconds: 5
      readinessProbe:
        httpGet:
          path: /
          port: 80
        initialDelaySeconds: 5
        periodSeconds: 5
```

### 1.2 Resource Requests and Limits

- **Requests** — What the scheduler guarantees. The pod is scheduled on a node with at least this much free CPU/memory.
- **Limits** — The hard cap. The pod cannot use more.

```yaml
resources:
  requests:
    cpu: 100m          # 100 millicores = 0.1 CPU
    memory: 128Mi      # 128 mebibytes
  limits:
    cpu: 500m
    memory: 512Mi
```

If a container exceeds its memory limit, it is OOM-killed (exit code 137). If it exceeds its CPU limit, it is throttled (runs slower).

### 1.3 Liveness and Readiness Probes

- **Liveness probe** — "Is the container healthy?" If it fails, Kubernetes restarts the container.
- **Readiness probe** — "Is the container ready to serve traffic?" If it fails, Kubernetes removes the pod from the Service's endpoints (no traffic), but does not restart it.

Both support `httpGet`, `tcpSocket`, and `exec` checks.

### 1.4 Multi-Container Pods (Sidecars)

A pod can have multiple containers. They share networking and volumes. Common patterns:

- **Sidecar** — A helper container (e.g., log shipper, metrics exporter).
- **Ambassador** — A proxy container that handles outbound traffic.
- **Adapter** — A container that transforms the main container's output.

```yaml
spec:
  containers:
    - name: app
      image: myapp:1.0
    - name: log-shipper
      image: fluent-bit:latest
      volumeMounts:
        - name: logs
          mountPath: /var/log/app
  volumes:
    - name: logs
      emptyDir: {}
```

## 2. Deployments

A **Deployment** manages a ReplicaSet, which manages pods. It handles:

- Ensuring N replicas are running.
- Rolling updates (gradually replace old pods with new).
- Rollbacks (undo a bad deployment).

### 2.1 A Simple Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
```

The `selector` says "this deployment manages pods with label `app: nginx`." The `template` defines what those pods look like.

### 2.2 Rolling Updates

When you change `image: nginx:1.25` to `image: nginx:1.26`:

1. The Deployment creates a new ReplicaSet with 0 replicas.
2. It scales up the new ReplicaSet by 1 and scales down the old by 1.
3. It waits for the new pod to become ready.
4. It repeats until the new ReplicaSet has 3 replicas and the old has 0.

Default strategy: `RollingUpdate` with `maxSurge=25%` and `maxUnavailable=25%`.

For zero-downtime, you need:
- A readiness probe (so Kubernetes waits for the new pod to be ready).
- Multiple replicas (so traffic continues during the update).
- A Service that load-balances across ready pods.

### 2.3 Rollback

```bash
# View rollout history
kubectl rollout history deployment/nginx-deployment

# Roll back to the previous version
kubectl rollout undo deployment/nginx-deployment

# Roll back to a specific revision
kubectl rollout undo deployment/nginx-deployment --to-revision=2

# Watch the rollout
kubectl rollout status deployment/nginx-deployment
```

### 2.4 Scaling

```bash
kubectl scale deployment nginx-deployment --replicas=5
```

Or use a HorizontalPodAutoscaler (HPA) to scale based on CPU/memory/custom metrics:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx-deployment
  minReplicas: 3
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

## 3. Services

A **Service** provides a stable network identity for a set of pods.

### 3.1 Why Services Exist

Pods come and go. Each pod has a different IP. If your backend calls the database at `10.0.0.5`, and that pod is replaced by one at `10.0.0.7`, the backend breaks.

A Service has a stable IP and DNS name. It load-balances traffic to its associated pods (selected by label). As pods come and go, the Service's endpoint list updates, but the Service's IP and name stay the same.

### 3.2 ClusterIP (Default)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  type: ClusterIP
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
```

Reachable only from within the cluster at `nginx-service.default.svc.cluster.local:80` (or just `nginx-service:80` within the same namespace).

### 3.3 NodePort

```yaml
spec:
  type: NodePort
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30080
```

Exposes the service on each node's IP at port 30080 (must be in 30000-32767). Reachable from outside the cluster at `<any-node-ip>:30080`.

### 3.4 LoadBalancer

```yaml
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
```

Provisions a cloud load balancer (ELB on AWS) that routes to the service. The cloud controller manager handles provisioning. On AWS, this creates a Classic Load Balancer by default; use the AWS Load Balancer Controller annotation to create an ALB or NLB.

### 3.5 Headless Service

```yaml
spec:
  clusterIP: None
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
```

A headless service does not allocate a ClusterIP. DNS queries return the pod IPs directly. Used with StatefulSets (e.g., for databases where each pod has a stable identity).

## 4. StatefulSets

A **StatefulSet** is like a Deployment, but for stateful applications. Pods have:

- Stable identities (`app-0`, `app-1`, `app-2`).
- Stable DNS names (`app-0.nginx.default.svc.cluster.local`).
- Stable persistent storage (each pod gets its own volume).

Use StatefulSets for:

- Databases (MySQL, PostgreSQL, MongoDB, Cassandra).
- Distributed systems (Kafka, ZooKeeper, Elasticsearch).
- Any application where pod identity matters.

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:15
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 10Gi
```

## 5. DaemonSets

A **DaemonSet** runs a pod on every node (or a subset). Use cases:

- Log collection (Fluent Bit, Filebeat).
- Monitoring (Prometheus Node Exporter).
- Network plugins (Calico, AWS VPC CNI).
- Storage daemons (Ceph, GlusterFS).

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: log-collector
spec:
  selector:
    matchLabels:
      app: log-collector
  template:
    metadata:
      labels:
        app: log-collector
    spec:
      containers:
        - name: fluent-bit
          image: fluent/fluent-bit:latest
```

## 6. Jobs and CronJobs

- **Job** — Runs a pod to completion (e.g., a database migration).
- **CronJob** — Runs a Job on a schedule (e.g., a daily backup).

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  template:
    spec:
      containers:
        - name: migrate
          image: myapp:1.0
          command: ["python", "manage.py", "migrate"]
      restartPolicy: OnFailure
```

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: backup
              image: myapp:1.0
              command: ["/scripts/backup.sh"]
          restartPolicy: OnFailure
```

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Deploying Pods Without Deployments
> A bare Pod is not restarted when it dies. Use a Deployment (or StatefulSet/DaemonSet).

> [!warning] Mistake 2 — Forgetting Resource Requests
> Without requests, the scheduler does not know how much CPU/memory the pod needs. It may schedule too many pods on one node, causing OOM kills.

> [!warning] Mistake 3 — Forgetting Probes
> Without a readiness probe, traffic is sent to a pod before it is ready. Without a liveness probe, hung pods are never restarted.

> [!warning] Mistake 4 — Using Services Incorrectly
> ClusterIP for internal, NodePort/LoadBalancer for external. Use Ingress for HTTP routing instead of many LoadBalancers.

> [!warning] Mistake 5 — Using a Deployment for a Database
> Databases need stable identities and storage. Use a StatefulSet (or, even better, use a managed database like RDS).

> [!warning] Mistake 6 — Mismatched Labels
> A Service's `selector` must match the pod's `labels`. A typo means no traffic.

## 8. Summary Checklist

- [ ] Pod = smallest unit; one or more containers sharing networking/storage.
- [ ] Resource requests: scheduler guarantee. Limits: hard cap.
- [ ] Liveness probe: restart on failure. Readiness probe: stop traffic on failure.
- [ ] Deployment = manages ReplicaSet = manages pods. Handles rolling updates and rollbacks.
- [ ] Service = stable IP/DNS for a set of pods.
- [ ] Service types: ClusterIP (internal), NodePort, LoadBalancer (cloud LB), Headless (for StatefulSets).
- [ ] StatefulSet = for stateful apps (databases). Stable identity, stable storage.
- [ ] DaemonSet = one pod per node.
- [ ] Job = run to completion. CronJob = scheduled Job.
