# 4. ConfigMaps, Secrets, and Storage

> [!info] Chapter Context
> Applications need configuration (URLs, feature flags), secrets (passwords, API keys), and persistent storage. Kubernetes provides ConfigMaps, Secrets, and PersistentVolumes for these. This note covers all three.

Related: [[3. Pods, Deployments, and Services]] | [[5. EKS and Managed Kubernetes]] | [[07 - Identity and Security/4. Secrets Management in AWS]]

## 1. ConfigMaps

A ConfigMap stores non-sensitive configuration as key-value pairs.

### 1.1 Creating a ConfigMap

```bash
# From literal values
kubectl create configmap app-config \
  --from-literal=LOG_LEVEL=debug \
  --from-literal=PORT=8080

# From a file
kubectl create configmap app-config --from-file=config.yaml

# From a directory
kubectl create configmap app-config --from-file=./config-dir/
```

### 1.2 YAML Form

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: debug
  PORT: "8080"
  config.yaml: |
    database:
      host: db.example.com
      port: 5432
    features:
      new_ui: true
```

### 1.3 Using a ConfigMap in a Pod

**As environment variables:**

```yaml
spec:
  containers:
    - name: app
      image: myapp:1.0
      envFrom:
        - configMapRef:
            name: app-config
```

Or selectively:

```yaml
spec:
  containers:
    - name: app
      image: myapp:1.0
      env:
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
```

**As a volume mount:**

```yaml
spec:
  containers:
    - name: app
      image: myapp:1.0
      volumeMounts:
        - name: config
          mountPath: /etc/app
  volumes:
    - name: config
      configMap:
        name: app-config
```

This mounts each key as a file in `/etc/app/` (e.g., `/etc/app/LOG_LEVEL`, `/etc/app/config.yaml`).

## 2. Secrets

A Secret is like a ConfigMap, but for sensitive data. Values are base64-encoded (NOT encrypted by default — you must enable encryption at rest for true security).

### 2.1 Creating a Secret

```bash
# From literal values
kubectl create secret generic db-secret \
  --from-literal=username=admin \
  --from-literal=password=password123

# From a file (e.g., an SSH key)
kubectl create secret generic ssh-key --from-file=ssh-privatekey=~/.ssh/id_rsa

# Docker registry secret (for pulling from ECR)
kubectl create secret docker-registry ecr-secret \
  --docker-server=123456789012.dkr.ecr.us-east-1.amazonaws.com \
  --docker-username=AWS \
  --docker-password=$(aws ecr get-login-password) \
  --docker-email=unused
```

### 2.2 YAML Form

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  username: YWRtaW4=             # base64("admin")
  password: cGFzc3dvcmQxMjM=    # base64("password123")
```

Generate the base64:

```bash
echo -n 'admin' | base64
echo -n 'password123' | base64
```

### 2.3 Using a Secret

**As environment variables:**

```yaml
spec:
  containers:
    - name: app
      image: myapp:1.0
      env:
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: username
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password
```

**As a volume mount:**

```yaml
spec:
  containers:
    - name: app
      image: myapp:1.0
      volumeMounts:
        - name: secrets
          mountPath: /etc/secrets
          readOnly: true
  volumes:
    - name: secrets
      secret:
        secretName: db-secret
```

### 2.4 Secret Security Best Practices

- **Enable encryption at rest** — By default, Secrets in etcd are only base64-encoded. Enable encryption at rest with a KMS provider.
- **Use RBAC** — Restrict who can read Secrets.
- **Use external secret managers** — Tools like External Secrets Operator sync secrets from AWS Secrets Manager / Parameter Store into Kubernetes Secrets, so you don't manage them manually.
- **Never commit Secrets to Git** — Use `.gitignore` or tools like Sealed Secrets / SOPS.

> [!warning] Base64 Is Not Encryption
> Anyone with `kubectl get secret` can decode your secrets with `base64 -d`. Base64 is just encoding. Enable encryption at rest and use RBAC.

## 3. PersistentVolumes and PersistentVolumeClaims

Pods are ephemeral. When a pod dies, its filesystem is gone. For databases and other stateful apps, you need persistent storage.

### 3.1 The Three Resources

- **PersistentVolume (PV)** — A piece of storage in the cluster (e.g., a 100 GB EBS volume). Provisioned by an admin or dynamically by a StorageClass.
- **PersistentVolumeClaim (PVC)** — A user's request for storage (e.g., "I need 10 GB").
- **StorageClass** — Defines how PVs are provisioned (e.g., "Use gp2 EBS volumes in us-east-1a").

### 3.2 Dynamic Provisioning

Most clusters use dynamic provisioning. You create a PVC; the StorageClass provisions a PV automatically.

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: gp2
  resources:
    requests:
      storage: 10Gi
```

### 3.3 Using a PVC in a Pod

```yaml
spec:
  containers:
    - name: app
      image: myapp:1.0
      volumeMounts:
        - name: data
          mountPath: /data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: data-pvc
```

### 3.4 Access Modes

- **ReadWriteOnce (RWO)** — Mounted read-write by one node. Most EBS volumes.
- **ReadOnlyMany (ROX)** — Mounted read-only by many nodes.
- **ReadWriteMany (RWX)** — Mounted read-write by many nodes. Requires a shared filesystem (EFS, NFS).
- **ReadWriteOncePod (RWOP)** — Mounted by one pod only (Kubernetes 1.22+).

EBS is RWO (block storage, attached to one node at a time). EFS is RWX (file storage, mounted by many nodes).

### 3.5 StorageClasses on AWS

- **gp2** — General Purpose SSD (default for EKS).
- **gp3** — Newer, cheaper, faster than gp2.
- **io1 / io2** — Provisioned IOPS SSD (high performance).
- **st1** — Throughput-optimized HDD.
- **sc1** — Cold HDD (cheapest).

For EFS, install the EFS CSI driver and create a StorageClass that provisions EFS access points.

## 4. StatefulSets and Storage

StatefulSets use `volumeClaimTemplates` to give each pod its own PVC:

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:15
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: gp3
        resources:
          requests:
            storage: 20Gi
```

This creates 3 PVCs (`data-postgres-0`, `data-postgres-1`, `data-postgres-2`), each with 20 GB. Each pod mounts its own PVC, which persists across pod restarts.

## 5. Common Student Mistakes

> [!warning] Mistake 1 — Treating Secrets as Encrypted
> Secrets are base64-encoded, not encrypted. Enable encryption at rest. Use external secret managers for production.

> [!warning] Mistake 2 — Forgetting to Set Resource Requests on Storage
> A PVC must specify `resources.requests.storage`. Without it, the PVC is invalid.

> [!warning] Mistake 3 — Using EBS for Multi-Node Access
> EBS is ReadWriteOnce — only one node can mount it. For multi-node access, use EFS (ReadWriteMany).

> [!warning] Mistake 4 — Committing Secrets to Git
> Never commit Secret YAMLs to Git. Use External Secrets Operator, Sealed Secrets, or SOPS.

> [!warning] Mistake 5 — Forgetting That ConfigMap Updates Don't Restart Pods
> If you change a ConfigMap mounted as environment variables, the pod does not pick up the change. You must restart the pod. (Volume-mounted ConfigMaps auto-update, but env-var ConfigMaps do not.)

> [!warning] Mistake 6 — Using a Deployment for a Database
> Databases need stable storage. Use a StatefulSet (or use a managed database like RDS).

## 6. Summary Checklist

- [ ] ConfigMaps: non-sensitive configuration. Mounted as env vars or volumes.
- [ ] Secrets: sensitive data, base64-encoded (NOT encrypted by default). Enable encryption at rest.
- [ ] PersistentVolume (PV): storage in the cluster. PersistentVolumeClaim (PVC): user's request for storage.
- [ ] StorageClass: how PVs are provisioned (e.g., gp2, gp3 EBS).
- [ ] Access modes: RWO (one node), ROX (read-only many), RWX (read-write many, needs EFS), RWOP (one pod).
- [ ] StatefulSets use `volumeClaimTemplates` for per-pod PVCs.
- [ ] Use External Secrets Operator to sync from AWS Secrets Manager.
- [ ] ConfigMap env vars do not auto-update; restart the pod to pick up changes.
