# 1. Database Fundamentals

> [!info] Chapter Context
> AWS offers many database services. Choosing the right one requires understanding the SQL/NoSQL divide, ACID vs. BASE, consistency models, and access patterns. This note covers the foundations; later notes cover RDS, DynamoDB, and ElastiCache.

## 1. SQL vs. NoSQL

### 1.1 SQL (Relational)

- **Data model** — Tables with rows and columns; relationships via foreign keys.
- **Schema** — Strict; defined upfront.
- **Query language** — SQL (Structured Query Language).
- **Transactions** — ACID (Atomic, Consistent, Isolated, Durable).
- **Scaling** — Vertical (bigger server); horizontal scaling is hard.
- **Examples** — PostgreSQL, MySQL, Oracle, SQL Server, Aurora.

Best for:

- Complex queries with JOINs.
- Multi-row transactions (banking, e-commerce checkout).
- Well-structured, consistent data.
- Ad-hoc reporting.

### 1.2 NoSQL (Non-Relational)

- **Data model** — Key-value, document, column-family, or graph.
- **Schema** — Flexible; each record can have different fields.
- **Query language** — Varies (often API-based, not SQL).
- **Transactions** — BASE (Basically Available, Soft state, Eventual consistency); some support ACID on a single item.
- **Scaling** — Horizontal (sharding built-in).
- **Examples** — DynamoDB, MongoDB, Cassandra, Redis, Neo4j.

Best for:

- High-velocity reads/writes at scale.
- Simple access patterns (key-based lookup).
- Flexible or evolving schemas.
- Real-time big data.

## 2. ACID vs. BASE

### 2.1 ACID (SQL)

- **Atomicity** — A transaction is all-or-nothing. Either all changes happen, or none.
- **Consistency** — The database always satisfies its constraints (foreign keys, unique, etc.).
- **Isolation** — Concurrent transactions do not interfere; the result is the same as if they ran sequentially.
- **Durability** — Once committed, the change survives crashes.

ACID is essential for financial transactions, inventory management, anywhere data correctness is critical.

### 2.2 BASE (NoSQL)

- **Basically Available** — The system is always available (may return stale data during partitions).
- **Soft state** — State can change without input (due to eventual consistency).
- **Eventual consistency** — The system will become consistent over time, but not immediately.

BASE trades strong consistency for high availability and scale. Suitable for social media feeds, shopping carts, recommendation systems.

## 3. The CAP Theorem (Recap)

Covered in [[00 - Foundations/2. Distributed Systems Basics]]. For databases:

- **CP** — Strong consistency; may be unavailable during partitions. Examples: HBase, Spanner.
- **AP** — High availability; eventual consistency. Examples: DynamoDB (default), Cassandra.

DynamoDB offers **tunable consistency** — you can choose strong (costs more) or eventual per request.

## 4. Access Patterns Drive Database Choice

Before picking a database, list your access patterns:

- "Get user by ID."
- "List orders for a user, sorted by date."
- "Find users by email."
- "Get the top 10 products by sales."

For SQL, you can query any way you want (with indexes). For NoSQL, you design the schema around the access patterns — if you cannot query by a field, you cannot query by it (efficiently).

> [!tip] NoSQL Requires Upfront Access Pattern Design
> With SQL, you add an index later. With NoSQL, you must design the schema around your access patterns from the start. Adding a new access pattern to DynamoDB may require a new table or a Global Secondary Index.

## 5. Common Database Services on AWS

| Service | Type | Use case |
| :--- | :--- | :--- |
| **RDS** | Managed SQL (MySQL, PostgreSQL, etc.) | Traditional relational databases |
| **Aurora** | AWS proprietary SQL (MySQL/PostgreSQL-compatible) | High-performance SQL |
| **DynamoDB** | Managed NoSQL (key-value) | High-scale, low-latency key-based access |
| **ElastiCache** | Managed Redis / Memcached | Caching, session store, leaderboards |
| **DocumentDB** | Managed MongoDB-compatible | Document storage (MongoDB API) |
| **Neptune** | Managed graph database | Social networks, recommendation engines |
| **Timestream** | Time-series database | IoT, metrics, logs |
| **QLDB** | Quantum ledger database | Immutable, verifiable logs |
| **Redshift** | Data warehouse | Analytics, BI |

## 6. When to Use Which

- **RDS / Aurora** — Traditional SQL needs; complex queries; multi-row transactions.
- **DynamoDB** — Massive scale, single-digit-ms latency, key-based access.
- **ElastiCache (Redis)** — Caching, session storage, real-time leaderboards.
- **DocumentDB** — If you're already using MongoDB.
- **Neptune** — Graph queries (e.g., "find friends of friends of friends").
- **Redshift** — OLAP (analytics) on petabytes.

For most web apps, a single RDS/Aurora database + an ElastiCache cache is sufficient. Add DynamoDB for high-scale, low-latency needs (e.g., session storage, IoT telemetry).

## 7. Indexing

An index is a data structure that speeds up queries on a specific field. Without an index, the database must scan every row (slow).

- **SQL** — Create indexes on columns you query by:
  ```sql
  CREATE INDEX idx_users_email ON users(email);
  ```
- **DynamoDB** — Use Global Secondary Indexes (GSIs) for non-key queries.
- **Trade-off** — Indexes speed up reads but slow down writes (the index must be updated). Don't over-index.

---

## 8. Connection Pooling

Opening a new database connection for every request is slow (TCP handshake, TLS handshake, authentication). A **connection pool** keeps a set of open connections and reuses them.

- **In the application** — Most frameworks (Django, Rails, Spring) have built-in pooling.
- **At the database** — Use **RDS Proxy** (managed connection pooling) for Lambda and high-concurrency apps.

Without pooling, you may hit the database's `max_connections` limit and start getting "too many connections" errors.

## 9. Common Student Mistakes

> [!warning] Mistake 1 — Using NoSQL for Complex Queries
> NoSQL is bad at JOINs and ad-hoc queries. If you need complex queries, use SQL.

> [!warning] Mistake 2 — Using SQL for Massive Scale
> SQL databases are hard to scale horizontally. If you need millions of writes per second, use DynamoDB.

> [!warning] Mistake 3 — Not Designing Access Patterns for NoSQL
> DynamoDB requires upfront access pattern design. Adding new patterns later is expensive.

> [!warning] Mistake 4 — Over-Indexing
> Every index slows down writes. Add only the indexes you need.

> [!warning] Mistake 5 — Not Using Connection Pooling
> Opening a new connection per request exhausts the database. Use a pool or RDS Proxy.

> [!warning] Mistake 6 — Using the Database as a Queue
> Polling a database for new rows is inefficient. Use SQS or a message queue.

## 10. Summary Checklist

- [ ] SQL: structured, ACID, complex queries, hard to scale horizontally.
- [ ] NoSQL: flexible schema, BASE (eventual consistency), horizontal scaling.
- [ ] Access patterns drive database choice, especially for NoSQL.
- [ ] AWS SQL: RDS (managed), Aurora (high-performance).
- [ ] AWS NoSQL: DynamoDB (key-value), DocumentDB (MongoDB-compatible), Neptune (graph).
- [ ] AWS Cache: ElastiCache (Redis / Memcached).
- [ ] AWS Analytics: Redshift (data warehouse).
- [ ] Use indexes judiciously (speed up reads, slow down writes).
- [ ] Use connection pooling (RDS Proxy for Lambda).
