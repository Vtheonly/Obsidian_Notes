# 3. DynamoDB Fundamentals

> [!info] Chapter Context
> DynamoDB is AWS's managed NoSQL database. It is designed for massive scale (millions of requests per second) and single-digit-millisecond latency. This note covers tables, partition keys, sort keys, GSIs, and the Query vs. Scan distinction.

Related: [[2. RDS and Relational Databases]] | [[4. ElastiCache and In-Memory Stores]] | [[10 - Event Driven Systems/1. Events and Pub-Sub]]

## 1. What DynamoDB Is

DynamoDB is a **key-value and document database**. It is:

- **Fully managed** — No servers to provision, patch, or maintain.
- **Massively scalable** — Handles trillions of requests per day for Amazon's own retail site.
- **Single-digit-millisecond latency** at any scale (microseconds with DAX).
- **Highly available** — Data replicated across 3 AZs.
- **Schemaless** (mostly) — Each item can have different attributes, but the key schema is fixed.
- **Pay-per-use** — On-demand (per request) or provisioned (per capacity unit).

## 2. Core Concepts

### 2.1 Table

A collection of items. Like a SQL table, but schemaless (except for the key).

### 2.2 Item

A single data record, identified by its primary key. Each item can have any number of attributes. Max item size: 400 KB.

```json
{
  "id": {"N": "1"},                  // Partition key
  "email": {"S": "alice@example.com"},
  "name": {"S": "Alice"},
  "age": {"N": "30"},
  "interests": {"L": [{"S": "hiking"}, {"S": "reading"}]}
}
```

### 2.3 Primary Key

Two options:

- **Partition key only** — A single attribute uniquely identifies the item. Like a HashMap.
- **Partition key + sort key (composite)** — The combination uniquely identifies the item. Multiple items can share a partition key; they differ by sort key.

Examples:

- `UserID` (partition only) — Each user is one item.
- `UserID` + `OrderDate` (composite) — Multiple orders per user, sorted by date.

### 2.4 Partition Key (HASH)

DynamoDB uses the partition key to determine which partition (physical storage) an item goes to. It computes `hash(partition_key) % N_partitions`.

This means:

- All items with the same partition key are stored together (on the same partition).
- You can efficiently query all items for a given partition key.
- You CANNOT query by a non-partition key without a scan or GSI.

### 2.5 Sort Key (RANGE)

Within a partition, items are sorted by the sort key. You can:

- Query items with a specific partition key and sort key in a range.
- Get the top N items by sort key.
- Use sort key conditions: `=`, `<`, `<=`, `>`, `>=`, `BETWEEN`, `begins_with`.

Example: `UserID` + `Timestamp` lets you query "all events for user 123 in the last hour."

## 3. Creating a Table

```bash
aws dynamodb create-table \
  --table-name Users \
  --attribute-definitions \
    AttributeName=id,AttributeType=N \
    AttributeName=email,AttributeType=S \
  --key-schema \
    AttributeName=id,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --global-secondary-indexes '[
    {
      "IndexName": "email-index",
      "KeySchema": [{"AttributeName": "email", "KeyType": "HASH"}],
      "Projection": {"ProjectionType": "ALL"}
    }
  ]'
```

Key points:

- `AttributeType` can be `S` (string), `N` (number), or `B` (binary).
- `KeyType` is `HASH` (partition) or `RANGE` (sort).
- `PAY_PER_REQUEST` (on-demand) is simplest; `PROVISIONED` is cheaper for steady workloads.

## 4. CRUD Operations

### 4.1 PutItem

```bash
aws dynamodb put-item \
  --table-name Users \
  --item '{
    "id": {"N": "1"},
    "email": {"S": "alice@example.com"},
    "name": {"S": "Alice"},
    "age": {"N": "30"}
  }'
```

### 4.2 GetItem

```bash
aws dynamodb get-item \
  --table-name Users \
  --key '{"id": {"N": "1"}}'
```

Returns the item (or empty if not found). Strongly consistent by default in the latest version; previously eventual by default.

### 4.3 UpdateItem

```bash
aws dynamodb update-item \
  --table-name Users \
  --key '{"id": {"N": "1"}}' \
  --update-expression "SET age = :newAge" \
  --expression-attribute-values '{":newAge": {"N": "31"}}' \
  --return-values UPDATED_NEW
```

UpdateExpression uses a custom syntax (`SET`, `ADD`, `REMOVE`, `DELETE`).

### 4.4 DeleteItem

```bash
aws dynamodb delete-item \
  --table-name Users \
  --key '{"id": {"N": "1"}}'
```

## 5. Query vs. Scan

These are the two ways to read multiple items. They are very different.

### 5.1 Query

`Query` retrieves items by partition key (optionally filtered by sort key).

```bash
aws dynamodb query \
  --table-name Orders \
  --key-condition-expression "user_id = :uid AND order_date BETWEEN :start AND :end" \
  --expression-attribute-values '{
    ":uid": {"S": "alice"},
    ":start": {"S": "2024-01-01"},
    ":end": {"S": "2024-12-31"}
  }'
```

- **Efficient** — Reads only the matching partition.
- **Requires** the partition key.
- Optional sort key condition.
- Optional filter expression (applied AFTER the read; doesn't reduce consumed capacity).

### 5.2 Scan

`Scan` reads the entire table.

```bash
aws dynamodb scan --table-name Users
```

- **Inefficient** — Reads every item.
- **Expensive** — Consumes read capacity for every item scanned.
- Useful for: small tables, full exports, occasional reports.

> [!warning] Avoid Scan on Large Tables
> A Scan on a 10-million-item table is slow and expensive. Use Query with a partition key, or design a GSI for the access pattern.

### 5.3 FilterExpression

Filters results AFTER reading. Does not reduce consumed capacity.

```bash
aws dynamodb scan \
  --table-name Users \
  --filter-expression "age > :minAge" \
  --expression-attribute-values '{":minAge": {"N": "18"}}'
```

This still scans the whole table — `FilterExpression` just removes non-matching items from the response. If you find yourself using `FilterExpression` with Scan frequently, you need a GSI.

## 6. Global Secondary Indexes (GSIs)

A GSI lets you query by a different key. For example, if your table has `id` as the partition key but you also want to query by `email`, create a GSI on `email`.

```bash
aws dynamodb update-table \
  --table-name Users \
  --attribute-definitions AttributeName=email,AttributeType=S \
  --global-secondary-index-updates '[{
    "Create": {
      "IndexName": "email-index",
      "KeySchema": [{"AttributeName": "email", "KeyType": "HASH"}],
      "Projection": {"ProjectionType": "ALL"}
    }
  }]'
```

Query the GSI:

```bash
aws dynamodb query \
  --table-name Users \
  --index-name email-index \
  --key-condition-expression "email = :email" \
  --expression-attribute-values '{":email": {"S": "alice@example.com"}}'
```

GSI considerations:

- **Eventually consistent** only (cannot be strongly consistent).
- **Consumes its own capacity** (separate from the base table).
- **Projection** — Choose `ALL` (all attributes), `KEYS_ONLY` (only keys), or `INCLUDE` (specific attributes). Smaller projections are cheaper.
- **Can be created and deleted on the fly** (no downtime).

## 7. Consistency Models

### 7.1 Eventually Consistent (Default)

Reads may return stale data (within a second or two). Cheaper; uses half the capacity of strong reads.

### 7.2 Strongly Consistent

Reads return the most recent write. Costs 2x capacity. Not available on GSIs.

```bash
aws dynamodb get-item --table-name Users --key '{"id": {"N": "1"}}' --consistent-read
```

---

## 8. Capacity Modes

### 8.1 On-Demand

Pay per request. Simple; no capacity planning. ~3x more expensive per request than provisioned.

Best for: unpredictable workloads, development, low-traffic apps.

### 8.2 Provisioned

You specify Read Capacity Units (RCUs) and Write Capacity Units (WCUs) per second.

- 1 RCU = 1 strongly consistent read per second (or 2 eventually consistent reads) for items up to 4 KB.
- 1 WCU = 1 write per second for items up to 1 KB.

Best for: predictable, steady workloads. Can use **Auto Scaling** to adjust capacity based on usage.

## 9. DynamoDB Streams

DynamoDB Streams captures changes (inserts, updates, deletes) to a table. Each change is a record in a stream, which can be consumed by Lambda or Kinesis.

Use cases:

- Real-time aggregation (e.g., update a count when items are added).
- Replicate to another table or service.
- Trigger notifications.

```bash
aws dynamodb update-table \
  --table-name Users \
  --stream-specification StreamEnabled=true,StreamViewType=NEW_AND_OLD_IMAGES
```

The Lambda trigger pattern: DynamoDB Streams → Lambda function. Each batch of records invokes the function.

## 10. DynamoDB Global Tables

Multi-region, multi-active DynamoDB. Writes in any region are replicated to all others (typically <1 second).

```bash
aws dynamodb create-global-table \
  --global-table-name Users \
  --replication-group '[{"RegionName":"us-east-1"},{"RegionName":"eu-west-1"}]'
```

Use cases:

- Disaster recovery.
- Low-latency global access (writes from any region).

## 11. Common Student Mistakes

> [!warning] Mistake 1 — Using Scan Instead of Query
> Scan reads the entire table. Always prefer Query (with a GSI if needed).

> [!warning] Mistake 2 — Forgetting to Design for Access Patterns
> DynamoDB requires you to design the schema for your queries. If you need a new query later, you may need a new GSI.

> [!warning] Mistake 3 — Storing Large Items
> Max item size is 400 KB. For larger data, store metadata in DynamoDB and the actual data in S3.

> [!warning] Mistake 4 — Hot Partitions
> If one partition key is much more popular than others, that partition becomes a bottleneck. Use a random suffix on the key to spread load.

> [!warning] Mistake 5 — Forgetting GSI Eventually Consistent
> GSIs are eventually consistent. If you write then immediately read from the GSI, you may not see the new item.

> [!warning] Mistake 6 — Using FilterExpression to Reduce Cost
> FilterExpression is applied AFTER the read. It does not reduce consumed capacity. Use a GSI.

## 12. Summary Checklist

- [ ] DynamoDB is a managed NoSQL key-value/document database.
- [ ] Primary key: partition key (HASH) only, or partition + sort key (HASH + RANGE).
- [ ] Items up to 400 KB; attributes can be S, N, B, L, M, etc.
- [ ] Query: efficient, requires partition key. Scan: reads the whole table (avoid on large tables).
- [ ] FilterExpression filters after reading (does not save capacity).
- [ ] GSIs let you query by a different key; eventually consistent; consume their own capacity.
- [ ] Consistency: eventual (default, cheap) or strong (2x capacity).
- [ ] Capacity: on-demand (simple, expensive) or provisioned (cheaper, requires planning).
- [ ] DynamoDB Streams capture changes; Lambda can consume them.
- [ ] Global Tables: multi-region, multi-active replication.
