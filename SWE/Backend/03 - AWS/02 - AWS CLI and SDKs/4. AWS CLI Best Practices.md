# 4. AWS CLI Best Practices

> [!info] Chapter Context
> This note consolidates best practices for using the AWS CLI effectively: scripting, error handling, output parsing, performance, and security.

Related: [[1. The AWS CLI]] | [[2. Boto3 and AWS SDKs]] | [[3. Credentials and Profiles]]

## 1. Scripting with the CLI

### 1.1 Use `--query` to Extract What You Need

```bash
# Get just the instance IDs of running EC2 instances
aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query 'Reservations[*].Instances[*].InstanceId' \
  --output text
```

### 1.2 Use `--output text` for Shell Pipelines

JSON is great for humans; text is better for shell scripts:

```bash
# Iterate over running instance IDs
for id in $(aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query 'Reservations[*].Instances[*].InstanceId' \
  --output text); do
  echo "Stopping $id"
  aws ec2 stop-instances --instance-ids "$id"
done
```

### 1.3 Use `--no-paginate` When You Want a Single Page

By default, the CLI auto-paginates (fetches all pages). For performance, use `--no-paginate` and handle pagination manually if needed:

```bash
aws s3api list-objects-v2 --bucket my-bucket --no-paginate
```

### 1.4 Use `--no-cli-pager` in Scripts

By default, long output is piped through `less`. In scripts, disable this:

```bash
aws s3 ls --no-cli-pager
# Or globally
aws configure set cli_pager ""
```

### 1.5 Use `set -e` in Scripts to Fail on Errors

```bash
#!/bin/bash
set -e   # exit on any error
set -o pipefail  # treat pipe failures as errors

aws s3 mb s3://my-bucket
aws s3 cp file.txt s3://my-bucket/
```

## 2. Error Handling

### 2.1 Check Exit Codes

```bash
if aws s3 ls s3://nonexistent-bucket; then
  echo "Bucket exists"
else
  echo "Bucket does not exist or no access (exit code $?)"
fi
```

### 2.2 Capture Errors with `2>&1`

```bash
output=$(aws s3 ls s3://nonexistent-bucket 2>&1) || {
  echo "Error: $output"
  exit 1
}
```

### 2.3 Use `wait` for Asynchronous Operations

```bash
# Stop an instance and wait for it to stop
aws ec2 stop-instances --instance-ids i-12345
aws ec2 wait instance-stopped --instance-ids i-12345
echo "Instance stopped"
```

## 3. Output Parsing with `jq`

For complex JSON, pipe to `jq`:

```bash
# Get instance ID, name, and state as a table
aws ec2 describe-instances | jq -r '
  .Reservations[].Instances[] |
  [.InstanceId, (.Tags[]? | select(.Key=="Name").Value), .State.Name] |
  @tsv
'
```

Learn `jq` — it is invaluable for AWS CLI output.

## 4. Performance

### 4.1 Use High-Level Commands Where Possible

```bash
# Fast (high-level, parallel)
aws s3 sync ./local s3://my-bucket/

# Slow (low-level, one file at a time)
for file in local/*; do
  aws s3api put-object --bucket my-bucket --key "$(basename $file)" --body "$file"
done
```

### 4.2 Use Multipart Upload for Large Files

`aws s3 cp` automatically uses multipart upload for files > 8 MB. For more control:

```bash
aws s3 cp largefile.bin s3://my-bucket/ \
  --expected-size 5000000000 \
  --multipart-threshold 100MB \
  --multipart-chunksize 50MB
```

### 4.3 Use `--page-size` for Fewer API Calls

```bash
aws s3api list-objects-v2 --bucket my-bucket --page-size 1000
```

### 4.4 Use Multiple Profiles in Parallel

```bash
# Run the same command across all profiles in parallel
for profile in dev staging prod; do
  aws s3 ls --profile $profile &
done
wait
```

## 5. Security

### 5.1 Never Hardcode Credentials in Scripts

```bash
# BAD
AWS_ACCESS_KEY_ID=AKIA... aws s3 ls

# GOOD (use a profile or env var set elsewhere)
aws s3 ls --profile production
```

### 5.2 Use IAM Roles on EC2/ECS/Lambda

If your script runs on EC2, ECS, or Lambda, use the attached IAM role. No credentials needed.

### 5.3 Audit with CloudTrail

Every CLI call is logged in CloudTrail. Use it to audit who did what.

## 6. Common Mistakes

> [!warning] Mistake 1 — Forgetting `--output text` for Shell Scripts
> JSON output is hard to parse in shell. Use `--output text` and iterate over lines.

> [!warning] Mistake 2 — Not Using `--query` to Filter Early
> Pulling all fields and parsing later is slow. Filter with `--query` to get only what you need.

> [!warning] Mistake 3 — Hardcoding Instance IDs
> Instance IDs change. Use `--query` and `--filters` to find them dynamically.

> [!warning] Mistake 4 — Forgetting to Handle Pagination
> If you have more than 1000 S3 objects, a single `list-objects-v2` call does not return them all. Use the Paginator or `--no-paginate` and iterate manually.

> [!warning] Mistake 5 — Using Real AWS Keys in CI
> Use OIDC federation (GitHub Actions, GitLab CI) or assume-role. Never store long-lived keys in CI secrets if you can avoid it.

## 7. Summary Checklist

- [ ] Use `--query` and `--output text` for shell scripts.
- [ ] Use `--no-cli-pager` in scripts to avoid `less`.
- [ ] Use `set -e` and `set -o pipefail` in shell scripts.
- [ ] Check exit codes and capture stderr for error handling.
- [ ] Use `wait` subcommands for asynchronous operations.
- [ ] Use `jq` for complex JSON parsing.
- [ ] Use high-level commands (`aws s3 sync`) for performance.
- [ ] Use IAM roles instead of hardcoded credentials.
- [ ] Audit all CLI calls via CloudTrail.
