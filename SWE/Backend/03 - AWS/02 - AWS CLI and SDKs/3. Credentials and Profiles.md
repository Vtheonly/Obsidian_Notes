# 3. Credentials and Profiles

> [!info] Chapter Context
> Managing credentials across multiple AWS accounts, environments, and LocalStack is one of the most confusing parts of AWS for beginners. This note covers credential types, profiles, environment variables, IAM roles, and the precedence rules.

Related: [[1. The AWS CLI]] | [[2. Boto3 and AWS SDKs]] | [[07 - Identity and Security/2. IAM Users, Groups, and Roles]] | [[07 - Identity and Security/3. IAM Policies]]

## 1. Credential Types

### 1.1 Long-Lived Access Keys

A pair of strings:

- **Access Key ID** — e.g., `AKIAIOSFODNN7EXAMPLE`. Public, safe to log.
- **Secret Access Key** — e.g., `wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY`. Secret, never log.

Long-lived access keys are created for IAM users. They do not expire (unless you rotate them manually). They are the most common cause of cloud security incidents — leaked keys are picked up by attackers within minutes.

### 1.2 Temporary Credentials (STS)

Issued by AWS STS (Security Token Service) when you assume a role. They include:

- Access Key ID (starts with `ASIA`).
- Secret Access Key.
- Session Token (required; long-lived keys do not have one).
- Expiration (typically 1-12 hours).

Temporary credentials are safer because they expire. Use them whenever possible. IAM roles issue temporary credentials via STS.

### 1.3 SSO Credentials

When using AWS IAM Identity Center (formerly AWS SSO), you log in via a browser and get temporary credentials. The CLI caches them in `~/.aws/sso/cache/`.

## 2. Where Credentials Live

### 2.1 `~/.aws/credentials`

```ini
[default]
aws_access_key_id = AKIAIOSFODNN7EXAMPLE
aws_secret_access_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY

[production]
aws_access_key_id = AKIAPRODUCTIONEXAMPLE
aws_secret_access_key = productionsecretkey

[localstack]
aws_access_key_id = test
aws_secret_access_key = test
```

Each `[profile-name]` section is a profile. `[default]` is the default profile (used when no `--profile` is specified).

### 2.2 `~/.aws/config`

```ini
[default]
region = us-east-1
output = json

[profile production]
region = us-west-2
output = table
role_arn = arn:aws:iam::123456789012:role/ProductionRole
source_profile = default

[profile localstack]
region = us-east-1
endpoint_url = http://localhost:4566
```

Note the `profile` prefix: sections in `~/.aws/config` are `[profile name]` (except `[default]`).

### 2.3 The `source_profile` Mechanism

The `production` profile uses `source_profile = default`, which means: "Use the credentials from the `default` profile to assume the role `arn:aws:iam::123456789012:role/ProductionRole`." This lets you assume a role without storing separate credentials.

## 3. Environment Variables

The CLI and SDKs check these env vars first:

```bash
export AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE
export AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
export AWS_SESSION_TOKEN=...            # for temporary credentials
export AWS_DEFAULT_REGION=us-east-1
export AWS_DEFAULT_OUTPUT=json
export AWS_PROFILE=production           # use this profile
export AWS_ENDPOINT_URL=http://localhost:4566   # for LocalStack (SDK v2+)
```

Environment variables override the config files. Useful for:

- CI/CD pipelines (set in the runner's env).
- Docker containers (set in `docker run -e`).
- Switching profiles temporarily.

> [!warning] Environment Variables Can Leak
> Env vars are visible in `/proc/<pid>/environ`, in `ps -e` output (with some commands), and in child processes. Do not set secret keys in env vars on shared systems. Use IAM roles instead.

## 4. IAM Roles

The safest credential is no credential at all. IAM roles let EC2 instances, ECS tasks, Lambda functions, and other AWS resources obtain temporary credentials automatically.

### 4.1 EC2 Instance Profiles

1. Create an IAM role with a trust policy allowing EC2 to assume it.
2. Attach the role to an EC2 instance (an "instance profile").
3. On the instance, Boto3 and the CLI automatically fetch temporary credentials from the EC2 metadata service.

```python
# On an EC2 instance with an attached role, no credentials needed:
import boto3
s3 = boto3.client('s3')   # uses the instance role
s3.list_buckets()
```

```bash
# From the instance, view the role credentials
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

### 4.2 Lambda Execution Roles

Each Lambda function has an execution role. The function's code can call AWS APIs without managing credentials — Boto3 picks up the role automatically.

### 4.3 ECS Task Roles

ECS tasks can have IAM roles. Containers in the task get temporary credentials via the ECS credential provider (set in the environment).

### 4.4 Assuming a Role Manually

```bash
# Assume a role and use the temporary credentials
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/MyRole \
  --role-session-name MySession
```

Returns:

```json
{
  "Credentials": {
    "AccessKeyId": "ASIA...",
    "SecretAccessKey": "...",
    "SessionToken": "...",
    "Expiration": "2024-01-15T12:00:00Z"
  }
}
```

For convenience, use `aws configure` with `source_profile`:

```ini
# ~/.aws/config
[profile production]
role_arn = arn:aws:iam::123456789012:role/ProductionRole
source_profile = default
region = us-east-1
```

Now `aws s3 ls --profile production` automatically assumes the role and uses the temporary credentials.

## 5. Credential Resolution Order

When Boto3 or the CLI needs credentials, they check in this order:

1. **Explicit parameters** — `boto3.client('s3', aws_access_key_id=...)`.
2. **Environment variables** — `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`.
3. **Shared credentials file** — `~/.aws/credentials`.
4. **AWS config file** — `~/.aws/config` (for `role_arn` and `source_profile`).
5. **SSO** — If using IAM Identity Center.
6. **IAM role on EC2/ECS/Lambda** — Via the metadata service.
7. **Container credentials** — ECS tasks.

The first source that provides valid credentials wins. This is why an EC2 instance with a role and no `~/.aws/credentials` file uses the role — the file is checked first but does not exist.

## 6. Multiple Accounts and Profiles

### 6.1 The Common Setup

```ini
# ~/.aws/credentials
[default]
aws_access_key_id = AKIAYOURPERSONALACCT
aws_secret_access_key = yourpersonalsecret

[work-dev]
aws_access_key_id = AKIAWORKDEVACCT
aws_secret_access_key = workdevsecret

[work-prod]
aws_access_key_id = AKIAWORKPRODACCT
aws_secret_access_key = workprodsecret
```

```ini
# ~/.aws/config
[default]
region = us-east-1

[profile work-dev]
region = us-east-1

[profile work-prod]
region = us-west-2

[profile localstack]
region = us-east-1
endpoint_url = http://localhost:4566
```

### 6.2 Using Profiles

```bash
# Default profile
aws s3 ls

# Specific profile
aws s3 ls --profile work-prod

# Set profile for the current shell
export AWS_PROFILE=work-prod
aws s3 ls

# Switch back
export AWS_PROFILE=default
```

### 6.3 Cross-Account Roles

For organizations with many AWS accounts, a common pattern is:

1. One "identity account" with IAM users.
2. Other accounts have roles that trust the identity account.
3. Users assume roles in other accounts via `source_profile`.

```ini
[profile identity]
aws_access_key_id = AKIAIDENTITYACCT
aws_secret_access_key = identitysecret

[profile production]
role_arn = arn:aws:iam::PROD-ACCOUNT:role/ProductionAccess
source_profile = identity
region = us-east-1

[profile staging]
role_arn = arn:aws:iam::STAGING-ACCOUNT:role/StagingAccess
source_profile = identity
region = us-east-1
```

Now `aws s3 ls --profile production` uses the identity credentials to assume the ProductionAccess role, then uses those temporary credentials to list S3.

## 7. Using `aws-vault` for Better Security

[`aws-vault`](https://github.com/99designs/aws-vault) is a third-party tool that stores AWS credentials in your OS keychain (instead of plaintext in `~/.aws/credentials`) and uses STS to issue temporary credentials. Highly recommended for developers with multiple AWS accounts.

```bash
brew install aws-vault   # macOS
# Or: https://github.com/99designs/aws-vault/releases

aws-vault add work-dev   # stores in keychain
aws-vault exec work-dev -- aws s3 ls   # uses temporary credentials
```

## 8. Common Student Mistakes

> [!warning] Mistake 1 — Hardcoding Access Keys in Code
> Use environment variables, profiles, or IAM roles. Never hardcode access keys in source code (especially code committed to Git).

> [!warning] Mistake 2 — Using IAM User Keys on EC2
> Don't put long-lived access keys on EC2 instances. Use IAM instance profiles — they issue temporary credentials automatically.

> [!warning] Mistake 3 — Forgetting That `~/.aws/credentials` Is Plaintext
> The credentials file is plaintext. Anyone with read access can steal your keys. Use `aws-vault` for better security, or set restrictive permissions (`chmod 600 ~/.aws/credentials`).

> [!warning] Mistake 4 — Confusing `~/.aws/config` Profile Prefix
> In `~/.aws/credentials`, profiles are `[name]`. In `~/.aws/config`, they are `[profile name]` (except `[default]`). Forgetting the `profile` prefix in config is a common mistake.

> [!warning] Mistake 5 — Forgetting to Set the Region
> Without a region, many commands fail. Set it in `~/.aws/config` or via `AWS_DEFAULT_REGION`.

> [!warning] Mistake 6 — Using Real AWS Keys with LocalStack
> For LocalStack, you do not need real AWS keys. Use `test` / `test`. Set `endpoint_url=http://localhost:4566`.

> [!warning] Mistake 7 — Not Rotating Keys
> Rotate access keys periodically (every 90 days). Use IAM credential reports to find old keys.

## 9. Summary Checklist

- [ ] Long-lived access keys (AKIA…) never expire; rotate manually.
- [ ] Temporary credentials (ASIA…) expire; preferred for security.
- [ ] `~/.aws/credentials` has profiles; `~/.aws/config` has `[profile name]` sections (except `[default]`).
- [ ] Environment variables override the config files.
- [ ] IAM roles issue temporary credentials via STS; use them on EC2/ECS/Lambda.
- [ ] Credential resolution order: parameters → env vars → credentials file → config file → SSO → IAM role.
- [ ] Use `source_profile` to assume a role using another profile's credentials.
- [ ] Use `aws-vault` for better security (keychain storage, temporary credentials).
- [ ] For LocalStack, use `test`/`test` and set `endpoint_url`.
