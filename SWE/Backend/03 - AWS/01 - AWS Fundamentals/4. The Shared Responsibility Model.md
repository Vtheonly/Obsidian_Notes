# 4. The Shared Responsibility Model

> [!info] Chapter Context
> AWS secures the cloud; you secure what you put in the cloud. This note covers the shared responsibility model in detail, including the exact split for each service category (IaaS, PaaS, SaaS).

Related: [[3. Regions and Availability Zones]] | [[5. AWS Pricing and Cost Management]] | [[07 - Identity and Security/1. IAM Overview]]

---

## 1. The Model in One Sentence

> **AWS is responsible for security OF the cloud; you are responsible for security IN the cloud.**

AWS secures the physical infrastructure. You secure your data, your configurations, your access controls, and your applications.

---

## 2. What AWS Always Manages

Regardless of service model, AWS is responsible for:

- **Physical security** — Data center guards, biometric access, video surveillance.
- **Hardware** — Servers, storage, networking gear.
- **Network infrastructure** — The physical network connecting data centers.
- **Virtualization layer** — Hypervisors (Xen on older EC2, Nitro on newer).
- **Host OS for managed services** — The OS that runs RDS, DynamoDB, Lambda, etc.
- **Patch management for the host OS** — AWS patches the underlying systems.
- **Facilities** — Power, cooling, fire suppression.

You never see or touch any of this. AWS guarantees it is secure.

---

## 3. What You Always Manage

Regardless of service model, you are responsible for:

- **Your data** — What data you store, how you encrypt it, who can access it.
- **IAM** — Users, roles, policies. Who can do what.
- **Client-side encryption** — If you encrypt before uploading, you manage the keys.
- **Network traffic protection** — TLS for data in transit.
- **Application code** — Security of your application logic.

---

## 4. The Split Depends on the Service

### 4.1 IaaS — EC2, EBS, VPC

You manage:
- Guest OS (Linux distribution, version, patches).
- Application code and runtime.
- Security groups, NACLs.
- IAM roles for the instance.
- Data encryption (EBS encryption, S3 SSE).

AWS manages:
- Hypervisor.
- Physical server.
- Network infrastructure.
- Data center.

### 4.2 PaaS — RDS, Elastic Beanstalk, EKS

You manage:
- Database schema, queries.
- Application code.
- IAM access to the service.
- Database-level users and permissions.
- Encryption settings (often one-click).

AWS manages:
- OS installation and patching.
- Database engine installation and patching.
- Backups (configurable).
- Multi-AZ replication.
- Scaling (for some services).

### 4.3 SaaS — DynamoDB, S3, Lambda

You manage:
- Data (what you store).
- IAM policies (who can access).
- Encryption settings.

AWS manages:
- Everything else (OS, runtime, scaling, replication, patching).

### 4.4 Comparison Table

| Layer | IaaS (EC2) | PaaS (RDS) | SaaS (DynamoDB) |
| :--- | :--- | :--- | :--- |
| Physical security | AWS | AWS | AWS |
| Hardware | AWS | AWS | AWS |
| Virtualization | AWS | AWS | AWS |
| Host OS | AWS | AWS | AWS |
| Guest OS | **You** | AWS | AWS |
| Database engine | N/A | AWS | AWS |
| Database configuration | N/A | **You** | AWS |
| Application code | **You** | **You** | **You** |
| Data | **You** | **You** | **You** |
| IAM access | **You** | **You** | **You** |

## 5. Real-World Security Incident Patterns

Most cloud security incidents are caused by customer misconfiguration, not AWS breaches:

### 5.1 Public S3 Buckets

Default S3 bucket settings have changed (buckets are private by default since 2023), but historically many buckets were accidentally made public, exposing sensitive data. **Customer responsibility**: configure bucket policies correctly.

### 5.2 Overly Permissive IAM Policies

`AdministratorAccess` on a role used by an EC2 instance means anyone who compromises that instance has full AWS access. **Customer responsibility**: apply least privilege.

### 5.3 Unpatched EC2 Instances

If you do not apply security updates to your EC2 instance's OS, it is vulnerable to known exploits. **Customer responsibility**: patch your instances (or use managed services where AWS patches).

### 5.4 Hardcoded Secrets in Code

If you commit AWS access keys to GitHub, anyone can use them. **Customer responsibility**: use IAM roles, secrets managers, and `.gitignore`.

### 5.5 Open Security Groups

If your security group allows 0.0.0.0/0 on port 22 (SSH), your instance is being brute-forced right now. **Customer responsibility**: restrict security groups to known IPs.

## 6. Tools AWS Provides to Help You

AWS provides many security tools; using them is your responsibility:

- **IAM Access Analyzer** — Identifies resources shared externally.
- **AWS Config** — Tracks configuration changes; can detect non-compliant resources.
- **AWS Security Hub** — Aggregated security findings.
- **AWS GuardDuty** — Threat detection (unusual API calls, unauthorized deployments).
- **AWS Macie** — Discovers and protects sensitive data in S3.
- **AWS Inspector** — Automated vulnerability scanning of EC2 and ECR.
- **AWS CloudTrail** — Logs every API call for auditing.

---

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Assuming AWS Secures Everything
> AWS secures the infrastructure. You secure your usage. Most incidents are customer misconfiguration.

> [!warning] Mistake 2 — Forgetting to Patch EC2 Instances
> With great power (IaaS) comes great responsibility. You must patch the guest OS. Use AWS Systems Manager Patch Manager to automate.

> [!warning] Mistake 3 — Public S3 Buckets
> S3 buckets are private by default since 2023, but you can still accidentally make them public. Use IAM Access Analyzer to verify.

> [!warning] Mistake 4 — `AdministratorAccess` Everywhere
> Apply least privilege. Use IAM Access Analyzer to identify unused permissions and remove them.

> [!warning] Mistake 5 — No CloudTrail
> Without CloudTrail, you have no audit trail of API calls. Enable it on day one (it is free for management events).

---

## 8. Summary Checklist

- [ ] AWS secures OF the cloud; you secure IN the cloud.
- [ ] AWS always: physical, hardware, virtualization, host OS for managed services.
- [ ] You always: data, IAM, application code, client-side encryption.
- [ ] IaaS (EC2): you manage guest OS and patches.
- [ ] PaaS (RDS): AWS manages OS and engine; you manage schema and data.
- [ ] SaaS (DynamoDB): AWS manages everything; you manage data and IAM.
- [ ] Most cloud security incidents are customer misconfiguration.
- [ ] Use AWS security tools: IAM Access Analyzer, Config, Security Hub, GuardDuty, Macie, Inspector, CloudTrail.
