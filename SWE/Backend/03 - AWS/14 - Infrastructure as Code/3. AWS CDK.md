# 3. AWS CDK

> [!info] Chapter Context
> The AWS CDK (Cloud Development Kit) lets you define infrastructure in real programming languages (TypeScript, Python, Java, C#, Go). It compiles to CloudFormation. This note covers CDK concepts, the standard workflow, and when to choose CDK over Terraform.

Related: [[2. Terraform Fundamentals]] | [[4. CloudFormation]] | [[11 - Serverless Computing/2. Lambda Fundamentals]]

---

## 1. Why CDK

CloudFormation YAML is verbose. CDK lets you use a real programming language with loops, conditionals, functions, and types — much more concise and reusable.

```typescript
// CDK (TypeScript) — 5 lines
const bucket = new s3.Bucket(this, 'MyBucket', {
  bucketName: 'my-bucket',
  versioned: true,
  encryption: s3.BucketEncryption.KMS,
});
```

Equivalent CloudFormation YAML would be 30+ lines.

## 3. The CDK Workflow

```bash
# Initialize a new project (TypeScript)
mkdir my-app && cd my-app
cdk init app --language typescript

# Write your stack in lib/my-app-stack.ts

# Synthesize to CloudFormation template
cdk synth

# Deploy
cdk deploy

# Destroy
cdk destroy
```

## 4. A CDK Stack

```typescript
// lib/my-app-stack.ts
import * as cdk from 'aws-cdk-lib';
import { Construct } from 'constructs';
import * as s3 from 'aws-cdk-lib/aws-s3';
import * as lambda from 'aws-cdk-lib/aws-lambda';
import * as s3notif from 'aws-cdk-lib/aws-s3-notifications';

export class MyAppStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    // S3 bucket
    const bucket = new s3.Bucket(this, 'MyBucket', {
      bucketName: 'my-app-bucket-12345',
      removalPolicy: cdk.RemovalPolicy.DESTROY,
      autoDeleteObjects: true,
    });

    // Lambda function
    const fn = new lambda.Function(this, 'MyFunction', {
      runtime: lambda.Runtime.PYTHON_3_11,
      handler: 'index.handler',
      code: lambda.Code.fromAsset('./lambda'),
      environment: {
        BUCKET_NAME: bucket.bucketName,
      },
    });

    // Trigger Lambda on S3 PUT
    bucket.addEventNotification(
      s3.EventType.OBJECT_CREATED,
      new s3notif.LambdaDestination(fn)
    );

    // Grant Lambda read access to the bucket
    bucket.grantRead(fn);
  }
}
```

## 5. Constructs

A **construct** is the basic building block. Three levels:

### 5.1 L1 (Layer 1) Constructs

Direct mapping to CloudFormation resources. Verbose, like CloudFormation.

```typescript
new s3.CfnBucket(this, 'MyBucket', {
  bucketName: 'my-bucket',
});
```

### 5.2 L2 (Layer 2) Constructs

AWS-curated, with sensible defaults and built-in best practices.

```typescript
new s3.Bucket(this, 'MyBucket', {
  bucketName: 'my-bucket',
});   // encryption, versioning, etc. have defaults
```

### 5.3 L3 (Patterns) Constructs

High-level patterns that combine multiple resources.

```typescript
new aws_s3_deployment.BucketDeployment(this, 'DeployWebsite', {
  sources: [aws_s3_deployment.Source.asset('./website')],
  destinationBucket: bucket,
});

// Or: a full Lambda + API Gateway + logs in one line
new lambda.FunctionUrl(this, 'MyFunctionUrl', {
  function: fn,
  authType: lambda.FunctionUrlAuthType.NONE,
});
```

Use L2 and L3 constructs where possible — they handle best practices automatically.

## 6. The `aws-cdk-lib` Modules

CDK has modules per AWS service:

- `aws-cdk-lib/aws-s3` — S3
- `aws-cdk-lib/aws-lambda` — Lambda
- `aws-cdk-lib/aws-dynamodb` — DynamoDB
- `aws-cdk-lib/aws-ec2` — EC2, VPC
- `aws-cdk-lib/aws-iam` — IAM
- `aws-cdk-lib/aws-sns` — SNS
- `aws-cdk-lib/aws-sqs` — SQS
- `aws-cdk-lib/aws-apigateway` — API Gateway
- ... and many more

## 7. Deployment

```bash
# Synthesize (generates CloudFormation template in cdk.out/)
cdk synth

# Deploy (creates/updates the CloudFormation stack)
cdk deploy

# With parameters
cdk deploy --parameters EnvName=prod

# Diff (what would change)
cdk diff

# Destroy
cdk destroy
```

### 7.1 Multiple Environments

```typescript
// bin/my-app.ts
const app = new cdk.App();

new MyAppStack(app, 'MyApp-Dev', {
  env: { account: '123456789012', region: 'us-east-1' },
  envName: 'dev',
});

new MyAppStack(app, 'MyApp-Prod', {
  env: { account: '123456789012', region: 'us-east-1' },
  envName: 'prod',
});
```

```bash
cdk deploy MyApp-Prod
```

## 8. CDK vs. Terraform

| Feature | CDK | Terraform |
| :--- | :--- | :--- |
| Cloud | AWS only | Multi-cloud |
| Language | TypeScript, Python, etc. | HCL |
| State | CloudFormation (managed) | State file (you manage) |
| Testing | Jest, pytest (real code) | `terraform validate` |
| Abstraction | L1/L2/L3 constructs | Modules |
| Learning curve | Steeper (programming) | Gentler (DSL) |

Use **CDK** if:
- You're AWS-only.
- You have developers who prefer real programming languages.
- You want to test infrastructure code with unit tests.

Use **Terraform** if:
- You're multi-cloud.
- You have ops/sysadmin teams familiar with HCL.
- You want a simpler, declarative DSL.

## 9. Common Student Mistakes

> [!warning] Mistake 1 — Using L1 Constructs When L2 Exists
> L1 constructs are verbose and miss best practices. Use L2/L3 where possible.

> [!warning] Mistake 2 — Forgetting `cdk synth` Before Deploy
> `cdk synth` shows the generated CloudFormation. Always review before deploying.

> [!warning] Mistake 3 — Hardcoding ARNs
> Use CDK's `grant` methods (`bucket.grantRead(fn)`) instead of writing IAM policies by hand.

> [!warning] Mistake 4 — Not Using `RemovalPolicy`
#  By default, CDK resources are retained on stack deletion. Use `RemovalPolicy.DESTROY` for dev environments.

> [!warning] Mistake 5 — Forgetting to Install Dependencies
> CDK projects use npm/pip/etc. Run `npm install` (or equivalent) before `cdk synth`.

> [!warning] Mistake 6 — Treating CDK as Multi-Cloud
#  CDK is AWS-only. For multi-cloud, use Terraform or Pulumi.
