# 3. HTTP and REST APIs

> [!info] Chapter Context
> Almost every AWS service is an HTTP API. When you call `aws s3 ls`, the CLI sends an HTTP request to `s3.amazonaws.com`. Understanding HTTP methods, status codes, headers, and REST conventions is essential for debugging cloud services and designing your own APIs.

Related: [[2. Distributed Systems Basics]] | [[4. DNS and TLS]] | [[02 - AWS CLI and SDKs/1. The AWS CLI]]

## 1. HTTP ‚Äî The Protocol of the Web

HTTP (HyperText Transfer Protocol) is a request-response protocol: a client sends a request, the server sends a response. It is stateless ‚Äî each request is independent, with no inherent memory of previous requests.

### 1.1 Anatomy of an HTTP Request

```http
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGc...
Content-Length: 52

{"name": "Alice", "email": "alice@example.com"}
```

Components:

1. **Method** ‚Äî `POST` (what to do).
2. **Path** ‚Äî `/api/users` (what to do it to).
3. **Version** ‚Äî `HTTP/1.1` (or `HTTP/2`).
4. **Headers** ‚Äî `Host`, `Content-Type`, `Authorization`, etc. (metadata).
5. **Body** ‚Äî `{"name": "Alice", ...}` (the payload).

### 1.2 Anatomy of an HTTP Response

```http
HTTP/1.1 201 Created
Content-Type: application/json
Location: /api/users/123
Content-Length: 38

{"id": 123, "name": "Alice"}
```

Components:

1. **Version** ‚Äî `HTTP/1.1`.
2. **Status code** ‚Äî `201` (what happened).
3. **Headers** ‚Äî `Content-Type`, `Location`, etc.
4. **Body** ‚Äî `{"id": 123, ...}`.

## 2. HTTP Methods

| Method | Purpose | Idempotent? | Safe? |
| :--- | :--- | :--- | :--- |
| `GET` | Retrieve a resource. | Yes | Yes |
| `POST` | Create a new resource. | No | No |
| `PUT` | Replace a resource (full update). | Yes | No |
| `PATCH` | Partially update a resource. | No (usually) | No |
| `DELETE` | Remove a resource. | Yes | No |
| `HEAD` | Like GET, but only headers (no body). | Yes | Yes |
| `OPTIONS` | Ask the server what methods are allowed. | Yes | Yes |

- **Idempotent** ‚Äî Calling the method multiple times has the same effect as calling it once. `DELETE /users/123` once is the same as calling it 100 times (the user is deleted).
- **Safe** ‚Äî The method does not modify server state. GET, HEAD, OPTIONS are safe.

> [!warning] POST vs PUT
> Use POST to create (server assigns the ID). Use PUT to update (client specifies the ID). PUT `/users/123` creates or replaces user 123. POST `/users` creates a new user with a server-assigned ID.

## 3. HTTP Status Codes

Status codes are 3-digit numbers grouped by the first digit:

| Range | Meaning | Examples |
| :--- | :--- | :--- |
| 1xx | Informational | 100 Continue |
| 2xx | Success | 200 OK, 201 Created, 204 No Content |
| 3xx | Redirection | 301 Moved Permanently, 302 Found, 304 Not Modified |
| 4xx | Client error | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests |
| 5xx | Server error | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout |

### 3.1 The Most Common Codes

- **200 OK** ‚Äî Request succeeded.
- **201 Created** ‚Äî Resource was created (POST that succeeded).
- **204 No Content** ‚Äî Request succeeded, no body to return (DELETE that succeeded).
- **301 Moved Permanently** ‚Äî Resource has a new URL; cache the new location.
- **302 Found** ‚Äî Temporary redirect.
- **304 Not Modified** ‚Äî Cached version is still valid (used with `If-Modified-Since`).
- **400 Bad Request** ‚Äî Malformed request (e.g., invalid JSON).
- **401 Unauthorized** ‚Äî Authentication required or failed.
- **403 Forbidden** ‚Äî Authenticated, but not allowed.
- **404 Not Found** ‚Äî Resource does not exist.
- **409 Conflict** ‚Äî Request conflicts with current state (e.g., duplicate email).
- **429 Too Many Requests** ‚Äî Rate limit exceeded.
- **500 Internal Server Error** ‚Äî Generic server error.
- **502 Bad Gateway** ‚Äî A proxy/upstream returned an invalid response.
- **503 Service Unavailable** ‚Äî Server is down or overloaded.
- **504 Gateway Timeout** ‚Äî Upstream did not respond in time.

> [!tip] 401 vs 403
> **401 Unauthorized** = "Who are you?" (no authentication, or invalid credentials).
> **403 Forbidden** = "I know who you are, but you cannot do that." (authenticated, but not authorized).
>
> AWS API uses these consistently. `aws s3 ls` returning 403 means your IAM role does not have permission, not that your access key is wrong.

## 4. HTTP Headers

Headers are key-value metadata sent with requests and responses. Common ones:

### 4.1 Request Headers

- `Host` ‚Äî The hostname (required in HTTP/1.1).
- `User-Agent` ‚Äî Identifies the client (e.g., `aws-cli/2.15.0`).
- `Accept` ‚Äî What content types the client can handle.
- `Content-Type` ‚Äî What the request body is (e.g., `application/json`).
- `Authorization` ‚Äî Credentials (e.g., `Bearer <token>`, `Basic <base64>`).
- `Cookie` ‚Äî Sent cookies.
- `If-None-Match` / `If-Modified-Since` ‚Äî Conditional requests (caching).

### 4.2 Response Headers

- `Content-Type` ‚Äî What the response body is.
- `Content-Length` ‚Äî Body size in bytes.
- `Cache-Control` ‚Äî Caching directives (e.g., `max-age=3600`).
- `ETag` ‚Äî A hash of the content (for caching).
- `Location` ‚Äî Used with 3xx redirects and 201 Created.
- `Set-Cookie` ‚Äî Set a cookie on the client.
- `Access-Control-Allow-Origin` ‚Äî CORS (cross-origin resource sharing).

### 4.3 The `Content-Type` Header

Tells the recipient how to interpret the body. Common types:

- `application/json` ‚Äî JSON data.
- `application/xml` ‚Äî XML data.
- `text/html` ‚Äî HTML page.
- `text/plain` ‚Äî Plain text.
- `multipart/form-data` ‚Äî File uploads.
- `application/x-www-form-urlencoded` ‚Äî Form data.

AWS APIs typically use `application/x-amz-json-1.0` or `application/x-amz-json-1.1` for JSON, with custom headers like `X-Amz-Target` to specify the operation.

## 5. REST ‚Äî Representational State Transfer

REST is an architectural style for HTTP APIs, defined by Roy Fielding in 2000. It is a set of conventions, not a protocol. Most public APIs (Twitter, GitHub, Stripe) are RESTful.

### 5.1 REST Principles

1. **Resource-based URLs** ‚Äî URLs identify resources, not actions. `/users/123` (a user), not `/getUser?id=123`.
2. **HTTP methods express intent** ‚Äî GET to read, POST to create, PUT to update, DELETE to remove.
3. **Stateless** ‚Äî Each request contains all the information needed; the server does not store session state.
4. **Layered** ‚Äî The client does not know if it is talking to the end server or a proxy.
5. **Cacheable** ‚Äî Responses should be cacheable when possible (use `Cache-Control`, `ETag`).

### 5.2 A RESTful URL Structure

| Method | URL | Action |
| :--- | :--- | :--- |
| `GET` | `/users` | List users |
| `POST` | `/users` | Create a user |
| `GET` | `/users/123` | Get user 123 |
| `PUT` | `/users/123` | Replace user 123 |
| `PATCH` | `/users/123` | Update user 123 partially |
| `DELETE` | `/users/123` | Delete user 123 |
| `GET` | `/users/123/orders` | List user 123's orders |

### 5.3 REST vs. Other API Styles

- **REST** ‚Äî Resource-oriented, HTTP-based, JSON payloads. Most common.
- **GraphQL** ‚Äî Client specifies exactly what fields it wants; one endpoint (`/graphql`). Solves over-fetching/under-fetching.
- **gRPC** ‚Äî Binary protocol over HTTP/2, uses Protocol Buffers. Faster than REST, used for internal microservice communication.
- **SOAP** ‚Äî XML-based, verbose, used in enterprise/legacy systems.

For external APIs, REST is the default. For internal microservice communication, gRPC is increasingly common.

## 6. AWS APIs Are HTTP

Every AWS service exposes an HTTP API. When you run `aws s3 ls`, the CLI constructs an HTTP request:

```http
GET / HTTP/1.1
Host: my-bucket.s3.amazonaws.com
X-Amz-Date: 20240115T103000Z
X-Amz-Content-SHA256: e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855
Authorization: AWS4-HMAC-SHA256 Credential=AKIAIOSFODNN7EXAMPLE/20240115/us-east-1/s3/aws4_request, SignedHeaders=host;x-amz-content-sha256;x-amz-date, Signature=...
```

The CLI handles the signing (AWS Signature Version 4) for you. If you call AWS APIs from your own code (via SDK), the SDK signs each request. See [[02 - AWS CLI and SDKs/2. Boto3 and AWS SDKs]].

### 6.1 AWS API Conventions

- Most AWS APIs are POST-based, even for read operations. The action is specified in a header (`X-Amz-Target`) or in the URL.
- Requests are signed with AWS Signature V4 (HMAC-SHA256 of the request, using your access key).
- Responses are JSON (for most modern services) or XML (for S3 and some legacy services).
- Errors are returned as HTTP 4xx or 5xx with a JSON error body.

## 7. Authentication and Authorization

### 7.1 Common API Authentication Methods

- **API keys** ‚Äî A string sent in a header (`Authorization: Bearer <key>` or `X-API-Key: <key>`). Simple but easy to leak.
- **Bearer tokens (JWT)** ‚Äî A signed JSON Web Token. Self-contained (carries claims). Common in OAuth 2.0.
- **OAuth 2.0** ‚Äî A framework for delegated authorization. User grants a third-party app access to their data without sharing their password.
- **mTLS (mutual TLS)** ‚Äî Both client and server present certificates. Used in high-security environments.
- **AWS Signature V4** ‚Äî Request signing with HMAC-SHA256. Each request is signed with your access key.

### 7.2 Authorization vs. Authentication

- **Authentication** = "Who are you?" (verifying identity).
- **Authorization** = "What can you do?" (verifying permissions).

A user can be authenticated (we know who they are) but not authorized (they cannot perform the action). AWS separates these: IAM handles authentication (who you are) and authorization (what you can do) in one system, but the concepts are distinct.

## 8. Tools for Working with HTTP

- **`curl`** ‚Äî The universal CLI HTTP client. See [[04 - Linux/06 - Networking/1. Networking Fundamentals]].
- **Postman / Insomnia** ‚Äî GUI HTTP clients with request collections.
- **HTTPie** ‚Äî A friendlier `curl` alternative (`http GET https://api.example.com/users`).
- **`httpstat`** ‚Äî Shows timing breakdown of an HTTP request (DNS, connect, TLS, etc.).
- **Browser DevTools** ‚Äî F12 ‚Üí Network tab. Inspect requests/responses from a web page.

Example with `curl`:

```bash
# GET request
curl https://api.example.com/users

# GET with headers
curl -H "Authorization: Bearer TOKEN" https://api.example.com/users

# POST with JSON body
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "Alice", "email": "alice@example.com"}'

# Verbose (shows request and response headers)
curl -v https://api.example.com/users

# Just headers (HEAD request)
curl -I https://api.example.com/users
```

## 9. Common Student Mistakes

> [!warning] Mistake 1 ‚Äî Confusing 401 and 403
> 401 = not authenticated (no/invalid credentials). 403 = authenticated but not authorized. AWS APIs follow this strictly.

> [!warning] Mistake 2 ‚Äî Forgetting `Content-Type`
> If you POST JSON without `Content-Type: application/json`, the server may not parse it. Always set the Content-Type header.

> [!warning] Mistake 3 ‚Äî Using GET for State Changes
> GET should be safe (no side effects). Use POST/PUT/PATCH/DELETE for modifications. Browsers prefetch GET requests; if GET modifies state, you have a bug (and a CSRF vulnerability).

> [!warning] Mistake 4 ‚Äî Ignoring Idempotency
> If a client retries a POST (e.g., network timeout), it may create duplicate resources. Design your APIs to be idempotent (e.g., use idempotency keys for payment APIs).

> [!warning] Mistake 5 ‚Äî Returning 500 for Client Errors
> 500 means "the server broke." For "you sent bad data," return 400. For "you are not allowed," return 403. Reserve 500 for actual server bugs.

## 10. Summary Checklist

- [ ] HTTP is request-response, stateless, text-based (in HTTP/1.1).
- [ ] Methods: GET (read), POST (create), PUT (replace), PATCH (partial update), DELETE (remove).
- [ ] Status codes: 2xx success, 3xx redirect, 4xx client error, 5xx server error.
- [ ] Common: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 500 Internal Server Error.
- [ ] Headers carry metadata: Content-Type, Authorization, Cache-Control, ETag, etc.
- [ ] REST principles: resource-based URLs, methods express intent, stateless, cacheable.
- [ ] AWS APIs are HTTP-based, signed with AWS Signature V4.
- [ ] Authentication = who you are; Authorization = what you can do.
