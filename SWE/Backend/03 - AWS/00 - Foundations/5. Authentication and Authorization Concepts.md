# 5. Authentication and Authorization Concepts

> [!info] Chapter Context
> Before diving into AWS IAM (which has its own chapter), this note covers the general concepts of authentication and authorization that apply across all systems: passwords, tokens, OAuth 2.0, OIDC, API keys, and the principle of least privilege.

## 1. Authentication vs. Authorization

These two terms are often confused but are fundamentally different:

- **Authentication (AuthN)** — "Who are you?" Verifying identity.
- **Authorization (AuthZ)** — "What can you do?" Verifying permissions.

You can be authenticated without being authorized (you log in, but cannot access a resource). You cannot be authorized without being authenticated (the system needs to know who you are before deciding what you can do).

Example: When you SSH to an EC2 instance:

1. You present your SSH key. The server verifies it. → **Authentication** ("This is Alice").
2. The server checks `~/.ssh/authorized_keys` and `/etc/sudoers` to decide what Alice can do. → **Authorization** ("Alice can run sudo").

In AWS:

1. You provide an access key. The API verifies the signature. → **Authentication**.
2. IAM evaluates the attached policies to decide if you can call `s3:GetObject`. → **Authorization**.

## 2. Authentication Methods

### 2.1 Passwords

The oldest and most common method. The user provides a password; the server hashes it and compares to the stored hash.

**Best practices:**

- Hash passwords with a slow function: bcrypt, scrypt, Argon2. Never MD5 or SHA-1.
- Use a unique salt per password (bcrypt does this automatically).
- Enforce minimum length (12+ characters), not complexity rules (NIST 800-63B).
- Rate-limit login attempts.
- Support multi-factor authentication.

**Never store passwords in plaintext.** Never store them with reversible encryption. Only store the hash.

### 2.2 Multi-Factor Authentication (MFA)

Requires two of:

- Something you **know** (password).
- Something you **have** (phone, hardware token).
- Something you **are** (biometric: fingerprint, face).

Common MFA methods:

- **TOTP** (Time-based One-Time Password) — Google Authenticator, Authy. A 6-digit code that changes every 30 seconds, derived from a shared secret and the current time.
- **SMS** — A code texted to your phone. Less secure (SIM swapping attacks).
- **Hardware tokens** — YubiKey, Google Titan. The most secure; resistant to phishing.
- **Push notifications** — An app prompts you to approve the login.

AWS supports MFA for IAM users and root accounts. Always enable MFA on the root account.

### 2.3 API Keys

A long string that identifies the caller. Sent in a header (`Authorization: Bearer <key>` or `X-API-Key: <key>`).

- Simple to implement.
- Easy to leak (committed to Git, hardcoded in client-side code).
- No built-in expiration (must be rotated manually).
- No built-in scope (the key has full access unless the API enforces scopes).

For server-to-server communication, API keys are acceptable if treated like passwords (rotated, scoped, never committed). For client-side apps, they are dangerous.

### 2.4 Tokens (JWT)

A **JSON Web Token (JWT)** is a self-contained token with three base64-encoded parts:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsaWNlIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
\_________________________________./\_________________________________________________________./\______________________________________________________/
              Header                                       Payload                                              Signature
```

- **Header** — Algorithm and token type (`{"alg": "HS256", "typ": "JWT"}`).
- **Payload** — Claims (user ID, roles, expiration, etc.).
- **Signature** — HMAC of the header + payload, using a secret key.

The server signs the JWT with its secret. Clients present the JWT; the server verifies the signature. JWTs are stateless — the server does not need to look up the token in a database.

**Caveats:**

- JWTs cannot be revoked (they are valid until they expire). Keep expirations short (15 minutes).
- The payload is base64-encoded, not encrypted. Do not put secrets in a JWT (unless you use JWE — encrypted JWTs).
- The `alg: none` attack: never accept JWTs with `alg: none`. Always verify the signature.

### 2.5 OAuth 2.0

OAuth 2.0 is a framework for **delegated authorization**: a user grants a third-party app access to their data on another service, without sharing their password.

Example: "Log in with Google" — a third-party app asks Google for access to your email and profile. You authenticate with Google (not the third-party app) and authorize the app to access specific data.

OAuth 2.0 roles:

- **Resource Owner** — The user.
- **Client** — The third-party app.
- **Authorization Server** — Issues tokens (e.g., Google).
- **Resource Server** — Hosts the user's data (e.g., Gmail).

The flow:

1. The client redirects the user to the authorization server.
2. The user logs in and grants the client permission.
3. The authorization server redirects back to the client with an authorization code.
4. The client exchanges the code for an access token.
5. The client uses the access token to access the resource server.

### 2.6 OpenID Connect (OIDC)

OIDC is an identity layer on top of OAuth 2.0. While OAuth 2.0 is for authorization (what can the app do), OIDC is for authentication (who is the user). It adds an **ID token** (a JWT) that contains the user's identity claims.

AWS Cognito, Auth0, and Okta support OIDC. AWS IAM Identity Center (formerly SSO) supports OIDC for federated login.

### 2.7 SAML

SAML (Security Assertion Markup Language) is an XML-based protocol for single sign-on (SSO), predating OIDC. Used in enterprise environments. AWS IAM Identity Center supports SAML for federated login with corporate identity providers (Okta, Azure AD, etc.).

## 3. The Principle of Least Privilege

Every user, role, and service should have the **minimum permissions needed to do its job**, and no more.

### 3.1 Why Least Privilege Matters

- **Reduces blast radius** — If an account is compromised, the attacker can only do what that account can do.
- **Reduces accidents** — A developer cannot accidentally delete production data if they do not have delete permissions.
- **Simplifies auditing** — Fewer permissions to review.
- **Compliance** — Most security frameworks (SOC 2, ISO 27001, PCI DSS) require it.

### 3.2 How to Apply It

- Start with **no permissions** and add only what is needed.
- Prefer **specific resources** over wildcards (`arn:aws:s3:::my-bucket/*` instead of `*`).
- Prefer **specific actions** over wildcards (`s3:GetObject` instead of `s3:*`).
- Use **roles** instead of long-lived access keys.
- **Audit** regularly with IAM Access Analyzer — it identifies unused permissions.
- Use **permission boundaries** to cap what a role can do, even if a policy grants more.

### 3.3 Common Anti-Patterns

- `AdministratorAccess` for everyone.
- `*` for resources.
- `s3:*` for actions.
- Long-lived access keys on EC2 instances (use instance profiles instead).
- One IAM user shared by a team (no individual accountability).

## 4. Secrets Management

Secrets (passwords, API keys, certificates) must be:

- **Stored securely** — Encrypted at rest, access-controlled.
- **Transmitted securely** — Over TLS.
- **Rotated regularly** — Reduce the impact of a leak.
- **Auditable** — Track who accessed what when.

### 4.1 AWS Secrets Management Services

- **AWS Secrets Manager** — Stores secrets (database credentials, API keys), supports automatic rotation for RDS, integration with Lambda.
- **AWS Systems Manager Parameter Store** — Stores configuration data and secrets. Cheaper than Secrets Manager for non-secret config.
- **AWS KMS (Key Management Service)** — Manages encryption keys. Used by other services (S3, EBS, Secrets Manager) to encrypt data.

### 4.2 Anti-Patterns

- Hardcoding secrets in source code.
- Committing secrets to Git (use `.gitignore` and tools like `git-secrets` or `trufflehog`).
- Storing secrets in environment variables in plaintext config files.
- Sharing secrets via Slack/email.
- Using the same secret across environments (dev, staging, prod).

## 5. Session Management

After authentication, the server must remember the user across requests. Options:

### 5.1 Server-Side Sessions

The server stores session data in memory or a database. The client gets a session ID (in a cookie). On each request, the server looks up the session by ID.

- **Pros:** Easy to revoke (delete the session). Session data is hidden from the client.
- **Cons:** Server must store sessions; does not scale horizontally without a shared session store (Redis, DynamoDB).

### 5.2 Stateless Tokens (JWT)

The server signs a JWT containing the user's identity and claims. The client sends the JWT on each request. The server verifies the signature.

- **Pros:** Stateless; scales horizontally. No shared session store needed.
- **Cons:** Cannot revoke until expiration. Keep expirations short and use refresh tokens.

### 5.3 Refresh Tokens

For apps using JWTs, the access token expires quickly (15 minutes). When it expires, the client uses a **refresh token** (longer-lived, stored server-side, revocable) to get a new access token. This balances statelessness with revocation.

## 6. Federation and Single Sign-On (SSO)

In an enterprise, users should log in once and access all applications. This is **Single Sign-On (SSO)**.

### 6.1 How SSO Works

1. User logs in to an **Identity Provider (IdP)** — e.g., Okta, Azure AD, Google Workspace.
2. When the user accesses an application (Service Provider, SP), the SP redirects to the IdP.
3. The IdP confirms the user is logged in and sends an assertion (SAML) or token (OIDC) to the SP.
4. The SP trusts the assertion and logs the user in.

### 6.2 AWS IAM Identity Center

AWS IAM Identity Center (formerly AWS SSO) is AWS's SSO solution. It supports:

- SAML and OIDC for federated login.
- Integration with external IdPs (Okta, Azure AD, Google Workspace).
- Centralized access to multiple AWS accounts.
- Integration with on-premises Active Directory via AWS Directory Service.

For organizations with multiple AWS accounts, IAM Identity Center is the recommended way to manage access.

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Confusing AuthN and AuthZ
> Authentication = who you are. Authorization = what you can do. They are different problems with different solutions.

> [!warning] Mistake 2 — Storing Passwords in Plaintext or with Weak Hashing
> Use bcrypt, scrypt, or Argon2. Never MD5 or SHA-1. Never store plaintext.

> [!warning] Mistake 3 — Not Using MFA
> MFA dramatically reduces the impact of password leaks. Enable it on every account that supports it, especially root accounts and admin accounts.

> [!warning] Mistake 4 — Hardcoding Secrets
> Never commit secrets to Git. Use environment variables, secret managers, or cloud-native secret services.

> [!warning] Mistake 5 — Over-Permissive Policies
> `AdministratorAccess` for everyone violates least privilege. Grant only the permissions needed.

> [!warning] Mistake 6 — Long-Lived Access Keys
> Long-lived access keys leak. Use IAM roles (assumed via STS) for temporary credentials whenever possible.

> [!warning] Mistake 7 — JWTs with Long Expirations
> JWTs cannot be revoked. Keep expirations short (15 minutes) and use refresh tokens.

## 8. Summary Checklist

- [ ] Authentication = "Who are you?" Authorization = "What can you do?"
- [ ] Passwords must be hashed with bcrypt/scrypt/Argon2, never stored plaintext.
- [ ] MFA: TOTP (Google Authenticator), SMS (less secure), hardware tokens (most secure).
- [ ] API keys: simple but easy to leak; rotate and scope them.
- [ ] JWTs: self-contained, stateless, short-lived. Use refresh tokens for longer sessions.
- [ ] OAuth 2.0: delegated authorization (third-party app accesses user's data).
- [ ] OIDC: identity layer on OAuth 2.0 (used for "Log in with X").
- [ ] SAML: XML-based SSO for enterprises.
- [ ] Principle of least privilege: minimum permissions needed, no more.
- [ ] Secrets: store in a secrets manager, rotate regularly, never hardcode.
- [ ] AWS Secrets Manager, Parameter Store, and KMS for secrets in AWS.
- [ ] IAM Identity Center for SSO across multiple AWS accounts.
