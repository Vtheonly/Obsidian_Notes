# 3. S3 Permissions and Security

> [!info] Chapter Context
> S3 security is one of the most common sources of cloud incidents — public buckets, overly permissive policies, leaked access keys. This note covers bucket policies, ACLs (deprecated but still exist), Block Public Access, encryption, and access logging.

## 1. The Three Access Control Mechanisms

S3 has three ways to control access, evaluated in this order:

1. **IAM policies** — Attached to the user/role making the request. "Who can do what."
2. **Bucket policies** — Attached to the bucket. "What can be done with this bucket, by whom."
3. **ACLs (Access Control Lists)** — Legacy, per-object permissions. Mostly deprecated. Use bucket policies instead.

### 1.1 When to Use Which

- **IAM policies** — When access is for users/roles in your account.
- **Bucket policies** — When access is for cross-account, public, or anonymous. When you want to enforce rules at the bucket level (e.g., "all objects must be encrypted").
- **ACLs** — Avoid. Use bucket policies.

### 1.2 Evaluation Order

For an S3 request, IAM evaluates:

1. Is the action explicitly denied anywhere? → Deny.
2. Is the action allowed by IAM policy AND bucket policy? → Allow.
3. Otherwise → Deny (default).

> [!note] Bucket Policy Can Grant Access Without IAM
> If a bucket policy allows an action, but the IAM policy does not, the action is allowed (within the same account). For cross-account, BOTH the bucket policy AND the caller's IAM policy must allow.

## 2. Bucket Policies

### 2.1 Allow Public Read

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicReadGetObject",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::my-bucket/*"
  }]
}
```

> [!warning] Public Buckets Are Dangerous
> Only use this for genuinely public content (e.g., a website's static assets). Even then, consider using CloudFront instead of direct S3 access.

### 2.2 Allow a Specific AWS Account

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"AWS": "arn:aws:iam::123456789012:root"},
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-bucket/*"
  }]
}
```

### 2.3 Enforce Encryption (Deny Unencrypted Puts)

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyUnEncryptedObjectUploads",
    "Effect": "Deny",
    "Principal": "*",
    "Action": "s3:PutObject",
    "Resource": "arn:aws:s3:::my-bucket/*",
    "Condition": {
      "StringNotEquals": {
        "s3:x-amz-server-side-encryption": "AES256"
      }
    }
  }]
}
```

### 2.4 Enforce HTTPS Only

```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {"Bool": {"aws:SecureTransport": false}}
}
```

### 2.5 Apply a Bucket Policy

```bash
aws s3api put-bucket-policy --bucket my-bucket --policy file://policy.json
```

## 3. Block Public Access

S3 Block Public Access (enabled by default since 2023) prevents:

- New ACLs that grant public access.
- New bucket policies that grant public access.
- Existing public ACLs from being honored.
- Existing public bucket policies from being honored.

```bash
# Enable on a bucket
aws s3api put-public-access-block --bucket my-bucket \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true

# Enable at the account level (affects all buckets)
aws s3control put-public-access-block --account-id 123456789012 \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

> [!tip] Keep Block Public Access Enabled
> Unless you specifically need public access (e.g., a static website — but use CloudFront instead), keep all four options enabled.

## 4. Encryption

S3 supports three encryption modes:

### 4.1 SSE-S3 (Server-Side Encryption with S3-Managed Keys)

- AWS manages the keys.
- Free (no additional cost).
- Default since January 2023 (every bucket has encryption enabled by default).
- Suitable for most use cases.

### 4.2 SSE-KMS (Server-Side Encryption with KMS-Managed Keys)

- You manage the keys in KMS.
- $0.03 per 10,000 KMS API calls + $1/month per key.
- Use when you need: audit trail of key usage, ability to disable keys, cross-account access control.

### 4.3 SSE-C (Server-Side Encryption with Customer-Provided Keys)

- You provide the encryption key with each request.
- S3 uses it to encrypt and discards it.
- You must manage keys yourself (and never lose them).
- Use when you must control keys entirely.

### 4.4 Client-Side Encryption

- You encrypt before uploading.
- S3 stores ciphertext only.
- Use when you cannot trust S3 with plaintext (rare).

### 4.5 Default Encryption

```bash
aws s3api put-bucket-encryption --bucket my-bucket --server-side-encryption-configuration \
  '{"Rules": [{"ApplyServerSideEncryptionByDefault": {"SSEAlgorithm": "AES256"}}]}'
```

For SSE-KMS:

```json
{
  "Rules": [{
    "ApplyServerSideEncryptionByDefault": {
      "SSEAlgorithm": "aws:kms",
      "KMSMasterKeyID": "arn:aws:kms:us-east-1:123456789012:key/abc-123"
    }
  }]
}
```

## 5. Access Logging

Enable server access logging to track every request to the bucket:

```bash
aws s3api put-bucket-logging --bucket my-bucket --bucket-logging-status file://logging.json
```

```json
// logging.json
{
  "LoggingEnabled": {
    "TargetBucket": "log-bucket",
    "TargetPrefix": "s3-access/my-bucket/"
  }
}
```

Logs are written to the target bucket with details: requester, bucket name, time, request type, key, response code, bytes transferred.

For richer analytics, use **S3 Server Access Logs** with Athena to query them with SQL.

## 6. Object Lock (WORM)

Object Lock prevents objects from being deleted or overwritten for a retention period. Used for compliance (SEC 17a-4, FINRA, HIPAA).

Two modes:

- **Governance mode** — Most users cannot delete; privileged users can.
- **Compliance mode** — No one (including root) can delete until the retention period expires.

```bash
# Enable Object Lock when creating the bucket
aws s3api create-bucket --bucket my-bucket --object-lock-enabled-for-bucket

# Apply a retention period to an object
aws s3api put-object-retention --bucket my-bucket --key important.pdf \
  --retention '{"Mode": "COMPLIANCE", "RetainUntilDate": "2025-12-31T00:00:00Z"}'
```

## 7. Auditing with IAM Access Analyzer

IAM Access Analyzer scans your S3 buckets and reports:

- Buckets accessible by external accounts.
- Buckets accessible publicly.

```bash
aws accessanalyzer list-analyzers
aws accessanalyzer start-resource-scan --analyzer-arn arn:... --resource-arn arn:aws:s3:::my-bucket
```

Review findings weekly.

## 8. Common Student Mistakes

> [!warning] Mistake 1 — Public Bucket Without Realizing
> A bucket policy granting `Principal: "*"` makes the bucket public. Use IAM Access Analyzer to find these.

> [!warning] Mistake 2 — Forgetting Cross-Account Requires Both Sides
> For cross-account access, the bucket policy must allow AND the caller's IAM policy must allow.

> [!warning] Mistake 3 — Using ACLs
> ACLs are legacy and hard to manage. Use bucket policies.

> [!warning] Mistake 4 — Disabling Block Public Access Without Reason
> Keep Block Public Access enabled. For public websites, use CloudFront (which can use OAC to access the private bucket).

> [!warning] Mistake 5 — Forgetting to Enable Access Logging
> Without access logs, you cannot audit who accessed what. Enable logging for sensitive buckets.

> [!warning] Mistake 6 — Not Enforcing Encryption
> Even though SSE-S3 is default, use a bucket policy to DENY unencrypted PUTs to enforce it.

## 9. Summary Checklist

- [ ] Three access mechanisms: IAM policies, bucket policies, ACLs (avoid ACLs).
- [ ] Bucket policy can grant access without IAM (same account); both must allow (cross-account).
- [ ] Block Public Access: enabled by default; keep it that way.
- [ ] Encryption: SSE-S3 (free, default), SSE-KMS (audit trail), SSE-C (customer keys), client-side.
- [ ] Object Lock for WORM compliance.
- [ ] Enable access logging for sensitive buckets.
- [ ] Use IAM Access Analyzer to audit external access.
