# 4. STS and Temporary Credentials

> [!info] Chapter Context
> AWS STS (Security Token Service) issues temporary credentials. This is how roles work, how EC2 instances get credentials, and how cross-account access happens. This note covers STS APIs, credential lifetimes, and common patterns.

Related: [[3. IAM Policies]] | [[2. IAM Users, Groups, and Roles]] | [[5. Secrets Management in AWS]]

---

## 1. Why Temporary Credentials

Long-lived access keys never expire. If leaked, they are valid until manually rotated. Temporary credentials:

- **Expire automatically** — Typically 15 minutes to 12 hours.
- **Are scoped** — Limited to a specific role's permissions, optionally further scoped.
- **Are issued by STS** — Via `AssumeRole`, `GetSessionToken`, etc.

Temporary credentials include three values:

- `AccessKeyId` (starts with `ASIA`).
- `SecretAccessKey`.
- `SessionToken` (required; long-lived keys do not have one).

The SessionToken is what tells AWS "this is a temporary credential, not a long-lived one."

## 2. The STS APIs

### 2.1 `AssumeRole`

The most common STS API. Returns temporary credentials for a role.

```bash
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/MyRole \
  --role-session-name MySession \
  --duration-seconds 3600
```

Returns:

```json
{
  "Credentials": {
    "AccessKeyId": "ASIAIOSFODNN7EXAMPLE",
    "SecretAccessKey": "...",
    "SessionToken": "...",
    "Expiration": "2024-01-15T11:00:00Z"
  },
  "AssumedRoleUser": {
    "AssumedRoleId": "AROA123EXAMPLE:MySession",
    "Arn": "arn:aws:sts::123456789012:assumed-role/MyRole/MySession"
  }
}
```

### 2.2 `GetSessionToken`

Returns temporary credentials for the calling identity (yourself). Useful when you want to require MFA for sensitive operations.

```bash
aws sts get-session-token --duration-seconds 3600 --serial-number arn:aws:iam::123456789012:mfa/alice --token-code 123456
```

### 2.3 `GetCallerIdentity`

Returns who you currently are. Invaluable for debugging "which credentials am I using?"

```bash
aws sts get-caller-identity
# {
#   "UserId": "AIDAIOSFODNN7EXAMPLE",
#   "Account": "123456789012",
#   "Arn": "arn:aws:iam::123456789012:user/alice"
# }
```

### 2.4 `AssumeRoleWithWebIdentity`

For federated login via OIDC (Google, Facebook, Cognito, GitHub Actions OIDC).

### 2.5 `AssumeRoleWithSAML`

For federated login via SAML (Okta, Azure AD).

## 3. How EC2 Instances Get Credentials

When you attach an IAM role to an EC2 instance (via an instance profile), the instance gets credentials automatically:

1. The EC2 metadata service (`169.254.169.254`) is available on the instance.
2. AWS rotates temporary credentials for the role and publishes them to the metadata service.
3. The AWS SDK (Boto3, etc.) fetches credentials from the metadata service when needed.
4. Credentials are cached and refreshed before they expire.

```bash
# From an EC2 instance with a role:
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
# Returns the role name

curl http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRole
# Returns the temporary credentials JSON
```

```python
# Boto3 on EC2 — no credentials needed in code:
import boto3
s3 = boto3.client('s3')   # uses the instance role automatically
```

### 3.1 IMDSv2

The original metadata service (IMDSv1) is vulnerable to SSRF attacks. IMDSv2 requires a token (PUT request first), making SSRF harder. Always use IMDSv2 on new instances.

## 4. How Lambda Gets Credentials

Each Lambda function has an execution role. When Lambda invokes your function:

1. Lambda calls STS to assume the execution role.
2. Lambda passes the temporary credentials to your function via environment variables (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`).
3. The AWS SDK in your function uses these credentials automatically.

```python
# In a Lambda function — no credentials needed:
import boto3
s3 = boto3.client('s3')   # uses the execution role
```

## 5. Cross-Account Access with AssumeRole

A common pattern: account A needs to access resources in account B.

### 5.1 In Account B (the trusting account)

Create a role with:
- Trust policy allowing account A to assume it.
- Permissions policy granting the needed access.

```json
// trust-policy.json (in account B)
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"AWS": "arn:aws:iam::ACCOUNT-A:root"},
    "Action": "sts:AssumeRole"
  }]
}
```

### 5.2 In Account A (the trusted account)

Attach a policy to a user/role in account A allowing them to assume the role in account B:

```json
{
  "Effect": "Allow",
  "Action": "sts:AssumeRole",
  "Resource": "arn:aws:iam::ACCOUNT-B:role/CrossAccountRole"
}
```

### 5.3 Using It

```bash
# In account A
aws sts assume-role --role-arn arn:aws:iam::ACCOUNT-B:role/CrossAccountRole --role-session-name MySession
```

Or via a profile:

```ini
# ~/.aws/config
[profile account-b]
role_arn = arn:aws:iam::ACCOUNT-B:role/CrossAccountRole
source_profile = default
```

```bash
aws s3 ls --profile account-b
```

## 6. OIDC Federation (GitHub Actions)

GitHub Actions can use OIDC to assume an IAM role in your AWS account — no long-lived access keys needed.

### 6.1 Set Up the Trust

In AWS:

1. Create an OIDC provider for GitHub (`https://token.actions.githubusercontent.com`).
2. Create a role with a trust policy allowing GitHub Actions to assume it.

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::ACCOUNT:oidc-provider/token.actions.githubusercontent.com"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
      },
      "StringLike": {
        "token.actions.githubusercontent.com:sub": "repo:my-org/my-repo:*"
      }
    }
  }]
}
```

### 6.2 In the GitHub Actions Workflow

```yaml
permissions:
  id-token: write   # required for OIDC
  contents: read

steps:
  - uses: aws-actions/configure-aws-credentials@v4
    with:
      role-to-assume: arn:aws:iam::123456789012:role/github-actions
      aws-region: us-east-1

  - run: aws s3 ls
```

No AWS access keys stored in GitHub secrets!

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Forgetting the SessionToken
> Temporary credentials require three values: AccessKeyId, SecretAccessKey, AND SessionToken. Without the SessionToken, the credentials are rejected.

> [!warning] Mistake 2 — Using Long-Lived Keys on EC2
> EC2 instances should use instance profiles (roles). No access keys needed.

> [!warning] Mistake 3 — Not Using `aws sts get-caller-identity` for Debugging
> If you're unsure which credentials you're using, run `aws sts get-caller-identity`. It shows the ARN of the caller.

> [!warning] Mistake 4 — Forgetting Cross-Account Requires Both Sides
> Account B's trust policy AND account A's user policy must both allow the assume.

> [!warning] Mistake 5 — Using IMDSv1 on EC2
> IMDSv1 is vulnerable to SSRF. Use IMDSv2 (require hop limit 1, session tokens).

> [!warning] Mistake 6 — Hardcoding AWS Keys in GitHub Actions
> Use OIDC federation. No keys to leak.
