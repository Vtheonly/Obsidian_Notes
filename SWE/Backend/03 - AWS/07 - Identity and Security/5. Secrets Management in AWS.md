# 5. Secrets Management in AWS

> [!info] Chapter Context
> Secrets (database passwords, API keys, certificates) must be stored securely, rotated regularly, and accessed only by authorized services. AWS provides three main services for this: Secrets Manager, Systems Manager Parameter Store, and KMS. This note covers all three.

Related: [[4. STS and Temporary Credentials]] | [[08 - Object Storage/1. S3 Fundamentals]] | [[09 - Databases/1. Database Fundamentals]]

---

## 1. The Three Services

| Service | Purpose | Cost | Rotation | Encryption |
| :--- | :--- | :--- | :--- | :--- |
| **Secrets Manager** | Store and rotate secrets | $0.40/secret/month + $0.05/10K API calls | Built-in (RDS, etc.) | KMS |
| **Parameter Store** | Store config and secrets | Free (standard); $0.05/advanced parameter/month | Manual | KMS (optional) |
| **KMS** | Manage encryption keys | $1/key/month + $0.03/10K API calls | Manual (annual) | N/A (it IS the encryption) |

For most use cases:

- Use **Secrets Manager** for database credentials and other rotate-able secrets.
- Use **Parameter Store** for non-sensitive config and inexpensive secrets.
- Use **KMS** for encryption keys (envelope encryption of your data).

## 2. AWS Secrets Manager

### 2.1 Create a Secret

```bash
# A plaintext secret
aws secretsmanager create-secret --name my-app/db-password \
  --secret-string "password123"

# A key-value secret
aws secretsmanager create-secret --name my-app/db-creds \
  --secret-string '{"username":"admin","password":"password123","host":"db.example.com"}'
```

### 2.2 Retrieve a Secret

```bash
aws secretsmanager get-secret-value --secret-id my-app/db-password
# Returns JSON with the secret value

# With jq to extract just the secret string
aws secretsmanager get-secret-value --secret-id my-app/db-password \
  --query 'SecretString' --output text
```

### 2.3 Use in Code (Boto3)

```python
import boto3
import json

client = boto3.client('secretsmanager')
response = client.get_secret_value(SecretId='my-app/db-creds')
secret = json.loads(response['SecretString'])

db_url = f"postgres://{secret['username']}:{secret['password']}@{secret['host']}/myapp"
```

### 2.4 Rotation

Secrets Manager supports automatic rotation for RDS via Lambda:

1. Create a Lambda function that rotates the secret (AWS provides templates for RDS).
2. Configure Secrets Manager to call the Lambda every N days.

```bash
aws secretsmanager rotate-secret --secret-id my-app/db-password \
  --rotation-lambda-arn arn:aws:lambda:us-east-1:123456789012:function:rotate-db-password \
  --rotation-rules AutomaticallyAfterDays=30
```

For non-RDS secrets, you write the rotation Lambda yourself.

### 2.5 Cross-Account Access

To let another account read a secret:

1. Update the secret's resource policy to allow the other account.
2. The other account's IAM role/user must have `secretsmanager:GetSecretValue` permission.

## 3. AWS Systems Manager Parameter Store

Parameter Store is simpler and cheaper than Secrets Manager. It stores parameters (strings) with optional encryption.

### 3.1 Parameter Types

- **String** — Plain text.
- **StringList** — Comma-separated values.
- **SecureString** — Encrypted with KMS.

### 3.2 Create Parameters

```bash
# Plain string
aws ssm put-parameter --name /my-app/db-host --value "db.example.com" --type String

# SecureString (encrypted)
aws ssm put-parameter --name /my-app/db-password --value "password123" --type SecureString

# With a specific KMS key
aws ssm put-parameter --name /my-app/api-key --value "sk-..." --type SecureString \
  --key-id arn:aws:kms:us-east-1:123456789012:key/abc-123
```

### 3.3 Retrieve Parameters

```bash
# Get one parameter
aws ssm get-parameter --name /my-app/db-host

# Get a SecureString (decrypted)
aws ssm get-parameter --name /my-app/db-password --with-decryption

# Get multiple parameters by path
aws ssm get-parameters-by-path --path /my-app/ --recursive --with-decryption
```

### 3.4 Use in Code

```python
import boto3

ssm = boto3.client('ssm')

def get_param(name, decrypt=False):
    response = ssm.get_parameter(Name=name, WithDecryption=decrypt)
    return response['Parameter']['Value']

db_host = get_param('/my-app/db-host')
db_password = get_param('/my-app/db-password', decrypt=True)
```

### 3.5 Parameter Hierarchies

Use a hierarchical naming convention:

- `/my-app/dev/db-host`
- `/my-app/prod/db-host`
- `/my-app/dev/db-password`
- `/my-app/prod/db-password`

`get-parameters-by-path --path /my-app/prod/` retrieves all prod parameters.

### 3.6 Parameter Store vs. Secrets Manager

| Feature | Parameter Store | Secrets Manager |
| :--- | :--- | :--- |
| Cost | Free (standard) | $0.40/secret/month |
| Rotation | Manual | Built-in |
| Encryption | Optional (SecureString) | Always (KMS) |
| Cross-account | Via KMS key policy | Via resource policy |
| Best for | Config, non-sensitive secrets | Database passwords, rotate-able secrets |

## 4. AWS KMS (Key Management Service)

KMS manages encryption keys. Other AWS services (S3, EBS, Secrets Manager, Parameter Store) use KMS to encrypt your data.

### 4.1 Key Types

- **AWS managed keys** — Free; one per service per region (e.g., `aws/s3` for S3 default encryption). Cannot be deleted; rotated annually by AWS.
- **Customer managed keys** — $1/month; you create, control rotation, and set key policies. Use for sensitive data.
- **Data keys** — Used to encrypt your actual data (envelope encryption). KMS generates, encrypts with the master key, and returns both the plaintext and ciphertext.

### 4.2 Envelope Encryption

Direct KMS encryption has a 4 KB limit. For larger data, use envelope encryption:

1. Request a data key from KMS (encrypted with your master key).
2. Use the plaintext data key to encrypt your data locally (e.g., with AES-256).
3. Discard the plaintext key; store the encrypted data key alongside the ciphertext.
4. To decrypt: send the encrypted data key to KMS, get the plaintext key back, decrypt.

Most AWS services (S3, EBS, etc.) do this for you automatically.

### 4.3 Creating a Customer Managed Key

```bash
aws kms create-key --description "My app encryption key" --key-usage ENCRYPT_DECRYPT --key-spec SYMMETRIC_DEFAULT
# Returns the key ARN

# Create an alias (friendlier name)
aws kms create-alias --alias-name alias/my-app-key --target-key-id <key-id>
```

### 4.4 Key Policies

Every KMS key has a resource-based policy. The default policy allows the root account full access. For cross-account or service access, modify the policy.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "Enable IAM policies",
      "Effect": "Allow",
      "Principal": {"AWS": "arn:aws:iam::123456789012:root"},
      "Action": "kms:*",
      "Resource": "*"
    },
    {
      "Sid": "Allow Lambda to use the key",
      "Effect": "Allow",
      "Principal": {"AWS": "arn:aws:iam::123456789012:role/lambda-role"},
      "Action": ["kms:Encrypt", "kms:Decrypt", "kms:ReEncrypt*", "kms:GenerateDataKey*", "kms:DescribeKey"],
      "Resource": "*"
    }
  ]
}
```

## 5. Putting It Together: A Database Password

Recommended setup for a database password:

1. **Create a KMS customer managed key** (`alias/my-app-key`).
2. **Create a secret in Secrets Manager** (`my-app/db-password`) encrypted with the key.
3. **Configure rotation** to call a Lambda function every 30 days.
4. **Grant the application's IAM role** access to:
   - `secretsmanager:GetSecretValue` on the secret.
   - `kms:Decrypt` on the KMS key.

```python
# In the application:
import boto3
import json

client = boto3.client('secretsmanager')
response = client.get_secret_value(SecretId='my-app/db-password')
password = json.loads(response['SecretString'])['password']
```

The application never sees the password in config files or environment variables — it fetches it from Secrets Manager at startup.

## 6. Common Student Mistakes

> [!warning] Mistake 1 — Storing Secrets in Environment Variables in Plaintext
> Env vars are visible in `/proc/<pid>/environ`, in process listings, and to anyone with shell access. Use a secrets manager.

> [!warning] Mistake 2 — Forgetting to Grant KMS Decrypt Permission
> If a secret is encrypted with a customer managed KMS key, the caller needs both `secretsmanager:GetSecretValue` AND `kms:Decrypt` permissions.

> [!warning] Mistake 3 — Hardcoding Secrets in Code
> Never commit secrets to Git. Use a secrets manager, or environment variables at runtime (set via the deploy pipeline).

> [!warning] Mistake 4 — Not Rotating Secrets
> Even with secure storage, secrets should be rotated. If one leaks, the rotation limits the exposure window.

> [!warning] Mistake 5 — Using Parameter Store for Everything
> Parameter Store is fine for non-sensitive config. For database passwords, use Secrets Manager (rotation support).

> [!warning] Mistake 6 — Using the Default KMS Key for Sensitive Data
> The default `aws/s3` key is shared across all S3 buckets in your account. For sensitive data, use a customer managed key with a strict key policy.

## 7. Summary Checklist

- [ ] Secrets Manager: $0.40/secret/month; built-in rotation for RDS; best for database credentials.
- [ ] Parameter Store: free (standard); SecureString for encrypted params; best for config.
- [ ] KMS: manages encryption keys; $1/month for customer managed keys; envelope encryption for large data.
- [ ] Use hierarchical names: `/my-app/prod/db-password`.
- [ ] Grant both `secretsmanager:GetSecretValue` AND `kms:Decrypt` permissions.
- [ ] Rotate secrets regularly (Secrets Manager can automate this for RDS).
- [ ] Never commit secrets to Git; never hardcode in source.
