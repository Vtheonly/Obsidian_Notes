# 2. IAM Users, Groups, and Roles

> [!info] Chapter Context
> Building on [[1. IAM Overview]], this note is a practical guide to creating and managing IAM users, groups, and roles. Includes CLI commands, common patterns, and the role-vs-user decision.

Related: [[1. IAM Overview]] | [[3. IAM Policies]] | [[02 - AWS CLI and SDKs/3. Credentials and Profiles]]

---

## 1. When to Use a User vs. a Role

### 1.1 Use a User When

- A long-lived identity for a human (e.g., "Alice the developer").
- A long-lived identity for an external application that cannot assume roles (rare; prefer roles).
- The entity needs Console access.

### 1.2 Use a Role When

- An AWS service needs to access other AWS services (EC2 → S3, Lambda → DynamoDB).
- A user in another AWS account needs access to your account.
- A federated user (via SAML/OIDC) needs access.
- You want to issue temporary credentials (more secure than long-lived keys).

> [!tip] Default to Roles
> Whenever possible, use roles. They issue temporary credentials, are easier to audit, and reduce the risk of leaked keys.

## 2. Managing Users

### 2.1 Create a User

```bash
aws iam create-user --user-name alice

# Create access keys
aws iam create-access-key --user-name alice
# Returns: AccessKeyId, SecretAccessKey, status, creation date
# Save the secret; it's only shown once.

# Allow Console login
aws iam create-login-profile --user-name alice --password 'TempPass123!' --password-reset-required

# Add to a group
aws iam add-user-to-group --user-name alice --group-name Developers
```

### 2.2 Attach a Policy

```bash
# Attach an AWS managed policy
aws iam attach-user-policy --user-name alice \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

# Attach an inline policy (specific to this user)
aws iam put-user-policy --user-name alice --policy-name MyPolicy \
  --policy-document file://policy.json
```

### 2.3 List and Inspect

```bash
aws iam list-users
aws iam get-user --user-name alice
aws iam list-access-keys --user-name alice
aws iam list-attached-user-policies --user-name alice
aws iam list-groups-for-user --user-name alice
```

### 2.4 Delete a User

```bash
# Must delete access keys, detach policies, remove from groups first
aws iam delete-access-key --user-name alice --access-key-id AKIA...
aws iam detach-user-policy --user-name alice --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
aws iam remove-user-from-group --user-name alice --group-name Developers
aws iam delete-login-profile --user-name alice
aws iam delete-user --user-name alice
```

## 3. Managing Groups

Groups are collections of users. Users inherit the group's policies.

### 3.1 Create and Manage

```bash
# Create a group
aws iam create-group --group-name Developers

# Add users
aws iam add-user-to-group --user-name alice --group-name Developers
aws iam add-user-to-group --user-name bob --group-name Developers

# Attach a policy
aws iam attach-group-policy --group-name Developers \
  --policy-arn arn:aws:iam::aws:policy/PowerUserAccess

# List members
aws iam get-group --group-name Developers

# Remove a user
aws iam remove-user-from-group --user-name alice --group-name Developers
```

### 3.2 Common Group Patterns

- **Developers** — PowerUserAccess (no IAM changes).
- **Admins** — AdministratorAccess.
- **DBAs** — Custom policy: full RDS, read-only others.
- **Billing** — AWS Billing access only.

## 4. Managing Roles

### 4.1 Create a Role for an AWS Service

The trust policy defines who can assume the role. For an EC2 role:

```json
// trust-policy.json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {"Service": "ec2.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }
  ]
}
```

```bash
aws iam create-role --role-name EC2-S3-Access \
  --assume-role-policy-document file://trust-policy.json

# Attach a permissions policy
aws iam put-role-policy --role-name EC2-S3-Access \
  --policy-name S3Access --policy-document file://s3-policy.json

# Create an instance profile (required for EC2)
aws iam create-instance-profile --instance-profile-name EC2-S3-Access
aws iam add-role-to-instance-profile --role-name EC2-S3-Access \
  --instance-profile-name EC2-S3-Access
```

### 4.2 Create a Role for Cross-Account Access

Trust policy for account `123456789012` to assume this role:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {"AWS": "arn:aws:iam::123456789012:root"},
      "Action": "sts:AssumeRole"
    }
  ]
}
```

Then in account `123456789012`, attach a policy to a user/role allowing them to assume it:

```json
{
  "Effect": "Allow",
  "Action": "sts:AssumeRole",
  "Resource": "arn:aws:iam::OTHER-ACCOUNT:role/CrossAccountRole"
}
```

### 4.3 Assume a Role

```bash
# Via the CLI
aws sts assume-role --role-arn arn:aws:iam::123456789012:role/MyRole \
  --role-session-name MySession

# Returns temporary credentials:
# {
#   "Credentials": {
#     "AccessKeyId": "ASIA...",
#     "SecretAccessKey": "...",
#     "SessionToken": "...",
#     "Expiration": "2024-01-15T12:00:00Z"
#   }
# }

# Or via a profile (in ~/.aws/config):
# [profile production]
# role_arn = arn:aws:iam::123456789012:role/MyRole
# source_profile = default
# Then: aws s3 ls --profile production
```

## 5. Roles for AWS Services

Each AWS service that needs to access other services has a "service role":

- **EC2** — Instance profile (a role attached to the instance).
- **Lambda** — Execution role.
- **ECS** — Task role (per-task) and task execution role (for pulling images, logging).
- **CodeBuild** — Build project role.
- **CodePipeline** — Pipeline role.

These roles are created via the Console's "Create role" flow, or via CloudFormation / Terraform.

## 6. IAM Identity Center (formerly AWS SSO)

For organizations with multiple AWS accounts, IAM Identity Center is the recommended way to manage access:

- Users log in once via a corporate identity (Okta, Azure AD, Google Workspace, or AWS's internal store).
- They can then access multiple AWS accounts without separate logins.
- Permissions are managed centrally.

IAM Identity Center creates IAM roles in each account and lets users assume them via the SSO portal. This avoids creating IAM users in each account.

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Creating Access Keys for EC2
> Don't put long-lived access keys on EC2 instances. Use an instance profile (a role).

> [!warning] Mistake 2 — Forgetting the Instance Profile
> EC2 roles require an instance profile (a wrapper around the role). The role alone is not enough.

> [!warning] Mistake 3 — Not Cleaning Up Before Deleting a User
> You cannot delete a user with access keys, attached policies, or group memberships. Delete those first.

> [!warning] Mistake 4 — Using a Single User for a Team
> Each person should have their own user (for accountability). Don't share credentials.

> [!warning] Mistake 5 — Forgetting to Assume the Role in Cross-Account Access
> Both accounts must allow the access: the trusting account's trust policy AND the trusted account's user policy.

## 8. Summary Checklist

- [ ] Users: long-lived identities for humans/apps. Have passwords (Console) and access keys (API).
- [ ] Groups: collections of users. Users inherit group policies.
- [ ] Roles: assumable identities. Issue temporary credentials via STS.
- [ ] Default to roles; reserve users for cases where roles don't fit.
- [ ] EC2 roles require an instance profile (a wrapper).
- [ ] Lambda, ECS, CodeBuild, CodePipeline have service-specific roles.
- [ ] Cross-account: trusting account's trust policy + trusted account's user policy.
- [ ] IAM Identity Center for multi-account SSO.
