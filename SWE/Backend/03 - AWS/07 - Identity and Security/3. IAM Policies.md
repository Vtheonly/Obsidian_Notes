# 3. IAM Policies

> [!info] Chapter Context
> Policies are the heart of IAM. This note covers policy structure, common patterns, conditions, variables, and how to write policies that follow least privilege.

Related: [[1. IAM Overview]] | [[2. IAM Users, Groups, and Roles]] | [[4. STS and Temporary Credentials]]

---

## 1. Policy Structure Recap

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ReadBucket",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {
        "IpAddress": {"aws:SourceIp": "10.0.0.0/8"}
      }
    }
  ]
}
```

- `Sid` — Statement ID (optional, for documentation).
- `Effect` — `Allow` or `Deny`.
- `Action` — AWS API actions.
- `Resource` — ARN(s) the action applies to.
- `Condition` — Optional conditions.

## 2. Common Action Patterns

### 2.1 Wildcards

- `s3:*` — All S3 actions.
- `s3:Get*` — All S3 actions starting with "Get" (GetObject, GetBucketLocation, etc.).
- `*` — All actions on all services (very dangerous).

### 2.2 Common S3 Actions

- `s3:ListBucket` — List objects in a bucket (resource: bucket).
- `s3:GetObject` — Read an object (resource: object).
- `s3:PutObject` — Write an object.
- `s3:DeleteObject` — Delete an object.
- `s3:CreateBucket` — Create a bucket.
- `s3:DeleteBucket` — Delete a bucket.

### 2.3 Common EC2 Actions

- `ec2:RunInstances` — Launch an instance.
- `ec2:StartInstances` / `ec2:StopInstances` / `ec2:TerminateInstances`.
- `ec2:DescribeInstances` — List instances.

### 2.4 Common DynamoDB Actions

- `dynamodb:PutItem`, `dynamodb:GetItem`, `dynamodb:UpdateItem`, `dynamodb:DeleteItem`.
- `dynamodb:Query`, `dynamodb:Scan`.
- `dynamodb:CreateTable`, `dynamodb:DeleteTable`.

## 3. Common Policy Patterns

### 3.1 Read-Only Access to a Bucket

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}
```

### 3.2 Read-Write Access to a Specific Prefix

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::my-bucket/users/${aws:userid}/*"
    }
  ]
}
```

The `${aws:userid}` variable makes this a "user-specific folder" policy. Each user can only access their own folder.

### 3.3 Deny All Except Encrypted Objects

```json
{
  "Effect": "Deny",
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {
    "StringNotEquals": {"s3:x-amz-server-side-encryption": "AES256"}
  }
}
```

### 3.4 Require MFA for Sensitive Actions

```json
{
  "Effect": "Allow",
  "Action": "iam:DeleteUser",
  "Resource": "*",
  "Condition": {
    "Bool": {"aws:MultiFactorAuthPresent": true}
  }
}
```

### 3.5 Allow Within Office Hours

```json
{
  "Effect": "Allow",
  "Action": "ec2:RunInstances",
  "Resource": "*",
  "Condition": {
    "DateGreaterThan": {"aws:CurrentTime": "2024-01-01T09:00:00Z"},
    "DateLessThan": {"aws:CurrentTime": "2024-01-01T17:00:00Z"}
  }
}
```

## 4. Policy Variables

IAM supports variables in policies, substituted at evaluation time:

- `${aws:userid}` — The user's unique ID.
- `${aws:username}` — The user's name.
- `${aws:PrincipalAccount}` — The account ID of the principal.
- `${s3:prefix}` — The S3 prefix being accessed.
- `${ec2:SourceInstanceArn}` — The ARN of the EC2 instance making the call.

Example: a policy that lets each user access only their own S3 prefix:

```json
{
  "Effect": "Allow",
  "Action": "s3:*",
  "Resource": "arn:aws:s3:::my-bucket/home/${aws:username}/*"
}
```

## 5. Conditions

Conditions refine when a statement applies. Common condition keys:

### 5.1 IP Address

```json
"Condition": {
  "IpAddress": {"aws:SourceIp": "10.0.0.0/8"},
  "NotIpAddress": {"aws:SourceIp": "10.0.0.1"}
}
```

### 5.2 MFA

```json
"Condition": {
  "Bool": {"aws:MultiFactorAuthPresent": true}
}
```

### 5.3 Time

```json
"Condition": {
  "DateGreaterThan": {"aws:CurrentTime": "2024-01-01T00:00:00Z"},
  "DateLessThan": {"aws:CurrentTime": "2024-12-31T23:59:59Z"}
}
```

### 5.4 String Comparison

```json
"Condition": {
  "StringEquals": {"s3:prefix": "home/"},
  "StringLike": {"s3:prefix": "home/${aws:username}/*"}
}
```

### 5.5 Source ARN (for service-to-service)

```json
"Condition": {
  "ArnEquals": {"aws:SourceArn": "arn:aws:lambda:us-east-1:123456789012:function:my-func"}
}
```

Use this to restrict which Lambda functions can trigger your SNS topic, for example.

## 6. Managed vs. Inline Policies

### 6.1 AWS Managed Policies

Pre-built by AWS. Updated by AWS (sometimes with breaking changes). Examples: `AmazonS3ReadOnlyAccess`, `AdministratorAccess`.

### 6.2 Customer Managed Policies

Created by you, reusable across users/roles. Recommended for most use cases — easy to update and audit.

```bash
aws iam create-policy --policy-name MyS3Policy --policy-document file://policy.json
aws iam attach-user-policy --user-name alice --policy-arn arn:aws:iam::ACCOUNT:policy/MyS3Policy
```

### 6.3 Inline Policies

Embedded in a single user/group/role. Not reusable. Use for one-off, specific permissions.

```bash
aws iam put-user-policy --user-name alice --policy-name MyInline --policy-document file://policy.json
```

## 7. Tools for Policy Management

### 7.1 IAM Policy Simulator

Test what a user/role can do:

```bash
aws iam simulate-principal-policy \
  --policy-source-arn arn:aws:iam::123456789012:user/alice \
  --action-names s3:GetObject s3:PutObject \
  --resource-arns arn:aws:s3:::my-bucket/*
```

Or use the Console: IAM → Policy Simulator.

### 7.2 IAM Access Analyzer

Identifies resources shared externally (e.g., an S3 bucket accessible from another account). Also identifies unused permissions in a policy.

### 7.3 AWS IAM Policy Validator

Validates JSON syntax and policy language. Built into the Console; CLI tool available.

## 8. Common Student Mistakes

> [!warning] Mistake 1 — Using `Resource: "*"` When a Specific ARN Works
> Always scope to specific resources. `s3:GetObject` with `Resource: "*"` allows reading every object in every bucket.

> [!warning] Mistake 2 — Forgetting That Deny Wins
> An explicit deny overrides all allows. Use deny to enforce hard boundaries.

> [!warning] Mistake 3 — Forgetting the Bucket vs. Object Distinction
> `arn:aws:s3:::my-bucket` is the bucket (for ListBucket). `arn:aws:s3:::my-bucket/*` is the objects (for GetObject). Different actions need different resources.

> [!warning] Mistake 4 — Not Using Conditions
> Conditions (MFA, IP, time) let you scope permissions further. Use them.

> [!warning] Mistake 5 — Using AWS Managed Policies in Production
> AWS managed policies (like `AmazonS3FullAccess`) are often broader than needed. Write custom policies for production.

> [!warning] Mistake 6 — Forgetting to Test with the Policy Simulator
> Use the simulator to verify what a user can actually do before deploying.

## 9. Summary Checklist

- [ ] Policies: `Version`, `Statement[]` with `Effect`, `Action`, `Resource`, `Condition`.
- [ ] Wildcards: `s3:*`, `s3:Get*`, `*` (very dangerous).
- [ ] Variables: `${aws:username}`, `${aws:userid}`, `${aws:PrincipalAccount}`.
- [ ] Conditions: IP, MFA, time, string comparison, source ARN.
- [ ] Policy types: AWS managed, customer managed (recommended), inline.
- [ ] Use IAM Policy Simulator to test what a user can do.
- [ ] Use IAM Access Analyzer to identify external access and unused permissions.
- [ ] Always scope to specific resources; never use `*` when a specific ARN works.
