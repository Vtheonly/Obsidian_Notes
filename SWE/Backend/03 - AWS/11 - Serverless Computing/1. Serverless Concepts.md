# 1. Serverless Concepts

> [!info] Chapter Context
> Serverless is a cloud execution model where the cloud provider manages the infrastructure, and you pay only for actual usage. This note covers what serverless is, when to use it, the trade-offs vs. containers, and the AWS serverless services.

Related: [[10 - Event Driven Systems/4. Dead Letter Queues and Fan-Out]] | [[2. Lambda Fundamentals]] | [[15 - Architecture Patterns/4. Serverless Patterns]]

## 1. What "Serverless" Means

"Serverless" does not mean "no servers." It means:

1. **No server management** — You don't provision, patch, or scale servers. The provider does.
2. **Pay-per-use** — You pay for actual invocations/requests, not for idle capacity.
3. **Auto-scaling** — The provider scales to handle any load, instantly.
4. **Event-driven** — Serverless functions are triggered by events (HTTP requests, S3 uploads, scheduled timers).

In AWS, "serverless" includes:

- **Lambda** — Functions.
- **Fargate** — Containers (serverless ECS/EKS).
- **DynamoDB** — Database.
- **S3** — Storage.
- **API Gateway** — HTTP API.
- **Aurora Serverless** — Database.
- **Step Functions** — Workflow orchestration.

## 2. Serverless vs. Containers vs. VMs

| Aspect | VM (EC2) | Container (ECS/EKS) | Serverless (Lambda) |
| :--- | :--- | :--- | :--- |
| Granularity | Whole OS | Process | Function |
| Scaling | Manual or auto-scaling group | Task/pod-level | Per-request, instant |
| Idle cost | Yes (always running) | Yes (always running) | None (pay per invocation) |
| Cold start | Seconds (boot) | Milliseconds (container start) | 100ms-1s (function init) |
| Max duration | Unlimited | Unlimited | 15 minutes |
| Max memory | Instance size | Container limit | 10 GB |
| Vendor lock-in | Low | Low | High (Lambda API is AWS-specific) |
| Best for | Long-running apps, legacy | Microservices | Event-driven, bursty workloads |

## 3. When to Use Serverless

### 3.1 Good Fits

- **Event-driven workloads** — React to S3 uploads, DynamoDB changes, SNS messages.
- **Bursty traffic** — Traffic that spikes occasionally (e.g., a viral link).
- **Scheduled tasks** — Cron-like jobs (daily reports, backups).
- **HTTP APIs** — REST APIs with variable traffic.
- **Quick prototypes** — No infrastructure to provision.

### 3.2 Bad Fits

- **Long-running processes** — Lambda max is 15 minutes.
- **Constant high traffic** — Serverless is more expensive at peak; containers are cheaper.
- **Real-time / low-latency** — Cold starts add 100ms-1s.
- **Heavy stateful apps** — Serverless functions are stateless; external state is required.
- **Legacy apps** — Refactoring is usually more work than containerizing.

## 4. The Serverless Cost Model

Lambda pricing:

- **Per request** — $0.20 per million requests (first 1 million free).
- **Per GB-second** — Memory allocated × execution time. $0.0000166667 per GB-second (first 400,000 GB-seconds free).

Example: A function with 512 MB memory that runs for 200ms, invoked 1 million times:

- Requests: 1,000,000 × $0.20 / 1,000,000 = $0.20
- Compute: 1,000,000 × (512/1024) GB × (200/1000) s = 100,000 GB-seconds
- Compute cost: 100,000 × $0.0000166667 = $1.67
- **Total: ~$1.87** for 1 million invocations.

For comparison, a t3.micro EC2 instance costs ~$7/month 24/7. If your traffic is below ~3.7 million requests/month, Lambda is cheaper. Above that, EC2 becomes cheaper.

## 5. Cold Starts

When a Lambda function is invoked for the first time (or after being idle), AWS:

1. Allocates a container.
2. Loads your code.
3. Initializes the runtime (Python, Node.js, etc.).
4. Runs your initialization code (e.g., `boto3.client('s3')`).
5. Calls the handler.

Steps 1-4 are the **cold start** — typically 100ms-1s, depending on the runtime and code size.

Subsequent invocations reuse the container (warm start), which is much faster (1-50ms).

### 5.1 Reducing Cold Starts

- **Use interpreted languages** — Python, Node.js have faster cold starts than Java, C#.
- **Minimize dependencies** — Smaller deployment package = faster load.
- **Use provisioned concurrency** — Pre-warm instances (extra cost).
- **Initialize outside the handler** — Boto3 clients, DB connections created at module level are reused across invocations.

```python
# GOOD: Boto3 client created once, reused
import boto3
s3 = boto3.client('s3')   # runs on cold start only

def handler(event, context):
    s3.get_object(Bucket='my-bucket', Key='file')   # reuses the client

# BAD: Boto3 client created on every invocation
def handler(event, context):
    s3 = boto3.client('s3')   # created every time
    s3.get_object(Bucket='my-bucket', Key='file')
```

## 6. The Serverless Application Stack

A typical serverless application:

```mermaid

    APIGW -->|HTTP request| Lambda
    Cognito -->|auth| APIGW
```

- **API Gateway** receives HTTP requests and routes them to Lambda.
- **Lambda** runs your business logic.
- **DynamoDB** stores state (serverless database).
- **S3** stores files.
- **Cognito** handles user authentication.
- **CloudWatch** captures logs and metrics.

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Using Serverless for Long-Running Tasks
> Lambda max is 15 minutes. For longer tasks, use ECS/Fargate or Step Functions (which can run for hours).

> [!warning] Mistake 2 — Forgetting About Cold Starts
> Cold starts add latency to the first request after idle. If your latency budget is tight, use provisioned concurrency.

> [!warning] Mistake 3 — Initializing Clients Inside the Handler
> Boto3 clients, DB connections, etc. should be initialized at module level (outside the handler) to be reused across invocations.

> [!warning] Mistake 4 — Forgetting That Lambda Scales Per-Request
> Each concurrent request invokes a separate Lambda instance. 1000 concurrent requests = 1000 instances. This can overwhelm downstream resources (databases). Use connection pooling (RDS Proxy) and concurrency limits.

> [!warning] Mistake 5 — Treating Lambda as Stateful
> Lambda invocations are stateless. The container may be reused, but it may also be destroyed. Don't store state in module-level variables (except as a cache).

> [!warning] Mistake 6 — Not Setting Concurrency Limits
> Without limits, a Lambda under load can exhaust downstream resources (DB connections, API rate limits). Set a reserved concurrency.
