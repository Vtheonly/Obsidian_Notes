# 5. Step Functions

> [!info] Chapter Context
> AWS Step Functions is a serverless workflow orchestration service. It coordinates multiple Lambda functions (and other AWS services) into multi-step business processes, with retries, error handling, and state management.

Related: [[4. API Gateway]] | [[11 - Serverless Computing/1. Serverless Concepts]] | [[15 - Architecture Patterns/3. Saga Pattern]]

---

## 1. What Step Functions Is

Step Functions lets you define **state machines** — workflows with steps, branches, parallel paths, retries, and error handling. Each step is a task (Lambda, ECS, SNS, DynamoDB, etc.).

Use cases:

- Order processing (validate → charge → ship → notify).
- ETL pipelines (extract → transform → load).
- Human approval workflows (submit → wait for approval → proceed).
- Long-running processes (Lambda is limited to 15 minutes; Step Functions can run for up to a year).

---

## 2. State Machine Definition

A state machine is defined in JSON (Amazon States Language):

```json
{
  "Comment": "Order processing workflow",
  "StartAt": "ValidateOrder",
  "States": {
    "ValidateOrder": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:validate-order",
      "Next": "ChargeCard"
    },
    "ChargeCard": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:charge-card",
      "Retry": [{
        "ErrorEquals": ["States.TaskFailed"],
        "IntervalSeconds": 2,
        "MaxAttempts": 3,
        "BackoffRate": 2.0
      }],
      "Catch": [{
        "ErrorEquals": ["States.ALL"],
        "Next": "RefundAndNotify"
      }],
      "Next": "ShipOrder"
    },
    "ShipOrder": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:ship-order",
      "Next": "NotifyCustomer"
    },
    "NotifyCustomer": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:notify-customer",
      "End": true
    },
    "RefundAndNotify": {
      "Type": "Task",
      "Resource": "arn:aws:lambda:us-east-1:123456789012:function:refund-and-notify",
      "End": true
    }
  }
}
```

## 3. State Types

- **Task** — A single unit of work (Lambda, ECS, SNS, etc.).
- **Choice** — Branch based on input.
- **Parallel** — Run multiple branches simultaneously.
- **Map** — Iterate over a list (process each item).
- **Wait** — Wait for a duration or until a timestamp.
- **Pass** — Pass input to output (no-op, useful for transformation).
- **Succeed** — Terminal success state.
- **Fail** — Terminal failure state.

### 3.1 Choice

```json
"CheckAmount": {
  "Type": "Choice",
  "Choices": [{
    "Variable": "$.amount",
    "NumericGreaterThan": 1000,
    "Next": "ManualReview"
  }, {
    "Variable": "$.amount",
    "NumericLessThanEquals": 1000,
    "Next": "AutoApprove"
  }],
  "Default": "AutoApprove"
}
```

### 3.2 Parallel

```json
"ProcessInParallel": {
  "Type": "Parallel",
  "Branches": [{
    "StartAt": "ProcessImage",
    "States": {"ProcessImage": {"Type": "Task", "Resource": "...", "End": true}}
  }, {
    "StartAt": "ProcessMetadata",
    "States": {"ProcessMetadata": {"Type": "Task", "Resource": "...", "End": true}}
  }],
  "Next": "CombineResults"
}
```

### 3.3 Map

```json
"ProcessItems": {
  "Type": "Map",
  "ItemsPath": "$.items",
  "MaxConcurrency": 10,
  "Iterator": {
    "StartAt": "ProcessOne",
    "States": {"ProcessOne": {"Type": "Task", "Resource": "...", "End": true}}
  },
  "Next": "AllDone"
}
```

## 4. Error Handling

### 4.1 Retry

```json
"Retry": [{
  "ErrorEquals": ["States.TaskFailed"],
  "IntervalSeconds": 2,
  "MaxAttempts": 3,
  "BackoffRate": 2.0
}]
```

- `IntervalSeconds` — Initial wait before first retry.
- `MaxAttempts` — Maximum retries.
- `BackoffRate` — Multiplier for each subsequent wait.

### 4.2 Catch

```json
"Catch": [{
  "ErrorEquals": ["CustomError"],
  "Next": "HandleCustomError"
}, {
  "ErrorEquals": ["States.ALL"],
  "Next": "HandleAnyError"
}]
```

### 4.3 Predefined Error Names

- `States.ALL` — Any error.
- `States.Timeout` — Task timed out.
- `States.TaskFailed` — Task failed.
- `States.Permissions` — Insufficient permissions.
- Custom errors thrown by Lambda (raise an exception with a specific name).

## 5. Standard vs. Express Workflows

| Feature | Standard | Express |
| :--- | :--- | :--- |
| Duration | Up to 1 year | Up to 5 minutes |
| Rate | 2,000 executions/second | 100,000 executions/second |
| Pricing | Per state transition | Per invocation + duration |
| At-least-once | Yes | At-least-once (async) or at-most-once (sync) |
| Execution history | 90 days | None (use CloudWatch Logs) |
| Use case | Long-running, audit-required | High-throughput, short |

Use Express for high-throughput, short workflows (e.g., API Gateway → Step Functions → Lambda). Use Standard for long-running or audit-required workflows.

## 6. Invoking Step Functions

### 6.1 From Lambda

```python
import boto3
sf = boto3.client('stepfunctions')

response = sf.start_execution(
    stateMachineArn='arn:aws:states:us-east-1:123456789012:stateMachine:OrderWorkflow',
    input=json.dumps({'order_id': 123, 'amount': 99.99})
)
execution_arn = response['executionArn']
```

### 6.2 From API Gateway (Synchronous)

API Gateway can call `StartSyncExecution` (Express workflows only) for synchronous invocation.

### 6.3 From EventBridge

EventBridge can start a Step Functions execution on a schedule or event.

## 7. Creating a State Machine

```bash
aws stepfunctions create-state-machine \
  --name OrderWorkflow \
  --definition file://workflow.json \
  --role-arn arn:aws:iam::123456789012:role/stepfunctions-role \
  --type STANDARD
```

The role must allow Step Functions to invoke the Lambda functions, etc.

## 8. Common Student Mistakes

> [!warning] Mistake 1 — Using Lambda for Long Workflows
> Lambda max is 15 minutes. For longer workflows, use Step Functions (up to 1 year for Standard).

> [!warning] Mistake 2 — Not Using Retries
> Lambda can fail transiently. Use Step Functions' `Retry` for resilient workflows.

> [!warning] Mistake 3 — Forgetting the IAM Role
> Step Functions needs a role with permissions to invoke the Lambda functions (or other services).

> [!warning] Mistake 4 — Using Standard for High-Throughput
> Standard is rate-limited to 2,000 executions/second. For higher throughput, use Express.

> [!warning] Mistake 5 — Not Using Map for Batch Processing
> If you're iterating over a list, use the `Map` state. It handles concurrency and error handling per item.

> [!warning] Mistake 6 — Hardcoding ARNs in the Definition
> Use CloudFormation/SAM/CDK to substitute ARNs at deploy time.

## 9. Summary Checklist

- [ ] Step Functions is a serverless workflow orchestrator.
- [ ] State types: Task, Choice, Parallel, Map, Wait, Pass, Succeed, Fail.
- [ ] Error handling: Retry (with backoff), Catch (with error matching).
- [ ] Standard workflows: up to 1 year, 2,000/s. Express: up to 5 min, 100,000/s.
- [ ] Use Express for high-throughput, short workflows. Use Standard for long-running.
- [ ] The IAM role must allow Step Functions to invoke the resources (Lambda, etc.).
- [ ] Use Map for batch processing (iterate over a list).
