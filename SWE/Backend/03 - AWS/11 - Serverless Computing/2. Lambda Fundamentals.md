# 2. Lambda Fundamentals

> [!info] Chapter Context
> AWS Lambda is the most-used serverless compute service. This note covers the Lambda programming model, deployment, the execution environment, and how to write Lambda functions in Python and Node.js.

Related: [[1. Serverless Concepts]] | [[3. Lambda Triggers and Events]] | [[4. API Gateway]] | [[5. Step Functions]]

## 1. The Lambda Programming Model

A Lambda function is:

- A **handler** — The function AWS calls when invoking.
- An **event** — JSON input (the trigger data).
- A **context** — Metadata about the invocation (request ID, deadline, function ARN).
- An optional **return value** — What the function returns (for synchronous invocations).

```python
# Python handler
def lambda_handler(event, context):
    name = event.get('name', 'World')
    return {
        'statusCode': 200,
        'body': f'Hello, {name}!'
    }
```

```javascript
// Node.js handler
exports.handler = async (event) => {
    const name = event.name || 'World';
    return {
        statusCode: 200,
        body: `Hello, ${name}!`
    };
};
```

## 2. The Handler Signature

### 2.1 Python

```python
def handler(event, context):
    # event: dict (the trigger data)
    # context: LambdaContext (metadata)
    return result
```

The handler name in the Lambda config is `module.function`. For `def lambda_handler(event, context)` in `index.py`, the handler is `index.lambda_handler`.

### 2.2 Node.js

```javascript
// Async handler (modern)
exports.handler = async (event, context) => {
    return result;
};

// Callback handler (legacy)
exports.handler = (event, context, callback) => {
    callback(null, result);
};
```

## 3. The `event` Object

The event is a dict (Python) / object (Node.js) with the trigger data. Its shape depends on the trigger:

### 3.1 API Gateway

```json
{
  "httpMethod": "GET",
  "path": "/users/123",
  "headers": {"Content-Type": "application/json"},
  "queryStringParameters": {"filter": "active"},
  "pathParameters": {"id": "123"},
  "body": "{\"name\": \"Alice\"}"
}
```

### 3.2 S3

```json
{
  "Records": [{
    "eventName": "ObjectCreated:Put",
    "s3": {
      "bucket": {"name": "my-bucket"},
      "object": {"key": "photos/alice.jpg", "size": 12345}
    }
  }]
}
```

### 3.3 DynamoDB Streams

```json
{
  "Records": [{
    "eventName": "INSERT",
    "dynamodb": {
      "Keys": {"id": {"S": "1"}},
      "NewImage": {"id": {"S": "1"}, "name": {"S": "Alice"}}
    }
  }]
}
```

### 3.4 SQS

```json
{
  "Records": [{
    "messageId": "abc123",
    "body": "{\"order_id\": 123}",
    "receiptHandle": "...",
    "attributes": {"ApproximateReceiveCount": "1"}
  }]
}
```

AWS provides event schemas in the docs and in the EventBridge Schema Registry.

---

## 4. The `context` Object

The `context` object has metadata about the invocation:

- `function_name` — The function's name.
- `function_version` — The version (`$LATEST` or an alias).
- `invoked_function_arn` — The ARN used to invoke.
- `memory_limit_in_mb` — Memory allocated.
- `aws_request_id` — Unique ID for this invocation (useful for log correlation).
- `get_remaining_time_in_millis()` — Milliseconds until Lambda kills the function (Python).

Use `get_remaining_time_in_millis()` to decide whether to process more items or return early.

## 5. Deploying a Lambda Function

### 5.1 With the CLI

Zip your code and upload:

```bash
zip function.zip index.py

aws lambda create-function \
  --function-name my-func \
  --runtime python3.11 \
  --handler index.lambda_handler \
  --role arn:aws:iam::123456789012:role/lambda-role \
  --zip-file fileb://function.zip
```

Update:

```bash
aws lambda update-function-code --function-name my-func --zip-file fileb://function.zip
```

### 5.2 With Container Images

For functions larger than 50 MB (or with custom runtime), package as a container image:

```dockerfile
FROM public.ecr.aws/lambda/python:3.11
COPY index.py ${LAMBDA_TASK_ROOT}/
COPY requirements.txt .
RUN pip install -r requirements.txt
CMD ["index.lambda_handler"]
```

```bash
docker build -t my-func .
docker tag my-func:latest 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-func:latest
docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-func:latest

aws lambda create-function \
  --function-name my-func \
  --package-type Image \
  --code ImageUri=123456789012.dkr.ecr.us-east-1.amazonaws.com/my-func:latest \
  --role arn:aws:iam::123456789012:role/lambda-role
```

### 5.3 With the Serverless Framework or SAM

For complex functions (multiple resources, API Gateway, etc.), use:

- **AWS SAM (Serverless Application Model)** — AWS's IaC for serverless.
- **Serverless Framework** — Third-party, popular.
- **AWS CDK** — Programmatic IaC.

```yaml
# SAM template.yaml
Resources:
  MyFunction:
    Type: AWS::Serverless::Function
    Properties:
      CodeUri: ./src
      Handler: index.lambda_handler
      Runtime: python3.11
      Events:
        Api:
          Type: Api
          Properties:
            Path: /hello
            Method: get
```

## 6. The Execution Environment

Each Lambda function runs in a **container** (a "execution environment") managed by AWS:

- The container has the runtime (Python, Node.js) and your code.
- The container is reused across invocations (warm starts).
- After ~15 minutes of idle, the container is destroyed (next invocation is a cold start).
- Each concurrent invocation gets its own container (1000 concurrent invocations = 1000 containers).

### 6.1 What Is Reused Across Invocations

- Module-level code (Boto3 clients, DB connections).
- The `/tmp` directory (512 MB to 10 GB). Persist files here for reuse, but don't rely on it (the container may be destroyed).

### 6.2 What Is NOT Reused

- The handler's local variables.
- Environment variables (these can be updated via the API).
- The event (each invocation has its own).

## 7. Environment Variables

Lambda functions can have environment variables (set via the API or Console):

```bash
aws lambda update-function-configuration --function-name my-func \
  --environment "Variables={LOG_LEVEL=info,TABLE_NAME=users}"
```

In the function:

```python
import os
log_level = os.environ['LOG_LEVEL']
table_name = os.environ['TABLE_NAME']
```

Lambda also sets reserved environment variables automatically:

- `_HANDLER` — The handler name.
- `_X_AMZN_TRACE_ID` — The X-Ray trace ID.
- `AWS_REGION` — The region.
- `AWS_LAMBDA_FUNCTION_NAME` — The function name.
- `AWS_LAMBDA_FUNCTION_MEMORY_SIZE` — Memory in MB.
- `AWS_LAMBDA_LOG_GROUP_NAME`, `AWS_LAMBDA_LOG_STREAM_NAME` — CloudWatch Logs locations.

## 8. Memory and Timeout

### 8.1 Memory

- 128 MB to 10 GB (1 MB increments).
- **More memory = more CPU** — Lambda allocates CPU proportionally to memory. At ~1.8 GB, you get 1 full vCPU; above that, multiple vCPUs.
- The optimal memory is workload-dependent. Use AWS Compute Optimizer or experiment.

### 8.2 Timeout

- 1 second to 15 minutes (default 3 seconds).
- If the function runs longer, Lambda kills it.

> [!warning] Always Set the Timeout Carefully
> Too short: the function is killed mid-processing. Too long: a hung function runs up costs. Monitor duration and adjust.

## 9. Logging

Lambda automatically sends `stdout`/`stderr` to CloudWatch Logs. Each function has a log group `/aws/lambda/my-func` with log streams per container.

```python
import logging
logger = logging.getLogger()
logger.setLevel(logging.INFO)

def lambda_handler(event, context):
    logger.info(f"Processing event: {event}")
    # ...
```

```javascript
exports.handler = async (event) => {
    console.log('Processing event:', JSON.stringify(event));
    // ...
};
```

## 10. Common Student Mistakes

> [!warning] Mistake 1 — Initializing Clients in the Handler
> Boto3 clients, DB connections created inside the handler are recreated every invocation. Create them at module level.

> [!warning] Mistake 2 — Forgetting the 15-Minute Limit
> Lambda max is 15 minutes. For longer tasks, use Step Functions or Fargate.

> [!warning] Mistake 3 — Not Using `/tmp` for Caches
> `/tmp` persists across invocations within the same container. Use it for downloaded files, ML models, etc.

> [!warning] Mistake 4 — Forgetting That Lambda Scales Per-Request
> 1000 concurrent requests = 1000 containers, each with its own DB connection. Use RDS Proxy or DynamoDB.

> [!warning] Mistake 5 — Not Setting Reserved Concurrency
> Without limits, a Lambda under load can exhaust downstream resources. Set a reserved concurrency limit.

> [!warning] Mistake 6 — Forgetting to Handle Idempotency
> Lambda may be invoked multiple times for the same event (especially with retries). Make your function idempotent.

> [!warning] Mistake 7 — Not Logging Enough
> Without logs, debugging Lambda is impossible. Use `logging` (Python) or `console.log` (Node.js) liberally.

## 11. Summary Checklist

- [ ] Lambda handler signature: `def handler(event, context)` (Python) or `exports.handler = async (event, context) => {...}` (Node.js).
- [ ] Event shape depends on the trigger (API Gateway, S3, SQS, DynamoDB Streams, etc.).
- [ ] Deploy via zip (small) or container image (large).
- [ ] Use SAM, Serverless Framework, or CDK for complex functions.
- [ ] Initialize Boto3 clients and DB connections at module level.
- [ ] Memory: 128 MB - 10 GB (more memory = more CPU).
- [ ] Timeout: 1 second - 15 minutes.
- [ ] `/tmp` (up to 10 GB) persists across invocations in the same container.
- [ ] Lambda scales per-request; set reserved concurrency to protect downstream.
- [ ] Make functions idempotent (may be invoked multiple times).
