## 1. Image Selection

### 1.1 Pin Versions Explicitly

```dockerfile
# Bad
FROM node:latest

# Good
FROM node:18.19.0-alpine3.19
```

`latest` is a moving target. Today's `node:latest` is Node 22; tomorrow it might be Node 23 with breaking changes. Pin to a specific version for reproducible builds. Pinning by digest (`FROM node@sha256:abc123...`) is even more reproducible.

### 1.2 Choose the Right Variant

Official images usually come in several variants:

| Variant | Size | Use case |
| :--- | :--- | :--- |
| `node:18` | ~400 MB | Full Debian-based. Has everything, large. |
| `node:18-slim` | ~70 MB | Stripped Debian. No build tools. |
| `node:18-alpine` | ~50 MB | Alpine Linux (musl libc). Smallest. |
| `node:18-bookworm-slim` | ~75 MB | Specific Debian release, slim. |

> [!warning] Alpine and musl libc
> Alpine uses `musl` instead of `glibc`. Most Python packages with C extensions (NumPy, Pandas, lxml) and some Node packages with native bindings do not have prebuilt musl wheels and must be compiled from source â€” which requires installing `gcc`, `musl-dev`, etc. If you see long Python install times on Alpine, switch to `python:3.12-slim` (Debian-based) instead.

### 1.3 Use Official Images

Official images (`node`, `python`, `nginx`, `postgres`, `redis`) are maintained by Docker and the upstream project. They get security patches promptly. Avoid random community images for production unless you have audited them.

## 2. Layer Caching

### 2.1 Order Instructions by Change Frequency

Put rarely-changing instructions first, frequently-changing last:

```dockerfile
FROM node:18-alpine          # changes rarely
WORKDIR /app                  # never changes
COPY package*.json ./         # changes when deps change
RUN npm ci                    # changes when deps change
COPY . .                      # changes every code edit
CMD ["node", "server.js"]     # changes rarely
```

### 2.2 Combine `RUN` Commands

Each `RUN` is a layer. Combine related commands to reduce layers and avoid "deleted file still in earlier layer" bloat:

```dockerfile
# Bad: 3 layers, the rm doesn't shrink the apt install layer
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# Good: 1 layer, cleanup in same layer
RUN apt-get update \
    && apt-get install -y --no-install-recommends curl \
    && rm -rf /var/lib/apt/lists/*
```

### 2.3 Use `--no-install-recommends` for `apt`

By default, `apt install` installs "recommended" packages that are not strictly required. `--no-install-recommends` skips them, often saving tens of MB.

## 3. Security

### 3.1 Run as a Non-Root User

```dockerfile
RUN addgroup -S app && adduser -S app -G app
USER app
```

By default, containers run as root. If an attacker escapes the container, they have root on the host. Create a dedicated user and switch to it before `CMD`.

### 3.2 Do Not Bake Secrets

Never `COPY .env` or `RUN echo "API_KEY=xxx" > /etc/secrets`. Secrets in image layers can be extracted. Pass secrets at runtime via:

- Environment variables (`docker run -e API_KEY=xxx`).
- Secret files mounted as volumes.
- Docker secrets (Swarm) or Kubernetes secrets.

For build-time secrets, use BuildKit `--mount=type=secret`.

### 3.3 Use `COPY --chown` to Set Ownership in One Layer

```dockerfile
COPY --chown=app:app . /app
```

This sets ownership at copy time, avoiding a separate `RUN chown -R app:app /app` layer.

### 3.4 Drop Capabilities (At Runtime)

```bash
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE myapp
```

Drops all Linux capabilities and adds back only what's needed. We cover this in detail in [[8. Docker Security]].

### 3.5 Scan Images for Vulnerabilities

```bash
docker scan myapp:1.0           # Docker Desktop built-in (uses Snyk)
trivy image myapp:1.0           # Open-source scanner
```

Run scans in CI to catch known CVEs in your base images and dependencies.

## 4. Image Size

### 4.1 Use Multi-Stage Builds

Covered in [[4.2 Multi-Stage Builds]]. This is the single biggest size reduction.

### 4.2 Clean Up in the Same Layer

```dockerfile
RUN apk add --no-cache curl \
    && curl -o /tmp/file.tar.gz https://example.com/file.tar.gz \
    && tar xzf /tmp/file.tar.gz -C /opt \
    && rm /tmp/file.tar.gz
```

If you download and extract in one `RUN`, then `rm` in another `RUN`, the download still occupies a layer.

### 4.3 Use `.dockerignore`

Keep the build context small. See [[4.1 The .dockerignore File]].

### 4.4 Use `npm ci` Instead of `npm install`

- `npm ci` requires `package-lock.json`.
- It installs exactly what's in the lockfile â€” no resolving, no updates.
- It is faster and reproducible.
- It removes `node_modules` before installing, ensuring a clean state.

## 5. Reliability

### 5.1 Use Exec Form for `CMD` and `ENTRYPOINT`

```dockerfile
# Bad (shell form â€” sh becomes PID 1, signals not forwarded)
CMD node server.js

# Good (exec form â€” node is PID 1)
CMD ["node", "server.js"]
```

### 5.2 Add a `HEALTHCHECK`

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD wget --quiet --tries=1 --spider http://localhost:3000/health || exit 1
```

Orchestrators (ECS, Kubernetes) use this to detect unhealthy containers and restart them.

### 5.3 Handle Signals Properly

If your application is PID 1 in the container (which it should be with exec form `CMD`), it must handle `SIGTERM` to allow graceful shutdown. Most web frameworks do this; some (especially Java apps) need explicit configuration.

If your app does not handle signals well, prepend `tini` or `dumb-init` as PID 1:

```dockerfile
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["node", "server.js"]
```

### 5.4 Set `WORKDIR` Instead of `cd`

```dockerfile
# Bad
RUN cd /app && npm install

# Good
WORKDIR /app
RUN npm install
```

`WORKDIR` persists across instructions; `cd` only works within the same `RUN`.

## 6. Reproducibility

### 6.1 Pin Base Image Versions

Already covered in 1.1.

### 6.2 Pin Application Dependencies

- `package-lock.json` (Node.js)
- `requirements.txt` with `==` (Python)
- `Gemfile.lock` (Ruby)
- `go.sum` (Go)

### 6.3 Use `npm ci` / `pip install --require-hashes`

`npm ci` strictly installs from the lockfile. `pip install --require-hashes` verifies package hashes against the lockfile.

### 6.4 Document the Build in the Dockerfile

Add `LABEL` metadata:

```dockerfile
LABEL org.opencontainers.image.source="https://github.com/myorg/myapp"
LABEL org.opencontainers.image.version="1.2.3"
LABEL org.opencontainers.image.revision="${GIT_SHA}"
```

You can use `ARG GIT_SHA=default` and pass `--build-arg GIT_SHA=$(git rev-parse HEAD)` at build time.

## 7. The Production-Ready Dockerfile Checklist

- [ ] `FROM` uses a pinned version, not `latest`.
- [ ] `.dockerignore` exists and excludes `node_modules`, `.git`, `.env`, build artifacts.
- [ ] Multi-stage build separates build tools from runtime.
- [ ] Dependencies are installed before code is copied (for caching).
- [ ] Application runs as a non-root user.
- [ ] `CMD` and `ENTRYPOINT` use exec form.
- [ ] `HEALTHCHECK` is defined.
- [ ] Image is scanned for vulnerabilities in CI.
- [ ] No secrets in any layer.
- [ ] Base image and dependencies are pinned for reproducibility.

---

## 8. Common Mistakes Summary

> [!warning] Top 10 Mistakes
> 1. Using `latest` tag.
> 2. Running as root.
> 3. Baking secrets into the image.
> 4. `COPY . .` before dependency install (breaks caching).
> 5. No `.dockerignore`.
> 6. Shell form `CMD` (breaks signal handling).
> 7. No `HEALTHCHECK`.
> 8. No multi-stage build (huge images).
> 9. Not pinning dependency versions.
> 10. Using `apt-get install` without `--no-install-recommends` and cleanup.
