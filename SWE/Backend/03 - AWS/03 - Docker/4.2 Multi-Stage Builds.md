# 4.2 Multi-Stage Builds

> [!info] Chapter Context
> A typical Docker image ends up bloated because it contains the build tools (compilers, SDKs, dev dependencies) needed to build the application, even though only the runtime artifacts are needed at runtime. **Multi-stage builds** solve this by letting you use multiple `FROM` lines in one Dockerfile: one stage for building, another for running. The final image contains only what the final stage provides.

Related: [[4. The Dockerfile]] | [[4.1 The .dockerignore File]] | [[4.3 Dockerfile Best Practices]] | [[3.1 Image Layers and Storage Drivers]]

---

## 1. The Problem Multi-Stage Solves

Consider a Go web application. To build it, you need:

- The Go toolchain (~500 MB).
- Your source code.
- Build dependencies (cached in `$GOPATH/pkg`).

But to **run** the compiled binary, you only need:

- The compiled binary itself (often <20 MB).
- A minimal OS base (Alpine, ~5 MB).

Without multi-stage, your image contains both — 500 MB of Go toolchain you never use at runtime, plus the binary. The image is 50× larger than necessary.

The same applies to:

- **Java**: Build with Maven/Gradle (huge), run with JRE only (smaller).
- **Node.js**: Build with `devDependencies` (testing, transpilation), run with `dependencies` only.
- **Python**: Build Cython extensions, run with the compiled `.so` files only.
- **Rust**: Build with `cargo` (huge toolchain), run with the binary only.

## 2. The Multi-Stage Syntax

A multi-stage Dockerfile has multiple `FROM` lines. Each `FROM` starts a new, independent stage. You name stages with `AS <name>` and copy artifacts between them with `COPY --from=<name>`.

```dockerfile
# Stage 1: Build
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN go build -o /app/server ./cmd/server

# Stage 2: Runtime
FROM alpine:3.19
RUN adduser -D -u 1001 app
WORKDIR /app
COPY --from=builder /app/server /app/server
USER app
EXPOSE 8080
CMD ["/app/server"]
```

The final image is the **last stage** — `alpine:3.19` plus the 20 MB binary. The Go toolchain, source code, and build dependencies exist only in the builder stage; they are **not** included in the final image.

### 2.1 Naming Stages

```dockerfile
FROM node:18 AS deps
# ...
FROM node:18 AS builder
# ...
FROM nginx:alpine AS runtime
# ...
COPY --from=builder /app/dist /usr/share/nginx/html
```

Naming stages makes the Dockerfile readable. You can also reference stages by their numeric index (0-based), but names are clearer.

### 2.2 Stopping at a Specific Stage

```bash
# Build only up to the "builder" stage (useful for testing the build)
docker build --target builder -t myapp:builder .
```

This is useful for debugging intermediate stages or for running tests in CI.

## 3. Concrete Examples by Ecosystem

### 3.1 Node.js (Frontend with Vite/React)

```dockerfile
# Build stage
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build    # produces /app/dist

# Runtime stage
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

Final image: Nginx + static files. No Node.js, no source code, no `node_modules`. Typically 20–30 MB instead of 800 MB.

### 3.2 Python (with compiled extensions)

```dockerfile
# Build stage
FROM python:3.12 AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user -r requirements.txt
COPY . .
RUN pip install --user .

# Runtime stage
FROM python:3.12-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY --from=builder /app /app
ENV PATH=/root/.local/bin:$PATH
CMD ["python", "-m", "myapp"]
```

Final image: Python slim + installed packages. No build tools, no `gcc`, no `.pyc` caches.

### 3.3 Java (Spring Boot with JReleaser)

```dockerfile
# Build stage
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# Runtime stage
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
```

Final image: JRE only (no JDK, no Maven). Typically 200 MB instead of 800 MB.

### 3.4 Rust

```dockerfile
FROM rust:1.75 AS builder
WORKDIR /app
COPY Cargo.toml Cargo.lock ./
RUN mkdir src && echo "fn main() {}" > src/main.rs && cargo build --release
COPY . .
RUN cargo build --release

FROM debian:bookworm-slim
WORKDIR /app
COPY --from=builder /app/target/release/myapp /app/myapp
CMD ["/app/myapp"]
```

Final image: Debian slim + binary. No Rust toolchain. Often <100 MB instead of 1.5 GB.

## 4. Why Multi-Stage Is Better Than Squashing

Docker has an experimental `--squash` flag that combines all layers into one. Multi-stage is almost always better:

| Feature | `--squash` | Multi-stage |
| :--- | :--- | :--- |
| Reduces image size | Yes, but still includes everything built | Yes, and excludes build tools entirely |
| Removes build tools from image | No (just flattens them) | Yes |
| Removes secrets from intermediate layers | No | Yes |
| Lets you use different base images for build vs. runtime | No | Yes |
| Status | Experimental, may not work everywhere | Stable, widely supported |

## 5. Copying from External Images

`COPY --from` can also reference an external image, not just a previous stage:

```dockerfile
FROM alpine:3.19
COPY --from=nginx:alpine /usr/sbin/nginx /usr/sbin/nginx
COPY --from=nginx:alpine /etc/nginx /etc/nginx
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

This lets you compose an image from pieces of other images.

## 6. BuildKit Secret Mounts

When you need secrets during the build (e.g., an NPM token to install private packages), do **not** use `ARG` — `ARG` values are visible in `docker history`. Instead, use BuildKit secret mounts:

```dockerfile
# syntax=docker/dockerfile:1.4
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm install
```

Build with:

```bash
docker build --secret id=npmrc,src=$HOME/.npmrc -t myapp .
```

The secret is mounted into the build container but is **not** persisted in any layer of the image.

## 7. Common Mistakes

> [!warning] Mistake 1 — Forgetting to Copy a File
> If the application crashes at runtime with "file not found," you probably forgot to `COPY --from=builder` a needed artifact. Multi-stage strips everything not explicitly copied.

> [!warning] Mistake 2 — Copying Source Code into the Runtime Stage
> The whole point of multi-stage is to keep source code out of the runtime image. If you `COPY . .` in the final stage, you have undone the benefit.

> [!warning] Mistake 3 — Using `latest` for the Build Stage
> Pin the build stage's base image too: `FROM golang:1.21.5-alpine3.19 AS builder`. Otherwise, builds are not reproducible.

> [!warning] Mistake 4 — Putting Secrets in `ARG`
> `ARG NPM_TOKEN=xxx` makes the token visible in `docker history`. Use BuildKit `--mount=type=secret` instead.

> [!warning] Mistake 5 — Building with `--no-cache` When Iterating
> Multi-stage builds cache each stage independently. If you `--no-cache`, every stage rebuilds from scratch, including slow dependency installs. Iterate without `--no-cache`; only use it for clean release builds.

---

## 8. Summary Checklist

- [ ] Multi-stage builds use multiple `FROM` lines, each starting an independent stage.
- [ ] The final image is the **last** stage; earlier stages are not included.
- [ ] Use `COPY --from=<stage>` to bring artifacts from earlier stages.
- [ ] Name stages with `AS <name>` for readability.
- [ ] Pin versions for **all** stages, not just the final one.
- [ ] Multi-stage is better than `--squash` for reducing size and removing build tools.
- [ ] Use `--target <stage>` to build up to a specific stage (useful for testing).
- [ ] Use BuildKit secret mounts (`--mount=type=secret`) for secrets during build; never use `ARG`.
