## 1. The Compose CLI (V2)

Compose V2 is a Docker CLI plugin invoked as `docker compose` (with a space). V1 was a standalone binary invoked as `docker-compose` (with a hyphen). All examples below use V2 syntax; V1 is equivalent.

```bash
docker compose <command> [options]
```

---

## 2. Lifecycle Commands

### 2.1 `up`

```bash
docker compose up                              # build, create, start, attach
docker compose up -d                           # detached
docker compose up --build                      # force rebuild
docker compose up backend                      # just one service + deps
docker compose up --scale backend=3            # scale backend to 3 instances
docker compose up --no-deps backend            # just backend, no dependencies
docker compose up --force-recreate             # recreate containers even if config unchanged
docker compose up --renew-anon-volumes         # recreate anonymous volumes
```

### 2.2 `down`

```bash
docker compose down                            # stop + remove containers + networks
docker compose down -v                         # also remove named volumes (destructive!)
docker compose down --rmi all                  # also remove images
docker compose down --rmi local                # remove only images built by Compose (not pulled)
docker compose down --remove-orphans           # remove containers not in current compose file
```

### 2.3 `stop`, `start`, `restart`

```bash
docker compose stop                            # stop all services (containers stay on disk)
docker compose stop backend                    # stop one service
docker compose start                           # restart stopped services
docker compose restart                         # full restart (stop + start)
docker compose restart backend                 # restart one service
```

### 2.4 `pause`, `unpause`

```bash
docker compose pause                           # freeze all container processes
docker compose unpause                         # resume
```

## 3. Inspection Commands

### 3.1 `ps`

```bash
docker compose ps                              # list services
docker compose ps -a                           # include stopped
docker compose ps --services                   # just service names
docker compose ps --format json                # JSON output
```

### 3.2 `logs`

```bash
docker compose logs                            # all services
docker compose logs -f                         # follow live
docker compose logs -f backend                 # one service
docker compose logs --tail 100                 # last 100 lines per service
docker compose logs --since 30m                # last 30 minutes
docker compose logs -t                         # with timestamps
```

### 3.3 `top`

```bash
docker compose top                             # show running processes in each service
docker compose top backend                     # one service
```

### 3.4 `config`

```bash
docker compose config                          # validate + print the merged YAML
docker compose config --services               # list service names after merge
docker compose config --volumes                # list volume names after merge
docker compose config --profiles               # list available profiles
```

Useful for debugging "why is my variable not being substituted?" — `config` shows the final YAML after all substitutions and merges.

### 3.5 `images`

```bash
docker compose images                          # list images used by the services
```

## 4. Building

```bash
docker compose build                           # build all services with a `build:` section
docker compose build backend                   # one service
docker compose build --no-cache                # disable cache
docker compose build --pull                    # pull latest base images
docker compose build --parallel                # build services in parallel
docker compose build --progress plain          # plain text output (no fancy progress bars)
```

## 5. Running One-Off Commands

### 5.1 `run`

```bash
docker compose run backend python manage.py migrate     # run a one-off command in a NEW container
docker compose run --rm backend python manage.py migrate  # remove the container after
docker compose run --no-deps backend npm test            # don't start dependencies
docker compose run -e DEBUG=1 backend bash               # set env var, drop into shell
```

`run` creates a new container with the service's config. It does NOT start dependencies unless you omit `--no-deps`.

### 5.2 `exec`

```bash
docker compose exec backend sh                 # open shell in RUNNING backend
docker compose exec db psql -U postgres        # run psql in running db
docker compose exec -e DEBUG=1 backend bash    # set env var
```

`exec` runs a command in the **already-running** container (like `docker exec`).

### 5.3 Difference Between `run` and `exec`

| Command | Container | When to use |
| :--- | :--- | :--- |
| `run` | New container | One-off tasks (migrations, tests, scripts). |
| `exec` | Running container | Debugging, inspecting state, running a quick command. |

## 6. Profiles

Profiles let you define services that only start when explicitly requested.

### 6.1 Defining Profiles

```yaml
services:
  db:
    # no profile -> always starts
  backend:
    # no profile -> always starts
  debug-tools:
    profiles: ["debug"]
    image: nicolaka/netshoot
  load-test:
    profiles: ["test"]
    image: grafana/k6
  metrics:
    profiles: ["debug", "monitoring"]
    image: prom/prometheus
```

### 6.2 Using Profiles

```bash
docker compose up                                # starts only non-profile services
docker compose --profile debug up                # starts non-profile + debug
docker compose --profile debug --profile monitoring up   # both
docker compose --profile test run load-test run script.js
```

### 6.3 When to Use Profiles

- **Debug tools** — netshoot, tcpdump, etc.
- **Test runners** — load tests, integration tests.
- **Monitoring stack** — Prometheus, Grafana (only when you want to inspect metrics).
- **Database admin tools** — pgadmin, adminer.

## 7. Scaling

```bash
docker compose up --scale backend=3             # 3 backend containers
docker compose up --scale backend=3 --scale worker=2
```

> [!warning] Scaling and Port Conflicts
> If a service publishes a host port (`ports: - "8000:8000"`), you cannot scale it beyond 1 — the second instance cannot bind to the same host port. Either:
> - Remove the `ports` section (the service is reachable only on the internal network).
> - Use an internal port mapping (`- "8000"` lets Docker pick a random host port).
> - Put a load balancer in front.

## 8. Project Names

```bash
docker compose -p staging up -d                 # project name: staging
docker compose -p production up -d              # project name: production
```

All resources (containers, networks, volumes) are prefixed with the project name. This lets you run multiple instances of the same compose file on the same host.

You can also set the project name in the compose file:

```yaml
name: myapp
services:
  # ...
```

Or via environment variable: `COMPOSE_PROJECT_NAME=myapp`.

## 9. Working with Multiple Compose Files

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
docker compose -f docker-compose.yml -f docker-compose.override.yml config
```

You can set `COMPOSE_FILE` env var to avoid typing `-f` repeatedly:

```bash
export COMPOSE_FILE=docker-compose.yml:docker-compose.prod.yml
docker compose up -d
```

## 10. Common Everyday Workflows

### 10.1 Start Dev Environment

```bash
cd ~/projects/myapp
docker compose up -d
docker compose logs -f backend
```

### 10.2 Run Migrations

```bash
docker compose run --rm backend python manage.py migrate
```

### 10.3 Open a Database Shell

```bash
docker compose exec db psql -U admin -d myapp
```

### 10.4 Inspect a Container

```bash
docker compose exec backend sh
$ ls /app
$ cat /etc/hosts
$ env
```

### 10.5 Tear Down and Restart

```bash
docker compose down                  # stop and remove containers (keeps volumes)
docker compose up -d --build         # rebuild and start
```

### 10.6 Full Reset (Lose All Data!)

```bash
docker compose down -v               # also remove volumes
docker compose up -d --build
```

### 10.7 View Resource Usage

```bash
docker compose stats                 # live CPU/mem for all services
```

## 11. Common Mistakes

> [!warning] Mistake 1 — Using `docker-compose` (V1) When V2 Is Installed
> V2 is `docker compose` (space). V1 is `docker-compose` (hyphen). If you have both, prefer V2.

> [!warning] Mistake 2 — Running `up` Without `-d` and Being Stuck
> Without `-d`, your terminal is attached to all services' logs. Ctrl+C stops everything. Use `-d` for long-running services.

> [!warning] Mistake 3 — Forgetting `--rm` with `run`
> `docker compose run backend bash` creates a container that stays around after you exit. Use `--rm` for one-off commands.

> [!warning] Mistake 4 — Using `exec` When You Meant `run`
> `exec` requires the container to be running. `run` creates a new container. For migrations, use `run`.

> [!warning] Mistake 5 — Scaling Services with Host Ports
> Cannot scale beyond 1 if the service has `ports: - "8000:8000"`. Use random host ports or no port mapping.

## 12. Summary Checklist

- [ ] Compose V2 is `docker compose` (space); V1 is `docker-compose` (hyphen).
- [ ] `up -d` starts in background; `down` stops + removes containers + networks.
- [ ] `down -v` is destructive (removes volumes).
- [ ] `run` creates a new container; `exec` runs in a running container.
- [ ] `config` validates and shows the merged YAML after substitution.
- [ ] Profiles let you define optional services (debug, test, monitoring tools).
- [ ] `--scale service=N` scales a service, but cannot exceed 1 if it publishes a fixed host port.
- [ ] Project names (`-p`) prefix all resources, enabling multiple instances on one host.
- [ ] Multiple compose files (`-f`) merge: scalars replaced, maps merged, lists concatenated.
