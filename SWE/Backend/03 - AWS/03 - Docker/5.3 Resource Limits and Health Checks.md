# 5.3 Resource Limits and Health Checks

> [!info] Chapter Context
> Without limits, a single misbehaving container can consume all CPU and memory on a host, starving everything else. Without health checks, Docker cannot tell whether your application is actually serving traffic or just hung. This note covers both: how to set cgroup limits via `docker run` flags, and how to define `HEALTHCHECK` so orchestrators know when to restart your container.

Related: [[5. Container Lifecycle and Management]] | [[5.2 Port Mapping and Environment Variables]] | [[1.1 Container Isolation Internals]] | [[8. Docker Security]]

## 1. Resource Limits — Why They Matter

On a Linux host, all processes share the same CPU and memory. Without cgroups, a Node.js app that infinite-loops would consume 100% of one CPU core forever. A Java app with a memory leak would consume all RAM until the kernel's OOM killer fired — possibly killing your database instead of the misbehaving app.

Cgroups (control groups) are the kernel feature that limits how much CPU, memory, I/O, and other resources a process (or group of processes) can use. Docker exposes these limits via `docker run` flags.

In production (ECS, EKS, Fargate), you must always specify resource limits. A task definition without limits is a recipe for noisy-neighbor problems and unpredictable performance.

## 2. Memory Limits

### 2.1 The `--memory` Flag

```bash
docker run -d --memory=512m nginx
docker run -d --memory=1g --memory-swap=2g nginx
docker run -d --memory=512m --memory-reservation=256m nginx
```

| Flag | Meaning |
| :--- | :--- |
| `--memory` (`-m`) | Hard limit. Container cannot exceed this. |
| `--memory-reservation` | Soft limit. Kernel tries to keep the container under this when there's memory pressure. |
| `--memory-swap` | Total memory + swap the container can use. Must be ≥ `--memory`. |
| `--memory-swappiness` | 0–100. How willing the kernel is to swap this container's pages. 0 = never. |
| `--oom-kill-disable` | Disable OOM killer for this container (dangerous). |
| `--oom-score-adj` | -1000 to 1000. Lower = less likely to be killed by OOM. |

### 2.2 What Happens When the Container Hits the Limit

When a container's memory usage approaches the hard limit:

1. The kernel reclaims cache pages (file-backed memory) from the container.
2. If still over, the kernel triggers the OOM killer.
3. The OOM killer picks the process inside the container using the most memory and sends it SIGKILL.
4. The container exits with code 137.

> [!tip] Exit Code 137 Means OOM-Killed
> If your container keeps dying with exit code 137, it is almost certainly being OOM-killed. Check `docker inspect <container> --format '{{.State.OOMKilled}}'` to confirm. The fix is to either increase `--memory` or fix the memory leak in your app.

### 2.3 Setting Both Memory and Swap

If you set `--memory=512m` without `--memory-swap`, the swap limit defaults to 2× memory (so 1 GB total). If you want to disable swap entirely:

```bash
docker run -d --memory=512m --memory-swap=512m nginx
# memory-swap equals memory, so 0 swap allowed
```

If you set `--memory-swap=-1`, the container can use unlimited swap.

## 3. CPU Limits

### 3.1 The `--cpus` Flag

```bash
docker run -d --cpus=1.5 nginx          # 1.5 CPU cores
docker run -d --cpus=2 nginx            # 2 CPU cores
docker run -d --cpus=0.5 nginx          # half a CPU core
```

`--cpus` is a fractional limit. On a 4-core host, `--cpus=2.5` means the container can use at most 2.5 cores worth of CPU time per scheduling period.

### 3.2 CPU Shares (Relative Weight)

```bash
docker run -d --cpu-shares=512 nginx    # default is 1024
docker run -d --cpu-shares=2048 nginx   # twice the default
```

CPU shares are a **relative** weight. They only matter when there's CPU contention. If two containers have shares 512 and 1024 and the host is busy, they get 1/3 and 2/3 of CPU. If the host is idle, both can use 100% of available CPU.

### 3.3 Pinning to Specific Cores

```bash
docker run -d --cpuset-cpus=0,1 nginx   # only cores 0 and 1
docker run -d --cpuset-cpus=0-3 nginx   # cores 0 through 3
```

Useful for performance-sensitive workloads where you want to pin a container to specific cores for cache locality.

### 3.4 How CPU Limits Are Enforced

The kernel uses the Completely Fair Scheduler (CFS) with bandwidth control. The container is allowed to use up to its quota per period (default period is 100ms). If `--cpus=1.5`, the quota is 150ms per 100ms period. If the container uses 150ms of CPU in the first 50ms of the period, it is **throttled** for the remaining 50ms.

> [!warning] CPU Throttling Is Silent
> Unlike memory OOM, CPU throttling does not crash the container. It just runs slower. Your P99 latency might spike with no obvious error. If your app is slow in production but fast locally, check for CPU throttling.

### 3.5 Verifying Limits Inside the Container

```bash
# Inside a cgroup v2 container
cat /sys/fs/cgroup/cpu.max
# Output: 150000 100000  (means quota=150ms, period=100ms, i.e., 1.5 CPUs)

cat /sys/fs/cgroup/memory.max
# Output: 536870912  (512 MB in bytes)
```

## 4. Other Resource Limits

### 4.1 PIDs

```bash
docker run -d --pids-limit=200 nginx
```

Limits the number of processes (and threads) the container can create. Prevents fork bombs from crashing the host.

### 4.2 Block I/O

```bash
docker run -d --device-read-bps=/dev/sda:10mb nginx
docker run -d --device-write-iops=/dev/sda:100 nginx
```

Limits disk read/write throughput or IOPS.

### 4.3 Network

```bash
docker run -d --network=my-net --network-alias=web nginx
```

Not a "limit" per se, but lets you control the container's network namespace. See [[6. Docker Networking]].

## 5. Health Checks

### 5.1 The Problem

A container can be "running" (PID 1 is alive) but not actually serving traffic. For example:

- A web server that has crashed internally but its process is still alive.
- A database that is in a deadlock.
- An app that is stuck in an infinite loop and not accepting connections.

Without a health check, Docker (and orchestrators like ECS, Kubernetes) cannot distinguish "running and healthy" from "running but hung." They will not restart the container.

### 5.2 The `HEALTHCHECK` Instruction

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD wget --quiet --tries=1 --spider http://localhost:3000/health || exit 1
```

| Flag | Default | Meaning |
| :--- | :--- | :--- |
| `--interval` | 30s | How often to run the check. |
| `--timeout` | 30s | How long to wait for the check to return. |
| `--start-period` | 0s | Grace period before failures count (for slow-starting apps). |
| `--retries` | 3 | Number of consecutive failures before marking unhealthy. |

The check command:

- Exits 0 → healthy.
- Exits 1 → unhealthy.
- Exits 2 → reserved (do not use).

### 5.3 Health Status

```bash
docker ps
# STATUS column shows: "Up 5 minutes (healthy)" or "Up 5 minutes (unhealthy)"

docker inspect web --format '{{.State.Health.Status}}'
# healthy / unhealthy / starting
```

### 5.4 Overriding Health Checks at Runtime

```bash
docker run --health-cmd="curl -f http://localhost/ || exit 1" \
           --health-interval=10s \
           --no-healthcheck \
           myapp
```

`--no-healthcheck` disables any `HEALTHCHECK` from the image.

### 5.5 Health Check Best Practices

- The endpoint should be **fast** (<100ms). A slow health check degrades performance.
- The endpoint should test the application's real dependencies (database, cache) — not just "the process is alive."
- A `/health` endpoint that always returns 200 is useless. It should return 200 only if the app can actually serve traffic.
- Distinguish **liveness** (is the app hung?) from **readiness** (is the app ready to receive traffic?). Kubernetes has separate probes for these; Docker's `HEALTHCHECK` is closer to liveness.

### 5.6 Health Check Example: Database

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --retries=5 \
  CMD pg_isready -U postgres || exit 1
```

For PostgreSQL, `pg_isready` is the official health check tool.

### 5.7 Health Check Example: Redis

```dockerfile
HEALTHCHECK --interval=10s --timeout=3s --retries=3 \
  CMD redis-cli ping | grep -q PONG || exit 1
```

## 6. Common Student Mistakes

> [!warning] Mistake 1 — Setting Memory Limits Too Low
> A Node.js app that runs fine with 1 GB might crash at 256 MB. Set the limit 20–30% above the app's peak usage.

> [!warning] Mistake 2 — Setting CPU Limits Too Low
> A `--cpus=0.1` limit means the container can only use 10ms of CPU per 100ms. Even a fast web framework will struggle. Start with `--cpus=0.5` for low-traffic apps and scale up.

> [!warning] Mistake 3 — No Health Check
> Without `HEALTHCHECK`, Docker cannot restart hung containers. Always define one for production images.

> [!warning] Mistake 4 — Health Check That Always Returns 200
> `HEALTHCHECK CMD curl -f http://localhost/` is useless if the app always returns 200 even when broken. Make the endpoint test real functionality.

> [!warning] Mistake 5 — Health Check Too Aggressive
> `--interval=1s --retries=1` will mark the container unhealthy on a single missed check. Use a sensible interval (10–30s) and retries (3+).

> [!warning] Mistake 6 — Forgetting `--start-period` for Slow Apps
> A Spring Boot app might take 30 seconds to start. Without `--start-period`, health check failures during startup will count and may mark the container unhealthy prematurely.

## 7. Summary Checklist

- [ ] `--memory` sets the hard memory limit; exceeding it triggers OOM kill (exit 137).
- [ ] `--cpus` is a fractional CPU limit (e.g., `--cpus=1.5` = 1.5 cores).
- [ ] CPU throttling is silent — the container runs slower, not crashes.
- [ ] `--pids-limit` prevents fork bombs.
- [ ] `HEALTHCHECK` lets Docker detect hung containers.
- [ ] Health check exit codes: 0 = healthy, 1 = unhealthy.
- [ ] Use `--start-period` for slow-starting apps.
- [ ] Always set resource limits and health checks in production.
