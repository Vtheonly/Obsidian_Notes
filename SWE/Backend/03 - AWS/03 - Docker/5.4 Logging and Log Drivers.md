# 5.4 Logging and Log Drivers

> [!info] Chapter Context
> `docker logs` shows you the stdout/stderr of a container, but where do those logs actually go? This note explains Docker's logging drivers, why the default `json-file` driver fills up disks, and how to ship logs to external systems (syslog, journald, Fluentd, CloudWatch).

Related: [[5. Container Lifecycle and Management]] | [[5.3 Resource Limits and Health Checks]] | [[6. Docker Networking]]

---

## 1. How Container Logging Works

A container's main process writes to two streams:

- **STDOUT** (file descriptor 1) — normal output.
- **STDERR** (file descriptor 2) — error output.

Docker captures these streams and routes them through a **logging driver**. The driver decides where the logs go: a JSON file on disk, syslog, journald, an external log aggregator, or nowhere.

```bash
docker logs web
```

This command asks the current logging driver for the container's logs. Some drivers (like `json-file`) support this; others (like `syslog`) do not — you have to read the logs from the destination system instead.

---

## 2. The Default Driver: `json-file`

By default, Docker uses the `json-file` driver. Each container gets a file at `/var/lib/docker/containers/<container-id>/<container-id>-json.log`. Each line is a JSON object with the log message, stream, and timestamp.

### 2.1 The Disk-Fill Problem

Without rotation, this file grows forever. A chatty app can fill your disk in hours. The fix is to configure log rotation in `/etc/docker/daemon.json`:

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

This keeps at most 3 files of 10 MB each per container (30 MB max). Restart Docker after changing this: `sudo systemctl restart docker`. The new settings apply to newly created containers only.

### 2.2 Per-Container Log Options

```bash
docker run -d \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx
```

### 2.3 Reading `json-file` Logs

```bash
docker logs web                    # all logs
docker logs -f web                 # follow (live)
docker logs --tail 100 web         # last 100 lines
docker logs --since 30m web        # last 30 minutes
docker logs --since 2024-01-01T00:00:00 web
docker logs -t web                 # with timestamps
```

## 3. Other Built-in Drivers

| Driver | Destination | `docker logs` works? |
| :--- | :--- | :--- |
| `json-file` (default) | JSON file on disk | Yes |
| `local` | Custom binary format on disk, with built-in rotation | Yes |
| `syslog` | Syslog daemon | No |
| `journald` | systemd journal | No (use `journalctl`) |
| `none` | Discarded | No |
| `fluentd` | Fluentd daemon | No |
| `gelf` | Graylog Extended Log Format | No |
| `awslogs` | AWS CloudWatch Logs | No |
| `gcplogs` | Google Cloud Logging | No |
| `splunk` | Splunk HTTP Event Collector | No |

### 3.1 The `none` Driver

```bash
docker run -d --log-driver none nginx
```

Discards all logs. Useful for high-throughput containers where you do not need logs (e.g., a sidecar that proxies traffic).

### 3.2 The `local` Driver

```bash
docker run -d --log-driver local --log-opt max-size=10m nginx
```

Stores logs in a custom binary format that is more efficient than JSON. Built-in rotation (default 100 MB × 5 files). `docker logs` works.

### 3.3 The `awslogs` Driver (for AWS)

```bash
docker run -d \
  --log-driver awslogs \
  --log-opt awslogs-region=us-east-1 \
  --log-opt awslogs-group=myapp-logs \
  --log-opt awslogs-stream=web-1 \
  nginx
```

Ships logs directly to CloudWatch Logs. This is how ECS tasks send their logs to CloudWatch in AWS.

### 3.4 The `fluentd` Driver

```bash
docker run -d \
  --log-driver fluentd \
  --log-opt fluentd-address=localhost:24224 \
  nginx
```

Sends logs to a Fluentd daemon (which can then forward to Elasticsearch, S3, Kafka, etc.). Common in Kubernetes clusters.

## 4. Structured Logging

The `json-file` driver wraps each line in JSON, but the **content** of the line is whatever the app printed. If your app prints:

```
2024-01-15 10:30:45 INFO User logged in user_id=123
```

That whole string becomes the `log` field in the JSON. To get truly structured logs, your app should print JSON itself:

```json
{"timestamp":"2024-01-15T10:30:45Z","level":"INFO","message":"User logged in","user_id":123}
```

Then log aggregators (ELK, Loki, CloudWatch Logs Insights) can parse the fields and let you query them.

> [!tip] Configure Your Logger for JSON
> Most modern loggers (Winston for Node.js, structlog for Python, logrus for Go) can be configured to output JSON. Do this in production — it makes log queries vastly more powerful.

---

## 5. The `--tag` Option

```bash
docker run -d \
  --log-driver json-file \
  --log-opt tag="{{.Name}}/{{.ID}}" \
  nginx
```

The `tag` option lets you customize how the container is identified in logs. Useful for distinguishing containers in shared log streams.

Available template variables:

- `{{.ID}}` — Full container ID.
- `{{.IDShort}}` — First 12 chars of ID.
- `{{.Name}}` — Container name.
- `{{.ImageName}}` — Image name.

## 6. Multi-Line Logs

Some applications (especially Java stack traces) produce log entries that span multiple lines:

```
Exception in thread "main" java.lang.NullPointerException
    at com.example.MyApp.main(MyApp.java:42)
    at sun.reflect.NativeMethodAccessorImpl.invoke0(Native Method)
```

Docker treats each line as a separate log entry, which breaks stack traces into pieces in the JSON file. Solutions:

- Use a logging framework that combines multi-line entries into a single log event before printing.
- Use a log aggregator (Fluentd, Filebeat) with multi-line detection rules.

---

## 7. Viewing Logs in Production

In production, you typically do not SSH into hosts to read `docker logs`. Instead:

- **ECS/Fargate** — Use the `awslogs` driver; logs go to CloudWatch Logs. Query them via the AWS Console or `aws logs`.
- **EKS** — Use Fluent Bit (DaemonSet) to ship container logs to CloudWatch, Elasticsearch, or Loki.
- **Self-hosted** — Run Fluentd/Filebeat as a sidecar or DaemonSet, ship to ELK/Loki/Splunk.

The principle: container logs should be shipped to a central system, not read locally. `docker logs` is a development convenience; production uses aggregated logging.

## 8. Common Student Mistakes

> [!warning] Mistake 1 — Letting `json-file` Logs Grow Unbounded
> The default driver has no rotation. Always configure `max-size` and `max-file` in `daemon.json` or per container. A single chatty container can fill a host's disk in hours.

> [!warning] Mistake 2 — Writing Logs to Files Inside the Container
> If your app writes to `/var/log/myapp.log`, those logs are inside the container's writable layer and are lost when the container is removed. Write to STDOUT/STDERR instead — Docker captures them and routes them through the logging driver.

> [!warning] Mistake 3 — Expecting `docker logs` to Work with `syslog` Driver
> Only `json-file`, `local`, and `journald` drivers support `docker logs`. For other drivers, read logs from the destination system (syslog, CloudWatch, etc.).

> [!warning] Mistake 4 — Mixing stdout and stderr
> Most apps write errors to stderr and normal output to stdout. `docker logs` shows both by default. Use `docker logs web 2>/dev/null` to see stdout only, or `docker logs web 2>&1 1>/dev/null` for stderr only.

> [!warning] Mistake 5 — Forgetting `--since` for Long Log Histories
> `docker logs web` can return millions of lines. Use `--since 5m` or `--tail 100` to limit output.

## 9. Summary Checklist

- [ ] Docker captures STDOUT and STDERR and routes them through a logging driver.
- [ ] The default driver `json-file` writes to `/var/lib/docker/containers/<id>/<id>-json.log`.
- [ ] Always configure log rotation (`max-size`, `max-file`) — the default has none.
- [ ] Other drivers: `local`, `syslog`, `journald`, `fluentd`, `awslogs`, `none`.
- [ ] `docker logs` only works with `json-file`, `local`, and `journald` drivers.
- [ ] For production, ship logs to a central system (CloudWatch, ELK, Loki, Splunk).
- [ ] Apps should write to STDOUT/STDERR (not files), in JSON format for structured logging.
