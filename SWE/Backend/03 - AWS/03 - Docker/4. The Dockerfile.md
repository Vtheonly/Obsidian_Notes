# 4. The Dockerfile

> [!info] Chapter Context
> A Dockerfile is the recipe that produces a Docker image. This note covers every common Dockerfile instruction, the difference between `RUN`/`CMD`/`ENTRYPOINT`, layer caching, and the build context. After this, you will be able to read and write Dockerfiles fluently.

Related: [[3.1 Image Layers and Storage Drivers]] | [[3.2 Volumes and Bind Mounts]] | [[4.1 The .dockerignore File]] | [[4.2 Multi-Stage Builds]] | [[4.3 Dockerfile Best Practices]]

## 1. What Is a Dockerfile

A Dockerfile is a plain-text file named exactly `Dockerfile` (no extension) that contains a sequence of instructions. Docker reads these instructions top-to-bottom and builds an image layer by layer. The Dockerfile is the recipe; the image is the cake baked from it.

A minimal Dockerfile looks like this:

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

Each line beginning with an uppercase keyword (`FROM`, `WORKDIR`, `COPY`, etc.) is an instruction. Each instruction creates either a new layer (`RUN`, `COPY`, `ADD`) or modifies the image metadata (`WORKDIR`, `ENV`, `EXPOSE`, `CMD`, etc.).

---

## 2. The Build Command

```bash
docker build -t myapp:1.0 .
```

Breaking this down:

- `docker build` — The build command.
- `-t myapp:1.0` — Tag the resulting image as `myapp:1.0`. You can specify multiple tags with multiple `-t` flags.
- `.` — The **build context**. This is the directory Docker sends to the daemon before building. All `COPY` and `ADD` instructions can only reference files inside this directory.

> [!warning] The Build Context Can Be Huge
> If your build context is your home directory, Docker sends GB of files to the daemon before it even starts building. Always run `docker build` from a small, focused directory and use `.dockerignore` (see [[4.1 The .dockerignore File]]) to exclude unwanted files.

You can also specify a Dockerfile at a different path:

```bash
docker build -t myapp:1.0 -f docker/Dockerfile.prod .
```

## 3. Dockerfile Instructions — The Comprehensive List

### 3.1 `FROM` — The Base Image

Every Dockerfile must start with `FROM`. It specifies the parent image to build on.

```dockerfile
FROM node:18-alpine
```

You can use any image — official images from Docker Hub (`node`, `python`, `nginx`), private registry images (`registry.example.com/mybase:1.0`), or scratch (`FROM scratch` for an empty image, used to build truly minimal images).

Multi-stage builds use multiple `FROM` lines, each starting a new stage (see [[4.2 Multi-Stage Builds]]).

### 3.2 `WORKDIR` — Set Working Directory

```dockerfile
WORKDIR /app
```

Sets the working directory for all subsequent `RUN`, `CMD`, `ENTRYPOINT`, `COPY`, and `ADD` instructions. If the directory does not exist, Docker creates it. Subsequent `COPY . .` will copy files into `/app`.

> [!tip] Always Use `WORKDIR` Instead of `cd`
> Do not write `RUN cd /app && npm install`. Use `WORKDIR /app` followed by `RUN npm install`. `WORKDIR` is more reliable, persists across instructions, and is clearer in `docker history`.

### 3.3 `COPY` — Copy Files Into the Image

```dockerfile
COPY package.json .
COPY ./src /app/src
COPY *.py /app/
```

Copies files or directories from the **build context** (the host directory passed to `docker build`) into the image. The destination is an absolute path inside the image (relative to `WORKDIR` if not absolute).

### 3.4 `ADD` — Like `COPY` but with Magic

`ADD` does everything `COPY` does, plus:

- Auto-extracts tarballs: `ADD rootfs.tar.gz /` extracts the tarball into `/`.
- Can fetch URLs: `ADD https://example.com/file.txt /app/`.

> [!warning] Prefer `COPY` over `ADD`
> `ADD`'s magic (especially auto-extraction) is surprising and has caused many bugs. The official Docker best practice is to use `COPY` for plain file copies and reserve `ADD` for the rare case where you actually need tarball extraction or URL fetching.

### 3.5 `RUN` — Execute a Command at Build Time

```dockerfile
RUN apt-get update && apt-get install -y curl
RUN npm install
RUN pip install -r requirements.txt
```

`RUN` executes a command **during the build**, and the resulting filesystem change becomes a new layer in the image. This is where you install packages, compile code, generate files, etc.

There are two forms:

- **Shell form** (default): `RUN apt-get install -y curl` runs as `/bin/sh -c "apt-get install -y curl"`.
- **Exec form**: `RUN ["apt-get", "install", "-y", "curl"]` runs the binary directly, no shell. Use this when you need JSON-escaping or when there is no shell available (e.g., scratch-based images).

> [!tip] Combine `RUN` Commands
> Each `RUN` creates a layer. Combining `apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*` into one `RUN` keeps the layer small. Doing them in separate `RUN` instructions creates bloated layers because the `rm` in a later layer does not erase files from an earlier layer.

### 3.6 `CMD` — Default Command at Runtime

```dockerfile
CMD ["node", "server.js"]
CMD node server.js              # shell form
CMD ["node", "server.js", "--port", "3000"]  # exec form
```

`CMD` specifies the default command to run when the container starts. It does **not** execute during the build. There can be only one `CMD` per Dockerfile; if you specify multiple, only the last one takes effect.

`CMD` can be overridden from the command line:

```bash
docker run myapp node --inspect    # overrides CMD
```

### 3.7 `ENTRYPOINT` — The Executable

`ENTRYPOINT` specifies the executable that always runs when the container starts. Unlike `CMD`, it is **not** overridden by command-line arguments — instead, command-line arguments are **appended** to `ENTRYPOINT`.

A common pattern is to use `ENTRYPOINT` for the executable and `CMD` for default arguments:

```dockerfile
ENTRYPOINT ["node"]
CMD ["server.js"]
```

- `docker run myapp` runs `node server.js`.
- `docker run myapp --inspect` runs `node --inspect` (overrides the `CMD` but not `ENTRYPOINT`).

### 3.8 `ENV` — Environment Variables

```dockerfile
ENV NODE_ENV=production
ENV PATH="/usr/local/myapp/bin:$PATH"
```

Sets environment variables that persist in the image and are available to all subsequent instructions and to the running container.

### 3.9 `EXPOSE` — Document a Port

```dockerfile
EXPOSE 3000
EXPOSE 80 443
```

`EXPOSE` is **documentation only**. It does not actually publish the port — you still need `-p 3000:3000` on `docker run`. However, it tells Docker Desktop and orchestration systems which ports the container intends to listen on.

### 3.10 `USER` — Switch User

```dockerfile
RUN adduser -D appuser
USER appuser
```

Sets the user (and optionally group) for subsequent `RUN`, `CMD`, and `ENTRYPOINT` instructions. By default, containers run as root; switching to a non-root user is a critical security best practice.

### 3.11 `VOLUME` — Declare a Mount Point

```dockerfile
VOLUME /var/lib/postgresql/data
```

Declares that the given path should be persisted. If a user runs the container without `-v`, Docker creates an anonymous volume. See [[3.2 Volumes and Bind Mounts]] for details.

### 3.12 `ARG` — Build-Time Variable

```dockerfile
ARG NODE_VERSION=18
FROM node:${NODE_VERSION}-alpine
```

`ARG` defines a variable that can be set at build time with `--build-arg`:

```bash
docker build --build-arg NODE_VERSION=20 -t myapp .
```

`ARG` values are visible in `docker history` — do not use them for secrets.

### 3.13 `HEALTHCHECK` — Container Health

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1
```

Tells Docker how to check whether the container is healthy. Docker runs the check periodically; if it fails, the container is marked unhealthy. Orchestration systems (ECS, Kubernetes) use this to restart unhealthy containers.

### 3.14 `LABEL` — Metadata

```dockerfile
LABEL maintainer="dev@example.com"
LABEL version="1.0"
LABEL description="My FastAPI app"
```

Adds key-value metadata to the image. Inspect with `docker inspect <image>`.

## 4. `RUN` vs `CMD` vs `ENTRYPOINT` — The Big Confusion

This is the most confusing part of Dockerfiles for beginners. The table below clarifies:

| Instruction | When it runs | Purpose | Overridable by `docker run` args? |
| :--- | :--- | :--- | :--- |
| `RUN` | Build time | Install packages, create files. Each `RUN` makes a layer. | N/A (already happened) |
| `CMD` | Run time (container start) | Default command + args. | Yes — `docker run myapp other-cmd` replaces it. |
| `ENTRYPOINT` | Run time (container start) | The executable that always runs. | No — args are appended. |

### 4.1 The Three Common Patterns

**Pattern A — `CMD` only (simplest):**
```dockerfile
CMD ["node", "server.js"]
```
Easy to override: `docker run myapp sh`.

**Pattern B — `ENTRYPOINT` only (fixed executable):**
```dockerfile
ENTRYPOINT ["my-app"]
```
Container always runs `my-app`. Args are appended: `docker run myapp --debug` runs `my-app --debug`.

**Pattern C — `ENTRYPOINT` + `CMD` (flexible):**
```dockerfile
ENTRYPOINT ["node"]
CMD ["server.js"]
```
Default: `node server.js`. Override args: `docker run myapp --inspect` runs `node --inspect`.

> [!warning] Shell Form vs Exec Form
> `CMD node server.js` (shell form) runs as `/bin/sh -c "node server.js"`. PID 1 is `/bin/sh`, not `node`. Signals (SIGTERM) go to `/bin/sh`, which does not forward them to `node`. The container cannot be gracefully stopped.
>
> `CMD ["node", "server.js"]` (exec form) runs `node` directly as PID 1. Signals work correctly.
>
> **Always use exec form for `CMD` and `ENTRYPOINT`.**

## 5. Layer Caching — How to Make Builds Fast

Docker caches each layer. If you rebuild an image and an instruction's inputs have not changed (same base image, same files being copied, same command), Docker reuses the cached layer instead of re-running the instruction.

The cache is invalidated for an instruction and **all subsequent instructions** as soon as one instruction's inputs change.

### 5.1 The Bad Order

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY . .              # any code change invalidates this layer
RUN npm install       # so this layer also gets re-run (slow!)
CMD ["node", "server.js"]
```

Change one line of code → `COPY . .` is invalidated → `RUN npm install` is invalidated → npm install runs from scratch (could take minutes).

### 5.2 The Good Order

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./  # only invalidates if package.json changes
RUN npm install        # cached as long as package.json is unchanged
COPY . .               # code changes only invalidate this layer
CMD ["node", "server.js"]
```

Now, code changes only invalidate the `COPY . .` layer. The expensive `npm install` is cached. Rebuilds take seconds, not minutes.

This pattern — copy the dependency manifest first, install deps, then copy the rest of the code — applies to every ecosystem:

- **Node.js**: `COPY package*.json ./` → `RUN npm install` → `COPY . .`
- **Python**: `COPY requirements.txt .` → `RUN pip install -r requirements.txt` → `COPY . .`
- **Go**: `COPY go.mod go.sum ./` → `RUN go mod download` → `COPY . .`
- **Ruby**: `COPY Gemfile Gemfile.lock ./` → `RUN bundle install` → `COPY . .`

## 6. Build Context and `.dockerignore`

When you run `docker build .`, Docker sends the entire current directory (the build context) to the daemon. This includes:

- `node_modules/` (if present)
- `.git/` (potentially huge)
- `.env` files (potentially containing secrets)
- Build artifacts (`dist/`, `build/`, `*.pyc`)
- IDE config (`.vscode/`, `.idea/`)

Sending all of this is slow and can leak secrets into the image. The solution is `.dockerignore`, covered in detail in [[4.1 The .dockerignore File]].

## 7. A Complete Realistic Example

Here is a production-grade Dockerfile for a Node.js app:

```dockerfile
# Use a specific version, not latest
FROM node:18.19-alpine3.19

# Create a non-root user
RUN addgroup -S app && adduser -S app -G app

# Set working directory
WORKDIR /app

# Copy dependency manifests first (for caching)
COPY package*.json ./

# Install production dependencies only
RUN npm ci --only=production

# Copy application code
COPY --chown=app:app . .

# Switch to non-root user
USER app

# Document the port
EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget --quiet --tries=1 --spider http://localhost:3000/health || exit 1

# Run as PID 1 (exec form)
CMD ["node", "server.js"]
```

Key best practices applied:

- Pinned version (`18.19-alpine3.19`), not `latest`.
- Non-root user.
- `npm ci` instead of `npm install` (faster, reproducible, requires `package-lock.json`).
- `--chown` in `COPY` to set file ownership in one layer.
- `HEALTHCHECK` for orchestration.
- Exec form for `CMD`.

## 8. Common Student Mistakes

> [!warning] Mistake 1 — `RUN` vs `CMD` Confusion
> `RUN npm install` installs dependencies **at build time**. `CMD ["node", "server.js"]` starts the app **at runtime**. Mixing these up is the most common Dockerfile mistake.

> [!warning] Mistake 2 — Using Shell Form for `CMD`
> `CMD node server.js` makes `/bin/sh` PID 1, breaking signal handling. Always use exec form: `CMD ["node", "server.js"]`.

> [!warning] Mistake 3 — Not Pinning Base Image Tags
> `FROM node:latest` means "whatever was latest when I built this." Rebuilding tomorrow might give a different Node version. Pin: `FROM node:18.19-alpine3.19`.

> [!warning] Mistake 4 — Running as Root
> By default, containers run as root. If an attacker escapes the container, they have root on the host (modulo user namespaces). Always create a non-root user and `USER` it before `CMD`.

> [!warning] Mistake 5 — Putting `COPY . .` Before Dependency Install
> This breaks caching. Every code change re-runs `npm install` (or equivalent). Always copy the dependency manifest first.

> [!warning] Mistake 6 — Using `ADD` When You Mean `COPY`
> `ADD app.tar.gz /app` extracts the tarball. `COPY app.tar.gz /app` copies the file as-is. Use `COPY` unless you specifically want extraction.

## 9. Summary Checklist

- [ ] A Dockerfile is a sequence of instructions; each `RUN`/`COPY`/`ADD` creates a layer.
- [ ] `FROM` is mandatory and must be first.
- [ ] `RUN` runs at build time; `CMD`/`ENTRYPOINT` run at container start.
- [ ] `CMD` is overridable; `ENTRYPOINT` is not (args are appended).
- [ ] Always use **exec form** (`CMD ["node", "server.js"]`) for proper signal handling.
- [ ] Order instructions to maximize cache hits: copy dependency manifests first, install, then copy code.
- [ ] Use `.dockerignore` to keep the build context small and to keep secrets out.
- [ ] Create a non-root user and `USER` it before `CMD`.
- [ ] Pin base image versions; never use `latest` in production.
- [ ] Add a `HEALTHCHECK` so orchestrators can detect unhealthy containers.
