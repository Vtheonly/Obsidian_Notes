# 7. Docker Compose

> [!info] Chapter Context
> Running multi-container apps with `docker run` quickly becomes unmanageable â€” you end up with shell scripts full of long `docker run` commands. **Docker Compose** lets you declare your entire application stack in a YAML file and bring it up with a single command. This note covers the YAML structure, common options, and how Compose handles networking, volumes, and dependencies.

Related: [[6. Docker Networking]] | [[3.2 Volumes and Bind Mounts]] | [[7.1 Compose Files in Depth]] | [[7.2 Compose Commands and Profiles]]

## 1. Why Docker Compose

Real applications are multi-container:

- A **frontend** (React static files served by Nginx).
- A **backend** (Python FastAPI or Node.js Express).
- A **database** (PostgreSQL).
- A **cache** (Redis).
- Maybe a **queue** (RabbitMQ), a **search engine** (Elasticsearch), etc.

Managing these with `docker run`:

```bash
docker network create app-net
docker volume create pgdata
docker run -d --name db --network app-net -v pgdata:/var/lib/postgresql/data -e POSTGRES_PASSWORD=secret postgres:15
docker run -d --name cache --network app-net redis:7
docker run -d --name backend --network app-net -p 8000:8000 --depends-on db myapp-backend
docker run -d --name frontend --network app-net -p 3000:80 myapp-frontend
```

That is one mistake away from a broken setup. Compose replaces all of this with a single declarative file.

## 2. The `docker-compose.yml` File

A Compose file describes services, networks, and volumes in YAML. Here is a complete example:

```yaml
version: "3.9"

services:
  db:
    image: postgres:15
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data
    networks:
      - back-net
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U admin"]
      interval: 10s
      timeout: 5s
      retries: 5

  cache:
    image: redis:7-alpine
    networks:
      - back-net
    command: redis-server --save 60 1

  backend:
    build: ./backend
    environment:
      DATABASE_URL: postgres://admin:secret@db:5432/myapp
      REDIS_URL: redis://cache:6379
    ports:
      - "8000:8000"
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    networks:
      - front-net
      - back-net
    volumes:
      - ./backend:/app         # bind mount for live reload
    develop:
      watch:
        - action: rebuild
          path: ./backend

  frontend:
    build: ./frontend
    ports:
      - "3000:80"
    depends_on:
      - backend
    networks:
      - front-net

networks:
  front-net:
  back-net:

volumes:
  pgdata:
```

## 3. The Anatomy of a Compose File

### 3.1 `version`

```yaml
version: "3.9"
```

Specifies the Compose file format version. Modern Compose (V2) does not strictly require this â€” `version` is optional and ignored if omitted. Use `"3.9"` for compatibility, or omit it entirely.

### 3.2 `services`

The core of the file. Each key under `services` defines one container. The key is the **service name** â€” this becomes the DNS name on the Compose network.

### 3.3 `networks`

Declares networks. If you do not declare any, Compose creates one default network and attaches all services to it. If you declare networks, you must explicitly attach each service to the ones it should join.

### 3.4 `volumes`

Declares named volumes. Each volume must be referenced by at least one service's `volumes` section.

## 4. Common Service Options

### 4.1 `image` vs `build`

```yaml
services:
  db:
    image: postgres:15          # pull this image

  backend:
    build: ./backend            # build from ./backend/Dockerfile

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile.prod
      args:
        NODE_ENV: production
```

### 4.2 `ports`

```yaml
ports:
  - "8000:8000"                  # host:container
  - "127.0.0.1:9000:9000"        # bind to localhost only
  - "53:53/udp"                  # UDP
```

### 4.3 `environment`

```yaml
environment:
  DATABASE_URL: postgres://admin:secret@db:5432/myapp
  REDIS_URL: redis://cache:6379
  NODE_ENV: production

# Or from a file
env_file:
  - .env.production
```

### 4.4 `volumes`

```yaml
volumes:
  - pgdata:/var/lib/postgresql/data       # named volume
  - ./backend:/app                         # bind mount
  - ./config:/etc/app:ro                   # bind mount, read-only
```

### 4.5 `depends_on`

```yaml
depends_on:
  - db                                      # wait for db to start (not necessarily be ready)
  - cache

# Long form (Compose 3.9+) for health-based waiting
depends_on:
  db:
    condition: service_healthy
  cache:
    condition: service_started
```

> [!warning] `depends_on` Only Waits for Container Start
> The short form (`depends_on: [db]`) only waits for the `db` container to start â€” not for Postgres to actually be ready to accept connections. Use the long form with `condition: service_healthy` and a `healthcheck` on the database service to wait until it is actually ready.

### 4.6 `networks`

```yaml
networks:
  - front-net
  - back-net
```

Attaches the service to one or more networks.

### 4.7 `command` and `entrypoint`

```yaml
command: ["python", "-m", "debugpy", "--listen", "0.0.0.0:5678", "manage.py", "runserver", "0.0.0.0:8000"]
entrypoint: ["/app/entrypoint.sh"]
```

Overrides the image's `CMD` and `ENTRYPOINT`.

### 4.8 `restart`

```yaml
restart: unless-stopped
```

Same policies as `docker run --restart`.

### 4.9 `healthcheck`

```yaml
healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
  interval: 30s
  timeout: 3s
  retries: 3
  start_period: 10s
```

### 4.10 `user`, `working_dir`, `hostname`

```yaml
user: "1000:1000"
working_dir: /app
hostname: backend-1
```

### 4.11 `deploy` (for Swarm and Compose v2)

```yaml
deploy:
  replicas: 3
  resources:
    limits:
      cpus: "0.5"
      memory: 512M
  restart_policy:
    condition: on-failure
```

The `deploy` key is used by `docker stack deploy` (Swarm). Plain `docker-compose up` ignores most of it, but Compose V2 does apply `resources.limits` and `resources.reservations`.

## 5. The Magic of Compose Networking

When you run `docker-compose up`, Compose:

1. Creates a network named `<project>_default` (where `<project>` is the directory name by default).
2. Attaches all services to this network.
3. Sets up DNS so each service is reachable by its service name.

In the backend code, you connect to the database with:

```python
DATABASE_URL=postgres://admin:secret@db:5432/myapp
#                                ^^
#                       just the service name
```

Not `localhost`, not an IP â€” just `db`. Compose's DNS server resolves it.

### 5.1 Multiple Networks

If you declare multiple networks and assign services to them, Compose creates each network and attaches services accordingly. The frontend service can talk to the backend but not to the database; the backend can talk to both. This is the multi-tier pattern from [[6.1 Custom Networks and DNS]].

## 6. Compose Commands

| Command | What it does |
| :--- | :--- |
| `docker-compose up` | Build (if needed), create, start, and attach to all services. |
| `docker-compose up -d` | Detached â€” run in background. |
| `docker-compose up --build` | Force rebuild images before starting. |
| `docker-compose down` | Stop and remove containers, networks. |
| `docker-compose down -v` | Also remove named volumes (destructive!). |
| `docker-compose stop` | Stop services without removing them. |
| `docker-compose start` | Restart previously stopped services. |
| `docker-compose restart` | Restart all services. |
| `docker-compose ps` | List services and their status. |
| `docker-compose logs -f` | Tail logs from all services. |
| `docker-compose logs -f backend` | Tail logs from one service. |
| `docker-compose exec backend sh` | Open a shell in the backend service. |
| `docker-compose run backend python manage.py migrate` | Run a one-off command in a new container for the backend service. |
| `docker-compose build` | Rebuild all images. |
| `docker-compose build backend` | Rebuild just the backend. |
| `docker-compose config` | Validate and print the final YAML (after variable substitution). |

> [!tip] Compose V1 vs V2
> Compose V1 used `docker-compose` (with hyphen). Compose V2 is a Docker CLI plugin and uses `docker compose` (with space). Both work, but V2 is faster and more integrated. The commands are mostly identical.

## 7. Variable Substitution

Compose supports environment variables in the YAML:

```yaml
services:
  db:
    image: postgres:${POSTGRES_VERSION:-15}    # defaults to 15
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}        # required (errors if unset)
```

Variables come from your shell environment or a `.env` file in the same directory as `docker-compose.yml`.

### 7.1 The `.env` File

```bash
# .env
POSTGRES_VERSION=16
DB_PASSWORD=supersecret
```

Compose automatically loads this file. You can also specify a different file:

```bash
docker-compose --env-file .env.production up -d
```

## 8. Project Names

Compose groups services under a **project name**. By default, the project name is the name of the directory containing the compose file (with hyphens replacing spaces and underscores). All containers, networks, and volumes are prefixed with the project name:

```
myapp-backend-1   (container name: <project>-<service>-<instance>)
myapp-db-1
myapp_default     (network name)
myapp_pgdata      (volume name)
```

Override the project name with `-p`:

```bash
docker-compose -p staging up -d
docker-compose -p production up -d
```

This lets you run multiple instances of the same stack on the same host (e.g., staging and production side-by-side).

## 9. Profiles

Profiles let you define optional services that only start when explicitly requested:

```yaml
services:
  db:
    # always starts
  backend:
    # always starts
  debug-tools:
    profiles: ["debug"]
    # only starts with --profile debug
  load-test:
    profiles: ["test"]
    # only starts with --profile test
```

```bash
docker-compose up                          # starts db and backend only
docker-compose --profile debug up          # starts db, backend, and debug-tools
docker-compose --profile test run load-test
```

Useful for keeping debug and test tools out of normal startup.

## 10. Common Student Mistakes

> [!warning] Mistake 1 â€” Using `localhost` for Inter-Service Communication
> Inside Compose, services do NOT share a localhost. The backend must connect to `db` (the service name), not `localhost`.

> [!warning] Mistake 2 â€” Expecting `depends_on` to Wait for Readiness
> `depends_on: [db]` waits for the db container to start, not for Postgres to accept connections. Use `condition: service_healthy` with a healthcheck.

> [!warning] Mistake 3 â€” Forgetting to Declare Volumes
> If you reference `pgdata:/data` in a service's `volumes` but do not declare `pgdata` under the top-level `volumes` key, Compose will create it as an anonymous volume (or error, depending on version).

> [!warning] Mistake 4 â€” Using `docker-compose down -v` Carelessly
> `-v` removes named volumes. This deletes your database data. Only use `-v` when you want a clean slate.

> [!warning] Mistake 5 â€” Running Multiple Projects with Default Names
> If you have two projects in directories named `backend`, both will use project name `backend`, and their containers will conflict. Use `-p` to give each project a unique name.

> [!warning] Mistake 6 â€” Confusing `docker-compose run` and `docker-compose exec`
> `run` creates a NEW container (with the same config as the service). `exec` runs a command in the already-running container. For migrations, `run` is usually correct (one-off, isolated).

## 11. Summary Checklist

- [ ] Compose defines multi-container apps in a single YAML file.
- [ ] Services, networks, and volumes are the three top-level keys.
- [ ] Compose automatically creates a network and provides DNS by service name.
- [ ] Use `condition: service_healthy` with `healthcheck` to wait for dependency readiness.
- [ ] Variable substitution: `${VAR:-default}` syntax; `.env` file is auto-loaded.
- [ ] `docker-compose up -d` starts in background; `down` stops and removes.
- [ ] `down -v` removes volumes (destructive).
- [ ] Project names default to the directory name; override with `-p`.
- [ ] Profiles let you define optional services (debug, test tools).
- [ ] Compose V2 uses `docker compose` (space); V1 used `docker-compose` (hyphen).
