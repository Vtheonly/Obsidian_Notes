# 10. Debugging and Troubleshooting

> [!info] Chapter Context
> When things go wrong — containers crash, builds fail, networking is broken, performance is bad — you need a systematic way to diagnose the issue. This note is a practical debugging playbook: the commands to run, the order to run them in, and how to interpret the output.

Related: [[5. Container Lifecycle and Management]] | [[5.4 Logging and Log Drivers]] | [[6. Docker Networking]] | [[3.1 Image Layers and Storage Drivers]]

---

## 1. The Debugging Workflow

When a container is misbehaving, follow this sequence:

1. **Check the state** — Is it running? Exited? Restarting?
2. **Check the logs** — What did the application say?
3. **Check the exit code** — How did it die?
4. **Inspect the configuration** — Env vars, mounts, ports, image.
5. **Run a shell inside** — Look at files, test connectivity.
6. **Check the host** — Disk space, network, daemon logs.
7. **Reproduce minimally** — Strip the problem down to the smallest reproducer.

---

## 2. Container State

```bash
docker ps -a                                    # all containers, including stopped
docker ps -a --filter "name=web"                # filter by name
docker ps -a --filter "status=exited"           # only exited
docker inspect web --format '{{.State.Status}}' # current state
docker inspect web --format '{{json .State}}' | jq
```

States to watch for:

- **`created`** — Container exists but never started. Unusual; usually a config issue.
- **`running`** — Healthy (or at least, PID 1 is alive).
- **`restarting`** — Crash loop. Check the logs.
- **`exited`** — Process terminated. Check the exit code.
- **`dead`** — Daemon lost track of the container. Restart the daemon.

---

## 3. Exit Codes

The exit code tells you why the process terminated:

| Code | Meaning | Likely cause |
| :--- | :--- | :--- |
| 0 | Success | Normal exit. |
| 1 | General error | Application error. Check logs. |
| 125 | Docker error | `docker run` itself failed (e.g., invalid flag). |
| 126 | Command not executable | Permission denied on the entrypoint. |
| 127 | Command not found | Wrong path or missing binary. |
| 137 | SIGKILL | Usually OOM-kill. Check `--memory` and `docker inspect .State.OOMKilled`. |
| 139 | SIGSEGV | Segmentation fault. Often a native library bug or wrong architecture. |
| 143 | SIGTERM | Graceful shutdown (e.g., `docker stop` worked). |

```bash
docker inspect web --format '{{.State.ExitCode}}'
docker inspect web --format '{{.State.OOMKilled}}'
docker inspect web --format '{{.State.Error}}'
```

> [!tip] Exit 137 in Production
> If your container keeps dying with exit 137, it is almost always OOM. Either:
> - Increase `--memory`.
> - Fix a memory leak in your app.
> - Use a profiler (e.g., `node --inspect`, `py-spy`) to find what is consuming memory.

## 4. Logs

```bash
docker logs web                                 # all logs
docker logs -f web                              # follow live
docker logs --tail 100 web                      # last 100 lines
docker logs --since 30m web                     # last 30 minutes
docker logs --since 2024-01-15T10:00:00 web
docker logs -t web                              # with timestamps
docker logs web 2>&1 | grep ERROR               # filter for errors
```

> [!warning] "No logs" Does Not Mean No Problem
> If `docker logs` returns nothing, the app may be writing logs to a file instead of STDOUT/STDERR. Either fix the app to log to STDOUT, or `docker exec` in and read the file.

---

## 5. Inspecting Configuration

```bash
docker inspect web
docker inspect web --format '{{json .Config.Env}}' | jq
docker inspect web --format '{{json .Mounts}}' | jq
docker inspect web --format '{{json .NetworkSettings}}' | jq
docker inspect web --format '{{.Config.Image}}'
docker inspect web --format '{{.Config.Cmd}}'
docker inspect web --format '{{.Config.Entrypoint}}'
```

Common things to verify:

- **Env vars** — Did the secret get passed? Is `DATABASE_URL` correct?
- **Mounts** — Is the volume attached? Is the source path correct?
- **Network settings** — Is the container on the right network? What is its IP?
- **Image** — Did you accidentally run the wrong image?
- **Cmd/Entrypoint** — Are they what you expect?

## 6. Running a Shell Inside

```bash
docker exec -it web sh                         # Alpine
docker exec -it web bash                       # Debian/Ubuntu
docker exec -it --user root web bash           # as root (if image runs as non-root)
```

Once inside:

```bash
# Check environment
env
cat /etc/hosts
cat /etc/resolv.conf

# Check files
ls /app
cat /app/config.json

# Check network
ping -c1 db                                    # DNS resolution
nslookup db
curl -v http://backend:8000/health
ss -tlnp                                       # what ports are listening

# Check processes
ps aux
top
```

### 6.1 If the Container Has No Shell

Distroless images have no `sh` or `bash`. You cannot `exec` in. Options:

- Use `docker debug` (Docker Desktop 4.27+) which provides a debug shell.
- Use ephemeral debug containers with `kubectl debug` (in Kubernetes).
- Add a shell temporarily by copying from a debug image:
  ```bash
  docker create --name debug myapp:1.0
  docker cp debug-image:/bin/sh debug:/bin/sh
  docker start debug
  docker exec -it debug /bin/sh
  ```

## 7. Networking Issues

### 7.1 "Connection Refused"

The destination is not listening on the expected port. Check:

```bash
docker exec backend ss -tlnp                   # is the app actually listening?
docker exec backend curl -v http://localhost:8000/   # can it reach itself?
docker exec app nslookup backend               # can the client resolve the name?
docker exec app ping -c1 backend               # can it reach the IP?
docker exec app curl -v http://backend:8000    # can it connect?
```

Common causes:

- The app is listening on `127.0.0.1` instead of `0.0.0.0`. Inside a container, `127.0.0.1` is the container itself — other containers cannot reach it.
- The app is not yet started (race condition). Use healthchecks.
- The port number is wrong (e.g., app listens on 8000, you tried 8080).

### 7.2 "Name Resolution Failed"

```bash
docker exec app cat /etc/resolv.conf           # should show nameserver 127.0.0.11
docker exec app nslookup db                    # try resolving
docker network inspect my-net                  # is db on the same network?
```

If `resolv.conf` does not show `127.0.0.11`, the container is on the default bridge (which has no DNS). Move it to a user-defined network.

### 7.3 "Port Already Allocated"

```bash
sudo lsof -i :8080                             # what is using host port 8080?
sudo ss -tlnp | grep :8080
docker ps --format "table {{.Names}}\t{{.Ports}}"   # which container has it?
```

Either stop the conflicting process/container or use a different host port.

## 8. Disk Space Issues

Docker can fill your disk with old images, stopped containers, and unused volumes.

```bash
docker system df                               # disk usage overview
docker system df -v                            # detailed per-image/container/volume

# Clean up
docker container prune                         # remove stopped containers
docker image prune                             # remove dangling images
docker image prune -a                          # remove ALL unused images
docker volume prune                            # remove unused volumes (CAREFUL)
docker system prune                            # all of the above (no volumes)
docker system prune -a --volumes               # nuclear option
```

> [!danger] Verify Before Pruning
> `docker volume prune` deletes volumes not currently attached to a running container. If you have a stopped database container with a volume, the volume — and all the data — will be deleted. Always check `docker ps -a` first.

### 8.1 The "Build Cache Is Huge" Problem

BuildKit caches build layers separately from images. Check:

```bash
docker buildx du                               # build cache usage
docker buildx prune                            # clean build cache
```

## 9. Build Failures

### 9.1 "Cannot find package"

```dockerfile
COPY package*.json ./
RUN npm install
COPY . .
```

Did you copy `package.json` before running `npm install`? If you `COPY . .` first and then `npm install`, it might work — but if `node_modules` is in the build context (despite `.dockerignore`), it can cause issues.

### 9.2 "Permission denied"

```dockerfile
USER app
COPY . /app/        # files copied as root, app user can't read them
```

Fix: `COPY --chown=app:app . /app/`.

### 9.3 Caching Not Working

If every rebuild re-runs `npm install`, check the order of instructions:

```dockerfile
# BAD
COPY . .
RUN npm install

# GOOD
COPY package*.json ./
RUN npm install
COPY . .
```

### 9.4 Multi-Architecture Build Issues

If an image works on Intel but not Apple Silicon (or vice versa), you may have architecture-specific binaries:

```bash
# Build for both architectures
docker buildx build --platform linux/amd64,linux/arm64 -t myapp:1.0 --push .
```

## 10. Performance Issues

### 10.1 Slow Builds

- Use `.dockerignore` to shrink the build context.
- Order instructions to maximize cache hits.
- Use BuildKit (`DOCKER_BUILDKIT=1`) for parallel builds.
- Use multi-stage builds to avoid reinstalling build tools.

### 10.2 Slow Containers

```bash
docker stats                                   # live CPU/mem/IO
docker inspect web --format '{{.HostConfig.Memory}}'   # check limits
```

Common causes:

- CPU throttling (limit too low).
- Memory pressure (limit too low, app GCing constantly).
- Bind mounts on macOS (use VirtioFS).
- Disk I/O contention (use named volumes instead of bind mounts for hot data).

### 10.3 macOS-Specific Issues

- File synchronization between the host filesystem and the Linux VM is slow.
- Use VirtioFS (Settings → General → "Choose file sharing implementation").
- Avoid bind-mounting `node_modules` — install it inside the image.

## 11. The `docker events` Command

```bash
docker events                                  # stream events live
docker events --since 1h                       # events from the last hour
docker events --filter type=container
docker events --filter event=die
```

`docker events` shows everything the daemon is doing: container starts, stops, dies, image pulls, network creates, etc. Useful for diagnosing "why did my container restart?" or "did the daemon actually receive my command?"

## 12. The `docker inspect` Cheat Sheet

```bash
# Container exit info
docker inspect web --format 'Exit={{.State.ExitCode}} OOM={{.State.OOMKilled}} Error={{.State.Error}}'

# Network
docker inspect web --format 'IP={{.NetworkSettings.IPAddress}} Networks={{range $k,$v := .NetworkSettings.Networks}}{{$k}} {{end}}'

# Mounts
docker inspect web --format '{{range .Mounts}}{{.Type}} {{.Source}} -> {{.Destination}}{{println}}{{end}}'

# Image
docker inspect web --format 'Image={{.Config.Image}} Cmd={{.Config.Cmd}} Entrypoint={{.Config.Entrypoint}}'

# Resources
docker inspect web --format 'Memory={{.HostConfig.Memory}} CPUs={{.HostConfig.NanoCpus}}'
```

## 13. Common Mistakes

> [!warning] Mistake 1 — Not Checking `docker logs` First
> 90% of container problems are explained in the logs. Always check logs before more complex debugging.

> [!warning] Mistake 2 — Not Setting a Healthcheck
> Without a healthcheck, Docker cannot tell when your app is hung. Add `HEALTHCHECK` to your Dockerfile.

> [!warning] Mistake 3 — Ignoring Exit Codes
> Exit 137 = OOM. Exit 127 = command not found. Exit 139 = segfault. These point to specific root causes.

> [!warning] Mistake 4 — Trying to Debug a Container That Has No Shell
> Distroless images have no `sh`. Use `docker debug` or copy a shell in. Or use ephemeral debug containers.

> [!warning] Mistake 5 — Forgetting to Check the Host
> Sometimes the problem is the host, not the container. Check disk space (`df -h`), memory (`free -m`), and the daemon logs (`journalctl -u docker`).

---

## 14. Summary Checklist

- [ ] Workflow: state → logs → exit code → inspect → exec → host.
- [ ] Exit 137 = OOM-killed; exit 127 = command not found; exit 139 = segfault.
- [ ] `docker logs -f --tail 100 web` is your first stop.
- [ ] `docker inspect` has everything: env, mounts, network, image, exit info.
- [ ] `docker exec -it web sh` lets you look inside (if the image has a shell).
- [ ] Use `ss -tlnp`, `nslookup`, `curl`, `ping` to debug networking inside the container.
- [ ] `docker system df` and `docker system prune` manage disk space.
- [ ] `docker events` shows live daemon activity.
- [ ] For distroless images, use `docker debug` or ephemeral debug containers.
