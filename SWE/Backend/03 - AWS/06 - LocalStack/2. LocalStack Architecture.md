# 2. LocalStack Architecture

> [!info] Chapter Context
> Understanding LocalStack's architecture helps you debug issues, configure persistence, and know when to use LocalStack vs. real AWS. This note covers the runtime model, service implementations, and how LocalStack differs from AWS internally.

Related: [[1. Installing LocalStack]] | [[3. LocalStack vs AWS]] | [[11 - Serverless Computing/2. Lambda Fundamentals]]

## 1. The Big Picture

LocalStack is a Python application that listens on port 4566 and routes incoming HTTP requests to per-service emulators. Each emulator implements (a subset of) the AWS API for that service.

```mermaid
graph TD
    Client[Client: CLI / SDK]
    Client -->|HTTP request| Edge["Edge port (4566)<br/>HTTP router"]
    Edge --> Router["Service router<br/>(parses AWS SigV4, headers)"]
    Router --> S3[S3 emulator<br/>(files on disk)"]
    Router --> DDB[DynamoDB emulator<br/>(DynamoDB Local / SQLite)"]
    Router --> Lambda[Lambda emulator<br/>(runs in Docker)"]
    Router --> SQS[SQS emulator<br/>(in-memory)"]
    Router --> SNS[SNS emulator<br/>(in-memory)"]
    Router --> IAM[IAM emulator<br/>(in-memory)"]

    style Edge fill:#bbf,stroke:#333,stroke-width:2px
    style Router fill:#ffa,stroke:#333
```

## 2. The Unified Endpoint

AWS has a different hostname for each service (`s3.amazonaws.com`, `dynamodb.us-east-1.amazonaws.com`, etc.). LocalStack uses a **single endpoint** (`localhost:4566`) for all services.

The router determines which service to dispatch to based on:

- The `Authorization` header (which contains the service name in SigV4).
- The URL path (e.g., `/` for S3 list buckets).
- Custom headers (`X-Amz-Target` for some services).

This is why you can use `http://localhost:4566` for both S3 and DynamoDB calls.

## 3. Service Implementations

### 3.1 S3

- Stores objects as files on disk under `/var/lib/localstack/s3/`.
- Supports most S3 APIs: PUT, GET, DELETE, list, presigned URLs.
- Limited support for: bucket policies, lifecycle rules, replication (Pro).
- Cross-region replication is not really emulated (single region only).

### 3.2 DynamoDB

- Uses [DynamoDB Local](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DynamoDBLocal.html) under the hood.
- Data stored in SQLite (default) or in-memory.
- Supports all DynamoDB APIs (create table, put, get, query, scan, etc.).
- Streams supported (limited).

### 3.3 Lambda

- Each invocation runs in a Docker container (pulled from your local Docker).
- For interpreted languages (Python, Node.js), the runtime is bundled in LocalStack's container.
- For compiled languages, you provide a self-contained binary.
- Cold starts are faster than real AWS (containers are reused).
- Environment variables, IAM roles (simulated), and triggers (S3, SQS, API Gateway) work.

### 3.4 SQS and SNS

- In-memory message queues and topics.
- Standard and FIFO queues supported.
- SNS fan-out to SQS works.
- Message persistence depends on `DATA_DIR` configuration.

### 3.5 IAM

- In-memory user/role/policy store.
- Authentication is simulated — any access key works.
- Authorization is partially simulated — some policy evaluations work, but it is not a complete reimplementation. Do not rely on IAM for security testing.

### 3.6 CloudFormation (Pro)

- Pro edition supports a significant subset of CloudFormation.
- Useful for testing infrastructure-as-code locally.

## 4. Persistence

By default, all LocalStack state is in-memory and lost on restart. To persist:

### 4.1 `DATA_DIR`

```bash
docker run -d -p 4566:4566 \
  -e DATA_DIR=/var/lib/localstack \
  -v localstack-data:/var/lib/localstack \
  localstack/localstack
```

LocalStack saves state to disk under this directory. On restart, it restores.

### 4.2 What Persists

- S3 objects and buckets.
- DynamoDB tables and items.
- SQS queues.
- SNS topics.
- IAM entities.
- Lambda function definitions (but not running state).

### 4.3 What Does Not Persist

- Lambda invocation state.
- CloudWatch Logs (unless configured).
- In-flight messages.

## 5. Lambda Execution Modes

LocalStack can run Lambda functions in two ways:

### 5.1 Container Mode (Default)

Each invocation runs in a Docker container. The container has the runtime (Python, Node.js) installed. Cold start is the time to start the container.

Pros: realistic. Cons: slower (container startup).

### 5.2 Host Mode (Pro)

Lambdas run as direct processes on the LocalStack host. No container per invocation.

Pros: fast. Cons: less isolation; some Lambda features behave differently.

For most development, container mode is fine. Use host mode for tight dev loops.

## 6. Debugging LocalStack

### 6.1 Enable Debug Logging

```bash
docker run -d -p 4566:4566 -e DEBUG=1 localstack/localstack
```

Now LocalStack logs every API call.

### 6.2 View Logs

```bash
docker logs -f localstack
```

### 6.3 The Health Endpoint

```bash
curl http://localhost:4566/_localstack/health | jq
```

Shows running services and any errors.

### 6.4 The LocalStack Web Dashboard

Open `http://localhost:4566/_localstack/` in a browser for a GUI view of resources (Pro feature).

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Forgetting That LocalStack Is Not AWS
> LocalStack emulates the API, but some behaviors differ. Always test on real AWS before production.

> [!warning] Mistake 2 — Expecting IAM to Enforce Authorization
> LocalStack's IAM is partial. It does not strictly enforce all policy evaluations. Do not use it for security testing.

> [!warning] Mistake 3 — Forgetting That Cross-Region Replication Doesn't Work
> LocalStack is single-region. Cross-region features are not emulated.

> [!warning] Mistake 4 — Forgetting to Persist Data
> Without `DATA_DIR`, all data is lost on restart. Set it for development environments.

> [!warning] Mistake 5 — Expecting EC2/VPC to Work
> LocalStack does not emulate VMs or networking. Use real Docker containers as your "EC2 instances."

## 8. Summary Checklist

- [ ] LocalStack is a Python app listening on port 4566.
- [ ] All services share a single endpoint; routing is based on the SigV4 service name.
- [ ] S3 stores files on disk; DynamoDB uses DynamoDB Local (SQLite); Lambda runs in Docker containers.
- [ ] IAM is partial — any access key authenticates; authorization is not strictly enforced.
- [ ] Persistence requires `DATA_DIR`; without it, all data is lost on restart.
- [ ] Enable `DEBUG=1` for verbose logging.
- [ ] Some AWS behaviors differ — always test on real AWS before production.
