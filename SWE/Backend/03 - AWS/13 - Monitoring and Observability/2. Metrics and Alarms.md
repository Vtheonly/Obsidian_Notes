# 2. Metrics and Alarms

> [!info] Chapter Context
> CloudWatch Metrics collects numeric data from AWS services; CloudWatch Alarms notifies you when a metric crosses a threshold. This note covers standard metrics, custom metrics, dashboards, and alarm configuration.

Related: [[1. Logging]] | [[3. Tracing with X-Ray]] | [[13 - Monitoring and Observability/4. SNS Notifications]]

---

## 1. CloudWatch Metrics

A **metric** is a time-ordered set of data points. Examples:

- `AWS/Lambda/Invocations` — Number of Lambda invocations.
- `AWS/EC2/CPUUtilization` — EC2 CPU usage percentage.
- `AWS/RDS/CPUUtilization` — RDS CPU usage.
- `AWS/S3/BucketSizeBytes` — S3 bucket size.

Each metric has:

- **Namespace** — Container (e.g., `AWS/Lambda`, `AWS/EC2`, `MyApp`).
- **Metric name** — The metric (e.g., `Invocations`).
- **Dimensions** — Name/value pairs (e.g., `FunctionName=my-func`).
- **Timestamp** — When the data point was recorded.
- **Unit** — Optional unit (Count, Bytes, Percent, Seconds).

## 2. Standard Metrics

AWS services emit metrics automatically (no configuration needed). Common ones:

### 2.1 EC2

- `CPUUtilization` — CPU usage (%).
- `NetworkIn`/`NetworkOut` — Network bytes.
- `DiskReadOps`/`DiskWriteOps` — Disk IOPS.
- `StatusCheckFailed` — 1 if instance or system status check failed.

> [!warning] EC2 Memory and Disk Usage Are NOT Automatic
> EC2 metrics include CPU, network, and disk IOPS — but NOT memory or disk space usage. To get those, install the CloudWatch agent and configure it to push memory/disk metrics.

### 2.2 Lambda

- `Invocations` — Number of invocations.
- `Duration` — Execution time.
- `Errors` — Number of errors.
- `Throttles` — Number of throttled invocations.
- `ConcurrentExecutions` — Current concurrency.
- `IteratorAge` — For stream triggers, the age of the oldest record.

### 2.3 RDS

- `CPUUtilization`, `FreeableMemory`, `FreeStorageSpace`.
- `DatabaseConnections`.
- `ReadIOPS`/`WriteIOPS`.
- `ReplicaLag` — For read replicas.

### 2.4 DynamoDB

- `ConsumedReadCapacityUnits`/`ConsumedWriteCapacityUnits`.
- `ThrottledRequests`.
- `ReturnedItemCount`.

### 2.5 S3

- `BucketSizeBytes` — Daily bucket size.
- `NumberOfObjects` — Daily object count.
- `AllRequests`, `GetRequests`, `PutRequests` — Request counts (requires request metrics config).

## 3. Custom Metrics

Push your own metrics with `PutMetricData`:

```bash
aws cloudwatch put-metric-data \
  --namespace MyApp \
  --metric-name OrdersProcessed \
  --value 42 \
  --dimensions Environment=Production,Region=us-east-1
```

With Boto3:

```python
import boto3
cw = boto3.client('cloudwatch')

cw.put_metric_data(
    Namespace='MyApp',
    MetricData=[{
        'MetricName': 'OrdersProcessed',
        'Dimensions': [{'Name': 'Environment', 'Value': 'Production'}],
        'Value': 42,
        'Unit': 'Count'
    }]
)
```

For high-resolution metrics (1-second resolution):

```python
cw.put_metric_data(
    Namespace='MyApp',
    MetricData=[{
        'MetricName': 'RequestsPerSecond',
        'Value': 100,
        'Unit': 'Count/Second',
        'StorageResolution': 1   # 1-second resolution (default is 60)
    }]
)
```

> [!warning] Custom Metrics Cost Money
> Each custom metric costs $0.30/month. Don't create one per user or per request — use dimensions instead.

## 4. CloudWatch Alarms

An alarm watches a metric and triggers an action when it crosses a threshold.

```bash
aws cloudwatch put-metric-alarm \
  --alarm-name HighCPU \
  --alarm-description "Alarm when CPU exceeds 80% for 5 minutes" \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 60 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 5 \
  --dimensions Name=InstanceId,Value=i-12345 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:alerts
```

Key fields:

- `period` — The granularity in seconds (must be a multiple of 60).
- `statistic` — How to aggregate the data points in each period (Average, Sum, Maximum, Minimum, SampleCount).
- `evaluation-periods` — How many consecutive periods must breach the threshold.
- `threshold` — The value to compare against.
- `comparison-operator` — `GreaterThanThreshold`, `LessThanThreshold`, `GreaterThanOrEqualToThreshold`, etc.
- `alarm-actions` — SNS topics to notify when the alarm fires.

In the example, the alarm fires when the average CPU is >80% for 5 consecutive 1-minute periods (5 minutes total).

### 4.1 Alarm States

- **OK** — The metric is within the threshold.
- **ALARM** — The metric breached the threshold for the evaluation period.
- **INSUFFICIENT_DATA** — Not enough data points yet.

## 5. Common Alarms to Set Up

### 5.1 EC2

- `CPUUtilization > 90%` for 5 minutes.
- `StatusCheckFailed > 0` for 2 minutes.
- Memory usage > 90% (with the CloudWatch agent).

### 5.2 Lambda

- `Errors > 0` for 1 minute.
- `Throttles > 0` for 1 minute.
- `Duration > p99` (use anomaly detection).

### 5.3 RDS

- `CPUUtilization > 80%` for 10 minutes.
- `FreeStorageSpace < 10 GB` (database is about to run out of disk).
- `DatabaseConnections > 80%` of max.

### 5.4 Billing

- `EstimatedCharges > $50` (Billing alarm, in `us-east-1` only).

### 5.5 DynamoDB

- `ThrottledRequests > 0` for 1 minute.

## 6. Anomaly Detection

Instead of a fixed threshold, CloudWatch can learn the metric's normal pattern and alarm on anomalies:

```bash
aws cloudwatch put-anomaly-detector \
  --namespace AWS/Lambda \
  --metric-name Duration \
  --dimensions Name=FunctionName,Value=my-func \
  --stat Average

aws cloudwatch put-metric-alarm \
  --alarm-name LambdaDurationAnomaly \
  --metric-name Duration \
  --namespace AWS/Lambda \
  --statistic Average \
  --period 60 \
  --evaluation-periods 1 \
  --threshold-metric-id ad1 \
  --metrics '[{"Id":"ad1","Expression":"ANOMALY_DETECTION_BAND(m1, 2)","ReturnData":true},{"Id":"m1","MetricStat":{"Metric":{"Namespace":"AWS/Lambda","MetricName":"Duration","Dimensions":[{"Name":"FunctionName","Value":"my-func"}]},"Period":60,"Stat":"Average"},"ReturnData":false}]' \
  --comparison-operator LessThanLowerOrGreaterThanUpperThreshold
```

Useful for metrics without a fixed "normal" (e.g., traffic varies by time of day).

## 7. CloudWatch Dashboards

Create dashboards to visualize metrics:

```bash
aws cloudwatch put-dashboard \
  --dashboard-name MyApp \
  --dashboard-body file://dashboard.json
```

The dashboard JSON defines widgets (metrics, logs, text). Easier to create via the Console and then export.

## 8. Common Student Mistakes

> [!warning] Mistake 1 — Forgetting to Set Billing Alarms
> Without a billing alarm, you may not notice a runaway resource. Set one on day one.

> [!warning] Mistake 2 — No Alarm on Lambda Errors
> Lambda errors are easy to miss without an alarm. Set one (Errors > 0 for 1 minute).

> [!warning] Mistake 3 — Forgetting to Install the CloudWatch Agent for EC2 Memory
> EC2 does not push memory metrics by default. Install the agent.

> [!warning] Mistake 4 — Too Many Custom Metrics
#  Each custom metric costs $0.30/month. Use dimensions instead of separate metrics.

> [!warning] Mistake 5 — Alarm Period Not a Multiple of 60
> `--period` must be a multiple of 60 (60, 120, 180, ..., 86400). Other values cause errors.

> [!warning] Mistake 6 — Setting Alarms Without SNS Actions
> An alarm without an SNS action just changes state — no one is notified. Always set an action.

## 9. Summary Checklist

- [ ] Metrics are time-ordered data points with namespace, name, dimensions.
- [ ] AWS services emit standard metrics automatically.
- [ ] EC2 memory/disk usage requires the CloudWatch agent.
- [ ] Push custom metrics with `PutMetricData` ($0.30/metric/month).
- [ ] Alarms watch metrics and trigger SNS actions on threshold breach.
- [ ] Alarm states: OK, ALARM, INSUFFICIENT_DATA.
- [ ] Common alarms: CPU, errors, throttles, billing, free disk space.
- [ ] Use anomaly detection for metrics without a fixed normal.
- [ ] Always set an SNS action on alarms.
