# 3. Tracing with X-Ray

> [!info] Chapter Context
> Logs tell you what happened in one service; metrics tell you if something is wrong. **Traces** show you the full path of a request across services. AWS X-Ray is the distributed tracing service.

Related: [[2. Metrics and Alarms]] | [[1. Logging]] | [[15 - Architecture Patterns/1. Microservices]]

---

## 1. Why Distributed Tracing

In a microservices architecture, a single user request may touch many services:

```
Browser → API Gateway → Auth Service → Order Service → DB
                                       → Inventory Service → DB
                                       → Payment Service → External API
```

If the request is slow, which service is the bottleneck? Logs in each service don't help — you need to see the full path.

**Distributed tracing** adds a unique trace ID to each request. Each service records spans (segments of work) tagged with the trace ID. You can see the full timeline.

```mermaid
gantt
    title Trace Timeline
    dateFormat ss
    axisFormat %Ss
    
    section API Gateway
    Receive request     :a1, 0, 1s
    Return response     :a2, after b3, 1s
    
    section Auth Service
    Validate token      :b1, 1s, 2s
    
    section Order Service
    Create order        :b2, 3s, 1s
    Call inventory      :b3, 4s, 3s
    Call payment        :b4, 7s, 2s
    Save to DB          :b5, 9s, 1s
    
    section Inventory
    Reserve items       :c1, 4s, 2s
    
    section Payment
    Charge card         :d1, 7s, 1s
    External API call   :d2, 8s, 1s
```

## 2. X-Ray Concepts

- **Trace** — The end-to-end path of a request. Identified by a trace ID.
- **Segment** — A span of work in one service (e.g., the Order Service's handling).
- **Subsegment** — A finer-grained span within a segment (e.g., a DB call).
- **Annotation** — A key-value pair indexed for search (e.g., `user_id=123`).
- **metadata** — Non-indexed key-value pairs (e.g., the full request body).
- **sampling** — X-Ray samples a percentage of requests (100% is expensive).

## 3. Enabling X-Ray

### 3.1 IAM Permissions

The Lambda execution role (or EC2 instance role) needs `AWSXrayWriteOnlyAccess` (or the `xray:PutTraceSegments` and `xray:PutTelemetryRecords` permissions).

### 3.2 Lambda

Enable X-Ray tracing on the function:

```bash
aws lambda update-function-configuration \
  --function-name my-func \
  --tracing-config Mode=Active
```

In Active mode, Lambda automatically traces invocations and sends them to X-Ray.

In your code, use the X-Ray SDK to add subsegments:

```python
from aws_xray_sdk.core import xray_recorder

def lambda_handler(event, context):
    # Lambda auto-creates a segment; create a subsegment for the DB call
    with xray_recorder.capture('db_query'):
        result = db.query(...)
    return result
```

### 3.3 API Gateway

Enable X-Ray on the API Gateway stage:

```bash
aws apigateway update-stage \
  --rest-api-id abc123 \
  --stage-name prod \
  --patch-operations op=replace,path=/tracingEnabled,value=true
```

### 3.4 ECS/EKS

Install the X-Ray daemon as a sidecar container. Your application uses the SDK to send traces to the daemon.

### 3.5 EC2

Install the X-Ray daemon:

```bash
sudo yum install -y xray-daemon   # Amazon Linux
sudo systemctl start xray
sudo systemctl enable xray
```

## 4. The X-Ray SDK

### 4.1 Python

```python
from aws_xray_sdk.core import xray_recorder
from aws_xray_sdk.core import patch_all

# Patch all supported libraries (boto3, requests, etc.)
patch_all()

def handler(event, context):
    # Auto-traced: boto3 calls, HTTP requests
    
    # Manual subsegment
    with xray_recorder.capture('process_order'):
        order = process(event['order_id'])
    
    # Annotations (indexed, searchable)
    xray_recorder.put_annotation('user_id', event['user_id'])
    
    # Metadata (not indexed)
    xray_recorder.put_metadata('request_body', event)
    
    return order
```

### 4.2 Node.js

```javascript
const AWSXRay = require('aws-xray-sdk');
const AWS = AWSXRay.captureAWS(require('aws-sdk'));

exports.handler = async (event) => {
    const segment = AWSXRay.getSegment();
    const subsegment = segment.addNewSubsegment('process_order');
    try {
        const result = await processOrder(event.order_id);
        subsegment.addAnnotation('user_id', event.user_id);
        return result;
    } catch (err) {
        subsegment.addError(err);
        throw err;
    } finally {
        subsegment.close();
    }
};
```

## 5. Viewing Traces

In the Console: X-Ray → Service Map. You see a graph of services and their interactions. Click any service to see traces.

Each trace shows the timeline of segments and subsegments. Look for:

- Slow segments (bottlenecks).
- Errors (red segments).
- Throttles.
- Calls to external APIs.

## 6. Sampling

X-Ray samples requests to control cost. Default: 5% of requests (or the first 5 per second, whichever is higher).

Configure sampling rules:

```bash
aws xray create-sampling-rule --cli-input-json file://sampling-rule.json
```

```json
{
  "SamplingRule": {
    "RuleName": "high-traffic",
    "Priority": 100,
    "Rate": 0.01,
    "ReservoirSize": 10,
    "ServiceName": "*",
    "ServiceType": "*",
    "Host": "*",
    "HTTPMethod": "*",
    "URLPath": "*"
  }
}
```

For high-traffic apps, sample 1% or less. For low-traffic apps, sample 100% (so you see every request).

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Forgetting IAM Permissions
> The execution role needs X-Ray write permissions. Without them, traces are silently dropped.

> [!warning] Mistake 2 — Sampling Too Much in Production
> 100% sampling on a high-traffic app generates huge costs. Sample 1-5%.

> [!warning] Mistake 3 — Not Patching Libraries
> Without `patch_all()` (Python) or `captureAWS()` (Node.js), AWS SDK and HTTP calls are not traced.

> [!warning] Mistake 4 — Adding Too Many Subsegments
> Subsegments add overhead. Don't add one for every line of code — add them for significant operations (DB calls, external API calls, business logic).

> [!warning] Mistake 5 — Forgetting to Add Annotations for Search
#  Annotations are indexed; metadata is not. Use annotations for fields you want to search by (user_id, order_id).
