# 2. System Design Questions

> [!info] Chapter Context
> System design interviews ask you to design a system from scratch (e.g., "Design Twitter"). This note covers the framework, common questions, and model answers.

Related: [[1. AWS Interview Questions]] | [[3. Scenario Based Questions]] | [[15 - Architecture Patterns/1. Microservices]]

## 1. The System Design Framework

Use this 5-step framework for any system design question:

### Step 1: Clarify Requirements (5 minutes)

Ask:

- Functional: What does the system do? (e.g., "Tweet, follow, timeline.")
- Non-functional: Scale (users, RPS), latency, consistency, availability.
- Constraints: Read-heavy or write-heavy? Geographic distribution? Mobile/web?

### Step 2: Estimate Scale (5 minutes)

Back-of-the-envelope estimates:

- 100M users, 10% active daily = 10M DAU.
- Each user tweets 1/day = 10M tweets/day = ~115 tweets/second (average), ~1000/second (peak).
- 5-year storage: 10M × 365 × 5 = 18B tweets. At 1 KB each = 18 TB.

### Step 3: High-Level Design (10 minutes)

Draw the major components:

```
Client → CDN → API Gateway → Load Balancer → Application Servers → Database
                                            → Cache
                                            → Queue (for async work)
```

### Step 4: Detailed Design (15 minutes)

Drill into specific components:

- Database choice (SQL vs. NoSQL).
- Caching strategy.
- Async vs. sync processing.
- Sharding strategy.
- Replication.

### Step 5: Identify Bottlenecks and Trade-offs (5 minutes)

- Single points of failure.
- Scaling bottlenecks.
- Cost vs. performance.
- Consistency vs. availability.

## 2. Sample Question: Design a URL Shortener

### Step 1: Requirements

- Functional: User submits a long URL; gets a short URL; visiting the short URL redirects to the long URL.
- Non-functional: Low latency (<50ms), high availability, 100M URLs, 10K new URLs/day, 10M redirects/day.

### Step 2: Scale

- 10M redirects/day = 115/second average, ~1000/second peak.
- 100M URLs × 500 bytes each = 50 GB storage.

### Step 3: High-Level

```
Client → API Gateway → Lambda → DynamoDB
                              → (presigned URLs not needed)
```

### Step 4: Detailed

- **Database**: DynamoDB (key-value, low latency, auto-scaling). Partition key: `short_code`.
- **Short code generation**: Random 7-character base62 string. Or counter + hash.
- **Caching**: ElastiCache for hot URLs (further reduce latency).
- **Redirection**: 301 (permanent, browser-cached) for analytics-light, 302 for click tracking.
- **Custom URLs**: Allow users to pick (check collision).

### Step 5: Bottlenecks

- Hot keys in DynamoDB (one URL going viral). Use cache (ElastiCache) to absorb.
- Short code collisions. Handle by retrying with a new code.
- Analytics: write to Kinesis async (don't block the redirect).

## 3. Sample Question: Design Twitter

### Step 1: Requirements

- Functional: Tweet, follow/unfollow, timeline (see tweets from people you follow).
- Non-functional: 300M users, 100M DAU, 500M tweets/day, timeline <200ms latency.

### Step 2: Scale

- 500M tweets/day = ~6000 tweets/second average, ~30K peak.
- Storage: 500M × 1 KB × 5 years = 250 TB.
- Timeline reads: 100M DAU × 10 timeline views/day = 1B reads/day = 12K reads/second.

### Step 3: High-Level

```
Client → CDN (for media) → API Gateway → App Servers → Tweet Service
                                              ↓
                                            Tweet DB (DynamoDB or sharded RDS)
                                              ↓
                                            Fanout Service (writes to timelines)
                                              ↓
                                            Timeline Cache (Redis)
                                              ↓
                                            Timeline Read
```

### Step 4: Detailed

- **Tweet storage**: DynamoDB (massive scale). Partition by `tweet_id`.
- **User graph**: Separate service (who follows whom).
- **Timeline generation**:
  - Pull model: When user requests timeline, fetch recent tweets from everyone they follow. Slow for users following many people.
  - Push model (fan-out on write): When a user tweets, push to all followers' timelines. Fast for readers; expensive for users with millions of followers (e.g., Justin Bieber).
  - Hybrid: Push for normal users; pull for celebrities (followers > 100K).
- **Caching**: Redis for timelines (precomputed in fan-out).
- **Media**: S3 + CloudFront.

### Step 5: Bottlenecks

- Celebrity tweets cause huge fan-out. Hybrid model.
- Hot tweets. Cache aggressively.
- Eventual consistency for timeline (acceptable).

## 4. Sample Question: Design a Chat Application

### Step 1: Requirements

- Functional: 1:1 chat, group chat, online status, message history.
- Non-functional: Real-time delivery (<1 second), 10M users, 1M concurrent.

### Step 2: Scale

- 1M concurrent connections.
- 100M messages/day = 1K/second average.

### Step 3: High-Level

```
Client (WebSocket) → WebSocket Server (ECS/EKS) → Message Service → Database
                                                  → Push Notification Service
                                                  → Online Status Service
```

### Step 4: Detailed

- **Real-time**: WebSockets. Server: ECS/EKS with sticky sessions.
- **Message storage**: DynamoDB (partition by `conversation_id`).
- **Online status**: Redis (key: user_id, value: server_id). Heartbeat every 30 seconds.
- **Push notifications**: SNS for mobile; WebSocket reconnect for web.
- **Group chat**: Send to all members; track read receipts per member.

### Step 5: Bottlenecks

- WebSocket connection limits per server. Use ECS/EKS Auto Scaling.
- Group chat with millions of members. Use fan-out via SNS/SQS.
- Message ordering. Use sequence numbers per conversation.

## 5. Sample Question: Design a Rate Limiter

### Step 1: Requirements

- Functional: Limit N requests per user per minute.
- Non-functional: <10ms latency, distributed, 10M users.

### Step 2: Scale

- 10M users × 10 requests/minute = 100M requests/minute = 1.6M/second.

### Step 3: High-Level

```
Client → API Gateway → Rate Limiter (Redis) → App
```

### Step 4: Detailed

- **Token bucket**: Each user has a bucket of N tokens. Each request consumes 1 token. Tokens refill at rate R/minute.
- **Implementation**: Redis INCR with EXPIRE. Atomic operations.
- **Distributed**: Redis cluster. Consistent hashing for user → shard mapping.
- **Headers**: Return `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`.

### Step 5: Bottlenecks

- Redis throughput. Use Redis cluster; pre-shard by user ID.
- Clock skew (for reset). Use server time.

## 6. Common Mistakes in System Design Interviews

> [!warning] Mistake 1 — Jumping to Solution Without Requirements
> Always clarify requirements first. Interviewers will guide you to the right design.

> [!warning] Mistake 2 — Not Estimating Scale
> Scale affects every design decision. Estimate before designing.

> [!warning] Mistake 3 — Over-Engineering
> Don't propose Kafka + Cassandra + Spark for a 1000-user app. Match the design to the scale.

> [!warning] Mistake 4 — No Trade-off Discussion
> Every choice has a trade-off. Articulate them.

> [!warning] Mistake 5 — Forgetting the Database Choice
> The database is the most important decision. Justify SQL vs. NoSQL.

> [!warning] Mistake 6 — Not Drawing
> Use a whiteboard or drawing tool. Visual communication is critical.

## 7. Summary Checklist

- [ ] Use the 5-step framework: requirements → scale → high-level → detailed → bottlenecks.
- [ ] Always clarify functional AND non-functional requirements.
- [ ] Estimate scale (RPS, storage, bandwidth).
- [ ] Draw the architecture.
- [ ] Justify database choice (SQL vs. NoSQL, which one).
- [ ] Discuss caching, async, sharding, replication.
- [ ] Articulate trade-offs.
- [ ] Match the design to the scale (don't over-engineer).
