## 1. Foundational Questions

> [!question] Q1: What is the difference between a region and an Availability Zone (AZ)?
> **A:** A region is a geographical area (e.g., "us-east-1" N. Virginia) that contains multiple Availability Zones. An AZ is one or more data centers within a region with independent power, cooling, and networking. Regions are connected via high-speed fiber; AZs within a region are connected with low-latency links (<2ms). For high availability, deploy across multiple AZs in a region.

> [!question] Q2: What is the shared responsibility model?
> **A:** AWS is responsible for security OF the cloud (physical infrastructure, virtualization, host OS for managed services). You are responsible for security IN the cloud (your data, IAM, application code, OS for IaaS). The exact split depends on the service model: IaaS gives you more responsibility; SaaS gives AWS more.

> [!question] Q3: What's the difference between S3 and EBS?
> **A:** S3 is object storage (files up to 5 TB, accessed via HTTP API, infinite scale, 11 nines durability). EBS is block storage (volumes attached to EC2 instances, like a virtual hard drive, single-AZ). Use S3 for files, backups, static assets. Use EBS for OS disks and databases on EC2.

> [!question] Q4: When would you use DynamoDB vs. RDS?
> **A:** DynamoDB for high-scale, low-latency, key-based access (millions of TPS, single-digit-ms latency). RDS for traditional SQL with complex queries, JOINs, multi-row transactions. DynamoDB requires upfront access pattern design; RDS allows ad-hoc queries.

> [!question] Q5: What is an IAM role, and when would you use one instead of a user?
> **A:** A role is an identity that can be assumed (via STS) to get temporary credentials. Use roles for AWS services (EC2, Lambda) that need to access other services, for cross-account access, and for federated users. Use users for long-lived identities (humans) that need CLI/Console access. Roles are safer because credentials are temporary.

## 2. Architecture Questions

> [!question] Q6: How would you design a highly available web application on AWS?
> **A:** 
> - Route 53 for DNS.
> - CloudFront for CDN (caches static assets at edge locations).
> - Application Load Balancer across multiple AZs.
> - EC2 Auto Scaling Group across 2-3 AZs (or ECS/Fargate, or Lambda).
> - RDS Multi-AZ for the database.
> - DynamoDB (already multi-AZ) or ElastiCache (multi-AZ) for caching.
> - S3 for static assets.
> - CloudWatch for monitoring.
> - IaC (Terraform/CDK) for deployment.
> - Multi-region for DR if needed.

> [!question] Q7: How do you secure an S3 bucket?
> **A:**
> - Enable Block Public Access (default since 2023).
> - Use bucket policies to grant specific access.
> - Enable encryption (SSE-S3 default, SSE-KMS for audit trail).
> - Enable versioning for recovery.
> - Use VPC endpoints to keep traffic within AWS.
> - Enable access logging for audit.
> - Use IAM Access Analyzer to find unintended public access.
> - For public websites, use CloudFront + OAC (not direct public bucket).

> [!question] Q8: How do you scale a Lambda function that processes SQS messages?
> **A:** Lambda scales automatically based on the queue depth. Each concurrent invocation processes a batch. To control scaling:
> - Set reserved concurrency (upper limit).
> - Adjust batch size and batching window.
> - For high throughput, ensure downstream (e.g., DynamoDB) can handle the write rate.
> - Watch `ApproximateAgeOfOldestMessage` — if growing, increase concurrency or batch size.

> [!question] Q9: What's the difference between an ALB and an NLB?
> **A:** ALB is L7 (HTTP/HTTPS), supports path/host routing, WebSocket, Lambda targets. NLB is L4 (TCP/UDP), ultra-low latency, static IPs, preserves source IP. Use ALB for HTTP apps; NLB for TCP/UDP or extreme performance.

> [!question] Q10: How would you migrate a monolithic application to microservices?
> **A:**
> 1. Don't rewrite everything at once — extract services incrementally.
> 2. Identify bounded contexts (domain-driven design).
> 3. Use the Strangler Fig pattern: route some traffic to the new service via an API gateway, gradually increase.
> 4. Keep the monolith running during migration.
> 5. Use events to sync data between the monolith and new services.
> 6. Each new service owns its database (no shared DB).
> 7. Maintain backward compatibility at the API level.

## 3. Networking Questions

> [!question] Q11: What is a VPC, and why do you need one?
> **A:** A VPC is your private network in AWS. You need one to: isolate resources, control inbound/outbound traffic with security groups and NACLs, connect to on-premises via VPN/Direct Connect, host private databases, and use private subnets for sensitive resources.

> [!question] Q12: What's the difference between a security group and a NACL?
> **A:** Security groups are stateful (return traffic auto-allowed), allow-only, attached to ENIs. NACLs are stateless (must allow return traffic explicitly), allow and deny, attached to subnets. Use SGs for most firewalling; NACLs for explicit denies (e.g., block a known-bad IP).

> [!question] Q13: How do you let a private subnet access the internet?
> **A:** Deploy a NAT Gateway in a public subnet. Update the private subnet's route table to send `0.0.0.0/0` traffic to the NAT Gateway. The NAT Gateway forwards to the IGW (in the public subnet). For production, deploy a NAT Gateway per AZ for HA.

> [!question] Q14: What is a VPC endpoint, and when would you use one?
> **A:** A VPC endpoint lets your VPC access AWS services (S3, DynamoDB, etc.) without traversing the internet. Gateway endpoints (free) for S3 and DynamoDB. Interface endpoints (paid) for other services. Use them for security (no internet traversal) and to avoid NAT Gateway data processing fees.

## 4. Serverless Questions

> [!question] Q15: What is a Lambda cold start, and how do you reduce it?
> **A:** A cold start is the latency for Lambda to allocate a container, load your code, and initialize the runtime (100ms-1s). Reduce by:
> - Using interpreted languages (Python, Node.js) instead of Java.
> - Minimizing dependencies (smaller package).
> - Initializing clients (Boto3, DB) at module level (not in the handler).
> - Using provisioned concurrency (pre-warm instances, extra cost).

> [!question] Q16: What's the maximum Lambda execution time?
> **A:** 15 minutes. For longer tasks, use Step Functions (up to 1 year), Fargate, or EC2.

> [!question] Q17: How do you handle errors in a Lambda function triggered by SQS?
> **A:**
> - If the function fails, the messages become visible again after the visibility timeout (retried).
> - After `maxReceiveCount` retries, SQS moves them to a DLQ.
> - For partial failures, use `ReportBatchItemFailures` to fail only specific messages.
> - Make the function idempotent (messages may be delivered twice).

> [!question] Q18: What's the difference between Step Functions Standard and Express?
> **A:** Standard: up to 1 year duration, 2,000 executions/second, at-least-once, 90-day history. Express: up to 5 minutes, 100,000 executions/second, lower cost per execution. Use Standard for long/audit-required workflows; Express for high-throughput, short workflows (e.g., API Gateway synchronous invocations).

## 5. Database Questions

> [!question] Q19: How does DynamoDB scale?
> **A:** DynamoDB partitions data by partition key. Each partition has limited throughput (3000 RCUs / 1000 WCUs). As your data or throughput grows, DynamoDB adds partitions. For hot partitions (one key much more popular), use a random suffix on the key to spread load. GSIs have their own throughput.

> [!question] Q20: What's the difference between Multi-AZ and Read Replicas in RDS?
> **A:** Multi-AZ is for high availability: a synchronous standby in another AZ, automatic failover, NOT readable. Read Replicas are for read scaling: asynchronous copies, readable, can be in another region. Use Multi-AZ for HA; Read Replicas for read-heavy workloads.

> [!question] Q21: When would you use ElastiCache?
> **A:** For caching (reduce DB load), session storage, rate limiting, real-time leaderboards, pub/sub. Use Redis (most features) or Memcached (simple cache only). Don't use it as a primary data store — it's in-memory (volatile without persistence).

## 6. Cost Optimization Questions

> [!question] Q22: How would you reduce AWS costs?
> **A:**
> - Right-size resources (use Compute Optimizer).
> - Use Reserved Instances or Savings Plans for steady workloads (30-72% off).
> - Use Spot Instances for flexible workloads (up to 90% off).
> - Use auto-scaling to match capacity to demand.
> - Use serverless (Lambda, Fargate) for variable workloads.
> - Use S3 lifecycle policies to move old data to Glacier.
> - Delete unused EBS volumes, EIPs, snapshots.
> - Use VPC endpoints to avoid NAT Gateway fees for AWS service traffic.
> - Set up AWS Budgets to alert on spending.

> [!question] Q23: What's the cost difference between Lambda and EC2?
> **A:** Lambda is pay-per-invocation ($0.20/million + compute time). EC2 is pay-per-hour (regardless of usage). For low/variable traffic, Lambda is cheaper. For steady high traffic, EC2 is cheaper. Break-even is roughly 3-4 million Lambda requests/month for a t3.micro-equivalent workload.

## 7. Behavioral / Scenario Questions

> [!question] Q24: Tell me about a time you debugged a production issue on AWS.
> **A:** [Use STAR: Situation, Task, Action, Result.]
> - Situation: Users reported 500 errors on the checkout API.
> - Task: Identify and fix the issue ASAP.
> - Action: Checked CloudWatch metrics — saw Lambda errors spiking. Checked CloudWatch Logs — saw "database connection limit reached." Checked RDS — saw `DatabaseConnections` at max. Identified that Lambda was opening a new DB connection per invocation (no pooling). Implemented RDS Proxy to pool connections. Deployed via CodePipeline.
> - Result: Errors dropped to zero; p99 latency decreased by 30% (fewer connection handshakes).

> [!question] Q25: How do you stay current with AWS?
> **A:** AWS Blog, AWS What's New, AWS re:Invent videos, AWS Weekly podcast, hands-on experimentation with new services, certifications, peer discussion. Demonstrate specific recent learning.

## 8. Common Student Mistakes in Interviews

> [!warning] Mistake 1 — Memorizing Without Understanding
> Interviewers follow up with "why?" If you can't explain the why, you sound rehearsed.

> [!warning] Mistake 2 — Not Asking Clarifying Questions
> For system design questions, ask about scale, latency requirements, consistency needs. Don't assume.

> [!warning] Mistake 3 — Jumping to Solutions Without Considering Trade-offs
> Every architecture decision has trade-offs. Articulate them.

> [!warning] Mistake 4 — Forgetting Cost
> AWS interviews often ask about cost. Don't propose 10 t3.2xlarge instances for a dev workload.

> [!warning] Mistake 5 — Not Knowing Your Past Projects
> Be ready to discuss any project on your resume in depth.
