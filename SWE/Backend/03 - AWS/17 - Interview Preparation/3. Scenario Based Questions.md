# 3. Scenario Based Questions

> [!info] Chapter Context
> Scenario-based questions present a real-world situation and ask how you'd respond. This note covers common scenarios and model answers.

Related: [[2. System Design Questions]] | [[1. AWS Interview Questions]] | [[13 - Monitoring and Observability/1. Logging]]

## 1. Scenario: Production Database is Slow

> [!question] Users are reporting that the app is slow. You check CloudWatch and see RDS CPU at 95%. What do you do?
> **A:**
> 1. **Immediate**: Scale up the RDS instance (vertical scaling, faster than fixing the root cause).
> 2. **Diagnose**: 
>    - Check RDS Performance Insights for slow queries.
>    - Check CloudWatch for connection count (might be too many connections).
>    - Check for recent deploys (might have introduced a bad query).
> 3. **Fix**:
>    - Add indexes for slow queries.
>    - Add ElastiCache in front of the DB (cache frequent reads).
>    - Use RDS Proxy to pool connections.
>    - Create Read Replicas for read-heavy workloads.
>    - Optimize queries (avoid N+1, use EXPLAIN).
> 4. **Long-term**: Consider sharding, moving to Aurora, or migrating to DynamoDB for specific workloads.

## 2. Scenario: S3 Bucket Leaks Data

> [!question] You discover that an S3 bucket with customer data is publicly accessible. What do you do?
> **A:**
> 1. **Immediate**: Enable Block Public Access on the bucket.
> 2. **Audit**: Use CloudTrail to see who accessed the bucket while it was public. Use S3 access logs for details.
> 3. **Notify**: Inform security and legal teams. Notify affected customers (legal requirement in many jurisdictions).
> 4. **Investigate**: How did it become public? Bucket policy? ACL? Misconfigured CloudFront? Fix the root cause.
> 5. **Prevent**: 
>    - Enable Block Public Access at the account level.
>    - Use AWS Config rules to detect public buckets.
>    - Use IAM Access Analyzer for continuous monitoring.
>    - Require bucket policies to deny public access.

## 3. Scenario: Lambda Function Timing Out

> [!question] A Lambda function is timing out intermittently. How do you debug?
> **A:**
> 1. **Check CloudWatch Logs**: Look for the last log line before the timeout. Identify where it's stuck.
> 2. **Check CloudWatch Metrics**: `Duration` (is it close to the timeout?), `Errors` (correlated?).
> 3. **Check downstream**: Is the function calling an external API or DB that's slow? Check their metrics.
> 4. **Check concurrency**: Is the function being throttled? (`Throttles` metric.)
> 5. **Common causes**:
>    - DB connection pool exhausted (use RDS Proxy).
>    - External API slow (add retries with backoff; reduce timeout).
>    - Cold start (increase memory or use provisioned concurrency).
>    - Event payload too large (process in batches).
> 6. **Fix**: Increase timeout if appropriate. Add X-Ray tracing for visibility. Optimize the slow step.

## 4. Scenario: Cost Spike

> [!question] Your AWS bill jumped from $5K to $20K last month. How do you investigate?
> **A:**
> 1. **Open Cost Explorer**: Filter by service, then by region, then by tag. Identify the biggest contributor.
> 2. **Check for new resources**: Did someone create a large EC2 instance? An RDS instance? A Lambda with high invocation count?
> 3. **Check data transfer**: Cross-AZ ($0.01/GB) and cross-region ($0.02/GB) data transfer can spike if a config changed.
> 4. **Check NAT Gateway**: NAT Gateway data processing fees ($0.045/GB) can spike if traffic patterns changed.
> 5. **Check S3**: Did someone upload a lot of data? Are there old objects not lifecycle-managed?
> 6. **Check unused resources**: EBS volumes of stopped instances, unattached EIPs, old snapshots.
> 7. **Fix**: Delete unused resources. Right-size over-provisioned instances. Add lifecycle policies. Set up AWS Budgets to alert earlier.

## 5. Scenario: Multi-Region Failover

> [!question] Your primary region (us-east-1) has a major outage. How do you fail over to us-west-2?
> **A:**
> 1. **DNS**: Update Route 53 to point to the secondary region (use a failover routing policy for automatic failover).
> 2. **Database**: Promote the cross-region read replica in us-west-2 (it becomes the primary).
> 3. **Application**: Switch traffic to the standby ECS/EKS/Lambda in us-west-2.
> 4. **Queues**: If using SQS, the messages in us-east-1 are lost (SQS is regional). Use SNS cross-region or replicate to both regions.
> 5. **After us-east-1 recovers**: 
>    - Set up replication from us-west-2 (new primary) back to us-east-1.
>    - Once in sync, optionally fail back to us-east-1 (or keep us-west-2 as primary).
> 6. **Test the failover regularly**: DR plans that aren't tested usually fail when needed.

## 6. Scenario: Deploying Without Downtime

> [!question] How do you deploy a new version of your application without downtime?
> **A:**
> - **Blue/Green**: Deploy to a new environment (green); switch traffic via Route 53 or ALB; keep blue running for quick rollback.
> - **Canary**: Deploy to a small % of instances; monitor; gradually increase to 100%.
> - **Rolling**: Replace old instances with new ones gradually (Auto Scaling Group with `minHealthyPercentage`).
> - **Lambda**: Traffic shifting (10% → 100% over 5 minutes), with alarms that auto-rollback on errors.
> - **Database migrations**: 
>   - Forward-compatible schema changes (add column, not remove).
>   - Deploy new code that uses the new column.
>   - Backfill data.
>   - Deploy code that stops using the old column.
>   - Remove the old column.

## 7. Scenario: Implementing a New Feature Behind a Flag

> [!question] You want to deploy new code but only enable it for 5% of users. How?
> **A:**
> - Use a feature flag service (LaunchDarkly, AWS AppConfig, or roll your own with DynamoDB).
> - The code checks the flag: if enabled for this user, run new code; else, run old code.
> - Enable for 5% (random by user ID hash); monitor for issues.
> - Gradually increase to 100%.
> - If issues arise, disable the flag (instant rollback without redeployment).

## 8. Scenario: Securing Access for a Third-Party Vendor

> [!question] A vendor needs access to specific S3 objects in your account. How do you grant it securely?
> **A:**
> 1. **Don't share IAM users**: Use cross-account role assumption.
> 2. **Create a role** in your account with a trust policy allowing the vendor's account to assume it.
> 3. **Attach a policy** to the role granting `s3:GetObject` on the specific bucket/prefix only.
> 4. **The vendor assumes the role** via STS in their account; gets temporary credentials.
> 5. **Alternative**: Generate presigned URLs for the specific objects (time-limited, no IAM needed).
> 6. **Audit**: Use CloudTrail to log all access by the vendor's role.

## 9. Scenario: Disaster Recovery Strategy

> [!question] Design a DR strategy for a 3-tier app with 1-hour RTO and 15-minute RPO.
> **A:**
> - RTO (Recovery Time Objective) = max downtime = 1 hour.
> - RPO (Recovery Point Objective) = max data loss = 15 minutes.
> - **Strategy**:
>   - Multi-AZ within the primary region (handles AZ failure, no RPO).
>   - Cross-region replication: RDS cross-region read replica (asynchronous, typically <1 second lag → RPO met). S3 cross-region replication. DynamoDB Global Tables.
>   - Standby infrastructure in the secondary region: minimal (Auto Scaling Group size 0; pre-configured IaC).
>   - On DR: scale up the standby, promote the read replica, switch DNS (Route 53 failover).
>   - Test the DR plan quarterly.
> - **Pilot light**: Keep the minimal core (database replica) running; scale up everything else on DR.
> - **Warm standby**: Keep a scaled-down version running; scale up on DR.
> - **Multi-site active-active**: Both regions serve traffic; most expensive.

## 10. Common Student Mistakes in Scenario Interviews

> [!warning] Mistake 1 — Jumping to Solution Without Diagnosis
> For "database is slow," don't immediately say "shard it." Diagnose first.

> [!warning] Mistake 2 — Not Considering the Blast Radius
> When fixing production, consider what could break. Roll out fixes gradually.

> [!warning] Mistake 3 — No Post-Incident Follow-up
> After fixing, mention what you'd do to prevent recurrence (alerts, runbooks, code review).

> [!warning] Mistake 4 — Forgetting Cost
> "Scale up the RDS instance" — yes, but mention cost. "Add 10 read replicas" — expensive; mention as a trade-off.

> [!warning] Mistake 5 — Not Testing the DR Plan
> An untested DR plan is a hope, not a plan. Mention regular testing.

---

## 11. Summary Checklist

- [ ] For "slow/buggy production": diagnose first (logs, metrics, tracing), then fix, then prevent.
- [ ] For security incidents: contain immediately, audit, notify, fix root cause, prevent.
- [ ] For cost spikes: Cost Explorer → identify the spike → unused resources or new traffic patterns.
- [ ] For multi-region DR: DNS failover, promote read replica, switch traffic.
- [ ] For zero-downtime deploys: blue/green, canary, or rolling. Handle DB migrations carefully.
- [ ] For feature flags: enable for % of users, monitor, gradually increase.
- [ ] For DR strategy: match RTO/RPO with the right pattern (pilot light, warm standby, multi-site).
