# 1. The Shell and Bash Basics

> [!info] Chapter Context
> The **shell** is the program that interprets your commands. **Bash** (Bourne Again Shell) is the default on most Linux distributions. This note covers the shell prompt, command structure, variables, expansion, quoting, and the everyday features you need to use the shell effectively.

## 1. What the Shell Does

When you open a terminal, you see a prompt like:

```
alice@ubuntu:~$
```

This is the **shell prompt** (bash's default). It tells you:

- `alice` — your username.
- `ubuntu` — the hostname.
- `~` — your current directory (`~` = home).
- `$` — you are a regular user (`#` would mean root).

When you type a command and press Enter, the shell:

1. **Reads** the line.
2. **Parses** it into tokens (command, arguments, operators).
3. **Expands** variables, globs, and other shortcuts.
4. **Executes** the command (a builtin, a function, or an external program).
5. **Displays** the output.
6. **Returns** to the prompt.

---

## 2. Command Structure

A typical command looks like:

```bash
command -option --long-option argument1 argument2
```

- `command` — The program to run (e.g., `ls`, `cp`, `grep`).
- `-option` — Short option (single letter, prefixed with `-`).
- `--long-option` — Long option (multi-letter, prefixed with `--`).
- `argument` — Positional argument (e.g., a filename).

Examples:

```bash
ls -l -a /etc                 # short options, separately
ls -la /etc                   # short options combined
ls --all --color=auto /etc    # long options
cp -r src/ dest/              # -r = recursive
```

Most commands accept `--help` to show usage:

```bash
ls --help
man ls                        # full manual
```

## 3. Variables

### 3.1 Setting Variables

```bash
NAME="Alice"
AGE=30
PI=3.14
```

Important rules:

- **No spaces around `=`.** `NAME="Alice"` is correct; `NAME = "Alice"` is wrong (the shell interprets `NAME` as a command).
- Use quotes if the value contains spaces.
- By convention, variable names are uppercase for environment variables, lowercase for local variables.

### 3.2 Reading Variables

```bash
echo $NAME                    # Alice
echo "My name is $NAME"       # My name is Alice
echo 'My name is $NAME'       # My name is $NAME (single quotes do not expand)
```

### 3.3 Environment Variables

A **variable** is local to the shell. An **environment variable** is exported to child processes.

```bash
export DATABASE_URL=postgres://localhost/myapp
node server.js                # the node process inherits DATABASE_URL
```

Without `export`, the variable is shell-local and child processes do not see it:

```bash
LOCAL_VAR="hello"
node -e "console.log(process.env.LOCAL_VAR)"    # undefined
```

### 3.4 Common Built-in Variables

| Variable | Meaning |
| :--- | :--- |
| `$HOME` | Your home directory. |
| `$USER` | Your username. |
| `$PATH` | Directories the shell searches for commands. |
| `$PWD` | Current working directory. |
| `$SHELL` | Your shell. |
| `$?` | Exit code of the last command. |
| `$$` | Current shell's PID. |
| `$!` | PID of the last background process. |
| `$0` | The name of the script. |
| `$1`, `$2`, ... | Positional arguments to a script/function. |
| `$#` | Number of positional arguments. |
| `$@` | All positional arguments (as separate words). |
| `$*` | All positional arguments (as a single string). |

## 4. Expansion

The shell performs several types of **expansion** before executing the command:

### 4.1 Variable Expansion

```bash
echo "Hello, $USER"          # Hello, alice
echo "Home is ${HOME}"        # Home is /home/alice
```

The `${VAR}` form is needed when the variable name is followed by characters that could be part of the name:

```bash
echo "${USER}_file"           # alice_file (correct)
echo "$USER_file"             # (empty, because $USER_file is not defined)
```

### 4.2 Command Substitution

```bash
DATE=$(date +%Y-%m-%d)
echo "Today is $DATE"

# Old syntax (backticks) - avoid, does not nest well
DATE=`date +%Y-%m-%d`
```

`$(command)` runs the command and substitutes its output.

### 4.3 Arithmetic Expansion

```bash
echo $((2 + 3))               # 5
echo $((10 / 3))              # 3 (integer division)
echo $((10 % 3))              # 1 (modulo)
X=5
echo $((X * 2))               # 10
```

### 4.4 Pathname Expansion (Globbing)

```bash
echo *.txt                    # all .txt files in the current directory
echo /etc/*.conf              # all .conf files in /etc
echo ???.txt                  # 3-char names ending in .txt
echo [a-c]*.txt               # names starting with a, b, or c
echo .[a-z]*                  # dotfiles starting with a lowercase letter
```

- `*` matches any sequence (including empty).
- `?` matches any single character.
- `[abc]` matches any one of a, b, c.
- `[a-z]` matches any one character in the range a-z.

Globbing is done by the shell, not by the command. So `ls *.txt` works because the shell expands `*.txt` to a list of filenames before running `ls`.

### 4.5 Brace Expansion

```bash
echo {a,b,c}                  # a b c
echo {1..5}                   # 1 2 3 4 5
echo {a..e}                   # a b c d e
echo file{1..3}.txt           # file1.txt file2.txt file3.txt
mkdir -p project/{src,tests,docs}
# creates project/src, project/tests, project/docs
```

Brace expansion is purely textual — it does not check whether the resulting paths exist.

## 5. Quoting

Quoting controls how the shell interprets special characters.

### 5.1 Single Quotes

Single quotes preserve everything literally. No expansion happens.

```bash
echo 'Hello, $USER'           # Hello, $USER
echo 'Today is $(date)'       # Today is $(date)
```

### 5.2 Double Quotes

Double quotes allow variable and command expansion, but prevent word splitting and globbing.

```bash
echo "Hello, $USER"           # Hello, alice
echo "Today is $(date)"       # Today is Wed Jan 15 ...
echo "Files: *.txt"           # Files: *.txt (no globbing inside double quotes)
```

### 5.3 No Quotes

Without quotes, the shell performs word splitting (on whitespace) and globbing.

```bash
FILES="a.txt b.txt c.txt"
for f in $FILES; do echo "$f"; done
# Iterates over a.txt, b.txt, c.txt (word split)

for f in "$FILES"; do echo "$f"; done
# One iteration: "a.txt b.txt c.txt"
```

> [!tip] The Golden Rule of Quoting
> **Always quote variables** unless you specifically want word splitting or globbing. `"$VAR"` is safe; `$VAR` is risky.

### 5.4 Escape Characters

Use `\` to escape a single character:

```bash
echo "He said \"hello\""      # He said "hello"
echo 'It'\''s a test'         # It's a test (single quote escaped by ending the string)
```

## 6. The `PATH` Variable

When you type `ls`, the shell does not know where `ls` is. It searches directories in `PATH`, in order, for an executable named `ls`:

```bash
echo $PATH
# /usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
```

To add a directory to `PATH`:

```bash
export PATH=$PATH:/opt/myapp/bin       # temporary (current shell only)
# To make permanent, add to ~/.bashrc or ~/.profile
echo 'export PATH=$PATH:/opt/myapp/bin' >> ~/.bashrc
```

### 6.1 The `hash` Cache

Bash caches the full path of commands it has found. If you install a new version of a command in a different directory, the shell may still use the old cached path:

```bash
hash -r                        # clear the cache
hash -d ls                     # clear the cache for ls only
```

## 7. Shell Configuration Files

| File | When it's sourced | Purpose |
| :--- | :--- | :--- |
| `/etc/profile` | Login shells (system-wide) | System-wide login setup. |
| `~/.bash_profile` | Login shells (per-user) | User's login setup. Typically sources `~/.bashrc`. |
| `~/.bash_login` | Login shells (if no `.bash_profile`) | Alternative to `.bash_profile`. |
| `~/.profile` | Login shells (if no `.bash_profile` or `.bash_login`) | Generic shell setup (used by other shells too). |
| `~/.bashrc` | Interactive non-login shells (every new terminal) | User's interactive setup (aliases, functions, prompt). |

The distinction between login and non-login shells is confusing:

- **Login shell** — Started after SSH login or `su -`. Reads `~/.bash_profile` (or `.bash_login` or `.profile`).
- **Non-login shell** — Started by opening a new terminal in a GUI. Reads `~/.bashrc`.

Most users put their customization in `~/.bashrc` and have `~/.bash_profile` source it:

```bash
# ~/.bash_profile
if [ -f ~/.bashrc ]; then
  source ~/.bashrc
fi
```

### 7.1 Common `~/.bashrc` Customizations

```bash
# Aliases
alias ll='ls -la'
alias gs='git status'
alias dc='docker compose'

# Prompt customization
export PS1='\u@\h:\w\$ '

# PATH additions
export PATH=$PATH:$HOME/bin

# Editor
export EDITOR=vim
```

## 8. Aliases and Functions

### 8.1 Aliases

```bash
alias ll='ls -la'
alias gs='git status'
alias ..='cd ..'
alias ...='cd ../..'
```

Aliases are simple text substitutions. They are expanded before the command is parsed.

### 8.2 Functions

Functions are more powerful — they can take arguments and contain logic.

```bash
mkcd() {
  mkdir -p "$1"
  cd "$1"
}

# Usage:
mkcd /tmp/my/new/directory
```

Add functions to `~/.bashrc` to make them permanent.

## 9. Command History

```bash
history                        # show command history
!123                           # re-run command 123 from history
!!                             # re-run the last command (e.g., `sudo !!`)
!$                             # the last argument of the previous command
!^                             # the first argument of the previous command
!grep                          # the most recent command starting with "grep"
Ctrl+R                         # reverse search through history (type to filter)
```

The history is stored in `~/.bash_history` (one command per line). The size is controlled by `HISTSIZE` and `HISTFILESIZE` environment variables.

## 10. Common Student Mistakes

> [!warning] Mistake 1 — Spaces Around `=` in Variable Assignment
> `NAME = "Alice"` is interpreted as the command `NAME` with arguments `=` and `Alice`. Use `NAME="Alice"` (no spaces).

> [!warning] Mistake 2 — Forgetting to Quote Variables
> `echo $FILE` is unsafe if `$FILE` contains spaces or glob characters. Use `echo "$FILE"`.

> [!warning] Mistake 3 — Using `export` for Shell-Local Variables
> `export` makes a variable visible to child processes. For shell-local variables, omit `export` (it is harmless but unnecessary).

> [!warning] Mistake 4 — Forgetting `hash -r` After Installing a New Command
> If you install `docker` to `/usr/local/bin/docker` but `hash` has cached `/usr/bin/docker`, the shell still uses the old one. Run `hash -r` to clear the cache.

> [!warning] Mistake 5 — Editing `~/.bash_profile` and Expecting It in New Terminals
> New GUI terminals read `~/.bashrc`, not `~/.bash_profile`. Put your customizations in `~/.bashrc` (or have `~/.bash_profile` source it).

> [!warning] Mistake 6 — Forgetting to `source` After Editing `~/.bashrc`
> Edits to `~/.bashrc` do not take effect in the current shell until you `source ~/.bashrc` (or open a new terminal).

- [ ] `export` makes a variable an environment variable (visible to child processes).

- [ ] `PATH` controls where the shell looks for commands.
