# 2. sudo and Privilege Escalation

> [!info] Chapter Context
> Running everything as root is dangerous. `sudo` lets you run individual commands as root (or as another user) without logging in as root. This note covers `sudo` configuration, `su`, and best practices for privilege escalation.

Related: [[05 - Users and Security/1. Users and Groups]] | [[02 - File System and Permissions/3. Special Permission Bits]] | [[05 - Users and Security/3. SSH and Key Management]]

---

## 1. Why `sudo` Instead of `root`

Logging in as root and running every command as root is dangerous:

- A typo (`rm -rf / etc` instead of `rm -rf /etc`) can destroy the system.
- A compromised application running as root has full system access.
- There is no audit trail — you cannot tell which user ran which command.

`sudo` solves these problems:

- You run as a regular user most of the time.
- You prefix specific commands with `sudo` to run them as root.
- `sudo` logs every command (via syslog / journald).
- You can grant fine-grained permissions (e.g., "alice can run `systemctl restart nginx` but nothing else").

---

## 2. Basic `sudo` Usage

```bash
sudo command                     # run command as root
sudo -u alice command            # run command as user alice
sudo -u postgres psql            # run psql as the postgres user
sudo -i                          # start an interactive root shell
sudo -s                          # start a shell as root (non-login)
sudo -k                          # forget the cached password (force re-prompt)
sudo -l                          # list what you are allowed to do with sudo
```

After you run `sudo` once and enter your password, subsequent `sudo` commands within 15 minutes (configurable) do not re-prompt.

---

## 3. The `sudo` Group

On Debian/Ubuntu, members of the `sudo` group can run any command as root. On RHEL/CentOS, the group is called `wheel`.

```bash
# Grant alice sudo privileges
sudo usermod -aG sudo alice      # Debian/Ubuntu
sudo usermod -aG wheel alice     # Fedora/RHEL/CentOS

# alice must log out and back in for the group change to take effect
```

Verify with `groups` or `id` after re-login.

---

## 4. The `sudoers` File

`sudo` configuration is in `/etc/sudoers` and `/etc/sudoers.d/`. The format is:

```
# User privilege specification
root    ALL=(ALL:ALL) ALL

# Members of the admin group may gain root privileges
%admin ALL=(ALL) ALL

# Allow members of group sudo to execute any command
%sudo   ALL=(ALL:ALL) ALL

# Specific command permission
alice   ALL=(ALL) /usr/bin/systemctl restart nginx
bob     ALL=(ALL) NOPASSWD: /usr/bin/apt update
```

Format: `who where=(as_who) what`

- `who` — Username or `%group`.
- `where` — Hostname (usually `ALL`).
- `(as_who)` — Which user to run as (`ALL` = any, `root` = just root).
- `what` — Command(s) allowed. `ALL` = anything.

### 4.1 `NOPASSWD:`

`NOPASSWD:` means no password is required. Useful for scripts and CI:

```
deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart myapp
```

### 4.2 Editing sudoers

> [!danger] Never Edit `/etc/sudoers` Directly
> A syntax error in `/etc/sudoers` can lock you out of sudo (and therefore root). Always use `visudo`, which validates the syntax before saving.

```bash
sudo visudo                      # edit /etc/sudoers
sudo visudo -f /etc/sudoers.d/myapp   # edit a file in /etc/sudoers.d/
```

`visudo` opens the file in your default editor (`EDITOR` env var). When you save, it checks the syntax. If there is an error, it asks "What now?" — you can re-edit, quit without saving, or save anyway (do not).

### 4.3 Best Practice: Use `/etc/sudoers.d/`

Instead of editing `/etc/sudoers`, create a file in `/etc/sudoers.d/`:

```bash
sudo visudo -f /etc/sudoers.d/myapp
```

```ini
# /etc/sudoers.d/myapp
deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart myapp
deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl status myapp
```

The main `/etc/sudoers` includes `#includedir /etc/sudoers.d`, so these files are loaded automatically. This is cleaner than editing the main file (which package upgrades may overwrite).

## 5. `su` — Switch User

`su` (switch user) lets you switch to another user's account by entering their password.

```bash
su                               # switch to root (prompts for root's password)
su -                             # switch to root with a login shell (loads root's env)
su alice                         # switch to alice (prompts for alice's password)
su - alice                       # switch to alice with a login shell
```

`su` requires the target user's password. `sudo su` requires your password (and grants you root). On modern systems, root has no password (login disabled), so `su` alone does not work — you must use `sudo su` or `sudo -i`.

### 5.1 `su` vs `sudo`

| Feature | `su` | `sudo` |
| :--- | :--- | :--- |
| Prompts for | Target user's password | Your password |
| Granularity | Full shell as target user | Per-command |
| Logging | Minimal | Detailed (via syslog) |
| Modern use | Rare | Standard |

## 6. Best Practices

### 6.1 Disable Root Login

On a fresh server, disable root login over SSH:

```bash
# /etc/ssh/sshd_config
PermitRootLogin no
```

And ensure your user is in the `sudo` group so you can still become root.

### 6.2 Use `sudo -i` Instead of `su -`

`sudo -i` gives you a root login shell using your own password. `su -` requires root's password (which should be disabled).

### 6.3 Grant Least Privilege

Do not put every user in the `sudo` group. For specific needs, use `/etc/sudoers.d/` to grant specific commands:

```
# Allow deploy user to restart only myapp
deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart myapp
```

### 6.4 Audit `sudo` Usage

`sudo` logs to `/var/log/auth.log` (Debian/Ubuntu) or `/var/log/secure` (RHEL) and to journald. View with:

```bash
journalctl -t sudo
sudo grep sudo /var/log/auth.log
```

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Editing `/etc/sudoers` Without `visudo`
> A syntax error locks you out of sudo. Always use `visudo`, which validates before saving.

> [!warning] Mistake 2 — Forgetting to Log Out After Adding to `sudo` Group
> Group changes do not take effect until you log out and back in. Verify with `id` after re-login.

> [!warning] Mistake 3 — Using `NOPASSWD: ALL` Carelessly
> `NOPASSWD: ALL` means the user can run any command as root with no password. This is dangerous. Use only for specific commands.

> [!warning] Mistake 4 — Forgetting `sudo` Aliases
> `alias sudo="sudo "` (with a trailing space) makes sudo respect aliases for the following command. Without this, `sudo ll` fails because `ll` is an alias.

> [!warning] Mistake 5 — Confusing `sudo su` and `sudo -i`
> Both give you a root shell. `sudo -i` is cleaner (loads root's login profile). `sudo su` is older and less idiomatic.

> [!warning] Mistake 6 — Forgetting That `sudo` Caches the Password
> After you enter your password once, sudo does not re-prompt for 15 minutes. This is a security risk if you walk away from your terminal. Use `sudo -k` to force re-prompt.

---

## 8. Summary Checklist

- [ ] `sudo` runs individual commands as root (or another user) using your password.
- [ ] Members of `sudo` (Debian/Ubuntu) or `wheel` (RHEL) groups can sudo to root.
- [ ] `sudo -i` opens a root login shell; `sudo -s` opens a non-login shell.
- [ ] Configure sudo in `/etc/sudoers` (with `visudo`) and `/etc/sudoers.d/`.
- [ ] `NOPASSWD:` skips the password prompt (use sparingly).
- [ ] Always use `visudo` to edit sudoers — it validates the syntax.
- [ ] Prefer `/etc/sudoers.d/` over editing the main `/etc/sudoers`.
- [ ] Disable root SSH login (`PermitRootLogin no`).
- [ ] Grant least privilege — only the specific commands a user needs.
- [ ] `sudo` logs to `/var/log/auth.log` and journald.
