# 3. systemd and journalctl

> [!info] Chapter Context
> **systemd** is the init system on virtually all modern Linux distributions. It starts services at boot, manages them, restarts them if they crash, and handles dependencies. **journalctl** is its log viewer. If you manage Linux servers, you will use these tools every day.

Related: [[03 - Processes and Services/1. Processes and the Process Tree]] | [[03 - Processes and Services/2. Signals and Job Control]] | [[08 - Automation/1. Cron and systemd Timers]]

## 1. What systemd Does

systemd is PID 1. It manages **units**, which can be:

- **Services** — Long-running daemons (Nginx, PostgreSQL, sshd).
- **Timers** — Scheduled tasks (like cron, but systemd-native).
- **Sockets** — Listen on a socket and start a service on demand.
- **Mounts** — Mount filesystems at boot.
- **Targets** — Groups of units (like runlevels in old SysV init).

Each unit is defined by a `.service` (or `.timer`, `.socket`, etc.) file, typically in:

- `/etc/systemd/system/` — System administrator's units (highest priority).
- `/lib/systemd/system/` or `/usr/lib/systemd/system/` — Distribution-provided units.
- `~/.config/systemd/user/` — Per-user units (for user services).

## 2. The `systemctl` Command

### 2.1 Service Lifecycle

```bash
sudo systemctl start nginx            # start the service now
sudo systemctl stop nginx             # stop the service now
sudo systemctl restart nginx          # stop + start
sudo systemctl reload nginx           # reload config without restarting (if supported)
sudo systemctl status nginx           # show current status
systemctl is-active nginx             # is it running? (returns "active" or "inactive")
systemctl is-enabled nginx            # will it start at boot? (returns "enabled" or "disabled")
sudo systemctl enable nginx           # start at boot
sudo systemctl disable nginx          # do not start at boot
sudo systemctl enable --now nginx     # enable AND start immediately
```

### 2.2 Listing Units

```bash
systemctl list-units                  # all loaded units
systemctl list-units --type=service   # only services
systemctl list-units --state=running  # only running
systemctl list-unit-files --type=service    # all installed service files (even if not loaded)
systemctl list-units --failed         # failed units (need attention)
```

### 2.3 Inspecting Units

```bash
systemctl cat nginx                   # show the unit file
systemctl show nginx                  # show all properties (low-level)
systemctl list-dependencies nginx     # what does nginx depend on?
```

### 2.4 The System Itself

```bash
systemctl reboot                      # reboot the system
systemctl poweroff                    # shut down
systemctl suspend                     # suspend to RAM
systemctl hibernate                   # hibernate to disk
systemctl rescue                      # enter rescue mode (single-user)
systemctl emergency                   # enter emergency mode (minimal)
```

## 3. Anatomy of a `.service` File

```ini
[Unit]
Description=Nginx Web Server
Documentation=https://nginx.org/en/docs/
After=network.target
Wants=network-online.target

[Service]
Type=forking
PIDFile=/run/nginx.pid
ExecStartPre=/usr/sbin/nginx -t -q -g 'daemon on; master_process on;'
ExecStart=/usr/sbin/nginx -g 'daemon on; master_process on;'
ExecReload=/usr/sbin/nginx -s reload
ExecStop=/usr/sbin/nginx -s quit
TimeoutStopSec=5
KillMode=mixed
PrivateTmp=true

[Install]
WantedBy=multi-user.target
```

### 3.1 The Three Sections

- **`[Unit]`** — Metadata and dependencies.
  - `Description` — Human-readable name.
  - `After=` — Start this unit after the listed units (does not require them to be active).
  - `Requires=` — Hard dependency (if the listed unit fails, this one fails too).
  - `Wants=` — Soft dependency (try to start the listed unit, but do not fail if it fails).
- **`[Service]`** — How to run the service.
  - `Type=` — `simple` (default, runs in foreground), `forking` (daemonizes), `oneshot` (runs once and exits), `notify` (service sends readiness notification to systemd).
  - `ExecStart=` — Command to start the service.
  - `ExecStop=` — Command to stop (default is to send SIGTERM).
  - `ExecReload=` — Command to reload.
  - `Restart=` — `no`, `on-failure`, `on-success`, `always`, `on-abnormal`.
  - `User=`, `Group=` — Run as this user/group.
  - `Environment=` — Set env vars.
  - `WorkingDirectory=` — Where to run.
- **`[Install]`** — How to enable the unit.
  - `WantedBy=multi-user.target` — Start at the multi-user runlevel (normal boot).

### 3.2 The `Type` Field

| Type | When to use |
| :--- | :--- |
| `simple` (default) | The process does not fork; it stays in the foreground. Most modern apps. |
| `forking` | The process forks and the parent exits, leaving a daemon child. The PIDFile is required so systemd can track the child. |
| `oneshot` | The process runs once and exits (e.g., a setup script). |
| `notify` | The service uses `sd_notify()` to tell systemd when it is ready. Modern services that support systemd integration. |
| `idle` | Like `simple`, but delayed until all jobs are dispatched. Rarely used. |

## 4. Creating Your Own Service

Create a unit file at `/etc/systemd/system/myapp.service`:

```ini
[Unit]
Description=My Custom Application
After=network.target

[Service]
Type=simple
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/node /opt/myapp/server.js
Restart=on-failure
RestartSec=5
Environment=NODE_ENV=production
Environment=PORT=3000

[Install]
WantedBy=multi-user.target
```

Then:

```bash
sudo systemctl daemon-reload            # reload systemd's config (after creating/editing unit files)
sudo systemctl enable --now myapp       # enable at boot AND start now
sudo systemctl status myapp             # verify
journalctl -u myapp -f                  # follow logs
```

> [!warning] Always Run `daemon-reload` After Editing Unit Files
> systemd caches unit files. After editing any file in `/etc/systemd/system/`, run `sudo systemctl daemon-reload` so systemd picks up the changes. Otherwise, your edits are ignored.

## 5. `journalctl` — The Log Viewer

systemd collects logs from all services into a central journal (stored under `/var/log/journal/`). `journalctl` is the viewer.

### 5.1 Basic Usage

```bash
journalctl                              # all logs (paged)
journalctl -e                           # jump to the end (most recent)
journalctl -f                           # follow (live, like tail -f)
journalctl -n 100                       # last 100 lines
journalctl --since "1 hour ago"
journalctl --since "2024-01-15 10:00:00" --until "2024-01-15 11:00:00"
```

### 5.2 Filtering by Service

```bash
journalctl -u nginx                     # all logs from the nginx unit
journalctl -u nginx -f                  # follow nginx logs
journalctl -u nginx --since today
journalctl -u myapp -u nginx            # multiple units
```

### 5.3 Filtering by Priority

```bash
journalctl -p err                       # only errors and above
journalctl -p warning                   # warnings and above
journalctl -p err -u nginx              # errors from nginx
```

Priorities (syslog levels):

| Number | Name | Meaning |
| :--- | :--- | :--- |
| 0 | emerg | System is unusable. |
| 1 | alert | Action must be taken immediately. |
| 2 | crit | Critical conditions. |
| 3 | err | Error conditions. |
| 4 | warning | Warning conditions. |
| 5 | notice | Normal but significant. |
| 6 | info | Informational. |
| 7 | debug | Debug-level messages. |

### 5.4 Filtering by Time, User, PID

```bash
journalctl --since "2024-01-15" --until "2024-01-16"
journalctl _PID=1234                    # logs from PID 1234
journalctl _UID=1000                    # logs from UID 1000
journalctl _SYSTEMD_UNIT=nginx.service  # same as -u nginx
journalctl -k                           # kernel logs only (like dmesg)
journalctl -b                           # current boot only
journalctl -b -1                        # previous boot
journalctl --list-boots                 # list all boots
```

### 5.5 Output Format

```bash
journalctl -o json                      # JSON output (one object per line)
journalctl -o json-pretty               # pretty-printed JSON
journalctl -o cat                       # just the message, no metadata
journalctl -o short-iso                 # ISO timestamps
```

### 5.6 Persistence

By default, the journal is volatile (in `/run/log/journal/`, lost on reboot) on some distros. To make it persistent:

```bash
sudo mkdir -p /var/log/journal
sudo systemd-tmpfiles --create --prefix /var/log/journal
sudo systemctl restart systemd-journald
```

Or edit `/etc/systemd/journald.conf`:

```ini
[Journal]
Storage=persistent
```

## 6. Common systemd Patterns

### 6.1 Auto-Restart on Crash

```ini
[Service]
Restart=on-failure
RestartSec=5
```

If the service exits with a non-zero code, systemd restarts it after 5 seconds. Use `Restart=always` to restart even on success (for services that should never exit).

### 6.2 Sandboxing (Security)

```ini
[Service]
NoNewPrivileges=yes
ProtectSystem=strict
ProtectHome=yes
PrivateTmp=yes
PrivateDevices=yes
ReadWritePaths=/var/lib/myapp /var/log/myapp
CapabilityBoundingSet=
AmbientCapabilities=
SystemCallFilter=@system-service
SystemCallFilter=~@privileged @resources
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX
```

These directives harden the service by restricting what it can do — similar to Docker's security options.

### 6.3 Socket Activation

```ini
# /etc/systemd/system/myapp.socket
[Socket]
ListenStream=3000

[Install]
WantedBy=sockets.target
```

```ini
# /etc/systemd/system/myapp.service
[Service]
ExecStart=/usr/bin/node /opt/myapp/server.js
```

systemd listens on port 3000. When a connection arrives, it starts `myapp.service` and passes the socket as fd 0. This enables zero-downtime restarts and on-demand startup.

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Forgetting `daemon-reload` After Editing Unit Files
> systemd caches unit files. Run `sudo systemctl daemon-reload` after editing any file in `/etc/systemd/system/`.

> [!warning] Mistake 2 — Confusing `enable` and `start`
> `enable` makes the service start at boot. `start` starts it now. `enable --now` does both.

> [!warning] Mistake 3 — Using `service` Instead of `systemctl`
> The old `service nginx start` command still works (it is a compatibility wrapper) but `systemctl start nginx` is the modern way. Prefer `systemctl`.

> [!warning] Mistake 4 — Not Setting `Restart=on-failure`
> Without a restart policy, your service stays down if it crashes. For production, always set `Restart=on-failure` (or `Restart=always`).

> [!warning] Mistake 5 — Logs Not Persisting Across Reboots
> By default, the journal may be volatile. Set `Storage=persistent` in `/etc/systemd/journald.conf` to make logs persist.

> [!warning] Mistake 6 — Looking in `/var/log/messages` for systemd Logs
> Modern systemd-based distros do not write to `/var/log/messages` by default. Use `journalctl` instead. (Some distros also write to `/var/log/syslog` via rsyslog, but `journalctl` is the source of truth.)

## 8. Summary Checklist

- [ ] systemd is PID 1, manages units (services, timers, sockets, mounts, targets).
- [ ] `systemctl start/stop/restart/reload/enable/disable/status <unit>`.
- [ ] `systemctl list-units`, `systemctl list-unit-files`, `systemctl --failed`.
- [ ] Unit files live in `/etc/systemd/system/` (highest priority) or `/usr/lib/systemd/system/`.
- [ ] Always run `systemctl daemon-reload` after editing unit files.
- [ ] `journalctl` is the log viewer: `-u <unit>`, `-f`, `--since`, `-p err`.
- [ ] Use `Type=simple` for foreground processes, `Type=forking` for daemons, `Type=oneshot` for scripts.
- [ ] `Restart=on-failure` auto-restarts crashed services.
- [ ] systemd supports sandboxing (`NoNewPrivileges`, `ProtectSystem`, etc.) similar to Docker.
