# 2. Signals and Job Control

> [!info] Chapter Context
> **Signals** are the way Linux processes communicate simple events to each other: "please terminate," "your child died," "the user pressed Ctrl+C," "the terminal closed." This note covers the common signals, how to send them, and how to handle them in your applications.

Related: [[03 - Processes and Services/1. Processes and the Process Tree]] | [[03 - Processes and Services/3. systemd and journalctl]] | [[03 - Docker/5. Container Lifecycle and Management]]

---

## 1. What Signals Are

A **signal** is a small message sent to a process to notify it of an event. The kernel can send signals to processes; processes can send signals to themselves or to other processes (with permission).

When a signal arrives, the process either:

1. **Handles it** — The process has registered a handler function, which is called.
2. **Ignores it** — The process has explicitly ignored the signal (some signals cannot be ignored, like SIGKILL).
3. **Takes the default action** — Usually terminate, terminate with core dump, or stop.

---

## 2. Common Signals

| Signal | Number | Default action | Common meaning |
| :--- | :--- | :--- | :--- |
| `SIGHUP` | 1 | Terminate | "Hang up" — terminal closed. Many daemons interpret this as "reload config." |
| `SIGINT` | 2 | Terminate | "Interrupt" — user pressed Ctrl+C. |
| `SIGQUIT` | 3 | Terminate + core dump | User pressed Ctrl+\. |
| `SIGILL` | 4 | Terminate + core dump | Illegal CPU instruction. |
| `SIGABRT` | 6 | Terminate + core dump | `abort()` called (e.g., by `assert`). |
| `SIGFPE` | 8 | Terminate + core dump | Arithmetic error (division by zero). |
| `SIGKILL` | 9 | Terminate (cannot be caught) | Force kill. |
| `SIGSEGV` | 11 | Terminate + core dump | Segmentation fault (invalid memory access). |
| `SIGPIPE` | 13 | Terminate | Wrote to a pipe with no readers. |
| `SIGALRM` | 14 | Terminate | `alarm()` timer expired. |
| `SIGTERM` | 15 | Terminate | "Please terminate gracefully" (the default for `kill`). |
| `SIGSTOP` | 19 | Stop (cannot be caught) | Pause process (Ctrl+Z sends this). |
| `SIGTSTP` | 20 | Stop | "Terminal stop" — Ctrl+Z sends this (can be caught). |
| `SIGCONT` | 18 | Continue | Resume a stopped process. |
| `SIGCHLD` | 17 | Ignore | A child process terminated or stopped. |
| `SIGUSR1` | 10 | Terminate | User-defined signal 1. |
| `SIGUSR2` | 12 | Terminate | User-defined signal 2. |

### 2.1 The "Cannot Be Caught" Signals

**SIGKILL** (9) and **SIGSTOP** (19) cannot be caught, blocked, or ignored. The kernel handles them directly. This is why `kill -9` always works (unless the process is in uninterruptible sleep, state `D`).

### 2.2 Numbering Varies Across Architectures

The signal numbers above are for x86/x86_64 Linux. On other architectures (ARM, MIPS), the numbers may differ. Always use signal **names** in scripts (e.g., `kill -TERM`, not `kill -15`).

## 3. Sending Signals

### 3.1 The `kill` Command

```bash
kill 1234                     # send SIGTERM to PID 1234
kill -TERM 1234               # same, explicit
kill -15 1234                 # same, by number
kill -9 1234                  # send SIGKILL (force kill)
kill -KILL 1234               # same, by name
kill -HUP 1234                # send SIGHUP (often "reload config")
kill -l                       # list all signals
```

### 3.2 `pkill` and `killall` — By Name

```bash
pkill nginx                   # SIGTERM to all processes matching "nginx"
pkill -9 nginx                # SIGKILL
pkill -f "python script.py"   # match the full command line
pkill -u alice                # all processes owned by alice
pkill -P 1234                 # all child processes of PID 1234
killall nginx                 # older alternative (matches exact name)
```

### 3.3 From the Keyboard

| Key combination | Signal sent | To what |
| :--- | :--- | :--- |
| Ctrl+C | SIGINT | Foreground process group |
| Ctrl+Z | SIGTSTP | Foreground process group (suspend) |
| Ctrl+\ | SIGQUIT | Foreground process group (with core dump) |
| Ctrl+D | (EOF, not a signal) | Closes STDIN |

### 3.4 Sending Signals to Yourself

```bash
kill -TERM $$                 # $$ is the current shell's PID
```

This is how scripts can self-terminate.

## 4. Signal Handling in Applications

When you write an application, you should handle SIGTERM gracefully — clean up resources, close files, exit. This is especially important for processes that run as PID 1 (e.g., in a Docker container), because `docker stop` sends SIGTERM to PID 1.

### 4.1 Example: Node.js Signal Handler

```javascript
process.on('SIGTERM', () => {
  console.log('SIGTERM received. Closing server...');
  server.close(() => {
    console.log('Server closed. Exiting.');
    process.exit(0);
  });
});

// Force exit after 10 seconds if graceful shutdown stalls
setTimeout(() => {
  console.error('Forcing exit after timeout.');
  process.exit(1);
}, 10000).unref();
```

### 4.2 Example: Python Signal Handler

```python
import signal
import sys

def handle_sigterm(signum, frame):
    print("SIGTERM received. Cleaning up...")
    cleanup()
    sys.exit(0)

signal.signal(signal.SIGTERM, handle_sigterm)

# Run the main loop
main_loop()
```

### 4.3 The PID 1 Problem

In a Docker container, the main process is PID 1. PID 1 has special responsibilities in Linux:

- It must handle SIGTERM (for graceful shutdown).
- It must reap zombie children (or they accumulate).

Many applications (especially Node.js, Java, Python scripts) do not handle these correctly. Solutions:

- Use a minimal init system like `tini`:
  ```dockerfile
  RUN apk add --no-cache tini
  ENTRYPOINT ["/sbin/tini", "--"]
  CMD ["node", "server.js"]
  ```
  `tini` becomes PID 1, handles signals, reaps zombies, and forwards signals to your app.

- Use Bash's `trap` for shell scripts:
  ```bash
  trap 'cleanup; exit' TERM INT
  ```

## 5. Job Control — Revisited

### 5.1 Suspending and Resuming

```bash
$ sleep 100
^Z
[1]+  Stopped                 sleep 100
$ bg
[1]+ sleep 100 &
$ jobs
[1]+  Running                 sleep 100 &
$ fg %1
sleep 100
^C
$
```

- `Ctrl+Z` sends SIGTSTP to the foreground process, suspending it.
- `bg` sends SIGCONT to resume it in the background.
- `fg` brings it back to the foreground.
- `jobs` lists background/suspended jobs.
- `kill %1` sends SIGTERM to job 1.

### 5.2 Why This Matters

Job control lets you multitask in a single terminal. You can:

- Suspend a long-running task to do something else, then resume it.
- Run multiple jobs in the background and switch between them.
- Use `tmux` or `screen` for even more flexibility (multiple windows, persists across disconnects).

## 6. SIGHUP — The "Reload" Convention

Originally, SIGHUP was sent when the terminal "hung up" (the user closed the connection). For daemons (which have no terminal), SIGHUP was repurposed as "reload configuration."

```bash
sudo kill -HUP $(cat /var/run/nginx.pid)
# Tells Nginx to reload its configuration without restarting
```

Most daemons (Nginx, Apache, sshd, syslog-ng) follow this convention. Check the daemon's documentation to confirm.

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Using `kill -9` as the Default
> `kill -9` (SIGKILL) is forceful. The process gets no chance to clean up. Always try `kill` (SIGTERM) first; only escalate to `kill -9` if the process does not exit after a few seconds.

> [!warning] Mistake 2 — Forgetting That SIGKILL Cannot Be Caught
> If you write a SIGTERM handler and your process still dies from `kill -9`, that is expected — SIGKILL cannot be caught, blocked, or ignored.

> [!warning] Mistake 3 — Not Handling SIGTERM in Long-Running Processes
> If your app does not handle SIGTERM, `docker stop` and `systemctl stop` will hang for 10 seconds (the default timeout) and then SIGKILL it. This means no graceful shutdown — open files, databases, and connections may be in an inconsistent state.

> [!warning] Mistake 4 — Closing the Terminal Kills Background Jobs
> Without `disown` or `nohup`, closing the terminal sends SIGHUP to all your jobs, killing them. Use `nohup command &` or `tmux` for long-running tasks.

> [!warning] Mistake 5 — Confusing SIGSTOP and SIGTSTP
> `Ctrl+Z` sends SIGTSTP (which can be caught). `kill -STOP` sends SIGSTOP (cannot be caught). Most apps do not catch SIGTSTP, so the effect is the same — but be aware of the difference.

> [!warning] Mistake 6 — Sending Signals to the Wrong Process
> `pkill -f "python"` matches every process with "python" in its command line, including your IDE, your CI runner, and your shell's `python` tab completion. Be specific.
