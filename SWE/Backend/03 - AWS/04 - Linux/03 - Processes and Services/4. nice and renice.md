# 4. nice and renice

> [!info] Chapter Context
> Linux schedules processes to share CPU time. The **niceness** of a process influences how much CPU time it gets relative to others. This note covers `nice`, `renice`, and when you might want to change a process's priority.

Related: [[03 - Processes and Services/1. Processes and the Process Tree]] | [[03 - Processes and Services/2. Signals and Job Control]] | [[03 - Processes and Services/3. systemd and journalctl]]

---

## 1. The Nice Value

Every process has a **nice value** between -20 and 19. The default is 0.

- **Lower nice values = higher priority** (more CPU time).
- **Higher nice values = lower priority** (less CPU time).

Only root can set a negative nice value (higher priority than default). Regular users can only increase their processes' nice values (lower priority).

| Nice value | Meaning |
| :--- | :--- |
| -20 | Highest priority (root only). |
| 0 | Default. |
| 19 | Lowest priority ("be nice to others"). |

---

## 2. `nice` — Start a Process with a Given Niceness

```bash
nice -n 10 tar czf backup.tar.gz /home        # tar with niceness 10 (lower priority)
nice -n -5 myapp                              # myapp with niceness -5 (higher priority, requires root)
sudo nice -n -10 heavy-computation            # very high priority (root)
```

Without `-n`, `nice` defaults to 10 (a small bump down in priority).

---

## 3. `renice` — Change the Niceness of a Running Process

```bash
renice 10 -p 1234                             # set PID 1234 to niceness 10
renice -5 -p 1234                             # set to -5 (requires root)
renice 5 -u alice                             # all of alice's processes to 5
renice -n 10 -p 1234 -p 5678                  # multiple PIDs
```

---

## 4. Viewing Niceness

```bash
ps -l                                         # the NI column is the niceness
ps ax -o pid,ni,cmd                           # explicitly show nice value
top                                           # the NI column
```

In `top`, you can renice a process by pressing `r` and entering the PID and new nice value.

---

## 5. When to Change Niceness

In practice, you rarely need to change niceness. The Linux Completely Fair Scheduler (CFS) does a good job by default. Cases where it matters:

### 5.1 Background Batch Jobs

A nightly backup or a video transcoding job should not steal CPU from interactive users. Start it with `nice`:

```bash
nice -n 19 rsync -a /home /backup &
```

The backup runs in the background with the lowest priority, so it does not slow down interactive work.

### 5.2 Real-Time / Latency-Sensitive Processes

Audio processing, video rendering, or trading systems might need higher priority:

```bash
sudo nice -n -10 audio-server
```

But usually, you would use real-time scheduling (`chrt`) instead of nice for true real-time requirements.

### 5.3 Fairness on Shared Servers

On a shared dev server, you might renice your heavy compilation to be nice to others:

```bash
renice 10 -p $$
# Now my shell's processes default to lower priority
```

---

## 6. Nice vs. CPU Limits

Note that **nice is not a CPU limit**. A nice-19 process can still use 100% CPU if no other process wants it. Nice only affects scheduling priority — who gets the CPU when there is contention.

If you want to actually limit a process to 50% CPU, use `cpulimit` or cgroups (in Docker, `--cpus=0.5`).

| Tool | What it does |
| :--- | :--- |
| `nice` | Sets scheduling priority. No hard limit. |
| `cpulimit` | Throttles a process to a percentage of CPU. |
| `cgroups` | Kernel-level CPU limits (used by Docker, systemd). |
| `chrt` | Real-time scheduling (SCHED_FIFO, SCHED_RR). |

## 7. systemd and Niceness

You can set the niceness of a systemd service in its unit file:

```ini
[Service]
Nice=10
```

Or via the command line:

```bash
sudo systemctl edit myapp.service
# In the editor:
[Service]
Nice=10
# Save and exit
sudo systemctl restart myapp
```

`systemctl edit` creates an override file at `/etc/systemd/system/myapp.service.d/override.conf`, which is layered on top of the original unit file. This is the preferred way to customize a service without modifying the original unit file (which may be overwritten by package upgrades).

## 8. Common Student Mistakes

> [!warning] Mistake 1 — Confusing Nice Direction
> Lower nice value = higher priority. It is counterintuitive. Think "a nice person lets others go first" — high nice = lets others use CPU first.

> [!warning] Mistake 2 — Expecting Nice to Limit CPU
> `nice 19` does not limit a process to 5% CPU. It only schedules it last when there is contention. If the CPU is otherwise idle, the nice-19 process uses 100%.

> [!warning] Mistake 3 — Forgetting That Regular Users Cannot Decrease Niceness
> If alice sets her process to nice 10, she cannot later renice it back to 0. Only root can decrease niceness (after it has been increased).

> [!warning] Mistake 4 — Using Negative Nice for Production Apps
> Setting a service to nice -10 can starve other services. Most production services should stay at nice 0. Use systemd's resource controls (CPUWeight, CPUQuota) for finer control.

- [ ] Nice values range from -20 (highest priority) to 19 (lowest). Default is 0.
- [ ] Lower nice = higher priority (counterintuitive).
- [ ] `nice -n 10 command` starts a process with niceness 10.
- [ ] `renice 10 -p <pid>` changes a running process's niceness.
- [ ] Only root can set negative nice values or decrease niceness.
- [ ] Nice is a scheduling hint, not a CPU limit. Use cgroups for hard limits.
