# 3. Special Permission Bits

> [!info] Chapter Context
> Beyond the basic `rwx` permissions, Linux has three special bits: **setuid**, **setgid**, and the **sticky bit**. These are essential for understanding how `sudo` works, how shared directories work, and why some files have weird-looking permissions like `-rwsr-xr-x`.

Related: [[02 - File System and Permissions/1. Paths, Inodes, and Links]] | [[02 - File System and Permissions/2. Permissions and Ownership]] | [[05 - Users and Security/1. Users and Groups]]

---

## 1. The Three Special Bits

| Bit | Symbolic | Octal | Purpose |
| :--- | :--- | :--- | :--- |
| **setuid** | `s` in owner's execute position | 4000 | Run executable as the file's owner. |
| **setgid** | `s` in group's execute position | 2000 | Run executable as the file's group; for directories, new files inherit the directory's group. |
| **sticky** | `t` in others' execute position | 1000 | In a directory, only the file owner (or directory owner or root) can delete files. |

When you see permissions like `-rwsr-xr-x`, the `s` in the owner's execute position is the setuid bit. The file is executable by everyone, but when run, it executes **as the file's owner** (usually root) instead of as the user who ran it.

---

## 2. setuid — "Run as Owner"

### 2.1 The Concept

Normally, when you run a program, it runs **as you**. Your UID determines what the program can access.

With setuid, the program runs **as the file's owner**. If the file is owned by root, the program runs as root — regardless of who invoked it.

### 2.2 The Classic Example: `passwd`

The `passwd` command changes your password. To do that, it must modify `/etc/shadow`, which is only readable by root. How can a regular user change their password?

```bash
ls -l /usr/bin/passwd
# -rwsr-xr-x 1 root root 59976 ... /usr/bin/passwd
```

The `s` in the owner's execute position is the setuid bit. When alice runs `passwd`, the program runs **as root**, allowing it to modify `/etc/shadow`.

### 2.3 Other setuid Binaries

```bash
# Find all setuid binaries on the system
sudo find / -perm -4000 -type f 2>/dev/null

# Common ones:
# /usr/bin/sudo
# /usr/bin/passwd
# /usr/bin/su
# /usr/bin/newgrp
# /usr/bin/chsh
# /usr/bin/chfn
# /usr/bin/mount
# /usr/bin/umount
# /usr/lib/openssh/ssh-keysign
```

### 2.4 Security Implications

setuid is powerful and dangerous. A setuid-root program with a vulnerability gives an attacker root access. This is why:

- The number of setuid binaries on a system should be minimal.
- setuid binaries should be carefully audited.
- You should NEVER create your own setuid-root scripts (shell scripts can be manipulated via PATH, IFS, etc.).

> [!danger] Never Make Shell Scripts setuid
> Shell scripts are not safe to run as setuid. The shell interprets the script in ways that can be manipulated (PATH hijacking, IFS manipulation, race conditions). Linux actually ignores the setuid bit on scripts for this reason. Use a compiled binary (C, Go, Rust) if you need setuid behavior, or use `sudo` with proper configuration.

### 2.5 Setting setuid

```bash
chmod u+s myapp             # symbolic
chmod 4755 myapp            # octal (4 = setuid, 755 = rwxr-xr-x)
```

A typical setuid binary has permissions `4755` (`-rwsr-xr-x`).

## 3. setgid — "Run as Group" (or Inherit Group)

setgid has two behaviors, depending on whether it is set on a file or a directory.

### 3.1 On a File: Run with the File's Group

Same concept as setuid, but for group. When you run a setgid program, your effective GID becomes the file's group, not your own group.

Rare in practice. Some games use it to write to a shared high-score file owned by the `games` group.

### 3.2 On a Directory: Inherit Group for New Files

When setgid is set on a directory, any new file or subdirectory created inside inherits the directory's **group** (instead of the creator's primary group).

This is extremely useful for shared directories. Without setgid:

```bash
mkdir -p /shared
sudo chown :developers /shared
sudo chmod 770 /shared

# alice (in developers group) creates a file:
touch /shared/alice-file.txt
ls -l /shared/alice-file.txt
# -rw-r--r-- 1 alice alice 0 ... alice-file.txt
# Note: group is "alice" (her primary group), not "developers"
```

Bob (also in developers) might not be able to write to alice's file, because the group is `alice`, not `developers`.

With setgid:

```bash
sudo chmod 2770 /shared         # 2 = setgid, 770 = rwxrwx---

# alice creates a file:
touch /shared/alice-file.txt
ls -l /shared/alice-file.txt
# -rw-r--r-- 1 alice developers 0 ... alice-file.txt
# Note: group is "developers" (inherited from the directory)
```

Now Bob (in developers) can write to the file, because the group is `developers`.

### 3.3 Setting setgid

```bash
chmod g+s directory          # symbolic
chmod 2770 directory         # octal (2 = setgid, 770 = rwxrwx---)
```

A setgid directory has permissions like `drwxrws---` (the `s` is in the group's execute position).

### 3.4 Real-World Example: `/var/www`

If multiple developers need to edit files in `/var/www`, set the group to `www-developers` and set the setgid bit:

```bash
sudo groupadd www-developers
sudo usermod -aG www-developers alice
sudo usermod -aG www-developers bob
sudo chown -R :www-developers /var/www
sudo chmod -R 2775 /var/www
```

Now any file created in `/var/www` (or its subdirectories) is owned by the `www-developers` group, and any member of that group can edit it.

## 4. The Sticky Bit — "Only Owners Can Delete"

### 4.1 The Concept

When the sticky bit is set on a directory, only the file's owner (or the directory's owner, or root) can delete or rename files inside — even if other users have write permission to the directory.

### 4.2 The Classic Example: `/tmp`

`/tmp` is world-writable (`drwxrwxrwt`), so any user can create files there. Without the sticky bit, any user could delete any other user's files in `/tmp`. With the sticky bit (`t`), users can only delete their own files.

```bash
ls -ld /tmp
# drwxrwxrwt 25 root root 4096 ... /tmp
```

The `t` in the others' execute position is the sticky bit.

### 4.3 Setting the Sticky Bit

```bash
chmod +t directory           # symbolic
chmod 1777 directory         # octal (1 = sticky, 777 = rwxrwxrwx)
```

A sticky directory has permissions like `drwxrwxrwt`.

### 4.4 When to Use the Sticky Bit

- **Shared directories** where multiple users can write, but each user's files should be protected from others.
- **`/tmp`** (already set by default on all Linux systems).
- **Upload directories** for shared services.

## 5. Combining Special Bits

You can combine all three bits in the octal notation:

| Octal prefix | Meaning |
| :--- | :--- |
| `0xxx` | No special bits. |
| `1xxx` | Sticky bit only. |
| `2xxx` | setgid only. |
| `4xxx` | setuid only. |
| `6xxx` | setuid + setgid. |
| `7xxx` | setuid + setgid + sticky. |

Example: `chmod 6755 myapp` sets setuid + setgid on an executable.

In symbolic notation, the special bits appear in the execute position of the affected party:

- setuid on owner's execute: `s` (or `S` if execute is not set).
- setgid on group's execute: `s` (or `S` if execute is not set).
- sticky on others' execute: `t` (or `T` if execute is not set).

The capital `S` and `T` indicate the special bit is set but execute is not — usually a sign of misconfiguration.

---

## 6. Why These Bits Matter for Cloud Engineers

- **`/tmp` and `/var/tmp`** use the sticky bit to prevent users from deleting each other's files.
- **Shared project directories** use setgid to ensure all files inherit the project group.
- **setuid binaries** are a common privilege escalation vector. Auditing them is part of hardening a server:
  ```bash
  sudo find / -perm -4000 -type f 2>/dev/null
  ```
  Compare against a known-good list. Unexpected setuid binaries may indicate a compromise.
- **Containers** usually do not have setuid binaries (for security). If your container needs root-like behavior, use capabilities (`--cap-add`) instead of setuid.

## 7. Common Student Mistakes

> [!warning] Mistake 1 — Forgetting the Sticky Bit on Shared Directories
> If you create a world-writable directory without the sticky bit, any user can delete any other user's files. Always use `chmod 1777` (or `chmod +t`) for shared writable directories.

> [!warning] Mistake 2 — Making Shell Scripts setuid
> Linux ignores setuid on scripts. Do not try `chmod 4755 myscript.sh` expecting it to run as root. Use `sudo` with proper `/etc/sudoers` configuration instead.

> [!warning] Mistake 3 — Forgetting setgid on Shared Project Directories
> Without setgid, files created in a shared directory are owned by the creator's primary group, not the project group. Other project members may not be able to edit them.

> [!warning] Mistake 4 — Confusing Capital `S`/`T` with Lowercase `s`/`t`
> Capital `S` or `T` means the special bit is set but execute is not — usually a mistake. The special bit without execute is rarely useful. Check with `ls -l`.

> [!warning] Mistake 5 — Not Auditing setuid Binaries
> A new setuid-root binary on your system is a red flag. Audit regularly with `find / -perm -4000 -type f`.

## 8. Summary Checklist

- [ ] **setuid** (4000, `u+s`): executable runs as file's owner. Example: `/usr/bin/passwd`.
- [ ] **setgid on a file** (2000, `g+s`): executable runs as file's group.
- [ ] **setgid on a directory**: new files inherit the directory's group.
- [ ] **sticky bit** (1000, `+t`): in a directory, only file owners can delete their own files. Example: `/tmp`.
- [ ] Octal prefix: 4=setuid, 2=setgid, 1=sticky. Combine by adding (e.g., 6755 = setuid + setgid + rwxr-xr-x).
- [ ] Capital `S`/`T` means the special bit is set but execute is not — usually a mistake.
- [ ] Never make shell scripts setuid — Linux ignores it for security reasons.
- [ ] Audit setuid binaries regularly: `find / -perm -4000 -type f`.
