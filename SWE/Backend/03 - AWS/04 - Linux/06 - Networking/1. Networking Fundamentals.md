# 1. Networking Fundamentals

> [!info] Chapter Context
> Linux servers live on networks. To debug "why cannot my app connect to the database?" or "why is my server unreachable?", you need to understand IP addresses, ports, DNS, and the tools Linux provides (`ip`, `ss`, `ping`, `curl`, `dig`).

## 1. IP Addresses and Ports

Every network interface has an **IP address** (e.g., `192.168.1.100`). Every network service listens on a **port** (e.g., `80` for HTTP, `443` for HTTPS, `22` for SSH).

A connection is identified by the 4-tuple: `(source IP, source port, destination IP, destination port)`. For example, when your laptop connects to a web server:

- Source: `192.168.1.50:54321` (your laptop's IP, random port)
- Destination: `93.184.216.34:443` (the server's IP, HTTPS port)

### 1.1 IPv4 vs IPv6

- **IPv4** — 32-bit address (e.g., `192.168.1.100`). The familiar format. Running out globally.
- **IPv6** — 128-bit address (e.g., `2001:db8::1`). Much larger space. Adoption is growing but still less common in internal networks.

Most cloud networking still uses IPv4 internally.

### 1.2 Private IP Ranges

These ranges are not routable on the public internet; they are for internal networks:

- `10.0.0.0/8` (10.x.x.x) — Big networks.
- `172.16.0.0/12` (172.16.x.x to 172.31.x.x) — Medium networks. Docker uses `172.17.0.0/16` by default.
- `192.168.0.0/16` (192.168.x.x) — Small networks (home routers).

### 1.3 Common Ports

| Port | Service |
| :--- | :--- |
| 22 | SSH |
| 25 | SMTP (email) |
| 53 | DNS |
| 80 | HTTP |
| 443 | HTTPS |
| 3306 | MySQL |
| 5432 | PostgreSQL |
| 6379 | Redis |
| 8080 | HTTP alternate (often used by dev servers) |

## 2. The `ip` Command

`ip` (from `iproute2`) is the modern network configuration tool. It replaced the older `ifconfig`.

### 2.1 Viewing Interfaces

```bash
ip addr                           # show all interfaces and IPs
ip addr show eth0                 # just eth0
ip a                              # shorthand for `ip addr`
ip -br a                          # brief format
```

Output:

```
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN group default qlen 1000
    inet 127.0.0.1/8 scope host lo
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP group default qlen 1000
    inet 10.0.0.42/24 brd 10.0.0.255 scope global eth0
```

- `lo` — Loopback (`127.0.0.1`). Used for local connections (the machine talking to itself).
- `eth0` — The primary network interface. May also be named `ens3`, `enp0s3`, `wlan0`, etc.

### 2.2 Viewing Routes

```bash
ip route                          # show routing table
ip route get 8.8.8.8              # what route would be used to reach 8.8.8.8?
```

Output:

```
default via 10.0.0.1 dev eth0           # default gateway
10.0.0.0/24 dev eth0 proto kernel scope link src 10.0.0.42
```

The "default via" line is your default gateway — where traffic for unknown IPs goes.

### 2.3 Modifying (Requires Root)

```bash
sudo ip addr add 10.0.0.43/24 dev eth0       # add an IP
sudo ip addr del 10.0.0.43/24 dev eth0       # remove an IP
sudo ip link set eth0 up                     # bring interface up
sudo ip link set eth0 down                   # take interface down
sudo ip route add default via 10.0.0.1       # set default gateway
```

These changes are temporary — they do not survive a reboot. For persistent config, use `netplan` (Ubuntu), `NetworkManager`, or `/etc/network/interfaces` (Debian).

## 3. The `ss` Command (Replaces `netstat`)

`ss` (socket statistics) shows open sockets.

```bash
ss -tlnp                          # TCP listening sockets, with process info
ss -tln                           # TCP listening (no process info)
ss -tn                            # TCP established connections
ss -uln                           # UDP listening
ss -tlnp | grep :80               # what is listening on port 80?
ss -s                             # summary
```

Flags:

- `-t` — TCP.
- `-u` — UDP.
- `-l` — Listening.
- `-n` — Numeric (do not resolve IPs to hostnames).
- `-p` — Process info (requires root for processes you do not own).

Output:

```
State   Recv-Q   Send-Q   Local Address:Port   Peer Address:Port   Process
LISTEN  0        128      0.0.0.0:22           0.0.0.0:*           users:(("sshd",pid=1234,fd=3))
LISTEN  0        128      0.0.0.0:80           0.0.0.0:*           users:(("nginx",pid=5678,fd=6))
```

> [!tip] "What Is Listening on Port 80?"
> `sudo ss -tlnp | grep :80` is the fastest way to find out.

## 4. `ping` — Test Reachability

```bash
ping 8.8.8.8                      # ping until you Ctrl+C
ping -c 4 8.8.8.8                 # 4 pings, then exit
ping google.com                   # ping by name (tests DNS too)
```

`ping` sends ICMP Echo Request packets. If the destination replies, you have basic reachability. If `ping` fails, it usually means:

- The host is down.
- A firewall is blocking ICMP.
- You have no route to the host.

> [!note] ping Failure Does Not Mean the Service Is Down
> Many firewalls block ICMP (ping) but allow TCP (e.g., HTTP). A failed `ping` does not necessarily mean the service is unreachable. Use `curl` or `nc` to test the actual port.

## 5. `curl` — HTTP and More

```bash
curl https://example.com                       # GET
curl -X POST https://api.example.com -d '{"key":"value"}' -H "Content-Type: application/json"
curl -I https://example.com                    # HEAD (just headers)
curl -v https://example.com                    # verbose (shows TLS handshake, headers)
curl -o file.zip https://example.com/file.zip  # download to file
curl -L https://example.com                    # follow redirects
curl -u alice:password https://example.com     # HTTP basic auth
curl --cacert ca.pem https://example.com       # use a specific CA cert
curl -k https://self-signed.example.com        # ignore SSL errors (insecure)
curl -w "@curl-format.txt" -o /dev/null -s https://example.com    # timing breakdown
```

### 5.1 A Timing Format File

```text
# curl-format.txt
     time_namelookup:  %{time_namelookup}s\n
        time_connect:  %{time_connect}s\n
     time_appconnect:  %{time_appconnect}s\n
    time_pretransfer:  %{time_pretransfer}s\n
       time_redirect:  %{time_redirect}s\n
  time_starttransfer:  %{time_starttransfer}s\n
                     ----------\n
          time_total:  %{time_total}s\n
```

This shows you how long each phase of the request took (DNS, TCP connect, TLS handshake, server processing).

## 6. `wget` — Download Files

```bash
wget https://example.com/file.zip
wget -O custom-name.zip https://example.com/file.zip
wget -c https://example.com/file.zip              # continue (resume) a partial download
wget -r -l 1 https://example.com/                 # recursive (download linked pages)
wget --mirror https://example.com/                # mirror a site
```

`wget` is simpler than `curl` for downloads and handles large files and unreliable connections better (with `-c`).

## 7. DNS — `dig`, `nslookup`, `host`

```bash
dig example.com                              # A record (IPv4)
dig example.com AAAA                         # AAAA record (IPv6)
dig example.com MX                           # MX record (mail)
dig example.com NS                           # NS record (nameservers)
dig @8.8.8.8 example.com                     # use Google's DNS server
dig +short example.com                       # just the answer
dig +trace example.com                       # trace the DNS resolution path

nslookup example.com                         # alternative tool
host example.com                             # simpler alternative
```

### 7.1 The `/etc/hosts` File

Static hostname-to-IP mappings, overrides DNS:

```
# /etc/hosts
127.0.0.1   localhost
127.0.1.1   mylaptop
10.0.0.42   db.internal.example.com db
```

`ping db` resolves to `10.0.0.42` without querying DNS. Useful for testing and for pinning specific hosts.

### 7.2 The `/etc/resolv.conf` File

DNS resolver configuration:

```
# /etc/resolv.conf
nameserver 8.8.8.8
nameserver 8.8.4.4
search example.com
```

- `nameserver` — DNS server to query.
- `search` — Append this domain to bare hostnames (so `db` becomes `db.example.com`).

On modern systems, `/etc/resolv.conf` is often generated by `systemd-resolved` or `NetworkManager`. Editing it directly may be overwritten.

## 8. `nc` (netcat) — The Network Swiss Army Knife

```bash
# Test if a port is open
nc -zv example.com 443                       # -z (scan), -v (verbose)
nc -zv example.com 80 443 8080               # multiple ports

# Listen on a port (for testing)
nc -l 8080                                   # listen on 8080 (Ctrl+C to stop)

# Send a file
nc example.com 8080 < file.txt               # send file to remote
nc -l 8080 > file.txt                        # receive file (on the other end)

# Port scan a range
nc -zv example.com 1-1024 2>&1 | grep succeeded
```

`nc` is invaluable for debugging "is the port open?" and "is the firewall blocking me?".

## 9. `traceroute` and `mtr`

```bash
traceroute example.com                       # show each hop between you and the server
mtr example.com                              # continuous traceroute (interactive, like top)
```

`traceroute` shows each router between you and the destination, with the latency to each. Useful for diagnosing "where is the slowness?"

`mtr` is a continuous version of traceroute that updates in real time. Install with `sudo apt install mtr`.

## 10. Common Network Debugging Workflow

When "my app cannot connect to the database":

1. **Can you resolve the hostname?**
   ```bash
   nslookup db.example.com
   dig db.example.com +short
   ```
   If this fails, DNS is broken.

2. **Can you reach the IP?**
   ```bash
   ping 10.0.0.42
   ```
   If this fails, routing or a firewall is blocking ICMP.

3. **Is the port open?**
   ```bash
   nc -zv 10.0.0.42 5432
   ```
   If this fails, either the database is not listening or a firewall is blocking the port.

4. **Can you actually connect?**
   ```bash
   psql -h 10.0.0.42 -U postgres -c "SELECT 1"
   ```
   If this fails with a non-network error (e.g., "password authentication failed"), networking is fine.

5. **Is it listening on the right interface?**
   ```bash
   sudo ss -tlnp | grep 5432
   ```
   If the database is listening on `127.0.0.1:5432` (loopback only), other machines cannot reach it. It must listen on `0.0.0.0:5432` (all interfaces) or a specific IP.

## 11. Common Student Mistakes

> [!warning] Mistake 1 — Listening on `127.0.0.1` Instead of `0.0.0.0`
> A service listening on `127.0.0.1` (localhost) is reachable only from the same machine. To accept connections from other machines, listen on `0.0.0.0` (all interfaces) or a specific external IP.

> [!warning] Mistake 2 — Confusing Source and Destination Ports
> The server's port is fixed (e.g., 80 for HTTP). The client's port is random (e.g., 54321). When debugging, you usually care about the server's port.

> [!warning] Mistake 3 — Forgetting That `ping` Uses ICMP, Not TCP
> A service may be reachable via TCP even if `ping` fails (because ICMP is blocked). Use `nc -zv` or `curl` to test the actual port.

> [!warning] Mistake 4 — Editing `/etc/resolv.conf` Directly on Modern Systems
> On systems with `systemd-resolved` or `NetworkManager`, `/etc/resolv.conf` is auto-generated. Use `resolvectl` (systemd) or the network manager's config instead.

> [!warning] Mistake 5 — Using `ifconfig` Instead of `ip`
> `ifconfig` is deprecated. Use `ip addr` and `ip route`. The `net-tools` package (which includes `ifconfig` and `netstat`) is no longer maintained.

> [!warning] Mistake 6 — Forgetting That `/etc/hosts` Overrides DNS
> If you have `127.0.0.1 myapp.com` in `/etc/hosts`, every request to `myapp.com` goes to localhost, regardless of DNS. This is a common source of "why is my prod app going to localhost?" bugs.

## 12. Summary Checklist

- [ ] A connection = `(src IP, src port, dst IP, dst port)`.
- [ ] Private IP ranges: `10.x`, `172.16-31.x`, `192.168.x`.
- [ ] `ip addr` shows interfaces; `ip route` shows the routing table.
- [ ] `ss -tlnp` shows TCP listening sockets with process info (replaces `netstat`).
- [ ] `ping` tests reachability (ICMP); not all services respond to ping.
- [ ] `curl` is the universal HTTP client; `-v` for verbose, `-I` for headers, `-o` to save.
- [ ] `dig` queries DNS; `/etc/hosts` overrides DNS; `/etc/resolv.conf` configures the resolver.
- [ ] `nc -zv host port` tests if a TCP port is open.
- [ ] `traceroute`/`mtr` show each hop between you and the destination.
- [ ] Services must listen on `0.0.0.0` (or a specific IP) to accept external connections; `127.0.0.1` is loopback only.
