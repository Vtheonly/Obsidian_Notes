# 4. Load Balancers and CloudFront

> [!info] Chapter Context
> AWS has four load balancer types (ALB, NLB, GWLB, CLB) and the CloudFront CDN. This note covers each, when to use them, and how to configure them.

Related: [[3. DNS and Route 53]] | [[13 - Monitoring and Observability/1. Logging]] | [[12 - AWS Networking/1. VPC Fundamentals]]

---

## 1. The Four ELB Types

| Type | Layer | Use case |
| :--- | :--- | :--- |
| **ALB** (Application Load Balancer) | L7 (HTTP/HTTPS) | HTTP apps, microservices, Lambda targets |
| **NLB** (Network Load Balancer) | L4 (TCP/UDP) | High-throughput, low-latency, WebSocket, non-HTTP |
| **GWLB** (Gateway Load Balancer) | L3 (IP) | Third-party network appliances (firewalls, IDS) |
| **CLB** (Classic Load Balancer) | L4/L7 (legacy) | Legacy apps (avoid for new deployments) |

For most new HTTP apps, **use ALB**. For TCP/UDP or extreme performance, **use NLB**.

## 2. Application Load Balancer (ALB)

### 2.1 Features

- HTTP and HTTPS (TLS termination).
- Path-based routing (`/api/*` to one target group, `/static/*` to another).
- Host-based routing (`api.example.com` to one, `www.example.com` to another).
- WebSocket support.
- Lambda targets (invoke a Lambda function for a request).
- IP-based targets (route to IPs outside AWS).
- WAF integration.
- Sticky sessions (via cookies).
- Health checks.

### 2.2 Components

- **Listener** — A process that checks for connection requests (e.g., port 443, HTTPS).
- **Rule** — Routes requests to target groups based on conditions (path, host, headers).
- **Target Group** — A group of targets (EC2 instances, IPs, Lambda).
- **Target** — An endpoint (e.g., an EC2 instance on port 8080).

### 2.3 Creating an ALB (CLI)

```bash
# Create target group
TG_ARN=$(aws elbv2 create-target-group \
  --name my-tg \
  --protocol HTTP \
  --port 8080 \
  --vpc-id vpc-12345 \
  --health-check-path /health \
  --query 'TargetGroups[0].TargetGroupArn' --output text)

# Register targets
aws elbv2 register-targets \
  --target-group-arn $TG_ARN \
  --targets Id=i-12345 Id=i-67890

# Create the ALB
ALB_ARN=$(aws elbv2 create-load-balancer \
  --name my-alb \
  --subnets subnet-1 subnet-2 \
  --security-groups sg-12345 \
  --query 'LoadBalancers[0].LoadBalancerArn' --output text)

# Create a listener
aws elbv2 create-listener \
  --load-balancer-arn $ALB_ARN \
  --protocol HTTPS \
  --port 443 \
  --certificates CertificateArn=arn:aws:acm:... \
  --default-actions Type=forward,TargetGroupArn=$TG_ARN
```

### 2.4 Path-Based Routing

```bash
aws elbv2 create-rule \
  --listener-arn $LISTENER_ARN \
  --priority 10 \
  --conditions Field=path-pattern,Values='/api/*' \
  --actions Type=forward,TargetGroupArn=$API_TG_ARN

aws elbv2 create-rule \
  --listener-arn $LISTENER_ARN \
  --priority 20 \
  --conditions Field=path-pattern,Values='/static/*' \
  --actions Type=forward,TargetGroupArn=$STATIC_TG_ARN
```

## 3. Network Load Balancer (NLB)

### 3.1 Features

- TCP and UDP (no HTTP-level routing).
- Very high throughput, low latency (~100 microseconds).
- Static IPs (one per AZ) — useful for whitelisting.
- Preserves source IP (the target sees the client's IP, not the NLB's).
- WebSocket support (at TCP level).
- Cross-zone load balancing (optional, costs extra).

### 3.2 When to Use NLB

- Non-HTTP protocols (custom TCP, gaming, IoT).
- Extreme performance requirements.
- When you need static IPs.
- When you need to preserve the client IP at the target.

## 4. CloudFront (CDN)

CloudFront is AWS's Content Delivery Network. It caches content at edge locations worldwide, reducing latency for users and reducing load on your origin.

### 4.1 Use Cases

- Static website assets (CSS, JS, images).
- Video streaming (HLS, DASH).
- API acceleration (cache GET responses).
- Static website hosting (S3 origin).
- Dynamic content with edge compute (Lambda@Edge, CloudFront Functions).

### 4.2 Origins

- **S3 bucket** — For static content. Use Origin Access Control (OAC) to keep the bucket private.
- **ALB/NLB/EC2** — For dynamic content from your own servers.
- **Any HTTP endpoint** — Including on-premises or other clouds.

### 4.3 Creating a CloudFront Distribution

```bash
aws cloudfront create-distribution \
  --origin-domain-name my-bucket.s3.us-east-1.amazonaws.com \
  --default-cache-behavior TargetOriginId=my-bucket,ViewerProtocolPolicy=redirect-to-https,AllowedMethods=GET,HEAD,CachedMethods=GET,HEAD,Compress=true \
  --default-root-object index.html \
  --enabled
```

### 4.4 Cache Behaviors

- **Path patterns** — `/api/*` to one origin, `/*` to another.
- **TTLs** — Min/max/default TTL.
- **Allowed methods** — GET/HEAD (read-only), GET/HEAD/OPTIONS, or all methods.
- **Viewer protocol policy** — HTTP, HTTPS, or redirect-to-HTTPS.
- **Forwarded headers/cookies/query strings** — What to forward to the origin (for caching to work correctly).

### 4.5 Lambda@Edge and CloudFront Functions

Run code at edge locations:

- **CloudFront Functions** — Lightweight JavaScript; for HTTP header manipulation, URL rewrites. ~1ms execution.
- **Lambda@Edge** — Full Lambda functions (Node.js, Python); for more complex logic (auth, A/B testing, image resizing). Runs in regional edge caches.

## 5. Common Student Mistakes

> [!warning] Mistake 1 — Using CLB for New Apps
> Classic Load Balancers are legacy. Use ALB for HTTP, NLB for TCP/UDP.

> [!warning] Mistake 2 — Forgetting Health Checks
> Without health checks, the load balancer routes to dead targets. Configure a health check path that returns 200 when the app is healthy.

> [!warning] Mistake 3 — Cross-Zone Load Balancing Without Understanding the Cost
> Cross-zone load balancing on NLB costs extra ($0.01/GB cross-AZ). On ALB it's free and on by default.

> [!warning] Mistake 4 — Forgetting to Set `X-Forwarded-For`
> When the ALB terminates TLS and proxies, the target sees the ALB's IP. Use `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Forwarded-Port` headers to get the original client info.

> [!warning] Mistake 5 — Not Using CloudFront for Static Assets
> Serving static assets from your app server wastes bandwidth and adds latency. Use CloudFront (with S3 origin) for static assets.

> [!warning] Mistake 6 — Forgetting OAC for S3 Origins
> Without Origin Access Control, your S3 bucket must be public. Use OAC to keep the bucket private and let only CloudFront access it.

> [!warning] Mistake 7 — Forgetting to Forward Query Strings for API Caching
> If your API responses depend on query strings, configure CloudFront to forward them. Otherwise, all variations get the same cached response.

## 6. Summary Checklist

- [ ] ALB: L7, HTTP/HTTPS, path/host routing, Lambda targets.
- [ ] NLB: L4, TCP/UDP, high performance, static IPs, preserves source IP.
- [ ] GWLB: L3, for third-party network appliances.
- [ ] CLB: legacy, avoid for new apps.
- [ ] ALB components: listener, rules, target groups, targets.
- [ ] CloudFront: CDN; caches at edge locations worldwide.
- [ ] CloudFront origins: S3 (use OAC), ALB/NLB/EC2, any HTTP endpoint.
- [ ] Lambda@Edge (full Lambda) and CloudFront Functions (lightweight JS).
- [ ] Use `X-Forwarded-*` headers to get client info behind an ALB.
