# 2. Phase 2 — Building the Transition Function

> "Phase 1 gave you the state; Phase 2 gives you the engine's *behavior*. The transition function $F$ is where the engine's logic lives — where 'a chess position' becomes 'a chess move', where 'a search query' becomes 'a ranked result list', where 'a market event' becomes 'a trading decision'. Get $F$ right, and the engine does the right thing. Get it wrong, and no amount of optimization will fix the correctness."

This is the second note in the engine lifecycle. Phase 1 produced a well-designed state; Phase 2 builds the transition function $F$ that operates on that state.

---

## 6.2.1 Modular Functional Composition

The first principle of $F$ design is **modular composition**: $F$ should be a pipeline of small, independent, testable functions, not a monolithic block of code.

### The Three Stages of $F$

Recall from Chapter 1 that $F$ decomposes into three stages:

1. **Generate.** Produce the candidate set.
2. **Evaluate.** Score each candidate.
3. **Select.** Pick the best candidate and commit.

```python
def F(state, context):
    candidates = generate(state, context)
    scored = evaluate(candidates, state, context)
    next_state = select(scored, candidates, state, context)
    return next_state
```

Each stage is a separate function with a clear input and output. This separation enables:

- **Independent testing.** Each stage can be tested in isolation.
- **Independent optimization.** Each stage can be optimized without affecting the others.
- **Independent replacement.** A stage can be replaced with a different implementation (e.g., replacing a heuristic evaluator with a learned model).
- **Clear profiling.** Each stage's cost can be measured separately.

### Sub-Decomposition Within Stages

Each stage can itself be decomposed:

**Generate:**
```python
def generate(state, context):
    pseudo = generate_pseudo_candidates(state)
    legal = filter_illegal(pseudo, state)
    return legal
```

**Evaluate:**
```python
def evaluate(candidates, state, context):
    scores = []
    for c in candidates:
        score = sum(scorer.weight * scorer.fn(c, state, context) 
                    for scorer in context.scorers)
        scores.append(score)
    return scores
```

**Select:**
```python
def select(scored, candidates, state, context):
    best_idx = argmax(scored)
    return commit(candidates[best_idx], state, context)
```

Each sub-function is small, focused, and testable.

### The Pipeline Pattern

For more complex $F$ (e.g., search engine query evaluation), the pipeline may have many stages:

```mermaid
flowchart LR
    Q[Query] --> T[Tokenize]
    T --> A[Analyze: stem, stopwords]
    A --> R[Rewrite]
    R --> L[Lookup in index]
    L → M[Merge posting lists]
    M → S[Score with BM25]
    S → P[Personalize]
    P → F[Filter: safety, dedup]
    F → O[Output: top-k]
```

Each stage is a function: input → output. The stages are composed with simple function calls. This pattern is easy to read, easy to test, and easy to profile.

### When to Decompose

Decompose when:
- A function is more than ~50 lines of code.
- A function has multiple responsibilities.
- A function has multiple branches that are not mutually exclusive.
- A function is hard to test in isolation.

Do not decompose when:
- The function is already small and focused.
- Decomposition would create functions that are not reusable.
- The decomposition adds more overhead (function call cost) than it saves in clarity.

For hot loops, beware of over-decomposition: each function call has overhead. Inline the hottest functions manually or with compiler hints.

## 6.2.2 Guaranteeing Determinism

The second principle of $F$ design is **determinism**: given the same state and context, $F$ should produce the same output. Determinism is essential for:

1. **Testing.** You cannot test a non-deterministic function; the output changes between runs.
2. **Debugging.** A bug that appears only sometimes is nearly impossible to diagnose.
3. **Reproducibility.** If the engine's behavior cannot be reproduced, you cannot verify that a fix works.
4. **Concurrency.** Non-deterministic behavior in multi-threaded engines leads to race conditions.

### Sources of Non-Determinism

Common sources of non-determinism in $F$:

1. **Random number generation.** MCTS, sampling, randomized algorithms.
2. **Wall-clock time.** Time-based termination, time-based features.
3. **Hash table iteration order.** Hash tables do not preserve insertion order; iterating gives different orders across runs.
4. **Floating-point operations.** Floating-point arithmetic is not associative; `a + b + c` may differ from `a + (b + c)`.
5. **Multi-threading.** Without explicit synchronization, threads may execute in different orders.
6. **External state.** Reading from files, network, or environment variables.
7. **Memory allocation.** Allocators may return different addresses across runs; pointer comparison is non-deterministic.

### Achieving Determinism

To achieve determinism:

1. **Use seeded RNG.** Always seed the random number generator with a fixed seed. Store the seed in the engine's configuration.
2. **Encode time in state.** Instead of checking `now() > deadline`, check `state.iterations > max_iterations` or `state.deadline > state.current_time`. The "current time" is a field of state, set explicitly at startup.
3. **Use ordered collections.** Use sorted maps or vectors of pairs instead of hash tables when iteration order matters. Or sort the hash table's output before iterating.
4. **Use fixed-point or rational arithmetic.** For exact arithmetic, use integers (e.g., represent prices as integer ticks) or rational numbers.
5. **Synchronize explicitly.** Use locks, atomics, or single-threaded execution for the hot path.
6. **Avoid external state in the hot path.** Read all configuration at startup; do not read files or environment variables during execution.
7. **Avoid pointer comparison.** Compare by value, not by address.

### The Determinism Test

To verify determinism:

1. Run the engine twice with the same input and seed.
2. Compare the outputs byte-for-byte.
3. If they differ, the engine is non-deterministic. Find and fix the source.

For engines that use floating-point, this test may reveal subtle non-determinism from FP non-associativity. The fix is to use fixed-point or to accept the non-determinism (with documented error bounds).

### Determinism in Testing

For testing, determinism is mandatory. A test that fails sometimes is worse than no test. To make tests deterministic:

1. Use a fixed RNG seed in tests.
2. Mock the wall clock.
3. Use single-threaded execution.
4. Avoid floating-point in assertions (or use tolerance-based comparison).

### When Non-Determinism Is Acceptable

Non-determinism is acceptable when:
- The engine's output is judged statistically (e.g., win rate over many games), not per-decision.
- The non-determinism is bounded (e.g., MCTS gives different moves per run, but the move quality is similar).
- The user accepts approximate answers (e.g., search engines may return slightly different results per query due to A/B testing).

Even in these cases, the engine should be deterministic *given the same seed*. The seed is the lever for reproducibility.

## 6.2.3 The Build Process for $F$

Building $F$ is an iterative process:

### Step 1: Define the Interface

Start by defining the interface of $F$:

```python
def F(state: State, context: Context) -> State:
    """Apply one step of the engine to the state.
    
    Args:
        state: The current engine state.
        context: The engine context (caches, models, config).
    
    Returns:
        The next engine state.
    
    Raises:
        TerminalState: If the state is terminal (no further transitions).
    """
```

The interface should be clear, with documented inputs, outputs, and exceptions. This interface will not change much as the engine evolves.

### Step 2: Implement the Stages

Implement each stage (generate, evaluate, select) as a separate function. Initially, use the simplest correct implementation — do not optimize.

```python
def generate(state, context):
    # Simplest correct implementation
    return all_possible_actions(state)

def evaluate(candidates, state, context):
    # Simplest correct implementation
    return [simple_score(c, state) for c in candidates]

def select(scored, candidates, state, context):
    # Simplest correct implementation
    best_idx = max(range(len(scored)), key=lambda i: scored[i])
    return apply(candidates[best_idx], state)
```

The goal at this stage is **correctness**, not performance. A correct slow engine is a foundation for optimization; a fast incorrect engine is useless.

### Step 3: Test the Stages

Test each stage in isolation:

```python
def test_generate():
    state = create_test_state()
    candidates = generate(state, test_context())
    assert len(candidates) > 0
    assert all(is_valid(c, state) for c in candidates)

def test_evaluate():
    candidates = [test_candidate_1(), test_candidate_2()]
    scores = evaluate(candidates, test_state(), test_context())
    assert len(scores) == len(candidates)
    assert all(isinstance(s, (int, float)) for s in scores)

def test_select():
    candidates = [test_candidate_1(), test_candidate_2()]
    scores = [1.0, 2.0]
    next_state = select(scores, candidates, test_state(), test_context())
    assert next_state is not None
```

Each test should be small, focused, and deterministic.

### Step 4: Test $F$ End-to-End

Test $F$ as a whole:

```python
def test_F():
    state = create_test_state()
    next_state = F(state, test_context())
    assert next_state is not state  # F should produce a new state
    assert is_valid_state(next_state)
```

End-to-end tests verify that the stages compose correctly.

### Step 5: Test the Engine Loop

Test the full engine loop:

```python
def test_engine_loop():
    state = create_initial_state()
    context = create_test_context()
    while not is_terminal(state):
        state = F(state, context)
    assert is_valid_terminal_state(state)
```

This test runs the engine to completion and verifies the terminal state.

### Step 6: Add Optimizations (Phase 3)

Once the engine is correct, add optimizations (covered in the next note). Each optimization is tested against the unoptimized version to verify it does not break correctness.

### Step 7: Profile (Phase 4)

Profile the engine to identify bottlenecks (covered in the note after next). Optimize the bottlenecks, not the parts that are already fast.

## 6.2.4 The Discipline of Side Effects

$F$ is conceptually pure: it takes state and context, returns a new state. In practice, $F$ may have side effects: updating caches, modifying context, logging.

The discipline of side effects:

1. **Side effects must not affect the current call's output.** Two calls to $F$ with the same state and context must return the same next state, regardless of side effects.
2. **Side effects must be deferred until after the result is computed.** Compute the next state, then update caches and context.
3. **Side effects must be thread-safe.** Use atomics, locks, or per-thread buffers.

```python
def F(state, context):
    # Compute the next state (pure)
    candidates = generate(state, context)
    scored = evaluate(candidates, state, context)
    next_state = select(scored, candidates, state, context)
    
    # Side effects (after the result is computed)
    context.cache[state.hash] = next_state
    context.stats.nodes_processed += 1
    
    return next_state
```

This discipline ensures that $F$ remains deterministic (modulo side effects) and testable.

## 6.2.5 Common Pitfalls

### Pitfall 1: Monolithic $F$

A single function that does everything is unmaintainable, untestable, and unoptimizable. Decompose into stages.

### Pitfall 2: Premature Optimization

Optimizing $F$ before it is correct wastes time. Get a correct slow engine first, then optimize based on profiling.

### Pitfall 3: Non-Deterministic Behavior

Non-deterministic behavior makes the engine impossible to test and debug. Use seeded RNG, encoded time, ordered collections.

### Pitfall 4: Side Effects That Affect the Current Call

If a side effect changes the current call's output, the engine becomes non-deterministic. Defer side effects until after the result is computed.

### Pitfall 5: Untested Stages

Each stage should be tested in isolation. Without isolated tests, bugs in one stage are masked by other stages.

### Pitfall 6: Untested Engine Loop

The engine loop must be tested end-to-end. Without end-to-end tests, integration bugs go undetected.

### Pitfall 7: Wrong Abstraction Level

Decomposing too finely creates excessive function call overhead; decomposing too coarsely creates untestable monoliths. Find the right balance.

### Pitfall 8: Inlined Hot Functions

Inlined hot functions are fast but hard to test. Inline only after profiling identifies the function as a bottleneck.

### Pitfall 9: Forgetting to Handle Edge Cases

Edge cases (no candidates, all candidates tied, terminal state, etc.) must be handled explicitly. Unhandled edge cases cause crashes or wrong behavior.

### Pitfall 10: Not Documenting $F$

$F$'s interface, stages, and side effects should be documented. Without documentation, future changes will break the design.

---

## 6.2.6 Important Reminders

- **Decompose $F$ into generate, evaluate, select.** Each stage is independent and testable.
- **Use the simplest correct implementation first.** Optimize later.
- **Test each stage in isolation.** Then test $F$ end-to-end. Then test the engine loop.
- **Guarantee determinism.** Seeded RNG, encoded time, ordered collections.
- **Defer side effects until after the result is computed.** They affect future calls, not the current one.
- **Side effects must be thread-safe.** Atomics, locks, or per-thread buffers.
- **Do not optimize before profiling.** Get a correct slow engine first.
- **Document $F$'s interface, stages, and side effects.**

---

## 6.2.7 Summary

Phase 2 of the engine lifecycle is building the transition function $F$. The principles are:

1. **Modular functional composition.** $F$ decomposes into generate, evaluate, select. Each stage is a small, focused, testable function.
2. **Guaranteeing determinism.** $F$ produces the same output for the same input. Use seeded RNG, encoded time, ordered collections, and thread-safe side effects.
3. **The build process.** Define the interface, implement the stages (simply and correctly), test the stages, test $F$ end-to-end, test the engine loop, add optimizations, profile.
4. **The discipline of side effects.** Side effects must not affect the current call's output; they must be deferred until after the result is computed; they must be thread-safe.

Common pitfalls include monolithic $F$, premature optimization, non-deterministic behavior, side effects that affect the current call, untested stages, wrong abstraction level, inlined hot functions, unhandled edge cases, and missing documentation.

With a well-designed $F$, the engine is correct (if slow). Phase 3 (next note) covers structural optimizations to make it fast.
