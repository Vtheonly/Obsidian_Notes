# 4. Parser, Compiler, and Verification Engines

> "Parsing and compilation are the oldest engine domains — older than chess engines, older than search engines, older than trading engines. The first compiler was written in the 1950s; the techniques developed then (lexing, parsing, AST construction, code generation) are still in use today, refined by 70 years of theoretical and practical work. If you want to understand engines deeply, study compilers."

Parser and compiler engines transform *text* (source code, configuration, queries, protocols) into *executable meaning* (AST, bytecode, machine code, validated structures). They are unique among engine domains in that the input is fully specified upfront (no streaming, no real-time constraints) and correctness is paramount (a buggy compiler produces buggy programs).

This note covers the architecture of modern parser/compiler engines, with emphasis on the techniques that make them both correct and fast.

---

## 4.1 State Representation

The compiler's state evolves through several representations, each a transformation of the previous:

```mermaid
flowchart LR
    S[Source code: bytes] --> T[Token stream]
    T --> AST[Abstract Syntax Tree]
    AST --> IR[Intermediate Representation]
    IR --> OPT[Optimized IR]
    OPT --> CG[Generated code: bytecode / machine code]
```

### 4.1.1 Char Streams

The input is a sequence of bytes (or characters, after encoding decoding). The lexer reads from this stream, advancing a position pointer.

**Representation:** A `(pointer, length, position)` tuple — 24 bytes.

```c
struct CharStream {
    const char* data;
    size_t length;
    size_t position;
};
```

For zero-copy parsing, the data pointer references the original input buffer (e.g., a memory-mapped file). No copies are made.

### 4.1.2 Token Lists

The lexer produces a stream of tokens. Each token has a type (keyword, identifier, integer, operator, etc.) and a value (the substring of the input it represents).

**Representation:** A `(type, start_pos, end_pos, value_pointer)` tuple — 32 bytes per token.

```c
struct Token {
    TokenType type;       // 4 bytes
    uint32_t start_pos;   // 4 bytes
    uint32_t end_pos;     // 4 bytes
    const char* value;    // 8 bytes (pointer into source)
    uint32_t value_len;   // 4 bytes
    uint32_t padding;     // 4 bytes for alignment
};  // Total: 32 bytes
```

Tokens are stored in a **contiguous array** (not a linked list), arena-allocated. This gives O(1) random access and excellent cache behavior during parsing.

### 4.1.3 Abstract Syntax Trees (ASTs)

The AST is the hierarchical representation of the program's structure. Each node represents a syntactic construct (expression, statement, declaration).

**Representation:** Arena-allocated nodes with integer indices for children (not pointers).

```c
struct AstNode {
    AstNodeType type;        // 4 bytes
    uint32_t child1;          // 4 bytes (index into arena, 0 = no child)
    uint32_t child2;          // 4 bytes
    uint32_t child3;          // 4 bytes
    uint32_t source_loc;     // 4 bytes (for error messages)
    int64_t payload;         // 8 bytes (literal value, symbol ID, etc.)
};  // Total: 32 bytes
```

**Why arena + indices instead of pointers?**

1. **No heap allocation.** All nodes come from the arena; allocation is O(1) bump-pointer.
2. **No deallocation.** The arena is freed in bulk after compilation.
3. **No pointer chasing.** Indices are small integers; arithmetic on them is fast.
4. **Serializable.** The arena can be written to disk and read back; pointers cannot.
5. **Cache-friendly.** 32-byte nodes pack 2 per cache line; traversal is sequential.

### 4.1.4 Symbol Tables

The symbol table maps identifiers to their declarations. It is consulted during semantic analysis (type checking, scope resolution).

**Representation:** A hash table keyed by string ID (an integer representing the identifier's interned string).

```c
struct Symbol {
    uint32_t name_id;       // interned string ID
    SymbolKind kind;        // variable, function, type, etc.
    uint32_t type_id;       // type of the symbol
    uint32_t scope_id;      // scope in which it is declared
    uint32_t declaration_node; // AST node where it is declared
};

struct SymbolTable {
    Symbol* entries;        // hash table entries
    size_t capacity;
    size_t count;
};
```

**String interning:** All identifiers are interned — each unique string is stored once and assigned a 4-byte ID. Comparing identifiers becomes a 4-byte integer comparison instead of a string comparison. This is essential for fast semantic analysis.

### 4.1.5 Intermediate Representation (IR)

The IR is a lower-level representation of the program, between AST and machine code. It is simpler and more uniform than the AST, making it easier to optimize.

**Common IR forms:**

- **Three-address code (TAC).** Each instruction has at most three operands: two sources and one destination. Example: `t1 = a + b`.
- **Static Single Assignment (SSA).** Each variable is assigned exactly once. Example: `t1 = a + b; t2 = t1 * c`. SSA makes data flow explicit and enables powerful optimizations.
- **LLVM IR.** A specific SSA-based IR used by the LLVM compiler infrastructure. Widely used as a target IR for many languages (Rust, Swift, Clang/C++).

```c
// Example SSA IR instruction
struct IRInst {
    Opcode opcode;        // ADD, MUL, LOAD, STORE, BRANCH, etc.
    uint32_t dest;        // destination SSA value
    uint32_t operand1;    // source SSA value or literal
    uint32_t operand2;    // source SSA value or literal
    uint32_t type_id;     // type of the result
};
```

IR instructions are stored in **basic blocks** (sequences with no internal control flow). Basic blocks are stored in a flat array, with control-flow edges between them.

## 4.2 Transition Function $F$

The compiler's $F$ is a **multi-phase pipeline**: each phase transforms the representation from one form to the next.

### 4.2.1 Context-Free Grammar Parsing

The parser transforms the token stream into an AST according to a **context-free grammar (CFG)**. A CFG is a set of production rules:

```
expr    → expr '+' term | expr '-' term | term
term    → term '*' factor | term '/' factor | factor
factor  → NUMBER | IDENT | '(' expr ')'
```

**Parsing algorithms:**

- **LL(k) parsing.** Top-down, predictive. The parser decides which production to apply based on the next k tokens. Simple, fast, but limited (cannot handle left-recursive grammars).
- **LR(k) parsing.** Bottom-up, shift-reduce. The parser shifts tokens onto a stack and reduces when a production matches. Powerful, handles more grammars, but the parser tables are larger.
- **PEG (Parsing Expression Grammar).** A modern alternative with ordered choice. Easy to write by hand, but no left-recursion.
- **Recursive descent.** Hand-written parser where each grammar rule becomes a function. Most production compilers use this (GCC, Clang, V8) because it gives full control over error messages and performance.

```python
def parse_expr(tokens, pos):
    node, pos = parse_term(tokens, pos)
    while tokens[pos].type in (PLUS, MINUS):
        op = tokens[pos]
        right, pos = parse_term(tokens, pos + 1)
        node = BinaryOp(op, node, right)
    return node, pos
```

**Packrat parsing** is a memoized variant of PEG parsing. It guarantees linear-time parsing at the cost of memory (memoize every (rule, position) pair). Used in some modern parsers (e.g., Rust's macro parser).

### 4.2.2 Type Systems Resolution

After parsing, the compiler performs **semantic analysis**: type checking, scope resolution, variable resolution.

```python
def type_check(ast, symbol_table):
    for node in walk(ast):
        if node.type == BINARY_OP:
            left_type = type_check(node.left, symbol_table)
            right_type = type_check(node.right, symbol_table)
            node.result_type = resolve_binop(node.operator, left_type, right_type)
            if node.result_type is None:
                error(f"Cannot apply {node.operator} to {left_type} and {right_type}")
        elif node.type == IDENTIFIER:
            sym = symbol_table.lookup(node.name)
            if sym is None:
                error(f"Undefined identifier: {node.name}")
            node.result_type = sym.type
            node.resolved_symbol = sym
        # ... etc
```

Type checking is typically O(N) in the AST size, with constant-time lookups in the symbol table. More advanced type systems (Hindley-Milner type inference, dependent types) are more expensive.

### 4.2.3 Tree Walking and AST Traversal

Many compiler phases are tree walks over the AST. Common walks:

- **Pre-order.** Process the node, then its children. Used for declaration processing.
- **Post-order.** Process the children, then the node. Used for type checking (children's types are needed to determine the node's type).
- **In-order.** Process left child, then node, then right child. Used for expression evaluation.

Efficient tree walks use an **explicit stack** (avoiding recursion, which can overflow) and process nodes in cache-friendly order.

### 4.2.4 Intermediate Representation (IR) Optimization Passes

Once the AST is lowered to IR, the compiler applies a series of **optimization passes**, each transforming the IR to a more efficient form:

```mermaid
flowchart LR
    IR1[Initial IR] --> P1[Constant folding]
    P1 --> P2[Dead code elimination]
    P2 → P3[Common subexpression elimination]
    P3 → P4[Loop invariant code motion]
    P4 → P5[Strength reduction]
    P5 → P6[Inlining]
    P6 → P7[Register allocation]
    P7 → CG[Code generation]
```

**Common optimization passes:**

1. **Constant folding.** Evaluate constant expressions at compile time. `2 + 3` becomes `5`.
2. **Dead code elimination.** Remove code that can never execute (after `return`, unreachable branches).
3. **Common subexpression elimination (CSE).** If the same expression is computed multiple times, compute it once and reuse.
4. **Loop invariant code motion (LICM).** Move computations out of loops if their result does not change.
5. **Strength reduction.** Replace expensive operations with cheaper ones. `x * 2` becomes `x << 1`.
6. **Inlining.** Replace function calls with the function body. Eliminates call overhead and enables further optimizations.
7. **Register allocation.** Decide which values live in registers vs. memory. Use graph coloring or linear scan.

Each pass is typically O(N) in the IR size. With many passes, the total compile time can be O(N^2) or worse; production compilers limit the number of passes and use early-termination heuristics.

### 4.2.5 Code Generation

The final phase emits the target code — bytecode for a virtual machine, or machine code for a real CPU.

```python
def generate_code(ir):
    code = []
    for block in ir.blocks:
        for inst in block.instructions:
            if inst.opcode == ADD:
                code.append(emit_add(inst.dest, inst.operand1, inst.operand2))
            elif inst.opcode == LOAD:
                code.append(emit_load(inst.dest, inst.operand1))
            # ... etc
    return code
```

For machine code generation, the compiler uses a **instruction selector** (which picks the best instruction for each IR operation), a **register allocator** (which assigns physical registers to virtual registers), and an **instruction scheduler** (which reorders instructions for pipeline efficiency).

## 4.3 Dominant Optimizations

### 4.3.1 LL/LR Transition Tables

LL and LR parsers use precomputed **transition tables** that drive the parser. The table is computed once (at parser generation time) and stored as a 2D array indexed by (state, input symbol). Each entry specifies the next action: shift, reduce, accept, or error.

**Advantages:**

- Parsing is O(N) — each token is processed in O(1) (one table lookup).
- Tables are dense and cache-friendly.
- The same table-driven code works for any grammar.

**Disadvantages:**

- Tables can be large for complex grammars (megabytes).
- Error messages are notoriously bad (the parser knows *where* the error is, but not *what* the user intended).

Most production compilers (GCC, Clang) use **hand-written recursive descent parsers** instead, sacrificing some generality for better error messages and more control over performance.

### 4.3.2 Packrat Parsing with Memoization

Packrat parsing memoizes the result of each (rule, position) pair, guaranteeing linear-time parsing for PEG grammars. The cost is memory: O(N × R) where N is the input length and R is the number of rules.

**When to use:** when grammar simplicity matters more than memory. Common in prototype parsers and DSLs.

**When not to use:** for production compilers where memory and performance matter. Use hand-written recursive descent instead.

### 4.3.3 Arena Memory Allocation for Tree Nodes

AST and IR nodes are allocated from an **arena** — a large pre-allocated block of memory, with a bump-pointer allocator.

```c
struct Arena {
    char* block;
    size_t capacity;
    size_t used;
};

void* arena_alloc(Arena* a, size_t size) {
    // Align to 8 bytes
    size = (size + 7) & ~7;
    if (a->used + size > a->capacity) {
        // Grow arena (or fail)
        grow_arena(a);
    }
    void* ptr = a->block + a->used;
    a->used += size;
    return ptr;
}

void arena_free(Arena* a) {
    // Free everything at once
    free(a->block);
}
```

**Advantages:**

- Allocation is O(1) — one pointer bump.
- No fragmentation — all allocations are contiguous.
- No per-object free — the arena is freed in bulk.
- Excellent cache behavior — nodes are packed contiguously.

**Disadvantages:**

- Cannot free individual objects (only the whole arena).
- Arena grows monotonically — peak memory is high.

For compilers (where the entire AST is built and then discarded), arenas are ideal.

### 4.3.4 Memoization and Caching

Compilers cache the results of expensive computations:

- **Macro expansion.** Expanded macros are cached by their source location.
- **Template instantiation** (C++). Each template instantiation is cached by its template arguments.
- **Type inference** (Hindley-Milner). Type inference results are cached by AST node.

Without caching, these computations can be exponential (especially C++ templates). With caching, they are linear.

### 4.3.5 Parallel Compilation

Modern compilers parallelize at the file level (e.g., `make -j8` compiles 8 files in parallel). Within a single file, parallelization is harder due to data dependencies, but some passes (constant folding, dead code elimination) can be parallelized within basic blocks.

**Link-Time Optimization (LTO)** is a technique that performs optimization across compilation units, allowing inlining and other cross-module optimizations. LTO is expensive (link time increases 5–10×) but improves runtime performance by 5–15%.

## 4.4 Verification Engines

A related domain is **verification engines** — engines that prove properties of programs (e.g., "this function never returns negative", "this loop always terminates", "this protocol has no deadlocks").

### 4.4.1 SAT/SMT Solvers

SAT solvers determine whether a Boolean formula has a satisfying assignment. SMT (Satisfiability Modulo Theories) solvers extend SAT to theories (integers, arrays, bit vectors, etc.).

```python
# Example: verify that x * (y + z) == x*y + x*z for all integers
from z3 import *

x, y, z = Ints('x y z')
formula = (x * (y + z) != x*y + x*z)  # negation of the property
solver = Solver()
solver.add(formula)
result = solver.check()
if result == unsat:
    print("Property holds for all integers")
else:
    print("Counterexample:", solver.model())
```

SAT/SMT solvers use **conflict-driven clause learning (CDCL)**, a sophisticated backtracking algorithm that learns from conflicts to prune future search. Modern solvers (Z3, CVC4, Boolector) can solve formulas with millions of variables.

### 4.4.2 Model Checking

Model checking explores all states of a finite-state system, checking that a property holds in each state. Used for protocol verification, hardware verification, and embedded system verification.

The challenge is **state space explosion**: even small systems can have astronomically many states. Mitigation techniques:

- **Symbolic model checking** (BDDs). Represent sets of states compactly using Binary Decision Diagrams.
- **Partial order reduction.** Avoid exploring equivalent interleavings of concurrent operations.
- **Abstraction.** Verify an abstracted version of the system, then check that the abstraction is sound.

### 4.4.3 Abstract Interpretation

Abstract interpretation approximates the program's behavior by interpreting it over an **abstract domain** (e.g., intervals, signs, parity). It is conservative: if the abstract interpretation says a property holds, it holds; but it may report false positives.

Used by tools like Astree (verifies aerospace software), Infer (Facebook's static analyzer), and the Rust borrow checker (verifies memory safety).

## 4.5 Common Pitfalls

### Pitfall 1: Recursive Descent Without Backtracking Limits

Naive recursive descent can be exponential on ambiguous grammars (the parser tries every alternative). Use memoization (Packrat) or restructure the grammar to be unambiguous.

### Pitfall 2: Memory Leaks in Multi-Pass Compilers

Each pass allocates a new AST/IR. Without arenas, allocation dominates compile time. Use arenas; reset them between passes.

### Pitfall 3: Bad Error Messages

A compiler that says "syntax error" without context is unusable. Track source locations through every phase; produce error messages with file, line, column, and a snippet of the offending code.

### Pitfall 4: Phase Coupling

If phases are not cleanly separated (e.g., the parser does some type checking, the type checker does some optimization), the compiler becomes unmaintainable. Each phase should have a clear input and output; no phase should reach into another's internals.

### Pitfall 5: Insufficient Testing

Compilers must be correct. A buggy compiler produces buggy programs, which can cause real-world failures. Test extensively:

- **Unit tests** for each phase.
- **End-to-end tests** with known input/output pairs.
- **Differential testing** (compare output against another compiler).
- **Fuzz testing** (generate random inputs, verify the compiler does not crash).

### Pitfall 6: Slow Compile Times

Large projects (Chrome, LLVM itself) can take hours to compile. Slow compile times hurt developer productivity. Optimize compile time:

- Parallel compilation (`make -j`).
- Incremental compilation (only recompile changed files).
- Precompiled headers.
- Caching (ccache).

### Pitfall 7: IR Bloat

Each optimization pass can increase IR size (especially inlining). Without periodic IR compaction, compile time and memory usage grow unboundedly.

### Pitfall 8: Wrong Optimization Order

Some optimizations enable others (e.g., inlining enables constant folding). Putting optimizations in the wrong order can leave significant performance on the table. The right order is empirical — benchmark different orderings.

## 4.6 Important Reminders

- **Arena allocation for AST/IR nodes.** No per-node malloc.
- **Hand-written recursive descent for production parsers.** Best control over performance and error messages.
- **String interning for identifiers.** Fast symbol comparison.
- **SSA form for IR.** Enables most modern optimizations.
- **Multi-pass optimization.** Each pass does one thing well.
- **LTO for cross-module optimization.** Expensive but effective.
- **Test extensively.** Compiler bugs are catastrophic.
- **Track source locations through every phase.** Essential for error messages.
- **Parallel and incremental compilation.** For large projects.
- **SAT/SMT solvers for verification.** Industrial-strength tools exist (Z3, CVC4).

## 4.7 Summary

Parser and compiler engines transform text into executable meaning through a multi-phase pipeline: lexing, parsing, semantic analysis, IR generation, optimization, code generation. Each phase has a clear input and output, and the phases are connected by well-defined data structures (tokens, ASTs, IR).

The state evolves through these representations, each more abstract and more optimized than the previous. The transition function $F$ is the pipeline of phases, each implemented as a tree walk or a graph transformation.

Optimizations include arena allocation (no per-node malloc), string interning (fast identifier comparison), SSA form (enables modern optimizations), transition tables for LL/LR parsing, and memoization for ambiguous grammars.

Verification engines (SAT/SMT, model checking, abstract interpretation) extend the compiler's techniques to prove properties of programs, rather than just translate them.

The compiler architecture maps cleanly onto the six-layer model, with the multi-phase pipeline being the distinctive feature of the transition function layer.
