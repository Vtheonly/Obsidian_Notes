# 3. Vectorization and SIMD Libraries

> "SIMD is the hidden superpower of modern engines. A single AVX-512 instruction processes 16 floats in the same time a scalar instruction processes one — a 16× speedup, for free, if your data is laid out correctly. The libraries in this note make SIMD accessible without requiring you to write intrinsics."

This is the third note in Chapter 7. The previous notes covered languages and data structures; this note covers the libraries that make vectorization practical.

---

## 7.3.1 CPU Intrinsics and Vectorization Toolkits

### Intel Intrinsics (SSE, AVX, AVX-512)

Intel's SIMD instruction sets:

- **SSE (Streaming SIMD Extensions).** 128-bit registers. 4 floats or 2 doubles per instruction. Available on all x86-64 CPUs.
- **AVX / AVX2 (Advanced Vector Extensions).** 256-bit registers. 8 floats or 4 doubles per instruction. Available on most x86-64 CPUs since 2013.
- **AVX-512.** 512-bit registers. 16 floats or 8 doubles per instruction. Available on server x86 since 2017. Has quirks (frequency throttling on some CPUs).

**Intrinsics** are C/C++ functions that map directly to SIMD instructions. They are accessed via `<immintrin.h>` (x86) or `<arm_neon.h>` (ARM).

```c
#include <immintrin.h>

// Add 8 floats using AVX2
__m256 va = _mm256_loadu_ps(a);  // load 8 floats
__m256 vb = _mm256_loadu_ps(b);
__m256 vc = _mm256_add_ps(va, vb);  // add 8 floats in 1 instruction
_mm256_storeu_ps(c, vc);
```

**Pros:**
- Maximum control over code generation.
- Access to the full SIMD instruction set.
- Predictable performance.

**Cons:**
- Architecture-specific (AVX2 code does not run on ARM).
- Hard to read and maintain.
- Easy to introduce bugs (alignment, masking, etc.).

**When to use intrinsics:** only for the hottest loops where libraries are insufficient. For everything else, use libraries or auto-vectorization.

### ARM NEON

ARM's SIMD instruction set, available on all ARMv7+ processors (including all Apple Silicon, all Android phones, all AWS Graviton instances).

- 128-bit registers. 4 floats or 2 doubles per instruction.
- Similar functionality to SSE.

```c
#include <arm_neon.h>

// Add 4 floats using NEON
float32x4_t va = vld1q_f32(a);
float32x4_t vb = vld1q_f32(b);
float32x4_t vc = vaddq_f32(va, vb);
vst1q_f32(c, vc);
```

### SVE (ARM Scalable Vector Extension)

ARM's newer SIMD instruction set, available on ARMv9 and HPC processors. The key innovation: the vector length is **variable** (implemented by the CPU, not the programmer). The same code runs on 128-bit, 256-bit, 512-bit, or 2048-bit CPUs.

SVE is the future of ARM SIMD, but adoption is still limited (mainly HPC and server CPUs).

### SIMDe — Portable SIMD Abstraction

**SIMDe** is a header-only library that translates SIMD intrinsics across architectures. Write AVX2 code; SIMDe translates it to NEON on ARM, or vice versa.

```c
#include <simde/x86/avx2.h>

// This code uses AVX2 syntax but runs on any architecture
__m256 va = simde_mm256_loadu_ps(a);
__m256 vb = simde_mm256_loadu_ps(b);
__m256 vc = simde_mm256_add_ps(va, vb);
simde_mm256_storeu_ps(c, vc);
```

**Pros:**
- Write once, run on any architecture.
- Same code for x86 and ARM.
- No performance overhead (the translations are direct).

**Cons:**
- Does not abstract away differences in available operations (some AVX-512 operations have no NEON equivalent).
- Still uses intrinsic syntax (less readable than libraries).

**When to use SIMDe:** when you need cross-architecture SIMD with intrinsic-level control. For higher-level operations (linear algebra, JSON parsing), use specialized libraries.

### Highway (Google)

**Highway** is Google's portable SIMD library, providing a higher-level abstraction than SIMDe.

```cpp
#include <hwy/highway.h>
namespace hn = hwy::HWY_NAMESPACE;

void Add(const float* a, const float* b, float* c, size_t n) {
    for (size_t i = 0; i < n; i += hn::Lanes(hn::ScalableTag<float>())) {
        auto va = hn::Load(a + i);
        auto vb = hn::Load(b + i);
        auto vc = hn::Add(va, vb);
        hn::Store(vc, c + i);
    }
}
```

**Pros:**
- Higher-level than intrinsics.
- Portable across architectures.
- Supports "scalable" vectors (the vector length is determined at runtime).

**Cons:**
- Less mature than Intel intrinsics or SIMDe.
- Smaller community.

### std::simd (C++26)

The C++ standardization committee is working on `std::simd`, a portable SIMD abstraction in the standard library. It will be available in C++26 (or later).

```cpp
#include <simd>
namespace std::experimental::simd;

std::simd<float> a, b, c;
c = a + b;  // element-wise add
```

Until `std::simd` is widely available, use SIMDe or Highway for portable SIMD.

## 7.3.2 High-Performance Linear Algebra

For matrix and vector operations, dedicated linear algebra libraries are 10–100× faster than hand-written code.

### Eigen (C++)

**Eigen** is the dominant C++ linear algebra library. It is header-only (no linking required), expression-template-based (no temporaries), and supports SIMD automatically.

**Features:**
- Vectors, matrices, tensors of arbitrary size.
- All common linear algebra operations (multiply, invert, decompose, solve).
- SIMD auto-vectorization (uses SSE, AVX, NEON automatically).
- Expression templates (no temporaries for chained operations).
- Sparse matrices.
- Geometry module (quaternions, rotations).

**Usage:**
```cpp
#include <Eigen/Dense>
using namespace Eigen;

VectorXf a(1000), b(1000), c(1000);
// ... fill a and b ...
c = a + b;             // element-wise add, SIMD-vectorized
c = a.array() * b.array();  // element-wise multiply

MatrixXf M(100, 100);
VectorXf x(100), y(100);
y = M * x;  // matrix-vector multiply, SIMD-vectorized
```

**Performance:** Eigen uses SIMD automatically. For matrix multiplication, Eigen achieves > 90% of theoretical peak performance on modern CPUs.

**Use cases:**
- Machine learning (training and inference).
- Physics simulation.
- Computer graphics.
- Recommendation engines (similarity computations).

### Blaze (C++)

**Blaze** is another C++ linear algebra library, similar to Eigen but with a different design philosophy. Blaze is more focused on performance and less on generality.

**Pros:**
- Faster than Eigen for some operations (especially sparse matrices).
- Cleaner expression templates.

**Cons:**
- Smaller community than Eigen.
- Less feature-rich.

**Use cases:** same as Eigen; consider Blaze if Eigen's performance is insufficient.

### xtensor (C++)

**xtensor** is a C++ library inspired by NumPy. It provides N-dimensional arrays with NumPy-like syntax.

**Pros:**
- NumPy-like syntax (familiar to Python users).
- Supports SIMD via xtensor-blas.
- Interoperates with Python (via xtensor-python).

**Cons:**
- Slower than Eigen for pure linear algebra.
- Less mature.

**Use cases:** when NumPy-like syntax is important; when interoperating with Python.

### GSL (GNU Scientific Library)

**GSL** is a C library for scientific computing. It provides linear algebra, statistics, optimization, and more.

**Pros:** comprehensive; C interface (callable from many languages).
**Cons:** older; not as fast as Eigen for linear algebra.

## 7.3.3 Specialized SIMD Libraries

Beyond general linear algebra, specialized libraries provide SIMD-optimized implementations of specific operations.

### SIMDJSON (JSON Parsing)

**SIMDJSON** is a JSON parser that uses SIMD to achieve 1+ GB/s parsing speed. This is 10–100× faster than standard JSON parsers.

**Usage:**
```cpp
#include "simdjson.h"
using namespace simdjson;

ondemand::parser parser;
auto json = padded_string::load("data.json");
auto doc = parser.iterate(json);
for (auto item : doc["items"]) {
    std::string name = item["name"];
    int price = item["price"];
    // ...
}
```

**When to use:** whenever you parse JSON in the hot path. The speedup is dramatic.

### Hyperscan (Regex Matching)

**Hyperscan** is Intel's regex matching library, achieving 10+ GB/s for many patterns. Uses SIMD and automata-based matching.

**Use cases:**
- Network intrusion detection (Snort, Suricata).
- Log analysis.
- Filter pipelines.

**When to use:** when you need to match many regex patterns at high speed.

### simde (Fallback for Non-x86)

Already covered above. Use SIMDe when you need AVX/SSE code to run on ARM.

### OpenBLAS / MKL (BLAS Implementations)

**OpenBLAS** is an open-source BLAS (Basic Linear Algebra Subprograms) implementation. **MKL** (Intel Math Kernel Library) is Intel's proprietary BLAS, optimized for Intel CPUs.

Both provide highly optimized implementations of matrix multiply, vector operations, etc. They are typically faster than Eigen for large matrices.

**Use cases:**
- Large matrix operations (> 1000×1000).
- When you need BLAS-compatible API (e.g., for use with other scientific libraries).

**Eigen vs OpenBLAS/MKL:** for most engine use cases, Eigen is sufficient. For very large matrices, OpenBLAS or MKL may be faster.

## 7.3.4 Auto-Vectorization

Modern compilers (GCC, Clang) can auto-vectorize plain C/C++ loops. The loop must satisfy several conditions (Chapter 4.4).

### Enabling Auto-Vectorization

```bash
# GCC / Clang
g++ -O3 -march=native -fopt-info-vec ./engine.cpp
```

- `-O3` enables optimization level 3, which includes auto-vectorization.
- `-march=native` enables use of the host CPU's SIMD instruction sets.
- `-fopt-info-vec` (GCC) or `-Rpass=loop-vectorize` (Clang) prints which loops were vectorized.

### Helping the Compiler Vectorize

```c
// Use __restrict__ to indicate no aliasing
void add(float* __restrict__ a, float* __restrict__ b, float* __restrict__ c, int n) {
    for (int i = 0; i < n; i++) {
        c[i] = a[i] + b[i];
    }
}

// Use #pragma omp simd to force vectorization
#pragma omp simd
for (int i = 0; i < n; i++) {
    c[i] = a[i] + b[i];
}

// Use branchless operations
for (int i = 0; i < n; i++) {
    c[i] = std::max(std::min(a[i], hi), lo);  // branchless clamp
}
```

### Verifying Auto-Vectorization

Compile with `-fopt-info-vec` and verify that the hot loops were vectorized. If a loop was not vectorized, the compiler will print a reason (data dependency, aliasing, etc.).

For the truly curious, use Compiler Explorer (godbolt.org) to see the assembly output and verify SIMD instructions are used.

## 7.3.5 Common Pitfalls

### Pitfall 1: Hand-Writing Intrinsics When Libraries Suffice

Intrinsics are hard to read, hard to debug, and architecture-specific. Use libraries (Eigen, SIMDJSON, Hyperscan) wherever possible. Reserve intrinsics for the hottest loops where libraries are insufficient.

### Pitfall 2: Not Using Auto-Vectorization

Modern compilers are good at auto-vectorization. Use `-O3 -march=native` and let the compiler do its job. Verify with `-fopt-info-vec`.

### Pitfall 3: Forgetting `__restrict__`

Without `__restrict__`, the compiler must assume pointers alias and cannot vectorize. Always use `__restrict__` for distinct array parameters.

### Pitfall 4: Branches in Vectorized Loops

Branches prevent vectorization (or force expensive masked execution). Use branchless patterns.

### Pitfall 5: Not Using SoA Layout

SIMD requires consecutive memory. AoS layout cannot be vectorized. Always restructure to SoA first.

### Pitfall 6: AVX-512 Frequency Throttling

On some CPUs (Intel Skylake-X), AVX-512 instructions cause frequency throttling for several milliseconds. Benchmark on your target hardware.

### Pitfall 7: Not Benchmarking Different SIMD Widths

AVX-512 is not always faster than AVX2 (due to frequency throttling and lower clock speed). Benchmark both.

### Pitfall 8: Ignoring Alignment

Aligned loads are faster than unaligned. Align data to SIMD width boundaries (32 bytes for AVX2, 64 bytes for AVX-512).

### Pitfall 9: Not Using Libraries for Specialized Operations

For JSON parsing, use SIMDJSON (not a generic parser). For regex, use Hyperscan (not `std::regex`). For linear algebra, use Eigen (not hand-written loops).

## 7.3.6 Important Reminders

- **Use libraries (Eigen, SIMDJSON, Hyperscan) wherever possible.** Reserve intrinsics for the hottest loops.
- **Use auto-vectorization (`-O3 -march=native`).** Let the compiler do its job.
- **Use `__restrict__` and `#pragma omp simd` to help vectorization.**
- **Use SoA layout for SIMD-friendly data.**
- **Use branchless patterns in vectorized loops.**
- **Align data to SIMD width boundaries.**
- **Use SIMDe or Highway for cross-architecture portability.**
- **Benchmark different SIMD widths (AVX2 vs AVX-512).**
- **Verify auto-vectorization with `-fopt-info-vec`.**

---

## 7.3.7 Summary

SIMD is the hidden superpower of modern engines, providing 4–16× speedup for data-parallel operations. The libraries in this note make SIMD accessible:

- **Intrinsics (Intel SSE/AVX/AVX-512, ARM NEON/SVE)** for maximum control.
- **SIMDe and Highway** for cross-architecture portability.
- **Eigen, Blaze, xtensor** for linear algebra.
- **SIMDJSON** for JSON parsing (1+ GB/s).
- **Hyperscan** for regex matching (10+ GB/s).
- **OpenBLAS / MKL** for large matrix operations.

Auto-vectorization (`-O3 -march=native`) handles many cases automatically. Help the compiler with `__restrict__`, `#pragma omp simd`, SoA layout, and branchless patterns. Verify vectorization with `-fopt-info-vec`.

Reserve hand-written intrinsics for the hottest loops where libraries are insufficient. Always benchmark; AVX-512 is not always faster than AVX2.
